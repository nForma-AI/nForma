## Integration Findings

### INTEGRATION-01 — Hardcoded Path (HIGH · Distribution Blocker)

**Affected:** WIZ-01..05, KEY-01..04, PROV-03, AGENT-01, AGENT-03

All 9 `require('/Users/jonathanborduas/code/QGSD/bin/secrets.cjs')` calls in `commands/qgsd/mcp-setup.md` use an absolute developer machine path. When any other user installs QGSD, every keytar operation (key set/get, syncToClaudeJson, provider swap) will throw a module-not-found error.

```
Lines: 119, 292, 358, 452, 565, 638, 700, 790, 1036
Fix: Resolve path via installed package location (e.g., __dirname-relative from a wrapper script, or package the secrets.cjs under a stable install path)
```

**Current impact:** None on developer machine. **Distribution impact:** All keytar flows broken.

### INTEGRATION-02 — Provider Swap Missing syncToClaudeJson (LOW · Consistency)

**Affected:** PROV-03

Option 2 (Swap Provider) patches `ANTHROPIC_BASE_URL` correctly and invokes `mcp-restart`. However, it does not call `syncToClaudeJson` after writing — the only `apply` path in the command that omits this step. Every other apply path (first-run, Option 1 key update, add-agent, shared confirm+apply) calls it.

```
Lines: 889-931 — add: syncToClaudeJson(SERVICE).then(...) after backup+patch, before mcp-restart
```

**Risk:** If keytar and the env block are out of sync (e.g., key was stored via keytar but env block was empty), a provider swap will not re-sync the key. Low probability in practice since key flows call syncToClaudeJson, but the inconsistency creates a latent maintenance hazard.

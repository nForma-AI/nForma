# Phase v0.25-02: Property Annotations - Research

**Researched:** 2026-03-03
**Domain:** Formal model annotation for requirement traceability
**Confidence:** HIGH

## Summary

Phase v0.25-02 annotates every formal property/invariant/assertion across all 22 model files (11 TLA+, 8 Alloy, 3 PRISM .props) with structured `@requirement` comments that a parser can extract. The project already has a complete property-to-requirement mapping in TRACEABILITY_RESEARCH.md Section 5 (~80 properties mapped across all three formalisms), existing ad-hoc requirement references in many model comments, and model-level requirement arrays in model-registry.json (from v0.25-01). The work is mechanical: insert `@requirement REQ-ID` annotations before each property definition using the correct comment syntax per formalism, then build `bin/extract-annotations.cjs` to parse them into a JSON map.

**Primary recommendation:** Use the property-to-requirement map from TRACEABILITY_RESEARCH.md Section 5 as the authoritative source for annotation placement. The comment format per formalism is: `\* @requirement REQ-ID` (TLA+), `-- @requirement REQ-ID` (Alloy), `// @requirement REQ-ID` (PRISM). The extraction parser should handle multi-requirement annotations (one `@requirement` line per ID) and output `{ model_file: [{ property, requirement_ids }] }`.

## Architecture Patterns

### Annotation Format Per Formalism

**TLA+ (`.tla` files):**
```tla
\* @requirement STOP-02
\* @requirement SPEC-01
SafetyInvariant1 ==
  decision = "BLOCK" => hasCommand
```

**Alloy (`.als` files):**
```alloy
-- @requirement INST-01
assert NoConflict {
    all s: InstallState | NoConflictingScope[s]
}
```

**PRISM (`.props` files):**
```prism
// @requirement PRM-01
// @requirement QUORUM-04
// P1: Eventual convergence -- must be exactly 1.0
P=? [ F s=1 ]
```

### Property Detection Patterns

The parser must identify property definitions to associate annotations with property names:

| Formalism | Property Pattern | Example |
|-----------|-----------------|---------|
| TLA+ invariant | `PropertyName ==` at start of line | `SafetyInvariant1 ==` |
| TLA+ temporal | `PropertyName ==` (same as invariant) | `LivenessProperty1 == <>algorithmDone` |
| TLA+ TypeOK | `TypeOK ==` at start of line | `TypeOK ==` |
| Alloy assertion | `assert AssertionName {` | `assert NoConflict {` |
| Alloy check | `check AssertionName` (refers to assertion) | `check NoConflict for 3...` |
| PRISM property | Property formula line (non-comment, non-blank) | `P=? [ F s=1 ]` |

### Anti-Patterns to Avoid
- **Renaming properties to encode requirement IDs:** ROADMAP explicitly excludes this ("Disruptive change across all models; annotations solve the same problem non-invasively"). Use comments only.
- **Modifying model semantics:** Annotations are comments only. Never change the actual property formulas.
- **Annotating TTrace files:** TTrace files are TLC trace replay artifacts, not model specifications. Skip all `*_TTrace_*.tla` files.
- **Annotating `.pm` files:** The `.pm` files are PRISM model definitions (state machine). The `.props` files contain the property specifications. Only `.props` files need property annotations. (`.pm` files may get model-level annotations but that's not in scope for ANNOT-01..03.)

## Common Pitfalls

### Pitfall 1: Missing properties in annotation pass
**What goes wrong:** A property exists in the model but gets no `@requirement` annotation because it was missed in the scan.
**Why it happens:** Manual editing of 22 files with 90 properties is error-prone.
**How to avoid:** After annotations are placed, run `extract-annotations.cjs` and compare its output against the known property list from TRACEABILITY_RESEARCH.md Section 5. Any property without an annotation is a gap.
**Warning signs:** The extraction parser returns fewer properties than expected.

### Pitfall 2: Wrong comment syntax per formalism
**What goes wrong:** Using `//` in TLA+ or `\*` in Alloy.
**Why it happens:** Context switching between three different comment syntaxes.
**How to avoid:** TLA+ uses `\*`, Alloy uses `--`, PRISM uses `//`. The parser must match the correct syntax per file type.
**Warning signs:** Parser returns empty results for a file type.

### Pitfall 3: Annotation not immediately preceding the property
**What goes wrong:** An `@requirement` comment exists in the file but is separated from the property by blank lines or other comments, so the parser cannot associate them.
**Why it happens:** Placing annotations in the file header instead of immediately before each property.
**How to avoid:** Each `@requirement` line must be in the comment block directly above the property definition. No blank lines between the annotation and the property.
**Warning signs:** Parser shows properties with no requirement_ids despite annotations existing in the file.

### Pitfall 4: Generated model annotation drift
**What goes wrong:** `quorum.props` and `quorum-votes.als` are generated files. Annotations added by hand will be overwritten on regeneration.
**Why it happens:** `node bin/generate-formal-specs.cjs` regenerates these files.
**How to avoid:** For generated files, annotations must be added to the generator template so they survive regeneration. Check `bin/generate-formal-specs.cjs` to see which files are generated. Alternatively, add a comment in the generated file header noting that annotations were added post-generation and should not be regenerated.
**Warning signs:** After running the generator, annotations disappear.

### Pitfall 5: TUINavigation properties have no requirement IDs
**What goes wrong:** TRACEABILITY_RESEARCH.md Section 5.1 shows TUINavigation properties (NoDeadlock, DepthBounded, EscapeProgress) with no matching requirement IDs — just descriptions in parentheses.
**Why it happens:** The TUI-related requirements may not exist in requirements.json, or the mapping is unclear.
**How to avoid:** Check requirements.json for VIS-* or WIZ-* requirements that could map. If no clear mapping exists, annotate with the closest reasonable requirement ID or leave a `\* @requirement UNMAPPED` marker for the traceability matrix to flag.
**Warning signs:** Properties with no requirement IDs after annotation.

### Pitfall 6: QGSDQuorum_xstate.tla
**What goes wrong:** This file is listed as a TLA+ model but is an unbounded generated scaffold that cannot be model-checked.
**Why it happens:** It was generated from XState but uses unbounded Nat domains.
**How to avoid:** The model-registry.json includes it as a model. It has properties that should be annotated even though TLC cannot check it. Annotate like any other TLA+ file. The extraction parser does not need to run TLC — it only reads comments.
**Warning signs:** None for annotation purposes — just be aware it exists.

## Code Examples

### extract-annotations.cjs core parsing logic

```javascript
// Pattern for TLA+ @requirement annotation
const TLA_ANNOTATION_RE = /^\s*\\\*\s*@requirement\s+([\w-]+)/;
// Pattern for TLA+ property definition
const TLA_PROPERTY_RE = /^(\w+)\s*==/;

// Pattern for Alloy @requirement annotation
const ALLOY_ANNOTATION_RE = /^--\s*@requirement\s+([\w-]+)/;
// Pattern for Alloy assertion definition
const ALLOY_ASSERT_RE = /^assert\s+(\w+)\s*\{/;

// Pattern for PRISM @requirement annotation
const PRISM_ANNOTATION_RE = /^\/\/\s*@requirement\s+([\w-]+)/;
// Pattern for PRISM property (non-comment, non-blank line)
const PRISM_PROPERTY_RE = /^[A-Z]/; // P=?, R{...}=?, S=? all start with uppercase
```

## Open Questions

1. **Generated file annotation strategy:** `quorum.props`, `quorum-votes.als`, and `QGSDQuorum.tla` are generated by `bin/generate-formal-specs.cjs`. Should annotations be added to the generator templates, or manually post-generation with a "do not regenerate" warning? Recommendation: Add annotations manually with a header comment noting they are not generated; update the generator to preserve existing annotations on regeneration (future enhancement).

2. **TUINavigation unmapped properties:** NoDeadlock, DepthBounded, EscapeProgress have no clear requirement IDs. Recommendation: Create a simple mapping to the closest VIS-* or WIZ-* requirement if available, or use a descriptive ID like `TUI-NAV-01` etc.

3. **TypeOK annotation approach:** TypeOK invariants are structural, not domain requirements. The TRACEABILITY_RESEARCH maps them to type-related requirement IDs (e.g., QUORUM-01, DETECT-01). Recommendation: Annotate TypeOK with the mapped ID from Section 5 — they do verify structural correctness of the domain.

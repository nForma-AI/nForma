## Summary

Both v0.13-01 and v0.13-02 were fully executed — all 8 requirements have confirmed implementations in the workflow files — but neither phase produced a VERIFICATION.md artifact. The milestone audit (`v0.13-MILESTONE-AUDIT.md`) scored 0/8 because the QGSD trust+audit model requires formal verification documents, and "inline grep checks in SUMMARY do not constitute formal verification." This phase writes the missing artifacts and updates REQUIREMENTS.md and ROADMAP.md bookkeeping.

The task is entirely document authoring, not implementation. All 8 implementations are present and verifiable via grep against the source files right now. The planner's job is to direct an executor to: (1) read each workflow file, (2) extract the exact lines implementing each requirement, (3) write VERIFICATION.md files using the established project format, and (4) update the two bookkeeping files. No code is written.

The verification artifacts must follow the format established in existing VERIFICATION.md files (e.g., `v0.9-05-VERIFICATION.md`, `v0.10-04-VERIFICATION.md`): YAML frontmatter with phase/status/verified fields, a Requirements Check table, a Verification Commands section with actual grep output, and per-requirement evidence citations. The v0.13 verification is entirely grep-based with no executable tests — all verification is manual inspection of markdown workflow files.

**Primary recommendation:** Write two VERIFICATION.md files (one per phase) using the project's established format, with per-requirement checklists citing specific file paths and line numbers, then update REQUIREMENTS.md traceability and ROADMAP.md progress in a single bookkeeping plan.

The success criteria uses the bare `VERIFICATION.md` name (no phase prefix). The audit milestone checker likely checks for this exact path. The planner must decide: use the project convention (`v0.13-01-VERIFICATION.md`) or use the ROADMAP success criteria format (`VERIFICATION.md`).

**Resolution for planner:** Use `VERIFICATION.md` (no prefix) as specified in the ROADMAP success criteria. If the milestone audit script uses glob matching it will find both formats. The explicit success criteria check is the binding specification.

## Common Pitfalls

### Pitfall 1: Missing the REQUIREMENTS.md Checkboxes

**What goes wrong:** Updating the traceability table (bottom of REQUIREMENTS.md) but forgetting the top checkbox list. Both must be updated.

**Why it happens:** The traceability table has an explicit "Status" column so it's obvious. The checkboxes at the top look like "requirements spec" rather than status tracking.

**How to avoid:** Read REQUIREMENTS.md fully before editing. The checkboxes `- [ ] **LOOP-01**:` must become `- [x] **LOOP-01**:` for all 8 requirements.

**Warning signs:** After the edit, `grep -c '- \[ \]' .planning/REQUIREMENTS.md` returns non-zero for v1 requirements.

### Pitfall 2: ROADMAP.md Progress Table Row Format Inconsistency

**What goes wrong:** The Progress table has inconsistent column counts across rows (some rows have 4 columns, others 5). Editing without matching the existing row format corrupts the table.

**Why it happens:** The table was built incrementally; some rows have a trailing `| - |` (link column), others don't.

**How to avoid:** Look at the existing v0.13-02 row (`| v0.13-02. Quorum Gates | v0.13 | 0/3 | Planned | - |`) and match its exact column count when updating.

**Current v0.13-02 row (line 683):**
```
| v0.13-02. Quorum Gates | v0.13 | 0/3 | Planned | - |
```
**Target:**
```
| v0.13-02. Quorum Gates | v0.13 | 3/3 | Complete | 2026-02-25 |
```

### Pitfall 3: Omitting the `verified:` Date in Frontmatter

**What goes wrong:** The VERIFICATION.md frontmatter lacks a `verified:` field, causing the milestone audit to not recognize the file as formally verified.

**Why it happens:** Copying the v0.10-04 format which uses a longer timestamp format (`verified: 2026-02-25T00:00:00Z`).

**How to avoid:** Include `verified: 2026-02-25` in YAML frontmatter. Use the date format established in v0.9-05 (`2026-02-25`) for simplicity, or ISO timestamp — both are found in the project.

### Pitfall 4: Citing the Installed Copy Instead of Source Copy

**What goes wrong:** Evidence citations point to `~/.claude/qgsd/workflows/transition.md` (installed copy) instead of `qgsd-core/workflows/transition.md` (source copy). The installed copy is a deployment artifact; the source copy is the authoritative version.

**Why it happens:** grep checks during plan execution may run against either copy.

**How to avoid:** All evidence citations in VERIFICATION.md must point to the `qgsd-core/` source paths, not the `~/.claude/` installed paths.

### Pitfall 5: Wrong Status Value in REQUIREMENTS.md Traceability

**What goes wrong:** Updating the Status field to something other than `Complete` (e.g., "Done", "Verified", "PASSED").

**Why it happens:** The field value isn't specified in the table header.

**How to avoid:** The ROADMAP success criteria specifies the exact value: `Status = Complete`. Use exactly that string.

### Pitfall 6: Forgetting to Check the v0.13-02 Phase Checkbox (vs. Progress Table)

**What goes wrong:** Updating only the Progress table but not the `- [ ] **Phase v0.13-02:**` line in the v0.13 phase list (lines 168-175 area).

**Why it happens:** The checkbox and the Progress table are in different sections of ROADMAP.md.

**How to avoid:** The ROADMAP success criteria SC4 says "Phase v0.13-02 checkbox is `[x]`" — this refers to the `- [ ]` → `- [x]` at line 173. Both updates needed: line 173 (checkbox) and line 683 (Progress table).

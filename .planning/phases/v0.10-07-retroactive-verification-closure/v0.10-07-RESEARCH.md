### Pattern 3: `probeAllSlots` Unit Tests

`probeAllSlots(mcpServers, slots, secretsLib)` takes a mcpServers object, a slot list, and an optional secretsLib. Its key testable behaviors:

1. Slots without `ANTHROPIC_BASE_URL` in env immediately return the subprocess sentinel `{ healthy: null, error: 'subprocess' }` — no HTTP call, no await on probe
2. Returns an object keyed by slot name (via `Object.fromEntries`)
3. Result contains one entry per slot (length matches slots array)

Test approach using the optional `filePath` pattern: pass mock `mcpServers` with no `ANTHROPIC_BASE_URL` to force the subprocess fast-path. No HTTP calls needed for these unit tests.

```js
// Source: bin/manage-agents.cjs lines 1297-1316
test('probeAllSlots: subprocess slots return healthy:null without HTTP probe', async () => {
  const { probeAllSlots } = _pure;
  const mockMcpServers = {
    'subprocess-slot': { command: 'node', args: ['server.js'], env: {} },
  };
  const result = await probeAllSlots(mockMcpServers, ['subprocess-slot'], null);
  assert.strictEqual(result['subprocess-slot'].healthy, null);
  assert.strictEqual(result['subprocess-slot'].error, 'subprocess');
});

test('probeAllSlots: returns one entry per slot', async () => {
  const { probeAllSlots } = _pure;
  const mockMcpServers = {
    'slot-a': { env: {} },
    'slot-b': { env: {} },
  };
  const result = await probeAllSlots(mockMcpServers, ['slot-a', 'slot-b'], null);
  assert.strictEqual(Object.keys(result).length, 2);
});

test('probeAllSlots: empty slots returns empty object', async () => {
  const { probeAllSlots } = _pure;
  const result = await probeAllSlots({}, [], null);
  assert.deepStrictEqual(result, {});
});
```

### Pattern 4: `liveDashboard` Integration Smoke Test

`liveDashboard()` is TTY-dependent in production but can be driven through its non-TTY branch (which runs when `process.stdout.isTTY` is falsy). In CI/test contexts `process.stdout.isTTY` is typically `undefined` (falsy), so the non-TTY branch fires automatically.

The non-TTY code path (lines 1488-1494):
```js
if (!process.stdout.isTTY) {
  console.log('\n  Health check (non-TTY mode):\n');
  const healthMap = await probeAllSlots(mcpServers, slots, secretsLib);
  const lines = buildDashboardLines(slots, mcpServers, healthMap, Date.now());
  console.log(lines.join('\n'));
  return;
}
```

**Smoke test approach — non-TTY path (testable without mock stdin):**

The non-TTY code path is the correct smoke test target. It tests:
- Entry: `liveDashboard()` resolves without throwing
- Data flow: `probeAllSlots` is called and result passed to `buildDashboardLines`
- Exit: function returns (no infinite loop, no hanging promise)

Since the full TTY raw-mode loop (keypress refresh, exit on q) cannot be exercised without a real TTY or a complex stdin mock, the non-TTY path covers the smoke test requirement without requiring interactive TTY. The TTY behaviors are already covered by the manual-only verification section in v0.10-04-VALIDATION.md.

**Implementation:**

```js
test('liveDashboard smoke: non-TTY path resolves without throwing', async () => {
  const { liveDashboard } = _pure;
  // In node:test CI context, process.stdout.isTTY is falsy — non-TTY branch fires
  // Guard: skip if somehow running in a real TTY
  if (process.stdout.isTTY) {
    // mark as manual-only; return early
    return;
  }
  // Patch readClaudeJson to return empty mcpServers so no HTTP probes fire
  // liveDashboard reads claude.json internally — use a temp file approach
  // OR: mock via a _pure override if readClaudeJson is exported
  // SIMPLEST: run as-is; with empty real ~/.claude.json, probeAllSlots returns {}
  await assert.doesNotReject(liveDashboard());
});
```

**Important caveat on mock stdin for the TTY path:** Driving the raw-mode keypress loop requires either (a) a real PTY, (b) a subprocess spawned with `pty.js`, or (c) patching `process.stdin` to a `PassThrough` stream emitting keypress events. Option (c) is technically possible but dangerous — `process.stdin.setRawMode(true)` will throw if stdin is not a TTY, and replacing `process.stdin` globally in a test module causes interference between tests. The non-TTY fallback path is the safe, correct smoke test target. The TTY interactive path (keypress refresh, q exit) is explicitly marked manual-only in VALIDATION.md.

**If the success criterion requires testing entry, keypress refresh, and exit flow:** A subprocess-based approach works:

```js
// Alternative: spawn manage-agents.cjs as child process in non-TTY mode, pipe stdin
// Verify it prints health output and exits cleanly
test('liveDashboard smoke: subprocess non-TTY prints table and exits 0', (t, done) => {
  const child = require('child_process').spawn(
    process.execPath,
    [path.join(__dirname, 'manage-agents.cjs')],
    { stdio: ['pipe', 'pipe', 'pipe'] }  // non-TTY: stdout.isTTY = false
  );
  let output = '';
  child.stdout.on('data', (d) => { output += d; });
  child.on('close', (code) => {
    assert.strictEqual(code, 0);
    assert.ok(output.includes('Health check') || output.includes('No slots'), 'must print health output');
    done();
  });
  child.stdin.end();  // close stdin so liveDashboard can exit
});
```

This subprocess approach is the recommended pattern for the integration smoke test because it:
- Does not touch `process.stdin`/`process.stdout` in the test process
- Tests the real entry point routing (`const entry = process.stdout.isTTY ? mainMenu : liveDashboard`)
- Confirms the process exits with code 0
- Works in CI with no TTY

### Anti-Patterns to Avoid

- **Writing VERIFICATION.md without line-number evidence:** Every "WIRED" assertion in a verification table must cite a file line reference or commit hash. Assertions without evidence are not verifiable and fail the audit trail purpose.
- **Claiming `probeAllSlots` is a "pure function":** It is async and performs HTTP calls. Document it accurately in VERIFICATION.md as an "async integration function exported for test access" — not a pure function.
- **Using `mock.fn()` or `sinon` to stub `probeProviderUrl` inside `probeAllSlots` tests:** The project has zero external test dependencies. Use the subprocess fast-path (no `ANTHROPIC_BASE_URL`) to avoid HTTP calls in unit tests.
- **Running the TTY raw-mode path in `node:test`:** `process.stdin.setRawMode(true)` throws `ERR_INVALID_HANDLE_TYPE` when stdin is a pipe. The smoke test MUST target the non-TTY branch or use a subprocess.
- **Numbering VERIFICATION.md requirements as "SATISFIED" when human interaction was never performed:** Use `human_needed` status for TTY-interactive flows, consistent with v0.10-05's pattern.

## Common Pitfalls

### Pitfall 1: probeAllSlots `_pure` export breaks the module load
**What goes wrong:** `probeAllSlots` and `liveDashboard` are async functions defined before the `_pure` block — adding them to `_pure` should work. But if there's a hoisting issue (e.g., if either function were defined as `const` with a `require()` inside), it might not be available at module load time.
**Why it happens:** Both functions are defined with `async function` declarations (not `const` arrow functions), so they are hoisted and available at `module.exports._pure` assignment time.
**How to avoid:** After adding the exports, run `node -e "const {_pure}=require('./bin/manage-agents.cjs'); console.log(typeof _pure.probeAllSlots, typeof _pure.liveDashboard);"` — expect `function function`. This is the existing spot-check pattern from v0.10-03.
**Warning signs:** `undefined undefined` output from the spot-check.

### Pitfall 2: liveDashboard smoke test hangs in TTY context
**What goes wrong:** If `process.stdout.isTTY` is truthy (e.g., developer runs tests directly in terminal), `liveDashboard()` enters raw mode and waits for a keypress — the test hangs indefinitely.
**Why it happens:** The TTY guard inside `liveDashboard` is `if (!process.stdout.isTTY)` — when TTY is present, it takes the raw mode path.
**How to avoid:** Add a TTY guard at the top of the smoke test: `if (process.stdout.isTTY) { return; }`. For the subprocess variant, stdio: 'pipe' ensures `stdout.isTTY` is always false in the child.
**Warning signs:** Test suite hangs after the smoke test is added.

### Pitfall 3: VERIFICATION.md "score" line is inaccurate
**What goes wrong:** Writing `score: 4/4 automated truths verified` when some truths require human confirmation.
**Why it happens:** Copying the frontmatter format from Phase 18 (which had all-automated verification) into a phase with TTY-only flows.
**How to avoid:** Use `status: human_needed` when any truth requires human confirmation. Document each human-needed test in the `human_verification` YAML block. This is consistent with v0.10-05's VERIFICATION.md.

### Pitfall 4: Menu renumber breaks downstream documentation references
**What goes wrong:** Renaming `'13. Live health dashboard'` to `'12.'` causes the v0.10-05-VERIFICATION.md to reference the wrong number (it says `choice 15` for tune-timeouts, `choice 16` for set-update-policy).
**Why it happens:** Menu items are referenced by number in VERIFICATION.md prose and test expectations.
**How to avoid:** After renumbering: v0.10-05's numbers shift by -1 for items after position 12. Check v0.10-05-VERIFICATION.md for hardcoded references to item numbers 13-16 and update them in the v0.10-05 document as part of this phase.
**Status:** After the fix, the correct numbers will be: 12=Live health dashboard, 13=Update coding agents, 14=Tune timeouts, 15=Set update policy.

### Pitfall 5: VERIFICATION.md for v0.10-02 conflates human-verify checkpoint with final sign-off
**What goes wrong:** The v0.10-02 SUMMARY-02 has a `## Human Verification Gate (checkpoint:human-verify)` section with 4 tests listed as "Pending". If the VERIFICATION.md treats these as unverified, the status must be `human_needed`.
**Why it happens:** The checkpoint was flagged but no approval was recorded. Per the audit, this is "human-verify checkpoint was flagged in SUMMARY-02 but no approval recorded and no VERIFICATION.md created."
**How to avoid:** The VERIFICATION.md for v0.10-02 MUST use `status: human_needed` (not `passed`). The 4 TTY-interactive test cases from SUMMARY-02 are the human verification items.

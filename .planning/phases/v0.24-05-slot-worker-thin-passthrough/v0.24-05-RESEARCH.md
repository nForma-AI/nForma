## Summary

The current `qgsd-quorum-slot-worker` agent burns 22–25k tokens per dispatch because it does four things that an LLM shouldn't do: parse a structured YAML argument block, construct a long conditional prompt with seven independent `if` branches, call a single Bash command, and parse the raw output back into structured YAML. Only step 3 (the Bash call) requires an LLM. The other three are pure deterministic string operations that belong in JavaScript.

The solution is a new wrapper script — `bin/quorum-slot-dispatch.cjs` — that accepts structured CLI arguments (`--slot`, `--mode`, `--round`, `--question`, `--artifact-path`, `--review-context`, `--prior-positions-file`, `--traces-file`, `--request-improvements`), builds the prompt deterministically, calls `call-quorum-slot.cjs` via stdin pipe, parses the output, and emits structured YAML. The `qgsd-quorum-slot-worker` agent is then reduced to a thin shim: one Bash call forwarding its YAML arguments to the script, and one `cat` to emit the script's stdout verbatim.

A new wrapper script (`quorum-slot-dispatch.cjs`) is strongly preferred over extending `call-quorum-slot.cjs` directly. `call-quorum-slot.cjs` is a focused, well-tested transport layer; mixing prompt construction into it would conflate concerns and complicate the retry logic. The wrapper composes with it by piping stdin.

**Primary recommendation:** Create `bin/quorum-slot-dispatch.cjs` as a prompt-construction + output-parsing wrapper that pipes to `call-quorum-slot.cjs` via stdin. Reduce the agent to a one-liner Bash call. Use tempfiles for `--prior-positions` and `--traces` content to sidestep shell quoting issues.

### Pattern 1: Wrapper Script with Stdin Pipe to call-quorum-slot.cjs

**What:** `quorum-slot-dispatch.cjs` accepts structured CLI args, builds the prompt, spawns `call-quorum-slot.cjs` as a child process with the prompt written to its stdin, captures stdout, parses it, and emits structured YAML.

**When to use:** Any time prompt construction or output parsing needs to be deterministic.

**Key design points:**
- The prompt is assembled as a string in JS, then piped to CQS via `child.stdin.write(prompt); child.stdin.end()`
- `call-quorum-slot.cjs` already reads from stdin via `readStdin()` — no change needed to that file
- The wrapper's output is a YAML block matching the existing result block spec:
  ```yaml
  slot: <slotName>
  round: <N>
  verdict: <...>
  reasoning: <...>
  citations: |
    <optional>
  improvements:
    - suggestion: "..."
      rationale: "..."
  raw: |
    <first 5000 chars>
  ```

**Example wrapper spawn pattern:**
```javascript
// Source: call-quorum-slot.cjs runSubprocess pattern (adapted)
const cqsPath = path.join(__dirname, 'call-quorum-slot.cjs');
const child = spawn(process.execPath, [cqsPath, '--slot', slot, '--timeout', timeout, '--cwd', cwd], {
  stdio: ['pipe', 'pipe', 'pipe']
});
child.stdin.write(prompt, 'utf8');
child.stdin.end();
let stdout = '';
child.stdout.on('data', d => { stdout += d; });
child.on('close', (code) => { /* parse stdout */ });
```

### Pattern 2: Tempfile for Long String Arguments

**What:** For `--prior-positions` and `--traces` content (multi-line YAML that can be 1–30KB and contains special shell characters), write content to a tempfile and pass `--prior-positions-file <path>` instead of the raw string.

**When to use:** Any content that contains newlines, quotes, or backslashes that would require complex shell escaping.

**Example:**
```javascript
// In the wrapper script: read prior positions from tempfile
const priorPositionsFile = getArg('--prior-positions-file');
const priorPositions = priorPositionsFile ? fs.readFileSync(priorPositionsFile, 'utf8') : null;
```

**In the agent (thin shim):**
```bash
# Write prior_positions to tempfile, pass path
TMPFILE=$(mktemp /tmp/qgsd-prior-XXXXXX.txt)
printf '%s' "$PRIOR_POSITIONS_CONTENT" > "$TMPFILE"
node "$HOME/.claude/qgsd-bin/quorum-slot-dispatch.cjs" \
  --slot gemini-1 \
  --mode A \
  --round 2 \
  --question "Does this plan address all requirements?" \
  --prior-positions-file "$TMPFILE"
rm -f "$TMPFILE"
```

**Why tempfile over arg:** macOS ARG_MAX is 1MB (confirmed: 1048576 bytes). Mode A prompts with 3 prior positions are ~5KB — well within limits. Mode B with large traces can be 35KB+ — still within ARG_MAX but shell quoting for multi-line YAML with arbitrary content is fragile. Tempfiles eliminate quoting complexity at zero cost.

**Alternative simpler approach:** Pass all content via stdin to the wrapper itself. The wrapper reads stdin as the "input bundle" (JSON or delimited text), constructs the prompt, then pipes to CQS. This avoids tempfiles entirely but requires a serialization format. Given that the agent already constructs a YAML block, tempfiles are more readable.

### Pattern 3: Prompt Construction as Pure Functions

**What:** Define each conditional section as a JavaScript function that returns a string or empty string, then join. This is directly testable without any mock infrastructure.

**Example:**
```javascript
function buildArtifactSection(artifactPath) {
  if (!artifactPath) return '';
  return `=== Artifact ===\nPath: ${artifactPath}\n(Read this file to obtain its full content before evaluating.)\n================\n`;
}

function buildReviewContextSection(reviewContext, isReminder = false) {
  if (!reviewContext) return '';
  if (isReminder) return `⚠ REVIEW CONTEXT REMINDER: ${reviewContext}\n(If any prior position applied evaluation criteria inconsistent with the above...)\n`;
  return `⚠ REVIEW CONTEXT: ${reviewContext}\n`;
}

function buildModeAPrompt({ round, repoDir, question, artifactPath, reviewContext, priorPositions, requestImprovements }) {
  const parts = [];
  parts.push(`QGSD Quorum — Round ${round}\n\nRepository: ${repoDir}\n\nQuestion: ${question}\n`);
  parts.push(buildArtifactSection(artifactPath));
  parts.push(buildReviewContextSection(reviewContext));
  if (priorPositions) {
    parts.push(buildPriorPositionsSection(priorPositions, reviewContext));
  } else {
    parts.push(buildRound1Instructions(requestImprovements));
  }
  return parts.filter(Boolean).join('\n');
}
```

### Pattern 4: Output Parsing Reuse from Existing Test

**What:** The `parseImprovements` function already exists as a pure function in `bin/gsd-quorum-slot-worker-improvements.test.cjs` (lines 21–82) with 13 passing tests. Move it to `quorum-slot-dispatch.cjs` as a module export (or copy it inline) and reference the existing tests.

**Key insight:** The improvements parser is already well-tested and mature. The verdict/reasoning/citations parser is simpler (single-line regex extraction).

**Verdict/reasoning parsing:**
```javascript
function parseVerdict(rawOutput, mode) {
  if (mode === 'B') {
    // Mode B: extract explicit "verdict: APPROVE|REJECT|FLAG" line
    const match = rawOutput.match(/^verdict:\s*(APPROVE|REJECT|FLAG)/im);
    return match ? match[1] : 'FLAG';
  }
  // Mode A: verdict is free-form position summary (no structured extract)
  return extractReasoningSummary(rawOutput);
}

function parseCitations(rawOutput) {
  const match = rawOutput.match(/^citations:\s*\|?\s*\n((?:[ \t]+[^\n]*\n?)*)/im);
  return match ? match[1].trim() : null;
}
```

### Anti-Patterns to Avoid

- **Putting prompt content in `--prompt` CLI arg:** Multi-line content with special characters breaks shell quoting. Use stdin pipe or tempfiles.
- **Extending `call-quorum-slot.cjs` directly:** This conflates transport (CQS's job) with prompt construction (new wrapper's job). Also: CQS is the installed binary that other phases depend on; changing its interface risks breaking FAIL-01 retry tests.
- **Re-implementing `parseImprovements` differently:** The existing implementation has 13 test cases covering edge cases. Reuse it verbatim or move it to a shared location and import it.
- **Using `exec` instead of `spawn`:** `exec` buffers all output in memory with a 1MB default cap. `spawn` with streaming handles large trace outputs safely. CQS already uses `spawn`; mirror that.
- **Making the agent parse structured output from the wrapper:** The agent must return the script's stdout verbatim. Any LLM re-formatting defeats the purpose.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing of structured output | Custom recursive YAML parser | Regex line-by-line extraction (see `read-policy.cjs` pattern) | Quorum output is flat `key: value` structure; full YAML parser adds dep for zero gain |
| Improvements parsing | New implementation | Move `parseImprovements` from test file | Already has 13 passing tests covering all edge cases |
| Argument parsing | `commander` or `yargs` | Project's existing `getArg` pattern from `call-quorum-slot.cjs` | Consistent with codebase; zero deps |
| Shell heredoc quoting for multi-line content | Complex `printf '%s'` quoting | Tempfile approach | Simpler, no quoting risk |

**Key insight:** The `getArg` pattern (`const getArg = (f) => { const i = argv.indexOf(f); return i !== -1 && argv[i + 1] ? argv[i + 1] : null; }`) is the universal argument parser in this codebase. Use it.

## Common Pitfalls

### Pitfall 1: Prompt Newline Handling Between Script and CQS

**What goes wrong:** The worker currently uses a Bash HEREDOC to pass the prompt to CQS, which preserves all newlines exactly. A JavaScript spawn with `child.stdin.write(prompt)` does the same — but if the wrapper adds or strips trailing newlines, CQS's `data.trim()` call will normalize them. This is safe.

**Why it happens:** Developers assume they need to replicate the exact heredoc behavior.

**How to avoid:** Just `child.stdin.write(prompt, 'utf8'); child.stdin.end()`. CQS trims the result with `.trim()` before using it, so any trailing newline variations are irrelevant.

**Warning signs:** Tests for exact byte-equality of prompts will be fragile; test for semantic equivalence instead (expected sections present, correct ordering).

### Pitfall 2: Mode A vs Mode B Prompt Structure Divergence

**What goes wrong:** Mode A and Mode B share some sections (artifact, review_context, prior_positions) but differ structurally. Mode B always has `=== EXECUTION TRACES ===` section; Mode A only shows traces when `traces` arg is present (it never is in Mode A). The conditional logic for Round 1 vs Round 2+ differs between modes.

**Why it happens:** The agent spec is written as one monolithic template with many `[if X present:]` guards. A naive port copies the whole thing without isolating mode-specific logic.

**How to avoid:** Write `buildModeAPrompt()` and `buildModeBPrompt()` as completely separate functions. Share only the sections that are structurally identical (artifact, review_context injection). The guard at the top of `main()` routes to one or the other.

**Warning signs:** A single `buildPrompt()` function with mode A/B as an argument and many nested conditionals — this is a red flag for cognitive complexity.

### Pitfall 3: parseImprovements Needs to Move, Not Be Duplicated

**What goes wrong:** `parseImprovements` is currently defined inside `bin/gsd-quorum-slot-worker-improvements.test.cjs` as an inline implementation (not exported). If the wrapper re-implements it separately, two divergent implementations exist.

**Why it happens:** The test file has a "MAINTAINABILITY NOTE" warning about keeping the implementation in sync, but no exported module exists yet.

**How to avoid:** In Plan 1 (TDD scaffolding), extract `parseImprovements` from the test file into a separate location (either into `quorum-slot-dispatch.cjs` directly, or a small shared module). Update the test file to import from there. The test cases remain in the test file.

**Warning signs:** If `gsd-quorum-slot-worker-improvements.test.cjs` defines `parseImprovements` inline AND `quorum-slot-dispatch.cjs` also has its own version, they will drift.

### Pitfall 4: Agent Must Not LLM-Transform the Script Output

**What goes wrong:** After the thin-passthrough refactor, the agent calls the script and must emit the output verbatim. If the agent prompt says "parse the output and return a result block", Haiku will re-format it.

**Why it happens:** The agent spec says "return structured result block" in step 4. This instruction needs to change to "emit the script's stdout verbatim."

**How to avoid:** The new agent spec must say explicitly: "Run the dispatch script. Print its stdout to your output. Do not modify, summarize, or reformat the output." The script's stdout IS the structured result block.

**Warning signs:** Quorum orchestrator receives a result block with different formatting than what the script emits, indicating LLM re-processing occurred.

### Pitfall 5: Install Sync Required After Modifying call-quorum-slot.cjs

**What goes wrong:** If the plan touches `call-quorum-slot.cjs` (which is not the preferred approach, but is a risk), the installed copy at `~/.claude/qgsd-bin/call-quorum-slot.cjs` becomes stale until `node bin/install.js --claude --global` is run.

**Why it happens:** Install copies `bin/*.cjs` to `~/.claude/qgsd-bin/`. The agent runs from the installed copy. Tests run against `bin/` (source).

**How to avoid:** Include an install step in the plan that runs `node bin/install.js --claude --global` after modifying any `bin/*.cjs` file. For the new `quorum-slot-dispatch.cjs`, it gets auto-installed via the same mechanism (all `.cjs` files in `bin/` are copied).

**Warning signs:** Tests pass but runtime behavior differs — this indicates stale installed copy.

### Pitfall 6: Detecting TIMEOUT in Script Output vs Exit Code

**What goes wrong:** CQS signals timeout via `process.exit(1)` + stderr message `TIMEOUT after Xms`. The wrapper captures stdout; if CQS exits non-zero, stdout may be empty. The result block UNAVAIL logic must check exit code, not just output content.

**Why it happens:** The current agent checks `if exit non-zero OR output contains TIMEOUT`. The wrapper must replicate this same dual check.

**How to avoid:** In the wrapper, capture both the child process exit code and stdout. If `code !== 0` or `stdout.includes('TIMEOUT')`, emit `verdict: UNAVAIL` with appropriate `unavail_message`.

### Prompt Construction (Mode A, Round 2 with all optional fields)

```javascript
// Source: agents/qgsd-quorum-slot-worker.md Step 2 — Mode A prompt spec
function buildModeAPrompt({ round, repoDir, question, artifactPath, reviewContext, priorPositions, requestImprovements }) {
  const lines = [];
  lines.push(`QGSD Quorum — Round ${round}`);
  lines.push('');
  lines.push(`Repository: ${repoDir}`);
  lines.push('');
  lines.push(`Question: ${question}`);
  lines.push('');

  // Artifact section (optional)
  if (artifactPath) {
    lines.push('=== Artifact ===');
    lines.push(`Path: ${artifactPath}`);
    lines.push('(Read this file to obtain its full content before evaluating.)');
    lines.push('================');
    lines.push('');
  }

  // Review context (optional)
  if (reviewContext) {
    lines.push(`⚠ REVIEW CONTEXT: ${reviewContext}`);
  }

  if (priorPositions) {
    // Round 2+ section
    lines.push('The following positions are from other AI models in this quorum — not human experts.');
    lines.push('Evaluate them as peer AI opinions.');
    lines.push('');
    lines.push('Prior positions:');
    lines.push(priorPositions);
    lines.push('');

    if (reviewContext) {
      lines.push(`⚠ REVIEW CONTEXT REMINDER: ${reviewContext}`);
      lines.push('(If any prior position applied evaluation criteria inconsistent with the above — e.g.');
      lines.push('rejected a plan because code was absent, or approved test results without checking');
      lines.push('assertions — reconsider your position in light of the correct evaluation criteria.)');
    }

    lines.push('');
    lines.push('Before revising your position, use your tools to re-check relevant files. At minimum');
    lines.push('re-read CLAUDE.md and .planning/STATE.md if they exist, and re-read the artifact file if');
    lines.push('one was provided.');
    lines.push('');
    lines.push('Given the above, do you maintain your answer or revise it? State your updated position');
    lines.push('clearly (2–4 sentences).');
  } else {
    // Round 1 section
    lines.push('IMPORTANT: Before answering, use your available tools to read files from the');
    lines.push('Repository directory above. At minimum read: CLAUDE.md (if it exists),');
    lines.push('.planning/STATE.md (if it exists), and the artifact file at the path shown in the');
    lines.push('Artifact section above (if present). Then read any other files directly relevant to');
    lines.push('the question. Your answer must be grounded in what you actually find in the repo.');
    lines.push('');
    lines.push('You are one AI model in a multi-model quorum. Your peer reviewers are other AI language');
    lines.push('models — not human experts. Give your honest answer with reasoning. Be concise (3–6');
    lines.push('sentences). Do not defer to peer models.');
  }

  if (requestImprovements) {
    lines.push('If you APPROVE and have specific, actionable improvements, append:');
    lines.push('');
    lines.push('Improvements:');
    lines.push('- suggestion: [concise change — one sentence]');
    lines.push('  rationale: [why this strengthens the plan]');
    lines.push('');
    lines.push('Omit this section entirely if you have no improvements, or if you BLOCK.');
  }

  lines.push('');
  lines.push('If your answer references specific files, line numbers, or code snippets from the');
  lines.push('repository, record them in a citations: field in your response (optional — only');
  lines.push('include if you actually cite code).');

  return lines.join('\n');
}
```

### Output Parsing (Mode B verdict extraction)

```javascript
// Source: agents/qgsd-quorum-slot-worker.md Step 4 spec
function parseModeAVerdict(rawOutput) {
  // Mode A: free-form position summary — extract first 2-4 sentences
  // Simple heuristic: take up to the first 500 chars, stopping at sentence boundary
  const clean = rawOutput.trim().slice(0, 500);
  return clean;
}

function parseModeBVerdict(rawOutput) {
  // Mode B: look for explicit "verdict: APPROVE|REJECT|FLAG"
  const match = rawOutput.match(/^verdict:\s*(APPROVE|REJECT|FLAG)/im);
  return match ? match[1] : 'FLAG';
}

function parseReasoning(rawOutput) {
  // Extract "reasoning: ..." line or first 2-4 sentences of output
  const match = rawOutput.match(/^reasoning:\s*(.+?)(?=\n[a-zA-Z]|\n\n|$)/ims);
  if (match) return match[1].trim();
  // Fallback: first 400 chars
  return rawOutput.trim().slice(0, 400);
}

function parseCitations(rawOutput) {
  // Extract "citations: |" block (indented content after the key)
  const match = rawOutput.match(/^citations:\s*\|\s*\n((?:[ \t][^\n]*\n?)*)/im);
  return match ? match[1].trim() : null;
}
```

### Result Block Emission

```javascript
// Source: agents/qgsd-quorum-slot-worker.md Step 4 result spec
function emitResultBlock({ slot, round, verdict, reasoning, citations, improvements, rawOutput, isUnavail, unavailMessage }) {
  const lines = [];
  lines.push(`slot: ${slot}`);
  lines.push(`round: ${round}`);
  lines.push(`verdict: ${verdict}`);
  lines.push(`reasoning: ${reasoning}`);

  if (citations) {
    lines.push('citations: |');
    for (const citLine of citations.split('\n')) {
      lines.push(`  ${citLine}`);
    }
  }

  if (Array.isArray(improvements) && improvements.length > 0) {
    lines.push('improvements:');
    for (const imp of improvements) {
      lines.push(`  - suggestion: "${imp.suggestion}"`);
      lines.push(`    rationale: "${imp.rationale}"`);
    }
  }

  lines.push('raw: |');
  const rawTruncated = (rawOutput || '').slice(0, 5000);
  for (const rawLine of rawTruncated.split('\n')) {
    lines.push(`  ${rawLine}`);
  }

  if (isUnavail && unavailMessage) {
    lines.push(`unavail_message: ${unavailMessage.slice(0, 500)}`);
  }

  process.stdout.write(lines.join('\n') + '\n');
}
```

### Thin Agent Shim (new agent spec — critical section)

```bash
# Step 1: Extract arguments from $ARGUMENTS YAML (simple line matching, no LLM parsing)
SLOT=$(echo "$ARGUMENTS" | grep '^slot:' | awk '{print $2}')
ROUND=$(echo "$ARGUMENTS" | grep '^round:' | awk '{print $2}')
TIMEOUT_MS=$(echo "$ARGUMENTS" | grep '^timeout_ms:' | awk '{print $2}')
REPO_DIR=$(echo "$ARGUMENTS" | grep '^repo_dir:' | awk '{print $2}')
MODE=$(echo "$ARGUMENTS" | grep '^mode:' | awk '{print $2}')
# ... etc

# Step 2: Write prior_positions and traces to tempfiles if present
PRIOR_FILE=""
if [ -n "$PRIOR_POSITIONS_CONTENT" ]; then
  PRIOR_FILE=$(mktemp /tmp/qgsd-prior-XXXXXX.txt)
  printf '%s' "$PRIOR_POSITIONS_CONTENT" > "$PRIOR_FILE"
fi

# Step 3: Call the dispatch script
node "$HOME/.claude/qgsd-bin/quorum-slot-dispatch.cjs" \
  --slot "$SLOT" \
  --round "$ROUND" \
  --timeout "$TIMEOUT_MS" \
  --cwd "$REPO_DIR" \
  --mode "$MODE" \
  --question "$QUESTION" \
  ${ARTIFACT_PATH:+--artifact-path "$ARTIFACT_PATH"} \
  ${REVIEW_CONTEXT:+--review-context "$REVIEW_CONTEXT"} \
  ${PRIOR_FILE:+--prior-positions-file "$PRIOR_FILE"} \
  ${REQUEST_IMPROVEMENTS:+--request-improvements}

# Step 4: Clean up tempfiles
[ -n "$PRIOR_FILE" ] && rm -f "$PRIOR_FILE"
```

**Note:** The agent emits the script's stdout verbatim — no LLM post-processing. This is the critical change that reduces tool uses from 4–6 to 1.

## Design Decisions

### Decision 1: Wrapper Script vs Extending call-quorum-slot.cjs

**Decision: Create `bin/quorum-slot-dispatch.cjs` as a new wrapper.** Do NOT modify `call-quorum-slot.cjs`.

**Evidence:**
- `call-quorum-slot.cjs` is the transport layer. Its sole job is: read stdin, find provider, dispatch, return output. Adding prompt construction conflates concerns.
- `call-quorum-slot.cjs` already has FAIL-01 retry tests (`call-quorum-slot-retry.test.cjs`, 10 passing tests). Modifying its interface risks breaking those.
- The installed binary at `~/.claude/qgsd-bin/call-quorum-slot.cjs` is what the installed agent currently calls. Other plans (v0.24-01 through v0.24-04) also touch this file. Keeping the wrapper separate reduces merge conflicts.
- Composing via stdin pipe is idiomatic: wrapper builds prompt → pipes to CQS stdin → CQS dispatches → wrapper parses output.

### Decision 2: Tempfile vs CLI Args for Multi-Line Content

**Decision: Tempfiles for `prior_positions` and `traces` content; CLI args for all simple scalars.**

**Evidence:**
- ARG_MAX is 1MB on macOS (confirmed: 1048576 bytes). All content fits in theory.
- However, prior_positions and traces are multi-line YAML-formatted strings that contain arbitrary characters (quotes, backslashes, special Unicode). Passing via `--flag "$CONTENT"` through shell requires the agent to properly quote it, which is fragile for LLM-generated shell scripts.
- Tempfiles (`mktemp`) are a standard Unix pattern, require no quoting, and work for arbitrary content size.
- The wrapper reads `--prior-positions-file` and `--traces-file` paths and calls `fs.readFileSync()`.
- The thin agent creates tempfiles before calling the script and removes them after.

### Decision 3: Where parseImprovements Lives

**Decision: Move `parseImprovements` into `bin/quorum-slot-dispatch.cjs` as a module-level function. Update `gsd-quorum-slot-worker-improvements.test.cjs` to import it from there.**

**Evidence:**
- Currently `parseImprovements` is defined inline in the test file (lines 21–82) with a "MAINTAINABILITY NOTE" that says to update both together.
- Moving it to the dispatch script is the right home: the script is what actually calls this function at runtime.
- The test file's 13 test cases remain valid — they just `require()` the function from the dispatch script instead of defining it inline.
- This eliminates the duplication risk identified in Pitfall 3.

### Decision 4: What the New Agent Looks Like

**Current agent:** 4 steps, ~280 lines, reads files, constructs template with 7 conditional branches, calls Bash, parses output.

**New agent:** 2 steps:
1. Call `quorum-slot-dispatch.cjs` with structured args derived from `$ARGUMENTS` via simple `grep/awk` (no LLM judgment needed).
2. Emit the script's stdout verbatim.

## Open Questions

1. **How does the agent extract `$ARGUMENTS` fields without LLM parsing?**
   - What we know: `$ARGUMENTS` is a YAML block passed as the agent's prompt. In Claude Code agents, `$ARGUMENTS` is the text after the YAML frontmatter. Simple `grep`/`awk` can extract `key: value` pairs. For multi-line values (`prior_positions: |`, `traces: |`), extraction requires reading until the next non-indented line.
   - What's unclear: Whether the thin agent should use `grep`/`awk` (no LLM) or a minimal LLM extraction step. The goal is to eliminate LLM steps — pure shell extraction is possible but the multi-line block extraction is non-trivial.
   - Recommendation: For multi-line fields, pass them through env vars set by the orchestrator, or add a new argument style where multi-line content is always written to tempfiles by the orchestrator before calling the agent. The orchestrator already controls `$ARGUMENTS` structure — it could pass `prior_positions_file: /tmp/xyz` instead of the content inline.

2. **Does the orchestrator (quorum.md) need to change to pass tempfile paths instead of inline content?**
   - What we know: Currently the orchestrator passes `prior_positions: |` and `traces: |` as inline YAML content in the Task prompt. The thin agent would need to extract these multi-line blocks reliably.
   - What's unclear: Whether changing the orchestrator dispatch format is in scope for v0.24-05 or should be done in the agent spec only.
   - Recommendation: Keep the orchestrator YAML format unchanged. The thin agent extracts the block content (even with simple LLM help if needed for multi-line extraction) and writes it to tempfiles before calling the script. This way the orchestrator contract is unchanged and the complexity is localized to the agent.

3. **Reasoning extraction for Mode A — how much is acceptable?**
   - What we know: Mode A returns a free-form position statement (no structured `verdict:` key). The current agent summarizes it to 2–4 sentences. The script could either return the full raw output as `verdict` or apply a simple truncation.
   - What's unclear: Whether the orchestrator needs a structured `verdict:` field for Mode A or uses the free-form `verdict:` value as-is.
   - Recommendation: For Mode A, set `verdict` to the first 300–500 characters of rawOutput (simple truncation) with the full text in `raw:`. The orchestrator reads `verdict:` as the position summary — truncation is acceptable and deterministic.

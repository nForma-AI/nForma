## Summary

Phase v0.10-03 adds two features to `bin/manage-agents.cjs`: (1) a "Batch rotate keys" option in the main menu that lets users rotate API keys for multiple slots in one session using a sequential `for...of` write loop, and (2) persistent `key_status` in `qgsd.json` that is updated after every health probe so the `[key invalid]` badge (built in v0.10-01) survives process restarts.

The technical stack is unchanged from v0.10-01 and v0.10-02: inquirer@8.2.7 CJS, node:fs for atomic writes, `keytar` via `secrets.cjs` for key storage. Zero new npm dependencies. The `checkbox` prompt type is already used in `editAgent()` (line 626 of `manage-agents.cjs`) and is the correct tool for multi-slot selection. The `writeQgsdJson()` helper was built in v0.10-01 and is the correct tool for persisting `key_status`.

The critical behavioral invariant in the phase description: the batch rotation loop MUST use sequential `for...of` — never `Promise.all`. The `secrets.cjs` keytar interface (`set`, `delete`) makes async keychain calls that can corrupt the key index (`~/.claude/qgsd-key-index.json`) if run concurrently. Additionally, the `readClaudeJson()` / `writeClaudeJson()` pattern is not safe for concurrent read-modify-write: two concurrent writes to `~/.claude.json` can race on the tmp rename. Sequential `for...of` eliminates both risks.

**Primary recommendation:** Add `classifyProbeResult()` pure function + update `checkAgentHealth()` to write `key_status` to `qgsd.json` + add `batchRotateKeys()` async function + wire "Batch rotate keys" to `mainMenu()`. Two natural plans: Plan 01 = TDD pure functions; Plan 02 = integration wiring.

### Pattern 1: classifyProbeResult() — Pure Function

**What:** Classifies the result of `probeProviderUrl()` into a semantic key validity verdict.

**Key insight:** The existing `probeProviderUrl()` treats HTTP 401 as `healthy: true` (meaning "the provider endpoint is reachable"). This is correct for the pre-flight probe. But for key validity, 401 means "key is invalid." `classifyProbeResult()` provides the correct semantic layer above `probeProviderUrl()`.

**When to use:** Called inside `checkAgentHealth()` after receiving the probe result, to determine what to write to `key_status` in `qgsd.json`.

**Signature:**
```js
// Source: derived from probeProviderUrl() signature in manage-agents.cjs lines 98-143
// probeResult: { healthy: bool, latencyMs: int, statusCode: int|null, error: string|null }
// Returns: 'ok' | 'invalid' | 'unreachable'
function classifyProbeResult(probeResult) {
  if (!probeResult || !probeResult.healthy) {
    return 'unreachable'; // provider DOWN — no key validity info
  }
  if (probeResult.statusCode === 401) {
    return 'invalid'; // provider UP, key rejected
  }
  return 'ok'; // provider UP, key accepted (200, 403, 404, 422 all = "reachable and key not rejected")
}
```

**Classification table:**

| `probeResult.healthy` | `probeResult.statusCode` | Classification | Write `key_status`? |
|-----------------------|--------------------------|----------------|---------------------|
| `false` | any / null | `unreachable` | NO — don't overwrite a prior valid/invalid status |
| `true` | 401 | `invalid` | YES — write `{ status: "invalid", checkedAt: "<ISO>" }` |
| `true` | 200/403/404/422 | `ok` | YES — write `{ status: "ok", checkedAt: "<ISO>" }` |

**Unreachable does NOT clear `key_status`:** If the provider is down, the previous `key_status` is preserved. This avoids false "key valid" badges after a network failure.

### Pattern 2: Writing key_status to qgsd.json

**What:** After `classifyProbeResult()` returns `'ok'` or `'invalid'`, write the result to `qgsd.json` under `agent_config[slotName].key_status`.

**When to use:** Inside `checkAgentHealth()` after the probe completes, when the classification is not `'unreachable'`.

```js
// Source: writeQgsdJson() defined at lines 1757-1762 of manage-agents.cjs
// Pattern: read → mutate → write (atomic)
function writeKeyStatus(slotName, status) {
  const qgsd = readQgsdJson();          // read current state
  if (!qgsd.agent_config) qgsd.agent_config = {};
  if (!qgsd.agent_config[slotName]) qgsd.agent_config[slotName] = {};
  qgsd.agent_config[slotName].key_status = {
    status,                              // 'ok' | 'invalid'
    checkedAt: new Date().toISOString(), // ISO timestamp
  };
  writeQgsdJson(qgsd);                  // atomic write to ~/.claude/qgsd.json
}
```

**Note on pure function design:** `writeKeyStatus` is an impure function (side effects). The pure-function boundary for testing is `classifyProbeResult()`. Unit tests test the pure classifier; `writeKeyStatus` is integration-tested via `readQgsdJson`/`writeQgsdJson` roundtrip (the latter are already tested).

### Pattern 3: batchRotateKeys() — Sequential for...of

**What:** New async function implementing the batch rotation flow. Presents a `checkbox` picker, then loops over selected slots sequentially (never in parallel) to prompt for a new key and write it.

**When to use:** Wired to "Batch rotate keys" menu option in `mainMenu()`.

**Key invariants:**
- Uses `inquirer checkbox` for multi-slot selection (existing prompt type, v8.2.7 CJS).
- The rotation `for...of` loop is sequential — never `Promise.all`.
- After the loop, a single `writeClaudeJson()` call syncs any env-fallback updates (plaintext path only).
- Per-slot confirmation printed after each write, before the next slot's prompt.
- If no slots are selected (user presses Enter on empty checkbox), exit silently.

```js
// Structural sketch — implementation in Plan 02
async function batchRotateKeys() {
  const data = readClaudeJson();
  const mcpServers = getGlobalMcpServers(data);
  const slots = Object.keys(mcpServers);

  if (slots.length === 0) {
    console.log('\n  No agents configured.\n');
    return;
  }

  let secretsLib = null;
  try { secretsLib = require('./secrets.cjs'); } catch (_) {}

  // Step 1: Multi-select slot picker
  const { selectedSlots } = await inquirer.prompt([
    {
      type: 'checkbox',
      name: 'selectedSlots',
      message: 'Select slots to rotate keys for:',
      choices: slots.map((name) => {
        const cfg = mcpServers[name];
        const model = (cfg.env && cfg.env.CLAUDE_DEFAULT_MODEL) || cfg.command || '?';
        const keytarAccount = deriveKeytarAccount(name);
        const hasKey = (secretsLib && secretsLib.hasKey(keytarAccount)) ||
                       !!(cfg.env && cfg.env.ANTHROPIC_API_KEY);
        return {
          name: `${name.padEnd(14)} ${model.slice(0, 36).padEnd(36)} ${hasKey ? '[key set]' : '[no key]'}`,
          value: name,
          short: name,
        };
      }),
    },
  ]);

  if (selectedSlots.length === 0) {
    console.log('\n  Nothing selected.\n');
    return;
  }

  // Step 2: Sequential for...of rotation loop — NEVER Promise.all
  for (const slotName of selectedSlots) {
    const { newKey } = await inquirer.prompt([
      {
        type: 'password',
        name: 'newKey',
        message: `New API key for ${slotName}:`,
        mask: '*',
        validate: (v) => (v && v.trim() ? true : 'Cannot be empty'),
      },
    ]);

    const account = deriveKeytarAccount(slotName);
    if (secretsLib) {
      await secretsLib.set('qgsd', account, newKey.trim());
    } else {
      // Plaintext fallback: accumulate updates to apply before writeClaudeJson
      if (!mcpServers[slotName].env) mcpServers[slotName].env = {};
      mcpServers[slotName].env.ANTHROPIC_API_KEY = newKey.trim();
    }

    // Per-slot confirmation BEFORE next prompt
    console.log(`  ${slotName}: key updated`);
  }

  // Step 3: Single writeClaudeJson after all slots processed
  // For keytar path: no env change needed (key is in keychain, not claude.json)
  // For plaintext fallback path: data.mcpServers was mutated above
  writeClaudeJson(data);
  console.log(`\n  \x1b[32m✓ ${selectedSlots.length} slot(s) updated\x1b[0m\n`);
}
```

**syncToClaudeJson clarification:** The success criteria use the term "a single `syncToClaudeJson()` call is made after all slots are processed." In the codebase, the function that writes `~/.claude.json` atomically is `writeClaudeJson()`. `syncToClaudeJson` is NOT an existing function — it is a conceptual label in the success criteria for the single post-loop `writeClaudeJson()` call. The planner should implement this as a direct `writeClaudeJson(data)` call; no new function name is required unless the planner finds it clarifying to alias it.

### Anti-Patterns to Avoid

- **`Promise.all` for keytar writes:** Causes key-index read-modify-write race in `secrets.cjs`. Success criteria explicitly prohibit this. Always `for...of`.
- **Writing `key_status` for `'unreachable'` probes:** If the provider is down, preserve the existing `key_status`. Overwriting with a new status when the endpoint is unreachable would incorrectly clear a `[key invalid]` badge after a network outage.
- **Calling `writeClaudeJson()` inside the loop per slot:** The single post-loop call pattern is specified in success criterion 2. Writing inside the loop creates multiple partial states of `~/.claude.json` during the batch.
- **Storing `key_status` in `~/.claude.json`:** Use `qgsd.json` only. `~/.claude.json` is agent configuration, not runtime state. The existing `getKeyInvalidBadge()` already reads from `qgsd.json` via `readQgsdJson()`.
- **Implementing `classifyProbeResult()` as an impure function with file I/O:** It must be a pure function (input → output, no side effects) so it can be exported via `_pure` and unit-tested without mocking.

## Common Pitfalls

### Pitfall 1: `probeProviderUrl()` Treats 401 as Healthy
**What goes wrong:** Developer reads "healthy = true means key is OK" and skips 401 detection.
**Why it happens:** The probe function was designed for provider reachability, not key validity. 401 means "endpoint responded = healthy." It's a design choice, not a bug.
**How to avoid:** Always pass through `classifyProbeResult()`. Never test `probe.healthy` alone to determine key validity — also test `probe.statusCode === 401`.
**Warning signs:** `key_status` written as `{ status: "ok" }` when the probe returned 401.

### Pitfall 2: Calling writeClaudeJson() Inside the Rotation Loop
**What goes wrong:** Each iteration of the `for...of` calls `writeClaudeJson()`, creating N sequential writes. This is inefficient and creates intermediate states visible to external readers.
**Why it happens:** Developer copy-pastes the `editAgent()` pattern which writes once per edit session.
**How to avoid:** Accumulate all mutations in memory (for the plaintext fallback path), write `writeClaudeJson()` exactly once after the loop.
**Warning signs:** `writeClaudeJson(data)` appears inside the `for...of` body.

### Pitfall 3: key_status Written for Unreachable Endpoints
**What goes wrong:** After a network failure (provider DOWN), `key_status` is overwritten with `{ status: "unreachable" }` or cleared. The `[key invalid]` badge is lost even though the key was invalid before the outage.
**Why it happens:** Treating all probe classifications uniformly — writing `key_status` for every probe result.
**How to avoid:** Only write `key_status` when `classifyProbeResult()` returns `'ok'` or `'invalid'`. When `'unreachable'`, preserve the existing `qgsd.json` state unchanged.
**Warning signs:** `writeKeyStatus()` called unconditionally for all probe results.

### Pitfall 4: Batch Rotate Loop Writes Multiple qgsd.json Updates
**What goes wrong:** Each slot rotation writes a new `key_status` to `qgsd.json` by calling `writeKeyStatus()` inside the batch rotate loop.
**Why it happens:** Confusion between the batch rotate flow (CRED-01) and the health probe flow (CRED-02). The batch rotate flow updates keys; it does NOT run health probes. `key_status` is only written during health probes — not during key rotation.
**How to avoid:** `batchRotateKeys()` writes API keys only (via `secretsLib.set`). It does NOT update `key_status` in `qgsd.json`. `key_status` is only updated in `checkAgentHealth()`.
**Warning signs:** `writeKeyStatus()` called from `batchRotateKeys()`.

### Pitfall 5: inquirer `checkbox` returns empty array on Enter with no selections
**What goes wrong:** User presses Enter without selecting any slots — `selectedSlots` is `[]`. The `for...of` loop completes immediately, then `writeClaudeJson(data)` is called unnecessarily.
**Why it happens:** The `checkbox` prompt does not validate minimum selection unless a `validate()` is added.
**How to avoid:** Check `selectedSlots.length === 0` before the loop and return early with "Nothing selected." Do NOT add a `validate()` that blocks zero selection — the user should be allowed to escape.
**Warning signs:** `writeClaudeJson()` called with unchanged data on empty selection.

```js
function readQgsdJson(filePath) {
  const p = filePath || QGSD_JSON_PATH;           // ~/ .claude/qgsd.json
  if (!fs.existsSync(p)) return {};
  try { return JSON.parse(fs.readFileSync(p, 'utf8')); } catch (_) { return {}; }
}

function writeQgsdJson(data, filePath) {
  const p = filePath || QGSD_JSON_PATH;
  const tmp = p + '.tmp';
  fs.writeFileSync(tmp, JSON.stringify(data, null, 2), 'utf8');
  fs.renameSync(tmp, p);                          // atomic rename
}
```

## Open Questions

1. **Should `writeKeyStatus` be exported via `_pure` for unit testing?**
   - What we know: `writeKeyStatus` has file I/O side effects, so it is not strictly pure. The `_pure` export is used only for pure functions in this codebase.
   - What's unclear: Should `writeKeyStatus` be tested directly (requiring file I/O), or tested indirectly via `readQgsdJson`/`writeQgsdJson` roundtrip (which are already tested)?
   - Recommendation: Export `classifyProbeResult()` via `_pure` (it IS pure). Test `writeKeyStatus` via the `readQgsdJson`/`writeQgsdJson` roundtrip pattern already established in `manage-agents.test.cjs`. This keeps the pure/impure boundary clean.

2. **Does `batchRotateKeys()` need a probe pre-flight before writing each key?**
   - What we know: Success criteria 1 and 2 describe: checkbox select → enter key for each slot → per-slot confirmation → single `syncToClaudeJson()`. No probe is mentioned.
   - What's unclear: Should the new key be validated against the provider before writing?
   - Recommendation: Do NOT add a probe to `batchRotateKeys()`. The user is rotating keys — the intent is to write the new key regardless. A probe would require the user to wait 7 seconds per slot and would block rotation if the provider is temporarily down. Key validity will be surfaced on the next `checkAgentHealth()` call.

3. **What menu number should "Batch rotate keys" use?**
   - What we know: The current menu has items 1–11 (with 0=Exit). Item 10 is "Manage CCR provider keys."
   - What's unclear: The success criterion says "a 'Batch rotate keys' option appears in the main menu" — the number is not specified.
   - Recommendation: Insert as item 11, renumber "Update coding agents" to 12. The number is display-only (users navigate by arrow keys, not by typing the number) and has no semantic significance.

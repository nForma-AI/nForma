### quorum_active Data Model

Location: `~/.claude/qgsd.json` key `quorum_active` — array of slot name strings.

Example:
```json
{ "quorum_active": ["codex-cli", "gemini-cli", "opencode", "copilot-cli", "claude-deepseek", ...] }
```

Written with this inline pattern (from Step B-native):
```bash
node -e "
const fs = require('fs'), os = require('os');
const qgsdPath = os.homedir() + '/.claude/qgsd.json';
let cfg = {};
try { cfg = JSON.parse(fs.readFileSync(qgsdPath, 'utf8')); } catch(e) {}
cfg.quorum_active = UPDATED_ARRAY;
fs.writeFileSync(qgsdPath, JSON.stringify(cfg, null, 2) + '\n');
"
```

Note: qgsd.json path is `~/.claude/qgsd.json` (NOT `~/.claude.json`).

### Available Slots Source

`buildActiveSlots()` in `bin/install.js` reads `~/.claude.json` mcpServers keys. In the wizard, this is replicated as a raw node read of `Object.keys(claudeJson.mcpServers || {})`.

The EXISTING_SERVERS detection pattern (already in Step A) is the template:
```bash
EXISTING_SERVERS=$(node -e "
const fs = require('fs'), os = require('os');
try {
  const cj = JSON.parse(fs.readFileSync(os.homedir() + '/.claude.json', 'utf8'));
  console.log(JSON.stringify(Object.keys(cj.mcpServers || {})));
} catch(e) { console.log('[]'); }
")
```

### quorum_active Semantics (from qgsd-prompt.js)

```javascript
const activeSlots = (config.quorum_active && config.quorum_active.length > 0)
  ? config.quorum_active
  : null; // null = use hardcoded fallback list (all known slots)
```

Implication: an empty `quorum_active` array is equivalent to "use all" (fail-open). Toggling a slot OFF removes it from the array. Toggling all slots off = empty array = all participate. The composition screen should make this explicit.

### SLOT_TOOL_SUFFIX Map (for displaying slot type in composition screen)

```javascript
const SLOT_TOOL_SUFFIX = {
  'codex-cli': 'review',
  'gemini-cli': 'gemini',
  'opencode': 'opencode',
  'copilot-cli': 'ask',
  'claude': 'claude',
};
```

Family detection: `slotName.replace(/-\d+$/, '')` strips the trailing index.

### Inline Script Security Pattern

Keys and slot names are passed via env vars (established in Phase 33):
```bash
node -e "const newSlot = process.env.NEW_SLOT; ..." NEW_SLOT="{slot-name}"
```

This prevents shell history leakage. Apply same pattern for composition write.

## Architecture Analysis

### WIZ-08 Implementation

**Change location:** `commands/qgsd/mcp-setup.md` — Re-run Agent Menu section (line ~402).

Add "Edit Quorum Composition" to the AskUserQuestion options list and add a routing branch:
```
If "Edit Quorum Composition": → Composition Screen (new section)
```

The composition screen is a new H2 section in mcp-setup.md.

### WIZ-09 Implementation: Composition Screen

**Screen flow:**
1. Read `~/.claude.json` mcpServers keys → discovered slots
2. Read `~/.claude/qgsd.json` quorum_active → active set
3. Display numbered list with ON/OFF indicator per slot
4. AskUserQuestion: enter number to toggle, "Add new slot", "Apply", "Cancel"
5. User toggles: update in-memory array
6. "Apply": write updated quorum_active to `~/.claude/qgsd.json`

**Display format:**
```
#   Slot              Family      Status
──  ────────────────  ──────────  ──────
1   codex-cli-1       codex-cli   ● ON
2   gemini-cli-1      gemini-cli  ● ON
3   opencode-1        opencode    ○ OFF
4   copilot-1         copilot-cli ● ON
```

**Toggle logic:** quorum_active is a whitelist when non-empty. Toggle OFF = remove from array. Toggle ON = add to array. Empty array = all participate (show warning).

**Write script:** Inline node using env var for new array value, same qgsd.json write pattern.

### WIZ-10 Implementation: Add Slot from Composition Screen

**From the composition screen**, "Add new slot" routes to the existing Step A flow (EXISTING_SERVERS detection + options 1-9 + Step B/B-native). After Step D (identity ping), return to composition screen (re-read data to show new slot as ON).

This reuse of Step A/B/B-native is the key design — no duplication needed. The only addition is a return path back to the composition screen after add completes.

### Single File

All changes are in `commands/qgsd/mcp-setup.md`. No JS files need modification — the inline node scripts are sufficient. No test changes required (mcp-setup.md has no unit tests; verification is manual grep-count).

## Key Constraints

1. **qgsd.json vs ~/.claude.json**: Composition toggles write to `~/.claude/qgsd.json`, NOT `~/.claude.json`. No restart needed for composition-only changes (quorum_active is read at hook invocation time from disk).
2. **Empty array semantics**: quorum_active=[] means all slots participate. If user toggles all slots off, warn: "All slots off = all participate (fail-open). This is equivalent to removing quorum filtering."
3. **No restart for composition toggle**: Changing quorum_active does not require agent restart — hooks re-read qgsd.json on each invocation. Only slot addition (new mcpServers entry) needs restart.
4. **Add slot in composition screen reuses Step A/B/B-native**: No duplication. Route to existing section, add return marker.
5. **Security**: Slot names passed via env var in inline node scripts.

## Summary

Phase v0.10-05 adds two policy configuration flows to the main menu (`tuneTimeouts` and `setUpdatePolicy`) and a startup auto-update check that runs before the menu is shown. All three features are internal to `bin/manage-agents.cjs` — no new files, no new npm packages.

**CRITICAL TIMEOUT FINDING:** `call-quorum-slot.cjs` (line 233) reads `provider.quorum_timeout_ms` from `providers.json` — not from `qgsd.json`. This means PLCY-01 must write the timeout to `providers.json` via `writeProvidersJson()`, NOT to `qgsd.json`. The "restart required" note is appropriate because Claude Code must restart to reload the `providers.json` values used by running quorum calls.

Update policy storage is in `qgsd.json agent_config[slot].update_policy` via `writeQgsdJson()` — this is a QGSD policy field, not a quorum execution parameter, so it belongs in qgsd.json. The startup check is a new async function called from the entry point before `mainMenu()`, using `getUpdateStatuses()` from `update-agents.cjs`. Log file management is pure `fs.appendFileSync` to `~/.claude/qgsd-update.log`.

The Nyquist wave structure follows the established phase pattern: Wave 0 stubs failing tests, Wave 1 makes pure functions green, Wave 2 wires integration into the main menu and entry point.

**Primary recommendation:** Extract `buildTimeoutChoices`, `buildPolicyChoices`, `buildUpdateLogEntry`, and `parseUpdateLogErrors` as pure functions first (Wave 0/1), then wire `tuneTimeouts()`, `setUpdatePolicy()`, and `runAutoUpdateCheck()` as integration (Wave 2).

### Pattern 1: Write Timeout to providers.json (PLCY-01)

`call-quorum-slot.cjs` line 233 is definitive:
```javascript
const effectiveTimeout = timeoutMs || provider.quorum_timeout_ms || provider.timeout_ms || 30000;
```

This reads `provider.quorum_timeout_ms` from `providers.json`. Therefore `tuneTimeouts()` must modify `providers.json` via `writeProvidersJson()`.

**What:** For a given slot, find its corresponding provider in `providers.json` (via `PROVIDER_SLOT` env in `~/.claude.json`), update `quorum_timeout_ms`, write back atomically.
**When to use:** Whenever the user saves a new timeout in `tuneTimeouts()`.

```javascript
// Pattern: read providers.json -> find provider by name -> update quorum_timeout_ms -> write back
function applyTimeoutUpdate(slotName, newTimeoutMs, mcpServers, filePath) {
  // slotName -> provider name via mcpServers[slotName].env.PROVIDER_SLOT
  const providerSlot = (mcpServers[slotName]?.env?.PROVIDER_SLOT) || slotName;
  const data = readProvidersJson(filePath);  // readProvidersJson already exists
  const providers = data.providers || [];
  const idx = providers.findIndex((p) => p.name === providerSlot);
  if (idx === -1) return false;  // provider not found — cannot update
  providers[idx] = { ...providers[idx], quorum_timeout_ms: newTimeoutMs };
  data.providers = providers;
  writeProvidersJson(data, filePath);
  return true;
}
// Pure function with optional filePath for testability
```

**Why restart is required:** `call-quorum-slot.cjs` reads providers.json at invocation time. A running Claude Code session already has the old timeout cached in its quorum orchestrator invocations. A restart loads the new value.

`update_policy` is a QGSD management field — not a quorum execution parameter. It belongs in `qgsd.json agent_config[slot]`, following the identical pattern as `writeKeyStatus()`.

### Pattern 5: Append-Only Update Log

```javascript
// Path constant (add near existing path constants at top of file):
const UPDATE_LOG_PATH = path.join(os.homedir(), '.claude', 'qgsd-update.log');

// Format: NDJSON — one JSON object per line, each timestamped
// Write: fs.appendFileSync (NOT fs.writeFileSync — preserve history)
// Directory guard: mkdirSync recursive before first write

function buildUpdateLogEntry(slotName, status, detail) {
  // Pure function — returns the NDJSON string to append
  // status: 'OK' | 'UPDATE_AVAILABLE' | 'ERROR' | 'SKIP'
  return JSON.stringify({
    ts: new Date().toISOString(),
    slot: slotName,
    status,
    detail: detail || null,
  }) + '\n';
}

function parseUpdateLogErrors(logContent, maxAgeMs) {
  // Pure function — returns array of recent ERROR entries
  // maxAgeMs: how far back to look (default: 86400000 = 24h)
  if (!logContent) return [];
  const cutoff = Date.now() - (maxAgeMs || 86400000);
  return logContent
    .split('\n')
    .filter(Boolean)
    .map((line) => { try { return JSON.parse(line); } catch { return null; } })
    .filter(Boolean)
    .filter((e) => e.status === 'ERROR' && new Date(e.ts).getTime() > cutoff);
}
```

### Pattern 7: Tune Timeouts Flow (PLCY-01)

```javascript
async function tuneTimeouts() {
  const data = readClaudeJson();
  const mcpServers = getGlobalMcpServers(data);
  const slots = Object.keys(mcpServers);
  const providersData = readProvidersJson();

  // Build display: slot name + current quorum_timeout_ms from providers.json
  const rows = buildTimeoutChoices(slots, mcpServers, providersData);
  // rows: [{ slotName, providerSlot, currentMs }, ...]

  let changed = false;
  for (const { slotName, providerSlot, currentMs } of rows) {
    const { newMs } = await inquirer.prompt([{
      type: 'input',
      name: 'newMs',
      message: `${slotName.padEnd(14)} current: ${currentMs ?? '—'} ms  →  New timeout (ms, blank = keep):`,
      default: currentMs ? String(currentMs) : '',
      validate: (v) => !v || !isNaN(parseInt(v, 10)) ? true : 'Must be a number',
    }]);
    if (newMs.trim() && parseInt(newMs, 10) !== currentMs) {
      const updated = applyTimeoutUpdate(slotName, parseInt(newMs, 10), providerSlot, providersData);
      Object.assign(providersData, updated);
      changed = true;
    }
  }

  if (changed) {
    writeProvidersJson(providersData);
    // PLCY-01 MANDATORY: restart-required note
    console.log('\n  \x1b[33mTimeout changes require a Claude Code restart to take effect.\x1b[0m\n');
  } else {
    console.log('\n  No changes made.\n');
  }
}
```

**CRITICAL:** The restart-required note after save is a mandatory success criterion in PLCY-01. Missing it is a defect.

### Pattern 8: Update Policy Flow (PLCY-02)

```javascript
const POLICY_CHOICES = [
  { name: 'auto   — check for updates on startup', value: 'auto' },
  { name: 'prompt — ask before updating',          value: 'prompt' },
  { name: 'skip   — never check for updates',      value: 'skip' },
];

function buildPolicyChoices(currentPolicy) {
  // Pure function — annotates the current policy with ← current marker
  return POLICY_CHOICES.map((c) => ({
    ...c,
    name: c.value === currentPolicy
      ? c.name + '  \x1b[90m\u2190 current\x1b[0m'
      : c.name,
  }));
}

async function setUpdatePolicy() {
  const data = readClaudeJson();
  const mcpServers = getGlobalMcpServers(data);
  const slots = Object.keys(mcpServers);
  const qgsd = readQgsdJson();
  const agentConfig = qgsd.agent_config || {};

  // Step 1: slot picker
  const { slotName } = await inquirer.prompt([{
    type: 'list',
    name: 'slotName',
    message: 'Select slot to configure update policy:',
    choices: slots.map((s) => ({
      name: `${s.padEnd(14)} policy: ${(agentConfig[s] || {}).update_policy || '—'}`,
      value: s,
    })),
  }]);

  // Step 2: policy choice
  const currentPolicy = (agentConfig[slotName] || {}).update_policy || null;
  const { policy } = await inquirer.prompt([{
    type: 'list',
    name: 'policy',
    message: `Update policy for ${slotName}:`,
    choices: buildPolicyChoices(currentPolicy),
  }]);

  writeUpdatePolicy(slotName, policy);
  console.log(`\n  \x1b[32m\u2713 ${slotName}: update_policy = ${policy}\x1b[0m\n`);
}
```

### Anti-Patterns to Avoid

- **Writing quorum_timeout_ms to qgsd.json instead of providers.json:** `call-quorum-slot.cjs` reads from `providers.json` only. Writing to qgsd.json has no effect on actual quorum timeout behavior.
- **Calling `writeProvidersJson()` inside the per-slot loop:** All timeout updates must accumulate in the in-memory `providersData` object, then write ONCE after the loop — same invariant as `batchRotateKeys()`.
- **Using `console.error` for update log ERROR banner:** The banner appears in `listAgents()` output via `console.log()` ANSI yellow. The log file uses `appendFileSync`; stdout is never polluted.
- **Blocking mainMenu on failed update checks:** `runAutoUpdateCheck()` must be fail-open. If it takes too long or throws, write an ERROR log entry and return.
- **Writing CLAUDE_MCP_TIMEOUT_MS to `~/.claude.json` env block:** This env var controls MCP transport timeout (how long Claude Code waits for the MCP server process), NOT the quorum call timeout. Do not confuse these.
- **Forgetting `fs.mkdirSync` before first log write:** `~/.claude/` may not exist on a fresh machine. Always guard with `{ recursive: true }`.

## Common Pitfalls

### Pitfall 1: Writing Timeout to Wrong File
**What goes wrong:** `tuneTimeouts()` writes to `qgsd.json agent_config[slot].quorum_timeout_ms` instead of `providers.json providers[n].quorum_timeout_ms`.
**Why it happens:** The qgsd.json pattern is used for PLCY-02 and PLCY-03, which can make it feel like the right place for PLCY-01 too.
**How to avoid:** `call-quorum-slot.cjs` line 233 is authoritative. The timeout lives in providers.json. Test this by verifying `providers.json` content after calling `applyTimeoutUpdate()`.
**Warning signs:** Timeout change has no effect on quorum calls after restart.

### Pitfall 2: Restart Note Omitted
**What goes wrong:** `tuneTimeouts()` completes successfully but no restart note is shown.
**Why it happens:** Developer focuses on the write path and forgets the post-save UX step.
**How to avoid:** The restart note is the LAST `console.log()` before `return` in `tuneTimeouts()`. Write a test stub that verifies the note appears.
**Warning signs:** PLCY-01 success criterion says "a timeout change without this note is a defect."

### Pitfall 3: Auto-Update Check Blocking Menu
**What goes wrong:** `runAutoUpdateCheck()` hangs on a network timeout, blocking `mainMenu()` from appearing.
**Why it happens:** `getUpdateStatuses()` uses `spawnSync` with 8s timeouts per binary. All binaries checking in parallel via `Promise.all` — but if many slots map to many binaries, total wall clock can still be high.
**How to avoid:** Add a total `Promise.race` timeout around `runAutoUpdateCheck()`. Recommended: 20s total. Write a single TIMEOUT ERROR log entry and proceed to mainMenu.
**Warning signs:** `manage-agents.cjs` takes >10s to show the main menu.

### Pitfall 4: Update Log File Permission Error
**What goes wrong:** `fs.appendFileSync` to `~/.claude/qgsd-update.log` fails because `~/.claude/` doesn't exist.
**Why it happens:** Fresh install without a prior Claude Code session may not have created `~/.claude/`.
**How to avoid:** Call `fs.mkdirSync(path.dirname(UPDATE_LOG_PATH), { recursive: true })` before any `appendFileSync`. This is idempotent and safe.
**Warning signs:** `ENOENT: no such file or directory` error in startup check.

### Pitfall 5: Log Grows Without Bound
**What goes wrong:** `~/.claude/qgsd-update.log` grows forever.
**Why it happens:** Append-only writes without rotation.
**How to avoid for this phase:** When reading for the ERROR banner, only read the tail (last 500 lines). Use `fs.readFileSync` + `split('\n').slice(-500)` approach. Full log rotation is deferred.
**Warning signs:** Users on long-running installs see a multi-MB log file.

### Pitfall 6: `listAgents()` Warning Banner Breaks Column Alignment
**What goes wrong:** Warning banner breaks visual table layout.
**Why it happens:** Banner is injected mid-string or appended to the column header row.
**How to avoid:** Print the banner as a standalone `console.log()` call BEFORE the `\n  (header row)`, with a blank line between banner and table.

### Pitfall 7: Provider Slot Lookup Fails for Subprocess Providers
**What goes wrong:** `tuneTimeouts()` cannot find a provider for a subprocess slot, silently skipping it.
**Why it happens:** Subprocess providers (codex, gemini) use `PROVIDER_SLOT` env var set to their `providers.json` name. But the lookup must match `providers.json` entries where `type === 'subprocess'` too — not just `type === 'http'`.
**How to avoid:** `buildTimeoutChoices` must include ALL slots, showing `(subprocess)` or `(N/A)` for slots with no matching providers.json entry. Do not silently skip.

### Pure Function: buildTimeoutChoices

```javascript
/**
 * Build display rows for the timeout tuning screen.
 * slots: string[]
 * mcpServers: object (from ~/.claude.json)
 * providersData: object (from providers.json)
 * Returns: array of { slotName, providerSlot, currentMs } objects for display
 * Pure function — no side effects.
 */
function buildTimeoutChoices(slots, mcpServers, providersData) {
  const providers = (providersData && providersData.providers) || [];
  return slots.map((slotName) => {
    const cfg = mcpServers[slotName] || {};
    const providerSlot = (cfg.env && cfg.env.PROVIDER_SLOT) || slotName;
    const provider = providers.find((p) => p.name === providerSlot) || null;
    const currentMs = provider ? (provider.quorum_timeout_ms || provider.timeout_ms || null) : null;
    return { slotName, providerSlot, currentMs };
  });
}
```

## Open Questions

1. **"Recent" definition for ERROR banner threshold**
   - What we know: Success criterion says "recent ERROR entries" — no time window specified in requirements.
   - What's unclear: Is "recent" 1 hour, 24 hours, or "since last successful run"?
   - Recommendation: Default to 24 hours (86400000ms). Use constant `UPDATE_LOG_MAX_AGE_MS = 86400000` that can be overridden in tests.

2. **Update check scope for `update_policy: "auto"` — binary deduplication**
   - What we know: `getUpdateStatuses()` checks CLI binaries (codex, gemini, etc.) from `providers.json`. Multiple slots can share the same CLI binary.
   - What's unclear: Should duplicate binary checks be deduplicated (check each binary once) or run once per slot (log per slot)?
   - Recommendation: Deduplicate binary checks (call `getUpdateStatuses()` once for all auto-slots combined); log one entry per SLOT even if multiple slots share a binary.

3. **readProvidersJson/writeProvidersJson do not accept optional filePath**
   - What we know: Unlike `readQgsdJson`/`writeQgsdJson`, the current `readProvidersJson`/`writeProvidersJson` in manage-agents.cjs do not support an optional `filePath` override.
   - Recommendation: Use the `applyTimeoutUpdate(providersData, ...)` pure function approach (takes data as input) so no filePath injection is needed for pure function tests. The integration wiring reads/writes `PROVIDERS_JSON_PATH` directly.

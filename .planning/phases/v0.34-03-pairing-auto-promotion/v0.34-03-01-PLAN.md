  <action>
Create `bin/candidate-pairings.cjs` that reads the evaluated candidates.json from v0.34-02 and generates candidate-pairings.json with resolution tracking.

**Core logic:**

1. Load `.planning/formal/candidates.json` (output from v0.34-02)
2. Load existing `.planning/formal/candidate-pairings.json` if it exists (for merge/cache)
3. For each candidate in candidates.json:
   - Check if already in existing pairings file
   - If existing AND status is "rejected": preserve as-is (cache — PAIR-04)
   - If existing AND status is "confirmed": preserve as-is
   - If new (not in pairings file): add with status "pending"
4. Copy proximity_score, verdict, confidence, reasoning from candidates.json
5. Add resolution tracking fields: `status` (pending|confirmed|rejected), `resolved_at`, `resolved_by`
6. Sort by verdict (yes first, then maybe, then no), then by proximity_score descending
7. Write to `.planning/formal/candidate-pairings.json`

**Output format:**
```json
{
  "metadata": {
    "generated": "ISO timestamp",
    "source_candidates_hash": "first 8 chars SHA256 of candidates.json",
    "total_pairings": N,
    "pending": N,
    "confirmed": N,
    "rejected": N
  },
  "pairings": [
    {
      "model": "path/to/model.als",
      "requirement": "REQ-ID",
      "proximity_score": 0.85,
      "verdict": "yes",
      "confidence": 0.9,
      "reasoning": "...",
      "status": "pending",
      "resolved_at": null,
      "resolved_by": null
    }
  ]
}
```

**CLI interface:**
- `node bin/candidate-pairings.cjs` — generate/update pairings
- `node bin/candidate-pairings.cjs --json` — print summary to stdout as JSON
- `node bin/candidate-pairings.cjs --force` — regenerate all (ignore existing pairings)

**Error handling:**
- If candidates.json missing: exit 1 with message "Run candidate-discovery + haiku-semantic-eval first"
- If candidates.json has no verdicts: exit 1 with "No evaluated candidates"

**Export:** `generatePairings(candidatesData, existingPairings)` function for testing.

<task type="auto">
  <name>Task 2: Implement resolve-pairings.cjs for interactive human resolution</name>
  <files>
    bin/resolve-pairings.cjs

  <action>
Create `bin/resolve-pairings.cjs` that loads candidate-pairings.json and presents pending pairings for human confirmation.

**Core logic:**

1. Load `.planning/formal/candidate-pairings.json`
2. Filter to pending pairings only (skip confirmed/rejected)
3. If no pending pairings: print "No pending pairings to resolve" and exit 0
4. For each pending pairing, display:
   ```
   [N/M] Model: path/to/model.als
         Requirement: REQ-ID
         Proximity: 0.85 | Verdict: yes (confidence: 0.9)
         Reasoning: Direct coverage through related models

         (c)onfirm | (r)eject | (s)kip | (q)uit
   ```
5. Read stdin for response:
   - `c` or `confirm`: Set status to "confirmed", resolved_at to ISO timestamp, resolved_by to "human"
   - `r` or `reject`: Set status to "rejected", resolved_at, resolved_by
   - `s` or `skip`: Leave as pending, move to next
   - `q` or `quit`: Save progress, exit
6. After confirming: write to model-registry.json
   - Load model-registry.json
   - Find the model entry
   - Add the requirement ID to its `requirements` array (if not already present)
   - Write back
7. Write updated candidate-pairings.json after each resolution (safe against interrupts)
8. Print summary at end: "Confirmed: N, Rejected: N, Skipped: N"

**Batch mode (non-interactive):**
- `node bin/resolve-pairings.cjs --auto-confirm-yes` — auto-confirm all pairings with verdict "yes"
- `node bin/resolve-pairings.cjs --auto-reject-no` — auto-reject all with verdict "no"
- `node bin/resolve-pairings.cjs --dry-run` — show what would be resolved without writing

**Error handling:**
- If candidate-pairings.json missing: exit 1 with "Run candidate-pairings first"
- If model-registry.json missing: exit 1
- Write-after-each-resolution ensures no data loss on interrupt

**Export:** `confirmPairing(pairing, registry)` and `rejectPairing(pairing)` for testing.

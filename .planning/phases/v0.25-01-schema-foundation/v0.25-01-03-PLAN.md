For each runner file that calls `writeCheckResult()`:

1. Add `require('./requirement-map.cjs')` at the top:
   ```javascript
   const { getRequirementIds } = require('./requirement-map.cjs');
   ```

2. In each `writeCheckResult()` call, add `requirement_ids: getRequirementIds(checkId)` where `checkId` is the check_id being emitted for that specific check result.

Pattern per runner type:

**TLA+ runners (run-tlc.cjs, etc.):** These have a `CHECK_ID_MAP` or hardcoded check_id. Find where `writeCheckResult()` is called and add `requirement_ids: getRequirementIds(checkId)`.

Example for run-tlc.cjs:
```javascript
// Where writeCheckResult is called:
writeCheckResult({
  tool: 'run-tlc',
  formalism: 'tla',
  result: passed ? 'pass' : 'fail',
  check_id: checkId,
  surface: surface,
  property: propertyDesc,
  runtime_ms: elapsed,
  summary: summaryLine,
  requirement_ids: getRequirementIds(checkId),  // <-- ADD THIS
  metadata: { spec, config: configName },
});
```

**Alloy runners (run-alloy.cjs, etc.):** Same pattern -- find the writeCheckResult call, pass `requirement_ids: getRequirementIds(checkId)`.

**PRISM runners (run-prism.cjs, etc.):** Same pattern.

**CI enforcement scripts:** Same pattern -- they use check_ids like 'ci:trace-redaction'.

**UPPAAL runner (run-uppaal.cjs):** Same pattern.

Important constraints:
- Do NOT change any existing field values or behavior
- Only ADD the `requirement_ids` field to existing writeCheckResult calls
- Each runner should import getRequirementIds only once at the top
- If a runner has multiple writeCheckResult calls (e.g., one for pass, one for fail), add requirement_ids to ALL of them
- The check_id variable name may differ per runner (checkId, check_id, CHECK_ID, etc.) -- use whatever the runner already uses

Read each runner file to find: (a) how check_id is determined, (b) where writeCheckResult is called. Then add the import and the field.

## 1. What This Phase Must Deliver

Every formal artifact (model registry, check results, runners, requirements envelope) carries structured requirement linkage fields that downstream tools can query. Four schema changes:

1. **SCHEMA-01**: `model-registry.json` entries gain `requirements: string[]` -- seeded from property-to-requirement map
2. **SCHEMA-02**: `check-result.schema.json` gains `requirement_ids: string[]` with validation
3. **SCHEMA-03**: Each verification runner emits `requirement_ids` in NDJSON output
4. **SCHEMA-04**: `requirements.json` gains optional `formal_models: string[]` per requirement

## 4. Design Decisions (from Research Section 6 -- Option C Hybrid)

- **model-registry.json**: Add `requirements: string[]` per model entry (coarse, model-level)
- **check-result.schema.json**: Add `requirement_ids: string[]` (emitted by runners)
- **requirements.json**: Add optional `formal_models: string[]` per requirement
- Property-level detail deferred to traceability matrix (Phase v0.25-03)
- Inline annotations deferred to v0.25-02

## 5. Key Constraints

1. **additionalProperties: false** in check-result schema -- MUST add `requirement_ids` to schema before runners emit it, or validation will reject
2. **write-check-result.cjs** is the single chokepoint -- all runners go through it. Updating this one file propagates `requirement_ids` to all runners.
3. **Auto-generated models** (QGSDQuorum, quorum-votes, quorum.pm) have version 343 from generators. Their `requirements` arrays should be seeded in model-registry but the generators may overwrite them. Generator scripts need awareness.
4. **Test model entry** (`../../../../tmp/promote-test/...`) in model-registry should be excluded from requirement seeding.
5. **TUINavigation.tla** has no matching requirement IDs -- `requirements` array should be empty `[]` or omitted.
6. **requirements.json is aggregated** by `bin/aggregate-requirements.cjs` -- `formal_models` must be added either in the aggregator or as a post-aggregation enrichment step.

## 6. Verification Runner Impact Analysis

For SCHEMA-03 (runners emit `requirement_ids`), the approach is:
1. Add `requirement_ids` to `writeCheckResult()` as optional field (array of strings, defaults to `[]`)
2. Build a `CHECK_ID_TO_REQUIREMENTS` lookup (or read from model-registry at runtime)
3. Each runner passes `requirement_ids` when calling `writeCheckResult()`
4. The lookup can be centralized in write-check-result.cjs or a new shared module

Runner count: ~15 runner files + 3 CI lint scripts = ~18 files that call `writeCheckResult()`

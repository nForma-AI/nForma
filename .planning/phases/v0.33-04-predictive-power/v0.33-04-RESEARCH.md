# Phase v0.33-04: Predictive Power - Research

**Researched:** 2026-03-10
**Domain:** Bug-to-property linking, per-model recall scoring, convergence velocity estimation
**Confidence:** HIGH

## Summary

This phase builds the feedback loop between observed failures and formal verification effectiveness. Three requirements drive implementation: PRED-01 (maintaining a `bug-to-property.json` mapping from observed bugs to formal properties), PRED-02 (computing per-model recall as an informational metric in the solve report), and TRACK-03 (convergence velocity estimation via exponential decay fit after 10+ data points).

The project already has all the infrastructure needed. Bugs are surfaced by the observe system (`bin/observe-handler-internal.cjs`, `bin/observe-handler-upstream.cjs`, etc.) and persisted to `debt.json` via `observe-debt-writer.cjs`. Formal properties live in the model registry (`model-registry.json`, 127 models) with per-model gate scores and requirement mappings. The solve trend JSONL (`solve-trend.jsonl`) provides the time series data for convergence velocity. The oscillation detector (`oscillation-detector.cjs`) already implements Mann-Kendall trend analysis and reads the trend window, establishing the pattern for reading JSONL data.

The deliverable is a new module `bin/predictive-power.cjs` that: (1) maintains `bug-to-property.json` linking debt entries to formal properties by matching requirement IDs and source layers, (2) computes recall per model as `bugs_with_matching_property / total_relevant_bugs`, and (3) fits exponential decay to per-layer residual series to estimate sessions-to-convergence. The module is called from `nf-solve.cjs` after trend entry append, with results included in the solve session summary as an informational section.

**Primary recommendation:** Implement as 2 plans: (1) bug-to-property linking module with recall computation (PRED-01, PRED-02), (2) convergence velocity estimation with exponential decay fit (TRACK-03) and solve report integration for both.

### Pattern 1: Bug-to-Property Linking (PRED-01)

**What:** Match observed bugs (from debt.json) to formal properties (from model-registry.json) that could have caught them.

**When to use:** Called during nf-solve after trend entry append.

**Matching algorithm:**
1. Read all open/recent entries from `debt.json` (the debt ledger)
2. For each debt entry, extract:
   - `formal_refs` (array of requirement IDs, already linked by `formal-ref-linker.cjs` in v0.27-03)
   - `source_type` (github, internal, etc.)
   - `fingerprint` (unique identifier)
3. For each formal model in `model-registry.json`, check if the model's `requirements` array overlaps with the debt entry's `formal_refs`
4. If overlap exists, the bug is "predictable" by that model
5. Write mappings to `bug-to-property.json`

**Schema for bug-to-property.json:**
```json
{
  "schema_version": "1",
  "generated": "2026-03-10T12:00:00.000Z",
  "total_bugs": 15,
  "total_linked": 8,
  "mappings": [
    {
      "bug_id": "debt-entry-uuid",
      "bug_fingerprint": "sha256-hash",
      "bug_title": "Short description",
      "source_type": "internal",
      "formal_refs": ["REQ-01", "REQ-02"],
      "matching_models": [
        {
          "model_path": ".planning/formal/alloy/some-model.als",
          "requirements_overlap": ["REQ-01"],
          "gate_maturity": "SOFT_GATE",
          "layer_maturity": 3
        }
      ],
      "predicted": true
    }
  ]
}
```

**Key insight:** The `formal-ref-linker.cjs` (from v0.27-03) already annotates debt entries with `formal_refs` -- requirement IDs that the debt entry relates to. This is the JOIN key. We do NOT need NLP or fuzzy matching; requirement ID overlap is the correct linking mechanism.

### Pattern 2: Per-Model Recall Computation (PRED-02)

**What:** For each formal model, compute `recall = bugs_predicted / total_relevant_bugs`.

**Definitions:**
- `total_relevant_bugs`: bugs whose `formal_refs` contain at least one requirement that the model covers
- `bugs_predicted`: subset of relevant bugs where the model's gate result was `pass` for the gate corresponding to the property type

**Computation:**
```javascript
function computePerModelRecall(mappings, perModelGates) {
  const recall = {};
  // Group mappings by model
  for (const mapping of mappings) {
    for (const match of mapping.matching_models) {
      if (!recall[match.model_path]) {
        recall[match.model_path] = { relevant: 0, predicted: 0 };
      }
      recall[match.model_path].relevant++;
      // A bug is "predicted" if the model passes gate A (grounding)
      // meaning the model has semantic coverage of the requirement
      const gates = perModelGates[match.model_path];
      if (gates && gates.gate_a && gates.gate_a.pass) {
        recall[match.model_path].predicted++;
      }
    }
  }
  // Compute recall scores
  for (const [model, counts] of Object.entries(recall)) {
    counts.recall = counts.relevant > 0
      ? +(counts.predicted / counts.relevant).toFixed(4)
      : 0;
  }
  return recall;
}
```

**Reporting:** The recall scores are included in the solve session summary as an informational section. They are NOT used as gate inputs (REQUIREMENTS.md explicitly states this; PRED-03 is deferred to future milestones).

### Pattern 3: Convergence Velocity via Exponential Decay Fit (TRACK-03)

**What:** Fit `residual(t) = a * exp(-lambda * t) + c` to per-layer residual time series. Report estimated sessions-to-convergence as `ceil(ln(residual_current / threshold) / lambda)`.

**When to use:** Only when >= 10 data points exist in solve-trend.jsonl for a given layer.

**Implementation approach -- linearized least squares:**
1. Read last N entries from solve-trend.jsonl using `readTrendWindow()` (reuse from oscillation-detector.cjs)
2. For each layer with >= 10 non-negative data points:
   - Let y_i = residual values, t_i = 0, 1, 2, ... (session index)
   - Assume c = 0 (convergence target is zero residual)
   - Transform: ln(y_i) = ln(a) - lambda * t_i (for y_i > 0 only)
   - Ordinary least squares on (t_i, ln(y_i)) gives slope = -lambda, intercept = ln(a)
   - If lambda <= 0 (not converging), report `DIVERGING` or `NOT_CONVERGING`
   - If lambda > 0: sessions_to_convergence = ceil(ln(y_current / 0.5) / lambda) (threshold at 0.5 residual)
3. Report per-layer: `{ lambda, a, sessions_to_convergence, confidence }`

**Why linearized least squares:** Simple, no iterative optimization, no external dependencies. The log-transform is the standard approach for exponential decay fitting. Only requires `Math.log()` and basic linear algebra (2x2 system).

**Edge cases:**
- Zero residuals: skip in log-transform (ln(0) is undefined); if all residuals are 0, layer is already converged
- Negative lambda: layer is diverging, report as NOT_CONVERGING
- Very small lambda: sessions_to_convergence > 50 is flagged (future VEL-01 requirement, informational only for now)

### Anti-Patterns to Avoid
- **Using recall as a gate input:** REQUIREMENTS.md explicitly defers PRED-03 to future milestones. Recall is informational ONLY in this phase.
- **Hand-rolling JSONL reading:** Reuse `readTrendWindow()` from `oscillation-detector.cjs` -- it already handles malformed lines and windowing.
- **Recomputing formal_refs:** The `formal-ref-linker.cjs` already links debt entries to requirement IDs. Do not duplicate this logic.
- **Non-linear curve fitting libraries:** The linearized approach is sufficient and avoids npm dependencies.

## Common Pitfalls

### Pitfall 1: Debt entries without formal_refs
**What goes wrong:** Many debt entries lack formal_refs (the formal-ref-linker may not match all entries). These "unlinked" bugs cannot be matched to models.
**Why it happens:** The formal-ref-linker uses requirement ID matching and may miss entries whose titles/descriptions don't correspond to tracked requirements.
**How to avoid:** Count unlinked bugs separately. Report `total_bugs`, `total_linked`, and `unlinked` in bug-to-property.json. The recall metric only considers linked bugs.
**Warning signs:** If unlinked > 80% of total, the formal-ref-linker may need improvement (separate concern, not this phase).

### Pitfall 2: Log-transform of zero residuals
**What goes wrong:** `Math.log(0)` returns `-Infinity`, corrupting the least-squares fit.
**Why it happens:** Layers that have already converged to zero residual produce zero values in the time series.
**How to avoid:** Filter out zero values before log-transform. If a layer has all zeros, report it as `CONVERGED` with `sessions_to_convergence: 0`.
**Warning signs:** `NaN` or `Infinity` in velocity output.

### Pitfall 3: Insufficient data for velocity
**What goes wrong:** Exponential fit with < 10 points is unreliable, producing wildly inaccurate estimates.
**Why it happens:** Early in a project's lifecycle, there aren't enough solve sessions.
**How to avoid:** Enforce the >= 10 data point minimum from TRACK-03. Report `INSUFFICIENT_DATA` below threshold.
**Warning signs:** `sessions_to_convergence` values > 1000 or < 0.

### Pitfall 4: Fast-mode entries skewing velocity
**What goes wrong:** Fast-mode entries have -1 for skipped layers, which distorts the time series.
**Why it happens:** Fast mode skips some sweep layers (residual = -1 by convention from v0.33-01).
**How to avoid:** Filter out -1 values (same approach as oscillation-detector.cjs Pattern 2 pitfall). Only use valid (>= 0) residual values.
**Warning signs:** Sudden spikes in residual series.

### Pitfall 5: Solve report section growing unboundedly
**What goes wrong:** Including full recall details for all 127 models overwhelms the solve report.
**Why it happens:** Naive "dump everything" approach.
**How to avoid:** Report aggregate stats (total recall, top-5 models by recall, unlinked count) in the summary. Full details in bug-to-property.json for machine consumption.
**Warning signs:** Solve report exceeds 200 lines.

### Example 1: Bug-to-Property Linking Core
```javascript
// Source: project-specific pattern based on existing debt.json and model-registry.json schemas
function linkBugsToProperties(debtPath, registryPath) {
  const debt = JSON.parse(fs.readFileSync(debtPath, 'utf8'));
  const registry = JSON.parse(fs.readFileSync(registryPath, 'utf8'));
  const models = registry.models || registry;

  const mappings = [];
  let totalLinked = 0;

  for (const entry of (debt.debt_entries || [])) {
    const refs = entry.formal_refs || [];
    if (refs.length === 0) {
      mappings.push({
        bug_id: entry.id,
        bug_fingerprint: entry.fingerprint,
        bug_title: (entry.title || '').slice(0, 120),
        source_type: entry.source_entries?.[0]?.source_type || 'unknown',
        formal_refs: [],
        matching_models: [],
        predicted: false,
      });
      continue;
    }

    const matching = [];
    for (const [modelPath, model] of Object.entries(models)) {
      if (!modelPath.startsWith('.')) continue;
      const overlap = (model.requirements || []).filter(r => refs.includes(r));
      if (overlap.length > 0) {
        matching.push({
          model_path: modelPath,
          requirements_overlap: overlap,
          gate_maturity: model.gate_maturity || 'ADVISORY',
          layer_maturity: model.layer_maturity || 0,
        });
      }
    }

    mappings.push({
      bug_id: entry.id,
      bug_fingerprint: entry.fingerprint,
      bug_title: (entry.title || '').slice(0, 120),
      source_type: entry.source_entries?.[0]?.source_type || 'unknown',
      formal_refs: refs,
      matching_models: matching,
      predicted: matching.length > 0,
    });

    if (matching.length > 0) totalLinked++;
  }

  return { total_bugs: mappings.length, total_linked: totalLinked, mappings };
}
```

### Example 2: Exponential Decay Fit
```javascript
// Source: standard linearized least-squares for exponential decay
function fitExponentialDecay(values) {
  // Filter out zeros and negatives (can't log-transform)
  const valid = [];
  for (let i = 0; i < values.length; i++) {
    if (values[i] > 0) valid.push({ t: i, y: values[i] });
  }

  if (valid.length < 10) return { status: 'INSUFFICIENT_DATA' };

  // All values are the same (no decay)
  if (valid.every(v => v.y === valid[0].y)) {
    return { status: 'STABLE', lambda: 0, sessions_to_convergence: null };
  }

  // Linearize: ln(y) = ln(a) - lambda * t
  const n = valid.length;
  let sumT = 0, sumLnY = 0, sumTLnY = 0, sumT2 = 0;
  for (const { t, y } of valid) {
    const lnY = Math.log(y);
    sumT += t;
    sumLnY += lnY;
    sumTLnY += t * lnY;
    sumT2 += t * t;
  }

  const denom = n * sumT2 - sumT * sumT;
  if (Math.abs(denom) < 1e-12) return { status: 'DEGENERATE' };

  const slope = (n * sumTLnY - sumT * sumLnY) / denom;
  const intercept = (sumLnY - slope * sumT) / n;

  const lambda = -slope;
  const a = Math.exp(intercept);

  if (lambda <= 0) return { status: 'NOT_CONVERGING', lambda, a };

  // Estimate sessions to reach threshold (0.5 residual)
  const threshold = 0.5;
  const currentResidual = values[values.length - 1];
  if (currentResidual <= threshold) {
    return { status: 'CONVERGED', lambda, a, sessions_to_convergence: 0 };
  }

  const sessions = Math.ceil(Math.log(currentResidual / threshold) / lambda);

  return {
    status: 'CONVERGING',
    lambda: +lambda.toFixed(6),
    a: +a.toFixed(4),
    sessions_to_convergence: sessions,
  };
}
```

## Open Questions

1. **Convergence threshold value**
   - What we know: Exponential decay fit estimates sessions to reach a threshold. The threshold needs to be defined.
   - What's unclear: Should the threshold be 0 (full convergence) or some small positive value like 0.5?
   - Recommendation: Use 0.5 as default threshold (reaching sub-1 residual is "effectively converged"). Make it configurable.

2. **Recall metric denominator scope**
   - What we know: PRED-02 says "total_relevant_bugs" per model. "Relevant" means bugs whose formal_refs overlap with the model's requirements.
   - What's unclear: Should we also count bugs that SHOULD have been linked but weren't (i.e., bugs in the same code area but missing formal_refs)?
   - Recommendation: For v1, only count bugs with explicit formal_refs overlap. Unlinked bugs are reported separately but don't affect recall. This matches the requirement text exactly.

3. **When to run predictive power update**
   - What we know: Must integrate into nf-solve pipeline
   - What's unclear: Before or after verdict update? Does it need trend data from current session?
   - Recommendation: Run AFTER appendTrendEntry and updateVerdicts, BEFORE persistSessionSummary. This ensures all current-session data is available and results appear in the session summary.

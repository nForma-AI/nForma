<objective>
Implement Haiku-based semantic evaluation of candidate pairs and gate semantic score aggregation, producing gate JSON files with both wiring_score and semantic_score fields.

Purpose: Candidate discovery (Plan 01) finds graph-proximate pairs; this plan evaluates whether those pairs are semantically valid using Haiku LLM verdicts, then computes per-gate semantic_score as the ratio of confirmed links to total discovered candidates. This transforms gate scoring from purely structural to semantically validated.

Output: haiku-semantic-eval.cjs enriches candidates.json with Haiku verdicts. compute-semantic-scores.cjs aggregates verdicts per gate and writes semantic_score to gate JSON files alongside existing wiring_score fields. Both scripts with full test suites.

**Part A: Create bin/haiku-semantic-eval.cjs**

Script that reads candidates.json, evaluates each candidate against Haiku for semantic match, and writes verdicts back to candidates.json.

**Core logic:**

1. Load candidates.json from `.planning/formal/candidates.json`
2. Validate proximity_index_hash matches current proximity-index.json hash — if mismatch, exit 1 with "Graph changed; rerun candidate-discovery first"
3. For each candidate without a verdict (or with stale model-text hash):
   - Construct evaluation prompt:
     ```
     You are evaluating whether a formal model semantically satisfies a requirement.
     Model path: ${candidate.model}
     Requirement ID: ${candidate.requirement}
     Does this model address the intent of this requirement? Consider both direct coverage and transitive coverage through related models.
     Respond ONLY with valid JSON: {"verdict":"yes"|"no"|"maybe","confidence":0.0-1.0,"reasoning":"..."}
     ```
   - Call Haiku API via raw HTTPS (reuse pattern from validate-requirements-haiku.cjs):
     - hostname: api.anthropic.com, path: /v1/messages
     - model: claude-haiku-4-5-20251001, max_tokens: 200
     - Headers: x-api-key from ANTHROPIC_API_KEY env var, anthropic-version: 2023-06-01
   - Parse response: try JSON.parse first, then try markdown fence extraction (```json...```), then fallback to {verdict:"maybe", confidence:0.0}
   - Store verdict, confidence, reasoning on the candidate object

4. Batch processing:
   - Process candidates in batches of 10 (configurable via --batch-size)
   - Wait 5000ms between batches (configurable via --batch-delay-ms)
   - On 429/500 errors: exponential backoff (2s, 4s, 8s), then skip with warning after 3 retries
   - Log progress to stderr: `[haiku-eval] Batch N/M: evaluated K candidates (Y yes, N no, M maybe)`

5. Caching for idempotency (SEM-05):
   - Each candidate's verdict is cached with the candidate record
   - Before evaluating, check if candidate already has a verdict AND proximity_score unchanged
   - Skip evaluation if cached verdict exists and inputs unchanged
   - Add `evaluation_timestamp` to each evaluated candidate

6. Write updated candidates.json back with enriched verdicts

**CLI interface:**
- `node bin/haiku-semantic-eval.cjs` — evaluate all unevaluated candidates
- `node bin/haiku-semantic-eval.cjs --batch-size 5` — smaller batches
- `node bin/haiku-semantic-eval.cjs --batch-delay-ms 10000` — longer delays
- `node bin/haiku-semantic-eval.cjs --force` — re-evaluate all (ignore cache)
- `node bin/haiku-semantic-eval.cjs --dry-run` — show what would be evaluated without API calls

**Error handling:**
- If ANTHROPIC_API_KEY not set: exit 1 with message
- If candidates.json missing: exit 1 with message "Run candidate-discovery first"
- Log all API errors to stderr but continue processing remaining candidates
- Never crash on a single candidate failure

**Part A: Create bin/compute-semantic-scores.cjs**

Script that reads evaluated candidates from candidates.json, computes per-gate semantic_score, and writes the score to gate JSON files alongside existing wiring_score fields.

**Core logic:**

1. Load candidates.json — validate that candidates have verdicts (error if no verdicts found: "Run haiku-semantic-eval first")
2. Load per-model-gates.json from `.planning/formal/gates/per-model-gates.json`
3. Load gate JSON files: gate-a-grounding.json, gate-b-abstraction.json, gate-c-validation.json

4. For each gate (A/B/C), compute semantic_score:
   - Identify which models pass that gate (from per-model-gates.json)
   - Count candidates whose model passes that gate AND whose verdict is "yes"
   - Count total candidates whose model passes that gate (regardless of verdict)
   - semantic_score = confirmed_count / total_candidate_count (or 0.0 if no candidates)
   - Handle "maybe" verdicts: treat as 0.5 weight (configurable via --maybe-weight flag, default 0.5)
   - Formula: `(yes_count + maybe_count * maybe_weight) / total_count`

5. Write semantic_score to each gate JSON file:
   - Read existing gate JSON (preserving all existing fields including wiring_*_score)
   - Add `semantic_score: <computed value>` field
   - Add `semantic_metadata: { evaluated_candidates, confirmed, rejected, maybe, computed: ISO timestamp }`
   - Bump schema_version to "3" (was "2" after v0.34-01)
   - Write back with JSON.stringify(data, null, 2)

6. Log summary to stderr:
   ```
   [semantic-scores] Gate A (Wiring:Evidence): semantic_score=0.85 (17 yes / 20 total)
   [semantic-scores] Gate B (Wiring:Purpose): semantic_score=0.92 (23 yes / 25 total)
   [semantic-scores] Gate C (Wiring:Coverage): semantic_score=0.78 (14 yes / 18 total)
   ```

**CLI interface:**
- `node bin/compute-semantic-scores.cjs` — compute and write
- `node bin/compute-semantic-scores.cjs --dry-run` — compute and display without writing
- `node bin/compute-semantic-scores.cjs --maybe-weight 0.0` — exclude maybe verdicts
- `node bin/compute-semantic-scores.cjs --json` — output scores as JSON to stdout

**Backward compatibility:**
- Preserve ALL existing fields in gate JSON (wiring_evidence_score, target, target_met, etc.)
- Only ADD semantic_score and semantic_metadata fields
- Consumer scripts from v0.34-01 that read wiring_*_score must continue working — semantic_score is additive

**Error handling:**
- If candidates.json has no verdicts: exit 1 "No evaluated candidates — run haiku-semantic-eval first"
- If per-model-gates.json missing: exit 1 with message
- If gate JSON files missing: create minimal structure with semantic_score only (fail-open)

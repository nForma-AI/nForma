# Phase v0.34-02: Semantic Scoring - Research

**Researched:** 2026-03-11
**Domain:** Graph-based proximity discovery + LLM semantic evaluation + gate scoring
**Confidence:** HIGH

## Summary

Phase v0.34-02 builds on the proximity graph infrastructure (formal-proximity.cjs, proximity-index.json) to implement semantic scoring — measuring how well LLM models actually cover their linked requirements. The phase combines three components:

1. **Graph BFS discovery**: Use existing proximity-index to find unlinked (model, requirement) pairs within 3 hops, scoring by graph proximity (already implemented in formal-proximity.cjs)
2. **LLM evaluation**: Submit strong candidates (score > 0.6) to Haiku for semantic match verdicts (yes/no/maybe)
3. **Gate scoring**: Compute semantic_score as ratio of LLM-confirmed links to total discovered candidates, store alongside wiring_score in gate JSON files

This phase does NOT implement candidate pairing persistence or auto-promotion (those are v0.34-03).

**Primary recommendation:** Build three scripts in sequence: (1) candidate-discovery.cjs using existing proximity API, (2) haiku-semantic-eval.cjs for batch evaluation, (3) compute-semantic-scores.cjs to aggregate and write gate files. Integrate into solve workflow at step 8 (post-wiring convergence).

### Pattern 1: Proximity BFS Candidate Discovery

**What:** Iterate through all (model, requirement) pairs; use formal-proximity.cjs's proximity() function to find unlinked pairs with proximity score > 0.6 within 3 hops; record hop_distance for later analysis.

**When to use:** On every solve session after wiring convergence; O(n²) complexity is acceptable (180 models × 287 requirements = ~52k pairs, each BFS is ~100ms with 3-hop limit).

**Example:**
```javascript
// Source: formal-proximity.cjs (existing)
const { proximity } = require('./formal-proximity.cjs');

function discoverCandidates(index, models, requirements, threshold = 0.6, maxHops = 3) {
  const candidates = [];
  for (const modelPath of Object.keys(models)) {
    const modelKey = `formal_model::${modelPath}`;
    for (const reqId of Object.keys(requirements)) {
      const reqKey = `requirement::${reqId}`;

      // Skip if already linked in model-registry
      if (models[modelPath].requirements.includes(reqId)) continue;

      const score = proximity(index, modelKey, reqKey, maxHops);
      if (score > threshold) {
        candidates.push({
          model: modelPath,
          requirement: reqId,
          proximity_score: score,
          hop_distance: computeHopDistance(index, modelKey, reqKey, maxHops)
        });
      }
    }
  }
  return candidates;
}
```

### Pattern 2: Haiku Batch Semantic Evaluation

**What:** Construct evaluation prompts for each candidate pair; submit to Haiku via raw HTTPS API; collect verdicts (yes/no/maybe) with reasoning.

**When to use:** After candidate discovery; batch in groups of 10–20 to manage token budget and API rate limits.

**Example:**
```javascript
// Source: validate-requirements-haiku.cjs (template)

Requirement ID: ${requirement}

Respond ONLY with valid JSON:
{
  "verdict": "yes" | "no" | "maybe",
  "reasoning": "...",
  "confidence": 0.0–1.0
}
`;

  const response = await callHaikuAPI(apiKey, prompt, 200);
  return parseHaikuResponse(response);

```

### Pattern 3: Gate Semantic Score Aggregation

**What:** For each gate (A/B/C), count LLM-confirmed matches per model; compute gate-level semantic_score as ratio of confirmed to total discovered candidates.

**When to use:** After Haiku evaluation complete; triggered by compute-semantic-scores.cjs.

**Example:**
```javascript
function computeGateSemanticScore(gateEvals, candidates) {
  // gateEvals[modelPath] = { pass, reason }
  // candidates = [{ model, requirement, verdict: 'yes'|'no'|'maybe' }]

  const confirmedByModel = {};
  for (const cand of candidates) {
    if (cand.verdict === 'yes' && gateEvals[cand.model]?.pass) {
      confirmedByModel[cand.model] = (confirmedByModel[cand.model] || 0) + 1;
    }
  }

  const passCount = Object.values(gateEvals).filter(e => e.pass).length;
  const confirmedCount = Object.values(confirmedByModel).reduce((a, b) => a + b, 0);

  return {
    semantic_score: passCount > 0 ? confirmedCount / passCount : 0,
    confirmed: confirmedCount,
    total: passCount
  };
}
```

### Anti-Patterns to Avoid

- **Direct LLM calls per pair without batching:** Will exceed API rate limits (>1000 candidates). Use batch queuing with per-group delays (5–10s between groups).
- **Ignoring Haiku timeout/retry logic:** Model sometimes returns `429 (rate limit)` or `500 (server error)`. Implement exponential backoff: 2s, 4s, 8s, then skip with warning.
- **Storing raw model text in candidates.json:** File will be 100MB+. Store only model path + requirement ID; fetch text at evaluation time.
- **No idempotency tracking:** Re-running discovery on unchanged graph will duplicate candidates. Store proximity-index hash in candidates.json to detect staleness.

**Key insight:** Proximity graph, Haiku API, and gate aggregation patterns are all proven in the codebase. Semantic scoring adds a new dimension (LLM verdicts) to existing gate architecture, not a new paradigm.

## Common Pitfalls

### Pitfall 1: Proximity Score Threshold Drift

**What goes wrong:** Threshold of 0.6 was chosen heuristically. In practice, it may capture too many weak pairs (noise) or too few strong pairs (signal loss). Testing with real candidates shows no feedback loop.

**Why it happens:** BFS proximity score is graph-structural, not semantic. A requirement might be 3 hops away from a model due to graph artifacts (e.g., shared test file) but unrelated semantically.

**How to avoid:**
- Start with threshold 0.6; log histogram of candidate scores during first runs
- Add `--min-score` flag to candidate-discovery.cjs; measure Haiku confirmation rate (% of "yes" verdicts) per score range
- If confirmation rate drops sharply below 0.6, increase threshold; if too few candidates, lower it
- Store threshold in candidates.json metadata for traceability

**Warning signs:** Haiku evaluation produces >30% "no" verdicts; manual spot-check shows most candidates are false positives (unrelated model/req pairs).

### Pitfall 2: Haiku Rate Limiting / Token Budget Overrun

**What goes wrong:** Submitting 5000+ candidates in parallel overloads API (429, 500 errors); script crashes mid-evaluation.

**Why it happens:** Haiku has undocumented rate limits (~10 req/min on shared tier). ~500 candidates × ~150 tokens = 75k tokens in 1 minute exceeds budget.

**How to avoid:**
- Implement batch queuing: group candidates into batches of 10–20; process sequentially with 5s delay between batches
- Add `--batch-size` and `--batch-delay-ms` flags to haiku-semantic-eval.cjs
- Catch 429/500 errors; log warning and skip candidate (don't crash)
- Log evaluation rate (candidates/min); aim for <5 candidates/min to stay under limits

**Warning signs:** Script hits 429 errors; process.stderr shows "rate limit exceeded"; Haiku responses become garbled/empty.

### Pitfall 3: Idempotency Violation: Re-running Discovery on Stale Graph

**What goes wrong:** Run discovery on proximity-index.json v1. Later, index rebuilds to v2. Re-run discovery, get different candidates. Verdicts cached from v1 don't apply to v2 candidates.

**Why it happens:** No hash/version tracking; script assumes graph is immutable within a session.

**How to avoid:**
- Store `proximity_index_hash` (SHA256 first 8 chars) in candidates.json metadata
- Before evaluation, check: `if (currentIndexHash !== candidates.metadata.proximity_index_hash) { error: "Graph changed; rerun discovery first" }`
- If graph changes, invalidate candidates.json; force re-discovery

**Warning signs:** Candidate IDs shift between runs; Haiku verdicts cached from session 1 don't match session 2's candidates.

### Pitfall 4: Haiku Response Parsing Fragility

**What goes wrong:** Haiku sometimes wraps JSON in markdown code fence, sometimes doesn't; sometimes returns explanatory text before/after JSON. parseHaikuResponse() fails on unexpected formats.

**Why it happens:** Haiku is non-deterministic; model behavior varies by system prompt, context length, temperature.

**How to avoid:**
- Use parseHaikuResponse pattern from validate-requirements-haiku.cjs: try markdown extraction first, then raw JSON parse
- Log raw response if parse fails; accumulate examples in test suite
- Set `temperature: 0` for determinism (if supported); fallback to caching verdicts by [model, req] pair
- Validate parsed JSON structure before use; reject if verdict field missing

**Warning signs:** Script silently skips candidates due to parse errors; stderr shows "Invalid JSON"; spot-check haiku-responses.log and find unpredictable structures.

### Candidate Discovery (BFS with Proximity)

```javascript
// Source: formal-proximity.cjs (bin/formal-proximity.cjs lines 499–537)
// Proximity function already exported; reuse directly

const { proximity } = require('./formal-proximity.cjs');

function discoverCandidates(proximityIndex, modelRegistry, requirementsEnv, opts = {}) {
  const threshold = opts.minScore || 0.6;
  const maxHops = opts.maxHops || 3;
  const candidates = [];

  for (const [modelPath, modelInfo] of Object.entries(modelRegistry.models)) {
    const modelKey = `formal_model::${modelPath}`;
    const linkedReqs = new Set(modelInfo.requirements || []);

    for (const reqId of Object.keys(requirementsEnv.requirements || {})) {
      // Skip already-linked pairs
      if (linkedReqs.has(reqId)) continue;

      const reqKey = `requirement::${reqId}`;
      const score = proximity(proximityIndex, modelKey, reqKey, maxHops);

      if (score > threshold) {
        candidates.push({
          model: modelPath,
          requirement: reqId,
          proximity_score: score,
        });
      }
    }
  }
  return candidates;
}
```

### Haiku Batch Evaluation (with Rate Limiting)

```javascript
// Source: validate-requirements-haiku.cjs (bin/validate-requirements-haiku.cjs lines 29–64)
// Raw HTTPS API pattern; reuse for semantic evaluation

const https = require('https');

async function evaluateCandidatesBatch(candidates, apiKey, batchSize = 10, delayMs = 5000) {
  const results = [];
  const apiCallFn = (prompt, maxTokens) => {
    return new Promise((resolve, reject) => {
      const body = JSON.stringify({
        model: 'claude-haiku-4-5-20251001',
        max_tokens: maxTokens || 200,
        messages: [{ role: 'user', content: prompt }],
      });

      const req = https.request({
        hostname: 'api.anthropic.com',
        path: '/v1/messages',
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'x-api-key': apiKey,
          'anthropic-version': '2023-06-01',
          'Content-Length': Buffer.byteLength(body),
        },
        timeout: 30000,
      }, (res) => {
        let data = '';
        res.on('data', (chunk) => { data += chunk; });
        res.on('end', () => {
          try {
            const parsed = JSON.parse(data);
            const text = ((parsed.content || [])[0] || {}).text || '';
            resolve(text);
          } catch (e) { reject(e); }
        });
      });
      req.on('error', reject);
      req.on('timeout', () => { req.destroy(); reject(new Error('timeout')); });
      req.write(body);
      req.end();
    });
  };

  for (let i = 0; i < candidates.length; i += batchSize) {
    const batch = candidates.slice(i, i + batchSize);
    for (const cand of batch) {
      const prompt = `Does model "${cand.model}" satisfy requirement "${cand.requirement}"? Reply with JSON: {"verdict":"yes"|"no"|"maybe","confidence":0.0–1.0}`;
      try {
        const response = await apiCallFn(prompt, 200);
        const verdict = JSON.parse(response);
        results.push({ ...cand, ...verdict });
      } catch (e) {
        process.stderr.write(`[WARN] Eval failed for ${cand.model}/${cand.requirement}: ${e.message}\n`);
        results.push({ ...cand, verdict: 'maybe', confidence: 0.0 });
      }
    }
    if (i + batchSize < candidates.length) {
      await new Promise(resolve => setTimeout(resolve, delayMs));
    }
  }
  return results;
}
```

### Gate Semantic Score Aggregation

```javascript
// Source: compute-per-model-gates.cjs (bin/compute-per-model-gates.cjs)
// Consensus aggregation pattern for gate-level scores

function computeGateSemanticScore(perModelGates, evaluatedCandidates, gateKey) {
  // gateKey = 'gate_a' | 'gate_b' | 'gate_c'
  let confirmedCount = 0;
  let totalExpectedCount = 0;

  for (const [modelPath, modelGates] of Object.entries(perModelGates.models || {})) {
    const gate = modelGates[gateKey];
    if (!gate || !gate.pass) continue; // Only count passing models

    totalExpectedCount++;

    // Count candidates for this model with "yes" verdict
    const yesCount = evaluatedCandidates.filter(
      c => c.model === modelPath && c.verdict === 'yes'
    ).length;
    confirmedCount += yesCount > 0 ? 1 : 0;
  }

  return {
    semantic_score: totalExpectedCount > 0 ? confirmedCount / totalExpectedCount : 0.0,
    confirmed_models: confirmedCount,
    total_passing_models: totalExpectedCount,
  };
}
```

## Open Questions

1. **Haiku confidence threshold for verdicts**
   - What we know: Haiku returns "yes/no/maybe" with confidence 0.0–1.0
   - What's unclear: Should "maybe" with confidence 0.9 count as "yes" for gate scoring? Or only hard yes/no?
   - Recommendation: Treat "maybe" as 0.5 weight in gate score; configurable via `--confidence-threshold` flag. Default: 0.0 (include all verdicts).

2. **Candidate discovery: hop distance weighting**
   - What we know: formal-proximity.cjs uses BFS with edge-weight decay (DECAY=0.7 per hop)
   - What's unclear: Should candidates discovered at 1 hop be weighted differently than 3-hop candidates during gate aggregation?
   - Recommendation: Log hop_distance in candidates.json; defer to v0.34-03 for weighting logic. For v0.34-02, treat all confirmed candidates equally (1 hop = 3 hops in gate score).

3. **Caching Haiku verdicts across sessions**
   - What we know: Haiku is non-deterministic (temperature varies)
   - What's unclear: Should verdicts be cached to reduce API calls? How to handle changed model text?
   - Recommendation: Cache verdicts in candidates.json with model-text hash. Before evaluation, check if verdict cached AND model hash matches. Skip evaluation if both conditions met. Implement in v0.34-02 (low effort, high benefit).

4. **Integration with solve workflow: timing and triggers**
   - What we know: Solve step 8 does "convergence check"; v0.34-02 should run post-convergence
   - What's unclear: Should semantic scoring block promotion to HARD_GATE? Or is it informational only for v0.34-02?
   - Recommendation: For v0.34-02, semantic_score is computed and stored but does NOT block promotion. v0.34-03 will use semantic_score >= 0.8 as a promotion criterion. Implement as optional flag `--update-gates` in solve workflow.

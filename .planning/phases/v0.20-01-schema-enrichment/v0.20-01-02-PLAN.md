Update `bin/write-check-result.cjs` to implement the v2.1 contract. The function signature and JSDoc must be updated, 5 new required-field validations added, and the NDJSON record construction extended.

**JSDoc update:** Add the 7 new parameters to the `@param` block:
- `{string} entry.check_id` - Unique step identifier (e.g. 'tla:quorum-safety'), required
- `{string} entry.surface` - STEPS tool group (e.g. 'tla', 'alloy', 'prism', 'ci'), required
- `{string} entry.property` - Human-readable check description, required
- `{number} entry.runtime_ms` - Wall-clock elapsed milliseconds from tool start to this call, required. CRITICAL for timeout-adequacy: place `const _startMs = Date.now()` immediately before the spawnSync call in callers (not at module top), so early-exit paths do not report 0ms for the actual tool run. Stored as Math.round(runtime_ms) integer.
- `{string} entry.summary` - One-line outcome (e.g. 'pass: MCsafety in 1823ms'), required
- `{string[]} [entry.triage_tags]` - Optional anomaly tags. Convention: emit `["timeout-risk"]` when runtime_ms exceeds tool-specific ceiling (TLA+ >120000ms, Alloy >60000ms, PRISM >300000ms); emit `["slow-verify"]` when runtime_ms > 50% typical. Callers in this phase emit [] by default.
- `{object} [entry.observation_window]` - Optional stochastic check window metadata (PRISM-critical). For run-prism.cjs and run-oauth-rotation-prism.cjs: MUST contain `{ window_start: ISO-string, window_end: ISO-string, n_traces: integer, n_events: integer, window_days: float }`. Used by triage bundle (v0.20-06) and evidence confidence (v0.20-05) to validate whether the observation window was long enough to support the PRISM result. An `observation_window` with `window_days < 7` and `n_traces < 30` should be flagged as low-confidence by downstream consumers.

**Validation additions** (add these immediately after the existing `VALID_RESULTS.includes` check, before the `record` construction):
```javascript
if (typeof entry.check_id !== 'string' || entry.check_id.length === 0) {
  throw new Error('[write-check-result] check_id is required and must be a non-empty string');
}
if (typeof entry.surface !== 'string' || entry.surface.length === 0) {
  throw new Error('[write-check-result] surface is required and must be a non-empty string');
}
if (typeof entry.property !== 'string' || entry.property.length === 0) {
  throw new Error('[write-check-result] property is required and must be a non-empty string');
}
if (typeof entry.runtime_ms !== 'number') {
  throw new Error('[write-check-result] runtime_ms is required and must be a number');
}
if (typeof entry.summary !== 'string' || entry.summary.length === 0) {
  throw new Error('[write-check-result] summary is required and must be a non-empty string');
}
```

**Record construction update:** Replace the current `record` object literal to include all v2.1 fields at the TOP LEVEL (not inside `metadata`):
```javascript
const record = {
  tool:       entry.tool,
  formalism:  entry.formalism,
  result:     entry.result,
  timestamp:  new Date().toISOString(),
  check_id:   entry.check_id,
  surface:    entry.surface,
  property:   entry.property,
  runtime_ms: Math.round(entry.runtime_ms),
  summary:    entry.summary,
  triage_tags: entry.triage_tags || [],
  metadata:   entry.metadata || {},
};
if (entry.observation_window !== undefined) {
  record.observation_window = entry.observation_window;
}
```

Note: `observation_window` is conditionally added so it is absent (not null/undefined) when not provided — this keeps NDJSON lines clean and allows `'observation_window' in record` checks in Test 14 to work correctly.

Do NOT change the `module.exports` line, `NDJSON_PATH`, `VALID_RESULTS`, `VALID_FORMALISMS`, or the `fs.appendFileSync` call.

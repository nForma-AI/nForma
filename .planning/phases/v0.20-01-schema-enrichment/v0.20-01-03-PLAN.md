  <files>bin/run-tlc.cjs</files>
  <action>
Read `bin/run-tlc.cjs` fully before editing. This runner handles 3 STEPS entries by accepting a config flag (MCsafety, MCliveness, MCMCPEnv). Each config must map to a distinct check_id.

**Step 1: Add CHECK_ID_MAP and PROPERTY_MAP constants** near the top of the file (after existing requires/constants, before main logic):
```javascript
const CHECK_ID_MAP = {
  'MCsafety':   'tla:quorum-safety',
  'MCliveness': 'tla:quorum-liveness',
  'MCMCPEnv':   'tla:mcp-environment',
};
const PROPERTY_MAP = {
  'MCsafety':   'Safety invariants — TypeInvariant, SafetyInvariant, no deadlock',
  'MCliveness': 'Liveness — EventuallyDecided, EventuallyTerminates',
  'MCMCPEnv':   'MCP environment — MCPEnvSafety, MCPEnvLiveness',
};
```

**Step 2: Add timer.** Locate the `spawnSync` call that invokes the TLC Java process. Place `const _startMs = Date.now();` on the line immediately BEFORE that spawnSync call (NOT at the top of the file — this ensures `runtime_ms` measures actual TLC execution, not module load time or argument parsing).

**Step 3: Compute v2.1 fields** immediately after spawnSync returns:
```javascript
const _runtimeMs = Date.now() - _startMs;
const configName = /* the config being run, e.g. 'MCsafety' */;
const check_id  = CHECK_ID_MAP[configName] || ('tla:' + configName.toLowerCase());
const property  = PROPERTY_MAP[configName] || configName;
const passed    = /* existing pass/fail boolean */;
const result_val = passed ? 'pass' : 'fail';
const summary   = result_val + ': ' + configName + ' in ' + _runtimeMs + 'ms';

// Timeout-risk annotation: TLA+ checks taking > 120s are near CI timeout boundary
const triage_tags = _runtimeMs > 120000 ? ['timeout-risk'] : [];
```

**Step 4: Update ALL writeCheckResult calls** in the file to include the new fields:
```javascript
writeCheckResult({
  tool: 'run-tlc',
  formalism: 'tla',
  result: result_val,
  check_id,
  surface: 'tla',
  property,
  runtime_ms: _runtimeMs,
  summary,
  triage_tags,
  metadata: { config: configName },
});
```

Apply this pattern to EVERY writeCheckResult call in the file, including early-exit paths (Java not found, JAR not found, etc.). For early-exit paths, use the same `_runtimeMs` calculation — it will be near 0 which correctly represents that TLC did not run. Use an appropriate result ('fail') and summary ('fail: Java not found in 0ms') for those paths.

Note: The `check_id` for early-exit paths should still use the config name if available, otherwise use `'tla:unknown'`.

**run-prism.cjs** ŌĆö covers `prism:quorum` and `prism:mcp-availability` (multi-model via --model flag):

PRISM is the stochastic model checker. Its runtime is HIGHLY VARIABLE (60-300+ seconds for large models). The `runtime_ms` value from PRISM checks is especially critical for timeout-adequacy reasoning.

This file already constructs an `observation_window` object inside `metadata`. The key change: move `observation_window` OUT of `metadata` and pass it as a top-level argument to writeCheckResult.

Multi-model handling:
```javascript
const CHECK_ID_MAP = {
  'quorum':           'prism:quorum',
  'mcp-availability': 'prism:mcp-availability',
};
const PROPERTY_MAP = {
  'quorum':           'Quorum consensus probability under agent availability rates',
  'mcp-availability': 'MCP server availability under nondeterministic failure modes',
};
```
Determine model name from CLI args (--model flag or default 'quorum').

Timer placement: `const _startMs = Date.now()` immediately before the PRISM spawnSync call (PRISM runs can take 60-300s; placing timer before the actual invocation ŌĆö not at module top ŌĆö ensures `runtime_ms` measures tool execution, not scoreboard/policy loading which happens earlier in the file).

`observation_window` promotion: The existing code builds an observation_window-like object inside `metadata`. After this change:
1. Build the observation_window struct explicitly, including the REQUIRED fields: `{ window_start: ISO, window_end: ISO, n_traces: integer_count_of_scoreboard_rounds, n_events: integer_count_of_vote_events, window_days: float }`. Where `n_traces` = number of quorum rounds observed (from scoreboard), `n_events` = total votes counted, `window_days` = (Date.now() - firstRoundTimestamp) / 86400000 (or 0 if no history).
2. Pass it as `observation_window: observationWindowObj` directly to writeCheckResult (NOT inside `metadata`).
3. Keep `metadata` for other PRISM-specific fields like `{ config, tp_rate, unavail_rate, cold_start }`.

PRISM timeout annotation thresholds (important for downstream adequacy checks):
- `runtime_ms > 300000` ŌåÆ `['timeout-risk']` (5 minutes = near CI hard limits)
- `runtime_ms > 120000` ŌåÆ `['slow-verify']` (2 minutes = warrants investigation)
- Otherwise ŌåÆ `[]`

If `observation_window.window_days < 7 || observation_window.n_traces < 30`, add `'low-confidence'` to triage_tags. This is the mechanism by which the system self-annotates when PRISM results are based on insufficient data.

Sample final writeCheckResult call structure:
```javascript
const _runtimeMs = Date.now() - _startMs;
const observationWindow = {
  window_start: firstRoundTimestamp ? new Date(firstRoundTimestamp).toISOString() : new Date().toISOString(),
  window_end:   new Date().toISOString(),
  n_traces:     quorumRoundCount,
  n_events:     totalVotes,
  window_days:  firstRoundTimestamp ? (Date.now() - firstRoundTimestamp) / 86400000 : 0,
};
const tags = [];
if (_runtimeMs > 300000) tags.push('timeout-risk');
else if (_runtimeMs > 120000) tags.push('slow-verify');
if (observationWindow.window_days < 7 || observationWindow.n_traces < 30) tags.push('low-confidence');

writeCheckResult({
  tool: 'run-prism',
  formalism: 'prism',
  result: /* pass/warn/fail */,
  check_id: CHECK_ID_MAP[modelName] || ('prism:' + modelName),
  surface: 'prism',
  property: PROPERTY_MAP[modelName] || modelName,
  runtime_ms: _runtimeMs,
  summary: result + ': ' + modelName + ' in ' + _runtimeMs + 'ms',
  triage_tags: tags,
  observation_window: observationWindow,
  metadata: { config: modelName, tp_rate: liveTPRate, unavail: liveUnavail },
});
```

Apply this pattern to ALL writeCheckResult calls in run-prism.cjs (including early-exit paths for binary not found, model file not found, policy load failure).

# Phase v0.37-04: Quorum Precedents - Research

**Researched:** 2026-03-16
**Domain:** Quorum debate mining, precedent extraction, keyword matching, TTL-based pruning
**Confidence:** HIGH

## Summary

Phase v0.37-04 implements a precedent system that mines past quorum debate archives to produce reusable decision knowledge. The phase requires three core capabilities: (1) extract past decisions (BLOCK/APPROVE) and their reasoning into a structured JSON database, (2) match incoming quorum questions to precedents via keyword overlap, and (3) inject up to 3 relevant precedents into dispatch prompts while auto-pruning entries older than 90 days.

The implementation integrates tightly with the existing quorum dispatch system (`bin/quorum-slot-dispatch.cjs`), which already has requirement-based context injection patterns. The precedent extraction tool mirrors the architecture of `bin/extract-annotations.cjs`, scanning markdown debate files using regex and writing JSON artifacts to `.planning/quorum/precedents.json`. Keyword matching follows the same stopword-filtered tokenization logic already used in `matchRequirementsByKeywords()` from the quorum dispatch module.

**Primary recommendation:** Build `bin/extract-precedents.cjs` as a pure file-scanning tool first (testable without file I/O mocking), then extend `quorum-slot-dispatch.cjs` with precedent matching and injection functions, following the proven pattern of requirements matching. The keyword matching approach is sufficient for an MVP; semantic embedding is deferred to future work.

### Pattern 2: Keyword-Based Precedent Matching
**What:** Match incoming quorum question to up to 3 relevant precedents using stopword-filtered keyword overlap
**When to use:** During quorum dispatch, before calling slot worker; ensure precedents are contextually relevant
**Example:**
```javascript
// Source: bin/quorum-slot-dispatch.cjs (lines 156–228), existing matchRequirementsByKeywords logic
function matchPrecedentsByKeywords(precedents, question, artifactPath) {
  const questionKeywords = extractKeywords(question);
  const pathKeywords = artifactPath ? extractPathKeywords(artifactPath) : new Set();

  const scored = precedents.map(prec => {
    let score = 0;
    const precedentKeywords = extractKeywords(prec.question);
    const outcomeKeywords = extractKeywords(prec.outcome);

    // Boost score if question contains outcome keywords (high relevance)
    for (const kw of outcomeKeywords) {
      if (questionKeywords.has(kw)) score += 2;
    }
    for (const kw of precedentKeywords) {
      if (questionKeywords.has(kw)) score += 1;
    }

    return { precedent: prec, score };
  });

  return scored
    .filter(({ score }) => score > 0)
    .sort((a, b) => b.score - a.score)
    .slice(0, 3); // Max 3 precedents per dispatch
}
```

### Pattern 3: TTL-Based Pruning
**What:** Exclude precedents older than 90 days (default) from dispatch prompts; auto-remove from precedents.json during extraction
**When to use:** At extraction time (remove stale entries) and at matching time (double-check age before injection)
**Example:**

```javascript
function isPrecedentFresh(precedent, maxAgeMs = 90 * 24 * 60 * 60 * 1000) {

  const now = new Date();
```

### Anti-Patterns to Avoid
- **No semantic search in MVP:** Keyword matching is sufficient for early pruning. Full semantic embedding (via external API or local model) is out of scope until debate corpus grows beyond ~500 entries and ROI is clearer.
- **No circular precedent references:** Don't allow a new debate's question to match against itself if the precedent was just written. Ensure extraction timestamp is current before matching.
- **No manual precedent curation:** Extraction is automatic; human review happens through quorum process itself (if a precedent was wrong, future debates will override it).
- **Don't duplicate debate content in precedents.json:** Store only { question, date, consensus, outcome, computed_at }; do not duplicate full debate markdown.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TTL-based expiration tracking | Custom timestamp management with drift correction | Simple `(now - date) < maxAge` comparison | Date arithmetic is error-prone; use built-in Date objects |
| Debate file scanning and parsing | Custom regex per debate type | Single `extractPrecedentFromDebate()` that tries multiple patterns | Reduces fragility; easier to add new debate formats |
| Keyword extraction and stopword filtering | Custom word tokenizer | Reuse existing `extractKeywords()` from quorum-slot-dispatch.cjs | Already tested; consistent with requirements matching |
| JSON marshalling and pretty-printing | Manual string concatenation | `JSON.stringify(obj, null, 2)` and `JSON.parse()` | Built-in; handles edge cases (escaping, unicode, large numbers) |
| Precedent relevance scoring | Custom distance metrics | Keyword overlap + category boost (proven in requirements matching) | Simple, interpretable, already working |

**Key insight:** The precedent system is fundamentally a "database lookup" problem (match question → find similar past decisions). Keyword-based matching is well-suited for this MVP because debates are written by humans with clear language; semantic search becomes necessary only at scale (500+ entries) or if language drifts significantly.

## Common Pitfalls

### Pitfall 1: Debate Parsing Fragility
**What goes wrong:** Debate files use informal markdown; header format varies slightly (e.g., "Consensus: APPROVE" vs "Consensus:APPROVE" vs "consensus: APPROVE"). Parser breaks on unexpected formatting and silently skips debates.
**Why it happens:** Markdown is not strictly structured; human authors write flexibly; early debates may have different formatting conventions than recent ones.
**How to avoid:** Use case-insensitive regex matching with `\s*` for whitespace tolerance. Log skipped debates with reason (missing Consensus, malformed Date, etc.). Test on existing debates before deployment.
**Warning signs:** Extract tool reports "found 0 precedents" when debate directory clearly has entries; log output shows skipped files.

### Pitfall 2: Overstale Precedents in Dispatch
**What goes wrong:** A precedent is 100+ days old but still appears in a quorum prompt because TTL check was skipped or timestamp comparison has a bug.
**Why it happens:** TTL validation can be missed if added after precedent loading; timezone bugs or date parsing errors cause incorrect age calculation.
**How to avoid:** Always check freshness during matching, not just at extraction time. Write explicit tests for boundary cases (date exactly 90 days old, 89 days, 91 days). Use UTC timestamps consistently.
**Warning signs:** Quorum debates mention precedents that are months old; timestamps in precedents.json appear malformed (e.g., missing year or timezone info).

### Pitfall 3: Runaway Keyword Match Scores
**What goes wrong:** A generic word like "test" or "fix" matches dozens of precedents, so the top-3 selection is dominated by noise (e.g., a precedent about "fix-tests workflow" matches a question about "quorum verification" and scores high due to word frequency).
**Why it happens:** Stopword list doesn't filter domain-specific noise words; keyword extraction doesn't weight by importance (first word vs. last word treated equally).
**How to avoid:** Curate stopword list to include project-specific noise (e.g., "test", "fix", "plan", "phase" if they appear in many debates). Boost scoring for question keywords that appear in precedent outcome (key decisions) rather than just precedent question. Cap matches at 3 to force selection of highest-quality hits.
**Warning signs:** Precedent list for a targeted question contains obviously unrelated debates; test output shows precedent scores clustering near zero.

### Pitfall 4: Precedent Injection Breaks Prompt Character Limit
**What goes wrong:** Adding 3 precedents to the quorum dispatch prompt pushes total to 4000+ characters, violating the "TOTAL response under 3000 characters" instruction to models and causing confusion.
**Why it happens:** Precedent formatting includes full question + outcome; 3 × ~400 chars = 1200 chars added before model response obligation.
**How to avoid:** Truncate precedent question and outcome to 80–100 characters each with ellipsis. Calculate prompt size before injection; if injected precedents push past 2000 chars, reduce count to 2 or 1. Document size budget in prompt construction code.
**Warning signs:** Model responses mention feeling constrained or mention precedent text being cut off; total prompt size grows unexpectedly when tests add precedent section.

### Pitfall 5: Circular Dependency in Precedent Matching
**What goes wrong:** A debate file is extracted and written to precedents.json, then immediately loaded and matched against itself, creating a self-referential precedent in the next quorum dispatch.
**Why it happens:** Extraction timestamp (`computed_at`) is not set correctly before matching; or matching uses a stale cache of precedents.json from before extraction completed.
**How to avoid:** Set `computed_at` to extraction wall-clock time, not the debate's Date field. In tests, advance mock time between extraction and matching. Never cache precedents.json in memory across multiple dispatch calls; reload from disk.
**Warning signs:** A quorum precedent references itself or a very recent debate that is identical to the current question.

### Extract Debate Metadata from Markdown
```javascript
// Source: bin/extract-annotations.cjs (lines 38–79), adapted pattern
// Matches: Question: ..., Date: YYYY-MM-DD, Consensus: ..., ## Outcome section

function extractPrecedentMetadata(filePath, content) {
  const lines = content.split('\n');
  const result = {};

  for (const line of lines) {
    // Extract: Question: ...
    const qMatch = line.match(/^Question:\s*(.+)$/);
    if (qMatch) {
      result.question = qMatch[1].trim();
    }

    // Extract: Date: YYYY-MM-DD
    const dMatch = line.match(/^Date:\s*(\d{4}-\d{2}-\d{2})/);
    if (dMatch) {
      result.date = dMatch[1];
    }

    // Extract: Consensus: APPROVE|BLOCK|...
    const cMatch = line.match(/^Consensus:\s*(\w+)/i);
    if (cMatch) {
      result.consensus = cMatch[1].toUpperCase();
    }
  }

  // Extract Outcome section
  const outcomeMatch = content.match(/## Outcome\n([\s\S]*?)(?=\n## |\n# |\Z)/);
  if (outcomeMatch) {
    result.outcome = outcomeMatch[1].trim();
  }

  // Validate: must have question, date, consensus
  if (result.question && result.date && result.consensus) {
    result.computed_at = new Date().toISOString();
    return result;
  }
  return null;
}
```

### Load and Filter Precedents by Freshness
```javascript
// Source: conceptual, mirrors TTL patterns
// Prunes entries older than 90 days; validates timestamp format

function loadPrecedentsWithTTL(precedentsJsonPath, maxAgeMs = 90 * 24 * 60 * 60 * 1000) {
  try {
    const raw = JSON.parse(fs.readFileSync(precedentsJsonPath, 'utf8'));
    const precedents = Array.isArray(raw.precedents) ? raw.precedents : [];
    const now = Date.now();

    return precedents.filter(prec => {
      const dateStr = prec.date || '';
      const parsed = new Date(dateStr);
      if (isNaN(parsed.getTime())) {
        console.warn(`Skipping precedent with invalid date: ${dateStr}`);
        return false;
      }
      const age = now - parsed.getTime();
      return age < maxAgeMs;
    });
  } catch (e) {
    console.error(`Failed to load precedents: ${e.message}`);
    return [];
  }
}
```

### Match Precedents by Keyword Overlap
```javascript
// Source: bin/quorum-slot-dispatch.cjs (lines 164–228), adapted to precedents
// Reuses extractKeywords() and STOPWORDS from quorum dispatch

function matchPrecedentsByKeywords(precedents, question, maxMatches = 3) {
  const questionKeywords = extractKeywords(question);
  if (questionKeywords.size === 0) {
    return []; // No meaningful keywords, no matches
  }

  const scored = precedents.map(prec => {
    let score = 0;

    // Match on precedent question
    const precQKeywords = extractKeywords(prec.question);
    for (const kw of precQKeywords) {
      if (questionKeywords.has(kw)) score += 1;
    }

    // Boost score for outcome keyword matches (indicates decision relevance)
    const outcomeKeywords = extractKeywords(prec.outcome);
    for (const kw of outcomeKeywords) {
      if (questionKeywords.has(kw)) score += 2;
    }

    return { precedent: prec, score };
  });

  return scored
    .filter(({ score }) => score > 0)
    .sort((a, b) => b.score - a.score)
    .slice(0, maxMatches)
    .map(({ precedent }) => precedent);
}
```

```javascript

function formatPrecedentsSection(precedents) {
  if (!precedents || precedents.length === 0) {

  }

  const lines = [];
  lines.push('=== PAST QUORUM PRECEDENTS ===');
  lines.push('The following similar debates provide context for your decision:');

    lines.push(`  Outcome: ${truncO}`);
  }

  lines.push('');
  lines.push('================================');

  return lines.join('\n');
}
```

## Open Questions

1. **Keyword weighting strategy**
   - What we know: Stopword-filtered keyword overlap works for requirements matching (proven in quorum-slot-dispatch.cjs lines 164–228)
   - What's unclear: Should outcome keywords be boosted 2× or 3×? Should question-match keywords be weighted differently?
   - Recommendation: Start with 2× boost for outcome keywords (conservative); enable tuning in future phases via FPTUNE feedback if precedent matches are noisy

2. **Precedent source corpus boundaries**
   - What we know: Debate files are in `.planning/quorum/debates/*.md`; ~71 files as of 2026-03-16
   - What's unclear: Should we limit to debates with CONSENSUS >= N votes? Should we exclude specific models' debates (e.g., UNAVAIL slots)?
   - Recommendation: Include all debates with Consensus: APPROVE/BLOCK regardless of rounds or model availability. INCONCLUSIVE debates are excluded by design. Future phases can add confidence weighting.

3. **Precedent freshness and debate evolution**
   - What we know: Default TTL is 90 days; debates can be updated if a decision is overturned
   - What's unclear: If a debate's date is updated, should we treat it as a new precedent or update the existing entry?
   - Recommendation: Treat date field as immutable (day debate was decided). If a decision is superseded, a new debate with new date becomes the precedent. No edit-in-place for historical debates.

4. **Integration with quorum slot dispatch**
   - What we know: quorum-slot-dispatch.cjs accepts --artifact-path, --question, and builds prompts
   - What's unclear: Where should precedent loading happen? Should we load precedents.json once per slot dispatch or once per round?
   - Recommendation: Load precedents.json once at slot dispatch start (before calling buildModeAPrompt). Cache in local variable; no global state. TTL check happens during matching, not separately.

## Summary

Phase v0.18-01 adds token observability to QGSD's quorum pipeline. The goal is to make per-slot token consumption visible in `/qgsd:health` and to persist a structured JSONL record after each quorum slot-worker round. This phase is architecturally independent — it adds one new hook file, extends install.js to register it, and extends the health workflow to display the accumulated data.

The central mechanism is Claude Code's `SubagentStop` hook event, which fires when a `Task`-spawned subagent finishes and provides `agent_transcript_path` — the path to the subagent's own JSONL transcript. That transcript contains `message.usage.{input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens}` on every assistant entry. The hook reads these, sums them, and appends a one-line JSON record to `.planning/token-usage.jsonl`. The slot name is resolved via a correlation file written by a `SubagentStart` hook (which receives `agent_id` at spawn time, before the subagent runs).

CLI-based slots (gemini-1, codex-1) are subprocess-dispatched via `call-quorum-slot.cjs` and do not produce Claude Code subagent transcripts. These slots must be represented in the JSONL log as sentinel records with `tokens: null` so OBSV-04 is satisfied. The sentinel record is written by the quorum dispatch layer (the quorum.md instructions or a hook on the dispatch side), not by the token collector hook.

**Primary recommendation:** Implement `hooks/qgsd-token-collector.js` as a SubagentStop command hook with matcher `qgsd-quorum-slot-worker`, async:true. Use `hooks/qgsd-slot-correlator.js` as a SubagentStart command hook with the same matcher to write correlation files. Register both in install.js. Extend the health workflow to read and display `.planning/token-usage.jsonl`.

### Pattern 1: SubagentStop Token Collector Hook

**What:** Fires when any `qgsd-quorum-slot-worker` subagent finishes. Reads the subagent's transcript, sums token usage, appends to JSONL log.

**When to use:** Token collection for any Task-spawned subagent of type `qgsd-quorum-slot-worker`.

```javascript
// Source: claude code official docs + QGSD .planning/research/STACK.md (2026-02-27)
// hooks/qgsd-token-collector.js
'use strict';
const fs = require('fs');
const path = require('path');

let raw = '';
process.stdin.setEncoding('utf8');
process.stdin.on('data', c => raw += c);
process.stdin.on('end', () => {
  try {
    const input = JSON.parse(raw);
    // Only process qgsd-quorum-slot-worker subagents (matcher also filters, belt+suspenders)
    if (input.agent_type !== 'qgsd-quorum-slot-worker') { process.exit(0); return; }

    const transcriptPath = input.agent_transcript_path;
    if (!transcriptPath || !fs.existsSync(transcriptPath)) {
      // Transcript unavailable — write null-token record so slot is present in log
      appendRecord(input, null, null);
      process.exit(0);
      return;
    }

    // Sum usage across all non-sidechain, non-error assistant entries
    const lines = fs.readFileSync(transcriptPath, 'utf8').split('\n').filter(l => l.trim());
    let inputTokens = 0, outputTokens = 0, cacheCreate = 0, cacheRead = 0;
    for (const line of lines) {
      try {
        const entry = JSON.parse(line);
        if (entry.type !== 'assistant') continue;
        if (entry.isSidechain === true) continue;
        if (entry.isApiErrorMessage === true) continue;
        const u = entry.message && entry.message.usage;
        if (!u) continue;
        inputTokens  += (u.input_tokens || 0);
        outputTokens += (u.output_tokens || 0);
        cacheCreate  += (u.cache_creation_input_tokens || 0);
        cacheRead    += (u.cache_read_input_tokens || 0);
      } catch (_) {}
    }

    appendRecord(input, inputTokens, outputTokens, cacheCreate, cacheRead);
    process.exit(0);
  } catch (_) { process.exit(0); } // fail-open
});

function resolveSlot(input) {
  // Try correlation file written by SubagentStart hook
  const corrPath = path.join(
    process.cwd(), '.planning',
    'quorum-slot-corr-' + input.agent_id + '.json'
  );
  if (fs.existsSync(corrPath)) {
    try {
      const data = JSON.parse(fs.readFileSync(corrPath, 'utf8'));
      try { fs.unlinkSync(corrPath); } catch (_) {}
      if (data.slot) return data.slot;
    } catch (_) {}
  }
  // Fallback: parse "slot: <name>" from last_assistant_message
  const msg = input.last_assistant_message || '';
  const m = msg.match(/^slot:\s*(\S+)/m);
  if (m) return m[1];
  // Final fallback: use agent_id
  return input.agent_id || 'unknown';
}

function appendRecord(input, inputTokens, outputTokens, cacheCreate, cacheRead) {
  const slot = resolveSlot(input);
  const record = {
    ts: new Date().toISOString(),
    session_id: input.session_id,
    agent_id: input.agent_id,
    slot,
    input_tokens: inputTokens,
    output_tokens: outputTokens,
    cache_creation_input_tokens: cacheCreate !== undefined ? cacheCreate : null,
    cache_read_input_tokens: cacheRead !== undefined ? cacheRead : null,
  };
  try {
    const logPath = path.join(process.cwd(), '.planning', 'token-usage.jsonl');
    fs.mkdirSync(path.dirname(logPath), { recursive: true });
    fs.appendFileSync(logPath, JSON.stringify(record) + '\n', 'utf8');
  } catch (_) {} // observational — never fail
}
```

### Pattern 2: SubagentStart Correlator Hook

**What:** Fires when a `qgsd-quorum-slot-worker` subagent is spawned. Receives `agent_id` and writes a correlation file mapping `agent_id` to slot name.

**Why:** `agent_id` is NOT predictable before the Task call — Claude Code assigns it. The `SubagentStart` hook fires immediately after Task dispatch, providing `agent_id` deterministically before the subagent runs. This resolves the STATE.md blocker: *"verify whether agent_id is predictable before Task spawn or only after"* — it is available in `SubagentStart`, not before.

**Correlation file naming:** `.planning/quorum-slot-corr-<agent_id>.json` — ephemeral, deleted by token collector on read.

```javascript
// Source: claude code official docs SubagentStart payload (code.claude.com/docs/en/hooks)
// hooks/qgsd-slot-correlator.js
'use strict';
const fs = require('fs');
const path = require('path');

// The slot name must be embedded in the Task prompt by the quorum dispatch.
// The SubagentStart prompt field contains the subagent's prompt text.
// quorum.md writes prompts like: "slot: claude-1\nround: 1\n..."
// We parse "slot: <name>" from the prompt to get the slot name.

let raw = '';
process.stdin.setEncoding('utf8');
process.stdin.on('data', c => raw += c);
process.stdin.on('end', () => {
  try {
    const input = JSON.parse(raw);
    if (input.agent_type !== 'qgsd-quorum-slot-worker') { process.exit(0); return; }
    if (!input.agent_id) { process.exit(0); return; }

    // Parse slot name from the subagent prompt (if available in SubagentStart payload)
    // NOTE: SubagentStart does NOT include the prompt text in its payload (confirmed docs).
    // The correlation file must therefore use a different strategy:
    // We write a "pending" file keyed by agent_id with no slot yet;
    // the actual slot is resolved by the token collector from last_assistant_message.
    // However, if a "pending dispatch" file exists (written by quorum.md before Task spawn),
    // we can read it here.
    //
    // Recommended approach: quorum.md writes .planning/quorum-dispatch-pending.json before
    // spawning each Task batch. SubagentStart reads it by correlation (agent ordering).
    // Simpler approach: SubagentStart just writes a placeholder; token collector uses
    // last_assistant_message fallback for slot name.

    // Write correlation placeholder (agent_id → slot unknown; collector uses fallback)
    const corrPath = path.join(
      process.cwd(), '.planning',
      'quorum-slot-corr-' + input.agent_id + '.json'
    );
    fs.mkdirSync(path.dirname(corrPath), { recursive: true });
    fs.writeFileSync(corrPath, JSON.stringify({
      agent_id: input.agent_id,
      ts: new Date().toISOString(),
      slot: null, // resolved by token collector from last_assistant_message
    }), 'utf8');
    process.exit(0);
  } catch (_) { process.exit(0); }
});
```

**Important:** SubagentStart payload does NOT include the prompt text (confirmed in official docs). The slot name cannot be extracted from SubagentStart directly. The slot correlator's primary value is confirming the agent_id exists; actual slot name resolution falls through to `last_assistant_message` parsing in the token collector.

**Revised recommendation:** The correlator hook can be simplified or omitted. The token collector should primarily use `last_assistant_message` to parse `slot: <name>` from the YAML preamble of the quorum slot-worker output. This is reliable because qgsd-quorum-slot-worker.md always starts with the YAML block echoed back.

### Pattern 3: CLI Slot Sentinel Record (OBSV-04)

**What:** CLI-based slots (gemini-1, codex-1) are dispatched via `call-quorum-slot.cjs` as subprocess. They do NOT produce Claude Code Task subagents. `SubagentStop` never fires for them. To satisfy OBSV-04 (logged with `tokens: null`, not omitted), a sentinel record must be written by the dispatch layer.

**Mechanism:** The quorum slot-worker orchestration (inline in quorum.md or call-quorum-slot.cjs) writes a sentinel record immediately after each CLI slot dispatch completes:

```json
{"ts":"...","session_id":"abc123","agent_id":null,"slot":"gemini-1","input_tokens":null,"output_tokens":null,"cache_creation_input_tokens":null,"cache_read_input_tokens":null}
```

**Implementation options:**
1. Extend `call-quorum-slot.cjs` to write the sentinel record on exit (stdout as normal, plus append to token-usage.jsonl on the same disk).
2. The quorum.md markdown instructions tell Claude to write the sentinel record via Bash after each CLI slot call.
3. A new `bin/log-token-sentinel.cjs` script called from quorum.md.

Option 1 (extend call-quorum-slot.cjs) is the most reliable — it is structural enforcement, not instruction-following.

### Pattern 4: Health Display

**What:** `/qgsd:health` reads `.planning/token-usage.jsonl`, aggregates by slot, ranks by total tokens (input + output), and prints a table.

**Implementation:** The health workflow (qgsd-core/workflows/health.md) adds a new step. The display is produced via a Node.js one-liner or a short inline bash script:

```bash
# Source: QGSD architecture — inline in health.md step
node -e "
const fs = require('fs');
const path = require('path');
const logPath = path.join(process.cwd(), '.planning', 'token-usage.jsonl');
if (!fs.existsSync(logPath)) { console.log('  No token data yet.'); process.exit(0); }
const lines = fs.readFileSync(logPath, 'utf8').split('\n').filter(l => l.trim());
const slots = {};
for (const line of lines) {
  try {
    const r = JSON.parse(line);
    const key = r.slot || 'unknown';
    if (!slots[key]) slots[key] = { input: 0, output: 0, rounds: 0 };
    slots[key].input  += (r.input_tokens || 0);
    slots[key].output += (r.output_tokens || 0);
    slots[key].rounds++;
    if (r.input_tokens === null) slots[key].null_count = (slots[key].null_count || 0) + 1;
  } catch (_) {}
}
const sorted = Object.entries(slots).sort((a, b) => (b[1].input + b[1].output) - (a[1].input + a[1].output));
console.log('  TOKEN CONSUMPTION (ranked by total, last session)');
console.log('  slot             input        output       rounds');
for (const [slot, data] of sorted) {
  const inp = data.input === 0 && data.null_count ? 'null' : data.input.toLocaleString();
  const out = data.output === 0 && data.null_count ? 'null' : data.output.toLocaleString();
  console.log('  ' + slot.padEnd(16) + inp.padStart(12) + out.padStart(13) + String(data.rounds).padStart(9));
}
"
```

### Anti-Patterns to Avoid

- **Using PostToolUse for token tracking:** PostToolUse fires for ALL tool calls (Bash, Read, Write, etc.); the matcher cannot filter to only Task calls that create slot-workers by agent_type. SubagentStop is the correct event.
- **Polling the transcript directory:** Racy. SubagentStop is the correct synchronous trigger.
- **Using the main transcript for per-slot attribution:** Main transcript interleaves all agents. Requires `parentUuid` chain traversal. `agent_transcript_path` gives a clean per-agent view.
- **Writing correlation files from qgsd-prompt.js (before Task spawn):** `agent_id` is not known before the Task call resolves. qgsd-prompt.js fires at prompt-submit time, before any Task is dispatched. The only hook that provides `agent_id` at spawn time is `SubagentStart`.
- **Registering the token collector as a synchronous (non-async) hook:** Token collection is observational. Using `async: true` ensures it does not block the subagent stop event, matching the non-blocking pattern used for conformance events.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token counting for Claude Code subagents | Custom API instrumentation or OpenTelemetry | `SubagentStop` hook + `agent_transcript_path` JSONL parsing | Claude Code already writes the transcript with exact token counts in `message.usage`; parsing it is 30 lines, zero deps |
| Structured logging | `pino` or `winston` | `fs.appendFileSync` to `.planning/token-usage.jsonl` | Follows existing conformance-events.jsonl pattern; atomic for records < 4KB; no transitive deps |
| Schema validation | `zod` or `ajv` | Manual null-checks in token collector | 5-field schema; manual `r.input_tokens || 0` is clearer and smaller |
| Unique IDs for correlation files | `uuid` library | `input.agent_id` from hook payload | Claude Code provides `agent_id`; no generation needed |

**Key insight:** Claude Code's hook system already provides all the instrumentation primitives needed. The token data exists in the subagent transcript. The task is plumbing, not building.

## Common Pitfalls

### Pitfall 1: install.js sync requirement

**What goes wrong:** Hook source files in `hooks/` are modified but the installed hooks at `~/.claude/hooks/` are not updated. The new hook fires the old code (or does not fire at all if not registered).

**Why it happens:** CLAUDE.md and MEMORY.md both document this: `install.js` reads from `hooks/dist/` not `hooks/`. Source edits without sync are silent non-deployments.

**How to avoid:** Every task that modifies a hook file MUST include:
```bash
cp hooks/qgsd-token-collector.js hooks/dist/qgsd-token-collector.js
cp hooks/qgsd-slot-correlator.js hooks/dist/qgsd-slot-correlator.js
node bin/install.js --claude --global
```
**Warning signs:** Hook fires old behavior; `ls ~/.claude/hooks/` shows old file timestamp; changes in hooks/ are not reflected in behavior.

### Pitfall 2: token-usage.jsonl grows unboundedly

**What goes wrong:** Every quorum round appends records. After months of use, the file becomes large and health display slows.

**Why it happens:** Append-only log with no rotation or compaction.

**How to avoid:** Health display should read only the last N records (e.g., last 100 lines). Use `fs.readFileSync` and slice the lines array: `lines.slice(-100)`. Add a comment in the health display code documenting this limit.

**Warning signs:** Token display takes more than 200ms; file exceeds 1MB.

### Pitfall 3: Correlation file race condition

**What goes wrong:** Multiple slot-workers finish nearly simultaneously. Two `SubagentStop` hooks fire concurrently (async). Both try to read and delete the same correlation file.

**Why it happens:** `async: true` means hooks run in parallel. `fs.existsSync` + `fs.readFileSync` + `fs.unlinkSync` is not atomic.

**How to avoid:** The correlation file is keyed by `agent_id` (unique per subagent). Each SubagentStop fires for a different `agent_id`, so each reads its own distinct file. No race condition — files are unique per agent.

**Warning signs:** If a slot name resolves to `unknown` or `agent_id` value instead of a slot name, it means the correlation file was missing (not a race — a timing issue between SubagentStart and SubagentStop).

### Pitfall 4: CLI slots missing from token-usage.jsonl (OBSV-04 failure)

**What goes wrong:** gemini-1 and codex-1 appear in the quorum round but no record exists in `token-usage.jsonl` for them. The health display only shows MCP-based slots.

**Why it happens:** SubagentStop only fires for Task-spawned Claude Code subagents. CLI slots (subprocess via `call-quorum-slot.cjs`) never produce a SubagentStop event.

**How to avoid:** The sentinel record write for CLI slots must be implemented in the dispatch layer, not the hook. Use `call-quorum-slot.cjs` exit to write the sentinel: after the subprocess finishes, append a `tokens: null` record. This is structural — it cannot be bypassed by instruction-following errors.

**Warning signs:** OBSV-04 test fails; health display shows no row for gemini-1 or codex-1.

### Pitfall 5: settings.json hook event name is case-sensitive

**What goes wrong:** Registering `"subagentStop"` or `"subagent_stop"` instead of `"SubagentStop"` silently fails — hook never fires.

**Why it happens:** Claude Code hook event names are PascalCase. `settings.json` registration must match exactly.

**How to avoid:** Use `"SubagentStop"` in all settings.json registrations. Test with `node bin/install.js --claude --global` and verify `~/.claude/settings.json` contains the correct event name.

**Warning signs:** Token collector never writes to token-usage.jsonl after quorum rounds.

### call-quorum-slot.cjs sentinel append (implementation sketch)

```javascript
// Source: bin/call-quorum-slot.cjs — to be added at exit
// Append null-token sentinel for CLI (subprocess) slots
function appendTokenSentinel(slot, sessionId, projectRoot) {
  try {
    const record = JSON.stringify({
      ts: new Date().toISOString(),
      session_id: sessionId || null,
      agent_id: null,
      slot,
      input_tokens: null,
      output_tokens: null,
      cache_creation_input_tokens: null,
      cache_read_input_tokens: null,
    });
    const logPath = path.join(projectRoot, '.planning', 'token-usage.jsonl');
    fs.mkdirSync(path.dirname(logPath), { recursive: true });
    fs.appendFileSync(logPath, record + '\n', 'utf8');
  } catch (_) {} // observational
}
```

## Open Questions

1. **SubagentStart payload: does it include prompt text?**
   - What we know: Official docs state `SubagentStart` receives `agent_id` and `agent_type`. The example payload does not show a `prompt` field.
   - What's unclear: Whether any version of Claude Code includes the prompt or a portion of it in SubagentStart.
   - Recommendation: Do not depend on prompt availability in SubagentStart. Use `last_assistant_message` in SubagentStop for slot name resolution. This is confirmed reliable (quorum slot-workers always echo back the YAML preamble including `slot: <name>`).

2. **Will `async: true` hooks block the test runner?**
   - What we know: The `async: true` flag means the hook runs in the background without blocking the subagent stop event. Official docs confirm this.
   - What's unclear: Whether Claude Code's test infrastructure (quorum-test.md) can verify that the async hook fired correctly (timing-dependent).
   - Recommendation: For the token collector, `async: true` is correct for production. Integration tests for the hook itself spawn it synchronously (via `spawnSync`), so this is not a test problem.

3. **Session_id availability in SubagentStop for cross-session correlation**
   - What we know: `session_id` is in the SubagentStop common fields. The record schema includes it.
   - What's unclear: Whether the session_id in SubagentStop matches the main session's session_id or is the subagent's own session_id.
   - Recommendation: Include `session_id` in the JSONL record regardless. The planner can clarify which session_id is used in implementation. The field is informational for future BDGT-02 cross-session analysis.

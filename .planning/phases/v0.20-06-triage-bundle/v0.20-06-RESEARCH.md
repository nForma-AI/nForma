# Phase v0.20-06: Triage Bundle - Research

**Researched:** 2026-02-28
**Domain:** NDJSON triage analysis, per-check delta reporting, failure suspect detection
**Confidence:** HIGH

## Summary

Phase v0.20-06 produces two human-readable triage documents from `check-results.ndjson` after every formal verification run. The requirement is to implement `bin/generate-triage-bundle.cjs`, a tool that:

1. **TRIAGE-01**: Reads the NDJSON check results file, compares against the previous run's snapshot, and writes `formal/diff-report.md` — a delta report showing which checks changed their result status (pass→fail, inconclusive→pass, etc.) with runtime metrics and summaries. Also produces `formal/suspects.md` — a prioritized list of checks with `result=fail` or `triage_tags` set, suitable for immediate developer attention.

2. **TRIAGE-02**: Integrates this tool as the final step of `run-formal-verify.cjs` — after all 26 steps complete, `generate-triage-bundle.cjs` runs and its output is included in the run summary.

The implementation depends on v0.20-01 (Schema Enrichment), which enriched `check-results.ndjson` with v2.1 fields (`check_id`, `surface`, `property`, `runtime_ms`, `summary`, `triage_tags`). These fields enable structured triage analysis without stdout parsing.

When no previous run exists (first execution), `diff-report.md` notes "first run" rather than failing or producing empty output. The suspects list is always produced from the current run, regardless of history.

**Primary recommendation:** Task 1 implements `generate-triage-bundle.cjs` with NDJSON read/parse, per-check delta logic, and markdown generation. Task 2 creates the diff snapshot persistence mechanism. Task 3 integrates into `run-formal-verify.cjs` and adds it as the final step.

## Architecture Patterns

### Pattern 1: NDJSON Parsing and Fail-Open Behavior

**What:** Read `formal/check-results.ndjson`, parse line-by-line, skip malformed lines without throwing.

**When to use:** At module initialization in `generate-triage-bundle.cjs` main entry point.

**Established precedent (verify-formal-results.cjs, lines 16-36):**
```javascript
// Source: /Users/jonathanborduas/code/QGSD/bin/verify-formal-results.cjs
function parseNDJSON(ndjsonPath) {
  let content;
  try {
    content = fs.readFileSync(ndjsonPath, 'utf8');
  } catch (e) {
    // File missing or unreadable — fail-open, return empty
    return [];
  }
  const lines = content.trim().split('\n');
  const results = [];
  for (const line of lines) {
    if (!line.trim()) continue; // skip empty lines
    try {
      results.push(JSON.parse(line));
    } catch (e) {
      // Skip malformed NDJSON lines — warn but do not throw
      process.stderr.write('[verify-formal-results] warning: skipping malformed NDJSON line: ' + e.message + '\n');
    }
  }
  return results;
}
```

**Key insight:** If NDJSON file doesn't exist or is empty, parsing returns `[]`, not error. Missing file on first run is expected and normal.

**Reuse:** Call `verify-formal-results.parseNDJSON()` directly rather than reimplementing.

### Pattern 2: Delta Detection — Current vs Previous Run

**What:** Compare current run's check results against a snapshot of the previous run, identifying state changes.

**When to use:** After parsing current NDJSON, before generating diff-report.md.

**Structure:**
1. Read and parse current `formal/check-results.ndjson` → `currentRun: { check_id → result }`
2. Try to read previous snapshot (e.g., `formal/.check-results.prev.ndjson` or persisted JSON) → `previousRun: { check_id → result }`
3. Compute deltas:
   - **New checks:** Present in current but not in previous
   - **Removed checks:** Present in previous but not in current
   - **Transitioned checks:** Present in both but result differs (e.g., `fail` → `pass`)
   - **Unchanged checks:** Present in both with same result

**Snapshot persistence strategy (three options):**
- **Option A (Recommended):** Append a `### Previous Run` section to `diff-report.md` containing a JSON block with previous run's check_id→result map. On next run, extract that JSON to build previousRun. Always update with current run's data.
- **Option B:** Separate file `formal/.check-results.prev.ndjson` (gitignored). Write after generating diff-report.
- **Option C:** Git-based: compare against last committed `formal/check-results.ndjson` (allows tracking history in git).

**Recommendation:** Use Option A (embedded JSON block in diff-report.md) for self-contained output and no additional file management. Simpler for developers to reason about. Update this block every run.

### Pattern 3: Suspects List — Prioritizing Failed and Tagged Checks

**What:** Filter checks to identify those needing immediate developer attention (failures and anomalies).

**When to use:** After parsing current NDJSON, independently of delta detection.

**Suspect criteria:**
1. **Critical:** `result === 'fail'`
2. **Important:** `result === 'warn'` AND `triage_tags.length > 0`
3. **Informational:** `result === 'inconclusive'` AND `triage_tags` includes `'needs-fairness'` or other specific tags

**Sort order (high to low priority):**
1. `fail` (critical)
2. `warn` with tags
3. `inconclusive` with specific tags

**For each suspect, include:**
- `check_id`: Unique identifier (e.g., `tla:quorum-safety`)
- `result`: Current status
- `summary`: One-line outcome from NDJSON
- `property`: Check description (what was tested)
- `triage_tags`: Array of anomaly markers (e.g., `['needs-fairness', 'liveness-suspect']`)
- `runtime_ms`: Execution time (useful for timing anomalies)

**Example structure (in markdown):**
```markdown
## Suspects

### Critical Failures
| Check | Result | Property | Summary |
|-------|--------|----------|---------|
| tla:quorum-safety | fail | MCsafety — no deadlock | fail: invariant violation at depth 42 |

### Warnings with Tags
| Check | Tags | Summary |
|-------|------|---------|
| ci:liveness-fairness-lint | needs-fairness | inconclusive: fairness declarations missing... |
```

### Pattern 4: First-Run Handling

**What:** When no previous run snapshot exists, produce diff-report.md with a "first run" marker instead of failing.

**When to use:** At diff computation step, detect empty previousRun.

**Pattern:**
```javascript
if (Object.keys(previousRun).length === 0) {
  // First run: diff-report.md should note this but not fail
  diffReportContent = `# Formal Verification Diff Report

**Status:** First run — no previous snapshot to compare against.

[Current run data follows...]
`;
}
```

**Key insight:** The tool always produces suspects.md (from current run) even on first run. Diff-report gracefully handles missing history.

### Pattern 5: Markdown Output Structure

**What:** Generate human-readable markdown from NDJSON data, organized by formalism and result.

**When to use:** After delta and suspect computation, write to disk.

**diff-report.md structure:**
```markdown
# Formal Verification Diff Report

**Generated:** [ISO timestamp]
**Current Run:** [summary — X pass, Y fail, Z warn]
**Previous Run:** [summary from snapshot — or "first run"]
**Overall Status:** [pass/fail/mixed]

## Deltas by Formalism

### TLA+
#### Transitioned Checks
- check_id: previous_result → current_result

#### New Checks
- check_id: current_result (first seen)

#### Removed Checks
- check_id: was previous_result (no longer run)

## Unchanged Passing Checks
[Summary line count only — not verbose list]

## Previous Run (for next comparison)
\`\`\`json
{ "check_id": "result", ... }
\`\`\`
```

**suspects.md structure:**
```markdown
# Formal Verification Suspects

**Generated:** [ISO timestamp]
**Total Suspects:** N

## Critical Failures (result=fail)

### check_id
- **Property:** [what was tested]
- **Summary:** [outcome message]
- **Runtime:** [runtime_ms]ms
- **Tags:** [triage_tags array]

## Warnings (result=warn, tags present)
[same format as Critical]

## Inconclusive (result=inconclusive, specific tags)
[same format as Critical]
```

### Anti-Patterns to Avoid

- **Hardcoded formalism list in diff table:** Use `Object.keys()` on parsed results grouped by formalism. Let the data determine the table structure.
- **Throwing on malformed NDJSON lines:** Skip and warn (fail-open pattern). One bad line should not break the entire triage.
- **Storing delta state in git:** Diff snapshots should be computed on-the-fly from current + previous run data. Don't commit `.prev` files.
- **Verbose unchanged checks:** Don't list all 20 passing TLA+ checks in diff-report. Instead: "18 unchanged passing TLA+ checks".
- **Merging check_id with tool field:** Use `check_id` from v2.1 schema (e.g., `tla:quorum-safety`) not `tool` (e.g., `run-tlc`). They differ by design.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| NDJSON parsing | Custom split + map + parse | `verify-formal-results.parseNDJSON()` | Already handles malformed lines, fail-open, and correct line splitting |
| Grouping checks by formalism | Manual reduce/filter loops | `verify-formal-results.groupByFormalism()` | Tested, handles all result types, no hardcoded formalism list |
| Markdown table generation | Manual string concatenation | Template literals or simple array.map | Tables are error-prone; use consistent borders and cell alignment patterns |
| Delta computation | Custom object diffing | Explicit { new, removed, transitioned, unchanged } sets | Clear logic, easy to test, avoids bugs in property comparison |

**Key insight:** The NDJSON parsing and grouping logic is already proven in `verify-formal-results.cjs`. Export those functions (if not already exported) and reuse them to avoid duplication and bugs.

## Common Pitfalls

### Pitfall 1: Treating Previous Snapshot as Required
**What goes wrong:** Code throws if previous snapshot doesn't exist, blocking first-run execution.
**Why it happens:** Defensive coding without considering "first run" as a valid state.
**How to avoid:** Check if previousRun is empty and handle gracefully. Return "first run" note in diff-report, not error.
**Warning signs:** Error on fresh clone with no prior verify run; failing CI on initial setup.

### Pitfall 2: Confusing `formalism` vs `surface`
**What goes wrong:** Diff-report groups by `surface` (e.g., `'ci'`, `'tla'`) but queries use `formalism` (e.g., `'trace'`, `'tla'`). Results split across tables incorrectly.
**Why it happens:** v0.20-01 added both fields; they're semantically similar but structurally different.
**How to avoid:** Use `surface` (the STEPS tool group) for delta diff table headers. Use `formalism` for filtering internal validation logic. Document which field is used where.
**Warning signs:** Suspects.md shows check under wrong formalism section; diff-report has empty sections.

### Pitfall 3: Snapshot Persistence Strategy Silent Failure
**What goes wrong:** Diff-report.md is generated, but the snapshot isn't saved (or save fails), so next run has no previous data and reports "first run" again.
**Why it happens:** Snapshot write is in try-catch without logging; fails silently.
**How to avoid:** Always log snapshot persistence success/failure. Use fail-open: if snapshot write fails, continue (diff-report still produced, just with "first run" on next cycle). Consider embedding snapshot in diff-report.md itself (Option A) so read and write are atomic.
**Warning signs:** diff-report.md always says "first run"; delta counts never populate; no visible indication that snapshots are persisted.

### Pitfall 4: Incorrect Delta Logic for Unchanged Passing Checks
**What goes wrong:** Diff-report floods with a full list of 20 passing checks under "Unchanged Passing," making the report useless for finding real changes.
**Why it happens:** Not filtering or summarizing unchanged passing checks.
**How to avoid:** In diff-report, only list checks that changed result or are new/removed. Unchanged passing checks appear as a single count line: "18 unchanged passing TLA+ checks, no action needed."
**Warning signs:** Diff-report markdown file is >100 lines every run; scrolling past "unchanged passing" to reach important sections.

### Pitfall 5: Suspects.md Missing Early-Exit Cases
**What goes wrong:** Suspects.md is generated from current run, but if current NDJSON has only 2 checks (partial run with `--only=tla`), the report looks like most checks passed when they were actually not run.
**Why it happens:** Not distinguishing between "pass" and "not run".
**How to avoid:** Document in suspects.md header that list reflects the specific run (e.g., "Generated from `run-formal-verify --only=tla`"). For filtered runs, report only suspects from that subset. Never infer absent results as passing.
**Warning signs:** Suspects.md says "0 suspects found" after a partial run, giving false confidence.

## Open Questions

1. **Should generate-triage-bundle be exported as a reusable module or only CLI?**
   - What we know: Other tools like verify-formal-results.cjs export functions for reuse. generate-triage-bundle may be called from run-formal-verify.cjs or standalone.
   - What's unclear: Is the intention to import and call functions, or spawn as a subprocess?
   - Recommendation: Export `{ generateDiffReport, generateSuspectsList, saveDiffReport }` functions. Main entry point can call them. Allows both CLI and programmatic use.

2. **What is the exact snapshot persistence mechanism?**
   - What we know: TRIAGE-01 requires "per-check delta from last run". Option A (embed in diff-report.md) is simplest. Option B (separate .prev file) is cleanest for large datasets.
   - What's unclear: Should snapshot be updated even if diff-report.md write fails? Should it be gitignored?
   - Recommendation: Use Option A (embedded JSON in diff-report.md) for simplicity and atomicity. Update during diff-report write (same try-catch). Mark as internal implementation detail in comments.

3. **Should run-formal-verify.cjs call generate-triage-bundle as a STEPS entry or as a post-processing step?**
   - What we know: run-formal-verify has 26 STEPS entries executed in phases. TRIAGE-02 says "called as final step after all checks complete."
   - What's unclear: Should generate-triage-bundle be a 27th STEPS entry (appearing in summary table) or a post-execution cleanup (lines 376-377)?
   - Recommendation: Add as a 27th STEPS entry (`{ tool: 'ci', id: 'ci:triage-bundle', ... }`). This makes it visible in the run summary, allows conditional execution with `--only=ci`, and follows the established STEPS pattern. Mark it as non-critical (continue on failure).

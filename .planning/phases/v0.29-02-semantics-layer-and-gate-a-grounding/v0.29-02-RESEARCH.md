# Phase v0.29-02: Semantics Layer and Gate A Grounding - Research

**Researched:** 2026-03-06
**Domain:** Formal verification semantics layer (L2) + Gate A grounding metric
**Confidence:** HIGH

## Summary

Phase v0.29-02 builds four new CLI scripts and one gate script that produce Layer 2 (Semantics) artifacts grounded in the Layer 1 evidence infrastructure completed in v0.29-01. The four L2 scripts are: (1) an invariant catalog that aggregates declared invariants from TLA+/Alloy specs with observed invariants mined from traces, (2) a mismatch register that records L2-model vs L1-trace disagreements as JSONL, (3) an assumption register that catalogs assumptions from the existing 567-entry assumption-gaps.md with validation status, and (4) an operational state machine derivation that produces an observed-behavior FSM from traces. Gate A then measures how well L2 models explain L1 traces.

The critical technical insight is that the existing `validate-traces.cjs` already computes the core metric: 82.2% of 35,825 traces are "explained" by the XState machine (valid state transitions). All 6,369 divergences come from a single unmappable action (`quorum_fallback_t1_required`). Gate A needs to wrap this existing capability with classification of unexplained traces and a formal grounding_score output. The prior regression (69% -> 0% -> 17.9%) noted in STATE.md was likely caused by breaking the event mapping; the current 82.2% baseline already exceeds the 80% target, meaning Gate A's design should PRESERVE this score, not reinvent it.

**Primary recommendation:** Build Gate A FIRST (it reuses validate-traces.cjs output), then build the invariant catalog, mismatch register, and assumption register as independent scripts. The operational FSM derivation (SEM-04) is the most novel piece and should come last.

### Pattern 2: Invariant Extraction from TLA+
**What:** Parse TLA+ files with regex to extract INVARIANT declarations, TypeOK definitions, and PROPERTY lines. Do NOT use a full TLA+ parser.
**When to use:** SEM-01 invariant catalog -- declared invariants.
**Extraction rules:**
```
From TLA+ (.tla files):
  - Lines matching /^(\w+)\s*==\s*/ where name is referenced in .cfg INVARIANT line -> declared invariant
  - Lines matching /INVARIANT\s+(\w+)/ in .cfg files -> invariant name
  - Lines matching /PROPERTY\s+(\w+)/ in .cfg files -> liveness property (also an invariant)
  - TypeOK definitions -> type invariant

From Alloy (.als files):
  - `assert Name { ... }` blocks -> declared assertion (invariant)
  - `fact Name { ... }` blocks -> background constraint
  - `check Name` commands -> checked assertion

From spec/*/invariants.md:
  - Property definitions with **Property:** markers -> declared invariant with natural language description
```

### Pattern 3: Gate A Grounding Score Computation
**What:** Gate A reuses `validate-traces.cjs` output (deviation score) and classifies unexplained traces.
**When to use:** GATE-01 implementation.
**Critical design:**
```
grounding_score = traces_explained / total_traces
                = valid_traces / total_events
                = 29456 / 35825  (current: 82.2%)

Classification of unexplained traces (6,369 total):
  1. instrumentation_bug: unmappable_action where action exists in hooks but not in event vocabulary
     -> quorum_fallback_t1_required (5,649 events) -- hook emits action not in vocabulary
  2. model_gap: mappable action that fails XState replay (transition not modeled)
     -> 0 currently (all mapped actions replay successfully)
  3. genuine_violation: action maps to XState event but violates an invariant
     -> 0 currently (no invariant violations in valid traces)

KEY INSIGHT: Adding quorum_fallback_t1_required to event vocabulary AND to
mapToXStateEvent() would change the score. Gate A should report the score
AS-IS and classify reasons, not "fix" the score by changing mappings.
Gate A MEASURES alignment; it does not ALTER it.
```

### Pattern 6: Observed FSM from Traces (SEM-04)
**What:** Derive a state machine from observed trace transitions. Count transition frequencies to build an empirical FSM.
**When to use:** SEM-04 implementation.
**Algorithm:**
```
1. Read conformance events via planning-paths.cjs (raw JSONL, not trace-corpus-stats summary)
2. Use session boundaries from trace-corpus-stats.json to group events
3. For each session, extract (action_i, action_i+1) transition pairs
4. Build adjacency matrix: { from_action -> { to_action -> count } }
5. States = unique actions; Transitions = observed (from, to) pairs
6. Initial state = most common first action per session
7. Terminal states = actions that are last in sessions
8. Output as JSON with states, transitions, frequencies
9. Compare structure to hand-written XState machine (shared states, missing states, extra states)
```
**Important:** trace-corpus-stats.json only has action COUNTS per session, not sequences. The observed FSM script must read raw conformance-events.jsonl directly and filter events by session time boundaries.

### Anti-Patterns to Avoid
- **Modifying validate-traces.cjs to compute Gate A score:** Gate A should be a SEPARATE script that CONSUMES validate-traces output. Do not merge Gate A logic into the existing validation script. The validator is L1 infrastructure; Gate A is an L2->L1 gate.
- **"Fixing" the grounding score by adding mappings:** Gate A measures alignment, it does not alter it. If `quorum_fallback_t1_required` is unmapped, Gate A reports that as an instrumentation_bug classification. Fixing the mapping is a separate action tracked in the mismatch register.
- **Parsing TLA+ with a full parser:** Regex extraction is sufficient. The TLA+ files have consistent formatting (generated by `bin/generate-formal-specs.cjs`). A full parser would require Java (TLA+ SANY) and add complexity.
- **Storing mismatches in a database:** JSONL is the right format. The mismatch register has ~6,369 entries grouped into far fewer unique mismatches. This is trivially small for file-based storage.
- **Re-parsing assumption-gaps.md on every run:** Parse once into structured JSON (assumption-register.json). Subsequent runs UPDATE the register, not re-derive it.

## Common Pitfalls

### Pitfall 1: Gate A Score Regression
**What goes wrong:** Changing event mappings or vocabulary causes the grounding score to drop from 82.2% below the 80% target. Prior attempt regressed from 69% to 0% to 17.9% (noted in STATE.md blockers).
**Why it happens:** Gate A's score depends on `mapToXStateEvent()` in validate-traces.cjs. Any change to this function changes the denominator or numerator.
**How to avoid:** Gate A READS the validate-traces output but NEVER modifies mapToXStateEvent(). The grounding_score is computed from the existing deviation data. If the score needs improvement, that is tracked as a mismatch register entry, not a Gate A fix.
**Warning signs:** grounding_score changes when no L1 data changed. Always assert score stability in tests.

### Pitfall 2: Invariant Extraction False Positives
**What goes wrong:** Regex parsing of TLA+ files picks up variable definitions that look like invariants but are not (e.g., `TypeOK == ...` might appear in comments or as a helper).
**Why it happens:** TLA+ does not have a single "invariant" keyword; invariants are regular definitions referenced in .cfg files.
**How to avoid:** Cross-reference extracted definitions against .cfg INVARIANT/PROPERTY lines. Only definitions that appear in a .cfg file as INVARIANT or PROPERTY are declared invariants. Other definitions are helper predicates.
**Warning signs:** Invariant count much higher than expected (should be roughly 2-5 per TLA+ model, not dozens).

### Pitfall 3: Assumption Deduplication
**What goes wrong:** assumption-gaps.md has 567 entries but many share names (e.g., `TypeOK` appears 15+ times across different models). Naive dedup loses valid distinct assumptions.
**Why it happens:** Each TLA+ model has its own TypeOK invariant. The assumption-gaps.md lists them all individually.
**How to avoid:** Use (source_model, name, type) as the unique key. TypeOK from QGSDQuorum.tla and TypeOK from QGSDBreakerState.tla are DIFFERENT assumptions despite having the same name. Do NOT deduplicate across models.
**Warning signs:** Assumption count significantly different from 567 without clear explanation.

### Pitfall 4: Observed FSM Noise
**What goes wrong:** The trace-derived FSM has too many spurious transitions because all action pairs are treated as transitions, even when they span session boundaries or unrelated events.
**Why it happens:** Without round_id grouping, consecutive events may not represent causal transitions.
**How to avoid:** Only extract transitions within the same session (use trace-corpus-stats.json session boundaries). Additionally, filter by minimum frequency threshold (e.g., transitions seen < 3 times are likely noise). The existing validate-traces.cjs round_id grouping provides better granularity than raw event sequences.
**Warning signs:** FSM has 50+ transitions when XState machine has ~15.

### Pitfall 5: Mismatch Register Size
**What goes wrong:** 6,369 divergences all become individual mismatch entries, creating a register too large to be useful.
**Why it happens:** All divergences are the same type (unmappable `quorum_fallback_t1_required`). Each trace event generates one divergence.
**How to avoid:** Group mismatches by (model, claim, disagreement_type, action). The 6,369 individual divergences become ONE mismatch entry with `instance_count: 6369`. The register tracks unique disagreement TYPES, not individual events.
**Warning signs:** Mismatch register has thousands of entries instead of dozens.

### Pitfall 6: Gate A "explains" Definition Ambiguity
**What goes wrong:** "traces_explained" could mean different things: (a) trace maps to a valid XState transition, (b) trace action is in the vocabulary, (c) trace action exists in the invariant catalog. Different definitions give different scores.
**Why it happens:** STATE.md specifically calls out that "'explains' must be formally specified."
**How to avoid:** Use the validate-traces.cjs definition: a trace is "explained" if `mapToXStateEvent(event)` returns non-null AND the XState replay accepts the transition. This is the existing 82.2% baseline. Document this definition explicitly in gate-a-result.json as `explanation_criteria`.
**Warning signs:** Score differs from validate-traces deviation score.

## Code Examples

### Extracting Invariants from TLA+ Files
```javascript
// Source: derived from analysis of .planning/formal/tla/*.tla structure
const fs = require('fs');
const path = require('path');

function extractTLAInvariants(tlaDir) {
  const invariants = [];
  const cfgFiles = fs.readdirSync(tlaDir).filter(f => f.endsWith('.cfg'));

  // Build map of .cfg -> { invariants: [], properties: [] }
  const cfgMap = {};
  for (const cfg of cfgFiles) {
    const content = fs.readFileSync(path.join(tlaDir, cfg), 'utf8');
    cfgMap[cfg] = { invariants: [], properties: [] };
    let match;
    const invRegex = /INVARIANT\s+(\w+)/g;
    while ((match = invRegex.exec(content)) !== null) cfgMap[cfg].invariants.push(match[1]);
    const propRegex = /PROPERTY\s+(\w+)/g;
    while ((match = propRegex.exec(content)) !== null) cfgMap[cfg].properties.push(match[1]);
  }

  // For each TLA+ file, find definitions referenced in .cfg files
  const tlaFiles = fs.readdirSync(tlaDir).filter(f => f.endsWith('.tla') && !f.startsWith('MC'));
  for (const file of tlaFiles) {
    const content = fs.readFileSync(path.join(tlaDir, file), 'utf8');
    const defRegex = /^(\w+)\s*==/gm;
    const defs = new Set();
    let match;
    while ((match = defRegex.exec(content)) !== null) defs.add(match[1]);

    for (const [cfg, refs] of Object.entries(cfgMap)) {
      for (const inv of refs.invariants) {
        if (defs.has(inv)) {
          invariants.push({
            name: inv, source_file: file, config_file: cfg,
            type: 'declared_invariant', source_type: 'tla', layer: 'L2'
          });
        }
      }
      for (const prop of refs.properties) {
        if (defs.has(prop)) {
          invariants.push({
            name: prop, source_file: file, config_file: cfg,
            type: 'declared_property', source_type: 'tla', layer: 'L2'
          });
        }
      }
    }
  }
  return invariants;
}
```

function extractAlloyInvariants(alloyDir) {
  const invariants = [];
  const files = fs.readdirSync(alloyDir).filter(f => f.endsWith('.als'));

  for (const file of files) {
    const content = fs.readFileSync(path.join(alloyDir, file), 'utf8');
    let match;

    // Extract assert blocks
    const assertRegex = /assert\s+(\w+)\s*\{/g;
    while ((match = assertRegex.exec(content)) !== null) {
      invariants.push({
        name: match[1], source_file: file,
        type: 'declared_assertion', source_type: 'alloy', layer: 'L2'
      });
    }

    // Extract fact blocks
    const factRegex = /fact\s+(\w+)\s*\{/g;
    while ((match = factRegex.exec(content)) !== null) {
      invariants.push({
        name: match[1], source_file: file,
        type: 'background_constraint', source_type: 'alloy', layer: 'L2'
      });
    }
  }
  return invariants;
}

### Gate A Score Computation (using require, not subprocess)
```javascript
// Source: derived from validate-traces.cjs exports
const validateTraces = require('./validate-traces.cjs');
const fs = require('fs');
const path = require('path');

function computeGateAScore(projectRoot) {
  const pp = require('./planning-paths.cjs');
  const logPath = pp.resolve(projectRoot, 'conformance-events');
  const lines = fs.readFileSync(logPath, 'utf8').trim().split('\n');
  const events = lines.map(l => { try { return JSON.parse(l); } catch { return null; } }).filter(Boolean);

  let explained = 0;
  const classifications = { instrumentation_bug: 0, model_gap: 0, genuine_violation: 0 };

  for (const event of events) {
    const xstateEvent = validateTraces.mapToXStateEvent(event);
    if (xstateEvent !== null) {
      explained++;
    } else {
      classifications.instrumentation_bug++;
    }
  }

  const groundingScore = explained / events.length;
  return {
    schema_version: '1',
    generated: new Date().toISOString(),
    grounding_score: groundingScore,
    grounding_score_pct: (groundingScore * 100).toFixed(1) + '%',
    traces_explained: explained,
    total_traces: events.length,
    target: 0.80,
    target_met: groundingScore >= 0.80,
    explanation_criteria: 'mapToXStateEvent returns non-null AND XState replay accepts transition',
    unexplained_classifications: classifications
  };
}
```

// Source: derived from .planning/formal/assumption-gaps.md format
function parseAssumptionGaps(gapsPath) {
  const content = fs.readFileSync(gapsPath, 'utf8');
  const lines = content.split('\n');
  const assumptions = [];

  for (const line of lines) {
    const match = line.match(/^\|\s*(\d+)\s*\|\s*(\w+)\s*\|\s*(.+?)\s*\|\s*(\w+)\s*\|\s*(\w+)\s*\|\s*`?(.+?)`?\s*\|\s*(\w+)\s*\|/);
    if (match) {
      assumptions.push({
        id: `assumption-${match[1].padStart(3, '0')}`,
        source: match[2],
        name: match[3].trim(),
        type: match[4],
        original_coverage: match[5],
        proposed_metric: match[6],
        metric_type: match[7],
        validation_status: 'uncovered',
        linked_l2_states: []
      });
    }
  }
  return assumptions;
}

## Open Questions

1. **Should Gate A call validate-traces.cjs as a subprocess or import its functions?**
   - What we know: validate-traces.cjs exports `mapToXStateEvent`, `validateMCPMetadata`, etc. via `module.exports`. It also has a `main()` that writes to stdout.
   - What's unclear: Whether to `require()` the module and call exported functions, or run the script as a subprocess and parse stdout.
   - Recommendation: `require()` and call exported functions. This avoids stdout parsing fragility and gives direct access to the divergence data structure. The subprocess approach is brittle (parsing "[validate-traces] Deviation score: X%" from stdout).

2. **How to link assumptions to L2 states?**
   - What we know: assumption-gaps.md lists assumptions by TLA+ model name (e.g., "TypeOK from QGSDQuorum.tla"). The XState machine has named states.

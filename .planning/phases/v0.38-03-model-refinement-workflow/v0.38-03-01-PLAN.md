Create `bin/refinement-loop.cjs` — a testable helper that normalizes bug context input and runs the inverted verification loop for model refinement. This script both exports functions for unit testing and runs as a CLI tool when invoked directly.

**CLI interface:**
```
node bin/refinement-loop.cjs --model path --bug-context "text-or-file" --formalism tla|alloy [--max-attempts 3] [--verbose] [--format json|text]
```

**CLI JSON output schema** (contract for close-formal-gaps.md Step 6):
```json
{
  "status": "reproduced|not_reproduced",
  "attempts": 1,
  "model_path": ".planning/formal/tla/ModelName.tla",
  "counterexample": "trace text or null",
  "iterations": [
    { "attempt": 1, "passed": true, "summary": "model invariants hold, bug not captured" },
    { "attempt": 2, "passed": false, "summary": "invariant X violated at state 7" }
  ]
}
```

**Exported functions:**

1. `normalizeBugContext(value)` — Normalize --bug-context input:
   - If value is null/undefined/empty string, return ''
   - If `fs.existsSync(value.trim())` is true, read file contents via `fs.readFileSync(value.trim(), 'utf-8').trim()`
   - Otherwise return `value.trim()` as inline text
   - Never throw — return empty string on any error

2. `verifyBugReproduction(modelPath, bugContext, options)` — Main refinement loop:
   - `options.formalism`: 'tla' or 'alloy' (determines which checker to call)
   - `options.maxAttempts`: max iterations (default 3)
   - `options.verbose`: show full checker output (default false)
   - `options.onIteration`: callback for progress display (receives `{ attempt, passed, summary }`)

   **Loop logic (INVERTED VERIFICATION — this is the critical semantics):**
   For each attempt from 1 to maxAttempts:
   - Run the appropriate checker (TLA+ via `node bin/run-tlc.cjs modelPath`, Alloy via `node bin/run-alloy.cjs modelPath`) using `execFileSync` from child_process
   - If checker finds a violation (non-zero exit code): the model REPRODUCES the bug — return `{ status: 'reproduced', attempts, model_path, counterexample, iterations }`
   - If checker passes (zero exit code): the model does NOT capture the failure
     - Push iteration info to iterations array
     - Call onIteration callback if provided
     - If attempts remain, continue; otherwise return `{ status: 'not_reproduced', attempts: maxAttempts, ... }`

   **Timeout:** 60s per checker invocation (match existing defaults in run-tlc.cjs/run-alloy.cjs)
   **Fail-open:** If checker subprocess fails to spawn or crashes, treat as "not reproduced" for that attempt and continue to next iteration.

3. `formatIterationFeedback(iteration, verbose)` — Format progress display:
   - Default (verbose=false): `Attempt {N}: {reproduced|still passes} — {short summary}`
   - Verbose: includes full checker output and pointer to output file

**Module exports:**
```javascript
module.exports = { normalizeBugContext, verifyBugReproduction, formatIterationFeedback };
```

**Dependencies (all existing, no new installs):**
- `require('fs')`, `require('path')`, `require('child_process')` — built-ins only
- Calls bin/run-tlc.cjs and bin/run-alloy.cjs as subprocesses via execFileSync (no shell injection risk)

**Workflow modifications (core/workflows/close-formal-gaps.md):**

1. **Step 1 (detect_gaps) — Add bug-context parsing:**

   After the existing flag parsing (--batch, --category, --ids, --all), add:

   If `--bug-context` is provided:
   - Normalize the value using normalizeBugContext from bin/refinement-loop.cjs:
     `BUG_CONTEXT=$(node -e "const r=require('./bin/refinement-loop.cjs');console.log(r.normalizeBugContext('$RAW_VALUE'))")`
   - Store normalized text as $BUG_CONTEXT
   - Log: "Bug context loaded: {first 80 chars}..."

   If `--verbose` is provided, set $VERBOSE=true

2. **Step 5 (generate_model) — Inject bug context into spec generation:**

   In the spec generation instructions for each formalism (TLA+, Alloy, PRISM), add a conditional block AFTER the existing requirement text and module context, but BEFORE the formalism-specific instructions:

   If $BUG_CONTEXT is non-empty, append to the spec generation prompt:

   ```
   <bug_context>
   This model must capture the following failure mode:
   {$BUG_CONTEXT}

   Generate invariants/assertions that would be VIOLATED when this bug is present.
   The model should FAIL (find a counterexample) when the bug exists in the system.
   After the bug is fixed, the model should PASS (no violations).
   </bug_context>
   ```

   This biases the generated spec toward capturing the failure mode rather than just verifying requirements.

3. **Step 6 (run_checker) — Add refinement loop when bug-context is active:**

   After the existing checker execution logic, add a conditional refinement section:

   If $BUG_CONTEXT is non-empty (--bug-context was provided):

   ### Refinement Verification (MRF-02)

   Instead of treating "no errors" as success, use inverted verification:
   - Model checker finds error (violation) = model REPRODUCES the bug = SUCCESS
   - Model checker passes (no errors) = model does NOT capture the failure = RETRY

   Run: `node bin/refinement-loop.cjs --model "$MODEL_PATH" --bug-context "$BUG_CONTEXT" --formalism "$FORMALISM" --format json`

   Parse the JSON output:
   - If status == "reproduced": Log success, continue to Step 7 (registry update)
     Display: "Attempt {N}: reproduced -- {summary}"
   - If status == "not_reproduced" AND loop has retries remaining:
     Display: "Attempt {N}: still passes -- {summary}, retry {N} of 2"
     Return to Step 5 to regenerate the model with refinement context:
       Append to the generation prompt: "Previous model passed all checks -- it did not capture the failure. Checker output: {iteration summary}. Refine the invariants to be more specific to the described failure mode."
     Then re-run Step 6 (up to 2 additional retries, 3 total attempts)
   - If status == "not_reproduced" AND all attempts exhausted:
     Display: "Model refinement exhausted (3 attempts). Model does not reproduce the bug."
     Continue to Step 7 with a note that model is unverified for bug reproduction.

   If $VERBOSE is true, show full checker output for each iteration.
   Otherwise show summary only (per user decision: summary verdicts by default).
   Include pointer to full checker details in summary.

**Important constraints on modifications:**

- When --bug-context is NOT provided, behavior is identical to today (no regression)
- The refinement loop retry goes back to Step 5 (regeneration) then Step 6 (checking) — this is a small internal loop within the workflow, not a full workflow restart

# Phase v0.40-02: Workflow Modifications - Research

**Researched:** 2026-03-19
**Domain:** Workflow-level structural enforcement (scope contracts, approach declaration, quorum dispatch)
**Confidence:** HIGH

## Summary

Phase v0.40-02 adds two workflow-level enforcement mechanisms that eliminate planning surprises before execution begins: (1) **Intent Declaration** — quick tasks must declare their approach (what will be done, what is explicitly out of scope) before the planner spawns, storing this in a scope contract that subsequent guards can validate; (2) **Root Cause Enforcement** — the solve-diagnose workflow adds a quorum vote on root cause diagnosis after hypothesis synthesis, ensuring multi-model consensus on the causal chain before proceeding to remediation. Both mechanisms operate at the workflow level rather than as post-hoc checks, making them enforcement gates rather than advisory warnings.

The research reveals that both features follow established nForma patterns: scope contracts leverage existing `.claude/` directory conventions used for session flags and feature toggles; approach declaration reuses the YAML frontmatter parsing already present in quick.md; quorum dispatch for ROOT-02 follows the canonical R3 protocol established in v0.37 quorum work.

**Primary recommendation:** Implement INTENT-01, INTENT-02, INTENT-03 in quick.md as a pre-planner gate that derives approach from the task description via Haiku and persists to `.claude/scope-contract.json`. Implement ROOT-02 in solve-diagnose.md as a quorum-vote step after Step 0f (hypothesis synthesis) before wave dispatch to remediation.

## Architecture Patterns

### Pattern 1: Scope Contract Format & Lifecycle

**What:** A JSON file at `.claude/scope-contract.json` maps git branch names to declared task scope (approach, out-of-scope items). Created by quick.md during planning, read by the scope guard during execution.

**When to use:** Every quick task that needs scope enforcement. Scope contracts are non-blocking in v0.40-02 (contract created but guard is v0.40-03); they are informational for planning gates.

**Example:**

```json
{
  "quick-336-add-requirement-mappings": {
    "task_id": 336,
    "task_description": "Add requirement mappings for 4 models",
    "approach": "Use semantic matching to map 4 formal models (TLA+, Alloy, PRISM, SMT) to existing requirements in requirements.json. Existing requirements unchanged.",
    "out_of_scope": [
      "Model creation or fixes",
      "Changes to existing requirements ID/text",
      "Formal verification runs",
      "UI/documentation changes"
    ],
    "branches_affected": ["quick-336-add-requirement-mappings"],
    "created_at": "2026-03-19T20:00:00Z",
    "planner_model": "gpt-4.1",
    "created_by": "quick-orchestrator"
  }
}
```

**Key insight:** Contract must be small and scannable — planner's approach block becomes a single JSON entry. Use dot-separated keys for nested structure to keep parsing simple.

### Pattern 2: Haiku-Driven Approach Derivation

**What:** When quick tasks are invoked, spawn a Haiku subagent to analyze the task description and derive a structured APPROACH block (what WILL be done, what explicitly will NOT). Store this in the scope contract before planner spawn.

**When to use:** Every quick task initialization. The derivation is automatic (non-modal) — no user confirmation step.

**Example (orchestrator call):**

```bash
DESCRIPTION="Add requirement mappings for 4 models"

Task(
  subagent_type="general-purpose",
  model="haiku",
  description="Derive approach for quick task",
  prompt="
You are deriving a task scope contract from a quick task description.

## Task Description
${DESCRIPTION}

## Your Task
Analyze the description and produce a structured approach in this JSON format:
{
  \"approach\": \"[One sentence: what will be done]\",
  \"out_of_scope\": [\"[item 1]\", \"[item 2]\", ...]
}

Guidelines:
- approach: imperative, specific, one sentence
- out_of_scope: literal items that the task explicitly WILL NOT touch
- Be conservative — when uncertain, mark something as out-of-scope

Respond with ONLY the JSON object, no markdown fencing or extra text.
"
)
```

Parse Haiku response as JSON. If parsing fails, fall back to a generic approach:

```json
{
  "approach": "Complete the task as described",
  "out_of_scope": []
}
```

**Key insight:** Haiku is fast and cheap for parsing intent. Fail-open: if approach derivation errors, quick tasks proceed with an empty contract rather than blocking.

### Pattern 3: Quorum Vote on Root Cause Diagnosis

**What:** After hypothesis synthesis in solve-diagnose (Step 0f), add a new step (0g) that dispatches a quorum vote on the root cause diagnosis. Each model votes APPROVE (diagnosis is sound) or BLOCK (diagnosis is incomplete/wrong). Only on consensus can remediation proceed.

**When to use:** In solve-diagnose.md, only for complex diagnoses. Simple single-hypothesis cases may skip (per QRUM-01 v2 requirement — complexity gate to be added post-v0.40).

**Flow:**

```
Step 0f (current): Hypothesis synthesis → hypotheses_list
Step 0g (new):    Quorum vote on diagnosis
                  - Question: "Is the root cause diagnosis sound and complete?"
                  - Artifacts: hypotheses_list + prior diagnostic state
                  - Dispatch to nf-quorum-slot-worker x N
                  - Consensus required before remediation
Step 1:           Remediation dispatch (only if quorum approved)
```

**Example prompt block (YAML format per quorum.md):**

```yaml
slot: <slotName>
round: 1
timeout_ms: 30000
repo_dir: <project_root>
mode: A
question: "Review the hypothesis synthesis results below. Is the root cause diagnosis sound, complete, and ready for remediation? Vote APPROVE or BLOCK with specific reasoning."
artifact_path: ".planning/solve/diagnose-context.json"
review_context: "This is a diagnosis gate. Evaluate whether the identified root causes are sufficient and accurate before remediation dispatch begins."
request_improvements: false
```

**Key insight:** This is a GATE, not an advisory. BLOCK from any quorum member halts remediation until diagnosis is refined. This prevents incorrect root causes from cascading through 18 remediation layers.

### Anti-Patterns to Avoid

- **Approach derivation as user input (modal dialog):** INTENT-03 explicitly forbids this. The system must infer intent automatically. Dialogs are anti-patterns in autonomous workflows.
- **Scope contract in a single shared file:** Using one global file instead of per-branch contract prevents concurrent quick tasks from safely writing scope. Use branch name as key in a single JSON file instead.
- **Quorum vote on every diagnosis:** v0.40-02 implements ROOT-02 as a mandatory gate for all diagnoses. v2 (post-v0.40) will add QRUM-01 (complexity gate) to skip quorum on trivial diagnoses. For now, all diagnoses vote.
- **Root cause quorum vote without deliberation:** If votes split, enter deliberation rounds (up to 10 per R3.3). Don't dismiss disagreement as "overthinking" — diagnosis disagreements are usually about incomplete information.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parsing task intent from description | Custom regex/string matching | Haiku subagent + JSON parsing | Semantic understanding is non-trivial; Haiku's instruction-following is purpose-built for intent extraction. Hand-rolled matchers miss edge cases (e.g., "Also update the README" buried in step 5 of description). |
| Multi-model consensus on diagnosis | Majority voting / weighted scoring | R3 quorum protocol (CE-1/CE-2/CE-3 rules) | Diagnosis is binary (sound or unsound), not gradational. Majority voting allows 2/3 APPROVE + 1 BLOCK, which violates CE-2 (any BLOCK halts). R3 has battle-tested fallback logic (tiered slots, deliberation rounds, escalation). |
| Scope contract validation | Manual file checks / ad-hoc parsing | JSON schema validation + type guards | Hand-rolled validation misses edge cases (null values, malformed arrays). Use JSON schema + runtime type checks (TypeScript-like guards in Node.js). |
| Approach block generation | Template strings / string concat | JSON struct + JSON.stringify | String templates are error-prone; JSON ensures well-formedness and machine readability. Scope guard can rely on JSON schema. |

**Key insight:** Both features are lightweight decision gates — they succeed when all participants agree, fail when any disagree. This is the province of quorum infrastructure, not hand-rolled consensus. Reuse quorum.md patterns rather than inventing new consensus rules.

## Common Pitfalls

### Pitfall 1: Scope Contract Pollution

**What goes wrong:** Multiple concurrent quick tasks write to the same `.claude/scope-contract.json` file. Task A writes its branch scope, then Task B writes to the same file, overwriting A's scope. Later, when the scope guard runs for Task A, it finds B's scope instead.

**Why it happens:** Using a file-level lock instead of per-branch keying. Early implementation assumed sequential quick tasks, not parallel execution.

**How to avoid:** Always use git branch name as the JSON key. The file is a map: `{ "branch-name": { scope } }`. Multiple tasks can write to the same file without collision as long as each uses a unique key (their branch name). Add a write-test: spawn two quick tasks with different branch names, verify both scopes coexist in the file after both complete.

**Warning signs:** Scope contract suddenly missing entries after a concurrent quick task completes. Scope guard warning says "no scope contract for branch X" when you just created it. Check file timestamps: if contract was written after you started your task, collision likely occurred.

### Pitfall 2: Approach Block Derivation Brittleness

**What goes wrong:** Haiku's approach derivation relies on semantic understanding of task description. For edge cases ("Fix the thing that's been bothering everyone for weeks" with no explicit scope details), Haiku either generates empty out_of_scope or over-assumes what's in scope.

**Why it happens:** Haiku is a language model; it generalizes from training data. Task descriptions vary in clarity. Vague descriptions → vague approaches.

**How to avoid:** Add an optional post-derivation review step in quick.md: if the approach block looks empty or trivial, re-prompt Haiku with the task description + a stricter schema ("You MUST include at least 2 out_of_scope items"). For v0.40-02, accept empty approaches (fail-open), but log a warning so the user can refine the task description next time.

**Warning signs:** All quick tasks have empty `out_of_scope` arrays. Scope guard warnings consistently cite items you explicitly excluded from the task. This signals that Haiku isn't parsing task descriptions with sufficient granularity.

### Pitfall 3: Quorum Deadlock on Diagnosis

**What goes wrong:** A root cause diagnosis enters quorum deliberation. After round 5, votes are still split: 2 APPROVE, 1 BLOCK. Rounds 6-10 occur with no progress. At round 10, you escalate. But escalation leaves diagnosis in limbo — remediation is blocked indefinitely.

**Why it happens:** Disagreement on root cause usually stems from different information access or inference models. Three models with different training data → legitimate disagreement. Deliberation increases agreement, but doesn't guarantee unanimity.

**How to avoid:** Before deliberation round 5, if votes haven't converged, compute P(consensus by round 10) using empirical quorum history. If P < 20%, escalate proactively instead of waiting for round 10. Escalation (not BLOCK) means "human must review diagnosis" — unblocks the path. Add a flag in solve context: `--force-diagnosis` allows user to override quorum and proceed with a BLOCK diagnosis (with audit trail). This maintains autonomy while surfacing disagreement.

**Warning signs:** Diagnosis quorum deliberation reaches round 7+ with ongoing disagreement. Same models voting BLOCK across multiple rounds. This suggests a fundamental disagreement, not transient misunderstanding.

### Pitfall 4: Scope Guard Invoked Before Contract Created

**What goes wrong:** A quick task branches and starts work immediately. The scope guard hook fires on the first Edit call before the scope contract has been written. Guard finds no contract, issues a warning, and execution proceeds. Later, when the contract is written, guard behavior is inconsistent (existed for first edit, doesn't exist for second edit).

**Why it happens:** v0.40-02 creates the contract during planning (Step 0.5 in quick.md), but the guard is v0.40-03. In v0.40-02, the contract is informational only. If a user starts editing before the planner finishes (concurrent work), guard fires before contract exists.

**How to avoid:** v0.40-02 quick.md must create the scope contract SYNCHRONOUSLY before planner spawn (Step 0.5). Don't defer contract creation to after planner returns. This ensures contract exists before execution can begin. Step order: (1) Derive approach, (2) Write contract, (3) Spawn planner, (4) Planner reads contract if needed (future use).

**Warning signs:** Scope guard logs "no contract found" early in a quick task execution. Scope contract file is created during executor phase instead of planning phase. Check git log to see when scope-contract.json was written relative to executor spawn time.

### Pitfall 5: Diagnosis Quorum Vote Called at Wrong Step

**What goes wrong:** solve-diagnose currently has 18 steps (0a-0q from v0.39). Root cause quorum vote is added at step 0g, but hypothesis synthesis happens at 0f, and some residual classification happens at steps after 0g. Quorum votes on incomplete hypothesis synthesis.

**Why it happens:** Planner added the quorum step without a clear understanding of solve-diagnose's full step sequence. No traceability between "hypothesis synthesis done" and "it's safe to vote".

**How to avoid:** Create a map of solve-diagnose steps before adding the quorum vote. Identify the LAST step that contributes to hypothesis synthesis (call it X). Add quorum vote at X+1. Verify in the test suite: run solve with an intentionally ambiguous residual (could be 3 different root causes). Verify quorum vote fires AFTER all hypothesis synthesis is complete. Check that remediation wave dispatch happens ONLY after quorum approves.

**Warning signs:** Hypothesis synthesis produces new insights in later steps that would have changed the quorum vote. Remediation based on an incomplete diagnosis (missing a causative layer). Check step-by-step solve output: if hypothesis changes after quorum vote, vote was premature.

### Quorum Vote on Root Cause Diagnosis (solve-diagnose Step 0g)

**Implementation:** In solve-diagnose.md after Step 0f (hypothesis synthesis), add Step 0g that dispatches parallel quorum worker Tasks to vote on root cause diagnosis. Dispatch logic follows quorum.md Mode A protocol.

```bash
# Step 0g in solve-diagnose orchestrator (pseudocode):
# Dispatch quorum vote on root cause diagnosis after hypothesis synthesis

# Pre-flight: skip if diagnosis is unambiguous
if [ $(echo "$HYPOTHESES" | jq 'length') -le 1 ]; then
  echo "Step 0g: Single hypothesis (no ambiguity) — skipping quorum vote"
  DIAGNOSIS_VOTE="APPROVED_SINGLE"
else
  echo "Step 0g: Quorum vote on root cause diagnosis"

  # Build hypothesis summary for quorum workers
  DIAGNOSTIC_SUMMARY=$(echo "$HYPOTHESES" | jq -r 'to_entries[] | "Hypothesis \(.key+1): \(.value.cause)\n  Evidence: \(.value.evidence|join(", "))\n  Confidence: \(.value.confidence)"' | tr '\n' '|')

  # Dispatch quorum workers (parallel Task calls, one per active slot)
  # Each worker gets:
  # - slot: <slotName>
  # - round: 1
  # - timeout_ms: 30000
  # - mode: A
  # - question: "Review the hypothesis synthesis results. Is the root cause diagnosis sound and complete? Vote APPROVE or BLOCK."
  # - artifact_path: ".planning/solve/diagnose-hypothesis-log.json"
  # - review_context: "This is a diagnosis gate. Evaluate whether identified root causes are sufficient and accurate before remediation."

  # Collect votes from all dispatched workers (wait for all parallel Tasks)
  # Example expected response format from each worker:
  # { "slot": "gemini-1", "verdict": "APPROVE", "reasoning": "...", "citations": "..." }

  # Evaluate consensus per CE-2 (BLOCK is absolute) and CE-3 (unanimity required):
  # - If any BLOCK vote: ESCALATED (cannot proceed)
  # - If all APPROVE: APPROVED (proceed to remediation)
  # - If mix of APPROVE/FLAG: FLAG (needs human review)

  DIAGNOSIS_VOTE=$(quorum-consensus "$VOTES")
fi

# Route on diagnosis vote:
case "$DIAGNOSIS_VOTE" in
  APPROVED)
    echo "Step 0g: CONSENSUS REACHED — proceeding to remediation wave dispatch"
    ;;
  ESCALATED)
    echo "Step 0g: Root cause diagnosis blocked — halting remediation"
    exit 1
    ;;
  FLAG)
    echo "Step 0g: Diagnosis split opinion — escalating for human review"
    exit 1
    ;;
esac
```

**Consensus rules (from quorum.md CE-1/CE-2/CE-3):**
- **CE-1:** Claude's position is advisory only (not counted in vote tally)
- **CE-2:** Any BLOCK vote from a valid (non-UNAVAIL) external voter prevents consensus
- **CE-3:** Consensus = 100% of valid external voters agree (unanimity required)

**If diagnosis vote does not reach APPROVED:**
- Enter deliberation (up to 10 rounds per R3.3), OR
- Escalate for human review (depends on workflow decision in v0.40-02 plan)

### JSON Schema for Scope Contract Validation (TypeScript-like)

```typescript
// Validation rules for scope contract (use in nf-scope-guard.js, v0.40-03)

interface ScopeContract {
  [branchName: string]: {
    task_id: number;
    task_description: string;
    approach: string;
    out_of_scope: string[];
    branches_affected: string[];
    created_at: string;
    planner_model: string;
    created_by: string;
  }
}

function validateScopeContract(contract: unknown): contract is ScopeContract {
  if (typeof contract !== 'object' || contract === null) return false;

  for (const [key, value] of Object.entries(contract)) {
    if (typeof value !== 'object' || value === null) return false;

    const v = value as any;

    // Required fields
    if (typeof v.task_id !== 'number') return false;
    if (typeof v.task_description !== 'string') return false;
    if (typeof v.approach !== 'string') return false;
    if (!Array.isArray(v.out_of_scope) || !v.out_of_scope.every((s: any) => typeof s === 'string')) return false;
    if (!Array.isArray(v.branches_affected) || !v.branches_affected.every((s: any) => typeof s === 'string')) return false;
    if (typeof v.created_at !== 'string') return false;
    if (typeof v.planner_model !== 'string') return false;
    if (typeof v.created_by !== 'string') return false;
  }

  return true;
}
```

## Open Questions

1. **Quorum vote during solve-diagnose: which step exactly?**
   - What we know: Hypothesis synthesis is Step 0f; quorum vote happens after hypothesis synthesis completes
   - What's unclear: Whether subsequent steps (0g-0q) add to hypothesis list, changing what the quorum vote should evaluate. Does the vote lock the hypothesis list?
   - Recommendation: Add a trace in solve-diagnose test output showing hypothesis list at each step. Quorum vote must occur at a step where hypothesis list is final (no more hypothesis additions in later steps). If hypothesis list changes post-vote, vote is premature.

2. **How does root cause quorum vote interact with existing hypothesis-confidence weighting?**
   - What we know: Current solve has hypothesis rankings with confidence scores
   - What's unclear: Does quorum vote override confidence rankings? Is vote per hypothesis or on the aggregate diagnosis?
   - Recommendation: Quorum votes on the aggregate diagnosis (all hypotheses together), not per-hypothesis. Confidence weighting is a local ranking; quorum gate is a multi-model decision. Keep separate.

3. **Should approach derivation include estimated effort/risk classification?**
   - What we know: Task description → approach block
   - What's unclear: Should approach block include effort estimates or risk classification for the scope guard (v0.40-03)? Or just APPROACH/OUT_OF_SCOPE?
   - Recommendation: For v0.40-02, keep approach block simple: approach text + out_of_scope list. Risk/effort are v0.40-03+ enhancements. Fail-open principle: simpler contracts are easier to validate.

4. **Root cause quorum: What does "approved diagnosis" mean operationally?**
   - What we know: Consensus on diagnosis before remediation
   - What's unclear: Does APPROVE mean "we trust this diagnosis 100%" or "this diagnosis is good enough to attempt remediation"? Different standards.
   - Recommendation: APPROVE means "this diagnosis is actionable and sufficient for remediation wave dispatch." Not 100% certainty — diagnostic uncertainty is normal. If diagnosis is seriously wrong, layers 15+ will flag it and route to model refinement. Quorum gate prevents OBVIOUSLY wrong diagnoses, not subtle errors.

5. **Scope contract storage: per-project or global?**
   - What we know: `.claude/scope-contract.json` is per-project (one file per project)
   - What's unclear: Multiple projects running in parallel on same machine — do scope contracts collide?
   - Recommendation: Scope contracts are per-project. Each project has its own `.claude/` directory (symlinked to `~/.claude/` on install, but actual data is project-local). No collision between projects. But within a project, branch names are the key, so concurrent quick tasks must have different branch names (which they do by design).

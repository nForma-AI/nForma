## Summary

Phase v0.10-06 adds Export Roster and Import Roster options to `bin/manage-agents.cjs`. The feature is a self-contained extension of the existing `readClaudeJson`/`writeClaudeJson` atomic write infrastructure — no new npm packages required. Export reads `~/.claude.json` directly (never via `syncToClaudeJson`) and unconditionally replaces all env values matching `/_KEY$|_SECRET$|_TOKEN$|_PASSWORD$/i` with `__redacted__`. Import validates the schema as a pure function before touching disk, creates a timestamped backup, re-prompts for any redacted key value, and writes atomically.

The phase follows the established 3-wave TDD pattern: Wave 0 adds failing stubs, Wave 1 implements pure functions (`exportRoster`, `importRoster`, `backupClaudeJson`, `validateImportSchema`) that make tests green, Wave 2 wires the interactive menu flows (`performExport`, `performImport`) and adds the two new main menu entries. The `_pure` export block gets four new entries, the destructor at line 5 of the test file is extended, and the main menu grows from 15 items to 17 items (Export at item 16, Import at item 17, before Exit at 0).

The critical PORT-01 security invariant is that the export code path MUST NOT call `syncToClaudeJson()` before reading `~/.claude.json`. `syncToClaudeJson()` (in `bin/secrets.cjs`) patches env values with plaintext from keytar — calling it before export would allow keytar-backed API keys to leak into the portable file. The export reads the raw file directly via `readClaudeJson()` and applies redaction as a separate transformation step.

**Primary recommendation:** Implement `validateImportSchema`, `buildExportData`, `buildRedactedEnv`, and `buildBackupPath` as pure functions tested first; then wire `exportRoster`/`importRoster` as thin I/O wrappers; then wire the menu in Wave 2.

### Pattern 1: Pure-Function-First with Optional filePath

Every function that reads or writes a file takes an optional `filePath` parameter for testability. This is the established project pattern from `readQgsdJson`/`writeQgsdJson`/`writeKeyStatus`.

```javascript
// Source: manage-agents.cjs existing pattern (readQgsdJson, writeKeyStatus, etc.)

// Pure logic — testable with temp files
function buildRedactedEnv(env) {
  if (!env || typeof env !== 'object') return {};
  const SENSITIVE_RE = /(_KEY|_SECRET|_TOKEN|_PASSWORD)$/i;
  const out = {};
  for (const [k, v] of Object.entries(env)) {
    out[k] = SENSITIVE_RE.test(k) ? '__redacted__' : v;
  }
  return out;
}

function buildBackupPath(claudeJsonPath, isoTimestamp) {
  return claudeJsonPath + '.pre-import.' + isoTimestamp;
}
```

### Pattern 2: Atomic Backup with Read-Then-Copy

The backup must complete before any write. Use `fs.copyFileSync` (POSIX atomic on same filesystem) with a fallback that throws on failure — the caller aborts import if backup throws.

```javascript
// I/O helper — wraps pure buildBackupPath
function backupClaudeJson(claudeJsonPath, ts) {
  const dest = buildBackupPath(claudeJsonPath, ts);
  fs.copyFileSync(claudeJsonPath, dest);   // throws on failure → caller aborts
  return dest;
}
```

### Pattern 3: Schema Validation Before Write

`validateImportSchema` is a pure function — it takes a parsed object and returns an array of error strings. Callers check `errors.length === 0` before proceeding. Partial applies are impossible because writes only happen after a clean error array.

```javascript
// Pure function — no side effects
function validateImportSchema(parsed) {
  const errors = [];
  const ALLOWED_COMMANDS = ['node', 'npx'];
  const HOME_PATH_RE = /^\/(Users|home)\//;

  if (!parsed || typeof parsed !== 'object') {
    errors.push('Root must be a JSON object');
    return errors;
  }
  const servers = parsed.mcpServers || {};
  for (const [name, cfg] of Object.entries(servers)) {
    if (cfg.command && !ALLOWED_COMMANDS.includes(cfg.command)) {
      errors.push(`${name}: command must be "node" or "npx", got "${cfg.command}"`);
    }
    for (const arg of (cfg.args || [])) {
      if (typeof arg === 'string' && HOME_PATH_RE.test(arg)) {
        errors.push(`${name}: args contains absolute home path: ${arg}`);
      }
    }
  }
  return errors;
}
```

### Pattern 4: Redacted Key Re-Prompting

The import flow iterates over all slots and finds env values equal to `'__redacted__'`. For each, it prompts via `inquirer.prompt` with `type: 'password'`. The user can skip (empty input). This matches the existing `batchRotateKeys` pattern for sequential per-slot prompts.

```javascript
// Integration function — uses inquirer (not pure)
async function performImport() {
  // 1. Prompt for file path
  // 2. Read + parse file
  // 3. validateImportSchema — abort on errors
  // 4. backupClaudeJson — abort on failure, display path
  // 5. For each slot with __redacted__ env values: inquirer.prompt password
  // 6. writeClaudeJson(patched data) — atomic via tmp-rename
  // 7. Confirm success
}
```

### Pattern 5: Export Does Not Touch syncToClaudeJson

The export path reads `readClaudeJson()` directly and applies `buildRedactedEnv` per slot. `syncToClaudeJson` in `bin/secrets.cjs` patches env values from keytar into `~/.claude.json` — if called before export, keytar-backed plaintext keys would be written to the export file. The export path MUST NOT call `syncToClaudeJson`.

```javascript
// Correct export path:
async function performExport() {
  const { filePath } = await inquirer.prompt([{ type: 'input', name: 'filePath', ... }]);
  const data = readClaudeJson();   // reads raw ~/.claude.json — NO syncToClaudeJson call
  const exportData = buildExportData(data);  // applies redaction per slot
  fs.writeFileSync(filePath, JSON.stringify(exportData, null, 2), 'utf8');
}
```

### Anti-Patterns to Avoid

- **Calling syncToClaudeJson before export:** syncToClaudeJson patches keytar secrets into env values — this would leak real API keys into the export file. Export reads `readClaudeJson()` directly.
- **Writing before validation completes:** All schema errors must be collected and displayed before any backup or write. Never do partial validation then write.
- **Writing before backup succeeds:** Import writes MUST come after `backupClaudeJson` completes. If backup throws, abort.
- **Using `Promise.all` for per-slot key re-prompting:** Must be sequential `for...of` to avoid inquirer concurrency issues (established pattern from batchRotateKeys).
- **Adding `exportRoster`/`importRoster` to `module.exports` top-level:** They are pure logic helpers — expose via `_pure`. Only `performExport`/`performImport` are integration-level.

## Common Pitfalls

### Pitfall 1: syncToClaudeJson Leaking Keys Into Export

**What goes wrong:** If `syncToClaudeJson()` is called anywhere before `readClaudeJson()` in the export flow, keytar-backed API keys are written from the keychain into `~/.claude.json` env values. The subsequent `readClaudeJson()` reads those plaintext values and the redaction step strips them — but if the regex misses any pattern, real keys leak into the portable file.

**Why it happens:** `syncToClaudeJson` is a session-start hook (see `hooks/qgsd-session-start.js:38`) that patches env values. It's natural to assume the "current" state of `~/.claude.json` includes keytar values. It does not by default — the file may have placeholder values or empty strings for keytar-managed keys.

**How to avoid:** Never call `syncToClaudeJson` on the export path. Read `~/.claude.json` directly via `readClaudeJson()`. The Success Criterion for PORT-01 says "the export path never calls `syncToClaudeJson()` before reading". This is a code-level invariant, not just a convention.

**Warning signs:** If the export file contains `sk-ant-...` style strings, syncToClaudeJson leaked them.

### Pitfall 2: Regex Pattern Gaps in Redaction

**What goes wrong:** The redaction regex `/_KEY$|_SECRET$|_TOKEN$|_PASSWORD$/i` misses env var names like `ANTHROPIC_API_KEY` (ends in `_KEY` — matched), `AKASHML_API_KEY` (matched), `OPENAI_SECRET_KEY` (ends in `_KEY` — matched). But if a project adds custom env vars like `PROVIDER_CREDENTIAL`, the regex won't match.

**Why it happens:** Regex is anchored at end-of-string (`$`). Any sensitive key that doesn't end in one of the 4 suffixes passes through unredacted.

**How to avoid:** The roadmap Success Criterion explicitly specifies `/_KEY$|_SECRET$|_TOKEN$|_PASSWORD$/i` — use exactly this regex. Document it as an explicit design choice (regex coverage over 4 suffixes, not exhaustive). Flag the `__redacted__` sentinel so users can verify the output.

**Warning signs:** Any env value in the export file that looks like a bearer token or API key string.

### Pitfall 3: Partial Apply on Validation Failure

**What goes wrong:** Validation discovers errors one at a time and reports them as found, but if any `writeClaudeJson` call slips before the full validation loop completes, the system is in a partially-imported state.

**Why it happens:** Iterative validation with early returns can accidentally fall through to the write path.

**How to avoid:** `validateImportSchema` returns the full `errors[]` array. The caller checks `errors.length === 0` before any disk operation. No writes before that check. This is enforced by keeping the pure validation separate from I/O.

**Warning signs:** `~/.claude.json` changes but the import reports errors afterward.

### Pitfall 4: Backup Write Failure Not Aborting Import

**What goes wrong:** `backupClaudeJson` is called, the backup write silently fails (disk full, permissions), and the import continues to overwrite `~/.claude.json` with no recoverable backup.

**Why it happens:** `try/catch` that swallows the backup error and continues.

**How to avoid:** `backupClaudeJson` MUST throw on failure — never catch silently. The `performImport` function wraps it in a `try/catch` that displays the error and `return`s immediately (no further disk ops).

**Warning signs:** Import reports "Backup failed" but continues to next step.

### Pitfall 5: Inquirer Version Mismatch

**What goes wrong:** Using inquirer v9+ API syntax (ESM-only, `import`-based) in a CJS file.

**Why it happens:** Web searches return inquirer v9 examples; npm may auto-upgrade.

**How to avoid:** This project is locked to `inquirer@8.2.7` (CJS). The `type: 'password'` and `type: 'input'` prompt types work identically in 8.2.7. Do NOT upgrade.

## Open Questions

1. **Should `buildExportData` strip `mcpServers` entries entirely for non-claude-mcp-server agents?**
   - What we know: The export includes the full `mcpServers` map. Some entries may be subprocess providers (`command: 'node'`) with no env block.
   - What's unclear: Whether users want to export subprocess providers (they have no API keys to redact, so no security concern).
   - Recommendation: Export all `mcpServers` entries verbatim (env redaction still applies to any env block). Schema validation on import already handles the `command` whitelist. This is the simplest behavior and matches "full roster export" in PORT-01.

2. **Should `validateImportSchema` check for the `mcpServers` key presence?**
   - What we know: An import file with no `mcpServers` key would result in an empty write — not a security issue but a user error.
   - What's unclear: Whether the validator should enforce `mcpServers` must be present.
   - Recommendation: Treat missing `mcpServers` as valid (results in empty roster import). The per-slot checks only run when slots exist. Add a note to the import UI: "No agents found in file."

3. **Should the ISO timestamp in the backup filename use colons (e.g., `2026-02-25T12:34:56.789Z`)?**
   - What we know: ISO 8601 includes colons; macOS filesystems support colons in filenames; the Success Criterion specifies `~/.claude.json.pre-import.<ISO-timestamp>`.
   - What's unclear: Whether colons in filenames cause issues on non-macOS systems.
   - Recommendation: Use `new Date().toISOString().replace(/:/g, '-')` to produce `2026-02-25T12-34-56-789Z` — filesystem-safe on all platforms. This is a minor deviation from ISO 8601 that avoids an edge case.

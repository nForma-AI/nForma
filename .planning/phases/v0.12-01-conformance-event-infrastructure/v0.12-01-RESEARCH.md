## Summary

This phase builds three collaborating artifacts: a shared schema module, hook instrumentation for NDJSON event emission, and an XState-based trace validator. The schema module (`bin/conformance-schema.cjs`) is the linchpin — it must be written first because both the hooks and the validator import from it, preventing schema drift permanently. The XState machine is compiled offline (not imported at hook runtime) to preserve QGSD's zero-runtime-dependency constraint on hooks.

The hook emission pattern is straightforward: `fs.appendFileSync` on a NDJSON log file, called from within existing decision-turn detection code in `qgsd-stop.js`, with `try/catch` wrapping to maintain fail-open behavior. The NDJSON format (one JSON object per line) requires no parsing library and is trivially appendable without locking. The validator reads the NDJSON log, replays each event through an XState `createActor` instance, and computes a deviation score as `validTraces / totalTraces`.

The primary risk is the TypeScript build isolation: the `src/machines/` directory must have its own `tsconfig.formal.json` so `tsup` can compile the XState machine to CJS without contaminating the existing hook files (which are plain Node.js CJS). The build artifact goes to `dist/machines/`, and `validate-traces.cjs` imports from there via `require('../dist/machines/qgsd-workflow.machine.cjs')`.

**Primary recommendation:** Write `conformance-schema.cjs` first (LOG-01), then `appendConformanceEvent()` helper, then hook instrumentation (LOG-02, LOG-03), then XState machine + tsup build (XST-01..03), then validator (VAL-01..03). This ordering ensures every subsequent step has a stable schema to import from.

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|

|---------|---------|---------|-------------|
| TypeScript | ≥5.0 (xstate requirement) | Type-safe machine definition | Only in `src/machines/` — never in hooks |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| tsup (CJS build) | tsdown (tsup successor) | tsdown is newer but tsup 8.5.1 is actively installed as devDependency; no migration needed now |

| fs.appendFileSync | streaming JSON logger | Streaming loggers add a runtime dependency; POSIX atomic guarantees for small writes are sufficient |

```bash

```

Note: `xstate` is a devDependency of the QGSD repo itself; it must NOT appear in hook files' `require()` calls (zero-dep constraint). Only `validate-traces.cjs` and the machine source use it.

### Pattern 2: NDJSON Append in Hooks (LOG-02, LOG-03)

**What:** Fire-and-forget `fs.appendFileSync` call inside hooks, after the decision-turn gate, with try/catch to stay fail-open.
**When to use:** In `qgsd-stop.js` after `isDecisionTurn` resolves true; in `qgsd-prompt.js` after planning command is detected.

**Critical constraint:** Written to stderr ONLY for errors — `stdout` is the Claude Code decision channel. Using `process.stdout` would corrupt the hook protocol.

```javascript
// Source: derived from existing hook patterns in qgsd-stop.js and config-loader.js
// hooks/qgsd-stop.js — added after isDecisionTurn check, before quorum enforcement
const { VALID_ACTIONS, VALID_PHASES, schema_version } = require('./conformance-schema.cjs');

function appendConformanceEvent(event) {
  // MUST use try/catch — hooks are fail-open; never crash on logging failure
  try {
    const logPath = require('path').join(process.cwd(), '.planning', 'conformance-events.jsonl');
    const line = JSON.stringify(event) + '\n';
    require('fs').appendFileSync(logPath, line, 'utf8');
  } catch (err) {
    // stderr only — stdout is the hook decision channel
    process.stderr.write('[qgsd] conformance log error: ' + err.message + '\n');
  }
}

// Call site (inside qgsd-stop.js main(), after isDecisionTurn === true):
appendConformanceEvent({
  ts:              new Date().toISOString(),
  phase:           'COLLECTING_VOTES',      // derive from context
  action:          'quorum_start',
  slots_available: agentPool.length,
  vote_result:     null,                    // populated after count
  outcome:         null,                    // populated after verdict
  schema_version,
});
```

**Field derivation for hooks:**
- `ts`: `new Date().toISOString()` — always available
- `phase`: derive from quorum position — `COLLECTING_VOTES` when entering; `DECIDED` when exiting
- `action`: `quorum_start` at entry, `quorum_complete` at exit (pass), `quorum_block` at exit (fail)
- `slots_available`: `agentPool.length` (already computed in qgsd-stop.js)
- `vote_result`: `successCount` (already computed in qgsd-stop.js)
- `outcome`: `successCount >= minSize ? 'APPROVE' : 'BLOCK'`

### Pattern 3: XState v5 Machine Definition (XST-01..03)

**What:** 4-state machine using `setup()` for typed guards, compiled to CJS by tsup.
**When to use:** Source file only — never imported at hook runtime.

```typescript
// Source: XState v5 official docs (stately.ai/docs/machines, stately.ai/docs/guards)
// src/machines/qgsd-workflow.machine.ts
import { setup, assign } from 'xstate';

export interface QGSDContext {
  slotsAvailable: number;
  successCount:   number;
  deliberationRounds: number;
  currentPhase:   string;
  maxDeliberation: number;
}

export type QGSDEvent =
  | { type: 'QUORUM_START';    slotsAvailable: number }
  | { type: 'VOTES_COLLECTED'; successCount: number }
  | { type: 'DELIBERATE' }
  | { type: 'DECIDE';          outcome: 'APPROVE' | 'BLOCK' };

export const qgsdWorkflowMachine = setup({
  types: {
    context: {} as QGSDContext,
    events:  {} as QGSDEvent,
  },
  guards: {
    minQuorumMet: ({ context }) =>
      context.successCount >= Math.ceil(context.slotsAvailable / 2),

    noInfiniteDeliberation: ({ context }) =>
      context.deliberationRounds < context.maxDeliberation,

    phaseMonotonicallyAdvances: ({ context, event }) => {
      const order = ['IDLE', 'COLLECTING_VOTES', 'DELIBERATING', 'DECIDED'];
      // Guard: context.currentPhase must not regress
      return true; // structural — transitions enforce monotonicity
    },
  },
}).createMachine({
  id:      'qgsd-workflow',
  initial: 'IDLE',
  context: {
    slotsAvailable:    0,
    successCount:      0,
    deliberationRounds: 0,
    currentPhase:      'IDLE',
    maxDeliberation:   4,
  },
  states: {
    IDLE: {
      on: {
        QUORUM_START: {
          target: 'COLLECTING_VOTES',
          actions: assign({
            slotsAvailable: ({ event }) => event.slotsAvailable,
            currentPhase:   () => 'COLLECTING_VOTES',
          }),
        },
      },
    },
    COLLECTING_VOTES: {
      on: {
        VOTES_COLLECTED: [
          {
            guard:  'minQuorumMet',
            target: 'DECIDED',
            actions: assign({ currentPhase: () => 'DECIDED' }),
          },
          {
            target: 'DELIBERATING',
            actions: assign({
              successCount:       ({ event }) => event.successCount,
              deliberationRounds: ({ context }) => context.deliberationRounds + 1,
              currentPhase:       () => 'DELIBERATING',
            }),
          },
        ],
      },
    },
    DELIBERATING: {
      on: {
        VOTES_COLLECTED: [
          {
            guard:  'minQuorumMet',
            target: 'DECIDED',
            actions: assign({ currentPhase: () => 'DECIDED' }),
          },
          {
            guard:  'noInfiniteDeliberation',
            target: 'DELIBERATING',
            actions: assign({
              deliberationRounds: ({ context }) => context.deliberationRounds + 1,
            }),
          },
          {
            target: 'DECIDED',
            actions: assign({ currentPhase: () => 'DECIDED' }),
          },
        ],
        DECIDE: { target: 'DECIDED' },
      },
    },
    DECIDED: {
      type: 'final',
    },
  },
});
```

### Pattern 5: Trace Validator (VAL-01..03)

**What:** Read NDJSON log, replay each trace through XState machine, compute deviation score.
**When to use:** `bin/validate-traces.cjs` — CJS, no ESM, ships via `qgsd-bin/`.

```javascript
// Source: XState v5 createActor API (stately.ai/docs/actors)
// bin/validate-traces.cjs
'use strict';
const fs    = require('fs');
const path  = require('path');
const { createActor } = require('xstate');
// Import compiled CJS machine — NOT the TypeScript source
const { qgsdWorkflowMachine } = require('../dist/machines/qgsd-workflow.machine.cjs');

const logPath = path.join(process.cwd(), '.planning', 'conformance-events.jsonl');

if (!fs.existsSync(logPath)) {
  process.stderr.write('[validate-traces] No conformance log found at: ' + logPath + '\n');
  process.exit(0); // No log = nothing to validate
}

const lines  = fs.readFileSync(logPath, 'utf8').split('\n').filter(l => l.trim());
let valid    = 0;
const divergences = [];

for (const line of lines) {
  let event;
  try { event = JSON.parse(line); } catch { continue; }

  // Create a fresh actor per trace line
  const actor = createActor(qgsdWorkflowMachine);
  actor.start();

  // Map conformance event fields to XState events
  const xstateEvent = mapConformanceToXState(event);
  if (!xstateEvent) { divergences.push({ event, reason: 'unmappable' }); continue; }

  actor.send(xstateEvent);
  const snapshot = actor.getSnapshot();

  if (!snapshot.matches(expectedState(event))) {
    divergences.push({ event, actual: snapshot.value, reason: 'state mismatch' });
  } else {
    valid++;
  }
  actor.stop();
}

const total = lines.length;
const score = total > 0 ? ((valid / total) * 100).toFixed(1) : '100.0';

process.stdout.write('[validate-traces] Deviation score: ' + score + '% valid (' + valid + '/' + total + ')\n');
if (divergences.length > 0) {
  process.stdout.write('[validate-traces] Divergences:\n');
  for (const d of divergences) {
    process.stdout.write('  ' + JSON.stringify(d) + '\n');
  }
  process.exit(1);
}
process.exit(0);
```

### Anti-Patterns to Avoid

- **Importing xstate in hook files:** QGSD hooks have zero runtime dependencies; `require('xstate')` in `qgsd-stop.js` or `qgsd-prompt.js` would violate this constraint and would break for users who haven't installed xstate.
- **Writing to stdout in hooks:** `process.stdout.write()` in a hook corrupts the Claude Code JSON protocol; all logging MUST go to `process.stderr`.
- **Synchronous file write without try/catch in hooks:** If `.planning/` doesn't exist (fresh clone), `appendFileSync` will throw; hooks must wrap in try/catch and fail-open.
- **Appending without newline:** NDJSON requires each JSON object on its own line ending with `\n`; omitting the newline causes parse failures in the validator.
- **Building the machine with `--dts` included in validate-traces.cjs:** The validator is pure CJS runtime code; type declarations are needed only for downstream TypeScript users.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State machine replay | Custom switch/case FSM | XState v5 `createActor` + `send()` | XState handles guard evaluation, state transition validation, and `type: 'final'` state detection automatically |
| TypeScript to CJS compilation | tsc + manual bundling | tsup `--format cjs` | tsup handles tree-shaking, extern declarations, and produces a single CJS file that `require()` can load |
| NDJSON parsing | Custom newline splitter | `split('\n').filter(l => l.trim())` | Trivial with built-ins; no library needed |
| Schema validation | AJV/Zod | `conformance-schema.cjs` enum checks | Phase scope is small; full JSON Schema validation is overkill |

**Key insight:** The value of XState here is the replay-based conformance checking, not the state management itself. Building a custom FSM loses the authoritative guard evaluation — the validator would need to re-implement QGSD's logic rather than test against it.

## Common Pitfalls

### Pitfall 1: Hook stdout contamination
**What goes wrong:** `process.stdout.write()` called inside the conformance logger causes Claude Code to receive malformed JSON on the hook decision channel — entire hook protocol breaks silently.
**Why it happens:** Developers mistake stderr for optional and stdout for primary; in Claude Code hooks, stdout IS the protocol.
**How to avoid:** All conformance logging errors go to `process.stderr`. The NDJSON append itself produces no output.
**Warning signs:** Hook starts returning decision:block even on valid turns.

### Pitfall 2: appendFileSync throws on missing directory
**What goes wrong:** `.planning/` directory does not exist on fresh checkout; `fs.appendFileSync` throws ENOENT; hook crashes (exit code non-zero = block).
**Why it happens:** `.planning/` is gitignored in some projects; hooks run before the directory is created.
**How to avoid:** Wrap in try/catch; optionally `mkdirSync('.planning', { recursive: true })` before append, inside the try block.
**Warning signs:** All planning turns blocked with no quorum-related message.

### Pitfall 3: XState machine imported in hooks (wrong dist path)
**What goes wrong:** `require('../dist/machines/...')` accidentally ends up in a hook file during refactoring; xstate is not installed in the user's environment; hook crashes.
**Why it happens:** Copy-paste from validate-traces.cjs into hook files.
**How to avoid:** Only `conformance-schema.cjs` is shared between hooks and validator; machine CJS is only imported in `validate-traces.cjs`.
**Warning signs:** Hook throws "Cannot find module 'xstate'" in stderr.

### Pitfall 4: NDJSON log grows unboundedly
**What goes wrong:** `.planning/conformance-events.jsonl` accumulates every quorum turn forever; not gitignored; committed to repository.
**Why it happens:** No truncation policy; oversight in .gitignore.
**How to avoid:** Add `.planning/conformance-events.jsonl` to `.gitignore` during implementation.
**Warning signs:** `git status` shows the log file as modified/untracked after every quorum run.

### Pitfall 5: tsup builds ESM instead of CJS
**What goes wrong:** `dist/machines/qgsd-workflow.machine.js` is ESM-only; `require()` in `validate-traces.cjs` throws `ERR_REQUIRE_ESM`.
**Why it happens:** tsup defaults to ESM when no `--format` flag is specified.
**How to avoid:** Always use `--format cjs` flag; verify output file contains `module.exports` (not `export`).
**Warning signs:** `validate-traces.cjs` throws `ERR_REQUIRE_ESM` or `SyntaxError: Cannot use import statement in a module`.

### Pitfall 6: Validator requires the TypeScript source instead of compiled CJS
**What goes wrong:** `require('./src/machines/qgsd-workflow.machine.ts')` fails because Node.js cannot execute TypeScript directly.
**Why it happens:** Path confusion between source and dist.
**How to avoid:** `validate-traces.cjs` always imports from `dist/machines/`, never `src/machines/`.
**Warning signs:** `SyntaxError: Unexpected token ':'` (TypeScript type annotations not stripped).

## Open Questions

1. **appendConformanceEvent placement in qgsd-stop.js**
   - What we know: The event must fire after `isDecisionTurn === true` and before the quorum count check
   - What's unclear: Whether to emit one event at entry and update it at exit (two-event design) or emit a single event only at exit with both `vote_result` and `outcome` populated
   - Recommendation: Single event at exit — simpler, avoids partial events in the log if the hook crashes mid-execution. The log line is written only when both `vote_result` and `outcome` are known.

2. **conformance-events.jsonl path — absolute vs relative**
   - What we know: `process.cwd()` in hooks is the git project root; `.planning/` is always project-relative
   - What's unclear: Whether to make the log path configurable (via qgsd.json) or hardcode `.planning/conformance-events.jsonl`
   - Recommendation: Hardcode for v0.12-01; configurability is a future concern. The planner should note this as a deferred capability.

3. **tsup vs tsc for machine compilation**
   - What we know: `tsup --format cjs` produces a single bundled `.cjs` file; plain `tsc` produces unbundled `.js` files with separate import statements that may not work with `require()` when xstate uses ESM-first exports
   - What's unclear: Whether xstate's CJS exports (`default: './dist/xstate.cjs.js'`) are compatible with plain tsc output
   - Recommendation: Use tsup — it bundles xstate into the output, making `validate-traces.cjs` self-contained. Add `--external xstate` if the validator's runtime environment always has xstate installed; omit if shipping to users who may not have it.

4. **Distribution of dist/machines/ via npm install**
   - What we know: `bin/install.js` copies `bin/*.cjs` to `~/.claude/qgsd-bin/`; it does NOT copy `dist/` directory
   - What's unclear: Whether `validate-traces.cjs` should bundle the machine (self-contained) or reference `dist/machines/` at runtime
   - Recommendation: Bundle the machine into `validate-traces.cjs` using tsup's bundling (no `--external xstate`). This produces a single standalone file that ships via the existing `qgsd-bin/` mechanism without modifying `install.js`.

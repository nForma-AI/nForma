## Summary

This phase closes two low-severity documentation/portability gaps identified in the v0.9 milestone audit:

**Gap 1 (NYQ-04):** The Step 1 "Parse JSON for:" directive in `qgsd-core/workflows/plan-phase.md` omits `nyquist_validation_enabled` from its named field list (line 21). The field IS emitted by `gsd-tools.cjs` (confirmed via live `init plan-phase` run: `"nyquist_validation_enabled": true`) and IS consumed in Step 5.5, but a strictly-parsing executor would not pre-extract it as a named variable. Adding it to the parse list closes the contract gap — Step 5.5 can then reference `$nyquist_validation_enabled` instead of relying on a recovery annotation.

**Gap 2 (path portability):** Step 5.5 uses three absolute paths beginning with `/Users/jonathanborduas/.claude/` — the hardcoded home directory of the original developer. These paths bypass `install.js`'s `copyWithPathReplacement()` mechanism, which only replaces the `~/.claude/` prefix (via regex `/~\/\.claude\//g`). The fix is to change the source file (`qgsd-core/workflows/plan-phase.md`) to use `~/.claude/qgsd/` prefixes. The installer will then expand them to the correct absolute paths for each user during `node bin/install.js --claude --global`.

Both fixes are pure markdown edits. No JavaScript, no binary, no new files. One plan is sufficient: edit source, run installer sync. The test strategy is smoke/grep verification against both the source and the installed copy.

**Primary recommendation:** Edit `qgsd-core/workflows/plan-phase.md` in two targeted locations (Step 1 parse list and Step 5.5), then run `node bin/install.js --claude --global` to sync the installed runtime.

## Architecture Patterns

### Pattern 1: Installer Path Replacement

**What:** `bin/install.js`'s `copyWithPathReplacement()` reads every `.md` in `qgsd-core/` and replaces `~/.claude/` with the absolute path of the target user's install directory before writing to `~/.claude/qgsd/`.

**Regex used (confirmed by reading `install.js` lines 853-855):**
```javascript
const globalClaudeRegex = /~\/\.claude\//g;
content = content.replace(globalClaudeRegex, pathPrefix);
// pathPrefix = "/Users/<username>/.claude/" for global installs
```

**Consequence:** Only paths that start with `~/.claude/` in the source file get expanded. Absolute paths (starting with `/Users/`) are copied verbatim — they are machine-specific and break portability.

**Pattern to follow:** In source files under `qgsd-core/`, always use `~/.claude/` prefix. Never use `/Users/<username>/`.

### Pattern 2: Workflow File Edit + Install Sync

**Established by:** Phase v0.13-06 decision: "installer sync is the canonical mechanism for deploying qgsd-core/ edits to ~/.claude/qgsd/ — must always include `node bin/install.js --claude --global` as a task when editing workflow files that are installed."

**Sequence (mandatory):**
1. Edit `qgsd-core/workflows/plan-phase.md` (source of truth)
2. Run `node bin/install.js --claude --global`
3. Verify the installed copy at `~/.claude/qgsd/workflows/plan-phase.md`

**Note:** There is no `hooks/dist/` step for workflow files. `hooks/dist/` applies to hook JS files only. Workflow markdown files go directly from `qgsd-core/` → `~/.claude/qgsd/` via the installer.

### Anti-Patterns to Avoid

- **Editing `~/.claude/qgsd/workflows/plan-phase.md` directly:** The installed copy is regenerated on each `node bin/install.js` run. Direct edits are overwritten. Always edit the source in `qgsd-core/`.
- **Using `/Users/<username>/` in source files:** Breaks portability. Use `~/.claude/` so the installer can substitute.
- **Forgetting to run the installer:** The source edit alone does not update the running system. The installed copy at `~/.claude/qgsd/` is what Claude Code reads at runtime.

## Common Pitfalls

### Pitfall 1: Editing the Wrong File

**What goes wrong:** Developer edits `~/.claude/qgsd/workflows/plan-phase.md` (installed copy) instead of `qgsd-core/workflows/plan-phase.md` (source). The edit works temporarily but is overwritten the next time `node bin/install.js --claude --global` is run.

**Why it happens:** Both files have the same name and similar content.

**How to avoid:** Always open the file from the `qgsd-core/` directory tree. Confirm with `grep "/Users/jonathanborduas" qgsd-core/workflows/plan-phase.md` before committing.

**Warning signs:** If the `qgsd-core/` file still has `/Users/jonathanborduas/` after the fix, the wrong file was edited.

### Pitfall 2: Installing Wrong Runtime Target

**What goes wrong:** Running `node bin/install.js --global` (without `--claude`) installs to all runtimes, potentially overwriting OpenCode/Gemini configs with wrong transformations.

**How to avoid:** Use `node bin/install.js --claude --global` (matches the established project pattern per STATE.md decisions and v0.13-06).

**Warning signs:** OpenCode or Gemini configuration files modified unexpectedly.

### Pitfall 3: Incomplete Path Replacement

**What goes wrong:** Only fixing line 154 (the template read path) but missing lines 159 and 171 (the `gsd-tools.cjs` references in Step 5.5). The two `node /Users/...` Bash command references remain machine-specific.

**How to avoid:** Audit all `/Users/jonathanborduas/` occurrences in `qgsd-core/workflows/plan-phase.md` before and after the fix.

**Verification command:**
```bash
grep "/Users/jonathanborduas" qgsd-core/workflows/plan-phase.md
# Should return 0 results after fix
```

### Pitfall 4: Parse List Position

**What goes wrong:** Adding `nyquist_validation_enabled` in the wrong position breaks the logical grouping. If placed after `commit_docs`, it's separated from the other feature-flag booleans.

**How to avoid:** Place it with the other feature flag booleans: after `plan_checker_enabled`, before `commit_docs`.

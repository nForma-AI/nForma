## TASK-00: BROKEN-01 — xstate-to-tla.cjs Output Path Fix

### Problem
`xstate-to-tla.cjs` computes its output path as:
```js
const outDir = outDirArg
  ? path.resolve(outDirArg)
  : path.join(__dirname, '..', 'formal', 'tla');

const tlaOutPath = path.join(outDir, moduleName + '.tla');   // → QGSDQuorum.tla
const cfgOutPath = path.join(outDir, 'MC' + moduleName + '.cfg');  // → MCQGSDQuorum.cfg
```

When called with `--module=QGSDQuorum` (as in STEPS[0] of run-formal-verify.cjs), it writes directly to `formal/tla/QGSDQuorum.tla`, overwriting the hand-authored spec.

### Fix (Option A — quorum-chosen)
Change the generated `.tla` filename to append `_xstate` suffix:
```js
// Before:
const tlaOutPath = path.join(outDir, moduleName + '.tla');
// After:
const tlaOutPath = path.join(outDir, moduleName + '_xstate.tla');
```

The cfg output (`MCQGSDQuorum.cfg`) already has a distinct name — no conflict there.

Update `run-formal-verify.cjs` STEPS[0] comment/label to reflect new output path. No args change needed — the output path is derived from `--module`.

### Impact
- `formal/tla/QGSDQuorum.tla` (hand-authored, with `phase`, AgentSymmetry, MinQuorumMet) is never touched
- `formal/tla/QGSDQuorum_xstate.tla` becomes the generated artifact
- `MCQGSDQuorum.cfg` references `SPECIFICATION Spec` — this works against either model; it already exists as an untracked file from v0.14-01 execution
- `check-spec-sync.cjs` reads `formal/tla/QGSDQuorum.tla` (the canonical one) — continues to work correctly post-fix

### Files
- `bin/xstate-to-tla.cjs` — change `moduleName + '.tla'` to `moduleName + '_xstate.tla'`
- `bin/xstate-to-tla.test.cjs` — add test: `--dry` output says `QGSDQuorum_xstate.tla` not `QGSDQuorum.tla`

## TASK-01: Wire check-spec-sync.cjs into npm test (DRFT-01)

```json

```

`check-spec-sync.cjs` is an executable script (exit 0/1), not a test file. It cannot be directly added to `node --test`. Two integration approaches:

**Approach A (recommended): Add a test wrapper**
Create `bin/check-spec-sync.test.cjs` that calls `spawnSync('node', ['bin/check-spec-sync.cjs'])` and asserts exit code 0 when in sync. This matches the pattern used by `bin/run-formal-verify.test.cjs` and others.

Add `check-spec-sync` to `prelint` or prefix the test command. Less discoverable.

Approach A is consistent with the existing test architecture. Add `bin/check-spec-sync.test.cjs` to the `node --test` invocation in package.json.

## TASK-02: TypeScript Compiler/AST Walk for State Extraction (DRFT-02)

### Current Regex Approach (lines 39-48)
```js
const xstateStateNames = (machineSrc.match(/^    ([A-Z_]+):\s*\{/gm) || [])
  .map(line => line.trim().split(':')[0]);
```
This only extracts top-level state names with 4-space indent. It misses:
- Guard names (e.g., `minQuorumMet`, `noInfiniteDeliberation`)
- Transition names/event types (e.g., `VOTE_CAST`, `DELIBERATE`)
- Nested states
- Context variable names

### TypeScript Compiler API Approach
XState v5 machines are TypeScript. The TypeScript compiler API (`require('typescript')`) can parse `.ts` files into an AST without running esbuild. However, `check-spec-sync.cjs` needs to avoid build dependencies that would slow `npm test`.

**Recommended approach: require() the compiled machine**
`xstate-to-tla.cjs` already uses esbuild to compile + require() the machine at runtime to extract the live machine object. `check-spec-sync.cjs` can do the same:
1. Use esbuild (already a devDependency) to compile the machine to a temp CJS bundle
2. `require()` the bundle to get the live machine object
3. Walk `machine.config` to extract states, guards, events, transitions

This gives access to the full structural information without regex. It matches DRFT-02's requirement: "TypeScript compiler API to parse the XState machine".

**Alternative: ts-morph or typescript compiler API**
`typescript` is a devDependency (via tsconfig.formal.json). `ts-morph` is not installed. Using raw `typescript` compiler API adds complexity. The esbuild+require approach is simpler and already proven in xstate-to-tla.cjs.

**What to extract:**
- State names (all, including nested)
- Guard names (from `guards:` definition or inline guards in transitions)
- Event/transition names (from `on:` blocks)

### Files
- `bin/check-spec-sync.cjs` — replace regex extraction with esbuild+require() of the machine
- Refactor sections 1 and 2 to use `machine.config` instead of regex on raw source

## TASK-03: Orphaned Spec Detection (DRFT-03)

### What to detect
Specs (TLA+, Alloy, PRISM) that reference state/guard names that no longer exist in the XState machine.

Current check-spec-sync.cjs already does the forward check (XState → TLA+). The reverse (TLA+ → XState) is partially present via the `extra` array in Check 1. DRFT-03 requires formalizing this as an error (not a warning) and extending it to Alloy and PRISM.

### Implementation
After extracting the full XState machine structure (from TASK-02's esbuild+require approach):

**TLA+ orphan check:** Already partially done. Change `warn()` to `fail()` for extra TLA+ phases with no XState counterpart (currently `warn('TLA+ TypeOK has extra phases not in XState (may be intentional)')`).

**Alloy orphan check:** Read `formal/alloy/*.als` files, extract state/guard identifiers that match XState naming patterns, compare against XState facts.

**PRISM orphan check:** Read `formal/prism/*.pm` files, extract state labels and constants, compare against XState facts.

### Note
The Alloy/PRISM checks should be best-effort — Alloy and PRISM models may legitimately contain abstractions not 1:1 with XState states. The check should flag names that clearly correspond to XState states/guards but are absent from the machine, not all identifiers.

### Files
- `bin/check-spec-sync.cjs` — add orphan detection sections for TLA+, Alloy, PRISM

### Current
Two steps have `continue-on-error: true`:

Line 119-121: Quorum health check step
```yaml

  run: node bin/verify-quorum-health.cjs || echo "..."
  continue-on-error: true
```

Line 123-127: Full FV pipeline master runner
```yaml
- name: Run full FV pipeline via master runner (run-formal-verify.cjs)
  ...
  continue-on-error: true
```

### Fix
**Quorum health check:** Keep `continue-on-error: true` OR convert the `|| echo "..."` fallback to always succeed. The quorum health check legitimately has no scoreboard in CI. Using the `|| echo` pattern already makes it safe to remove `continue-on-error: true` since the command always exits 0 via the fallback.

**Master runner:** Remove `continue-on-error: true`. After TASK-00 fixes the BROKEN-01 overwrite issue, the master runner should fail the CI pipeline if any formal verification step fails. This is the intent of TASK-05.

### Files
- `.github/workflows/formal-verify.yml` — remove `continue-on-error: true` from master runner step (line 127); optionally remove from quorum health step (line 121) since `|| echo` already handles it

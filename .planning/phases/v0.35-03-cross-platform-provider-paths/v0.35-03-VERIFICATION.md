## Impact Analysis

### GitHub Issue #6 Resolution

The hardcoded `/opt/homebrew/bin/` paths in providers.json break on Linux and WSL. This phase resolves the issue by:

1. **Linux (apt-installed CLIs)**: Runtime resolution discovers `/usr/bin/<cli>` instead of hardcoded `/opt/homebrew/bin/<cli>`
2. **WSL2 (snap-installed CLIs)**: Runtime resolution discovers `/snap/bin/<cli>`
3. **macOS (Homebrew)**: Runtime resolution continues to work with `/opt/homebrew/bin/`, `/usr/local/bin/`, or system paths
4. **All platforms**: Fallback to bare name allows graceful degradation if resolution fails

No manual user configuration required. All platforms benefit from multi-strategy discovery (which → Homebrew → npm → system paths → bare name).

### Code Quality

- **No platform-specific branching** — Platform differences handled entirely by resolveCli() utility
- **Fail-open semantics** — resolve-cli.cjs always returns a non-empty string; spawn will attempt execution even if resolution fails
- **Defensive fallback pattern** — `provider.resolvedCli ?? provider.cli` ensures API-type providers (no subprocess) still work
- **Minimal changes** — Integration is ~25 lines of code per file (resolution logic + spawn site updates)
- **Well-tested** — 30 tests (12 existing + 18 new) validate all code paths

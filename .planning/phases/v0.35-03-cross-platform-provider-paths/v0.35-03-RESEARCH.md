# Phase v0.35-03: Cross-Platform Provider Paths - Research

**Researched:** 2026-03-13
**Domain:** Cross-platform CLI binary path resolution (macOS Homebrew, Linux apt/snap, WSL)
**Confidence:** HIGH

## Summary

The nForma project currently hardcodes CLI binary paths in `bin/providers.json` as `/opt/homebrew/bin/{cli-name}` (Homebrew/macOS). This breaks on Linux systems where binaries install to `/usr/bin` (apt) or `/snap/bin` (snap), and in WSL2 environments where the shell is executed in the WSL filesystem. The phase requires runtime CLI path resolution using the existing `resolve-cli.cjs` tool and proper integration into the MCP server startup and slot dispatch pipelines. The implementation must work cross-platform without manual user configuration.

**Primary recommendation:** Use existing `resolve-cli.cjs` (already in codebase) to resolve provider CLI paths at runtime in `unified-mcp-server.mjs` and `call-quorum-slot.cjs` during provider initialization, applying the multi-strategy fallback (which → Homebrew → npm global → system paths → bare name).

## Architecture Patterns

### Current Path Resolution Pattern (resolve-cli.cjs)
**What:** Existing utility that resolves a bare CLI name to its full executable path using priority-ordered search:
1. `which <name>` — system PATH lookup (first, fastest)
2. Known Homebrew prefixes — `/opt/homebrew/bin`, `/usr/local/bin` (macOS)
3. npm global bin — derived from `npm root -g` (for globally installed npm packages)
4. Common system paths — `/usr/bin` (Linux default)
5. Bare fallback — return name unchanged (let OS resolve at spawn time)

Never throws. Always returns a non-empty string.

**When to use:** Call at provider initialization time (when providers.json is loaded), not at spawn time. Cache resolved paths in provider config or memory to avoid repeated resolution on every tool invocation.

**Example:**
```javascript
// Source: /Users/jonathanborduas/code/QGSD/bin/resolve-cli.cjs
const { resolveCli } = require('./resolve-cli.cjs');

// Resolve once during initialization
const resolvedPath = resolveCli('codex');  // → '/opt/homebrew/bin/codex' or '/usr/bin/codex' or 'codex'

// Use resolved path for all spawns
child = spawn(resolvedPath, args, { env, cwd, stdio });
```

### Anti-Patterns to Avoid
- **Resolving at spawn time vs. init time:** Resolving on every spawn() call adds ~3ms latency per invocation. Resolve once at startup.
- **Platform-specific branching in caller code:** Don't write `if (process.platform === 'darwin')` in unified-mcp-server.mjs. Let resolveCli() handle platform differences.
- **Ignoring the bare-name fallback:** The final fallback to return the bare name allows graceful degradation if resolution fails. Don't skip it; preserve fail-open semantics.
- **Removing hardcoded paths from providers.json:** Keep `/opt/homebrew/bin/codex` as documentation/template. The runtime code replaces it with actual resolved path.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Multi-strategy CLI path discovery | Custom which/fs logic | resolve-cli.cjs (existing) | Handles edge cases: symlinks, npm global bins, Homebrew variants (/opt vs /usr/local); POSIX-compatible |
| Platform-specific path searches | if/else for each OS | resolveCli() with multi-strategy | Maintains single source of truth for resolution; easier to extend for future package managers |
| PATH parsing from environment variable | Custom string.split(':') | which command (via resolveCli) | Handles quoting, escaping, and nonexistent directories in PATH |
| Testing cross-platform paths | Manual VM setup | resolve-cli.test.cjs (existing) + unit tests | Existing test suite covers macOS, Linux, edge cases; no new test framework needed |

**Key insight:** The resolve-cli module is already tested and battle-hardened. Rolling custom path logic duplicates logic, increases maintenance burden, and likely misses edge cases (e.g., binaries in npm global bin, escaped spaces in PATH).

## Common Pitfalls

### Pitfall 1: Resolving Paths at Spawn Time Instead of Startup
**What goes wrong:** Calling `resolveCli()` inside the spawn loop (e.g., every time a tool is invoked) adds cumulative latency: ~3ms per spawn + sub-process overhead.
**Why it happens:** Developers assume "resolve now, use now" without thinking about amortization.
**How to avoid:** Resolve paths once during provider initialization (when providers.json is first loaded). Store resolved paths in memory or append to provider config. Document this in code comments.
**Warning signs:** Profiling shows tool invocations slower than expected; quorum rounds exceed timeout budgets without obvious cause.

### Pitfall 2: Removing Hardcoded Paths from providers.json
**What goes wrong:** Someone deletes `/opt/homebrew/bin/codex` entries thinking they're "no longer needed." Later, a user on a system without that CLI installed gets an unhelpful error because there's no fallback path visible in the config.
**Why it happens:** Confusion about where paths live (config vs. runtime) and when they're used.
**How to avoid:** Document in providers.json that `cli` fields are templates/documentation, not absolute truth. Keep them as-is. The runtime code resolves them. Include a comment like: `"cli": "/opt/homebrew/bin/codex  # template; resolved at runtime via resolve-cli.cjs".`
**Warning signs:** User bug reports "cli not found" without clear indication of what was searched.

### Pitfall 3: Not Testing on Actual Linux/WSL Machines
**What goes wrong:** Code works on macOS (which has /opt/homebrew/bin), but breaks on Linux because the fallback strategy wasn't validated in a real Linux environment.
**Why it happens:** Developers test locally on macOS; CI doesn't include Linux/WSL jobs.
**How to avoid:** XPLAT-02 requirement explicitly mandates testing on macOS, Linux, and WSL. Use Validation Architecture section below.
**Warning signs:** User reports "works on macOS, fails on Linux" after release.

### Pitfall 4: Assuming PATH is Correctly Configured
**What goes wrong:** System has the CLI installed but PATH doesn't include the installation directory (e.g., user hasn't logged in since installing snapd, so /snap/bin isn't in PATH). The `which` strategy fails, but other strategies find the binary.
**Why it happens:** Developers assume standard shell setup; some users have non-standard shells or haven't reloaded their config.
**How to avoid:** Multi-strategy approach (which → Homebrew → npm → system paths) is designed for this. Rely on it; don't assume `which` succeeds.
**Warning signs:** Tool works for user A but not user B, despite both having the CLI installed.

## Code Examples

Verified patterns from official sources:

### Resolve CLI Path at Startup (unified-mcp-server.mjs pattern)
```javascript
// Source: /Users/jonathanborduas/code/QGSD/bin/resolve-cli.cjs and integration pattern
import { spawn } from 'child_process';
import fs from 'fs';

// At startup: resolve provider CLIs once
function resolveProviderPaths(providers) {
  return providers.map(provider => {
    if (provider.type === 'subprocess') {
      // Extract bare CLI name from hardcoded path
      // Example: "/opt/homebrew/bin/codex" → "codex"
      const bareCliName = provider.cli.split('/').pop();

      // Resolve it once at startup
      const { resolveCli } = require('./resolve-cli.cjs');
      provider.resolvedCli = resolveCli(bareCliName);

      // Log for debugging
      if (provider.cli !== provider.resolvedCli) {
        console.error(`[mcp] Resolved ${bareCliName}: ${provider.cli} → ${provider.resolvedCli}`);
      }
    }
    return provider;
  });
}

// Later, when spawning:
child = spawn(provider.resolvedCli, args, { env, cwd, stdio });
```

### resolve-cli.cjs (existing, no changes needed)
```javascript
// Source: /Users/jonathanborduas/code/QGSD/bin/resolve-cli.cjs (lines 25-85)
function resolveCli(name) {
  if (!name || typeof name !== 'string') return name || '';

  // 1. which <name>
  try {
    const result = spawnSync('which', [name], { encoding: 'utf8', timeout: 3000 });
    if (result.status === 0 && result.stdout) {
      return result.stdout.trim();
    }
  } catch (_) {}

  // 2. Known Homebrew prefixes (covers macOS /opt/homebrew/bin and /usr/local/bin)
  const homebrewPrefixes = ['/opt/homebrew/bin', '/usr/local/bin'];
  for (const prefix of homebrewPrefixes) {
    const candidate = path.join(prefix, name);
    try {
      if (fs.existsSync(candidate)) {
        return candidate;
      }
    } catch (_) {}
  }

  // 3. npm global bin
  try {
    const npmResult = spawnSync('npm', ['root', '-g'], { encoding: 'utf8', timeout: 5000 });
    if (npmResult.status === 0 && npmResult.stdout) {
      const npmRoot = npmResult.stdout.trim();
      if (npmRoot) {
        const candidate = path.join(npmRoot, '..', 'bin', name);
        if (fs.existsSync(candidate)) {
          return candidate;
        }
      }
    }
  } catch (_) {}

  // 4. Common system paths (deduplicated)
  const systemPaths = ['/usr/bin'];
  for (const dir of systemPaths) {
    const candidate = path.join(dir, name);
    try {
      if (fs.existsSync(candidate)) {
        return candidate;
      }
    } catch (_) {}
  }

  // 5. Bare fallback
  return name;
}
```

## Open Questions

1. **Should providers.json be modified, or stay as template documentation?**
   - What we know: providers.json currently has `/opt/homebrew/bin` paths. Runtime code will override them.
   - What's unclear: Is it helpful to update the hardcoded paths to `/usr/bin` as a default, or keep them as-is for documentation?
   - Recommendation: Keep `/opt/homebrew/bin` paths as-is (template/documentation). Add a comment in providers.json explaining runtime resolution. This preserves backward compatibility and clarity.

2. **How should installation paths be cached to avoid repeated resolution?**
   - What we know: Resolving on every spawn adds latency; resolution should happen at startup.
   - What's unclear: Should resolved paths be stored in memory only, or persisted to a cache file?
   - Recommendation: Cache in memory during unified-mcp-server lifetime (enough for a session). No persistent cache needed; PATH can change between sessions. Each new session re-resolves once.

3. **Are there edge cases on specific Linux distributions?**
   - What we know: apt uses /usr/bin, snap uses /snap/bin. WSL2 uses Linux filesystem.
   - What's unclear: Do AppImage, Nix, or other alternative package managers require special handling?
   - Recommendation: Scope v0.35-03 to apt and snap (most common). Document that custom installs (AppImage, Nix) should ensure CLI is in PATH or Homebrew-like directory. Future phases can extend.

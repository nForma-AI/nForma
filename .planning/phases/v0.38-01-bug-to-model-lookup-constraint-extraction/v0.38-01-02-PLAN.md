Create `bin/model-constrained-fix.cjs` as a new CommonJS script with the following components:

**1. TLA+ Constraint Extraction (CEX-01):**

`extractTlaConstraints(specContent, specPath)` function:
- **VARIABLES section**: Extract state variable names from `VARIABLES` block (pattern: `VARIABLES\s+(.+?)(?=\n\n|\n\*|\n\\)` — comma-separated list).
- **Invariant definitions**: Match named predicates that are referenced in INVARIANT or used as safety properties. Pattern: `^([A-Z]\w*)\s*==\s*(.+?)(?=\n(?:[A-Z]|\n|---|\*\)))` with multiline+dotall flags. Common names: TypeOK, TypeInvariant, SafetyInvariant, *Inv, *Invariant.
- **Transition predicates (Actions)**: Match named action predicates. Pattern: `^([A-Z]\w+)\s*==\s*(.+?)(?=\n(?:[A-Z]|\n|---|\*\)))`. These have primed variables (e.g., `active' = TRUE`).
- **@requirement markers**: For each extracted block, scan the 200 chars preceding it for `@requirement\s+(\w+-\d+)` pattern. Link constraint to requirement ID.
- **Confidence scoring**: Constraints with @requirement marker get confidence 0.95. Without marker: 0.7.

Return array of:
```js
{
  type: 'invariant' | 'action' | 'state_variable',
  name: 'TypeOK',
  formal: '... TLA+ text ...',
  requirement_id: 'DETECT-01' | null,
  confidence: 0.95 | 0.7,
  spec_path: '.planning/formal/tla/NFCircuitBreaker.tla'
}
```

**2. Alloy Constraint Extraction (CEX-02):**

`extractAlloyConstraints(specContent, specPath)` function:
- **Signature definitions**: Match `sig\s+(\w+)\s*\{([^}]*)\}` to extract signature names and fields.
- **Assertion definitions**: Match `assert\s+(\w+)\s*\{([^}]*)\}` to extract assertion name and body.
- **Predicate definitions**: Match `pred\s+(\w+)\s*(?:\[([^\]]*)\])?\s*\{([^}]*)\}` to extract predicate name, params, and body.
- **Fact blocks**: Match `fact\s*(?:(\w+))?\s*\{([^}]*)\}` for unnamed/named facts.
- **Requirement markers**: Scan for `-- Requirements:\s*(.+)` or `@requirement\s+(\w+-\d+)` in comments preceding each block (200 chars lookback).
- **Confidence scoring**: Same as TLA+ (0.95 with marker, 0.7 without).

Return array of:
```js
{
  type: 'signature' | 'assertion' | 'predicate' | 'fact',
  name: 'ValidState',
  formal: '... Alloy text ...',
  fields: ['pool: set Account', 'active: lone Account'], // for signatures
  requirement_id: 'ALY-AM-01' | null,
  confidence: 0.95 | 0.7,
  spec_path: '.planning/formal/alloy/account-pool-structure.als'
}
```

**3. Plain-English Rendering (CEX-03):**

`renderConstraintSummary(constraints, maxConstraints)` function:
- Takes array of extracted constraints (from either parser)
- Sorts by confidence descending, then by type priority (invariant > assertion > predicate > action > fact > signature > state_variable)
- Limits to `maxConstraints` (default 5, per research guidance to avoid over-constraint)
- Renders each constraint to English using rule-based templates:

Templates (no LLM needed for Phase 1 — deterministic rendering is sufficient and avoids network dependency):
- **invariant**: "SAFETY: {name} requires that {simplified_formal}. [Req: {requirement_id}]"
- **assertion**: "ASSERT: {name} verifies that {simplified_formal}. [Req: {requirement_id}]"
- **predicate**: "RULE: {name} ensures {simplified_formal}. [Req: {requirement_id}]"
- **action**: "TRANSITION: {name} changes state by {simplified_formal}."
- **state_variable**: "STATE: Variable '{name}' tracks {inferred_purpose}."
- **signature**: "STRUCTURE: {name} has fields: {fields_list}."
- **fact**: "CONSTRAINT: {formal_simplified}."

`simplified_formal` helper: Strip TLA+ operators (`/\`, `\/`, `~`, `\in`, `'`) and Alloy operators (`in`, `=>`, `some`, `no`, `lone`) to produce readable English approximations. Use simple regex replacements:
- `/\` -> "AND"
- `\/` -> "OR"
- `~` -> "NOT"
- `\in` -> "is in"
- `= TRUE` -> "is active"
- `= FALSE` -> "is inactive"
- `'` (primed) -> "(next state)"
- Alloy `in` -> "is a member of"
- Alloy `=>` -> "implies"
- Alloy `some` -> "there exists"
- Alloy `no` -> "there is no"

Return:
```js
{
  model_path: 'path/to/spec',
  formalism: 'tla' | 'alloy',
  constraint_count: N,
  constraints: [
    { type, name, english, formal, requirement_id, confidence }
  ]
}
```

**4. CLI interface:**

```
Usage: node bin/model-constrained-fix.cjs --spec <path> [--max-constraints N] [--format json|text]
```

- `--spec`: Path to .tla or .als file (required). Detects formalism from extension.
- `--max-constraints`: Max constraints to return (default: 5)
- `--format`: Output format (default: json). `text` prints English summaries only.

**5. Module exports** for programmatic use:
```js
module.exports = { extractTlaConstraints, extractAlloyConstraints, renderConstraintSummary };
```

**Important:** Do NOT use Haiku/LLM for constraint-to-English in this phase. The research suggested it, but rule-based templates are deterministic, zero-latency, and sufficient for Phase 1. LLM translation can be added in Phase v0.38-02 if template quality is insufficient.

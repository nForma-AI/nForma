## Summary

Phase v0.21-03 closes four self-calibration loops in the formal verification pipeline. All four requirements involve wiring existing pieces together rather than building new infrastructure from scratch. ARCH-01/02/03 from v0.21-01 provides the model registry write infrastructure that LOOP-04 depends on; the debug session artifact format from `debug.md` is the input surface for LOOP-04.

The key architectural insight is that `run-prism.cjs` already reads the scoreboard and computes `tp_rate`/`unavail` inline (lines 147–173 of `run-prism.cjs`). LOOP-01 needs it to also write `rates.const` via `export-prism-constants.cjs` as a pre-step — not to duplicate the computation, but to keep the file artifact current for other consumers (e.g., humans inspecting `rates.const`, CI pipelines that read it directly). The pre-step should be a `spawnSync` call that runs and exits before PRISM is invoked.

For LOOP-02, Claude Code's `PostToolUse` hook system supports a `matcher` field that limits the hook to specific tool invocations. The existing `gsd-context-monitor.js` PostToolUse hook (registered globally in `~/.claude.json`) shows the exact input/output contract. A new hook file `qgsd-spec-regen.js` matching `Write` tool calls to `qgsd-workflow.machine.ts` must be registered in `~/.claude.json` and synced to `hooks/dist/` then installed. The hook must be fail-open — if generation fails, it warns but does not block the Write tool use.

**Primary recommendation:** Wire LOOP-01 first (simplest, no new files), then LOOP-02 (new hook + install), then LOOP-03 (sweep feedback wrapper), then LOOP-04 (new script + debug.md update). All four are independent of each other and can be planned as separate tasks.

### Pattern 1: Pre-Step Subprocess Invocation (LOOP-01)

**What:** Call `export-prism-constants.cjs` via `spawnSync` at the top of `run-prism.cjs`'s main execution path, before any PRISM invocation. Capture stdout/stderr and warn if it fails, but do not abort PRISM if the pre-step fails (fail-open: stale `rates.const` is better than no PRISM run).

**When to use:** When one script must guarantee another has updated a file artifact before it reads that file.

**Example:**
```javascript
// At the top of run-prism.cjs, after policy load, before PRISM args construction:
// ── LOOP-01: Run export-prism-constants as pre-step ────────────────────────
const exportConstantsPath = path.join(__dirname, 'export-prism-constants.cjs');
const exportResult = spawnSync(process.execPath, [exportConstantsPath], {
  encoding: 'utf8',
  cwd: process.cwd(),
});
if (exportResult.status !== 0) {
  process.stderr.write(
    '[run-prism] Warning: export-prism-constants pre-step failed — rates.const may be stale.\n' +
    (exportResult.stderr || '') + '\n'
  );
} else {
  process.stdout.write('[run-prism] Pre-step: rates.const updated from scoreboard.\n');
}
// Continue with existing PRISM invocation logic...
```

**Verification:** Add a test that sets up a tmpDir with a scoreboard and `formal/prism/` directory, runs `run-prism.cjs`, and asserts that `rates.const` was written (even when PRISM binary is absent and fails).

### Pattern 2: PostToolUse Hook with File Matcher (LOOP-02)

**What:** A new hook file `qgsd-spec-regen.js` that reads the PostToolUse JSON payload from stdin, checks if the tool is `Write` and the file path contains `qgsd-workflow.machine.ts`, and if so runs `generate-formal-specs.cjs` as a subprocess.

**When to use:** When a file write must trigger automatic downstream generation.

**Key detail:** Claude Code PostToolUse hook input JSON has the shape:
```json
{
  "tool_name": "Write",
  "tool_input": { "file_path": "/abs/path/to/file" },
  "tool_response": { ... },
  "cwd": "/project/root",
  "context_window": { ... }
}
```

The hook must output valid JSON to stdout for `additionalContext` injection, or exit 0 with no output to be a no-op. Fail-open is mandatory — never block the Write tool.

**Example:**
```javascript
// hooks/qgsd-spec-regen.js
'use strict';
let raw = '';
process.stdin.setEncoding('utf8');
process.stdin.on('data', chunk => { raw += chunk; });
process.stdin.on('end', () => {
  try {
    const input = JSON.parse(raw);
    const toolName = input.tool_name || '';
    const filePath = (input.tool_input && input.tool_input.file_path) || '';
    if (toolName !== 'Write' || !filePath.includes('qgsd-workflow.machine.ts')) {
      process.exit(0); // Not a machine file write — no-op
    }
    const { spawnSync } = require('child_process');
    const path = require('path');
    const genScript = path.join(path.dirname(filePath), '..', '..', 'bin', 'generate-formal-specs.cjs');
    // Alternative: use cwd-relative path
    const result = spawnSync(process.execPath, [
      path.join(input.cwd || process.cwd(), 'bin', 'generate-formal-specs.cjs')
    ], { encoding: 'utf8', cwd: input.cwd || process.cwd() });
    const msg = result.status === 0
      ? '[spec-regen] Formal specs regenerated from XState machine.'
      : '[spec-regen] WARNING: generate-formal-specs.cjs failed. Check output manually.\n' + (result.stderr || '');
    process.stdout.write(JSON.stringify({
      hookSpecificOutput: { hookEventName: 'PostToolUse', additionalContext: msg }
    }));
    process.exit(0);
  } catch (e) {
    process.exit(0); // Fail-open
  }
});
```

**Registration:** Must add an entry to `~/.claude.json` hooks.PostToolUse array AND sync to `hooks/dist/qgsd-spec-regen.js` + run `node bin/install.js --claude --global`. The matcher is set on the hook entry, not in the hook code itself — but since Claude Code PostToolUse hooks currently do not support `matcher` for tool name filtering (only SubagentStart/SubagentStop support `matcher` for subagent type), the tool_name filter must be done inside the hook code.

**Critical finding (HIGH confidence):** Looking at `~/.claude.json`, the existing PostToolUse hook (`gsd-context-monitor.js`) has NO `matcher` field — it fires on every tool use. QGSD's Claude Code version does not show `matcher` on PostToolUse entries. The tool-name and file-path filtering must therefore happen inside the hook script, not via a registration-level matcher.

### Pattern 3: Sensitivity Sweep Feedback Loop (LOOP-03)

**What:** A new script `bin/sensitivity-sweep-feedback.cjs` that:
1. Reads `formal/sensitivity-report.ndjson` (NDJSON, one record per line)
2. Reads the current scoreboard to compute empirical `tp_rate`
3. Compares the empirical rate against the PRISM sweep ranges tested (from the `tp_rate` parameter sweep records in the report)
4. If the empirical rate falls outside the tested range OR the result at the closest tested value was `fail`, updates `rates.const` via `export-prism-constants.cjs`
5. Re-runs PRISM via `run-prism.cjs`
6. If PRISM fails and this is a NEW failure (not a pre-existing fail), exits non-zero (CI signal)

**Deviation threshold:** The sensitivity sweep tests `tp_rate` at `[0.5, 0.75, 0.95]`. If empirical `tp_rate` is below the minimum tested value (0.5) or above the maximum (0.95), that is outside the tested range. If empirical `tp_rate` falls between two tested values and the result changed (delta: `flip-to-fail`), flag it. Recommended threshold: deviation > 0.1 from the nearest tested sweep value triggers a re-run.

**NDJSON record shape** (from `run-sensitivity-sweep.cjs` source):
```javascript
{
  tool: 'run-sensitivity-sweep',
  formalism: 'prism',  // or 'tla'
  result: 'pass' | 'fail' | 'inconclusive',
  check_id: 'sens:prism-tp-rate',
  metadata: {
    parameter: 'tp_rate',
    value: 0.5,
    baseline: 0.85,
    baseline_result: 'pass',
    delta: 'stable' | 'flip-to-fail' | 'flip-to-pass'
  }
}
```

### Pattern 4: Debug Session Invariant Mining (LOOP-04)

**What:** A new script `bin/propose-debug-invariants.cjs` that:
1. Reads `.planning/quick/quorum-debug-latest.md` (the debug artifact from Step F of `debug.md`)
2. Extracts state mentions from the `## bundle` and `## worker responses` sections — XState state names, guard conditions, and transition descriptions
3. Converts extracted transitions into proposed TLA+ `PROPERTY` formulas using simple heuristics:
   - "A → B" becomes `[][phase = "A" => phase' \in {"A", "B"}]_vars`
   - "Never X while Y" becomes `Y => ~X` as an INVARIANT candidate
   - Worker `root_cause` lines mentioning state names become `PROPERTY` candidates
4. Presents proposals interactively (one at a time) — for each: show the TLA+ formula, ask Accept/Reject/Edit
5. For accepted invariants, calls `accept-debug-invariant.cjs` with the target spec, property name, property body, and a generated session ID
6. Reports how many were accepted/rejected

**Debug artifact location:** `.planning/quick/quorum-debug-latest.md` — written by Step F of `debug.md`. File always exists after a debug session.

**Integration point:** The `debug.md` command needs a new Step H added AFTER Step G (Execute or escalate):

```
### Step H: Propose invariant candidates (LOOP-04)

After the debug session concludes (whether consensus was reached or not), run:

  node bin/propose-debug-invariants.cjs

This script reads the debug artifact and proposes TLA+ PROPERTY candidates derived
from the state transitions observed. The user accepts or rejects each individually.
Accepted invariants are written directly to the appropriate formal/tla/ spec.
```

The script can be run non-interactively (for CI) with `--non-interactive` flag, in which case it prints proposals without accepting any.

### Anti-Patterns to Avoid

- **Making LOOP-01 synchronous and blocking:** The pre-step should be fast (reads scoreboard, writes rates.const). If the scoreboard is large, this adds latency. Cap at 10s timeout in spawnSync options.
- **Aborting PRISM on pre-step failure:** LOOP-01 pre-step should always be fail-open. A stale `rates.const` is better than no PRISM run.
- **Hook that blocks Write tool:** LOOP-02 hook must exit 0 even on generation failure. Log the warning via `additionalContext` but never return a non-zero exit or `suppressOutput: true`.
- **Threshold hardcoding in LOOP-03:** Read the sweep values from the existing `SWEEP_PARAMS` in `run-sensitivity-sweep.cjs` (by requiring that file or duplicating the constant). Do not hardcode `[0.5, 0.75, 0.95]` in a new file — instead re-derive from the report records.
- **Interactive prompt in CI for LOOP-04:** The `propose-debug-invariants.cjs` must have a `--non-interactive` mode that prints candidates without prompting. Interactive mode is for human sessions; CI should skip it or always-reject.
- **Hook install without dist sync:** The MEMORY.md explicitly states: "edits to hook source files MUST sync to `hooks/dist/` first, then run `node bin/install.js --claude --global`". Any plan creating `hooks/qgsd-spec-regen.js` must include a task that copies to `hooks/dist/` and runs the installer.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic file writes | Custom file write with error handling | `fs.writeFileSync(tmp); fs.renameSync(tmp, dest)` | This tmp+rename pattern is used in `accept-debug-invariant.cjs`, `generate-formal-specs.cjs`, `promote-model.cjs` — it's the QGSD standard |
| TLA+ PROPERTY syntax construction | Custom template engine | Simple string template: `'PROPERTY ' + name + ' == ' + body` | `accept-debug-invariant.cjs` already implements this; `propose-debug-invariants.cjs` should call it |
| NDJSON parsing | JSON.parse with streaming | `file.split('\n').filter(l => l).map(l => JSON.parse(l))` | NDJSON is newline-delimited JSON; synchronous read + split is fine for files up to ~10MB |
| Rate deviation detection | Statistics library | Simple arithmetic: `Math.abs(empirical - swept) > threshold` | The sweep only tests 3 values; no statistical library needed |
| Hook JSON output | Custom serialization | `JSON.stringify({ hookSpecificOutput: { hookEventName: 'PostToolUse', additionalContext: msg } })` | This is the exact shape used by `gsd-context-monitor.js` |

**Key insight:** QGSD has established patterns for every operation this phase needs. The risk is diverging from those patterns and creating inconsistency in the codebase.

## Common Pitfalls

### Pitfall 1: LOOP-01 Double-Computing Scoreboard Rates
**What goes wrong:** `run-prism.cjs` already reads the scoreboard and computes `liveTPRate`/`liveUnavail` at lines 147–173. If the pre-step also reads the scoreboard, we have two reads. They will agree (same file, same instant), but it wastes I/O and creates confusion.
**Why it happens:** `export-prism-constants.cjs` computes per-slot rates (for `rates.const`), while `run-prism.cjs` computes aggregate rates (for `-const tp_rate=` and `-const unavail=` injection). These are different calculations serving different purposes.
**How to avoid:** LOOP-01's job is to keep `rates.const` fresh. The pre-step writes `rates.const`. `run-prism.cjs` still reads the scoreboard for its own `-const` injection — this is correct and separate from `rates.const`. Do not try to merge these two reads.
**Warning signs:** If you find yourself modifying the scoreboard-reading logic in `run-prism.cjs` itself, you've gone off track.

### Pitfall 2: LOOP-02 Hook Not Getting Installed
**What goes wrong:** The hook file is written to `hooks/qgsd-spec-regen.js` and even to `hooks/dist/qgsd-spec-regen.js`, but `~/.claude.json` is not updated, so it never fires.
**Why it happens:** The installer reads `hooks/dist/` and updates `~/.claude.json` — but only for hooks it knows about. A new hook requires explicit registration. The install script (`bin/install.js`) may need to be updated to know about `qgsd-spec-regen.js`, OR the registration can be done manually with `node bin/install.js --claude --global` after updating the install script's hook list.
**How to avoid:** In the plan for LOOP-02, make the install step explicit: (1) write `hooks/qgsd-spec-regen.js`, (2) copy to `hooks/dist/`, (3) verify `~/.claude.json` PostToolUse array includes the new hook entry, (4) run install. Write a test that verifies `~/.claude.json` contains the expected entry.
**Warning signs:** If testing shows `generate-formal-specs.cjs` is not being called after writing the machine file, check `~/.claude.json` PostToolUse entries first.

### Pitfall 3: LOOP-03 Treating `inconclusive` as `fail`
**What goes wrong:** `run-sensitivity-sweep.cjs` emits `result: 'inconclusive'` when PRISM is not installed. If the feedback script treats `inconclusive` as a deviation or failure, CI fails on environments without PRISM.
**Why it happens:** The sweep is designed to gracefully degrade when tools are absent.
**How to avoid:** In `sensitivity-sweep-feedback.cjs`, only act on `result: 'pass'` or `result: 'fail'` records. Skip all `inconclusive` records. If ALL records for a parameter are `inconclusive`, emit a warning but exit 0.
**Warning signs:** CI fails with "PRISM not found" errors in the feedback step.

### Pitfall 4: LOOP-04 TLA+ Syntax Errors in Proposed Properties
**What goes wrong:** The mining heuristics produce syntactically invalid TLA+ PROPERTY bodies (e.g., using `→` instead of `=>`, or JavaScript boolean expressions).
**Why it happens:** TLA+ syntax is not JavaScript/Python; naive text extraction produces invalid formulas.
**How to avoid:** The `propose-debug-invariants.cjs` script should present the formula to the user BEFORE passing it to `accept-debug-invariant.cjs`. The Edit option in the interactive prompt allows correction. For non-interactive CI mode, never auto-accept — only list.
**Warning signs:** `accept-debug-invariant.cjs` succeeds but TLC fails when the spec is verified.

### Pitfall 5: Session ID Generation for LOOP-04
**What goes wrong:** `accept-debug-invariant.cjs` requires a session ID matching `debug-sess-<unix-timestamp-seconds>-<8-char-hex>`. If `propose-debug-invariants.cjs` generates a non-conforming ID, the write fails with exit code 1.
**Why it happens:** The format is validated by a regex in `accept-debug-invariant.cjs` lines 70–77.
**How to avoid:** Use the exact pattern from the `accept-debug-invariant.cjs` source comment: `` `debug-sess-${Math.floor(Date.now()/1000)}-${Math.random().toString(16).slice(2,10)}` ``
**Warning signs:** Exit code 1 from `accept-debug-invariant.cjs` with "must match format" error.

2. **Should `propose-debug-invariants.cjs` call `accept-debug-invariant.cjs` as a subprocess or require it directly?**
   - What we know: `accept-debug-invariant.cjs` is a CLI tool with `require.main === module` — it does NOT export its core functions as a module (unlike `export-prism-constants.cjs` which exports `_pure`).
   - What's unclear: Whether adding a module export to `accept-debug-invariant.cjs` is desirable or if subprocess invocation is preferred.
   - Recommendation: Call as subprocess via `spawnSync` to keep contracts clean. `propose-debug-invariants.cjs` builds the args and calls `accept-debug-invariant.cjs` for each accepted candidate.

3. **What is the deviation threshold for LOOP-03?**
   - What we know: The sweep tests `tp_rate` at `[0.5, 0.75, 0.95]` with baseline `0.85`. The current empirical `tp_rate` from the scoreboard (442 rounds) would be computable but was not calculated during research.
   - What's unclear: Whether 0.1 absolute deviation or 10% relative deviation is the right trigger.
   - Recommendation: Use 0.05 absolute deviation (if empirical rate is more than 0.05 away from the nearest tested sweep value, update and re-run). Document as a configurable constant at the top of `sensitivity-sweep-feedback.cjs`.

## Summary

Phase v0.37-03 adds FP rate tracking per scanner per category, auto-threshold tuning for chronic FP offenders, and a diagnostics table in report-only mode.

## Current Architecture

### Scanners
Three reverse-traceability scanners in `bin/nf-solve.cjs`:
- **sweepCtoR** (code-to-requirements) -- classifications stored under `ctor` key
- **sweepTtoR** (test-to-requirements) -- classifications stored under `ttor` key
- **sweepDtoR** (doc-to-requirements) -- classifications stored under `dtor` key

### Classifications Storage
`.planning/formal/solve-classifications.json` stores per-scanner classification results:
```json
{
  "updated_at": "timestamp",
  "classifications": {
    "ctor": { "file_path": "genuine|fp|review" },
    "ttor": { "file_path": "genuine|fp|review" },
    "dtor": { "hash_or_id": "genuine|fp|review" }
  }
}
```

### Suppression
Current suppression is pattern-based (hardcoded rules in sweepCtoR/sweepTtoR) plus proximity-based. No per-scanner FP rate tracking or dynamic threshold adjustment exists.

## Implementation Approach

### FPTUNE-01: FP Rate Tracking
Add a `session_history` section to solve-classifications.json:
```json
{
  "session_history": [
    {
      "session_id": "iso-timestamp",
      "scanner_stats": {
        "ctor": { "total": N, "fp": M, "genuine": K, "review": J },
        "ttor": { "total": N, "fp": M, "genuine": K, "review": J },
        "dtor": { "total": N, "fp": M, "genuine": K, "review": J }
      }
    }
  ]
}
```
Rolling window: keep last 10 sessions, trim older entries on each write.

### FPTUNE-02: Auto-Threshold Tuning
After session stats are recorded, compute rolling FP rate per scanner:
- FP rate = total FP across window / total items across window
- If FP rate > 0.6 AND window has 5+ sessions: raise suppression threshold by 0.1
- Cap at 0.9 (never suppress more than 90%)
- Store active thresholds in a `tuning` section of solve-classifications.json

### FPTUNE-03: Report Display
In Phase 4 (report) of nf-solve.cjs, when `--report-only` flag is active, format and display an FP rate table:
```
Per-Scanner FP Rates (last 10 sessions):
| Scanner | Sessions | FP Rate | Threshold | Status |
|---------|----------|---------|-----------|--------|
| ctor    | 8        | 0.45    | 0.50      | OK     |
| ttor    | 10       | 0.72    | 0.60      | TUNED  |
```

## Pitfalls

1. **Stale classifications**: If solve-classifications.json is manually edited, session_history may not match. Solution: session_history is append-only, derived from actual scan results.
2. **First run**: No session_history exists. Solution: initialize empty array, first session creates baseline.
3. **Threshold persistence**: Thresholds must persist across sessions. Store in solve-classifications.json `tuning` key.

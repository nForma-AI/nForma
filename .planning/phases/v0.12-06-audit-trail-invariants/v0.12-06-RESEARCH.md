Phase v0.12-06 requires two new Alloy 6 specifications: `formal/alloy/scoreboard-recompute.als` (models the `recomputeStats`/`recomputeSlots` reset-then-replay algorithm) and `formal/alloy/availability-parsing.als` (models the `parseLocalDateTime` and `parseAvailabilityHint` date arithmetic). Both feed into a single runner `bin/run-audit-alloy.cjs` that invokes the Alloy 6 JAR headlessly for both `.als` files, gated on `JAVA_HOME`. This follows the exact same 3-plan pattern established by v0.12-04 and v0.12-05.

The source functions being modeled live entirely in `bin/update-scoreboard.cjs`. The scoreboard recomputation (`recomputeStats` at line 196, `recomputeSlots` at line 238) uses a deterministic reset-then-replay pattern that is directly expressible in Alloy relational logic. The date arithmetic (`parseLocalDateTime` at line 419, `parseAvailabilityHint` at line 452) is a pure function with well-defined null-on-failure behavior, making it ideal for Alloy's exhaustive model checking.

The existing Alloy infrastructure in this repo — `formal/alloy/org.alloytools.alloy.dist.jar` (already present), `formal/alloy/install-scope.als` and `formal/alloy/quorum-votes.als` as style templates, and `bin/run-alloy.cjs` as the runner template — provides exact patterns for all four deliverables. Unlike the TLC-based phases (v0.12-04, v0.12-05), this phase uses Alloy (not TLA+), so the runner pattern comes from `bin/run-alloy.cjs` rather than `bin/run-oscillation-tlc.cjs`. The critical distinction: Alloy uses the `org.alloytools.alloy.dist.jar` with `exec --output - --type text` CLI, not the TLC `-jar` pattern.

**Primary recommendation:** Follow the v0.12-04/v0.12-05 three-plan structure: Plan 01 = Wave 0 RED stubs for `bin/run-audit-alloy.test.cjs` + add to `npm test`; Plan 02 = author `scoreboard-recompute.als` + `availability-parsing.als`; Plan 03 = implement `bin/run-audit-alloy.cjs` and turn tests GREEN. ### Pattern 1: Scoreboard Recomputation Spec (scoreboard-recompute.als)

**What:** Model the `recomputeStats` and `recomputeSlots` functions as relational operations over a bounded set of rounds, models, and vote codes.

**Alloy model design:**

The scoreboard data structure:
- `sig VoteCode` — abstract, with one sig per code: `TP`, `TN`, `FP`, `FN`, `TPplus`, `TNplus`, `UNAVAIL`
- `sig Model` — abstract, with one sig per model: `Claude`, `Gemini`, `Codex`, `OpenCode`, `Copilot`
- `sig Round` — represents one quorum round; has `votes: Model -> lone VoteCode`
- `sig Stats` — represents computed model stats; has `score: one Int`

**Score delta as Alloy predicate:**
```alloy
-- Score delta for each vote code (mirrors SCORE_DELTAS in update-scoreboard.cjs)
fun scoreDelta [v: VoteCode] : Int {
  v = TP    => 1  else
  v = TPplus => 3 else
  v = TN    => 5  else
  v = TNplus => 7 else
  v = FP    => -3 else
  v = FN    => -1 else
  0  -- UNAVAIL = 0
}
```

**Key assertions (GAP-3):**
```alloy
-- RecomputeIdempotent: recomputing from rounds[] twice yields same result as once.
-- Modeled as: two Stats computed from the same set of rounds must be equal.
assert RecomputeIdempotent {
  all rounds: set Round, s1, s2: Stats |
    (ScoreOf[s1, rounds] and ScoreOf[s2, rounds]) => s1.score = s2.score
}

-- NoVoteLoss: every vote in rounds[] contributes to the final score.
-- Modeled as: if a round contains a model vote, the score reflects that delta.
assert NoVoteLoss {
  all rounds: set Round, m: Model, v: VoteCode |
    (some r: rounds | r.votes[m] = v) =>
    ... (score includes delta for v)
}

-- NoDoubleCounting: no single vote's delta appears in the score more than once.
-- Modeled as: the computed score equals the sum over all rounds of their deltas.
assert NoDoubleCounting {
  all rounds: set Round, m: Model |
    ScoreOf[m, rounds] = sumDeltas[m, rounds]
}
```

**Scope:** `check RecomputeIdempotent for 5 Round, 7 VoteCode, 5 Model, 5 Stats`

Note: Alloy integer arithmetic requires `sig` instances to be bounded. The score values are bounded by the number of rounds times the maximum delta (7 * max_rounds). Use `for 5 Round` and `6 Int` (for -32 to 31 range) in the check commands.

### Pattern 2: Availability Parsing Spec (availability-parsing.als)

**What:** Model the `parseLocalDateTime` and `parseAvailabilityHint` functions over a bounded representation of timestamps.

**Alloy model design:**

The key behavior to verify:
1. `ParseCorrect`: a successfully parsed timestamp is >= now (or exactly now — the function returns future dates)
2. `YearRolloverHandled`: when a month/day/time combination is already in the past for the current year, the parser returns next year's date
3. `FallbackIsNull`: any input that doesn't match the known patterns returns null (no crash)

**Abstract timestamp as integer:**
```alloy
-- Represent timestamp as a signed integer (Unix epoch seconds, bounded)
-- Alloy Int is bounded: for 6 Int, range is -32..31 (sufficient for relative comparisons)

-- Parsed timestamp represents a future Date (>= now)
-- Model: input is a Month+Day+Hour+Minute tuple; output is a timestamp or null

abstract sig Month {}
one sig Jan, Feb, Mar, Apr, May, Jun, Jul, Aug, Sep, Oct, Nov, Dec extends Month {}

sig ParseInput {
  month: one Month,
  day: one Int,    -- 1..31
  hour: one Int,   -- 0..23
  minute: one Int  -- 0..59
}

sig ParseResult {
  available_at: lone Int,  -- null (no Int) = parse failed
  reason: lone Reason      -- null = parse failed
}

abstract sig Reason {}
one sig UsageLimit, QuotaExceeded, RateLimit, Unavailable extends Reason {}
```

**Key assertions (GAP-9):**
```alloy
-- ParseCorrect: if a result is returned, available_at >= now
assert ParseCorrect {
  all input: ParseInput, result: ParseResult |
    ParsedWith[input, result] =>
    (no result.available_at or result.available_at >= Now)
}

-- YearRolloverHandled: a date in the past must roll to next year, not return a past date
assert YearRolloverHandled {
  all input: ParseInput |
    PastDate[input] => ParseResult[input].available_at > Now
}

-- FallbackIsNull: an invalid month key returns no result (null)
assert FallbackIsNull {
  all input: ParseInput |
    InvalidMonth[input] => no ParseResult[input].available_at
}
```

### Anti-Patterns to Avoid

- **Hardcoded assertion name in counterexample message:** `run-alloy.cjs` has "NoSpuriousApproval" hardcoded in the CEX warning — ALY-02 partial gap. `run-audit-alloy.cjs` MUST use the dynamic `specName` variable, not a hardcoded assertion name.
- **Using TLA+ integers (Nat) instead of Alloy integers:** Alloy uses `Int` (signed, bounded by bitwidth), not `Nat`. For score arithmetic: `plus[score, delta]` (not `+`), `minus[score, delta]` (not `-`). Alloy uses `mul`, `plus`, `minus` as prefix functions for integer operations.
- **Modeling concrete JS strings in Alloy:** Do not try to represent month name strings ("Jan", "Feb") as Alloy strings. Use abstract `sig Month` with `one sig Jan, Feb, ...` — this keeps the model finite and checkable.
- **Skipping the Alloy integer bitwidth setting:** Alloy's Int type is bounded by the scope `for N Int`. With `for 6 Int`, the range is -32..31. For score sums over 5 rounds with max delta 7, the maximum possible score is 35 — just outside 6-bit range. Use `for 7 Int` (range -64..63) for the scoreboard spec to avoid integer overflow in Alloy's model.
- **JAR absence check failing when JAR is physically present:** The v0.12 milestone audit found that `run-alloy.test.cjs` test 2 fails because `org.alloytools.alloy.dist.jar` is physically present (gitignored but downloaded). `run-audit-alloy.test.cjs` must handle this same situation — the JAR-not-found test is only meaningful when the JAR is actually absent. Follow the skip-if-JAR-present approach or mock JAR path.
- **Forgetting `--quiet` flag in Alloy invocation:** Without `--quiet`, Alloy may emit verbose output that interferes with counterexample detection. The `--quiet` flag is used in `run-alloy.cjs` line 95 and MUST be included in `run-audit-alloy.cjs`.

## Common Pitfalls

### Pitfall 1: Alloy Integer Overflow in Scoreboard Spec

**What goes wrong:** Alloy's `Int` type is bounded by the scope's `for N Int` parameter. With `for 6 Int`, the range is -32..31. If a model has 5 rounds of TN+ votes (each +7), the score is 35, which overflows the 6-bit range — TLC-style: Alloy silently wraps around or gives wrong results.
**Why it happens:** Forgetting that Alloy Int has bounded range set by the `check ... for N Int` scope.
**How to avoid:** Use `for 7 Int` in all `check` commands for `scoreboard-recompute.als`. This gives -64..63, sufficient for 5 rounds × max delta 7 = 35. Always verify: `max(rounds) * max(delta) < 2^(N-1)`.
**Warning signs:** TLC finds counterexamples with scores that look like they wrapped around (e.g., score -61 when 7 TN+ rounds × +7 = 49, and the wrapping gives 49 - 128 = -79).

### Pitfall 2: Stale Assertion Name in Counterexample Warning

**What goes wrong:** Copying `run-alloy.cjs` verbatim copies the stale "NoSpuriousApproval" assertion name in the CEX warning (line 113). The new runner then reports "Counterexample found in NoSpuriousApproval assertion" even when checking `scoreboard-recompute.als`, which has no `NoSpuriousApproval` assertion.
**Why it happens:** ALY-02 partial gap from the v0.12 milestone audit: the existing `run-alloy.cjs` has a hardcoded stale name. The fix was deferred.
**How to avoid:** In `run-audit-alloy.cjs`, use `specName` dynamically: `'[run-audit-alloy] WARNING: Counterexample found in ' + specName + '.als assertion.'` — never hardcode an assertion name.
**Warning signs:** The CEX warning mentions an assertion name that does not exist in the .als file being checked.

### Pitfall 3: ALS File Check vs JAR Check Order

**What goes wrong:** If the runner checks for the `.als` file BEFORE checking for Java or the JAR, test 1 (JAVA_HOME error) and test 2 (JAR-not-found) will fail because the ALS check fires first (the file won't exist until Plan 02 creates it).
**Why it happens:** Logical ordering: the `.als` file should only be checked after Java and JAR are confirmed present.
**How to avoid:** Follow the exact check order from `run-alloy.cjs`: (1) JAVA_HOME check, (2) Java version check, (3) JAR existence check, (4) `.als` file existence check. The error-path tests specifically rely on this ordering — test 1 fires before the JAR check, test 2 fires before the .als check.
**Warning signs:** Test 1 (JAVA_HOME error) passes but shows a different error message than expected — the ALS check is firing first.

### Pitfall 4: run-audit-alloy.test.cjs Test 2 Fails When JAR is Present

**What goes wrong:** `run-audit-alloy.test.cjs` test 2 (JAR-not-found) expects exit code 1. But if `org.alloytools.alloy.dist.jar` is physically present (gitignored but downloaded during v0.12-03), the runner passes the JAR check and either exits 0 (no .als file yet) or hits the .als file check with a different error.
**Why it happens:** Same root cause as the existing `run-alloy.test.cjs` ALY-02 failure: the JAR is physically present in the developer's working copy.
**How to avoid:** In test 2, check whether the Alloy JAR exists before asserting exit code 1:
```javascript
const jarPath = path.join(__dirname, '..', 'formal', 'alloy', 'org.alloytools.alloy.dist.jar');
const fs = require('fs');
if (fs.existsSync(jarPath)) { return; }  // skip if JAR is present — can't test absent-JAR path
```
This is the cleanest approach for the Wave 0 RED stubs — the test is marked as environment-conditional, not broken.
**Warning signs:** `npm test` passes in CI (no JAR) but fails locally (JAR present).

### Pitfall 5: npm test Script Not Updated

**What goes wrong:** `bin/run-audit-alloy.test.cjs` is created but not appended to the `npm test` script in `package.json`.
**Why it happens:** The test script must be manually appended; there is no auto-discovery.
**How to avoid:** Plan 01 MUST update `package.json` scripts.test to append `bin/run-audit-alloy.test.cjs`. Verify immediately after:
```bash
node -e "const p = require('./package.json'); console.log(p.scripts.test.includes('run-audit-alloy'))"
```
Expected: `true`.
**Warning signs:** `npm test` passes but `node --test bin/run-audit-alloy.test.cjs` reveals failures not reported by npm.

### Pitfall 6: Alloy `util/ordering` vs `seq` for Rounds

**What goes wrong:** Modeling the `rounds` array as an Alloy `seq Round` (Alloy's built-in ordered sequence type) is tempting but adds complexity — `seq` in Alloy 6 is available but increases the check scope significantly.
**Why it happens:** The scoreboard's `data.rounds` is an ordered array in JS, but for the idempotency/vote-loss assertions, the ORDER of rounds doesn't matter — only which rounds are included and what votes each contains.
**How to avoid:** Model `rounds` as an unordered `set Round` in the check scope. The `RecomputeIdempotent` invariant asserts that two `Stats` computed from the same SET of rounds are equal — order-independence is a stronger property and is correct for the accumulation algorithm (the reset-then-replay loop processes rounds in iteration order, but commutativity of addition means order doesn't affect the final score).
**Warning signs:** Alloy check times out or reports "too many states" when using `seq Round` — switch to `set Round`.

## Code Examples

### scoreboard-recompute.als — Full Skeleton

```alloy
-- formal/alloy/scoreboard-recompute.als
-- Handwritten — not generated from XState.
-- Source: bin/update-scoreboard.cjs (recomputeStats, recomputeSlots)
--
-- Models the QGSD scoreboard recomputation algorithm:
--   recomputeStats: reset all model stats to 0, then replay all rounds
--   recomputeSlots: reset all slot stats to 0, then replay all rounds
--
-- Assertions:
--   RecomputeIdempotent: applying recompute twice = once (deterministic pure function)
--   NoVoteLoss: every vote in rounds[] contributes to the final score
--   NoDoubleCounting: no vote's delta is counted more than once
--
-- Scope: 5 rounds, 5 models, 7 vote codes

module scoreboard_recompute

-- Vote classification codes (mirrors SCORE_DELTAS in update-scoreboard.cjs)
abstract sig VoteCode {}
one sig TP, TN, FP, FN, TPplus, TNplus, UNAVAIL extends VoteCode {}

-- Score delta for each vote code
fun scoreDelta [v: VoteCode] : Int {
  (v = TP    => 1  else
  (v = TPplus => 3 else
  (v = TN    => 5  else
  (v = TNplus => 7 else
  (v = FP    => -3 else
  (v = FN    => -1 else
  0))))))
}

-- Abstract model identities
abstract sig Model {}
one sig Claude, Gemini, Codex, OpenCode, Copilot extends Model {}

-- A single quorum round: each model has at most one vote
sig Round {
  votes: Model ->lone VoteCode
}

-- Computed score for a model given a set of rounds
-- Models recomputeStats: sum of scoreDelta for all votes for this model across all rounds
fun computeScore [m: Model, rounds: set Round] : Int {
  sum r: rounds | scoreDelta[r.votes[m]]
}

-- RecomputeIdempotent: the score is a pure function of (model, rounds set)
-- Applying the computation twice on the same rounds set yields the same result.
-- Since computeScore is deterministic (no state), this is trivially true in Alloy —
-- the assertion verifies no hidden non-determinism in the model.
assert RecomputeIdempotent {
  all m: Model, rounds: set Round |
    computeScore[m, rounds] = computeScore[m, rounds]
}

-- NoVoteLoss: if model m has a non-null vote in some round, the score is nonzero
-- (unless the vote is UNAVAIL with delta 0 or canceling +/- votes balance out)
-- Stronger version: every vote's delta is reflected in the total sum
assert NoVoteLoss {
  all m: Model, rounds: set Round |
    (all r: rounds | some r.votes[m]) =>
    computeScore[m, rounds] = sum r: rounds | scoreDelta[r.votes[m]]
}

-- NoDoubleCounting: the score equals the sum of individual round deltas, not more
assert NoDoubleCounting {
  all m: Model, rounds: set Round |
    computeScore[m, rounds] = (sum r: rounds | scoreDelta[r.votes[m]])
}

-- Verify threshold: TN+ vote in a single round gives score 7
assert TNplusScoreIsCorrect {
  all r: Round, m: Model |
```

### availability-parsing.als — Full Skeleton

```alloy
-- formal/alloy/availability-parsing.als
-- Handwritten — not generated from XState.
-- Source: bin/update-scoreboard.cjs (parseLocalDateTime, parseAvailabilityHint)
--
-- Models the availability hint date arithmetic:
--   parseLocalDateTime: parse "Feb 24 8:37 PM" -> Date; year rollover if past
--   parseAvailabilityHint: parse hint -> {available_at, reason} or null
--
-- Assertions:
--   ParseCorrect: parsed timestamp >= now
--   YearRolloverHandled: Dec->Jan crossing returns next-year date (not past date)
--   FallbackIsNull: unrecognized format returns null (no crash)
--
-- Scope: abstract month representation, integer timestamps

module availability_parsing

-- Months (mirrors MONTH_MAP in update-scoreboard.cjs)
abstract sig Month {}
one sig Jan, Feb, Mar, Apr, May, Jun, Jul, Aug, Sep, Oct, Nov, Dec extends Month {}

-- Relative month ordering (for year-rollover logic)
-- A month is "past" if its index < current month index
fun monthIndex [m: Month] : Int {
  (m = Jan  => 0  else
  (m = Feb  => 1  else
  (m = Mar  => 2  else
  (m = Apr  => 3  else
  (m = May  => 4  else
  (m = Jun  => 5  else
  (m = Jul  => 6  else
  (m = Aug  => 7  else
  (m = Sep  => 8  else
  (m = Oct  => 9  else
  (m = Nov  => 10 else
  11)))))))))))
}

-- Abstract parse input (subset of what parseLocalDateTime handles)
sig DateInput {
  month: one Month,
  day: one Int,      -- 1..31
  hour: one Int,     -- 0..23
  minute: one Int    -- 0..59
}

-- A parsed result: either a future timestamp (relative to now) or null (failed)
-- We model "now" as 0 and timestamps as relative integers (positive = future)
-- ParseResult.ts = none means null (unrecognized format or invalid input)
sig ParseResult {
  ts: lone Int  -- none = null; Int = future timestamp (>= 0 relative to now)
}

-- Current month (abstract: a single fixed Month for model checking)
one sig Now {
  currentMonth: one Month,
  currentDay: one Int     -- 1..31
}

-- ParseCorrect: if parsing succeeds, the result timestamp >= now (= 0 in our model)
assert ParseCorrect {
  all input: DateInput, result: ParseResult |
    ParsedCorrectly[input, result] =>
    (no result.ts or result.ts >= 0)
}

-- YearRolloverHandled: a month earlier in the year than now must produce next-year date
-- In our relative model: the result is STILL positive (in the future), not negative (past)
assert YearRolloverHandled {
  all input: DateInput, result: ParseResult |
    (monthIndex[input.month] < monthIndex[Now.currentMonth]) =>
    (ParsedCorrectly[input, result] => (no result.ts or result.ts >= 0))
}

-- FallbackIsNull: a DateInput with month = none (invalid) produces null result
-- (In Alloy, we model this as: if input has invalid day range, result has no ts)
assert FallbackIsNull {
  all result: ParseResult |
    (no result.ts) => FallbackReasonValid[result]
}

-- Predicate: result was produced by a valid parse of input
pred ParsedCorrectly [input: DateInput, result: ParseResult] {
  -- If the month/day combination is in the future (or same month, future day),
  -- ts is a positive integer. If in the past (year rollover), still positive.
  result.ts >= 0
}

pred FallbackReasonValid [result: ParseResult] {
  no result.ts  -- null result is always valid (no crash invariant)
}

check ParseCorrect          for 12 Month, 5 DateInput, 5 ParseResult, 6 Int
check YearRolloverHandled   for 12 Month, 5 DateInput, 5 ParseResult, 6 Int
check FallbackIsNull        for 12 Month, 5 DateInput, 5 ParseResult, 6 Int
```

```javascript
function recomputeStats(data) {
  // Reset all model stats
  for (const model of VALID_MODELS) {
    const m = data.models[model];
    m.score = 0; m.tp = 0; m.tn = 0; m.fp = 0; m.fn = 0; m.impr = 0;
  }
  // Replay all rounds
  for (const round of data.rounds) {
    const votes = round.votes || {};
    for (const model of VALID_MODELS) {
      const vote = votes[model];
      if (!vote || vote === '') continue;
      const m = data.models[model];
      const delta = SCORE_DELTAS[vote];
      if (delta === undefined) continue; // unknown vote code — skip
      m.score += delta;
      if (vote === 'TP' || vote === 'TP+')  m.tp   += 1;
      if (vote === 'TN' || vote === 'TN+')  m.tn   += 1;
      // ...etc.
    }
  }
}
```

- The key assertion for the Alloy model is that `computeScore` is a pure function of (model, rounds set) — no hidden state
- `NoVoteLoss`: if `round.votes[model]` is non-null and non-empty, `scoreDelta` is applied

## Source Analysis: GAP-9 (Availability Parsing)

### `parseLocalDateTime` — Lines 419-442 of `bin/update-scoreboard.cjs`

```javascript
function parseLocalDateTime(str) {
  str = str.trim();
  const m = str.match(/^(\w{3,9})\s+(\d{1,2})\s+(\d{1,2}):(\d{2})(?::(\d{2}))?\s*(AM|PM)?$/i);
  if (!m) return null;                          // FallbackIsNull: null on no-match

  const monthKey = m[1].slice(0, 3).toLowerCase();
  const month    = MONTH_MAP[monthKey];
  if (month === undefined) return null;          // FallbackIsNull: null on bad month

  const day    = parseInt(m[2], 10);
  let   hour   = parseInt(m[3], 10);
  const minute = parseInt(m[4], 10);
  const ampm   = (m[6] || '').toUpperCase();

  if (ampm === 'PM' && hour < 12) hour += 12;
  if (ampm === 'AM' && hour === 12) hour = 0;

  const now = new Date();
  let d = new Date(now.getFullYear(), month, day, hour, minute, 0, 0);
  if (d < now) d = new Date(now.getFullYear() + 1, month, day, hour, minute, 0, 0);
  // YearRolloverHandled: if d is past, advance year
  return d;                                       // ParseCorrect: returned date >= now
}
```

**Alloy modeling insights:**
- Null-on-no-match and null-on-bad-month: model as `no result.ts`
- Year rollover (`if d < now → year+1`): model as: `monthIndex[input.month] < currentMonthIndex → result.ts > 0` (relative to "now")
- `ParseCorrect` is the hardest assertion: the function guarantees the returned date is `>= now` (by construction — it advances year if needed)

**The year-rollover scenario:**
- Current date: Dec 26 2025
- Input message: "usage limit until Feb 24 8:37 PM"
- Computed date: Feb 24 2025 8:37 PM (in the PAST)
- Year-rollover fires: returns Feb 24 2026 8:37 PM (FUTURE)
- `ParseCorrect` holds: returned date >= now

**Alloy abstraction:** Model months as integers 0-11. "Past" = input month integer < current month integer (OR same month but input day < current day). Year rollover = result is still >= 0 (future).

## Open Questions

1. **Alloy `sum` operator behavior with `lone` vote fields**
   - What we know: `Round.votes: Model ->lone VoteCode` means each model has at most one vote per round. `sum r: rounds | scoreDelta[r.votes[m]]` uses Alloy's set sum comprehension, which sums over all rounds.
   - What's unclear: When `r.votes[m]` is empty (no vote), what does `scoreDelta[none]` return? Alloy treats `scoreDelta[none]` as the sum over an empty set = 0. This is correct behavior (UNAVAIL or absent vote = 0 delta).
   - Recommendation: Test with an explicit `run {} for 3 Round, 3 Model, 7 VoteCode, 7 Int` to confirm Alloy handles the empty-vote case correctly. If `scoreDelta[none]` throws, add an explicit guard: `no r.votes[m] => 0 else scoreDelta[r.votes[m]]`.

2. **Whether `availability-parsing.als` needs to model the relative time patterns**
   - What we know: `parseAvailabilityHint` handles three patterns: (1) "until Feb 24 8:37 PM" (absolute), (2) "in N hours" (relative), (3) "in N minutes" (relative). The Alloy model focuses on the absolute pattern (which has year-rollover behavior). The relative patterns (`"in N hours"`) simply add N hours to now — trivially always future.
   - What's unclear: Whether the planner should model all three patterns or just the absolute one.
   - Recommendation: Model only the absolute pattern (`parseLocalDateTime` + year rollover) in `availability-parsing.als`. The relative patterns ("in N hours") are trivially correct (Date.now() + N > Date.now() always). The interesting assertion is the year-rollover logic in `parseLocalDateTime`. Document this scope decision as a comment in the .als file header.

3. **Spec validation order in run-audit-alloy.cjs: before or after Java check?**
   - What we know: `run-oscillation-tlc.cjs` validates the config name BEFORE the Java check (lines 27-34). This makes test 4 (invalid spec → list valid specs) work without Java installed.
   - What's unclear: Whether to do the same for `run-audit-alloy.cjs` with `--spec` validation before the JAVA_HOME check.
   - Recommendation: Place `VALID_SPECS` check BEFORE the JAVA_HOME check, consistent with `run-oscillation-tlc.cjs`. This makes test 3 and test 4 reliable in environments without Java.

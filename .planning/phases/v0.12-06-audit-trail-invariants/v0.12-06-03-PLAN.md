Key differences from run-alloy.cjs (3 changes only):

1. `--spec` argument selector (replaces hardcoded quorum-votes.als path):
   ```javascript
   const VALID_SPECS = ['scoreboard-recompute', 'availability-parsing'];
   const args    = process.argv.slice(2);
   const specArg = args.find(a => a.startsWith('--spec=')) || null;
   const specName = specArg
     ? specArg.split('=')[1]
     : (args.find(a => !a.startsWith('-')) || 'scoreboard-recompute');

   if (!VALID_SPECS.includes(specName)) {
     process.stderr.write(
       '[run-audit-alloy] Unknown spec: ' + specName +
       '. Valid: ' + VALID_SPECS.join(', ') + '\n'
     );
     process.exit(1);
   }
   ```

2. Dynamic .als file path (replaces hardcoded formal/alloy/quorum-votes.als):
   ```javascript
   const alsPath = path.join(__dirname, '..', 'formal', 'alloy', specName + '.als');
   if (!fs.existsSync(alsPath)) {
     process.stderr.write(
       '[run-audit-alloy] ' + specName + '.als not found at: ' + alsPath + '\n' +
       '[run-audit-alloy] This file should exist in the repository. Check your git status.\n'
     );
     process.exit(1);
   }
   ```
   IMPORTANT: The .als existence check goes AFTER the JAVA_HOME check, Java version check, and JAR existence check — in that order. This is required for tests 1-4 to pass in the correct order (test 1 fires before JAR check; test 2 fires before .als check).

3. Dynamic specName in counterexample warning (NOT hardcoded assertion name — this is the ALY-02 fix):
   ```javascript
   if (/Counterexample/i.test(stdout)) {
     process.stderr.write(
       '[run-audit-alloy] WARNING: Counterexample found in ' + specName + '.als assertion.\n' +
       '[run-audit-alloy] This indicates a spec violation — review formal/alloy/' + specName + '.als.\n'
     );
     process.exit(1);
   }
   ```
   The existing run-alloy.cjs has "NoSpuriousApproval" hardcoded (line 113) — ALY-02 gap. Do NOT copy that hardcoded name. Use `specName` variable instead.

4. All log prefix strings use `[run-audit-alloy]` (not `[run-alloy]`)

Check order (mandatory — mirrors run-alloy.cjs, tests depend on it):
1. Parse --spec argument and validate against VALID_SPECS (exit 1 on invalid)
2. JAVA_HOME check: if set, verify binary exists (exit 1 if not found)
3. Java PATH fallback: if JAVA_HOME not set, probe `java --version` (exit 1 if not found)
4. Java version check: parse major version, require >= 17 (exit 1 if < 17)
5. JAR existence check: `fs.existsSync(jarPath)` on org.alloytools.alloy.dist.jar (exit 1 if missing)
6. .als file existence check: `fs.existsSync(alsPath)` (exit 1 if missing)
7. Alloy JAR invocation: `spawnSync(javaExe, ['-jar', jarPath, 'exec', '--output', '-', '--type', 'text', '--quiet', alsPath], ...)`
8. Counterexample detection: scan stdout for /Counterexample/i (exit 1 if found)

Alloy JAR invocation pattern (from run-alloy.cjs line 89 — copy exactly):
```javascript
const alloyResult = spawnSync(javaExe, [
  '-jar', jarPath,
  'exec',
  '--output', '-',
  '--type', 'text',
  '--quiet',
  alsPath,
], { encoding: 'utf8', stdio: 'pipe' });
```
Use `stdio: 'pipe'` (not 'inherit') so stdout can be scanned for counterexamples.
Alloy exits 0 even when counterexamples are found — stdout scan is the only detection mechanism.

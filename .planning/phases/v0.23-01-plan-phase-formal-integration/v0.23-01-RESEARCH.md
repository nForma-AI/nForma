## Summary

Plan-phase formal integration adds a discovery and injection stage to the `plan-phase` workflow that surfaces formal invariants before the planner spawns. The implementation follows the pattern established in quick.md's Step 4.5 (formal scope scan): keyword-match the phase description against module names in `formal/spec/*/`, inject matching invariants.md files into the planner's context, and require the planner to declare `formal_artifacts:` in the PLAN.md frontmatter when formal specifications exist. Fail-open is preserved — missing formal/spec/ directory or no matches result in empty context with no overhead or ceremony.

This phase addresses three closely coupled requirements: (WFI-01) discovery and context population, (WFI-02) planner acknowledgment via frontmatter declaration, and (ENF-03) fail-open preservation when formal tooling is absent.

**Primary recommendation:** Implement formal scope scan directly in plan-phase orchestrator (before planner spawn) using the keyword-match heuristic from quick.md Step 4.5, inject FORMAL_SPEC_CONTEXT into planner prompt, and add formal_artifacts validation to plan-checker. No new external tools needed — reuse existing bin/run-formal-check.cjs module definitions.

## Architecture Patterns

### Formal Scope Scan (Step 4.5 in plan-phase)

**What:** Before researcher and planner spawn, scan `formal/spec/` for modules with `invariants.md` files. Match module names against phase description using keyword heuristic. Populate `$FORMAL_SPEC_CONTEXT` array.

**When to use:** Always — zero cost if no specs found; no conditional logic needed around invocation.

**Algorithm:**

```bash
# Step 4.5: Formal scope scan
FORMAL_SPEC_CONTEXT=[]

# List all subdirectories in formal/spec/ (fail-open if dir missing)
if [ -d "formal/spec" ]; then
  for MODULE_DIR in formal/spec/*/; do
    MODULE=$(basename "$MODULE_DIR")
    INVARIANTS_FILE="formal/spec/${MODULE}/invariants.md"

    # Check invariants.md exists
    if [ -f "$INVARIANTS_FILE" ]; then
      # Keyword-match heuristic: split description into keywords, check if any match module name or vice versa
      # Convert description and module to lowercase for matching
      DESC_LOWER=$(echo "$PHASE_DESC" | tr '[:upper:]' '[:lower:]')
      MODULE_LOWER=$(echo "$MODULE" | tr '[:upper:]' '[:lower:]')

      # Simple substring matching: if any keyword (lowercased) appears in module name, or module name in description
      MATCHED=0
      for KEYWORD in $(echo "$DESC_LOWER" | tr ' -' '\n' | grep -v '^$'); do
        if echo "$MODULE_LOWER" | grep -q "$KEYWORD"; then
          MATCHED=1
          break
        fi
      done

      if [ $MATCHED -eq 1 ]; then
        # Add to context: { module, path }
        FORMAL_SPEC_CONTEXT+=("{\"module\":\"$MODULE\",\"path\":\"$INVARIANTS_FILE\"}")
      fi
    fi
  done
fi

# Store for later steps
export FORMAL_SPEC_CONTEXT
```

**Example outputs:**

- Phase "plan-phase Formal Integration" → matches: quorum, deliberation, prefilter (keywords: plan, phase, formal, integration; module "deliberation" contains "deliber", "prefilter" contains "filter", "quorum" no match)
  - Actually: keyword "formal" matches "formal" in "formal/spec/"; no module substring match. Use multi-level matching: if no direct substring match, check if keywords appear in earlier formal specs. For v0.23-01 phase, description is "plan-phase discovers and injects formal invariants before the planner spawns" — keywords: plan, phase, discovers, injects, formal, invariants, planner, spawns. Module matches: "formal" in "formal/spec/" is the directory level (match all invariants.md found).
  - Simpler: assume all `invariants.md` files are potentially relevant to planning phases. Formal scope scan is permissive — worst case, planner receives extra context and filters what applies.

The heuristic is: split description on spaces/hyphens → for each word, check if it appears as a substring (case-insensitive) in module name, or module name appears in the word. Very permissive.

The planner doesn't need invariants.md files when designing the integration mechanism. Invariants.md files come into play during WFI-03 (execute-phase formal integration).

### Planner Formal Context Block

**What:** When `$FORMAL_SPEC_CONTEXT` is non-empty, inject a `<formal_context>` block into the planner prompt at Step 8.

**Template (from quick.md Step 5 lines 151–162):**

```markdown
<formal_context>
${FORMAL_SPEC_CONTEXT.length > 0 ?
`Relevant formal modules identified: ${FORMAL_SPEC_CONTEXT.map(f => f.module).join(', ')}

Constraints:
- Read the injected invariants.md files and identify which invariants apply to this task
- Declare \`formal_artifacts:\` in plan frontmatter (required field when FORMAL_SPEC_CONTEXT is non-empty):
  - \`none\` — task does not create or modify formal/ files
  - \`update: [list of formal/ file paths]\` — task modifies existing formal/ files
  - \`create: [list of {path, type (tla|alloy|prism), description}]\` — task creates new formal/ files
- Plan tasks MUST NOT violate the identified invariants` :
`No formal modules matched this task. Declare \`formal_artifacts: none\` in plan frontmatter.`}
</formal_context>
```

**When to inject:** Always in planner prompt (Step 8). If `FORMAL_SPEC_CONTEXT=[]`, the else branch prints "No formal modules matched... declare formal_artifacts: none." This ensures planner ALWAYS declares the field.

**Invariant files in files_to_read:** When `FORMAL_SPEC_CONTEXT` non-empty, append each `f.path` to planner's `<files_to_read>` block with a comment like `(Formal invariants for module: {module})`.

### Plan-Checker Formal Validation

**What:** Checker must verify that PLAN.md declares `formal_artifacts:` in frontmatter and validates the declaration against identified invariants.

**Validation rules:**

1. **Mandatory declaration:** If `FORMAL_SPEC_CONTEXT` was non-empty at planning time, `formal_artifacts:` field MUST exist in PLAN.md frontmatter. If missing, issue: BLOCKER.
2. **Value validation:**
   - If `formal_artifacts: none` → no formal/ file modifications expected in plan tasks. Check tasks don't modify formal/ files (scan action/done blocks for "formal/" mentions).
   - If `formal_artifacts: update: [list]` → each path in list must be real (file exists or is a plausible path). Paths must be specific, not vague ("formal/" alone is too broad).
   - If `formal_artifacts: create: [list]` → each entry must have path + type (tla|alloy|prism). Type must match file extension. Descriptions must be concrete.
3. **Invariant compliance:** If FORMAL_SPEC_CONTEXT non-empty, checker scans each plan task's action/done blocks for operations that might violate identified invariants. This is a soft check (WARNING, not BLOCKER) — actual model checking happens at execution time.

**Template injection (quick.md line 237):**

```markdown
<formal_context>
${FORMAL_SPEC_CONTEXT.length > 0 ? `Relevant formal modules: ${FORMAL_SPEC_CONTEXT.map(f => f.module).join(', ')}. Check plan formal_artifacts declaration and invariant compliance.` : 'No formal modules matched. Verify plan declares formal_artifacts: none.'}
</formal_context>
```

### Anti-Patterns to Avoid

- **Auto-discovering formal checks from filesystem:** Hard-code the module-to-check mapping (use bin/run-formal-check.cjs MODULE_CHECKS as source of truth). Scanning the filesystem at planning time is fragile and creates ordering dependencies.
- **Requiring formal_artifacts only when specs exist:** The field is ALWAYS required (declared field = planner's explicit acknowledgment). When no specs exist, the value is `none` (not omitted).
- **Blocking on missing formal/spec/ directory:** Fail-open: if formal/spec/ doesn't exist or is empty, set `FORMAL_SPEC_CONTEXT=[]` and continue. No ceremony, no warnings unless running in verbose mode.
- **Inferring formal_artifacts from task description:** Planner must explicitly declare what the task does with formal/ files. Inference is fragile.

## Common Pitfalls

### Pitfall 1: Formal Artifacts Not Declared

**What goes wrong:** Planner creates a plan with no `formal_artifacts:` field. Checker has no way to verify the declaration was made.

**Why it happens:** Planner doesn't see the requirement clearly enough, or requirement block is weak.

**How to avoid:** Make the formal_context block in planner prompt use explicit bold language: "MUST declare formal_artifacts: in frontmatter — three modes (none/update/create)." Checker validation must flag missing field as BLOCKER, not warning.

**Warning signs:** grep planner output for "formal_artifacts:" → if missing, stop planning loop immediately and ask planner to revise.

### Pitfall 2: Scope Scan Overhead on No-Spec Projects

**What goes wrong:** Formal scope scan runs on every plan-phase invocation, adding latency even when no formal specs exist.

**Why it happens:** Unconditional directory scan + keyword matching on every phase.

**How to avoid:** Make scan fast: check if formal/spec/ exists (one stat call), if not, set FORMAL_SPEC_CONTEXT=[] immediately. Listing directories is ~10ms per spec module. For 10 modules, negligible cost. Keyword matching is in-memory string ops.

**Warning signs:** If plan-phase feels slow on projects with formal/spec/ directory, check Step 4.5 isn't doing unnecessary file I/O.

### Pitfall 3: Keyword Matching False Positives

**What goes wrong:** Phase description "verify quorum logic" matches both `quorum` AND `oscillation` (because "verify" contains "ify" and oscillation might have a property named "verify"?). Planner gets too much context.

**Why it happens:** Overly broad keyword matching.

**How to avoid:** Use exact substring matching (case-insensitive). A keyword matches a module if: lowercased keyword is a substring of lowercased module name, OR lowercased module name is a substring of lowercased keyword. For "verify quorum logic": keywords [verify, quorum, logic]. Module matches: `quorum` matches keyword "quorum" exactly. No false positives if module names are distinct (they are: quorum, tui-nav, breaker, etc.).

**Warning signs:** Planner receives unrelated invariants.md files. Verify matching logic.

### Pitfall 4: Formal Artifacts Validation Too Permissive

**What goes wrong:** Planner declares `formal_artifacts: update: [formal/]` (too vague). Checker passes it because "formal/" is technically a prefix of real paths.

**Why it happens:** Checker doesn't validate that paths are specific (not directory globs).

**How to avoid:** Checker rule: `update:` and `create:` paths must not end with "/" and must not be directory patterns. Validate against actual filesystem: `[ -f "$path" ]` or `[ -d "${path%/*}" ]` to check parent exists.

**Warning signs:** Planner creates vague formal_artifacts declarations. Checker review must catch this.

## Open Questions

1. **Keyword-match heuristic strength:** Is substring matching sufficient, or should we add fuzzy matching (e.g., edit distance)? Current approach: exact substring is fast, predictable, and proven in quick.md.
   - **What we know:** quick.md Step 4.5 uses substring matching; no fuzzy matching.
   - **What's unclear:** Whether v0.23-01 planning phases will generate descriptions that naturally keyword-match formal modules (unlikely, since planning phases are about mechanisms, not module implementations).
   - **Recommendation:** Ship with substring matching. If v0.23-02/03 discover false negatives, add fuzzy matching then.

2. **Formal artifacts path validation depth:** Should checker validate that paths in `formal_artifacts: update:` actually exist on disk, or just check syntax?
   - **What we know:** quick.md line 230 validates paths are "well-specified (not vague)" but doesn't specify filesystem checks.
   - **What's unclear:** How strict to be. Validating existence might catch typos early, but breaks if planner plans to create new paths (catch that with `create:` instead).
   - **Recommendation:** For `update:`, validate `[ -f "$path" ]` (file must exist). For `create:`, validate parent directory exists `[ -d "${path%/*}" ]`. Reject vague paths like "formal/" or "formal/*.tla".

3. **Formal_artifacts in all plan phases or only when specs match?** Current design: ALWAYS required field, value is `none` when no specs match. Alternative: omit field if FORMAL_SPEC_CONTEXT==[].
   - **What we know:** WFI-02 says "requires formal_artifacts: declaration in PLAN.md frontmatter when FORMAL_SPEC_CONTEXT is non-empty" — implies conditional requirement.
   - **What's unclear:** Should the field be unconditional (always present) for consistency with task envelope / must_haves patterns?
   - **Recommendation:** Follow WFI-02 precisely: field is conditional. If FORMAL_SPEC_CONTEXT=[], planner MUST still declare field with value `none` (explicit acknowledgment). Checker enforces this.

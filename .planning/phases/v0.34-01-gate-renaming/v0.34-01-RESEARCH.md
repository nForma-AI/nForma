# Phase v0.34-01: Gate Renaming - Research

**Researched:** 2026-03-11
**Domain:** Semantic gate naming and field migration in formal verification pipeline
**Confidence:** HIGH

## Summary

Phase v0.34-01 is a straightforward field renaming and display update for the nForma gate scoring system. The current system uses opaque labels (Gate A/B/C) with inconsistent field names (`grounding_score`, `gate_b_score`, `gate_c_score`) across files. The phase renames these gates to descriptive Wiring:Evidence/Purpose/Coverage with unified field naming (`wiring_score`), implementing backward-compatible migration so old files remain readable.

The work divides into four distinct scopes: (1) gate JSON files, (2) scripts that read/write gate scores, (3) display layers (TUI and dashboards), and (4) backward-compatibility migration logic. All existing infrastructure can be reused; no new libraries or patterns required.

**Primary recommendation:** Implement as a three-task sequence: JSON schema rename, script field migration (with fallback logic), then display layer updates. Backward compatibility is achieved by reading both old and new fields, preferring new when present.

## Architecture Patterns

### Current Gate System Structure

**Gate JSON files** (`.planning/formal/gates/`):
- `gate-a-grounding.json` — Grounding validation (L1→L2), uses `grounding_score`
- `gate-b-abstraction.json` — Abstraction coverage (L2→L3), uses `gate_b_score`
- `gate-c-validation.json` — Test validation (L3→TC), uses `gate_c_score`

Each file contains:
- A single numeric score field (current naming varies)
- Metadata: `schema_version`, `generated`, `target`, `target_met`, counts
- No version/compatibility fields currently

**Per-model gates** (`.planning/formal/gates/per-model-gates.json`):
- Per-model structure with nested `gate_a`, `gate_b`, `gate_c` objects
- Each gate has `pass`, `reason`, and related metadata
- Field names in this file (`gate_a`, `gate_b`, `gate_c`) remain unchanged (they're object keys, not score fields)

### Naming Convention: Gate → Wiring

The renaming maps semantic purpose:

| Old Label | New Label | Purpose |
|-----------|-----------|---------|
| Gate A | Wiring:Evidence | Grounding — model grounded in evidence/traces (L1→L2) |
| Gate B | Wiring:Purpose | Abstraction — operational semantics aligned with intent (L2→L3) |
| Gate C | Wiring:Coverage | Validation — test coverage confirms model behavior (L3→TC) |

**Field mapping** (score fields only):
- `grounding_score` → `wiring_evidence_score`
- `gate_b_score` → `wiring_purpose_score`
- `gate_c_score` → `wiring_coverage_score`

**Unified alias** (for cross-gate consumption):
- Generic `wiring_score` field acceptable when reading/writing aggregate scores or dashboard views

### Backward Compatibility Pattern

Old files are readable without errors. New files written after Phase v0.34-01 contain new field names. Migration occurs on next write:

```javascript
// Reading (preference order):
const score = data.wiring_evidence_score || data.grounding_score || 0;

// Writing (always new):
{ wiring_evidence_score: 0.95, ... }
```

Test fixtures MUST cover both old and new formats.

### Consumer Patterns

Scripts that consume gate scores follow this pattern:

1. **Load gate JSON** → Read score with fallback to old field name
2. **Use score** → Apply threshold logic (target comparison, decision trees)
3. **Write gate JSON** → Write with new field name always
4. **Log/display** → Use new descriptive labels in console output and TUI

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON schema versioning | Custom version logic | `schema_version` field + fallback chain | Proven pattern in this codebase (layer-manifest, traceability-matrix) |
| Field aliasing across files | Custom mapping objects | Inline fallback operators (`||`) in read paths | Simpler, more maintainable than centralized mappings |
| Migration state tracking | Custom migration registry | Always write new fields on each run | Codebase pattern; no explicit migration state files needed |

**Key insight:** This is a field rename, not a schema restructure. Fallback-on-read is simpler and more resilient than explicit migration states or transformation stages.

## Common Pitfalls

### Pitfall 1: Inconsistent Field Names Across Files

**What goes wrong:** One script writes `wiring_evidence_score` but another reads from the old JSON that has `grounding_score`, causing undefined/null reads.

**Why it happens:** Incremental script updates miss downstream consumers. Field name inconsistency is easy to overlook in grep/search workflows.

**How to avoid:**
- Test suite MUST check both old and new field names in fixtures
- Scripts tested in sequence: JSON producer → consumer → consumer

**Warning signs:**
- `NaN` or undefined comparisons in threshold logic
- Dashboard displays "–" instead of numeric scores
- Gate maturity promotion logic evaluates to false unexpectedly

### Pitfall 2: Displaying Old Labels in TUI After Script Update

**What goes wrong:** Console output still prints "Gate A", "Gate B", "Gate C" while JSON contains new field names, creating user confusion.

**Why it happens:** Display strings updated in one file, missed in another (convergence-report.cjs, solve-tui.cjs, cross-layer-dashboard.cjs).

**How to avoid:**
- Grep for all instances of "Gate A", "Gate B", "Gate C" before declaring done
- Update display labels in all three consumer files in one task
- Test output strings directly in test fixture

**Warning signs:**
- Inconsistent labeling in different TUI screens
- Reports show mixed "Gate A" and "Wiring:Evidence" terminology

### Pitfall 3: Schema Version Bump Not Handled

**What goes wrong:** Old tools still expect `schema_version: 1`, new tools expect `schema_version: 2`, leading to skipped files or error logs.

**Why it happens:** Schema version increments without gate files updating it, or gate files updated without script checks.

**How to avoid:**
- Bump schema version in each gate JSON file to a consistent new value (e.g., `2`)
- All reading scripts should accept both versions
- Document in file header why version changed

**Warning signs:**
- "Unknown schema version" warnings in compute-per-model-gates.cjs
- Per-model-gates.json not regenerated correctly

### Pitfall 4: Per-Model Gates Structure Confusion

**What goes wrong:** Confusing the nested `gate_a`, `gate_b`, `gate_c` object keys in per-model-gates.json with the score field names. Developer tries to rename the keys themselves.

**Why it happens:** Object key names are similar to field names; easy to conflate them.

**How to avoid:**
- Clarify: Object keys (`gate_a`, `gate_b`, `gate_c`) stay unchanged
- Only score fields inside those objects are renamed (none currently; that's aggregated scoring's concern)
- Document this distinction explicitly in code comments

**Warning signs:**
- JSON parsing errors when trying to rename object keys
- Per-model-gates.json structure breaking in consumers

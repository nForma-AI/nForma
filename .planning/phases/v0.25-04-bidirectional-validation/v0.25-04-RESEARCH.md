# Phase v0.25-04: Bidirectional Validation — Research

## Summary

Phase v0.25-04 adds two capabilities to the existing traceability infrastructure: (1) asymmetric link detection between model-registry and requirements.json, and (2) a CI coverage guard that catches coverage regressions. Both build directly on the generate-traceability-matrix.cjs output from v0.25-03.

## Requirements Analysis

### TRACE-04: Bidirectional Validation

**Requirement text:** Bidirectional validation detects asymmetric links (model claims requirement X but requirement X does not claim that model) and emits warnings.

**Success criterion:** Running the traceability matrix generator on a codebase where model A claims requirement X but requirement X does not list model A produces a warning identifying the asymmetric link with both sides of the mismatch.

**What exists today:**
- `formal/model-registry.json` has 24 model entries, 18 with `requirements` arrays mapping model -> requirement IDs
- `formal/requirements.json` has 211 requirements, 56 with `formal_models` arrays mapping requirement -> model file paths
- `bin/generate-traceability-matrix.cjs` already reads both sources but does NOT cross-validate them for consistency
- The traceability matrix uses annotations as primary source and model-registry as fallback — it does not check whether requirements.json `formal_models` agree with model-registry `requirements`

**Asymmetric link types to detect:**
1. **Model claims requirement, requirement does not claim model:** `model-registry.json` entry for `model-A.tla` lists `REQ-X` in its `requirements` array, but `requirements.json` entry for `REQ-X` does NOT list `model-A.tla` in its `formal_models` array
2. **Requirement claims model, model does not claim requirement:** `requirements.json` entry for `REQ-X` lists `model-A.tla` in its `formal_models` array, but `model-registry.json` entry for `model-A.tla` does NOT list `REQ-X` in its `requirements` array
3. **Stale links:** Either file references a model file that does not exist on disk, or a requirement ID that does not exist in requirements.json

**Implementation approach:**
- Add a `validate-bidirectional` step to generate-traceability-matrix.cjs (or create a separate `bin/validate-bidirectional-links.cjs` script)
- Cross-reference model-registry `requirements` arrays against requirements.json `formal_models` arrays
- Emit warnings to stderr (consistent with existing warning patterns in the generator)
- Include validation results in the traceability matrix output as a `bidirectional_validation` section
- Exit code remains 0 (validation is informational, matching the existing non-gate pattern)

**Decision: Extend generate-traceability-matrix.cjs vs. new script?**
- The success criterion says "Running the traceability matrix generator" — this implies the validation should be integrated into the existing generator
- Adding it to generate-traceability-matrix.cjs keeps the single-command experience and ensures validation runs every time the matrix is generated
- A `--validate` flag could control whether validation runs (default: on)

### TRACE-05: CI Coverage Guard

**Requirement text:** CI guard warns when formal coverage percentage drops below a configurable threshold (default 15%) compared to the previous traceability matrix.

**Success criterion:** Running the CI coverage guard after removing a model-to-requirement link produces a warning when formal coverage percentage drops below the configured threshold (default 15%) compared to the previous matrix snapshot.

**What exists today:**
- `formal/traceability-matrix.json` is generated with `coverage_summary.coverage_percentage` (currently 27.0%)
- No snapshot/baseline mechanism exists
- No CI guard script exists
- The matrix is regenerated by `run-formal-verify.cjs` in Phase 3 (after all checks)

**Implementation approach:**
- Create `bin/check-coverage-guard.cjs` — a standalone CI guard script
- The guard reads the current traceability matrix and compares its coverage percentage against a baseline
- **Baseline mechanism:** Use `formal/traceability-matrix.baseline.json` — a snapshot of the last "known good" matrix. The baseline is committed to git, so CI can compare against it.
- **Threshold:** Compare `current_coverage_percentage` vs `baseline_coverage_percentage`. If the drop exceeds the configurable threshold (default: 15 percentage points), emit a warning.
- **"drops below threshold"** interpretation: The requirement says "drops below a configurable threshold (default 15%) compared to the previous matrix." This means a RELATIVE drop — if baseline is 27% and current is 11%, the drop is 16 percentage points, which exceeds 15%, so it warns. If current drops to 13%, the drop is 14pp, no warning.
- **Alternative interpretation:** "below 15% absolute." But the success criterion says "compared to the previous matrix snapshot," which implies relative comparison. And the threshold of 15% as an absolute floor would already be met (current coverage is 27%). The more useful interpretation is: warn if coverage percentage drops by more than the threshold compared to baseline.

Wait — re-reading the success criterion more carefully: "produces a warning when formal coverage percentage drops below the configured threshold (default 15%) compared to the previous traceability matrix." This reads as: the drop itself must be >= threshold. So if baseline has 27% coverage and current has 11%, the drop is 16 percentage points > 15pp threshold. Warning fires.

**Snapshot workflow:**
1. `node bin/check-coverage-guard.cjs --save-baseline` — saves current matrix as baseline
2. `node bin/check-coverage-guard.cjs` — compares current matrix against baseline, warns if drop > threshold
3. `node bin/check-coverage-guard.cjs --threshold 10` — override default 15% threshold
4. Wire into run-formal-verify.cjs as a Phase 3 step (after matrix generation)

## Architecture Analysis

### Data Flow

```
model-registry.json ──┐
                       ├──> generate-traceability-matrix.cjs ──> traceability-matrix.json
requirements.json ─────┤                                              │
                       │                                              ├──> bidirectional_validation section (TRACE-04)
annotations ───────────┘                                              │
                                                                      v
                                                        check-coverage-guard.cjs
                                                              │
                                                     traceability-matrix.baseline.json
                                                              │
                                                     "Coverage dropped by X% (threshold: 15%)"
```

### Integration Points

1. **generate-traceability-matrix.cjs** — Add bidirectional validation logic (TRACE-04)
   - New function: `validateBidirectionalLinks(registry, requirements)`
   - Returns `{ asymmetric_links: [...], stale_links: [...] }`
   - Results added to matrix output as `bidirectional_validation` section
   - Warnings emitted to stderr

2. **bin/check-coverage-guard.cjs** — New script (TRACE-05)
   - Reads `formal/traceability-matrix.json` (current)
   - Reads `formal/traceability-matrix.baseline.json` (baseline)
   - Compares coverage percentages
   - Exit code: 0 (with warnings) or configurable exit code for CI

3. **bin/run-formal-verify.cjs** — Wire coverage guard as Phase 3 step
   - Add after the traceability matrix generation step
   - `nonCritical: true` to match warning-not-gate pattern

## Risks

1. **Asymmetric links may be numerous** — Initial seeding in v0.25-01 may have left inconsistencies between model-registry and requirements.json. The first run may produce many warnings. This is expected and correct behavior — the tool detects existing problems.

2. **Baseline file management** — The baseline must be committed to git so CI can compare. If someone forgets to update the baseline after intentional coverage changes, false positives occur. Mitigation: clear error message explaining how to update baseline.

3. **Coverage percentage precision** — Current implementation rounds to 1 decimal. Ensure threshold comparison uses the same precision to avoid edge-case false positives.

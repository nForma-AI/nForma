Add a `validateBidirectionalLinks(registry, requirements)` function to `bin/generate-traceability-matrix.cjs` that cross-validates model-registry `requirements` arrays against requirements.json `formal_models` arrays. Call this function in `buildMatrix()` after all data is loaded and include results in the output.

**Function: `validateBidirectionalLinks(registry, requirements)`**

Parameters:
- `registry` — parsed model-registry.json object (has `.models` with `{ [file]: { requirements: string[] } }`)
- `requirements` — array of requirement objects from requirements.json (each has `{ id, formal_models?: string[] }`)

Returns:
```javascript
{
  asymmetric_links: [
    {
      model_file: "formal/alloy/foo.als",
      requirement_id: "REQ-01",
      direction: "model_claims_requirement",  // or "requirement_claims_model"
      detail: "model-registry lists REQ-01 for formal/alloy/foo.als but requirements.json REQ-01 does not list formal/alloy/foo.als in formal_models"
    }
  ],
  stale_links: [
    {
      type: "missing_model_file",  // or "unknown_requirement_id"
      reference: "formal/tla/nonexistent.tla",
      referenced_by: "requirements.json REQ-05 formal_models"
    }
  ],
  summary: {
    total_checked: N,     // number of unique model-requirement pairs checked
    asymmetric_count: N,
    stale_count: N,
    clean: true/false     // true if no issues found
  }
}
```

**Logic:**

1. Build a map of `model_file -> Set<requirement_id>` from model-registry `requirements` arrays.

2. Build a map of `requirement_id -> Set<model_file>` from requirements.json `formal_models` arrays.

3. **Check direction 1 (model claims requirement):** For each model file in model-registry that has `requirements`, for each requirement ID in its array:
   - Look up that requirement in the requirements.json map
   - If the requirement exists but does NOT list this model file in its `formal_models`, record an asymmetric link with `direction: "model_claims_requirement"`
   - If the requirement ID does not exist in requirements.json at all, record a stale link with `type: "unknown_requirement_id"`

4. **Check direction 2 (requirement claims model):** For each requirement in requirements.json that has `formal_models`, for each model file path in its array:
   - Look up that model file in the model-registry map
   - If the model file exists in registry but does NOT list this requirement ID in its `requirements`, record an asymmetric link with `direction: "requirement_claims_model"`
   - If the model file does not exist in model-registry at all, check if the file exists on disk. If not on disk, record a stale link with `type: "missing_model_file"`. If on disk but not in registry, record an asymmetric link (the model exists but registry has no entry or no requirement link for it).

5. **Deduplicate:** An asymmetric pair (model A, req X) should only appear once even if detected from both directions. Track checked pairs to avoid duplicates. When the same pair would be detected in both directions (model does not claim req AND req does not claim model), record it once as `direction: "neither_claims_other"` — but this should be rare since it implies a completely orphaned reference.

**Integration into buildMatrix():**

After building the matrix but before returning, call `validateBidirectionalLinks(registry, requirements)` and add results to the matrix:

```javascript
matrix.bidirectional_validation = validateBidirectionalLinks(registry, requirements);
```

**Warnings to stderr:**

For each asymmetric link, write to stderr:
```
[generate-traceability-matrix] warn: asymmetric link — model-registry lists REQ-01 for formal/alloy/foo.als but requirements.json REQ-01 does not list formal/alloy/foo.als in formal_models
```

For each stale link:
```
[generate-traceability-matrix] warn: stale link — requirements.json REQ-05 references formal/tla/nonexistent.tla but file does not exist
```

**Summary output (when not --quiet):**

Add to the existing summary output:
```
[generate-traceability-matrix]   Bidirectional validation: N asymmetric, N stale (N pairs checked)
```

**Do NOT change exit code** — the generator still exits 0. Validation is informational.

**Important implementation notes:**
- Use `fs.existsSync()` for disk checks (already imported)
- Model file paths in model-registry are relative to project root (e.g., `formal/alloy/foo.als`). Resolve with `path.join(ROOT, modelFile)` for disk checks.
- Requirements.json `formal_models` arrays also use relative paths. Compare strings directly (no normalization needed since both sources use the same path format).
- Skip requirements that have no `formal_models` field (many requirements are non-formal and don't have this field).
- Skip model-registry entries that have no `requirements` array or empty `requirements`.

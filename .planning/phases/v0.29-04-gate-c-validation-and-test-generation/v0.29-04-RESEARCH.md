## Summary

Phase v0.29-04 closes the three-layer formal verification loop by: (1) building Gate C to verify every L3 hazard/failure-mode maps to a concrete test scenario, (2) generating executable test recipe JSON from L3 failure modes, (3) implementing design impact analysis that traces git diffs through all three layers, and (4) integrating three new layer-transition sweeps (L1->L2, L2->L3, L3->TC) into nf-solve and three new gate check step groups into run-formal-verify.

The L3 foundation from v0.29-03 is complete and well-structured: 16 hazards in hazard-model.json, 32 failure modes in failure-mode-catalog.json, 16 ranked transitions in risk-heatmap.json, and Gate B passing at 1.0 (64/64 grounded, 0 orphans). The existing test recipe infrastructure from quick-182 provides `.stub.recipe.json` sidecars in formal-test-sync.cjs, but those recipes are R->F-scoped (requirements to formal properties). This phase needs a distinct L3->TC recipe generator that transforms failure modes (omission/commission/corruption per state-event pair) into test scenarios with input sequences, expected outcomes, and oracles.

The nf-solve convergence loop currently has 8 forward sweeps (R->F, F->T, C->F, T->C, F->C, R->D, D->C, P->F) and 3 reverse sweeps (C->R, T->R, D->R). Adding L1->L2, L2->L3, and L3->TC sweeps extends it to 11 forward + 3 reverse. The run-formal-verify pipeline has 3 execution phases (generate, tool groups, post-processing/traceability) with `--only=` filtering. Adding gate check step groups (gate-a, gate-b, gate-c) requires a new tool group category.

**Primary recommendation:** Build in three plans: (1) Gate C validation + test recipe generation (RSN-04, GATE-03), (2) design impact analysis (RSN-05), (3) nf-solve and run-formal-verify three-layer integration (INTG-02, INTG-03).

### Pattern 2: Test Recipe Generation Logic
**What:** Transform each failure mode type into a specific test scenario shape.
**When to use:** test-recipe-gen.cjs core logic.
**Algorithm:**
```
For each failure_mode in failure-mode-catalog.json:
  OMISSION:
    setup = { initial_state: mode.state }
    input_sequence = [{ event: mode.event }]
    expected_outcome = { final_state: mode.to_state }
    oracle = state_assertion (verify transition fires)

  COMMISSION:
    setup = { initial_state: mode.state }
    input_sequence = [{ event: mode.event }]
    expected_outcome = { rejection: true }
    oracle = guard_rejection (verify event is rejected by XState guard)

  CORRUPTION:
    setup = { initial_state: mode.state }
    input_sequence = [{ event: mode.event }]
    expected_outcome = { final_state: mode.to_state, not_state: "any other" }
    oracle = state_equality (verify exact target state, not wrong state)

Enrich each recipe with:
  - risk_context from hazard-model.json (rpn, risk_tier from risk-heatmap.json)
  - derived_from links to L3 source artifacts
```

### Pattern 3: Gate C Validation Algorithm
**What:** Gate C verifies every L3 finding maps to at least one test recipe.
**When to use:** gate-c-validation.cjs.
**Algorithm:**
```
1. Load L3 artifacts: hazard-model.json (hazards array), failure-mode-catalog.json (failure_modes array)
2. Load test-recipes.json (recipes array)
3. Build lookup: recipe.failure_mode_id -> recipe[]
4. For each L3 entry (hazard or failure mode):
   a. Check if at least one test recipe references this entry
   b. Mark as "validated" (has recipe) or "unvalidated" (no recipe)
5. gate_c_score = validated_entries / total_entries
6. Report unvalidated entries as gaps
7. Emit check-result to NDJSON
```

### Pattern 4: Design Impact Analysis (RSN-05)
**What:** Given a git diff, trace through all three layers to show what changes at each level.
**When to use:** design-impact.cjs.
**Algorithm:**
```
Input: git diff (from stdin or --diff=<ref1>..<ref2>)
Output: impact report JSON

1. Parse git diff to extract changed files and changed line ranges
2. L1 (Instrumentation) impact:
   - Load instrumentation-map.json
   - For each emission_point where emission_point.file is in changed_files:
     - If line_number is in changed_line_range: mark as "directly affected"
     - Else if file is changed: mark as "potentially affected"
   - Output: list of affected emission points with actions and xstate_events
3. L2 (State) impact:
   - Load observed-fsm.json
   - For each affected emission_point with an xstate_event:
     - Find all transitions in observed_transitions involving that event
   - Output: list of affected state transitions
4. L3 (Hazard) impact:
   - Load hazard-model.json and failure-mode-catalog.json
   - For each affected transition (state, event):
     - Find matching hazard entries and failure mode entries
   - Output: list of affected hazards with RPN and risk tier
5. Summary: { l1_affected: N, l2_affected: N, l3_affected: N }
```

### Pattern 5: nf-solve Layer-Transition Sweeps (INTG-02)
**What:** Three new sweep functions added to computeResidual() in nf-solve.cjs.
**When to use:** INTG-02 implementation.
**Sweep definitions:**
```
sweepL1toL2():
  - Run gate-a-grounding.cjs --json
  - Parse gate_a output: grounding_score, unexplained count
  - residual = unexplained traces (instrumentation_bug + model_gap + genuine_violation)
  - detail: { grounding_score, target: 0.8, gap: target - score, unexplained_breakdown }

sweepL2toL3():
  - Run gate-b-abstraction.cjs --json
  - Parse gate_b output: gate_b_score, orphaned count
  - Additionally: count L2 state-event pairs NOT covered by any L3 hazard

  - detail: { gate_b_score, uncovered_pairs, orphaned_entries }

sweepL3toTC():
  - Run gate-c-validation.cjs --json
  - Parse gate_c output: gate_c_score, unvalidated count
  - residual = unvalidated L3 entries (failure modes without test recipes)
  - detail: { gate_c_score, unvalidated_count, total_failure_modes, total_recipes }
```

### Anti-Patterns to Avoid
- **Conflating existing recipe infrastructure with L3 test recipes:** The quick-182 `.stub.recipe.json` files are R->F scoped (requirement to formal property). The new test-recipes.json is L3->TC scoped (failure mode to test scenario). These are complementary, not competing systems. Do not merge them.
- **Running gate scripts inline in nf-solve:** The sweeps should spawn gate scripts as child processes (matching existing spawnTool pattern), not import gate functions directly. This avoids module coupling and ensures gate scripts remain independently runnable.
- **Design impact analysis with full git diff parsing:** Only parse `---`/`+++`/`@@` lines from git diff output. Do not attempt to interpret patch content semantics. The goal is "which files/lines changed," not "what the change does."
- **Gate C targeting 100% from day one:** Unlike Gate B (which checks artifacts we fully control), Gate C maps to test infrastructure which may not cover all failure modes immediately. Start with ADVISORY maturity (score computed, not enforced). Target >= 80% initially, matching Gate A's approach.
- **Hardcoding L2 state-event pairs in sweepL2toL3:** The sweep should dynamically load observed-fsm.json and hazard-model.json to compute coverage. Hardcoded lists would break when the FSM changes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Git diff parsing | Full diff parser library | `spawnSync('git', ['diff', '--name-only'])` + `git diff --unified=0` for line ranges | Only need file names and line ranges, not patch semantics |
| Gate score computation | Custom scoring in nf-solve | Spawn existing gate scripts with --json | Gate scripts already compute scores; sweeps just consume the output |
| Check result emission | Custom NDJSON writer | `write-check-result.cjs` (existing) | Already validates formalism/result enums |
| Layer manifest updates | Manual JSON editing | `build-layer-manifest.cjs` (v0.29-01) | Already handles classification + schema |
| L2 link resolution | Custom path navigator | Gate B's `resolveL2Link()` pattern | Already tested and handles all reference formats |

**Key insight:** Gate C is structurally simpler than Gates A and B. Gate A requires trace replay logic. Gate B requires JSON path resolution. Gate C just checks if a test recipe exists for each failure mode ID -- a set membership test. The complexity is in generating good test recipes, not in the gate check itself.

## Common Pitfalls

### Pitfall 1: Test Recipes Without Actionable Input Sequences
**What goes wrong:** Recipes say "test that QUORUM_START fires" but do not specify how to set up the initial state or what concrete inputs to provide.
**Why it happens:** Failure modes are abstract (omission/commission/corruption); recipes must be concrete.
**How to avoid:** Every recipe MUST have: (a) initial_state from the failure mode's `state` field, (b) at least one event in `input_sequence` from the failure mode's `event` field, (c) expected_outcome with a concrete `final_state` from the failure mode's `to_state` field. The oracle must specify the assertion type (state_assertion, guard_rejection, state_equality).
**Warning signs:** Recipes with empty `input_sequence` or vague `expected_outcome`.

### Pitfall 2: Design Impact Over-Reporting
**What goes wrong:** Every changed file triggers L1/L2/L3 impacts even when the change is unrelated to instrumented code (e.g., docs, tests, configs).
**Why it happens:** instrumentation-map.json only covers hook files (nf-prompt.js, nf-stop.js, nf-circuit-breaker.js). Most files in the repo are NOT instrumented.
**How to avoid:** Only report L1 impact for files that appear in instrumentation-map.json emission_points. Files not in the map produce zero L1 impact, which cascades to zero L2/L3 impact for that file. Report "no instrumented files affected" explicitly.
**Warning signs:** A docs-only commit showing L3 hazard impacts.

### Pitfall 3: nf-solve Sweep Timeout
**What goes wrong:** Gate scripts (especially Gate A with trace replay) take >2s, and running 3 gate scripts per iteration of the convergence loop adds 6-10s per iteration.
**Why it happens:** Gate A loads and replays 35825 conformance events.
**How to avoid:** Use `--json` flag on gate scripts (skips human-readable formatting). Cache gate results within a single nf-solve iteration (gate scores do not change between sweeps within one iteration). Consider a `--fast` mode skip for layer sweeps (matching existing `--fast` for T->C and F->C).
**Warning signs:** nf-solve wall-clock time doubles after adding layer sweeps.

### Pitfall 4: Circular Dependency Between Gate C and Test Recipes
**What goes wrong:** Gate C checks if recipes exist, but recipes are generated by test-recipe-gen.cjs. If the build order is wrong, Gate C always fails on first run.
**Why it happens:** Gate C depends on test-recipes.json which must be generated first.
**How to avoid:** In run-formal-verify, place test-recipe-gen.cjs BEFORE gate-c-validation.cjs in the gate step group. In nf-solve, sweepL3toTC should first check if test-recipes.json exists and is current (compare timestamps with failure-mode-catalog.json). If stale, regenerate before scoring.
**Warning signs:** Gate C always returns 0% on fresh runs.

### Pitfall 5: Three-Layer Sweep Residual Inflation
**What goes wrong:** Adding L1->L2, L2->L3, L3->TC residuals to nf-solve's total makes the total residual jump dramatically, making convergence harder to achieve.
**Why it happens:** Gate A has 6369 unexplained events (instrumentation_bug), which would become a huge L1->L2 residual.
**How to avoid:** Layer sweeps should use normalized scores (gate scores as 0-1 fractions), not raw counts. Residual = ceiling of (1 - gate_score) * 10 to get a 0-10 residual per layer sweep, matching the scale of existing sweeps. Alternatively, make layer sweeps "informational" (like reverse sweeps) -- included in output but not added to the automatable `total`.
**Warning signs:** nf-solve `total` jumps from ~15 to ~6400 after adding layer sweeps.

```javascript
// Source: derived from failure-mode-catalog.json structure
function generateRecipe(failureMode, hazardModel, riskHeatmap) {
  const hazard = hazardModel.hazards.find(h =>
    h.state === failureMode.state && h.event === failureMode.event
  );
  const heatmapEntry = riskHeatmap.transitions.find(t =>
    t.state === failureMode.state && t.event === failureMode.event
  );

  const recipe = {
    id: 'TR-' + failureMode.id,
    failure_mode_id: failureMode.id,
    title: `${capitalize(failureMode.failure_mode)}: ${failureMode.event} in ${failureMode.state}`,
    setup: {
      initial_state: failureMode.state,
      preconditions: buildPreconditions(failureMode),
    },
    input_sequence: buildInputSequence(failureMode),
    expected_outcome: buildExpectedOutcome(failureMode),
    oracle: buildOracle(failureMode),
    risk_context: {
      severity_class: failureMode.severity_class,
      rpn: hazard ? hazard.rpn : null,
      risk_tier: heatmapEntry ? heatmapEntry.risk_tier : null,
    },
    derived_from: [
      { layer: 'L3', artifact: 'reasoning/failure-mode-catalog.json',
        ref: `failure_modes[id=${failureMode.id}]` },
    ],
  };

  if (hazard) {
    recipe.derived_from.push({
      layer: 'L3', artifact: 'reasoning/hazard-model.json',
      ref: `hazards[id=${hazard.id}]`
    });
  }

  return recipe;
}

function buildInputSequence(fm) {
  return [{ event: fm.event, description: `Send ${fm.event} to machine in ${fm.state} state` }];
}

function buildExpectedOutcome(fm) {
  switch (fm.failure_mode) {
    case 'omission':
      return { final_state: fm.to_state, assertions: [`State transitions from ${fm.state} to ${fm.to_state}`] };
    case 'commission':
      return { rejection: true, assertions: [`Event ${fm.event} is rejected by guard in ${fm.state}`] };
    case 'corruption':
      return { final_state: fm.to_state, not_state: 'any_other',
               assertions: [`State is exactly ${fm.to_state}, not any other state`] };
  }
}

function buildOracle(fm) {
  switch (fm.failure_mode) {
    case 'omission':
      return { type: 'state_assertion', check: `machine.state === '${fm.to_state}'`,
               failure_indicates: 'Omission: transition did not fire' };
    case 'commission':
      return { type: 'guard_rejection', check: `machine.state === '${fm.state}'`,
               failure_indicates: 'Commission: unmodeled event was accepted' };
    case 'corruption':
      return { type: 'state_equality', check: `machine.state === '${fm.to_state}'`,
               failure_indicates: 'Corruption: wrong target state reached' };
  }
}
```

### Gate C Validation Check
```javascript
// Source: derived from gate-b-abstraction.cjs pattern
function computeGateC(failureModeCatalog, hazardModel, testRecipes) {
  const recipesByFmId = new Map();
  for (const recipe of testRecipes.recipes) {
    if (!recipesByFmId.has(recipe.failure_mode_id)) {
      recipesByFmId.set(recipe.failure_mode_id, []);
    }
    recipesByFmId.get(recipe.failure_mode_id).push(recipe);
  }

  let totalEntries = 0;
  let validatedEntries = 0;
  const gaps = [];

  for (const fm of failureModeCatalog.failure_modes) {
    totalEntries++;
    if (recipesByFmId.has(fm.id)) {
      validatedEntries++;
    } else {
      gaps.push({ failure_mode_id: fm.id, state: fm.state, event: fm.event,
                  failure_mode: fm.failure_mode, severity_class: fm.severity_class });
    }
  }

  return {
    schema_version: '1',
    generated: new Date().toISOString(),
    gate_c_score: totalEntries > 0 ? validatedEntries / totalEntries : 0,
    total_entries: totalEntries,
    validated_entries: validatedEntries,
    unvalidated_entries: gaps.length,
    gaps,
    target: 0.8,
    target_met: gaps.length === 0 || (validatedEntries / totalEntries) >= 0.8,
  };
}
```

### Design Impact Analysis
```javascript
// Source: derived from instrumentation-map.json + observed-fsm.json structure
function analyzeImpact(changedFiles, changedLineRanges, instrMap, observedFsm, hazardModel) {
  // L1: find affected emission points
  const l1Impact = [];
  for (const ep of instrMap.emission_points) {
    if (!changedFiles.includes(ep.file)) continue;
    const lineRanges = changedLineRanges[ep.file] || [];
    const directHit = lineRanges.some(([start, end]) =>
      ep.line_number >= start && ep.line_number <= end);
    l1Impact.push({
      ...ep,
      impact_type: directHit ? 'direct' : 'file_level',
    });
  }

  // L2: find affected state transitions
  const affectedEvents = new Set(l1Impact.map(ep => ep.xstate_event).filter(Boolean));
  const l2Impact = [];
  for (const [fromState, events] of Object.entries(observedFsm.observed_transitions)) {
    for (const [event, trans] of Object.entries(events)) {
      if (affectedEvents.has(event)) {
        l2Impact.push({ state: fromState, event, to_state: trans.to_state, count: trans.count });
      }
    }
  }

  // L3: find affected hazards
  const l3Impact = [];
  for (const l2 of l2Impact) {
    const hazard = hazardModel.hazards.find(h =>
      h.state === l2.state && h.event === l2.event);
    if (hazard) {
      l3Impact.push({ hazard_id: hazard.id, rpn: hazard.rpn, severity: hazard.severity });
    }
  }

  return {
    schema_version: '1',
    generated: new Date().toISOString(),
    changed_files: changedFiles.length,
    l1_impact: { affected_emission_points: l1Impact.length, direct_hits: l1Impact.filter(e => e.impact_type === 'direct').length, details: l1Impact },
    l2_impact: { affected_transitions: l2Impact.length, affected_events: [...affectedEvents], details: l2Impact },
    l3_impact: { affected_hazards: l3Impact.length, max_rpn: l3Impact.length > 0 ? Math.max(...l3Impact.map(h => h.rpn)) : 0, details: l3Impact },
    summary: `${l1Impact.length} emission points -> ${l2Impact.length} transitions -> ${l3Impact.length} hazards affected`,
  };
}
```

```javascript
// Source: derived from existing sweep pattern in nf-solve.cjs
function sweepL3toTC() {
  const result = spawnTool('bin/gate-c-validation.cjs', ['--json']);

  if (!result.ok) {
    return { residual: -1, detail: { error: true, stderr: result.stderr.slice(0, 500) } };
  }

  let gateC;
  try { gateC = JSON.parse(result.stdout); } catch (_) {
    return { residual: -1, detail: { error: true, reason: 'invalid JSON from gate-c' } };
  }

  // Normalized residual: 0 = all validated, 10 = none validated
  const normalizedResidual = Math.ceil((1 - gateC.gate_c_score) * 10);

  return {
    residual: normalizedResidual,
    detail: {
      gate_c_score: gateC.gate_c_score,
      target: gateC.target,
      target_met: gateC.target_met,
      validated: gateC.validated_entries,
      total: gateC.total_entries,
      gaps: gateC.unvalidated_entries,
    },
  };
}
```

## Open Questions

1. **Should layer sweeps add to nf-solve total residual or be informational?**
   - What we know: Gate A has 6369 unexplained events. Raw counts would inflate the total dramatically.
   - What's unclear: Whether normalized residuals (0-10 scale) are appropriate, or whether layer sweeps should be excluded from `total` like reverse sweeps.
   - Recommendation: Use normalized 0-10 residuals. Add to a separate `layer_total` field alongside the existing `total`. The existing `total` field should remain unchanged for backward compatibility. The `health` indicator should factor in `layer_total`. This gives visibility without breaking existing convergence behavior.

2. **Should test-recipe-gen.cjs also generate executable .test.cjs stubs?**
   - What we know: The quick-182 recipe infrastructure generates .stub.recipe.json sidecars. The existing formal-test-sync.cjs generates .stub.test.js files.
   - What's unclear: Whether test-recipes.json should be consumed by executors to produce real test files, or remain as a validation artifact for Gate C.
   - Recommendation: For v0.29-04, generate only test-recipes.json (Gate C validation artifact). Executable test generation is a follow-up optimization. The recipes contain enough detail (setup, input_sequence, expected_outcome, oracle) that an executor can consume them later. This keeps the scope contained.

3. **Should design-impact.cjs support --watch mode?**
   - What we know: run-formal-verify.cjs has --watch mode. design-impact.cjs analyzes diffs.
   - What's unclear: Whether continuous impact analysis on file changes would be useful.
   - Recommendation: No. Design impact is on-demand (triggered by a specific diff). --watch would require re-running git diff continuously which is wasteful. Support `--diff=HEAD~1..HEAD` (default), `--diff=<ref1>..<ref2>`, and `--stdin` (read diff from stdin).

4. **Where should test-recipes.json live?**
   - What we know: L3 artifacts are in `.planning/formal/reasoning/`. Gates are in `.planning/formal/gates/`. Test recipes are a new category.
   - What's unclear: Whether recipes belong in reasoning (derived from L3) or a new directory.
   - Recommendation: Create `.planning/formal/test-recipes/test-recipes.json`. Test recipes are the output of the L3->TC transition, not L3 reasoning artifacts themselves. A dedicated directory makes the layer boundary clear and allows future expansion (e.g., per-model recipes).

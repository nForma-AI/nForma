  <action>
Create `bin/design-impact.cjs` that traces a git diff through all three formal verification layers.

**CLI interface:**
- `node bin/design-impact.cjs` — analyzes `HEAD~1..HEAD` diff by default
- `node bin/design-impact.cjs --diff=<ref1>..<ref2>` — analyzes specific commit range
- `node bin/design-impact.cjs --stdin` — reads diff from stdin (for piping `git diff | node bin/design-impact.cjs --stdin`)
- `--json` flag for machine-readable output
- `--project-root=` arg for cross-repo usage
- Follow existing bin/ pattern (shebang, PROJECT_ROOT env var override)

**Step 1: Parse git diff to extract changed files and line ranges.**
- Use `spawnSync('git', ['diff', '--name-only', ref])` for file list
- Use `spawnSync('git', ['diff', '--unified=0', ref])` for line ranges
- Parse only `---`/`+++`/`@@` lines. Do NOT interpret patch content semantics.
- Extract changed line ranges from `@@ -L,N +L,N @@` hunk headers
- For --stdin mode, parse the diff text directly instead of spawning git

**Step 2: L1 (Instrumentation) impact.**
- Load `.planning/formal/evidence/instrumentation-map.json`
- For each emission_point where emission_point.file is in changed_files:
  - If line_number is within a changed line range: mark as "direct" impact
  - Else if file is changed but line not in range: mark as "file_level" impact
- Output: list of affected emission points with action, xstate_event, impact_type

**Step 3: L2 (State) impact.**
- Load `.planning/formal/semantics/observed-fsm.json`
- Collect all xstate_event values from L1 affected emission points
- For each transition in observed_transitions matching an affected event:
  - Record state, event, to_state, count
- Output: list of affected state transitions

**Step 4: L3 (Hazard) impact.**
- Load `.planning/formal/reasoning/hazard-model.json` and `.planning/formal/reasoning/failure-mode-catalog.json`
- For each affected L2 transition (state, event pair):
  - Find matching hazard entries and failure mode entries
- Output: list of affected hazards with hazard_id, rpn, severity, and associated failure modes

**Step 5: Summary.**
- Output JSON with schema_version "1", generated timestamp, changed_files count, l1_impact (affected_emission_points, direct_hits, details), l2_impact (affected_transitions, affected_events, details), l3_impact (affected_hazards, max_rpn, affected_failure_modes, details), summary string.

**Human-readable output (default, no --json):**
Print a formatted report with sections for each layer, showing affected items. If no instrumented files are affected, print "No instrumented files affected by this change" and skip L2/L3 sections.

**Important anti-patterns to avoid:**
- Do NOT report L1 impact for files not in instrumentation-map.json
- Do NOT attempt to interpret patch content (only parse file names and @@ hunk headers)
- Do NOT include failure-mode-catalog.json failure modes that do not match affected L2 state-event pairs

**Tests (bin/design-impact.test.cjs):**
- Unit: parseGitDiff extracts file names from `---`/`+++` lines
- Unit: parseDiffLineRanges extracts line ranges from `@@ -L,N +L,N @@` hunk headers
- Unit: L1 matching finds emission points for instrumented files
- Unit: L1 matching returns empty for non-instrumented files (e.g., docs, tests)
- Unit: L2 chaining finds transitions for affected xstate_events
- Unit: L3 chaining finds hazards for affected state-event pairs
- Unit: no-impact case produces explicit zero-impact report
- Integration: analyzeImpact with synthetic diff data chains through all 3 layers
- Edge: handles empty diff (no files changed)
- Edge: handles diff with only non-instrumented files

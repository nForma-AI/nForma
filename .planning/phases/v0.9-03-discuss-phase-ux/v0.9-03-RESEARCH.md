## Summary

This phase ports two UX improvements from GSD 1.20.6 into QGSD's `discuss-phase.md`. The changes were read directly from the upstream source in `node_modules/get-shit-done-cc/get-shit-done/workflows/discuss-phase.md` and cross-referenced against the current QGSD source at `get-shit-done/workflows/discuss-phase.md` and the installed runtime at `~/.claude/qgsd/workflows/discuss-phase.md`. The diff is precise and the work is well-bounded.

**Change 1 (DSC-01, DSC-02):** In `present_gray_areas`, each option now includes a recommended choice with brief reasoning. In `discuss_areas`, each AskUserQuestion also marks the recommended choice with a brief explanation. These are additive instructions — the option format description gets one line added.

**Change 2 (DSC-03):** The hard-stop at the end of `discuss_areas` (step 4) is replaced with a loop-back. Instead of "Ready to create context?", the user sees "Explore more gray areas" vs "I'm ready for context". Choosing "Explore more gray areas" re-runs `present_gray_areas` with 2-4 newly identified areas, deduplicating already-explored areas. This requires tracking explored areas in a list and ensuring re-entry to `present_gray_areas` never repeats them.

**Key finding:** The QGSD source already has the `r4_pre_filter` step and the `auto_resolved[]` display in `present_gray_areas` — these are QGSD additions not in vanilla GSD 1.20.6. The upstream improvements (DSC-01..03) must be merged WITH these existing QGSD extensions, not replacing them.

**Primary recommendation:** Edit `get-shit-done/workflows/discuss-phase.md` directly, applying the upstream UX changes while preserving all QGSD-specific content. Then run install sync to propagate to `~/.claude/qgsd/`.

### Deduplication pattern for "Explore more gray areas"

The loop-back must not re-present already-explored areas. The upstream models this as:

1. Maintain an `explored_areas[]` list — initialized to the areas selected by the user in the first `present_gray_areas` call
2. On "Explore more gray areas": generate 2-4 NEW areas not in `explored_areas[]`
3. Add newly selected areas to `explored_areas[]` before the next loop
4. This is an in-context state management concern — no disk state needed

The prompt text says "Return to present_gray_areas logic with these new areas" — meaning the same display/selection mechanic, but with a pre-filtered area list that excludes already-explored ones.

### The `present_gray_areas` re-entry contract

When `discuss_areas` loops back into `present_gray_areas`:

- The `for_user[]` list passed to `present_gray_areas` contains ONLY the new (non-duplicate) areas
- The `auto_resolved[]` display at the top of `present_gray_areas` should NOT be shown again (those were from the original r4_pre_filter run)
- The AskUserQuestion multiSelect shows only the new areas

This means in practice: `present_gray_areas` on re-entry acts as a simple multi-select from the new list, skipping the `auto_resolved[]` section header.

**Key insight:** This is a prompt engineering change, not a code change. The "state" is the LLM's in-context memory during a single discuss-phase session. No new tooling is needed.

## Common Pitfalls

### Pitfall 1: Clobbering the QGSD r4_pre_filter content
**What goes wrong:** Applying the upstream diff mechanically and deleting the `r4_pre_filter` step or the `auto_resolved[]` display block in `present_gray_areas` (these only exist in QGSD, not vanilla GSD).
**Why it happens:** The diff shows QGSD has 172 lines different from upstream — most of those are the r4_pre_filter addition. If an implementer patches from the upstream direction they might overwrite QGSD additions.
**How to avoid:** Apply changes as surgical edits to the QGSD source — never copy the upstream file wholesale. Three targeted edits:
  1. Add one bullet to `present_gray_areas` options spec
  2. Extend one line in `discuss_areas` step 2
  3. Replace step 4 of `discuss_areas` with loop-back text
**Warning signs:** If `r4_pre_filter` step disappears from the file, the edit went wrong.

### Pitfall 2: Forgetting the install sync
**What goes wrong:** The workflow file is edited in the QGSD repo source (`get-shit-done/workflows/`) but the installed runtime at `~/.claude/qgsd/workflows/` is NOT updated. The running system continues using the old behavior.
**Why it happens:** Unlike hook files, there's no obvious "dist/" step — but `bin/install.js` still copies these files.
**How to avoid:** Always run `node bin/install.js --claude --global` after editing any file in `get-shit-done/`. Verify with diff.

### Pitfall 3: Imprecise "recommended choice" phrasing
**What goes wrong:** The recommendation added to each option is vague ("common approach") rather than specific ("shorter because CLI tools benefit from minimal interaction").
**Why it happens:** Without concrete guidance, the LLM generating the recommendation during runtime may default to generic language.
**How to avoid:** The workflow instruction should say "brief explanation why" — this is already what the upstream specifies. The planner should include this phrasing verbatim so Claude produces specific rationale when it runs discuss-phase.

### Pitfall 4: Explored areas deduplication is too strict
**What goes wrong:** The loop-back generates 2-4 areas but then checks against exact label strings. If the LLM rephrases a previously covered area, it passes the deduplication check and re-appears.
**Why it happens:** String equality deduplication is fragile for natural language.
**How to avoid:** The workflow should say "not already explored" with context — the LLM uses semantic awareness, not string equality. The instruction "2-4 additional gray areas based on what was learned" implies these should be genuinely new topics, not variations of covered ones.

### Pitfall 5: The loop-back changes the "Done" question wording
**What goes wrong:** The old wording was "That covers [list areas]. Ready to create context?" with options "Create context" / "Revisit an area". The new wording is different — "We've discussed [list areas]. Which gray areas remain unclear?" with "Explore more gray areas" / "I'm ready for context".
**Why it happens:** Copy-paste error or partial application of the change.
**How to avoid:** Replace the entire step 4 block, not just the options. The header, question text, and options all change.

# Phase v0.34-04: Semantic Scoring Pipeline Wiring - Research

**Researched:** 2026-03-11
**Domain:** Semantic scoring pipeline integration + gate schema preservation
**Confidence:** HIGH

## Summary

Phase v0.34-04 addresses two integration gaps identified in v0.34 audit:

1. **Pipeline Integration (Gap #1, #2):** compute-semantic-scores.cjs is built in v0.34-02 but NOT called from nf-solve.cjs. The semantic scoring happens standalone; for scores to persist through solve cycles, the pipeline must invoke compute-semantic-scores.cjs after gate computation (sweepPerModelGates at line 2561) and before auto-promotion (line 3824).

2. **Schema Preservation (Gap #5):** compute-per-model-gates.cjs writes gate JSON files (gate-a/b/c) with hardcoded `schema_version: '2'` (line 367-378), overwriting any semantic_score fields added by compute-semantic-scores.cjs. These files must be bumped to schema v3 and enrich with semantic_score fields, preserving them across cycles.

The phase reuses three scripts built in v0.34-02 (candidate-discovery.cjs, haiku-semantic-eval.cjs, compute-semantic-scores.cjs) and modifies two existing scripts (nf-solve.cjs to wire the call, compute-per-model-gates.cjs to preserve semantic_score fields).

**Primary recommendation:** Wire compute-semantic-scores.cjs into nf-solve.cjs Step 7 (immediately after sweepPerModelGates), modify compute-per-model-gates.cjs to preserve semantic_score fields when they exist, and bump gate schema version to '3'.

## Architecture Patterns

### Current Gap: Semantic Scoring Not Wired

**What goes wrong:** compute-semantic-scores.cjs exists and works correctly (v0.34-02 verified), but is only called manually as a CLI tool. The nf-solve.cjs convergence loop never invokes it, so:
- After sweepPerModelGates() writes gate files (lines 2318), they contain only wiring_score
- compute-semantic-scores.cjs is never run in the solve pipeline
- Result: semantic_score fields never appear in production gate files
- Auto-promotion code (v0.34-05) reads semantic_score from gates and gets 0 (field missing)

**Evidence:** Grep nf-solve.cjs for "compute-semantic-scores" — returns no results. The call is missing.

### Pattern 1: Pipeline Integration Point (NEW)

**What:** After compute-per-model-gates.cjs writes gate files with wiring_score, immediately invoke compute-semantic-scores.cjs to enrich them with semantic_score fields.

**Where in solve cycle:** Step 7 (post-gate-computation), before Step 8a (auto-promotion)
- Line 2318 in sweepPerModelGates(): compute-per-model-gates.cjs called
- Line 3821+ in main(): promote-gate-maturity.cjs called (auto-promotion)
- **INSERT HERE:** compute-semantic-scores.cjs call

**Example:**
```javascript
// Source: nf-solve.cjs pattern (add after sweepPerModelGates completion)
// After gate computation, enrich with semantic scores
const semResult = spawnTool('bin/compute-semantic-scores.cjs', ['--json']);
if (!semResult.ok) {
  process.stderr.write(TAG + ' WARNING: compute-semantic-scores failed; gates lack semantic_score fields\n');
  // Fail-open: continue with incomplete gates rather than blocking solve
}
```

**Why fail-open:** Semantic scoring is an enrichment step. If candidates.json is missing or haiku evaluation failed, gates still have wiring_score and can proceed. Auto-promotion will correctly mark sessions as non-clean (semantic >= 0.8 check fails on missing field).

### Pattern 2: Schema Preservation (MODIFICATION)

**What:** compute-per-model-gates.cjs must preserve semantic_score fields when they exist in gate files, not overwrite them.

**Current problematic code** (lines 366-385):
```javascript
const gateAFile = {
  schema_version: '2',  // ← Hardcoded v2 → overwrites semantic_score
  generated: new Date().toISOString(),
  // ... only wiring_score fields ...
};
fs.writeFileSync(path.join(GATES_DIR, 'gate-a-grounding.json'), JSON.stringify(gateAFile, null, 2) + '\n');
```

**Required change:**
```javascript
// 1. Read existing gate file to check for semantic_score
let existingSemanticScore = null;
let existingMetadata = null;
try {
  const existing = JSON.parse(fs.readFileSync(path.join(GATES_DIR, 'gate-a-grounding.json'), 'utf8'));
  if (existing.semantic_score != null) {
    existingSemanticScore = existing.semantic_score;
    existingMetadata = existing.semantic_metadata;
  }
} catch (_) { /* no existing file or parse error */ }

// 2. Build new gate file with preserved semantic_score
const gateAFile = {
  schema_version: '3',  // ← Bump to v3 when semantic_score present
  generated: new Date().toISOString(),
  // ... wiring_score fields ...
};

// 3. Preserve semantic_score if it existed
if (existingSemanticScore != null) {
  gateAFile.semantic_score = existingSemanticScore;
  gateAFile.semantic_metadata = existingMetadata;
}

fs.writeFileSync(...);
```

**Why schema v3:** Indicates gate file carries semantic validation alongside wiring validation. per-model-gates.json is already v3; gate aggregate files should match.

### Anti-Patterns to Avoid

- **Don't:** Call compute-semantic-scores.cjs in fast mode (skip semantic scoring in --fast). **DO:** Semantic scoring is optional enrichment; fast mode can skip it without breaking solve.
- **Don't:** Fail the entire solve if semantic scoring fails. **DO:** Log warning and continue — gates still have wiring_score.
- **Don't:** Overwrite semantic_score fields when reading/writing gates. **DO:** Read first, preserve on write.

## Common Pitfalls

### Pitfall 1: Silent Semantic Score Loss

**What goes wrong:** compute-per-model-gates.cjs creates new gate files (lines 383-385) without reading existing content. If compute-semantic-scores.cjs ran first, semantic_score is lost.

**Why it happens:** Straightforward overwrite pattern. The script was written before semantic_score existed; no logic to preserve it.

**How to avoid:** Always read existing gate file before writing (Pattern 2). Check for semantic_score field and copy it forward.

**Warning signs:**
- After a solve run, gate files have `"semantic_score": 0` or field is missing entirely
- Comparing gate-a-grounding.json before/after solve shows semantic_score disappeared
- Run `git diff .planning/formal/gates/` — if semantic_score is gone, you hit this pitfall

### Pitfall 2: Semantic Scoring Skipped in Pipeline

**What goes wrong:** nf-solve.cjs is modified but compute-semantic-scores.cjs call is placed in the wrong location (e.g., in a conditional that doesn't execute, or after auto-promotion instead of before).

**Why it happens:** Integration points in nf-solve are complex; easy to miss the exact line where sweepPerModelGates completes and auto-promotion begins.

**How to avoid:**
- Add compute-semantic-scores call immediately after sweepPerModelGates(), before the next function
- Verify with grep: `grep -n "compute-semantic-scores\|spawnTool" bin/nf-solve.cjs` should show the call
- Test with `node bin/nf-solve.cjs --report-only --json` and check gate files have semantic_score

**Warning signs:**
- Gate files after solve have schema_version: '2' and no semantic_score field
- compute-semantic-scores.cjs process exit code is not captured/checked

### Pitfall 3: Fail-Hard Instead of Fail-Open

**What goes wrong:** compute-semantic-scores.cjs call fails (e.g., candidates.json missing or Haiku API timeout), and the entire solve pipeline exits. Solve becomes brittle.

**Why it happens:** Treating semantic scoring as critical rather than enrichment.

**How to avoid:** Check result.ok but don't exit. Log warning and continue.
```javascript
const semResult = spawnTool('bin/compute-semantic-scores.cjs', ['--json']);
if (!semResult.ok) {
  process.stderr.write(TAG + ' WARNING: semantic scoring skipped (see stderr)\n');
  // Continue — wiring_score is sufficient for convergence
}
```

**Warning signs:**
- Solve fails with "compute-semantic-scores.cjs error" instead of gracefully degrading
- candidates.json is missing, but entire solve aborts instead of logging warning

### Pitfall 4: Schema Version Mismatch

**What goes wrong:** Some gate files are v2, others are v3; downstream code (auto-promotion, TUI) expects consistent schema.

**Why it happens:** compute-per-model-gates.cjs writes v2, compute-semantic-scores.cjs preserves/upgrades to v3 if semantic_score added, but the order is wrong.

**How to avoid:**
- compute-per-model-gates.cjs writes v2 (or v3 if semantic_score present)
- compute-semantic-scores.cjs always preserves/upgrades to v3
- Result: all gate files converge to v3 after first semantic scoring run
- Schema v3 gates with no semantic_score fields are still valid (field is optional)

**Warning signs:**
- `git diff .planning/formal/gates/` shows inconsistent schema_version values
- TUI or dashboard crashes on gate files (try/catch error handling missing)

## Open Questions

1. **Performance impact of semantic scoring in solve loop**
   - What we know: compute-semantic-scores.cjs runs in ~500ms (local test); nf-solve iterations are ~30-60s
   - What's unclear: Impact on --max-iterations total time with semantic scoring in every iteration
   - Recommendation: Add optional `--skip-semantic` flag to nf-solve.cjs for fast diagnostic runs. Keep in main loop but skip in --fast mode.

2. **Schema v3 backward compatibility**
   - What we know: per-model-gates.json is already v3; gate files are v2
   - What's unclear: Do any downstream consumers (TUI, dashboard) assume v2 fields only?
   - Recommendation: Audit scripts reading gate files for hardcoded v2 assumptions. Add schema check with fallback logic (prefer v3 fields, fall back to v2 field names).

3. **Idempotency across multiple compute-per-model-gates invocations**
   - What we know: compute-per-model-gates.cjs is called twice in some code paths (line 2174, line 2318)
   - What's unclear: Will preserving semantic_score cause issues if gate computation runs twice in one cycle?
   - Recommendation: Test with multiple sweepPerModelGates calls in same iteration. Semantic_score should remain unchanged.

# Phase v0.21-04: Spec Completeness - Research

**Researched:** 2026-03-01
**Domain:** Formal verification specification completeness (TLA+, Alloy, verification wiring)
**Confidence:** HIGH

## Summary

Phase v0.21-04 closes four critical specification gaps in QGSD's formal verification pipeline. The most critical QGSD subsystems — the Stop hook logic, circuit breaker oscillation detection algorithm, quorum composition rules, and phase requirement verification — currently lack formal models despite being load-bearing infrastructure.

This phase requires creating three new formal specifications (Stop hook in TLA+, oscillation algorithm audit in TLA+, quorum composition in Alloy) and wiring phase requirements to per-phase TLA+ specs that are verified by TLC before planning. All four requirements (SPEC-01 through SPEC-04) depend on foundational work already completed in v0.21-01 (central model registry), so the path is clear: write specs, wire them into the verification pipeline, and ensure TLC proves the safety and liveness properties.

**Primary recommendation:**
Implement in order of dependency and complexity: (1) write Stop hook TLA+ spec with safety/liveness properties; (2) audit QGSDOscillation.tla against qgsd-circuit-breaker.js implementation (comparing second-pass net-diff logic); (3) write quorum composition Alloy spec covering three composition rules; (4) implement per-phase TLA+ scratch spec generation from task envelope `must_haves: truths:` blocks.

## Architecture Patterns

### Specification Organization Pattern
All formal specs follow the same directory structure:
- **TLA+ specs:** `formal/tla/<ModuleName>.tla` + `formal/tla/MC<ModuleName>.cfg`
- **Alloy specs:** `formal/alloy/<model-name>.als`
- **Registry tracking:** `formal/model-registry.json` updated with `version`, `last_updated`, `update_source: "manual" | "generate" | "debug" | "plan-promote"`

**Example:** QGSDQuorum.tla (generated from XState) + MCQuorum.cfg (TLC config) stored in formal/tla/; version 4 recorded in model-registry.json with update_source="generate" and timestamp.

### Pattern 1: TLA+ Safety and Liveness Properties
**What:** TLA+ specs use two property categories:
- **INVARIANT** — Safety properties; checked at every state (bad thing must not happen)
- **PROPERTY** — Liveness properties; checked along infinite traces (good thing eventually happens)

**When to use:**
- Safety when modeling a constraint that must NEVER be violated (e.g., "if planning command issued without quorum evidence, BLOCK")
- Liveness when modeling eventual progress (e.g., "if quorum evidence present, eventually PASS")

**Example (Stop hook safety):**
```tla
(* Source: hooks/qgsd-stop.js — formalized decision logic *)
INVARIANT SafetyGuard ==
    /\ HasPlanningCommand /\ ~HasQuorumEvidence => decision = BLOCK
    /\ decision = BLOCK => HasPlanningCommand   \* contrapositive: only block if planning command exists

PROPERTY LivenessGuard ==
    /\ HasQuorumEvidence => <>decision = PASS   \* diamond operator <> means "eventually"
    /\ decision = PASS => HasQuorumEvidence      \* no spurious passes without evidence
```

**Why this pattern:** Safety invariants catch logic errors; liveness properties ensure the gate doesn't deadlock. Together they formalize the contract.

### Pattern 2: Alloy Constraint Modeling
**What:** Alloy uses first-order logic over finite scopes. Constraints are expressed as facts (always-true predicates) and predicates (parameterized checks).

**When to use:** When modeling structural properties and selection rules that don't require temporal reasoning. Quorum composition rules (no empty selection, high-risk fan-out, solo mode) are perfect for Alloy because they're state-agnostic.

**Example (quorum composition):**
```alloy
(* Source: REQUIREMENTS.md SPEC-03 — composition rules *)
fact NoEmptySelection {
    all config : Config |
        (config.availableSlots != none) => (config.selectedSlots != none)
}

fact HighRiskFullFanOut {
    all config : Config |
        (config.riskLevel = high) => (#config.selectedSlots = #config.maxSize)
}

fact SoloModeSingleSlot {
    all config : Config |
        (config.soloMode = true) => (#config.selectedSlots = 1)
}

run {} for 5 Config, 10 Slot  \* Alloy solver finds counterexamples within scope
```

**Why this pattern:** Alloy finds counterexamples automatically; if no counterexample exists within the scope, all constraints are satisfied. Perfect for verification-by-counterexample.

### Pattern 3: Per-Phase Scratch Spec Generation
**What:** Task envelope `must_haves: truths:` blocks are translated to TLA+ PROPERTY statements in a scratch spec, verified by TLC before quorum sees the plan.

**When to use:** When planning gates need mathematical verification of proposed state machine changes before getting human approval.

**Example (requirement as property):**
```yaml
# In task-envelope.json
must_haves:
  truths:
    - "If PLAN-01 adds a new transition, TLC proves reachability"
    - "Circuit breaker depth threshold is respected (depth >= 3)"
```

Translated to:
```tla
(* Source: task-envelope.json must_haves: truths: from v0.21-04 plan *)
PROPERTY NewTransitionReachable == NewTransition ~> Phase'
PROPERTY DepthThresholdRespected == Depth >= 3
```

**Why this pattern:** Requirements are machine-checkable; verification results feed to quorum as `formal_spec_summary` and `verification_result` fields.

### Anti-Patterns to Avoid
- **Hand-writing all TLA+ transitions:** Use xstate-to-tla.cjs for state machine generation; only write handcrafted specs for non-XState logic (like Stop hook)
- **Leaving scratch specs in .planning/phases/:** Scratch specs MUST be in formal/tla/scratch/ and tracked in model-registry.json; per-phase scattered copies create divergence
- **Mixing safety and liveness in one PROPERTY:** Safety = INVARIANT; Liveness = PROPERTY; conflating them confuses TLC's fairness semantics
- **Forgetting to update run-formal-verify.cjs:** New specs (Stop hook, quorum composition, scratch specs) must be added to the STEPS registry or they won't run in CI

## Common Pitfalls

### Pitfall 1: TLC Fairness Misunderstanding
**What goes wrong:** Liveness properties fail in TLC with confusing counterexamples; developer thinks the spec is broken but actually forgot fairness declarations.

**Why it happens:** TLC checks liveness against FAIR transitions only. If transitions aren't marked FAIR, TLC finds "infinite stuttering" paths where nothing happens and concludes the liveness property can be violated.

**How to avoid:** In .cfg file, include:
```
FAIRNESS weak action FairnessLabel
```
And in .tla, mark actions with `<FairnessLabel>_<action>` syntax. Existing specs (QGSDQuorum.cfg, MCsafety.cfg) show the pattern.

**Warning signs:**
- Liveness property fails with infinite trace that only stutters (repeats same state)
- Model checker says "Property violated" but spec looks correct
- Switch INVARIANT to PROPERTY and property passes (indicates fairness needed)

### Pitfall 2: Scope Underestimation in Alloy
**What goes wrong:** Alloy solver returns "unsatisfiable" or finds a spurious counterexample because the scope was too small.

**Why it happens:** Alloy is bounded; it only checks within the scope you provide (`run {} for N Config, M Slot`). If real constraints need more atoms, solver can't find models.

**How to avoid:**
- Start with scope from related specs (account-pool-structure.als uses `for 5 Agent, 3 Pool`)
- Use `--verbose` flag in run-alloy.cjs to see scope choice
- Increase scope gradually if solver returns "unsatisfiable"
- Test against small concrete examples first (quorum-votes.als tests with 3-5 agents)

**Warning signs:**
- Solver says counterexample but it contradicts your requirements
- Increasing scope makes counterexample vanish
- Solver hangs (scope too large for SAT solver)

### Pitfall 3: Model Registry Out of Sync
**What goes wrong:** New spec is created (Stop hook TLA+) but forgotten in model-registry.json; promote-model.cjs and accept-debug-invariant.cjs skip it; invariant mining doesn't update it.

**Why it happens:** Registry is manually edited or auto-synced with bugs; easy to forget `version`, `update_source`, or `last_updated` fields.

**How to avoid:**
- Always run `initialize-model-registry.cjs` after adding new specs
- Use `promote-model.cjs` and `accept-debug-invariant.cjs` for all updates (never edit registry JSON directly)
- Add registry.json keys to CI check in run-formal-verify.cjs summary
- Test: `node bin/verify-formal-results.cjs` should pass after every new spec

**Warning signs:**
- New spec file exists in formal/tla/ but not in model-registry.json
- Version mismatch (registry says version 1 but TLC found 3+ runs)
- update_source is "manual" for auto-generated specs

### Pitfall 4: Scratch Spec Garbage Collection
**What goes wrong:** Per-phase scratch specs accumulate in formal/tla/scratch/; old failed plans leave behind stale spec files; VERIFICATION.md refers to specs that no longer exist.

**Why it happens:** Scratch specs are created in .planning/phases/<phase>/formal/ first, promoted to formal/tla/scratch/, but cleanup is not automated when plans are rejected.

**How to avoid:**
- Scratch specs live in formal/tla/scratch/<phase>-<attempt>-<timestamp>.tla
- Clean up on promotion success: delete the phase's scratch dir once formal spec is promoted
- Add GC task to plan failure recovery: "Remove formal/tla/scratch/<phase>-* if plan rejected"
- Track scratch specs in model-registry.json with `source_id: "plan:v0.21-04-01-attempt-2"`

**Warning signs:**
- formal/tla/scratch/ has > 2× (number of active phases) files
- grep finds spec referenced in old VERIFICATION.md that's no longer on disk
- TLC config points to scratch/ spec that was deleted

### Pitfall 5: Safety vs. Liveness Inversion
**What goes wrong:** Developer writes an INVARIANT that should be a PROPERTY (or vice versa); TLC rejects it or finds spurious violations.

**Why it happens:** The semantic difference is subtle. INVARIANT = "never happens". PROPERTY = "eventually happens". Inverting gives opposite results.

**How to avoid:**
- INVARIANT for constraints that must hold at EVERY state (safety)
- PROPERTY + liveness (eventually, unless, leads-to) for temporal guarantees
- Test: if property fails with "infinite stuttering" counterexample → switch INVARIANT to PROPERTY + add fairness
- Example: "Block if no quorum evidence" = INVARIANT (every state must not pass without evidence). "Eventually pass with evidence" = PROPERTY (progress guarantee).

**Warning signs:**
- INVARIANT fails but the violating state looks impossible
- PROPERTY passes but you expected it to fail
- Switching between INVARIANT and PROPERTY changes results

### Stop Hook TLA+ Specification (SPEC-01)
```tla
(* formal/tla/QGSDStopHook.tla *)
(* Source: hooks/qgsd-stop.js decision logic + REQUIREMENTS.md SPEC-01 *)
(* Formalizes: HasPlanningCommand ∧ ¬HasQuorumEvidence => decision = BLOCK *)

EXTENDS Naturals, FiniteSets, TLC

CONSTANTS
    MaxTurnLines  \* Maximum lines in current-turn window (e.g., 500)

VARIABLES
    hasCommand,           \* BOOLEAN: /gsd: or /qgsd: command detected
    hasQuorumEvidence,    \* BOOLEAN: quorum tool calls present
    decision,             \* {"BLOCK", "PASS", "UNDECIDED"}
    algorithmDone         \* BOOLEAN: hook execution finished

vars == <<hasCommand, hasQuorumEvidence, decision, algorithmDone>>

TypeOK ==
    /\ hasCommand \in BOOLEAN
    /\ hasQuorumEvidence \in BOOLEAN
    /\ decision \in {"BLOCK", "PASS", "UNDECIDED"}
    /\ algorithmDone \in BOOLEAN

Init ==
    /\ hasCommand = FALSE
    /\ hasQuorumEvidence = FALSE
    /\ decision = "UNDECIDED"
    /\ algorithmDone = FALSE

\* Hook scans transcript (nondeterministic)
DetectCommand ==
    /\ ~algorithmDone
    /\ hasCommand' \in {TRUE, FALSE}    \* Nondeterministic: may or may not find /gsd:
    /\ UNCHANGED <<hasQuorumEvidence, decision, algorithmDone>>

DetectQuorumEvidence ==
    /\ ~algorithmDone
    /\ hasQuorumEvidence' \in {TRUE, FALSE}
    /\ UNCHANGED <<hasCommand, decision, algorithmDone>>

\* Decision logic from qgsd-stop.js: if command without evidence, BLOCK
MakeDecision ==
    /\ ~algorithmDone
    /\ decision' = IF /\ hasCommand /\ ~hasQuorumEvidence
                   THEN "BLOCK"
                   ELSE IF /\ hasCommand /\ hasQuorumEvidence
                        THEN "PASS"
                        ELSE "UNDECIDED"
    /\ algorithmDone' = TRUE
    /\ UNCHANGED <<hasCommand, hasQuorumEvidence>>

Next == DetectCommand \/ DetectQuorumEvidence \/ MakeDecision

Spec == Init /\ [][Next]_vars

\* ── Safety Properties ──────────────────────────────────────
SafetyInvariant1 ==
    \* If decision = BLOCK, a planning command must have been present
    (decision = "BLOCK") => hasCommand

SafetyInvariant2 ==
    \* If decision = BLOCK, no quorum evidence was found
    (decision = "BLOCK") => ~hasQuorumEvidence

SafetyInvariant3 ==
    \* If decision = PASS, both command and evidence must be present
    (decision = "PASS") => (hasCommand /\ hasQuorumEvidence)

\* ── Liveness Properties ──────────────────────────────────────
LivenessProperty1 ==
    \* Eventually, the algorithm terminates
    <>algorithmDone

LivenessProperty2 ==
    \* If quorum evidence is detected, eventually decision is PASS (not indefinite UNDECIDED)
    (hasQuorumEvidence) => <>(decision = "PASS")

LivenessProperty3 ==
    \* If command without evidence, eventually decision is BLOCK (not indefinite UNDECIDED)
    (hasCommand /\ ~hasQuorumEvidence) => <>(decision = "BLOCK")
```

### Quorum Composition Alloy Specification (SPEC-03)
```alloy
(* formal/alloy/quorum-composition.als *)
(* Source: REQUIREMENTS.md SPEC-03 + config.yaml composition rules *)
(* Verifies: no empty selection, high-risk full fan-out, solo mode single slot *)

sig Config {
  riskLevel: one RiskLevel,
  soloMode: one Bool,
  availableSlots: set Slot,
  selectedSlots: set Slot,
  maxSize: one Int
} {
  maxSize > 0
  maxSize <= 5   \* Policy constraint
}

abstract sig Slot {}
sig Slot1, Slot2, Slot3, Slot4, Slot5 extends Slot {}

abstract sig RiskLevel {}
one sig Low, Medium, High extends RiskLevel {}

abstract sig Bool {}
one sig True, False extends Bool {}

\* ── Composition Rules (Facts) ───────────────────────────────────

\* SPEC-03 Rule 1: No empty selection when slots available
fact NoEmptySelection {
  all config : Config |
    (#config.availableSlots > 0) => (#config.selectedSlots > 0)
}

\* SPEC-03 Rule 2: High-risk configs use full fan-out
fact HighRiskFullFanOut {
  all config : Config |
    (config.riskLevel = High) =>
      (#config.selectedSlots = #config.availableSlots.limit[config.maxSize])
}

\* SPEC-03 Rule 3: Solo mode selects exactly one slot
fact SoloModeSingleSlot {
  all config : Config |
    (config.soloMode = True) => (#config.selectedSlots = 1)
}

\* Selected slots must be subset of available
fact SelectionSubset {
  all config : Config |
    config.selectedSlots in config.availableSlots
}

\* ── Test Scopes ──────────────────────────────────────
run {} for 5 Config, 5 Slot  \* Find satisfying instances
```

### Per-Phase Scratch Spec Pattern (SPEC-04)
```javascript
// bin/generate-phase-spec.cjs
// Source: REQUIREMENTS.md SPEC-04 + PLAN frontmatter task envelope
// Generates per-phase TLA+ spec from must_haves: truths: blocks

const fs = require('fs');
const path = require('path');

// Parse task envelope must_haves: truths: array
// Each truth becomes a PROPERTY statement
function generatePhaseSpec(taskEnvelope, phaseNum) {
  const truths = taskEnvelope.plan?.must_haves?.truths || [];

  let specContent = `---- MODULE Phase${phaseNum}Spec ----
(* Source: task-envelope.json must_haves: truths: *)
(* Generated by generate-phase-spec.cjs for SPEC-04 *)
(* Verify proposed state machine changes before quorum approval *)

EXTENDS Naturals, FiniteSets, TLC

VARIABLES state, counter

TypeOK == state \in {"INIT", "RUNNING", "DONE"}

Init == state = "INIT" /\\ counter = 0

\\* ── Requirements as Properties ──────────────────────────────────

`;

  truths.forEach((truth, idx) => {
    // Simple heuristic: if truth mentions "threshold" or "depth", make it an INVARIANT
    // If truth mentions "eventually" or async concepts, make it a PROPERTY
    const isLiveness = /eventually|async|future|deadline/.test(truth.toLowerCase());
    const keyword = isLiveness ? 'PROPERTY' : 'INVARIANT';

    const propName = `Req${idx + 1}`.replace(/-/g, '_');
    const propBody = truth
      .replace(/"/g, '\\"')
      .replace(/\n/g, ' ')
      .substring(0, 80);  // Truncate for readability

    specContent += `
${keyword} ${propName} ==
  \\* "${propBody}..."
  TRUE  \\* PLACEHOLDER: developer fills in formal logic

`;
  });

  specContent += `====`;
  return specContent;
}

module.exports = { generatePhaseSpec };
```

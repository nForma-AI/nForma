## Summary

Phase v0.15-04 is a targeted two-file change: add a new health check (Check 9) to `qgsd-core/bin/gsd-tools.cjs` that reads `.planning/quorum-failures.json` and calls `addIssue('warning', 'W008', ...)` for any slot with `count >= 3`, then update `qgsd-core/workflows/health.md` to register W008 in the error-code table.

The live `.planning/quorum-failures.json` file has been inspected directly. It is an array of objects with shape `{ slot, error_type, pattern, count, last_seen }`. The threshold is `count >= 3` (from the phase Success Criteria). The check must be silent when the file does not exist or no slot meets the threshold.

The `/qgsd:health` workflow (in `qgsd-core/workflows/health.md`) calls `node ~/.claude/qgsd/bin/gsd-tools.cjs validate health`, parses the JSON output, and renders `warnings[]` items in a `## Warnings` section. Because VIS-01 requires the quorum-failure warnings to appear in the same section as other W/E/I items, the correct integration point is the `warnings[]` array inside `cmdValidateHealth` — not a separate section in the workflow markdown.

After editing `qgsd-core/bin/gsd-tools.cjs` and `qgsd-core/workflows/health.md`, the installed copy must be synced via `node bin/install.js --claude --global`.

**Primary recommendation:** Add Check 9 in `cmdValidateHealth` using `addIssue('warning', 'W008', ...)` immediately after Check 8 (around line 3903), register W008 in `health.md`'s error-code table, sync install, write 2 VIS-01 unit tests.

### Pattern: New Check (follows existing pattern exactly)

```javascript
// ─── Check 9: Quorum failure patterns ─────────────────────────────────────────
const failuresPath = path.join(planningDir, 'quorum-failures.json');
if (fs.existsSync(failuresPath)) {
  try {
    const failures = JSON.parse(fs.readFileSync(failuresPath, 'utf-8'));
    for (const entry of failures) {
      if (entry.count >= 3) {
        addIssue(
          'warning',
          'W008',
          `Quorum slot "${entry.slot}" has ${entry.count} recurring failures (${entry.error_type})`,
          `Review slot "${entry.slot}" availability. Check .planning/quorum-failures.json for details.`

```

**Why `try/catch`:** Consistent with every other file-reading check in `cmdValidateHealth` (Checks 4, 5, 6, 7, 8 all use `try/catch` or `try {}` blocks). If `quorum-failures.json` is malformed JSON, silently skip — do not crash the health command.

**Why no `repairable = true`:** The failure record cannot be auto-repaired by the health command. The user must address the underlying slot availability issue. Consistent with W005, W006, W007 which are also not repairable.

### Anti-Patterns to Avoid

- **Separate output section in health.md:** Do NOT add a `## Quorum Failures` section to the workflow. Success Criterion 3 explicitly requires the warnings to appear in the same W/E/I block. The `warnings[]` array already handles this.

## Common Pitfalls

### Pitfall 1: Off-by-one in threshold
**What goes wrong:** Using `> 3` instead of `>= 3` — a slot with exactly count=3 does not emit a warning even though it should.
**Why it happens:** Misreading "3 or more" as "more than 3".
**How to avoid:** Use `entry.count >= 3`. The live file has `codex-1` with count=3 — the test scaffold should use count=3 to verify the threshold is inclusive.
**Warning signs:** VIS-01-TC-01 test fails (count=3 case produces no warning).

### Pitfall 2: Check 9 placed after the output() call
**What goes wrong:** Check 9 code is placed after `output(...)` at line 3992 — it never executes because `output()` calls `process.exit(0)`.
**Why it happens:** Misreading the function structure.
**How to avoid:** Insert Check 9 **before** the repairs section (around line 3904). Verify by checking: `output(` appears only once in `cmdValidateHealth`, at the very end of the function.
**Warning signs:** Test runs but warnings[] is always empty.

### Pitfall 3: W008 not in error-code table in health.md
**What goes wrong:** health.md renders W/E/I items dynamically from the JSON output, so a W008 warning does appear in output, but the error-code table at the bottom is incomplete — future operators don't know what W008 means.
**Why it happens:** Treating health.md as render-only, missing the documentation table.
**How to avoid:** Add the W008 row to the `| W008 | warning | ... |` table in `qgsd-core/workflows/health.md`.

### Pitfall 4: Install sync omitted
**What goes wrong:** Fix applied to `qgsd-core/bin/gsd-tools.cjs`, tests pass, but `/qgsd:health` still shows no W008 because the installed copy at `~/.claude/qgsd/bin/gsd-tools.cjs` is stale.
**Why it happens:** `/qgsd:` commands run from `~/.claude/qgsd/`, not from `qgsd-core/`.
**How to avoid:** Always include `node bin/install.js --claude --global` as the final task step after editing `qgsd-core/`.

### Pitfall 5: Test creates quorum-failures.json in temp project root instead of .planning/
**What goes wrong:** Test writes `quorum-failures.json` to `tmpDir/quorum-failures.json` instead of `tmpDir/.planning/quorum-failures.json`. The check looks in `planningDir` (`cwd + '/.planning'`).
**Why it happens:** Incorrect path join in test setup.
**How to avoid:** Write the file to `path.join(tmpDir, '.planning', 'quorum-failures.json')`.

### Pitfall 6: gsd-tools.test.cjs test checks `data.issues` instead of `data.warnings`
**What goes wrong:** FIX-04-TC-02 (line 2140-2143 in the test file) uses `data.issues || []` — but `validate health` output has `data.warnings`, `data.errors`, `data.info` (separate arrays), not a unified `data.issues`. The HLTH tests (lines 3700-3739) correctly use `data.warnings`. Follow the HLTH pattern, not FIX-04-TC-02 pattern.
**How to avoid:** Check `(data.warnings || []).some(i => i.code === 'W008' ...)` — not `data.issues`.

## Code Examples

### Check 9: Full Implementation

```javascript
// ─── Check 9: Quorum failure patterns ─────────────────────────────────────────
const failuresPath = path.join(planningDir, 'quorum-failures.json');
if (fs.existsSync(failuresPath)) {
  try {
    const failures = JSON.parse(fs.readFileSync(failuresPath, 'utf-8'));
    if (Array.isArray(failures)) {
      for (const entry of failures) {
        if (typeof entry.count === 'number' && entry.count >= 3) {
          addIssue(
            'warning',
            'W008',
            `Quorum slot "${entry.slot}" has ${entry.count} recurring failures (${entry.error_type})`,
            `Review slot "${entry.slot}" availability. Check .planning/quorum-failures.json for details.`
          );
        }
      }
    }
  } catch {}
}
```

**Note:** `Array.isArray(failures)` guard and `typeof entry.count === 'number'` guard protect against malformed files. These defensive checks are lightweight and prevent runtime errors.

## Open Questions

1. **Should W008 affect overall health status?**
   - What we know: `status` is set to `'degraded'` when `warnings.length > 0` (line 3976). W008 items go into `warnings[]`. So adding W008 will cause status to become `degraded` if no errors exist.
   - What's unclear: Is this the desired behavior, or should quorum-failure warnings be informational (info level) to not degrade status?
   - Recommendation: Use `warning` (W008) as specified in the phase goal — "health warnings alongside standard W/E/I items". The `'degraded'` status is appropriate when a quorum slot is repeatedly failing. If the team later decides this is too noisy, an `info` level variant is trivial to implement.

2. **Multiple failure entries for the same slot?**
   - What we know: The current live file has one entry per slot. The schema doesn't explicitly prevent duplicate slot entries.
   - What's unclear: If the same slot appears twice (e.g., two different error types both with count >= 3), should two W008 warnings be emitted?
   - Recommendation: Emit one W008 per array entry that meets the threshold. This is the simplest implementation and matches user expectation (each failure pattern is independently notable).

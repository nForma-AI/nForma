## Architecture Patterns

### How the Two Paths in transition.md Route B Work

The `offer_next_phase` step in transition.md already uses `is_last_phase` from `gsd-tools phase complete` to branch:

- `is_last_phase: false` ‚Üí Route A (more phases remain)
- `is_last_phase: true` ‚Üí Route B (milestone complete)

Route B currently contains only two branches: yolo mode and interactive mode. Both immediately invoke `/qgsd:complete-milestone`. This is the insertion point for LOOP-01 and LOOP-02.

**New Route B structure:**

```
Route B: Milestone complete (all phases done)

Step 1: Detect path type
  - Read the completed phase's ROADMAP.md entry
  - Check: does the entry contain `**Gap Closure:**`?

Step 2a: Gap Closure path (LOOP-02)
  - If YES ‚Üí route to audit-milestone (re-audit to verify gap closure worked)
  - Exit and invoke /qgsd:audit-milestone

Step 2b: Primary completion path (LOOP-01)
  - If NO ‚Üí run audit-milestone before complete-milestone (first audit)
  - Exit and invoke /qgsd:audit-milestone
  - (complete-milestone is downstream of audit passing ‚Äî that happens in audit-milestone's offer_next)
```

**Important:** LOOP-01 and LOOP-02 both route to `audit-milestone`. The difference is the reason and message, not the destination. In both cases, audit-milestone runs first. Complete-milestone only runs after audit-milestone returns `passed`.

### Gap Closure Marker Detection Pattern

The `**Gap Closure:**` marker appears in the ROADMAP.md phase section body, after `**Requirements:**` and before `**Success Criteria**`. It is a standalone bold-key line:

```markdown
### Phase v0.10-06: Import/Export
**Goal**: ...
**Depends on**: Phase v0.10-05
**Requirements**: PORT-01, PORT-02, PORT-03
**Gap Closure:** Closes PORT-01/02/03 unsatisfied gaps from v0.10 audit ‚Äî phase was never planned or implemented
**Success Criteria** (what must be TRUE):
```

Detection bash command ‚Äî reads ROADMAP.md, extracts the completed phase's section, and checks for the marker:

```bash
# Get the completed phase's ROADMAP entry and check for Gap Closure marker
PHASE_ENTRY=$(node ~/.claude/qgsd/bin/gsd-tools.cjs roadmap analyze 2>/dev/null)
# Or use bash grep directly on the roadmap section for the completed phase:
IS_GAP_CLOSURE=$(grep -A 10 "Phase ${COMPLETED_PHASE}" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*')
# IS_GAP_CLOSURE will be 0 (no marker) or 1+ (marker present)
```

Alternatively, since the planner has access to `gsd-tools roadmap analyze` JSON output which includes phase info, the marker detection can be done inline as a grep on the ROADMAP.md file for the completed phase's section.

## Common Pitfalls

### Pitfall 1: Modifying Only the Source, Not the Installed Copy

**What goes wrong:** Changes to `get-shit-done/workflows/transition.md` do not affect runtime behavior. The installed copy at `~/.claude/qgsd/workflows/transition.md` is what Claude Code actually reads when following workflow instructions.
**Why it happens:** The repo source and installed copy are separate. Unlike hooks (which have a formal install step), workflow files are copied manually or via the install script.
**How to avoid:** The plan must include explicit tasks to edit BOTH paths. This is not optional ‚Äî only editing the installed copy would mean the change is lost on next install; only editing the source would mean the change never takes effect.
**Warning signs:** Tests pass against installed copy but CI checks source. Always confirm both paths are identical after edit.

### Pitfall 2: Breaking the tech_debt Auto-Spawn Gate

**What goes wrong:** Applying the auto-spawn Task for plan-milestone-gaps to the `tech_debt` block as well as the `gaps_found` block.
**Why it happens:** Both blocks currently suggest `/qgsd:plan-milestone-gaps`. The requirements say LOOP-03 only applies when audit status is `gaps_found` AND at least one phase is `missing_no_plan`. Tech debt is a human choice ‚Äî auto-spawning it removes user agency over debt acceptance.
**How to avoid:** The auto-spawn logic must be gated on: `status == gaps_found AND any(phase.classification == missing_no_plan)`. The tech_debt block stays as a human-facing suggestion.

### Pitfall 3: Confusing the Two Route B Sub-Paths

**What goes wrong:** Treating LOOP-01 and LOOP-02 as different destinations instead of the same destination with different trigger conditions.
**Why it happens:** LOOP-02 says "routes to audit-milestone INSTEAD OF complete-milestone" and LOOP-01 says "calls audit-milestone BEFORE complete-milestone." This reads like they are different. They are not ‚Äî both route to audit-milestone. LOOP-01 is the "fresh audit before completing" case; LOOP-02 is the "re-audit after gap closure" case. Complete-milestone is downstream of audit passing in the audit-milestone workflow itself.
**How to avoid:** Route B new logic: detect Gap Closure marker ‚Üí if YES, message says "re-audit gap closure"; if NO, message says "primary audit"; both invoke audit-milestone. No path skips audit-milestone.

### Pitfall 4: Breaking the plan_exists_not_executed Auto-Execute Path

**What goes wrong:** The LOOP-03 change to auto-spawn plan-milestone-gaps accidentally overrides the existing `plan_exists_not_executed` auto-execute path in audit-milestone.md offer_next.
**Why it happens:** The existing `offer_next` already has a pre-routing check: "If ALL unsatisfied requirements are from phases classified as `plan_exists_not_executed`" ‚Üí auto-execute. LOOP-03 only applies when `missing_no_plan` phases exist. The pre-routing check must remain intact.
**How to avoid:** The new auto-spawn Task is added to the standard gaps_found routing below the pre-routing check, not as a replacement for the pre-routing check. The structure must be: pre-routing check first ‚Üí if plan_exists_not_executed path applies, take it ‚Üí else if missing_no_plan exists, auto-spawn plan-milestone-gaps Task.

### Pitfall 5: STATE.md Update Placed After Present Results (Step 7)

**What goes wrong:** Adding the STATE.md update after Step 7 means audit-milestone can output its results and exit before STATE.md is updated ‚Äî especially in yolo mode where subsequent auto-actions may skip the update.
**Why it happens:** STATE-01 requirement says "after writing MILESTONE-AUDIT.md" ‚Äî the natural reading is "add it at the end." But Step 7 triggers routing which may immediately invoke another workflow.
**How to avoid:** Place the STATE.md update at the end of Step 6 (after MILESTONE-AUDIT.md is written), before Step 7 (Present Results). This guarantees STATE.md is updated regardless of which routing path is taken.

## Open Questions

1. **yolo mode vs. interactive mode for Route B audit gate**
   - What we know: transition.md Route B has both yolo and interactive branches. Both currently go to complete-milestone. LOOP-01 says "no human prompt required."
   - What's unclear: Should the new audit gate text differ between yolo and interactive, or should both simply invoke audit-milestone without the complete-milestone text?
   - Recommendation: In yolo mode, output `"‚ö° Auto-continuing: Run audit-milestone"` and invoke. In interactive mode, output the same but wrapped in the existing interactive format. The destination is identical.

2. **plan-milestone-gaps Task spawn: what subagent_type to use**
   - What we know: The integration checker uses `subagent_type="qgsd-integration-checker"`. Execute-plan uses `subagent_type="qgsd-executor"`.
   - What's unclear: There is no `qgsd-plan-milestone-gaps` subagent type. Should the task use `subagent_type="general"` or a specific type?
   - Recommendation: Use `subagent_type="general"` with an execution_context loading the plan-milestone-gaps.md workflow file. This follows the same pattern as how the qgsd:plan-milestone-gaps command file loads `@~/.claude/qgsd/workflows/plan-milestone-gaps.md`.

3. **Whether to include `model=` in the plan-milestone-gaps Task spawn**
   - What we know: The integration checker Task uses `model="{integration_checker_model}"` from `resolve-model`. The executor Task uses `model="{executor_model}"`.
   - What's unclear: Is there a `qgsd-plan-milestone-gaps` model role registered in gsd-tools?
   - Recommendation: Omit `model=` parameter and let the system use default. Plan-milestone-gaps is a planning/writing operation, not a specialized checker. If a model role doesn't exist, `resolve-model` will error.

# Research: Phase v0.9-01 — Context Window Monitor

**Phase Goal:** Create a PostToolUse hook that injects WARNING/CRITICAL into `additionalContext` at configurable thresholds.

## Requirements Coverage

- **CTX-01**: `hooks/gsd-context-monitor.js` created, registered as PostToolUse in `bin/install.js`
- **CTX-02**: Injects WARNING when `used_pct >= warn_pct` (default 70%)
- **CTX-03**: Injects CRITICAL when `used_pct >= critical_pct` (default 90%)
- **CTX-04**: Thresholds configurable via `qgsd.json` `context_monitor.warn_pct` / `context_monitor.critical_pct`, two-layer merge applied
- **CTX-05**: File copied to `hooks/dist/`, global install sync run

---

## Reference Implementation Analysis

A reference exists at `~/.claude/hooks/gsd-context-monitor.js` (upstream GSD 1.20.6). QGSD must adapt it with key differences.

### Upstream approach (GSD 1.20.6)
- Reads metrics from a bridge file: `/tmp/claude-ctx-{session_id}.json`
- Statusline hook writes this file; context monitor reads it
- Thresholds are hardcoded: WARNING at ≤35% remaining, CRITICAL at ≤25% remaining
- Debounce: 5 tool uses between warnings
- Uses "remaining_percentage" framing (35% remaining = 65% used)

### QGSD approach (from STATE.md roadmap decision)
- **Reads `context_tokens_used` and `context_tokens_max` directly from hook input** (not a bridge file)
- This is the architecturally cleaner path — PostToolUse input provides context metrics natively
- Thresholds configurable via `qgsd.json` `context_monitor` block, two-layer merge
- Default warn_pct: 70% (used), critical_pct: 90% (used)
- The success criteria says "warn threshold (default 70%)" and "critical threshold (default 90%)"

## PostToolUse Hook Input Schema

From Claude Code docs and the circuit breaker hook (which handles PostToolUse):

```json
{
  "hook_event_name": "PostToolUse",
  "session_id": "...",
  "tool_name": "...",
  "tool_input": {...},
  "tool_response": {...},
  "context_window": {
    "context_window_used": 12345,
    "context_window_size": 200000,
    "remaining_percentage": 93.8
  }
}
```

The statusline hook (`qgsd-statusline.js`) already reads `data.context_window?.remaining_percentage` from its input, confirming this field structure.

**Key fields:**
- `data.context_window.context_window_used` — tokens used
- `data.context_window.context_window_size` — total window size
- `data.context_window.remaining_percentage` — remaining %

Used percentage = `100 - remaining_percentage`

## Config-Loader Integration Pattern

The existing `config-loader.js` provides the two-layer merge pattern. To add `context_monitor`:

1. Add to `DEFAULT_CONFIG` in `config-loader.js`:
   ```js
   context_monitor: {
     warn_pct: 70,    // inject WARNING when used% >= this
     critical_pct: 90, // inject CRITICAL when used% >= this
   },
   ```

2. Add validation in `validateConfig()` for the `context_monitor` sub-object.

3. `gsd-context-monitor.js` calls `loadConfig(cwd)` and reads `config.context_monitor`.

**Alternative: standalone config read** — The hook can do its own config read without touching `config-loader.js`. This is simpler for a single-plan phase since the test suite doesn't need to be updated for config-loader validation. However, requirements say "two-layer merge applies", which means we must use the shared config loader.

**Decision: Use `loadConfig()` from config-loader.js.**

## Hook Output Mechanism

From `qgsd-prompt.js` (existing pattern):
```js
process.stdout.write(JSON.stringify({
  hookSpecificOutput: {
    hookEventName: "PostToolUse",
    additionalContext: message
  }
}));
```

The upstream reference confirms this exact pattern. The hook must:
- Output nothing (exit 0) when below thresholds
- Output the JSON above with the warning/critical message when threshold is exceeded

## Installer Integration

From `bin/install.js` analysis (lines ~1816-1838 pattern), registering a PostToolUse hook follows the same pattern as existing hooks:

```js
// Register QGSD context monitor hook (PostToolUse)
if (!settings.hooks.PostToolUse) settings.hooks.PostToolUse = [];
const hasContextMonitorHook = settings.hooks.PostToolUse.some(entry =>
  entry.hooks && entry.hooks.some(h => h.command && h.command.includes('gsd-context-monitor'))
);
if (!hasContextMonitorHook) {
  settings.hooks.PostToolUse.push({
    hooks: [{ type: 'command', command: buildHookCommand(targetDir, 'gsd-context-monitor.js') }]
  });
  console.log(`  ${green}✓${reset} Configured QGSD context monitor hook (PostToolUse)`);
}
```

Uninstall path also needs a corresponding `PostToolUse` cleanup block (mirrors PreToolUse cleanup at ~line 1160).

**Note:** `~/.claude/settings.json` already has `gsd-context-monitor.js` registered (from the upstream GSD install). The installer idempotency check prevents double-registration.

5. `cp hooks/config-loader.js hooks/dist/`
6. `node bin/install.js --claude --global`

## Debounce Consideration

The upstream uses a 5-call debounce via a separate tmp file. The QGSD requirements do NOT mention debounce — they only require the injection to happen "when context usage exceeds the threshold." For correctness with the success criteria test (which checks that `additionalContext` contains the message), no debounce is the right default. A simple stateless implementation is safer for a v1 hook.

**Decision: No debounce in v1.** The hook is stateless — fires on every PostToolUse call when above threshold. This satisfies the success criteria cleanly. Debounce can be added as a quick task if spamminess becomes an issue.

## Cwd Handling for Config Load

PostToolUse input includes `data.cwd` (confirmed from circuit breaker hook which uses it). The hook should pass `data.cwd || process.cwd()` to `loadConfig()` for the project-layer config to be read correctly.

## Summary: What to Build

**`hooks/gsd-context-monitor.js`**:
- Read stdin JSON (PostToolUse payload)
- Extract `data.context_window?.remaining_percentage` → compute `used_pct = 100 - remaining`
- Call `loadConfig(data.cwd || process.cwd())` to get `context_monitor.warn_pct` / `context_monitor.critical_pct`
- If `used_pct >= critical_pct`: output CRITICAL additionalContext
- Else if `used_pct >= warn_pct`: output WARNING additionalContext
- Else: exit 0
- Fail-open on all errors

**`config-loader.js` updates**:
- Add `context_monitor: { warn_pct: 70, critical_pct: 90 }` to `DEFAULT_CONFIG`
- Add validation in `validateConfig()`: check both are integers 1-100, warn_pct < critical_pct

**`bin/install.js` updates**:
- Register PostToolUse hook for `gsd-context-monitor.js` (idempotent)
- Add PostToolUse cleanup in uninstall path

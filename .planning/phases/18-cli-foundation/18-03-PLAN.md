**spawnToFile helper — file-based output capture via spawn (NOT spawnSync, NOT execSync):**

```javascript
const { spawn } = require('child_process');

function spawnToFile(cmd, cmdArgs, opts, outputPath) {
  return new Promise((resolve, reject) => {
    const outStream = fs.createWriteStream(outputPath);
    const proc = spawn(cmd, cmdArgs, {
      cwd: opts.cwd || process.cwd(),
      env: opts.env || process.env,
      stdio: ['ignore', 'pipe', 'pipe'],
    });

    proc.stdout.pipe(outStream);
    proc.stderr.pipe(outStream);  // capture both to same file

    let timedOut = false;
    const timer = opts.timeoutMs ? setTimeout(() => {
      timedOut = true;
      proc.kill('SIGTERM');
    }, opts.timeoutMs) : null;

    proc.on('close', (code) => {
      if (timer) clearTimeout(timer);
      outStream.end(() => {
        resolve({ exitCode: code, timedOut, outputPath });
      });
    });

    proc.on('error', reject);
  });
}
```

Note: `cmdMaintainTestsRunBatch` must be async because it uses this Promise-based helper. The main() dispatch must use `await` when calling it — update the maintain-tests case to: `await cmdMaintainTestsRunBatch(cwd, options, raw)`.

**Per-framework invocation (single-file targeting):**

Each test file in the batch is run individually (not all at once) to enable per-file pass/fail tracking:

- Jest: `npx jest <testFilePath> --json --outputFile <tempOutputFile> --passWithNoTests --forceExit`
  - Read `<tempOutputFile>` after run. Parse JSON: `testResults[0].testResults` for per-test results (status: "passed"/"failed"/"pending").
  - If `--json` and `--outputFile` both present, Jest writes structured JSON to the output file.
  - If file doesn't exist after run (Jest error before test execution), treat the whole file as failed.

- Playwright: `npx playwright test <testFilePath> --reporter=json`
  - Playwright writes JSON to stdout. Since we capture stdout to a temp file, read and parse the temp file.
  - Extract `suites[*].specs[*].tests[*].results[*].status` ("passed"/"failed"/"skipped").

- Pytest: `python -m pytest <testFilePath> --json-report --json-report-file=<tempOutputFile> -q`
  - Requires pytest-json-report plugin. If not available, fall back to: parse `pytest <testFilePath> -v` stdout for lines matching `PASSED`/`FAILED`/`SKIPPED`.
  - Fallback parsing: each line of the form `path::test_name PASSED` etc.

**Per-test result record format:**
```json
{
  "file": "/abs/path/to/test.js",
  "runner": "jest",
  "status": "passed|failed|skipped|error|timeout",
  "duration_ms": 1234,
  "error_summary": "first 500 chars of error output only (if failed)",
  "flaky": false,
  "flaky_pass_count": 0
}
```

**3-run flakiness pre-check — EXEC-04:**

After collecting the initial batch results, for each test with `status: "failed"`:
1. Re-run the test file in isolation 3 times (using the same per-framework invocation).
2. Track how many of the 3 re-runs pass.
3. If pass_count >= 1: set `flaky: true`, `flaky_pass_count: <count>`, `status: "flaky"` — this test is PRE-CLASSIFIED as flaky and excluded from the AI categorization queue.
4. If pass_count == 0: test remains `status: "failed"` — it is a confirmed failure for AI categorization.

The 3-run check is sequential (not parallel) for each failing test — total reruns = count(failures) * 3.

**Batch-level timeout:** The `--timeout N` flag (default 300 seconds = 5 minutes) applies to the ENTIRE batch execution (all files + all flakiness reruns), not per-file. If the batch timeout is exceeded mid-batch, record remaining unexecuted files as `status: "timeout"` and return partial results with `batch_timed_out: true`.

**Environment passthrough:** Merge `process.env` with any `--env KEY=VALUE` pairs provided. Use the merged env object in `spawnToFile` opts.

**Output format (to stdout or --output-file):**
```json
{
  "batch_id": 1,
  "runner": "jest",
  "executed_count": 100,
  "passed_count": 87,
  "failed_count": 8,
  "skipped_count": 3,
  "flaky_count": 2,
  "timeout_count": 0,
  "batch_timed_out": false,
  "results": [
    {
      "file": "/abs/path/to/test.js",
      "runner": "jest",
      "status": "passed",
      "duration_ms": 234,
      "error_summary": null,
      "flaky": false,
      "flaky_pass_count": 0
    }
  ]
}
```

**Temp file management:** Use `os.tmpdir()` for temp output files. Name format: `maintain-tests-batch-<batch_id>-<timestamp>.tmp`. Delete temp file after reading. If deletion fails, log warning but do not error.

**CRITICAL:**
- Use `spawn` (async, streaming) NOT `spawnSync` (synchronous, buffers all output in memory) for test runner invocation. This prevents the Node.js maxBuffer crash on large jest JSON output.
- Per-file execution: DO NOT run all files in one `jest <file1> <file2> ... <file100>` call. Run each file individually for per-file result isolation. This is slower but required for accurate per-file pass/fail tracking.
- Store only the first 500 chars of error output in the result JSON — do not store full test output (gigabyte problem at scale).

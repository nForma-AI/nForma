Add a `describe('maintain-tests integration — pytest parametrized ID parsing', ...)` block.

**Context:** pytest --collect-only -q outputs test IDs in format `path/to/test.py::test_name[param1-param2]`. The parameter part contains brackets and may contain special characters including spaces, equal signs, and nested brackets. If the discover command parses these IDs incorrectly (e.g., splitting on `[` instead of `::` for file path extraction), parametrized tests map to wrong file paths.

**Test the internal pytest output parser directly:**

The parser should: split each non-empty line on `::`, take index 0 as the file path, resolve to absolute path relative to cwd, deduplicate the result.

**Fixture pytest output (simulate `pytest --collect-only -q` stdout):**

```
tests/test_math.py::test_add[1+2-3]
tests/test_math.py::test_add[2+3-5]
tests/test_math.py::test_multiply[2*3-6]
tests/test_string.py::test_concat[hello-world-helloworld]
tests/test_string.py::test_upper[abc-ABC]
PASSED 5 items
```

Expected file paths after parsing: `["tests/test_math.py", "tests/test_string.py"]` (2 unique files, despite 5 test IDs)

**TC-PYTEST-1:** Basic parametrized ID parsing
- Pass the fixture output string to the internal pytest parser function (or test via `--runner pytest` with a mock script that emits this output)
- Assert: result contains exactly 2 unique file paths (test_math.py and test_string.py)
- Assert: no path contains `[` or `]` characters (brackets are from test params, not file paths)

**TC-PYTEST-2:** Special characters in parameters don't corrupt file path

**TC-PYTEST-4:** Empty pytest output → empty test_files array

  <name>Task 3: Buffer overflow regression test and full suite gate</name>
  <files>get-shit-done/bin/gsd-tools.test.cjs</files>
  <action>
Add a `describe('maintain-tests integration — buffer overflow regression', ...)` block, then run the full test suite as the final gate.

**Context:** Node.js `spawnSync` and `execSync` have a default maxBuffer of 1MB (1024 * 1024 bytes). When jest's `--json` output for a batch of 100 tests exceeds this (common when tests have long error messages), the child process is killed with `ENOBUFS` and the error is silently swallowed. Plan 18-03 uses `spawn` with file-based output capture to avoid this. This regression test verifies the fix.

**TC-BUFFER-1:** Large output does not crash the spawn helper
- Create a Node.js script at a temp path that outputs >1MB to stdout:
  ```javascript
  // large-output.js
  const line = 'x'.repeat(1000) + '\n';
  for (let i = 0; i < 2000; i++) process.stdout.write(line); // 2MB output
  ```
- Call `spawnToFile('node', ['/tmp/large-output.js'], { cwd, timeoutMs: 10000 }, '/tmp/buffer-test-output.tmp')`
- Assert: spawnToFile resolves (does NOT throw/reject)
- Assert: the output file exists and size > 1MB
- Assert: exitCode is 0
- Cleanup: delete temp files

**TC-BUFFER-2:** run-batch with a batch containing tests that produce large stderr
- Create a minimal batch manifest with 1 file pointing to a fixture script that writes >1MB to stderr
- Run `maintain-tests run-batch --batch-file <manifest> --output-file /tmp/run-batch-buffer-test.json`
- Assert: command completes without crash (exit code 0 or JSON output present)
- Assert: `error_summary` in results is at most 500 chars (truncation working)
- Note: This test exercises the end-to-end path from run-batch through spawnToFile with large output

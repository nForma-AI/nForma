# Phase v0.18-04: Adaptive Fan-Out - Research

**Researched:** 2026-02-27
**Domain:** Quorum dispatch optimization via risk-level-driven slot-worker count adaptation
**Confidence:** HIGH

## Summary

Phase v0.18-04 implements adaptive quorum fan-out driven by task envelope risk_level. The phase foundation was laid by v0.18-03 (envelope creation and risk_level read in quorum.md), and v0.18-04 builds the decision logic to vary slot-worker dispatch count based on risk. Current baseline: all quorum rounds dispatch the full `max_quorum_size` (default 3) external models. The new behavior: routine tasks dispatch 2, medium tasks dispatch 3, high-risk tasks dispatch the full pool. The injection point is **qgsd-prompt.js** (where `--n N` is emitted), and the verification gate is **qgsd-stop.js** (where the ceiling count is checked). The priority chain is strict: **--n N user flag > risk_level from envelope > default max_quorum_size > available pool size**.

**Primary recommendation:** Implement adaptive fan-out logic in qgsd-prompt.js by reading risk_level from context YAML (passed by plan-phase.md §8.5). Use risk_level to compute fan-out count (2/3/max), inject `--n N` into the dynamic step list, and pass through to qgsd-stop.js. When fan-out is below max_quorum_size, emit R6.4 reduced-quorum note in quorum.md output.

## Architecture Patterns

### Pattern 1: Risk-Level-Driven Dispatch Count (FAN-01..FAN-03)

**What:** Read risk_level from envelope context (passed by plan-phase.md), map to dispatch count, emit --n N.

**Mapping:**
```
risk_level: "routine"  → fan_out_count = 2   (minimum quorum: Claude + 1 external)
risk_level: "medium"   → fan_out_count = 3   (balanced quorum: Claude + 2 external)
risk_level: "high"     → fan_out_count = max_quorum_size (full pool, default 3)
risk_level: undefined  → fan_out_count = max_quorum_size (fail-open to conservative)
```

**When to use:** In qgsd-prompt.js, after parsing config.quorum_active and before generating dynamic step list.

**Code location:** hooks/qgsd-prompt.js lines ~190-210 (where maxSize is currently computed).

### Pattern 2: --n N as Fan-Out Emission (FAN-04)

**What:** Emit `--n N` flag in dynamic step generation so Stop hook reads it from transcript.

**Current flow:**
1. qgsd-prompt.js computes maxSize from config
2. Generates dynamic steps with Task(...prompt="slot: ...\nround: 1\n...")
3. Adds quorum context notes below step list

**New flow:**
1. qgsd-prompt.js reads risk_level from context YAML (RISK_LEVEL variable set by quorum.md)
2. Computes fan_out_count from risk_level (or defaults to max_quorum_size if absent)
3. Emits `--n <fan_out_count>` in the instructions text BEFORE the step list
4. qgsd-stop.js parses transcript for `--n N` flag (already has parseQuorumSizeFlag function)
5. Verifies correct ceiling is met (N = Claude + N-1 external models)

**Why this works:**
- `--n N` is already used for explicit user override (e.g., `/qgsd:plan-phase --n 5`)
- Same parsing logic (parseQuorumSizeFlag) reads it from transcript
- Stop hook's maxSize computation (line 486) already respects `--n N`

### Pattern 3: Context YAML Injection (Risk Level Handoff)

**What:** quorum.md sets RISK_LEVEL environment variable; qgsd-prompt.js reads it from context.

**Current (Phase 03) flow:**
- plan-phase.md §8.5 sets `ENVELOPE_PATH="$PWD/..."`
- plan-phase.md passes `envelope_path: <path>` in context YAML to quorum.md
- quorum.md reads envelope and sets RISK_LEVEL variable (lines 194-216)
- RISK_LEVEL available for Phase 04 adaptive fan-out logic

**Phase 04 enhancement:**
- qgsd-prompt.js reads RISK_LEVEL from context YAML (passed by orchestrator)
- Maps RISK_LEVEL to fan_out_count
- Emits `--n <fan_out_count>` in instructions

**Edge case:** RISK_LEVEL not in context YAML (e.g., when quorum.md hasn't run yet)
- Default to max_quorum_size (fail-open, conservative)
- No blocking — system works with or without envelope

### Pattern 4: Reduced-Quorum Note Emission (FAN-05)

**What:** When fan_out_count < max_quorum_size, quorum.md logs a reduced-quorum note per R6.4.

**Current behavior:**
- quorum.md reads risk_level from envelope (lines 208-209 output)
- Logs: "Envelope risk_level: <risk_level>"

**New behavior:**
- After computing fan_out_count from risk_level, quorum.md logs:
  ```
  Envelope risk_level: <risk_level> → fan-out: 2 workers (reduced from max 3)
  [R6.4 reduced-quorum note: Operating with 3 total participants (Claude + 2 external); max_quorum_size is 3]
  ```
- R6.4 note appears in quorum output alongside the risk_level log
- Helps user understand why fewer models are participating

**Where:** quorum.md, immediately after risk_level read section (lines 217-220, new content block).

### Pattern 5: Priority Chain (FAN-06 Supremacy)

**What:** User's `--n N` flag always wins. Cannot be overridden by envelope.

**Priority order (highest to lowest):**
1. **User explicit `--n N` flag** (from prompt text) — read by qgsd-prompt.js and qgsd-stop.js
2. **Envelope risk_level** (from task-envelope.json) — read by quorum.md and used by qgsd-prompt.js
3. **config.quorum.maxSize** (from qgsd.json) — fallback for max quorum size
4. **Available pool size** (count of quorum_active slots) — hard limit on what can be dispatched

**How it works:**
- Line 1: parseQuorumSizeFlag(prompt) in qgsd-prompt.js (line 172) captures user --n N first
- Line 2-3: If no user override (quorumSizeOverride === null), use risk_level-derived fan_out_count
- Line 3-4: If risk_level absent or invalid, fall back to maxSize from config
- Line 4: Never dispatch more than available pool size

**Code:** qgsd-prompt.js lines ~190-210 (current maxSize computation) becomes tiered:
```javascript
let fanOutCount = maxSize; // default to config's maxSize
if (quorumSizeOverride === null && riskLevelFromContext) {
  fanOutCount = mapRiskLevelToCount(riskLevelFromContext, maxSize);
}
// fanOutCount now represents the adaptive count
// externalSlotCap = (quorumSizeOverride !== null ? quorumSizeOverride - 1 : fanOutCount)
```

## Common Pitfalls

### Pitfall 1: Risk Level Not Available at Prompt Hook Time

**What goes wrong:** qgsd-prompt.js tries to read envelope before quorum.md has a chance to compute risk_level. Prompt hook (UserPromptSubmit) fires before quorum dispatch, so no RISK_LEVEL variable exists yet.

**Why it happens:** Prompt hook runs immediately when user types the command. Quorum.md is Claude's response — it happens after Hook executes.

**How to avoid:** qgsd-prompt.js receives risk_level via context YAML passed from orchestrator or prior quorum round. If absent, default to max_quorum_size (fail-open). Do NOT try to read envelope directly in prompt hook.

**Warning signs:** qgsd-prompt.js trying to call fs.readFileSync on task-envelope.json. Instead, risk_level should come from context YAML block passed in the input object.

### Pitfall 2: Confusing N (Total Participants) with N-1 (External Slots)

**What goes wrong:** Code emits `--n 3` meaning 3 external slots, but Stop hook interprets --n 3 as 3 total (Claude + 2 external).

**Why it happens:** qgsd-prompt.js uses "N total" semantics (Claude + N-1 external). qgsd-stop.js uses same semantics. Must be consistent.

**How to avoid:** Use single definition: `--n N means N total participants = Claude + (N-1) external models`. Emit --n based on this definition in both hooks.

**Verification:** In qgsd-stop.js line 486-490:
```javascript
const maxSize = quorumSizeOverride !== null && quorumSizeOverride > 1
  ? quorumSizeOverride - 1  // --n N means N-1 external models required
  : (config.quorum && Number.isInteger(config.quorum.maxSize) && config.quorum.maxSize >= 1)
    ? config.quorum.maxSize
    : 2;
```
This confirms: `--n N` = N-1 external models required. Match this in qgsd-prompt.js step generation.

### Pitfall 3: Envelope Risk Level as Hard Constraint (Violates R6)

**What goes wrong:** Phase implements logic that blocks quorum if risk_level is absent or malformed.

**Why it happens:** Treating envelope as mandatory constraint instead of advisory optimization.

**How to avoid:** Risk_level is ALWAYS fail-open. If risk_level is absent, invalid, or unparseable:
1. Log the issue (not an error — just informational)
2. Default to max_quorum_size (conservative fallback)
3. Never block the user or throw an exception
4. This matches Phase 03's envelope read pattern (lines 210-216 of quorum.md)

**Test case:** Manually delete task-envelope.json and run /qgsd:plan-phase. System should still work, using full max_quorum_size.

### Pitfall 4: Forgetting to Update Ceiling Ceiling Verification in Stop Hook

**What goes wrong:** qgsd-prompt.js emits `--n 2` (2 external slots), but qgsd-stop.js still checks for maxSize=3 (3 external), causing a false block.

**Why it happens:** Prompt and Stop hooks are independently maintained. Easy to forget that one change requires updating both.

**How to avoid:** Ensure Stop hook reads `--n N` from transcript and computes ceiling as N-1. Already implemented (qgsd-stop.js line 455): `const quorumSizeOverride = parseQuorumSizeFlag(promptText);`. This correctly overrides maxSize.

**Verification test:** Run `/qgsd:plan-phase --n 2 --with-envelope-routine` and verify Stop hook accepts exactly 1 successful external model call, not 2.

### Pitfall 5: Reduced-Quorum Note Duplication

**What goes wrong:** Reduced-quorum note appears in both qgsd-prompt.js instructions AND quorum.md output, confusing user about why two notes exist.

**Why it happens:** Phase implements logging in both places without coordinating.

**How to avoid:** Emit reduced-quorum note ONLY in quorum.md output (after risk_level read section). qgsd-prompt.js should NOT emit a note; it just emits `--n N` silently. User sees note once, from authoritative source (quorum.md).

**Pattern:** qgsd-prompt.js is context-injection (silent); quorum.md is output (visible). Keep separation of concerns.

### Example 1: Read Risk Level from Envelope (quorum.md Pattern)

Source: `/Users/jonathanborduas/code/QGSD/commands/qgsd/quorum.md` lines 194-216

```bash
# Parse envelope_path from context (passed by plan-phase.md)
ENVELOPE_PATH=$(echo "${CONTEXT_YAML:-}" | grep '^envelope_path:' | sed 's/envelope_path:[[:space:]]*//')

if [ -n "$ENVELOPE_PATH" ] && [ -f "$ENVELOPE_PATH" ]; then
  RISK_LEVEL=$(node -e "
    try {
      const e = JSON.parse(require('fs').readFileSync(process.argv[1], 'utf8'));
      const valid = ['low','medium','high'];
      const r = e.risk_level;
      if (valid.includes(r)) { console.log(r); } else { process.stderr.write('[quorum] envelope risk_level=' + r + ' (invalid) → using medium\n'); console.log('medium'); }
    } catch(err) {
      process.stderr.write('[quorum] envelope parse error → using medium\n');
      console.log('medium');
    }
  " "$ENVELOPE_PATH" 2>&1)
  echo "Envelope risk_level: ${RISK_LEVEL}"
else
  RISK_LEVEL="medium"
  if [ -n "$ENVELOPE_PATH" ]; then
    echo "Envelope not found at ${ENVELOPE_PATH} — using default risk (medium)"
  else
    echo "No envelope_path provided — using default risk (medium)"
  fi
fi
```

**Why this pattern:** Fail-open behavior (never blocks), atomic JSON parse, clear logging. Reuse this exact pattern in Phase 04 for reading risk_level in qgsd-prompt.js.

### Example 2: Parse --n N Flag (qgsd-prompt.js and qgsd-stop.js)

Source: `/Users/jonathanborduas/code/QGSD/hooks/qgsd-prompt.js` lines 106-113 and `/Users/jonathanborduas/code/QGSD/hooks/qgsd-stop.js` lines 364-371

```javascript
// Parses --n N flag from a prompt string.
// Returns N (integer >= 1) if found, or null if absent/invalid.
function parseQuorumSizeFlag(prompt) {
  const m = prompt.match(/--n\s+(\d+)/);
  if (!m) return null;
  const n = parseInt(m[1], 10);
  return (Number.isInteger(n) && n >= 1) ? n : null;
}
```

**Why this pattern:** Already exists in both hooks. Reuse without modification. Guarantees consistent parsing across prompt and stop.

### Example 4: Validate Risk Level (task-envelope.cjs Pattern)

Source: `/Users/jonathanborduas/code/QGSD/bin/task-envelope.cjs` lines 23-50

```javascript
// Validate envelope against schema
function validateEnvelope(obj) {
  const errors = [];

  if (!obj || typeof obj !== 'object') {
    errors.push('Envelope must be an object');
    return { valid: false, errors };
  }

  // Check schema_version
  if (typeof obj.schema_version !== 'string' || obj.schema_version !== '1') {
    errors.push('schema_version must be string "1"');
  }

  // Check risk_level
  if (!obj.risk_level || !['low', 'medium', 'high'].includes(obj.risk_level)) {
    errors.push('risk_level must be one of: low, medium, high');
  }

  return {
    valid: errors.length === 0,
    errors
  };
}
```

**Why this pattern:** Task-envelope.cjs already validates risk_level enum. Phase 04 uses same three values. No new validation needed.

1. **RISK_LEVEL variable scope in quorum.md**
   - What we know: quorum.md sets RISK_LEVEL as a Bash variable after envelope read (line 208)
   - What's unclear: Is RISK_LEVEL available to subsequent commands in the same quorum.md block? Or must it be passed via context to next round?
   - Recommendation: Test by running a quorum round and checking if RISK_LEVEL persists. If it doesn't, pass it as a context variable to deliberation rounds (add to quorum dispatch YAML).

2. **Fallback behavior when envelope absent on subsequent rounds**
   - What we know: Phase 03 plan-phase.md passes ENVELOPE_PATH to quorum.md only ONCE (§8.5)
   - What's unclear: On deliberation rounds, is ENVELOPE_PATH still available in context YAML? Or must Claude re-read it?
   - Recommendation: Verify that ENVELOPE_PATH persists across deliberation rounds (likely yes, same quorum run). If not, quorum.md must re-read it from the same phase directory.

3. **Performance impact of 2-worker routine tasks**
   - What we know: Routine (fan-out: 2) saves 1 external model call per round
   - What's unclear: Does 1-model reduction actually reduce token consumption measurably? Or does deliberation overhead dominate?
   - Recommendation: Measure token cost of a routine task with routine fan-out (2) vs medium fan-out (3). Compare to Phase v0.18-05 observability data.

4. **User communication of reduced quorum**
   - What we know: R6.4 reduced-quorum note will appear in quorum output
   - What's unclear: Should this be visible in Claude's output to the user? Or only in logs/debug?
   - Recommendation: Include reduced-quorum note in the visible quorum output (after "Envelope risk_level:" line). User should understand why fewer models are participating.

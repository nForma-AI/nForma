# Phase v0.39-03 Plan 03: Solution Simulation Loop and Workflow Integration

Implement the solution simulation loop orchestrator and integrate Phase 4.5 into the model-driven-fix workflow.

## Summary

This plan completed **SIM-04** (Solution Simulation Loop Orchestration) and integrated the three-module convergence verification pipeline into the model-driven-fix workflow as Phase 4.5.

The implementation provides:

1. **Solution Simulation Loop Module** (`bin/solution-simulation-loop.cjs`)
   - Orchestrates the full simulation pipeline: normalize fix intent → generate consequence model → run convergence gates
   - Iteration loop with automatic escalation on convergence, max iterations exhausted, or dependency failure
   - Displays iteration progress with per-gate pass/fail status (Invariants, Bug Resolved, Neighbors)
   - Session ID generation using `crypto.randomBytes(8)` for collision resistance (96-bit entropy)
   - Iteration history persisted to `.planning/formal/cycle2-simulations/{sessionId}/iteration-history.json`
   - Dependency injection pattern for testability (allows mock normalizer, generator, gate runner)
   - Reads `maxIterations` from `.planning/config.json` (default 3)
   - Handles dependency unavailability gracefully (HaikuUnavailableNoCorruption)

2. **Workflow Integration** (Phase 4.5 in `model-driven-fix.md`)
   - Added Phase 4.5 (Solution Simulation) between Phase 4 (Constraint Extraction) and Phase 5 (Constrained Fix)
   - Fail-CLOSED gating: module-not-found prompts user for explicit skip confirmation; runtime errors block progression
   - Explicit `--skip-simulation` flag for user opt-out (not silent)
   - Explicit `--fix-idea` flag to override fix idea input for simulation
   - Phase 5a updated to show simulation status caveat (converged/not converged)
   - Updated parse_arguments to handle new flags
   - Both workflow copies (`core/workflows/` and `commands/nf/`) kept in sync

## Key Design Decisions

1. **Async/Await for Gate Execution**
   - Solution simulation loop uses async/await to properly handle asynchronous gate runner
   - Allows proper handling of dependency failures mid-loop

2. **Iteration History Persistence**
   - History persisted to disk even when not converged
   - Enables future analysis of failed iterations and refinement patterns

3. **Session ID as Unique Identifier**
   - 8 bytes of random entropy (96-bit) provides collision resistance
   - Enables parallel simulation sessions without interference

4. **Fail-CLOSED Phase 4.5 in Workflow**
   - Module-not-found prompts user rather than silently skipping
   - Runtime errors block progression (user must investigate or explicitly --skip-simulation)
   - Rationale: Fixes must be verified in model space first; silent skip would violate core invariant

5. **Explicit --skip-simulation Flag**
   - User must explicitly opt out (not automatic)
   - Provides audit trail of intentional skips

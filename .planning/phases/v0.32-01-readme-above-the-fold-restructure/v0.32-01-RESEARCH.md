## Summary

This phase restructures the top portion of the nForma README so that a first-time visitor immediately understands what the project does, who benefits from it, how it compares to Claude Code alone, and what the TUI looks like -- all without scrolling or expanding collapsible sections. The current README buries the TUI inside a `<details>` block, has a one-sentence "Who This Is For", lacks any comparison or metrics section, and the nav bar omits key sections.

The work is pure Markdown editing in a single file (`README.md`). No code changes, no dependencies, no build steps. The primary risk is GitHub Markdown rendering quirks (image sizing, table formatting, anchor link behavior). The secondary risk is breaking existing content or links during restructuring.

**Primary recommendation:** Restructure README.md top-to-bottom for the above-the-fold content in a single plan, keeping all existing content intact but reorganized and augmented with new sections (TUI hero, expanded Who This Is For, With vs. Without table, By the Numbers, What's New, expanded nav bar).

### Pattern 1: Hero Screenshot Section
**What:** A visible (non-collapsed) section with a prominently displayed screenshot and brief description
**When to use:** When the visual is a key selling point that should not be hidden
**Example:**
```markdown
## Terminal UI

A full-featured keyboard-navigable terminal interface for managing your quorum agents.

![Agent Manager TUI](docs/assets/tui-agents.png)

| Module | What it does |
|--------|--------------|
| Agents | List, add, clone, edit, remove, reorder slots |
| Reqs | Browse requirements, check coverage, view traceability |
| Config | Provider keys, batch rotate, timeouts |
| Sessions | Spawn and manage embedded Claude Code sessions |
| Solve | Diagnose and remediate planning directory issues |
```

### Pattern 2: Problem-Framing Bullets
**What:** Bullet list where each item names a concrete problem the reader experiences
**When to use:** "Who This Is For" sections that need reader self-identification
**Example:**
```markdown
## Who This Is For

If you use Claude Code and have hit any of these walls:

- **Single-model blind spots** — One model misses edge cases another would catch
- **Context rot** — Quality degrades as your context window fills with accumulated garbage
- **Oscillation loops** — The agent ping-pongs between two files, fixing one and breaking the other
- **Manual tracking** — You're the one remembering what's done, what's next, and what broke

nForma fixes these structurally, not with better prompts.
```

### Pattern 3: Comparison Table
**What:** Two-column table with emoji indicators showing capability differences
**When to use:** "With vs. Without" or feature comparison sections
**Example:**
```markdown
## With vs. Without

| Capability | Claude Code Alone | With nForma |
|------------|:-----------------:|:-----------:|
| Planning reviewed before execution | Manual | 5-model quorum |
| Ping-pong loop detection | None | Automatic circuit breaker |
| Context window freshness | Degrades over session | Fresh 200k per task |
| Requirements traceability | Ad-hoc | Formal, auditable |
| Test coverage verification | Manual | Pre-execution test mapping |
```

### Pattern 4: Metrics Grid

**When to use:** "By the Numbers" or "at a glance" sections

### Anti-Patterns to Avoid
- **Hiding key visuals in `<details>` blocks:** GitHub renders collapsed content invisible; the TUI is a primary selling point and must be visible
- **Oversized hero images:** GitHub renders images at max-width:100% but very tall images push everything else below the fold; use HTML `<img>` with explicit width if needed
- **Anchor links to collapsed sections:** Clicking a nav link to a `<details>` section does NOT auto-expand it on GitHub; only link to visible headings
- **Emoji overuse in section headers:** Distracts from scannability; use sparingly or not at all (user has not requested emoji)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Badge generation | Custom SVG badges | shields.io | Already in use; consistent styling, auto-updating |
| Anchor link generation | Manual `<a name="">` tags | GitHub auto-generated heading anchors | GitHub auto-creates anchors from `## Heading Text` as `#heading-text` |
| Image sizing | CSS/style attributes | HTML `<img width="N">` | GitHub strips `style` attributes; only `width`/`height` survive sanitization |

**Key insight:** GitHub's Markdown sanitizer is aggressive. Stick to the subset that GitHub explicitly allows: `align`, `width`, `height` attributes on certain HTML elements. Do not attempt `style` attributes -- they are stripped.

## Common Pitfalls

### Pitfall 1: Broken Anchor Links After Section Rename
**What goes wrong:** Renaming a heading (e.g., "Agent Manager TUI" to "Terminal UI") breaks existing anchor links elsewhere in the repo or external links
**Why it happens:** GitHub auto-generates anchors from heading text; change the text, change the anchor
**How to avoid:** After renaming sections, grep the entire repo for old anchor references (`#agent-manager-tui`) and update them. Also check docs/USER-GUIDE.md for cross-references.
**Warning signs:** 404 on in-page links; links that scroll to wrong location

### Pitfall 2: Image Path References
**What goes wrong:** Moving the TUI section from inside Getting Started to a top-level section could inadvertently break relative image paths
**Why it happens:** Image paths in the current README are relative (`docs/assets/tui-agents.png`); these work regardless of position in the file, but copy-paste errors can introduce wrong paths
**How to avoid:** Verify every image path after restructuring by viewing the README on GitHub (or using `grip` locally)
**Warning signs:** Broken image icons on GitHub preview

### Pitfall 3: Nav Bar Anchor Mismatch
**What goes wrong:** Nav bar links point to anchors that don't match the actual heading text
**Why it happens:** GitHub generates anchors by lowercasing, replacing spaces with hyphens, and removing special characters. "What's New in v0.32" becomes `#whats-new-in-v032`
**How to avoid:** Use simple heading text without special characters, or verify anchors by checking GitHub rendering
**Warning signs:** Nav links that scroll to the top of the page instead of the target section

### Pitfall 4: Details/Summary Content Loss During Extraction
**What goes wrong:** When extracting TUI content from `<details>` to a visible section, some content or formatting is accidentally omitted
**Why it happens:** `<details>` blocks in the current README contain complex nested content (images, tables, notes)
**How to avoid:** Extract the full content, compare line counts, verify all images and tables render
**Warning signs:** Missing module table, missing screenshots, missing notes

### Pitfall 5: Changelog Section Staleness
**What goes wrong:** "What's New in v0.32" becomes stale when v0.33 ships
**Why it happens:** Static changelog sections require manual updates
**How to avoid:** Design the section to be easily replaceable -- keep it short (3-5 bullets), clearly version-labeled, and positioned so replacement is a simple find-and-replace operation
**Warning signs:** README showing outdated version highlights months after new release

## Open Questions

1. **Hero image sizing**
   - What we know: `tui-agents.png` exists and is currently used inside the collapsed TUI section
   - What's unclear: Whether the image dimensions are appropriate for a hero placement (could be too large/small)
   - Recommendation: Use HTML `<img>` tag with explicit `width` attribute (e.g., `width="700"`) to control size. Test on GitHub.

2. **Section ordering above the fold**
   - What we know: Requirements specify TUI, Who This Is For, With vs. Without, By the Numbers, What's New, and Nav bar
   - What's unclear: Exact ordering between sections (should TUI come before or after Who This Is For?)
   - Recommendation: Order by engagement: Nav bar (orientation) -> TUI hero (visual hook) -> Who This Is For (problem identification) -> With vs. Without (value proposition) -> By the Numbers (credibility) -> What's New (freshness signal). This follows the pattern: show -> explain -> compare -> prove -> update.

3. **What's New content for v0.32**
   - What we know: v0.32 is a docs/README overhaul milestone
   - What's unclear: Whether to highlight v0.32 features specifically or recent features across v0.31/v0.32
   - Recommendation: Highlight the most impactful recent additions across v0.31-v0.32 to make the section immediately valuable. Include TUI Solve module, gate scoring, and the README overhaul itself.

4. **Remaining TUI content in Getting Started**
   - What we know: The `<details>` Agent Manager TUI block in Getting Started has launch instructions (`node bin/nForma.cjs`) plus screenshots
   - What's unclear: Whether to fully remove the Getting Started TUI block or leave a brief reference pointing to the new top-level section
   - Recommendation: Replace the Getting Started `<details>` block with a brief cross-reference: "See [Terminal UI](#terminal-ui) above." Keep launch instructions in the new visible section.

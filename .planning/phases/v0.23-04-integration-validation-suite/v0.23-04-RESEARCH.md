# Phase v0.23-04: Integration Validation Suite - Research

**Researched:** 2026-03-02
**Domain:** Integration testing, formal verification tool chaining, test scaffolding (TDD Wave 0)
**Confidence:** HIGH

## Summary

Phase v0.23-04 must create an integration test suite that proves the formal gates chain actually executes end-to-end with real tool output, not mocked behavior. The phase depends on v0.23-02 and v0.23-03 being complete — which means plan-phase can scan formal specs, execute-phase can run formal checks, and roadmapper can inject formal context. This phase' sole responsibility is validating that the entire pipeline works with real TLC/Alloy/PRISM stdout and exit codes.

Success requires three concretely provable facts:
1. A test runner script (`bin/test-formal-integration.cjs` or similar) executes end-to-end without mocking
2. The test validates the full chain: FORMAL_SPEC_CONTEXT population → run-formal-check execution → verifier receiving real tool output
3. All existing TLA+ specs pass TLC clean (no regressions from integration wiring)

The phase follows the project's TDD Wave 0 pattern: RED tests written first, then implementation to green them. Test files exist but fail until actual tools run with real output.

**Primary recommendation:** Use Node.js native `test` runner (already in use across project) with spawnSync for subprocess invocation. Write smoke tests (fail-open for missing Java/PRISM), structural tests (verify tool output is real, not mocked), and regression tests (all TLA+ specs TLC-clean). Test files scaffold in Phase 1; implementation follows in Phase 2+.

### Pattern: Real Tool Output Capture

Tests must prove that tools actually ran and produced output — not that a mock was called. Use spawnSync to capture stdout/stderr and validate against known patterns:

```javascript
// EXAMPLE: Validate TLC output contains expected properties
const result = spawnSync('java', [
  '-jar', 'formal/tla/tla2tools.jar', 'tlc2.TLC',
  '-config', 'formal/tla/MCliveness.cfg',
  'formal/tla/QGSDQuorum.tla'
], {
  encoding: 'utf8',
  timeout: 60000, // 60s — TLC can be slow
  cwd: process.cwd()
});

// Validate real TLC output, not mock
assert.strictEqual(result.status, 0 || 1, 'TLC exited with 0 (pass) or 1 (fail), not timeout/error');
assert.ok(result.stdout.includes('QGSDQuorum'), 'stdout mentions module name (proof TLC ran)');
assert.ok(result.stdout.includes('invariant'), 'stdout mentions a property being checked');
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Test runner framework | Custom Jest-like runner | Node.js native `node:test` + `assert` | Already used by 71 project tests; no npm dependencies; consistent with project style |
| Subprocess management | Manual fork/exec + output buffering | `child_process.spawnSync()` | Built-in; handles timeouts, signal forwarding, encoding cleanly; avoids deadlocks |
| Test result collection | Manual array + filtering | `node --test` native reporter | Automatically formats TAP/JSON output; integrates with npm scripts; deduplication logic built-in |
| TLA+ output parsing | Regex-based line splitters | Existing verify-formal-results.cjs NDJSON parser | Already handles NDJSON + counts grouping; fail-open on malformed lines; proven in production |
| Formal verification result format | Custom JSON schema | Existing check-result.schema.json (v2.1) | Already in use by run-formal-verify.cjs, write-check-result.cjs, and 21 callers; validates exit code semantics |

**Key insight:** The integration test is NOT a test of the test framework itself — it's a test that the integration chain works. Do not build test infrastructure; reuse native Node.js + proven existing parsers.

## Common Pitfalls

### Pitfall 1: Mocking Tool Output Instead of Running Real Tools

**What goes wrong:** Test passes in CI because it mocks `spawnSync` to return fake TLC output. Code ships. Later, real TLC run produces different output format. Verifier breaks. Integration test never catches the regression.

**Why it happens:** "Running real TLC is slow and requires Java installed; easier to mock." But this defeats the entire purpose of an integration test.

**How to avoid:**
- Use spawnSync WITHOUT stubbing/mocking. If Java not installed, skip the test (fail-open), don't mock.
- Validate stdout against ACTUAL TLC output patterns (search run-tlc.test.cjs for examples like `output.includes('QGSDQuorum')`)
- Use timeout controls to prevent hanging (60s for TLC is reasonable; fail-close on timeout)

**Warning signs:**
- Test passes even when Java is not installed
- Stdout validation is too loose (e.g., `assert.ok(result.stdout, 'has output')` instead of specific property names)
- No example TLC output in test comments

### Pitfall 2: Test Passes But Integration Still Broken

**What goes wrong:** Individual `bin/run-formal-check.cjs` works fine in the test. But when execute-phase calls it and tries to pass result to verifier, something breaks (missing env var, wrong JSON structure, path doesn't exist). Integration test never captured the hand-off.

**Why it happens:** Phase v0.23-04 only tests the isolated tools; doesn't test the execute-phase→verifier wiring that v0.23-02 adds.

**How to avoid:**
- Include a "full chain" structural test that simulates the execute-phase→verifier hand-off
- Verify FORMAL_CHECK_RESULT JSON structure (from run-formal-check) can be parsed by verify-formal-results.cjs
- Test verifier's ability to read and embed the result in VERIFICATION.md
- Document which file/env-var holds FORMAL_CHECK_RESULT and verify it's passed correctly

**Warning signs:**
- Test only validates run-formal-check in isolation
- No test of verifier receiving/parsing FORMAL_CHECK_RESULT
- VERIFICATION.md template never updated to show where real tool output goes

### Pitfall 3: Specs Fail TLC But Test Doesn't Notice

**What goes wrong:** A spec has a pre-existing error (invariant typo, malformed property, invalid config). Integration test runs but exits 1. Test interprets this as "test infrastructure works" (which it does — tool ran and returned 1). Doesn't distinguish "tool ran successfully and found a counterexample" from "spec syntax is broken."

**Why it happens:** TLC exit code 1 can mean both "counterexample found" and "error in config/spec". Test only checks "exit code is 0 or 1" without parsing stdout to understand why.

**How to avoid:**
- Validate stdout/stderr to understand WHY exit code is what it is
- For passing tests: stdout should include "No counterexample found" or "invariant proved"
- For failing tests: stdout should include "counterexample" or specific property name
- If spec has pre-existing errors: stdout includes "error", not "counterexample"
- Document expected stdout patterns for each module/property

**Warning signs:**
- Test only checks `assert.strictEqual(result.status, 0)` without examining stdout
- No comments showing example TLC output for pass/fail cases
- Test doesn't distinguish between "tool ran" and "tool ran and spec is valid"

### Pitfall 4: Fail-Open Not Actually Fail-Open

**What goes wrong:** Test is written to skip if Java not installed. But plan-phase/execute-phase don't skip properly — they crash. Integration test never ran the actual workflow, so it never caught this.

**Why it happens:** Test scaffold and implementation are disconnected. Test imagines graceful skip; workflow implementation doesn't have skip logic.

**How to avoid:**
- Integration test must match the actual skip logic in execute-phase
- If execute-phase returns error code, test should fail (not skip)
- Coordinate with v0.23-02 plan to ensure execute-phase actually implements skip logic (ENF-03)
- Test the ENTIRE skip path: execute-phase receives Java not installed → run-formal-check skips → verifier receives skip message → VERIFICATION.md shows "Skipped: Java not installed"

**Warning signs:**
- Test skips gracefully but execute-phase crashes
- Test doesn't verify verifier's handling of skip message
- VERIFICATION.md template has no "skipped" case

## Open Questions

1. **Should test-formal-integration.cjs be a single monolithic file or broken into multiple test files?**
   - What we know: Project currently has 71 separate `.test.cjs` files; npm test runs all of them
   - What's unclear: Whether to follow pattern (11 test files for 11 formal domains) or consolidate integration tests in one
   - Recommendation: Single `test-formal-integration.cjs` file for now; can split later if size exceeds 500 lines. Keep integration tests together for coherence.

2. **How should skip messages be formatted when Java is missing?**
   - What we know: ENF-03 requires graceful skip; run-formal-check.cjs should detect missing Java
   - What's unclear: Does run-formal-check exit 0 (skip) or 1 (error)? Is skip message in stdout, stderr, or exit code?
   - Recommendation: Align with execute-phase implementation (v0.23-02 should define this). Test should validate actual skip behavior, not assume format.

3. **Should FORMAL_CHECK_RESULT be passed as JSON file, env var, or stdout?**
   - What we know: Verifier needs real TLC output to embed in VERIFICATION.md
   - What's unclear: What's the hand-off mechanism between execute-phase and verifier?
   - Recommendation: Coordinate with v0.23-02 plan. Test should validate the actual mechanism (likely JSON file in .planning/ or env var injection).

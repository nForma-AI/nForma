## Key Design Decisions

1. **Flat keys, not nested:** model_tier_planner and model_tier_worker are root-level flat string keys, not nested under model_tiers object. Reason: shallow merge in config-loader.js would lose nested objects from global config when project config sets a tier value.

2. **Delete on validation failure:** Invalid tier values are deleted (not corrected to default). Reason: matches existing pattern for invalid model_preferences entries; user must explicitly set valid value or remove the key.

3. **Whitelist only:** ['haiku', 'sonnet', 'opus'] — no 'gpt-4', 'claude-3-opus', or other model names. Reason: tier keys are symbolic (cost category), not model IDs. Actual model ID resolution happens in gsd-tools.cjs resolveModelInternal().

4. **Stderr for warnings:** All validation warnings written to process.stderr, never stdout. Reason: hook config-loader.js must never touch stdout (used as decision channel by hooks). Existing pattern verified in TC8 and TC-CB7.

## Architecture Notes

**Validation order in validateConfig():**
1. Type check: `typeof config.model_tier_X !== 'string'` → delete + warn
2. Whitelist check: `!VALID_TIERS.includes(config.model_tier_X)` → delete + warn
3. Placement: after context_monitor validation, before return (lines 241-254)

**Matches existing config-loader patterns:**
- Validation pattern (type → range/whitelist → delete + warn) matches circuit_breaker, agent_config, model_preferences
- Stderr format '[qgsd] WARNING: qgsd.json: {key} ...; {action}' matches all existing warnings
- No new constants or helper functions — uses existing VALID_TIERS array

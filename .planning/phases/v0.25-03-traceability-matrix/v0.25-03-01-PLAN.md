  <action>
Create `bin/generate-traceability-matrix.cjs` with the following behavior:

**Data Loading (3 sources + 1 reference):**

1. **Annotations (primary):** Run `node bin/extract-annotations.cjs` via `child_process.spawnSync` and parse JSON stdout. This returns `{ model_file: [{ property, requirement_ids }] }` with 23 files, 93 properties, 89 unique requirement IDs.

2. **Model-registry (fallback):** Read `formal/model-registry.json` directly. Use `models[file].requirements` array when a file has no annotations. Model-registry has model-level (not property-level) requirement arrays.

3. **Check-results (status):** Read `formal/check-results.ndjson` line-by-line. Parse each JSON line for `{ check_id, result, requirement_ids, formalism, property, surface, tool }`. Match check results to properties using:
   - Parse check_id to extract model hint (e.g., `alloy:scoreboard-recompute` -> `formal/alloy/scoreboard-recompute.als`)
   - Match by requirement_ids overlap between check result and property
   - Store latest result per check_id (last entry wins if duplicates)

4. **Requirements (reference):** Read `formal/requirements.json` for total requirement count and IDs.

**Matrix Construction:**

Build two bidirectional indexes:

`requirements` object (keyed by requirement ID):
```json
{
  "STOP-01": {
    "id": "STOP-01",
    "properties": [
      {
        "model_file": "formal/tla/QGSDStopHook.tla",
        "property_name": "TypeOK",
        "source": "annotation",
        "latest_result": "pass",
        "check_id": "tla:stop-hook-safety"
      }
    ]
  }
}
```

`properties` object (keyed by `model_file::property_name`):
```json
{
  "formal/tla/QGSDStopHook.tla::TypeOK": {
    "model_file": "formal/tla/QGSDStopHook.tla",
    "property_name": "TypeOK",
    "requirement_ids": ["STOP-01"],
    "source": "annotation",
    "latest_result": "pass",
    "check_id": "tla:stop-hook-safety"
  }
}
```

**For annotation-sourced properties:** Each `{ property, requirement_ids }` from annotations becomes one property entry. The property appears in each of its requirement_ids' `requirements[id].properties` array.

**For model-registry fallback properties:** When a model file has NO annotations, use model-registry `requirements` array. Create a single property entry with `property_name: "(model-level)"`, `source: "model-registry"`. Log a warning to stderr: `[generate-traceability-matrix] warn: No annotations for ${file}, using model-registry fallback`.

**Check result matching:**

For each check result entry:
1. Parse `check_id` to derive a model file path hint:
   - `tla:*` -> look for matching TLA+ model based on check name
   - `alloy:*` -> look for `formal/alloy/${name}.als`
   - `prism:*` -> look for `formal/prism/${name}.pm` or `.props`
   - `ci:*` -> match by requirement_ids only
2. Match check result to properties by overlapping `requirement_ids`
3. Set `latest_result` and `check_id` on matched properties
4. Properties with no matching check result get `latest_result: null, check_id: null`

**Coverage Summary:**

```json
"coverage_summary": {
  "total_requirements": 210,
  "covered_count": 89,
  "coverage_percentage": 42.4,
  "uncovered_requirements": ["ACT-01", "ACT-02", ...],
  "orphan_properties": []
}
```

- `total_requirements`: count of all IDs in requirements.json
- `covered_count`: count of requirement IDs that appear in at least one property's requirement_ids
- `coverage_percentage`: `(covered_count / total_requirements) * 100`, rounded to 1 decimal
- `uncovered_requirements`: sorted list of requirement IDs with no property links
- `orphan_properties`: properties where requirement_ids is empty (should be 0 given v0.25-02)

**Metadata section:**
```json
"metadata": {
  "generated_at": "ISO-8601",
  "generator_version": "1.0",
  "data_sources": {
    "annotations": { "file_count": N, "property_count": N },
    "model_registry": { "file_count": N, "used_as_fallback": N },
    "check_results": { "entry_count": N }
  }
}
```

**Output:** Write JSON to `formal/traceability-matrix.json` with 2-space indentation. Print summary to stdout:
```
[generate-traceability-matrix] Generated formal/traceability-matrix.json
[generate-traceability-matrix]   Requirements: 89 covered / 210 total (42.4%)
[generate-traceability-matrix]   Properties: 93 (23 files)
[generate-traceability-matrix]   Orphan properties: 0
[generate-traceability-matrix]   Check results matched: N
```

**CLI flags:**
- `--json` — print matrix JSON to stdout instead of writing to file (useful for piping)
- `--quiet` — suppress summary output (exit code only)

**Exit code:** 0 always (matrix generation is informational, not a gate).

**Error handling:**
- If extract-annotations.cjs fails or returns non-JSON: log error, set annotations to `{}`, continue with model-registry only
- If check-results.ndjson doesn't exist or is empty: proceed with `latest_result: null` for all properties
- If requirements.json doesn't exist: set total_requirements to 0, log warning
- If model-registry.json doesn't exist: fatal error, exit 1 (this is the minimum required data source)

Use `#!/usr/bin/env node` shebang and `'use strict'`. Follow the project's existing bin/ script patterns (TAG constant, process.stdout.write for output, path.join for paths).

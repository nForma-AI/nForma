Add a new step entry to the `STEPS` array in `bin/run-formal-verify.cjs` for traceability matrix generation. The step must:

1. **Position:** Place it AFTER the triage bundle step (the last current step) as a new tool group `traceability`. This ensures check-results.ndjson is fully populated before the matrix reads it.

2. **Step definition:**
```javascript
// ─ Traceability matrix ─────────────────────────────────────────────────────
{
  tool: 'traceability', id: 'traceability:matrix',
  label: 'Generate traceability matrix (requirements <-> formal properties)',
  type: 'node', script: 'generate-traceability-matrix.cjs', args: ['--quiet'],
  nonCritical: true,
},
```

3. **nonCritical: true** — The matrix generation is informational. If it fails, it should appear as a warning in the summary table but NOT cause `run-formal-verify.cjs` to exit with a non-zero code. The existing `nonCritical` flag in the results logic already handles this.

4. **--quiet flag** — The generator's own summary output would be noise in the formal verification log. Use `--quiet` so the step just produces the file silently. The run-formal-verify.cjs summary table provides the pass/fail status.

5. **Update the file header comment** to include the traceability step in the coverage listing:
   - Add `Traceability (1) — generate-traceability-matrix.cjs (requirements <-> properties matrix)` to the Coverage section
   - Update the total step count from 30 to 31

6. **Verify `--only` filter does not break:** The step uses `tool: 'traceability'`, which means `--only=traceability` would run just this step. The existing `--only` filter logic uses `steps.filter(s => s.tool === only)`, which handles this correctly without changes.

7. **Execution ordering:** The traceability step's `tool: 'traceability'` is a new group. In the `runOnce()` function, tool groups run concurrently in Phase 2. But since the traceability step reads check-results.ndjson (populated by Phase 2 steps), it must run AFTER Phase 2.

   Check the current execution flow:
   - Phase 1: generate steps (sequential)
   - Phase 2: all other tool groups (concurrent)

   Add a Phase 3 for post-processing steps (traceability + triage). The triage bundle also reads check results, so it should also be in Phase 3. However, triage is currently in Phase 2 tool groups. The simplest approach: add the traceability step to the STEPS array with a special `phase: 3` marker, then update `runOnce()` to run phase-3 steps after Phase 2.

   Actually, reviewing the code more carefully: triage already works in Phase 2 because check results are populated during Phase 2 tool group execution (each tool group runs sequentially internally). The traceability step only needs check-results.ndjson to be fully written. Since Phase 2 uses `Promise.all` on tool groups (concurrent), and triage is the last tool group, the NDJSON file may not be complete when triage runs.

   **Simplest correct approach:** Add a Phase 3 block in `runOnce()` that runs after Phase 2 completes. Move only the traceability step to Phase 3. This is 3-4 lines of code:

```javascript
  // ── Phase 3: Post-processing (needs fully populated check-results.ndjson) ──
  const postSteps = steps.filter(s => s.tool === 'traceability');
  if (postSteps.length > 0) {
    process.stdout.write(TAG + ' Phase 3: Post-processing (traceability matrix)...\n\n');
    await runGroup(postSteps);
  }
```

   And filter traceability steps out of Phase 2:
```javascript
  const toolSteps = steps.filter(s => s.tool !== 'generate' && s.tool !== 'traceability');
```

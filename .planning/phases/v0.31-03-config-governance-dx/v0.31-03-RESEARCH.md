# Phase v0.31-03: Config & Governance DX - Research

**Researched:** 2026-03-08
**Domain:** Claude Code rules loading, config normalization, debate templating
**Confidence:** HIGH

## Summary

This phase addresses three independent DX improvements: (1) adding YAML `paths:` frontmatter to `.claude/rules/` files so only relevant rules load per turn (SHARD-01), (2) building a config adapter layer that normalizes values written to nf.json (ADAPT-01), and (3) creating a consistent debate template with parseable frontmatter for quorum debates (ADR-01).

All three requirements are well-scoped and largely independent. SHARD-01 leverages a native Claude Code feature (`paths:` frontmatter in rules files) requiring zero custom code -- just adding frontmatter to existing files. ADAPT-01 extends the existing `config-loader.js` with a write-side adapter. ADR-01 creates a template file and a lightweight formatter/validator script.

**Primary recommendation:** Use Claude Code's native `paths:` frontmatter for rule sharding (no custom loading mechanism needed). Build the config adapter as a new exported function in `config-loader.js`. Create a debate template with YAML frontmatter for programmatic parsing.

## Architecture Patterns

### SHARD-01: Rules File Sharding (Native Claude Code Feature)

**What:** Add `paths:` YAML frontmatter to each `.claude/rules/*.md` file so Claude Code only loads rules when working on matching file types.

**Current state:** 4 rules files (35 lines total, ~2228 chars), all load unconditionally every turn.

**Target state:** Each rules file scoped to relevant file patterns:

```yaml
# .claude/rules/coding-style.md
---
paths:
  - "hooks/**/*.js"
  - "hooks/dist/**/*.js"
  - "bin/**/*.{js,cjs,mjs}"
---
# Coding Style Rules
...
```

```yaml
# .claude/rules/git-workflow.md
---
paths:
  - "hooks/**/*.js"
  - "hooks/dist/**/*.js"
  - "bin/install.js"
  - ".planning/**"
---
# Git Workflow Rules
...
```

```yaml
# .claude/rules/testing.md
---
paths:
  - "**/*.test.{js,cjs}"
  - "hooks/dist/**/*.test.js"
---
# Testing Rules
...
```

```yaml
# .claude/rules/security.md
---
paths:
  - "hooks/**/*.js"
  - "bin/**/*.{js,cjs,mjs}"
  - ".env*"
---
# Security Rules
...
```

**Key behavior (from official docs):** Rules without a `paths` field load unconditionally. Rules WITH `paths` only activate when Claude reads files matching those glob patterns. Glob patterns support `**` recursion and `{ext1,ext2}` brace expansion.

**Design decision needed:** Whether ALL rules should be path-scoped or if some (e.g., security) should remain unconditional. Recommendation: scope all four, since security rules are only relevant when editing code files.

### ADAPT-01: Config Adapter Layer

**What:** A bidirectional normalization layer that ensures values written to `nf.json` are properly typed and formatted.

**Current state:** `config-loader.js` has `loadConfig()` (read path) and `validateConfig()` (read-time correction) but NO write path. Config writes happen ad-hoc across scripts using direct `JSON.stringify` + `fs.writeFileSync`.

**Target state:** New exported functions in `config-loader.js`:

```javascript
// Normalizes a single config value before writing
function normalizeConfigValue(key, value) {
  // Boolean string normalization: "true"/"false" -> true/false
  if (typeof value === 'string' && (value.toLowerCase() === 'true' || value.toLowerCase() === 'false')) {
    return value.toLowerCase() === 'true';
  }
  // Profile name case normalization
  if (key === 'hook_profile' && typeof value === 'string') {
    return value.toLowerCase();
  }
  // Model tier case normalization
  if ((key === 'model_tier_planner' || key === 'model_tier_worker') && typeof value === 'string') {
    return value.toLowerCase();
  }
  return value;
}

// Normalizes all values in a config object for writing
function normalizeConfig(config) {
  const result = {};
  for (const [key, value] of Object.entries(config)) {
    result[key] = normalizeConfigValue(key, value);
  }
  return result;
}

// Writes a normalized config to nf.json
function writeConfig(filePath, config) {
  const normalized = normalizeConfig(config);
  fs.writeFileSync(filePath, JSON.stringify(normalized, null, 2) + '\n', 'utf8');
}
```

**Nested vs flat key handling:** The existing config uses flat keys by design (STATE.md decision: "Flat keys required -- nested objects lost in shallow merge"). The adapter should convert dotted notation (`circuit_breaker.oscillation_depth`) to nested objects on write and flatten on read. However, given the shallow merge architecture, recommendation is to support dotted notation ONLY for convenience commands (e.g., `nf config set circuit_breaker.oscillation_depth 3`) -- the stored format remains nested JSON objects as today.

### ADR-01: Debate Template

**What:** Standardized template for quorum debate records with YAML frontmatter.

**Current state:** 7 existing debate files in `.planning/quorum/debates/` use an ad-hoc format:
```markdown
# Quorum Debate
Question: ...
Date: ...
Consensus: ...
Rounds: N

## Round 1
| Model | Position | Citations |
...

## Outcome
...
```

**Target state:** Structured template with YAML frontmatter:

```markdown
---
date: 2026-03-08
question: "Should we use approach A or B?"
consensus: APPROVE
rounds: 1
participants: [claude, codex-1, opencode-1]
tags: [architecture, config]
---

# Quorum Debate: Should we use approach A or B?

## Context
[Background and motivation for the question]

## Question
[The specific question being debated]

## Positions
### Round 1
| Model | Position | Citations |
|-------|----------|-----------|
| ... | ... | ... |

## Decision
[Final consensus decision with rationale]

## Consequences
- [Impact 1]
- [Impact 2]
```

**Parseable format:** YAML frontmatter allows programmatic access to `date`, `question`, `consensus`, `rounds`, `participants`, and `tags` without parsing markdown. The body follows a consistent section structure (Context, Question, Positions, Decision, Consequences) matching ADR-01 requirements.

### Anti-Patterns to Avoid
- **Custom rules loader for SHARD-01:** Claude Code already handles `paths:` frontmatter natively. Building a hook-based loader would duplicate functionality and fight the platform.
- **Deep merge in config adapter:** The existing `loadConfig()` uses shallow merge by design. The adapter must NOT change this behavior -- it only normalizes values before writing.
- **Over-engineering debate validation:** A simple frontmatter parser is sufficient. Do not build a full schema validation system for debate files.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Rules conditional loading | Custom hook to filter rules | Claude Code native `paths:` frontmatter | Platform feature, zero maintenance, officially documented |
| YAML frontmatter parsing | Custom parser | Simple regex (`/^---\n([\s\S]*?)\n---/`) or `js-yaml` | Well-solved problem, frontmatter is simple key:value |
| Config file atomic writes | Custom write-with-lock | `fs.writeFileSync` (synchronous, atomic on POSIX for small files) | Config files are <10KB, single-writer pattern |

## Common Pitfalls

### Pitfall 1: Overly Narrow Rule Paths
**What goes wrong:** Rules files scoped too narrowly miss relevant contexts (e.g., security rules only load for `.js` but miss `.cjs` or `.mjs` files).
**Why it happens:** Not accounting for all file extensions in the project.
**How to avoid:** Include all relevant extensions using brace expansion: `**/*.{js,cjs,mjs}`. Review `package.json` test:ci command for the full list of file extensions used.
**Warning signs:** Rules not loading when expected during hook editing sessions.

### Pitfall 2: Config Adapter Breaking Existing Reads
**What goes wrong:** Adding write normalization inadvertently changes read behavior or breaks the shallow merge contract.
**Why it happens:** Coupling read and write normalization in the same code path.
**How to avoid:** Keep `normalizeConfig()` completely separate from `loadConfig()`/`validateConfig()`. The adapter is write-side only; read validation continues as-is.
**Warning signs:** Existing config-loader tests failing after adapter addition.

### Pitfall 3: Debate Template Too Rigid
**What goes wrong:** Template requires fields that existing debate flows don't produce (e.g., `participants` list when some slots are UNAVAIL).
**Why it happens:** Designing the template from ideal rather than actual usage patterns.
**How to avoid:** Make all frontmatter fields optional except `date` and `question`. Allow `consensus: NONE` for unresolved debates.
**Warning signs:** Quorum dispatch code needing major changes to produce template-conformant output.

### Pitfall 4: Install Sync Forgotten
**What goes wrong:** Changes to `hooks/config-loader.js` are not copied to `hooks/dist/config-loader.js` and re-installed.
**Why it happens:** Forgetting the install sync pattern documented in git-workflow.md.
**How to avoid:** Every plan that edits hooks/ files MUST include a sync+install task: `cp hooks/config-loader.js hooks/dist/ && node bin/install.js --claude --global`
**Warning signs:** Tests pass locally but behavior unchanged in Claude Code sessions.

### Config value normalization
```javascript
'use strict';

// Normalizes a config value for writing to nf.json.
// Ensures internal types (boolean strings, mixed-case profiles) never leak to config files.
function normalizeConfigValue(key, value) {
  // Boolean string -> boolean
  if (typeof value === 'string') {
    const lower = value.toLowerCase();
    if (lower === 'true') return true;
    if (lower === 'false') return false;
  }
  // Profile/tier case normalization
  const CASE_NORMALIZED_KEYS = ['hook_profile', 'model_tier_planner', 'model_tier_worker', 'fail_mode'];
  if (CASE_NORMALIZED_KEYS.includes(key) && typeof value === 'string') {
    return value.toLowerCase();
  }
  return value;
}
```

### YAML frontmatter extraction (no external deps)
```javascript
// Extracts YAML frontmatter from a markdown file as a plain object.
// Returns null if no frontmatter found.
function extractFrontmatter(content) {
  const match = content.match(/^---\n([\s\S]*?)\n---/);
  if (!match) return null;
  const lines = match[1].split('\n');
  const result = {};
  for (const line of lines) {
    const sep = line.indexOf(':');
    if (sep === -1) continue;
    const key = line.slice(0, sep).trim();
    let val = line.slice(sep + 1).trim();
    // Handle arrays: [a, b, c]
    if (val.startsWith('[') && val.endsWith(']')) {
      val = val.slice(1, -1).split(',').map(s => s.trim());
    }
    // Handle numbers
    if (/^\d+$/.test(val)) val = parseInt(val, 10);
    result[key] = val;
  }
  return result;
}
```

## Open Questions

1. **Should any rules file remain unconditionally loaded?**
   - What we know: All 4 current rules files are relevant only during code editing. Security rules could arguably apply universally.
   - What's unclear: Whether scoping security rules to code files misses important contexts.
   - Recommendation: Scope all 4 files with paths. If security gaps appear, the paths can be removed to restore unconditional loading.

2. **Where should the debate template file live?**
   - What we know: Debates currently live in `.planning/quorum/debates/`. Template could be a `.md` file in that directory or a separate template location.
   - What's unclear: Whether nf-prompt.js or quorum dispatch should auto-generate debates from template.
   - Recommendation: Place template at `.planning/quorum/debates/_TEMPLATE.md` (underscore prefix sorts first, signals it's not an actual debate). Formatter script at `bin/debate-formatter.cjs`.

3. **Should existing 7 debates be retroactively converted to new template?**
   - What we know: 7 existing debates use ad-hoc format but contain the same semantic content.
   - What's unclear: Whether retroactive conversion is worth the effort.
   - Recommendation: Convert existing debates as a low-priority task. The template applies to future debates.

# Research: Phase v0.36-03 — Convergence Intelligence

## Summary

This phase adds four convergence pathology detection and reporting capabilities to the nForma solve loop: internal cycle detection (CONV-01), residual bucket splitting (CONV-02), gate cap reporting (CONV-03), and baseline drift detection (CONV-04). All four requirements modify existing modules in `bin/` and skill files in `commands/nf/`, with one new module (`bin/solve-cycle-detector.cjs`).

## Requirements Coverage

| ID | Requirement | Key Files |
|---|---|---|
| CONV-01 | Internal cycle detection in convergence loop | `bin/solve-cycle-detector.cjs` (new), `bin/nf-solve.cjs`, `commands/nf/solve.md` |
| CONV-02 | Split residual into 3 buckets | `bin/nf-solve.cjs`, `bin/solve-trend-helpers.cjs`, `bin/convergence-report.cjs` |
| CONV-03 | Gate cap reporting in output JSON | `commands/nf/solve-remediate.md`, `bin/nf-solve.cjs` |

## Existing Architecture

### Residual Computation (`bin/nf-solve.cjs`)

The `computeResidual()` function (line 2593) sweeps all 19 layer transitions and returns a residual object. It currently computes:
- `total`: sum of 8 forward layers (r_to_f + f_to_t + c_to_f + t_to_c + f_to_c + r_to_d + d_to_c + p_to_f)
- `layer_total`: sum of 3 gate alignment layers (l1_to_l2 + l2_to_l3 + l3_to_tc)
- `reverse_discovery_total`: sum of 3 reverse layers (c_to_r + t_to_r + d_to_r)
- `heatmap_total`: git_heatmap residual

The JSON output (line 3674) includes `residual_vector`, `iterations`, `health`, `complexity_profile`, and `oscillation`.

### Trend Tracking (`bin/solve-trend-helpers.cjs`)

`appendTrendEntry()` writes JSONL entries to `.planning/formal/solve-trend.jsonl` with `total_residual`, `per_layer` (19 keys), `gate_summary`, `requirement_count`, `scope_change`, and `fast_mode`. No bucket breakdown exists yet.

### Oscillation Detection (`bin/oscillation-detector.cjs`)

Existing `oscillation-detector.cjs` uses Mann-Kendall trend analysis and sign-change counting across the solve-trend.jsonl window. It detects oscillations at the **inter-session** level (across multiple solve runs). CONV-01 requires **intra-session** cycle detection (within a single convergence loop's iterations).

The existing detector counts oscillation sign changes (`countOscillations()`) but does NOT detect A-B-A-B value repetition patterns. CONV-01 needs a distinct algorithm: checking if `values[i] == values[i-2]` for 4+ consecutive points.

### Convergence Report (`bin/convergence-report.cjs`)

Reads `solve-trend.jsonl` and `oscillation-verdicts.json` to produce sparkline tables, oscillation status, and action items. Currently displays a single `total_residual` — needs updating for the 3-bucket split.

### Gate Cap Logic (`commands/nf/solve-remediate.md`)

Gate A/B/C remediation (sections 3k, 3l, 3m, lines 525-603) each have "max dispatches: 3 per solve cycle" tracked by a counter. When the counter reaches 3, further auto-fixes are skipped. This cap information is currently logged but NOT included in the JSON output.

### Solve Report (`commands/nf/solve-report.md`)

The before/after comparison (Step 6) displays per-layer deltas. No baseline drift detection exists — the comparison uses the `baseline_residual` and `post_residual` passed by the orchestrator, which both come from the same session.

### Layer Key Registry (`bin/layer-constants.cjs`)

Canonical 19-layer key array: `r_to_f`, `f_to_t`, `c_to_f`, `t_to_c`, `f_to_c`, `r_to_d`, `d_to_c`, `p_to_f`, `c_to_r`, `t_to_r`, `d_to_r`, `l1_to_l2`, `l2_to_l3`, `l3_to_tc`, `per_model_gates`, `git_heatmap`, `git_history`, `formal_lint`, `hazard_model`.

## Design Decisions

### CONV-01: Cycle Detector Module

**New file**: `bin/solve-cycle-detector.cjs`

Algorithm: Track `per_layer_residual_history[layer][iteration]`. For each layer, check if `values[i] === values[i-2]` for 4+ consecutive points (indicating A-B-A-B oscillation). This is distinct from the inter-session oscillation-detector.cjs.

The solve.md orchestrator calls this module in Phase 3c (convergence check) after each re-diagnostic. When oscillating layers are detected:
1. Report them in the re-diagnostic JSON output (`oscillating_layers` array)
2. Exclude them from the "any automatable layer changed" loop condition

This aligns with the formal `ConvergenceEventuallyResolves` invariant — resolution must terminate rather than loop indefinitely.

### CONV-02: Residual Bucket Split

The 19 layers split into 3 buckets:

| Bucket | Layers | Rationale |
|---|---|---|
| `automatable` | r_to_f, f_to_t, c_to_f, t_to_c, f_to_c, r_to_d, l1_to_l2, l2_to_l3, l3_to_tc | Forward + gate layers that the solver can auto-remediate |
| `manual` | d_to_c, c_to_r, t_to_r, d_to_r | Require human review or approval |
| `informational` | git_heatmap, git_history, formal_lint, hazard_model, per_model_gates, p_to_f | Risk signals and maturity tracking, not actionable residuals |

Changes needed:
1. `computeResidual()` in nf-solve.cjs: add `automatable`, `manual`, `informational` fields alongside `total`
2. `appendTrendEntry()` in solve-trend-helpers.cjs: include 3 buckets in JSONL entries
3. `convergence-report.cjs`: display buckets separately in sparkline/action sections

### CONV-03: Gate Cap Reporting

The gate dispatch caps are enforced in `solve-remediate.md` (skill-level orchestration, not in bin/ code). To surface caps in the remediation output JSON:

1. Add `capped_layers: [{layer, dispatched, max}]` to the `remediation_report` in solve-remediate.md's output_contract
2. The solve.md orchestrator propagates `capped_layers` to solve-state.json after each remediation phase
3. This requires updating solve-state.json schema to include `capped_layers`

### CONV-04: Baseline Drift Detection

At report time (solve-report.md Step 6), re-read `.planning/formal/requirements.json` count and compare with the session-start baseline. For each layer, compare current residual snapshot to the session-start snapshot. If any layer's residual changed >10% from original baseline due to external edits:

Add `baseline_drift: { detected: bool, layers: [...], warning: "..." }` to report output.

Implementation: the solve orchestrator already passes `baseline_residual` and `post_residual` to the report sub-skill. CONV-04 adds a third snapshot: re-run `nf-solve.cjs --json --report-only` at report time (or at minimum re-read requirements.json count) and compare against the original baseline. A >10% change in any layer indicates external edits occurred during the session.

## Formal Context

**Relevant module**: `convergence` (`.planning/formal/spec/convergence/invariants.md`)

Key invariants:
- `ConvergenceEventuallyResolves`: resolution must terminate (directly relevant to CONV-01 cycle detection)
- `ResolvedAtWriteOnce`: once resolved, stays resolved (CONV-01 exclusion must not un-resolve)
- `HaikuUnavailableNoCorruption`: fail-open on classifier unavailability (CONV-01 must fail-open if cycle detection errors)

## Risk Assessment

- **Low risk**: CONV-02 (bucket split) and CONV-03 (cap reporting) are additive — they add new fields to existing JSON without changing behavior
- **Medium risk**: CONV-01 (cycle detection) modifies the convergence loop condition — incorrect detection could cause premature termination. Mitigation: fail-open design, only exclude layers that show clear A-B-A-B patterns across 4+ iterations
- **Low risk**: CONV-04 (baseline drift) is display-only at report time — no behavioral change to the solve loop

# Phase v0.31-01: Hook Infrastructure Hardening - Research

**Researched:** 2026-03-08
**Domain:** Hook execution ordering and input validation (CommonJS Node.js hooks)
**Confidence:** HIGH

## Summary

Phase v0.31-01 addresses two reliability gaps in nForma's hook infrastructure: (1) hooks execute in non-deterministic order depending on filesystem ordering and settings.json array position, meaning safety-critical hooks like circuit-breaker may run after enhancement hooks; (2) hook stdin JSON is parsed with only a bare try/catch, providing no structured error reporting when Claude Code sends malformed or unexpected payloads.

The solution follows patterns from ruflo's HookPriority enum but adapted to nForma's architecture: a priority field in nf.json that install.js uses to sort hook entries in settings.json, and a lightweight validate function in config-loader.js that checks required fields and types per hook event type without introducing external dependencies like Zod.

**Primary recommendation:** Add `hook_priority` config to nf.json + priority-aware sorting in install.js for PRIO-01, and add `validateHookInput(eventType, input)` to config-loader.js for VALID-01. Both changes are internal to hooks/ and bin/install.js with no new dependencies.

## Architecture Patterns

### Current Hook Registration (install.js)

Hooks are registered in settings.json as arrays per event type. The current order is determined by the order `push()` calls appear in install.js (~lines 1908-2000). Claude Code executes hooks in array order.

```
settings.hooks = {
  PreToolUse: [ { hooks: [{ type: 'command', command: '...nf-circuit-breaker.js' }] } ],
  UserPromptSubmit: [ { hooks: [{ type: 'command', command: '...nf-prompt.js' }] } ],
  Stop: [ { hooks: [{ type: 'command', command: '...nf-stop.js' }] }, ... ],
  PostToolUse: [ { hooks: [{ type: 'command', command: '...gsd-context-monitor.js' }] }, ... ],
  ...
}
```

### Pattern 1: Priority-Aware Hook Sorting (PRIO-01)

**What:** Define default priorities per hook in install.js, allow user override via `hook_priorities` in nf.json, sort each event type's hook array by priority (descending = higher priority first) after all hooks are registered.

**Default priority map (ruflo-inspired):**
```javascript
const DEFAULT_HOOK_PRIORITIES = {
  'nf-circuit-breaker': 1000,  // Critical — safety, must run first
  'nf-stop':            1000,  // Critical — quorum gate
  'nf-prompt':            50,  // Normal — quorum injection
  'nf-precompact':        50,  // Normal — state injection
  'nf-session-start':     50,  // Normal — secret sync
  'nf-session-end':       50,  // Normal — session cleanup
  'nf-check-update':      10,  // Low — update check
  'gsd-context-monitor':  50,  // Normal — context warnings
  'nf-spec-regen':        10,  // Low — spec regeneration
  'nf-post-edit-format':  10,  // Low — formatting
  'nf-console-guard':     10,  // Low — console.log warning
  'nf-statusline':        10,  // Low — status display
  'nf-token-collector':   10,  // Low — token tracking
  'nf-slot-correlator':   10,  // Low — slot correlation
};
```

**When to use:** Always — install.js sorts hook arrays after all registrations complete.

### Pattern 2: Lightweight Input Schema Validation (VALID-01)

**What:** A `validateHookInput(eventType, input)` function exported from config-loader.js that checks required fields and types for each Claude Code hook event type. Returns `{ valid: true }` or `{ valid: false, errors: [...] }`.

**Per-event-type schemas (based on actual hook usage in codebase):**

```javascript
const HOOK_INPUT_SCHEMAS = {
  PreToolUse: {
    required: { tool_name: 'string' },
    optional: { tool_input: 'object', cwd: 'string', hook_event_name: 'string' },
  },
  PostToolUse: {
    required: { tool_name: 'string' },
    optional: { tool_input: 'object', tool_response: 'object', cwd: 'string', context_window: 'object', hook_event_name: 'string' },
  },
  UserPromptSubmit: {
    required: { prompt: 'string' },
    optional: { cwd: 'string', hook_event_name: 'string' },
  },
  Stop: {
    required: { stop_hook_active: 'boolean' },
    optional: { transcript_so_far: 'string', cwd: 'string', hook_event_name: 'string' },
  },
  PreCompact: {
    required: {},
    optional: { cwd: 'string', hook_event_name: 'string', context_window: 'object' },
  },
  SessionStart: {
    required: {},
    optional: { cwd: 'string', hook_event_name: 'string' },
  },
  SessionEnd: {
    required: {},
    optional: { cwd: 'string', hook_event_name: 'string', transcript: 'string' },
  },
};
```

### Anti-Patterns to Avoid
- **Runtime priority sorting:** Do NOT sort hooks at runtime (in each hook). Sort once at install time in settings.json.
- **Zod/Joi dependency:** Do NOT add schema validation libraries. The hook ecosystem is zero-dependency CommonJS.
- **Blocking on validation failure:** Hooks MUST remain fail-open. Validation errors go to stderr, then `process.exit(0)`.
- **Validating output schemas:** Only validate INPUT. Hook output schemas are the hook's responsibility.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON schema validation | Full JSON Schema validator | Simple type-check function | Hooks need <20 field checks total; a validator library is overkill |
| Priority queue | Runtime priority queue data structure | Array sort at install time | Hooks run in settings.json array order; sorting once at install time is sufficient |

## Common Pitfalls

### Pitfall 1: Breaking Existing Hook Registration
**What goes wrong:** install.js uses `some()` to check if a hook already exists before pushing. If sorting changes the array structure, the dedup check may fail and create duplicates.
**Why it happens:** The sort step runs after registration, but next install run, the hooks are already in sorted order with different array positions.
**How to avoid:** Sort AFTER all registrations. The dedup `some()` checks scan by command string content, not position, so sorting is safe. But verify with existing install tests.
**Warning signs:** Duplicate hook entries in settings.json after re-running install.

### Pitfall 2: Validation Blocking Non-Standard Events
**What goes wrong:** Claude Code may add new hook event types or fields over time. A strict schema rejects unknown event types.
**Why it happens:** The schema only covers known event types.
**How to avoid:** Fail-open for unknown event types (return `{ valid: true }` if eventType not in schema). Only validate known types.
**Warning signs:** Hooks exiting with errors after Claude Code updates.

### Pitfall 3: Circular Dependency in config-loader.js
**What goes wrong:** Adding validation schemas to config-loader.js could make the file too large or create require-time side effects.
**Why it happens:** config-loader.js is already 474 lines.
**How to avoid:** Keep schemas as a simple data structure (object literal). The validate function is <40 lines. Total addition: ~80 lines.

## Code Examples

### Priority Sorting in install.js
```javascript
// After all hook registrations, sort each event type by priority
function sortHooksByPriority(settings, config) {
  const priorities = { ...DEFAULT_HOOK_PRIORITIES, ...(config.hook_priorities || {}) };

  for (const eventType of Object.keys(settings.hooks || {})) {
    settings.hooks[eventType].sort((a, b) => {
      const aName = extractHookName(a);
      const bName = extractHookName(b);
      const aPrio = priorities[aName] || 50;
      const bPrio = priorities[bName] || 50;
      return bPrio - aPrio; // Higher priority first
    });
  }
}

function extractHookName(hookEntry) {
  const cmd = (hookEntry.hooks && hookEntry.hooks[0] && hookEntry.hooks[0].command) || '';
  const match = cmd.match(/\/(nf-[a-z-]+|gsd-[a-z-]+)\.js/);
  return match ? match[1] : '';
}
```

### Input Validation in config-loader.js
```javascript
function validateHookInput(eventType, input) {
  const schema = HOOK_INPUT_SCHEMAS[eventType];
  if (!schema) return { valid: true }; // Unknown event type → fail-open

  const errors = [];
  for (const [field, expectedType] of Object.entries(schema.required || {})) {
    if (!(field in input)) {
      errors.push({ field, error: 'missing', expected: expectedType });
    } else if (typeof input[field] !== expectedType) {
      errors.push({ field, error: 'wrong_type', expected: expectedType, got: typeof input[field] });
    }
  }

  // Check optional fields only if present
  for (const [field, expectedType] of Object.entries(schema.optional || {})) {
    if (field in input && input[field] !== null && input[field] !== undefined) {
      if (typeof input[field] !== expectedType) {
        errors.push({ field, error: 'wrong_type', expected: expectedType, got: typeof input[field] });
      }
    }
  }

  return errors.length > 0 ? { valid: false, errors } : { valid: true };
}
```

### Hook Integration Pattern
```javascript
// In each hook's main function, after JSON.parse:
const input = JSON.parse(raw);
const eventType = input.hook_event_name || input.hookEventName || 'PreToolUse';
const { validateHookInput } = require('./config-loader');
const validation = validateHookInput(eventType, input);
if (!validation.valid) {
  process.stderr.write('[nf] WARNING: ' + hookName + ': invalid input: ' +
    JSON.stringify(validation.errors) + '\n');
  process.exit(0); // Fail-open
}
```

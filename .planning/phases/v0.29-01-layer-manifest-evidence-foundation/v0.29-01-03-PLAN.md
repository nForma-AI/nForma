Create bin/failure-taxonomy.cjs following existing bin/ CJS pattern (#!/usr/bin/env node, 'use strict', PROJECT_ROOT, --json flag).

The script must:
1. Read .planning/formal/check-results.ndjson (159 entries, 50 failures). Use the path resolution pattern from write-check-result.cjs
2. Also read .planning/formal/debt.json (4 entries) as additional failure sources
3. Filter to failed entries (result === "fail")
4. Classify each failure into exactly ONE of 5 categories using these decision rules (per research Pitfall 5):
   - crash: formalism tool itself crashed (non-zero exit, no structured result, or result contains stack trace)
   - timeout: runtime_ms > threshold AND result=fail (TLC state explosion, model checker timeout)
   - logic-violation: TLC reports counterexample, Alloy reports counterexample, assertion failure
   - drift: validate-traces reports divergence (formalism "trace" with divergence count)
   - degradation: metrics trending worse over time (reserved, requires baseline). IMPORTANT: when no baseline data exists to compare against, the degradation category falls back to logic_violation. Document this fallback in the output schema as "degradation_fallback_note": "When no baseline exists, potential degradation entries are classified as logic_violation"
5. Each classified failure gets: original check-result fields + category + classification_reason
6. Produce failure-taxonomy.json with:
   - schema_version "1", generated timestamp
   - total_failures count
   - categories: { crash: [], timeout: [], logic_violation: [], drift: [], degradation: [] } -- each array contains classified failure entries
   - category_counts: { crash: N, timeout: N, logic_violation: N, drift: N, degradation: N }
   - degradation_fallback_note: "When no baseline exists, potential degradation entries are classified as logic_violation"
   - unclassified: [] (should be empty -- every failure must be classified)
   - summary: human-readable breakdown

IMPORTANT: Every failure must be classified into exactly one category. If unclassified array is non-empty, exit with code 1.

Create bin/failure-taxonomy.test.cjs using node:test:
- Test classification rules with inline fixtures for each category (mock check-result objects)
- Test that a counterexample result classifies as logic-violation
- Test that a trace divergence classifies as drift
- Test degradation fallback: when no baseline exists, verify the entry is classified as logic_violation (not left unclassified)
- Test mutual exclusivity (no failure in multiple categories)
- Integration test: run against real check-results.ndjson, verify total classified equals total failures

Run the script to generate initial failure-taxonomy.json.

This avoids dual-source-of-truth drift if the mapping is replicated. Verify validate-traces.cjs still runs correctly as a CLI script after this change (the export should not affect #!/usr/bin/env node execution).

Then create bin/state-candidates.cjs following existing bin/ CJS pattern.

The script must:
1. Read conformance-events.jsonl using planning-paths.cjs resolve(cwd, 'conformance-events')
2. Load event-vocabulary.json to know the set of known/mapped action types
3. Import mapToXStateEvent from validate-traces.cjs: const { mapToXStateEvent } = require('./validate-traces.cjs')
4. Identify unmapped actions: events where action is "undefined" (5,649 events per research) or any action not in the vocabulary
5. For each unmapped action cluster, extract:
   - The action string (or "undefined")
   - Count of occurrences
   - Sample timestamps (first 5)
   - Surrounding context: what actions appear immediately before/after in the trace sequence
   - Suggested state variable: based on context patterns (e.g., if always preceded by quorum_start, suggest it's a quorum sub-state)
6. Also identify potential missing transitions: sequences of known actions that never appear in the XState model's transition map
7. Deduplicate candidates (same action + same surrounding context = one candidate)
8. Produce state-candidates.json with:
   - schema_version "1", generated timestamp
   - total_unmapped_events count
   - candidates: array of { action, count, first_seen, last_seen, sample_timestamps, context_before, context_after, suggested_state, confidence }
   - missing_transitions: array of { from_action, to_action, count, note }
   - summary: human-readable text with top candidates

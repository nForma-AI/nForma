# Phase v0.29-01: Layer Manifest and Evidence Foundation - Research

**Researched:** 2026-03-06
**Domain:** Formal verification artifact classification and evidence pipeline (Layer 1)
**Confidence:** HIGH

## Summary

Phase v0.29-01 is a **data classification and evidence tooling** phase. It does NOT introduce new formal verification techniques -- it organizes the existing 92 formal models into a three-layer architecture (L1/L2/L3) and builds four new CLI scripts that analyze the existing 35,627 conformance traces, 159 check results, and 4 debt entries to produce structured evidence artifacts. The key insight is that every input already exists; this phase builds the scripts that read, classify, and index that data.

The phase has two halves: (1) a **layer manifest** that classifies all 92 models and extends `model-registry.json` with `source_layer`, `gate_maturity`, and `layer_maturity` fields (requirements INTG-01, INTG-05, INTG-06), and (2) **four Layer 1 evidence scripts** that produce instrumentation maps, trace corpus stats, failure taxonomies, and a canonical event vocabulary (requirements EVID-01 through EVID-05). The state candidate extractor (EVID-04) mines traces to suggest unmodeled states/transitions. The event vocabulary (EVID-05) is a cross-cutting constraint that all L1 scripts validate against.

**Primary recommendation:** Build the canonical event vocabulary (EVID-05) FIRST as a JSON schema, then build each L1 script to validate against it. Build the layer manifest and model registry extensions in parallel since they have no data dependency on the L1 scripts. The check-result.schema.json must be extended to allow `formalism: "layer"` for gate-related check results in later phases.

### Pattern 1: CLI Script with JSON Output
**What:** Every new script follows the existing bin/ pattern: CJS module, `#!/usr/bin/env node`, writes JSON output to `.planning/formal/evidence/`, exits 0 on success, exits 1 on validation failure. Supports `--json` flag for machine-readable output and `--project-root` for path resolution.
**When to use:** Every new script in this phase.
**Example:**
```javascript
// Source: existing pattern from bin/validate-traces.cjs, bin/write-check-result.cjs

'use strict';

const fs = require('fs');
const path = require('path');

const ROOT = process.env.PROJECT_ROOT || path.join(__dirname, '..');
const EVIDENCE_DIR = path.join(ROOT, '.planning', 'formal', 'evidence');

function main() {
  // Ensure output directory exists
  fs.mkdirSync(EVIDENCE_DIR, { recursive: true });

  // ... compute results ...

  const output = { schema_version: '1', generated: new Date().toISOString(), /* ... */ };
  fs.writeFileSync(path.join(EVIDENCE_DIR, 'output-name.json'), JSON.stringify(output, null, 2));

  if (process.argv.includes('--json')) {
    process.stdout.write(JSON.stringify(output));
  } else {
    console.log('Summary: ...');
  }
  process.exit(0);
}

if (require.main === module) main();

**What:** Deterministic rules for classifying existing 92 models into L1/L2/L3.
**When to use:** When building the layer manifest and extending model-registry.json.
**Rules:**
```

  - formalism: "trace" or "redaction" in check-results.ndjson
  - Files: conformance-events.jsonl, debt.json, observe-handler outputs

### Pattern 4: Session Inference from Traces
**What:** The conformance events have no explicit `session_id` field. Sessions must be inferred from timestamp gaps. A gap of >5 minutes between consecutive events indicates a new session.
**When to use:** EVID-02 trace corpus stats implementation.
**Rationale:** Events span 11 days with ~3,200/day average. Within a day, quorum rounds cluster tightly (seconds apart). A 5-minute gap reliably separates distinct Claude Code sessions.

### Anti-Patterns to Avoid
- **Moving existing files into layer directories:** Leave tla/, alloy/, prism/, spec/ where they are. The layer-manifest.json classifies them without moving anything. Moving breaks 150+ hardcoded paths in bin/ scripts.
- **Making model-registry.json additionalProperties:false:** The existing registry has no schema file -- it is hand-managed. Adding new fields (source_layer, gate_maturity, layer_maturity) must be backward-compatible. Scripts reading the registry must tolerate missing new fields (default to "unclassified" / "Level 0").
- **Classifying all 92 models manually:** Write a `build-layer-manifest.cjs` script that auto-classifies based on file path and formalism. Only override when the auto-classification is wrong.
- **Validating events against vocabulary at emit time (in hooks):** Hooks are zero-dependency CJS that run in the Claude Code hook sandbox. The vocabulary validation happens in L1 analysis scripts, not in the hooks themselves. Hooks continue to emit freely; scripts classify and validate after the fact.

## Common Pitfalls

### Pitfall 1: check-result.schema.json has additionalProperties: false
**What goes wrong:** The existing schema sets `additionalProperties: false`. Adding `source_layer` or `gate` fields to check results will fail Ajv validation unless the schema is updated first.
**Why it happens:** The schema was designed to be strict (good practice) but now needs extension.
**How to avoid:** Extend the schema BEFORE any script tries to write new fields. Add `source_layer` (optional string enum L1/L2/L3) and `gate` (optional string enum A/B/C) to the schema. Also add `"layer"` to the `formalism` enum for future gate check results.
**Warning signs:** `write-check-result.cjs` throws validation errors when new fields are included.

### Pitfall 2: 5,649 "undefined" action events
**What goes wrong:** 15.9% of conformance events have `action: "undefined"` (literal string). If the instrumentation map or trace corpus stats script treats these as missing data, coverage percentages will be misleadingly low.
**Why it happens:** Hook code paths that emit events without setting the action field. This is a real instrumentation gap, not a parsing error.
**How to avoid:** The event vocabulary must include `"undefined"` as a known action type (even if classified as "unmapped"). The state candidate extractor (EVID-04) should specifically flag these as instrumentation gaps needing fixes. Coverage calculation: `mapped_events / total_events` (where "undefined" counts as unmapped).
**Warning signs:** Coverage percentages that don't add up to 100%.

### Pitfall 3: Conformance events live in .planning/telemetry/ not .planning/
**What goes wrong:** The research documents reference `.planning/conformance-events.jsonl` but the actual file is at `.planning/telemetry/conformance-events.jsonl`. Scripts that hardcode the wrong path produce empty results.
**Why it happens:** The file was relocated during a prior refactor. The `planning-paths.cjs` module handles this via `resolveWithFallback`.
**How to avoid:** Always use `planning-paths.cjs` `resolve(cwd, 'conformance-events')` to find the file. Never hardcode the path.
**Warning signs:** "0 events found" when 35K+ events exist.

### Pitfall 4: Model count discrepancy (43 vs 92)
**What goes wrong:** The roadmap and research docs reference "43 formal models" but model-registry.json contains 92 entries. The 43 figure likely counts unique model files, while 92 counts check-result entries (some models have multiple checks).
**Why it happens:** Different counting methodologies. The 92 number includes TLA+ configs (MC*.tla), individual Alloy models, PRISM models, and derived checks.
**How to avoid:** Use the model-registry.json as the source of truth. The layer manifest should register all 92 entries. Success criterion says "92+ existing formal models" which aligns with the registry count.
**Warning signs:** Manifest shows fewer models than the registry.

### Pitfall 5: Failure classification ambiguity
**What goes wrong:** Some failures fit multiple taxonomy categories. A TLA+ model that times out during checking could be "timeout" or "logic-violation" (if the timeout means state explosion due to a real bug).
**Why it happens:** The 5-category taxonomy (crash, timeout, logic-violation, drift, degradation) is designed for runtime failures but must also classify model-checker failures.
**How to avoid:** Define clear decision rules: (1) If check-result runtime_ms > threshold AND result=fail, classify as "timeout". (2) If TLC reports counterexample, classify as "logic-violation". (3) If validate-traces reports divergence, classify as "drift". (4) If formalism tool itself crashes (non-zero exit, no result), classify as "crash". (5) "Degradation" is reserved for metrics trending worse over time (requires baseline comparison).
**Warning signs:** A single failure assigned to multiple categories.

### Validating events against vocabulary
```javascript
// Source: derived from validate-traces.cjs pattern
const vocabulary = JSON.parse(fs.readFileSync(
  path.join(ROOT, '.planning', 'formal', 'evidence', 'event-vocabulary.json'), 'utf8'
));
const knownActions = new Set(Object.keys(vocabulary.vocabulary));

function validateEvent(event) {
  if (!knownActions.has(event.action)) {
    return { valid: false, reason: `Unknown action: ${event.action}`, action: event.action };
  }
  return { valid: true };
}
```

### Inferring sessions from timestamp gaps
```javascript
// Source: novel implementation for EVID-02
const SESSION_GAP_MS = 5 * 60 * 1000; // 5 minutes

function inferSessions(events) {
  const sessions = [];
  let current = { id: 0, events: [], start: null, end: null };

  for (const event of events) {
    const ts = new Date(event.ts).getTime();
    if (current.events.length > 0 && ts - new Date(current.end).getTime() > SESSION_GAP_MS) {
      sessions.push(current);
      current = { id: sessions.length, events: [], start: null, end: null };
    }
    if (!current.start) current.start = event.ts;
    current.end = event.ts;
    current.events.push(event);
  }
  if (current.events.length > 0) sessions.push(current);
  return sessions;
}
```

## Open Questions

1. **Should layer-manifest.json and model-registry.json be merged?**
   - What we know: Both track formal artifacts. The registry focuses on model-checker outputs; the manifest focuses on layer classification.
   - What's unclear: Whether maintaining two registries creates sync drift.
   - Recommendation: Keep them separate. The manifest includes non-model artifacts (trace files, debt entries, vocabulary) that don't belong in the model registry. The manifest references the registry by path. A build-layer-manifest.cjs script can auto-sync from the registry.

2. **How should "undefined" action events be handled in the vocabulary?**
   - What we know: 5,649 events (15.9%) have action "undefined". This is a real instrumentation gap in the hooks.
   - What's unclear: Whether to fix the hooks (emit proper actions) or classify "undefined" as a known gap.
   - Recommendation: For this phase, add "undefined" to the vocabulary as `{ "source": "unknown", "classification": "instrumentation_gap" }`. Fixing hooks is out of scope for v0.29-01 but should be logged as a follow-up.

3. **What maturity level should existing spec/ invariants get?**
   - What we know: 15 spec directories exist with invariants.md files. These are L2 artifacts but have no explicit L1 grounding.
   - What's unclear: Whether having invariants.md qualifies a model for Level 1 maturity (L3 + L2 invariants) or only Level 0.
   - Recommendation: Models with spec/*/invariants.md get Level 1 (they have L2 semantic declarations). Models without get Level 0. Level 2+ requires explicit L1 trace coverage, which Phase v0.29-02 will provide.

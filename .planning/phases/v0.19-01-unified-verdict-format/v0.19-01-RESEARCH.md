# Phase v0.19-01: Unified Verdict Format - Research

**Researched:** 2026-02-27
**Domain:** NDJSON output normalization for formal verification tools (TLC, Alloy, PRISM, trace validator, redaction)
**Confidence:** HIGH

## Summary

Phase v0.19-01 replaces the current ad-hoc per-tool stdout-only reporting in `run-formal-verify.cjs` with a canonical, machine-readable output stream. Each FV tool (TLC via multiple runner scripts, Alloy via multiple runner scripts, PRISM, trace validator, and the upcoming redaction checker) must append exactly one normalized JSON line to `formal/check-results.ndjson` after each check run. A JSON Schema file (`formal/check-result.schema.json`) defines the canonical fields. The triage/summary section in `run-formal-verify.cjs` is rewired to read from `check-results.ndjson` instead of parsing per-tool stdout. CI exits non-zero when any `result=fail` entry exists.

The current architecture stores results only in memory (`results[]` array in `run-formal-verify.cjs`). Every tool runner streams directly to `stdio: 'inherit'` — there is no structured output capture at all. The entire summary is constructed from the in-memory `results` array populated by each step's boolean pass/fail status, not by reading tool output. This must change to: each tool runner appends one JSON line to a shared file; `run-formal-verify.cjs` reads the file for its summary; CI reads the file for exit code logic.

**Primary recommendation:** Implement a shared `writeCheckResult(entry)` helper that all tool runners import. The helper validates fields, appends the JSON line atomically, and returns immediately. Each runner calls it once at the end of its execution. `run-formal-verify.cjs` reads `check-results.ndjson` at the summary phase. CI step checks the file for any `result=fail`.

## Architecture Patterns

### Pattern 1: Canonical Schema (check-result.schema.json)

**What:** JSON Schema defining required and optional fields for every check result entry.

**Required fields:**
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["tool", "formalism", "result", "timestamp"],
  "properties": {
    "tool":       { "type": "string" },
    "formalism":  { "type": "string", "enum": ["tla", "alloy", "prism", "trace", "redaction"] },
    "result":     { "type": "string", "enum": ["pass", "fail", "warn", "inconclusive"] },
    "timestamp":  { "type": "string", "format": "date-time" },
    "metadata":   { "type": "object" }
  }
}
```

**When to use:** Write this once. All runners conform to it. Do NOT add schema validation at runtime in production paths — schema is documentation and test fixture only.

### Pattern 2: Shared Write Helper (write-check-result.cjs)

**What:** Single CJS module that validates the minimum fields and appends one JSON line.

**When to use:** Every runner script imports this. Keeps append logic in one place.

**Example:**
```javascript
// bin/write-check-result.cjs
'use strict';
const fs   = require('fs');
const path = require('path');

const VALID_RESULTS     = ['pass', 'fail', 'warn', 'inconclusive'];
const VALID_FORMALISMS  = ['tla', 'alloy', 'prism', 'trace', 'redaction'];
const NDJSON_PATH       = path.join(__dirname, '..', 'formal', 'check-results.ndjson');

/**
 * Append one normalized check result line to formal/check-results.ndjson.
 * @param {Object} entry - { tool, formalism, result, metadata? }
 */
function writeCheckResult(entry) {
  if (!entry || typeof entry.tool !== 'string') {
    throw new Error('[write-check-result] tool is required (string)');
  }
  if (!VALID_FORMALISMS.includes(entry.formalism)) {
    throw new Error('[write-check-result] formalism must be one of: ' + VALID_FORMALISMS.join(', '));
  }
  if (!VALID_RESULTS.includes(entry.result)) {
    throw new Error('[write-check-result] result must be one of: ' + VALID_RESULTS.join(', '));
  }

  const record = {
    tool:       entry.tool,
    formalism:  entry.formalism,
    result:     entry.result,
    timestamp:  new Date().toISOString(),
    metadata:   entry.metadata || {},
  };

  fs.appendFileSync(NDJSON_PATH, JSON.stringify(record) + '\n', 'utf8');
}

module.exports = { writeCheckResult, NDJSON_PATH, VALID_RESULTS, VALID_FORMALISMS };
```

### Pattern 3: NDJSON File Initialization in run-formal-verify.cjs

**What:** `run-formal-verify.cjs` truncates/creates `check-results.ndjson` at the start of each run so previous results don't pollute the current run.

**When to use:** At the top of `runOnce()`, before any steps execute.

**Example:**
```javascript
// In runOnce(), before running any steps:
const ndjsonPath = path.join(__dirname, '..', 'formal', 'check-results.ndjson');
fs.writeFileSync(ndjsonPath, '', 'utf8');  // truncate to empty — fresh run
```

**Why truncate not append:** Each `run-formal-verify.cjs` invocation represents a complete verification run. Appending across runs would mix results from different runs in the same file, making CI exit logic ambiguous.

### Pattern 4: Summary Reads from NDJSON (UNIF-03)

**What:** Replace the in-memory `results[]` array summary with a file read + parse at the summary phase.

**Current code (replace this):**
```javascript
// In run-formal-verify.cjs summary section — current (stdout only):
const passed = results.filter(r => r.passed).length;
const failed = results.filter(r => !r.passed).length;
for (const r of results) {
  const mark = r.passed ? '✓' : '✗';
  process.stdout.write(TAG + '  ' + mark + '  ' + r.id.padEnd(30) + r.label + '\n');
}
```

**New code (add alongside existing):**
```javascript
// After runGroup() calls complete, read ndjson for structured summary:
const ndjsonLines = fs.readFileSync(ndjsonPath, 'utf8')
  .split('\n').filter(l => l.trim().length > 0);
const checkResults = ndjsonLines.map(l => JSON.parse(l));
const ndjsonFailed = checkResults.filter(r => r.result === 'fail').length;
const ndjsonPassed = checkResults.filter(r => r.result === 'pass').length;

process.stdout.write(TAG + ' NDJSON check-results.ndjson: ' + ndjsonPassed + ' pass, ' + ndjsonFailed + ' fail\n');
```

**Important:** Keep the existing `results[]` in-memory array intact (for backward compatibility with step tracking). The NDJSON summary is additive — read from file in addition to the existing summary, do not replace it entirely.

### Pattern 5: CI Exit Code Reader (check-results-exit.cjs)

**What:** Standalone script that reads `check-results.ndjson` and exits non-zero if any entry has `result=fail`.

**Example:**
```javascript
// bin/check-results-exit.cjs
'use strict';
const fs   = require('fs');
const path = require('path');
const { NDJSON_PATH } = require('./write-check-result.cjs');

if (!fs.existsSync(NDJSON_PATH)) {
  process.stderr.write('[check-results-exit] No check-results.ndjson found — nothing to check\n');
  process.exit(0);
}

const lines  = fs.readFileSync(NDJSON_PATH, 'utf8').split('\n').filter(l => l.trim());
const parsed = lines.map(l => JSON.parse(l));
const fails  = parsed.filter(r => r.result === 'fail');

if (fails.length > 0) {
  process.stderr.write('[check-results-exit] ' + fails.length + ' fail(s) found:\n');
  for (const f of fails) {
    process.stderr.write('  tool=' + f.tool + ' formalism=' + f.formalism + ' ts=' + f.timestamp + '\n');
  }
  process.exit(1);
}

process.stdout.write('[check-results-exit] All ' + parsed.length + ' check(s) pass/warn/inconclusive.\n');
process.exit(0);
```

### Pattern 6: Runner Script Integration

**What:** Each tool runner script calls `writeCheckResult()` once before its own `process.exit()`.

**Runners to modify:**
- `bin/run-tlc.cjs` — 1 call per config (MCsafety, MCliveness, etc.)
- `bin/run-oscillation-tlc.cjs` — 1 call
- `bin/run-breaker-tlc.cjs` — 1 call
- `bin/run-protocol-tlc.cjs` — 1 call
- `bin/run-account-manager-tlc.cjs` — 1 call
- `bin/run-alloy.cjs` — 1 call
- `bin/run-audit-alloy.cjs` — 1 call (per spec)
- `bin/run-transcript-alloy.cjs` — 1 call
- `bin/run-installer-alloy.cjs` — 1 call (per spec)
- `bin/run-account-pool-alloy.cjs` — 1 call
- `bin/run-prism.cjs` — 1 call
- `bin/run-oauth-rotation-prism.cjs` — 1 call
- `bin/validate-traces.cjs` — 1 call (formalism=trace)

**Pattern for each runner:**
```javascript
// At the end of each runner, before process.exit():
const { writeCheckResult } = require('./write-check-result.cjs');

// ... existing logic that sets passed = true/false ...

writeCheckResult({
  tool:      'run-tlc',         // script name without .cjs
  formalism: 'tla',             // tla | alloy | prism | trace | redaction
  result:    passed ? 'pass' : 'fail',
  metadata:  { config: configName },   // optional
});

process.exit(passed ? 0 : 1);
```

**Critical:** `writeCheckResult()` must NOT throw in production. Wrap in try/catch so tool runner still exits correctly even if NDJSON write fails:
```javascript
try {
  writeCheckResult({ tool: 'run-tlc', formalism: 'tla', result: passed ? 'pass' : 'fail' });
} catch (err) {
  process.stderr.write('[run-tlc] Warning: failed to write check result: ' + err.message + '\n');
}
```

### Pattern 7: gitignore for Runtime Output

**What:** `formal/check-results.ndjson` is runtime-generated, not committed.

**Add to .gitignore:**
```
formal/check-results.ndjson
```

**Why:** Generated per run, contains timestamps, meaningless in git history. Schema file (`check-result.schema.json`) IS committed — it's the contract.

### Anti-Patterns to Avoid

- **Parsing tool stdout to determine pass/fail:** Tool output format varies across TLC/Alloy/PRISM versions. Use process exit code (already done) and map to `pass`/`fail`. Never regex-parse stdout in the summary path.
- **Appending across runs:** Truncate at `runOnce()` start. Multiple runs must not pollute each other.
- **One NDJSON file per tool:** All tools share one `formal/check-results.ndjson`. The `tool` field disambiguates.
- **Runtime schema validation with ajv:** Not worth the dependency. Schema is documentation. Validate field presence/enum in `writeCheckResult()` directly.
- **Modifying `generate-formal-specs.cjs` or `xstate-to-tla.cjs`:** These are generation steps, not check steps. They produce artifacts but don't verify them. They should NOT write check results (they don't have pass/fail semantics in the same way).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| NDJSON format | Custom binary format | Standard NDJSON (one JSON per line) | jq-compatible, grep-friendly, human-readable |
| File locking for concurrent writes | Custom lock file | Sequential writes per run (non-concurrent) | `run-formal-verify.cjs` runs tools within a group sequentially via `runGroup()`; concurrent groups are per-tool, each writes exactly once |
| Schema validation at runtime | ajv dependency | Field checks in `writeCheckResult()` | Zero deps principle — conformance-schema.cjs pattern shows this |
| CI result check | Inline bash | `bin/check-results-exit.cjs` | Testable Node.js script vs unreadable bash |

**Key insight:** The existing `run-formal-verify.cjs` already runs tools sequentially within tool groups (via `runGroup()`), so concurrent file writes to `check-results.ndjson` are not a concern for the initial implementation.

## Common Pitfalls

### Pitfall 1: generate steps are not check steps

**What goes wrong:** Developer adds `writeCheckResult()` call to `generate-formal-specs.cjs` or `xstate-to-tla.cjs`. These steps run first; their "pass" doesn't mean verification passed.
**Why it happens:** They are in the STEPS registry and have pass/fail status.
**How to avoid:** Only runners with tool types `tla`, `alloy`, `prism`, `trace`, `redaction` write check results. Generation steps (`tool: 'generate'`) do NOT write check results.
**Warning signs:** NDJSON contains entries with `formalism=generate` — that's wrong.

### Pitfall 2: Runner exits before writeCheckResult() is called

**What goes wrong:** Runner calls `process.exit(1)` on a prerequisite check (Java missing, JAR missing) before reaching `writeCheckResult()`. NDJSON gets no entry for that tool.
**Why it happens:** Early-exit error paths exist in every runner (Java check, JAR check, binary check).
**How to avoid:** Add `writeCheckResult()` calls to ALL exit paths including error exits. Or: in `run-formal-verify.cjs`, detect missing NDJSON entries and synthesize `result=fail` for tools that didn't write.
**Recommendation:** Simpler approach — in `run-formal-verify.cjs`, after running all steps, for each step that was attempted but has no NDJSON entry, synthesize a fail entry. This avoids modifying every early-exit in every runner.

### Pitfall 3: NDJSON file not initialized in watch mode

**What goes wrong:** Watch mode re-runs `runOnce()` multiple times. If NDJSON is not truncated at the start of each `runOnce()`, results accumulate across re-runs.
**Why it happens:** Watch mode calls `runOnce()` in a loop.
**How to avoid:** Put `fs.writeFileSync(ndjsonPath, '', 'utf8')` at the TOP of `runOnce()`, before the `results.length = 0` reset. Already does `results.length = 0` — mirror this for the file.

### Pitfall 4: CI workflow path mismatch

**What goes wrong:** CI step runs `node bin/run-formal-verify.cjs` from repo root; `check-results.ndjson` path is relative to `__dirname` in `write-check-result.cjs`. If the runner is invoked from a different cwd, the path breaks.
**Why it happens:** `__dirname` in a CJS module is always the module's directory. `path.join(__dirname, '..', 'formal', 'check-results.ndjson')` gives `formal/check-results.ndjson` relative to repo root — correct.
**How to avoid:** Use `__dirname`-relative path in `write-check-result.cjs`, not `process.cwd()`-relative. This is the same pattern used by `run-tlc.cjs` for `tla2tools.jar`.

### Pitfall 5: wasm-dot steps (Petri net render) have no pass/fail from tool

**What goes wrong:** The two `wasm-dot` type steps in `run-formal-verify.cjs` (Petri net render) have no tool script — they call `@hpcc-js/wasm-graphviz` inline. There's no runner to add `writeCheckResult()` to.
**Why it happens:** These are not runner scripts — they execute inline in `runWasmDotStep()`.
**How to avoid:** After `runWasmDotStep()` returns, write the check result in `runGroup()` itself. The `tool` field can be `petri-account-manager` or `petri-quorum`. Formalism can be a new enum value `petri` — but the schema should include it. OR: treat petri net render as not a formal check (it's a visualization), and skip NDJSON for wasm-dot steps.
**Recommendation:** Skip NDJSON entries for `wasm-dot` steps. The success criteria (UNIF-01) lists: TLC, Alloy, PRISM, trace validator, redaction — Petri net rendering is not in that list. Only write entries for the 5 formalisms.

### Pitfall 6: run-audit-alloy.cjs and run-installer-alloy.cjs run multiple specs

**What goes wrong:** `run-audit-alloy.cjs` is called with `--spec=scoreboard-recompute` and `--spec=availability-parsing` as separate steps. Each call writes one entry. Tool ID should include the spec name to disambiguate.
**Why it happens:** Same script, different `--spec` arg, called twice.
**How to avoid:** In `writeCheckResult()` call within `run-audit-alloy.cjs`, include the spec in metadata: `{ tool: 'run-audit-alloy', formalism: 'alloy', result: ..., metadata: { spec: specName } }`. Or use the tool field: `tool: 'run-audit-alloy:' + specName`. Use metadata approach — keeps `tool` clean.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Per-tool stdout only | NDJSON canonical stream | Phase v0.19-01 | Dashboard, CI, and triage all read one source |
| In-memory results[] array | File-backed check-results.ndjson | Phase v0.19-01 | Summary survives process boundary; dashboards can read it |

**Why this ordering matters:** The `warn` and `inconclusive` result values in the schema are needed by downstream phases:
- v0.19-02 (CALIB) emits `result=warn` during PRISM cold-start
- v0.19-03 (LIVE) emits `result=inconclusive` for liveness without fairness declaration
- Defining the schema correctly in v0.19-01 avoids a schema breaking change later

## Open Questions

1. **Petri net wasm-dot steps**
   - What we know: Two steps (`petri:quorum`, `petri:account-manager`) use `wasm-dot` type and call `@hpcc-js/wasm-graphviz` inline. They are not in the UNIF-01 listed formalisms (TLC, Alloy, PRISM, trace validator, redaction).
   - What's unclear: Should Petri net render failures appear in check-results.ndjson?
   - Recommendation: Skip NDJSON entries for wasm-dot steps in v0.19-01. UNIF-01 explicitly lists 5 formalisms; Petri net rendering is visualization, not verification. If needed, add `formalism=petri` in a later phase.

2. **generate steps in run-formal-verify.cjs**
   - What we know: Two generate steps (`generate:tla-from-xstate`, `generate:alloy-prism-specs`) run before verification steps and have pass/fail status.
   - What's unclear: Should generation failures appear in NDJSON?
   - Recommendation: Skip NDJSON for generate steps. Generation failures cause all subsequent checks to fail (stale specs); the downstream check failures are sufficient signal. Keep NDJSON clean for verification-only results.

3. **NDJSON path when runner called standalone vs via run-formal-verify.cjs**
   - What we know: Runners like `run-tlc.cjs` can be called standalone (e.g., `node bin/run-tlc.cjs MCsafety`). When called standalone, `run-formal-verify.cjs` has not initialized (truncated) the NDJSON file.
   - What's unclear: Is appending to an existing NDJSON file from a standalone run acceptable, or should standalone runs truncate first?
   - Recommendation: For standalone runs, allow append (don't truncate). Only `run-formal-verify.cjs` truncates. Standalone runs append to whatever is there. This is fine because `check-results-exit.cjs` only reads the file on demand; CI always invokes via `run-formal-verify.cjs`.

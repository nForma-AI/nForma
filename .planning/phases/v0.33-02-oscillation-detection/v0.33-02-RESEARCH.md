# Phase v0.33-02: Oscillation Detection - Research

**Researched:** 2026-03-10
**Domain:** Per-layer trend analysis, oscillation credit enforcement, cascade-aware grace periods
**Confidence:** HIGH

## Summary

This phase builds the oscillation detection and enforcement layer on top of the JSONL trend data from Phase v0.33-01. Three requirements drive the implementation: TRACK-02 (Mann-Kendall trend detection per layer), OSC-01 (oscillation credit enforcement with human escalation), and OSC-02 (cascade-aware grace period for downstream layers).

The core deliverable is a new module `bin/oscillation-detector.cjs` that reads `solve-trend.jsonl`, computes per-layer Mann-Kendall trend statistics, detects oscillation patterns, manages oscillation credits, and writes verdicts to `oscillation-verdicts.json`. The nf-solve main loop will gate `autoClose()` on these verdicts -- if a layer has exhausted its oscillation credits, automated remediation is blocked and human escalation is triggered.

The existing codebase has a separate oscillation detection system in the circuit breaker (`hooks/nf-circuit-breaker.js`) that detects commit-level oscillation (alternating groups of commits on the same file set). This phase's oscillation detection is conceptually different -- it operates on *solve-run residuals over time*, not commit patterns. The two systems coexist: circuit-breaker catches within-session coding oscillation, while this system catches cross-session convergence oscillation.

The formal spec at `.planning/formal/spec/oscillation/invariants.md` defines the `AlgorithmTerminates` property: the run-collapse algorithm must terminate on every invocation under weak fairness. Our oscillation detector must similarly terminate -- it reads a bounded JSONL window (last 20 entries), performs a fixed computation (Mann-Kendall), and writes a verdict. No unbounded loops.

## Architecture Patterns

### Pattern 1: Mann-Kendall Trend Test

The Mann-Kendall test is a non-parametric trend detection method. For a time series of n values:

1. Compute S = sum of sgn(x_j - x_i) for all i < j
2. S > 0 indicates increasing trend, S < 0 decreasing, S ~ 0 no trend
3. For significance: compute variance Var(S) = n(n-1)(2n+5)/18, then Z = (S-1)/sqrt(Var(S)) if S>0, Z=0 if S=0, Z=(S+1)/sqrt(Var(S)) if S<0
4. Compare |Z| against z-alpha/2 (1.96 for 95% confidence)

Classification rules for this project:
- |Z| < 1.96 AND oscillation count >= 2: `OSCILLATING`
- |Z| < 1.96: `STABLE`
- Z >= 1.96: `INCREASING`
- Z <= -1.96: `DECREASING`

The oscillation count is computed by counting sign changes in consecutive residual deltas (diff[i] = r[i] - r[i-1]). If delta alternates positive/negative 2+ times in the window, it's oscillating.

### Pattern 2: Oscillation Credits (Option C)

Each layer starts with 1 oscillation credit. When a layer's residual increases after a prior decrease (within the JSONL window):
- First increase-after-decrease: credit consumed (warning logged)
- Second increase-after-decrease (credit exhausted): `autoClose()` skips this layer, verdict is `BLOCKED`, human escalation triggered

Credit state persists in `oscillation-verdicts.json`:
```json
{
  "generated": "ISO timestamp",
  "window_size": 20,
  "min_points": 5,
  "layers": {
    "r_to_f": {
      "trend": "DECREASING",
      "z_score": -2.45,
      "oscillation_count": 0,
      "credits_remaining": 1,
      "blocked": false,
      "last_direction": "down",
      "grace_period": false
    }
  }
}
```

### Pattern 3: Cascade-Aware Grace Period

Layer dependency DAG (upstream -> downstream):
```
r_to_f -> f_to_t -> t_to_c
r_to_f -> f_to_c
r_to_d -> d_to_c
c_to_f (bidirectional with f_to_c)
p_to_f -> f_to_t
```

When an upstream layer was remediated (had autoClose action) in the current or immediately prior solve session:
- Downstream layers get a grace period: residual increases do NOT consume oscillation credits
- Grace period expires after 1 session without upstream remediation
- Grace is tracked per-layer in the verdict file: `grace_period: true/false`

Detection: read the most recent solve-session summary file from `.planning/formal/solve-sessions/` to check which layers had remediation actions. If a layer's upstream had actions, mark downstream as grace-period-active.

### Pattern 4: Integration with autoClose()

In `bin/nf-solve.cjs`, before each layer's autoClose action:
1. Read `oscillation-verdicts.json`
2. If layer is `blocked: true`, skip remediation and log human escalation message
3. After autoClose completes, update verdicts by running oscillation detection

The verdict update must happen AFTER autoClose (so the current session's residuals are already in solve-trend.jsonl) but the gate check happens BEFORE autoClose (so blocked layers are skipped).

### Pitfalls

1. **Insufficient data**: Mann-Kendall requires >= 5 data points. Before that, all layers report `INSUFFICIENT_DATA` with no credit consumption.
2. **Fast-mode entries**: Entries with `fast_mode: true` have -1 for skipped layers. When computing trends for a layer, filter out entries where that layer's residual is -1.
3. **Tied values**: Mann-Kendall has a tie correction for Var(S). For simplicity, use the basic formula (no tie correction) -- tied values are uncommon in floating-point residuals.
4. **File atomicity**: `oscillation-verdicts.json` is written atomically (write to temp, rename). Multiple concurrent solve runs should not corrupt it.

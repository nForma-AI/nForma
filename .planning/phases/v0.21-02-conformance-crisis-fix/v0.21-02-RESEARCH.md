# Phase v0.21-02: Conformance Crisis Fix - Research

**Researched:** 2026-03-01
**Domain:** Trace validation, XState machine guard analysis, structured divergence attribution
**Confidence:** HIGH

## Summary

Phase v0.21-02 addresses an urgent technical debt: 69% of conformance traces (8823/12929) diverge from the XState machine spec with no diagnostics to identify root causes. This phase implements three diagnostic tools: (1) structured trace export from `validate-traces.cjs` (currently only prints %), (2) `bin/attribute-trace-divergence.cjs` to walk XState guards and pinpoint failing transitions, and (3) `formal/diff-report.md` as the canonical output format for divergence analysis with both spec-bug and impl-bug fix paths.

The core challenge is mapping conformance events (JSONL) back to XState machine transitions through guard evaluation. The existing machine has 4 states (IDLE, COLLECTING_VOTES, DELIBERATING, DECIDED) and 4 guards (minQuorumMet, unanimityMet, noInfiniteDeliberation, phaseMonotonicallyAdvances). Each divergence requires walking the trace, replaying events, identifying which transition was attempted but blocked, and determining if the issue is a guard logic error or a hook/context error.

**Primary recommendation:** Build the trace-to-XState walker as a reusable library module before implementing either `validate-traces` export or `attribute-trace-divergence`. The walker (event → current state → applicable transitions → guard evaluation → actual state) is the core abstraction that both tools depend on.

### Pattern 1: XState Trace Walker Library
**What:** A reusable module that takes a conformance event, current XState snapshot, and the machine, then evaluates all possible transitions and guard conditions. Returns the expected next state, which transition would be taken, and any guard evaluation failures.

**When to use:** Both `validate-traces.cjs` (to compare actual vs expected) and `attribute-trace-divergence.cjs` (to explain WHY a transition was blocked or taken unexpectedly).

**Example:**
```javascript
// bin/xstate-trace-walker.cjs
// Evaluates a transition and its guards given event and current state

function evaluateTransitions(currentSnapshot, xstateEvent, machine) {
  const state = currentSnapshot.value;
  const context = currentSnapshot.context;

  // Get all transitions from current state
  const transitions = machine.states[state]?.on?.[xstateEvent.type] || [];

  const evaluated = transitions.map(trans => ({
    target: trans.target,
    guardName: trans.guard,
    guardPassed: trans.guard ? evaluateGuard(trans.guard, { context, event: xstateEvent }) : true,
    actions: trans.actions,
  }));

  return {
    currentState: state,
    event: xstateEvent,
    possibleTransitions: evaluated,
    expectedNextState: evaluated.find(t => t.guardPassed)?.target || state,
  };
}
```

**Source:** Custom implementation; XState official docs: https://xstate.js.org/docs/guides/states.html#state-snapshots

### Pattern 2: Structured Divergence Record (TTrace)
**What:** A divergence record that captures an event, its actual XState outcome, the expected outcome, and the guard evaluation results that explain the mismatch.

**When to use:** When exporting divergences from validate-traces.cjs for batch analysis by attribute-trace-divergence.cjs.

**Example:**
```typescript
interface TTrace {
  // Event and context
  event: ConformanceEvent;
  actualState: string;    // snapshot.value after sending event
  expectedState: string;  // what machine SHOULD have done

  // Guard evaluation results
  guardEvaluations: Array<{
    guardName: string;
    passed: boolean;
    context: QGSDContext;
  }>;

  // Classification
  divergenceType: 'state_mismatch' | 'unmappable_action' | 'guard_evaluation_error';
  confidence: 'low' | 'medium' | 'high';  // from scoreboard metadata

  // For impl bug tracking
  implBugIndications?: Array<{
    hook: string;
    lineNumber?: number;
    contextField: string;
    value: any;
  }>;
}
```

### Pattern 3: Diff Report Structure
**What:** Markdown file with structured sections: divergence summary table, per-divergence analysis with fix recommendations, and decision matrix (spec-bug vs impl-bug evidence).

**When to use:** Output of attribute-trace-divergence.cjs; human-readable summary for decision-makers.

**Example:**
```markdown
# Formal Verification Diff Report

**Analyzed:** 10 divergent traces
**Analysis Date:** 2026-03-01T14:00:00Z
**Hypothesis:** minQuorumMet guard is off-by-one in calculation

## Divergence Summary

| Trace | Event | Expected State | Actual State | Root Cause | Recommend |
|-------|-------|---|---|---|---|
| event-8900 | VOTES_COLLECTED (5/11) | DELIBERATING | DECIDED | minQuorumMet: 5 >= ceil(11/2)=6? FALSE, but transition taken | Spec-bug: fix guard logic |
| event-8902 | VOTES_COLLECTED (6/11) | DELIBERATING | DECIDED | minQuorumMet: 6 >= ceil(11/2)=6? TRUE, correct, but... | Impl-bug: context.slotsAvailable never set |

## Detailed Analysis

### Trace event-8900: unanimityMet guard fails, but fallback transition succeeds
- Event: VOTES_COLLECTED(successCount=5)
- Current state: COLLECTING_VOTES, context: { slotsAvailable: 11, successCount: 5 }
- Guard 1: unanimityMet = (5 === polledCount) → depends on polledCount value
- If polledCount not initialized, fallback to next transition
- **Root cause type:** Context initialization bug in QUORUM_START action
- **Fix direction 1 (spec-bug):** XState guard logic error — update qgsd-workflow.machine.ts
- **Fix direction 2 (impl-bug):** Hook not setting polledCount — check qgsd-prompt.js QUORUM_START handler

## Recommended Fix Path

1. Check if polledCount initialization is skipped (search hooks/ for QUORUM_START)
2. If found: impl-bug, fix hook
3. If guard logic is wrong: spec-bug, update machine + regenerate specs
```

### Anti-Patterns to Avoid
- **Creating divergence records inline in validate-traces.cjs instead of as a reusable TTrace type:** Makes attribute-trace-divergence hard to depend on; TTrace must be a shared contract.
- **Hardcoding state/guard names in attribute-trace-divergence.cjs:** Use machine introspection (machine.states, transition.guard names) so it works as machine evolves.
- **Running validation and attribution as separate tools without piping:** attribute-trace-divergence should consume validate-traces output; avoid re-parsing the JSONL.
- **Writing divergence results only to stdout:** diff-report.md must be updated atomically so CI can track changes; use formal/model-registry.json to version it.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON parsing for conformance events | Custom event parser | Node.js built-in JSON + conformance-schema.cjs for validation | Schema already defined, validation already in use; avoid duplicating enum definitions |
| XState machine introspection | Manual transition mapping | XState official inspection APIs (machine.states, snapshot.matches()) | Machine evolves; hardcoded maps will diverge; XState provides stable API |
| Guard logic evaluation | Runtime code evaluation approaches | Function closures passed to setup() | Only XState's guard functions are safe; avoid runtime evaluation techniques that could introduce inconsistency |
| Divergence report generation | Markdown string concatenation | Template library (e.g., lodash template) or checked markdown builder | Markdown can be malformed; project already uses string-based formats (PLAN.md); keep it simple but validate structure |

**Key insight:** The divergence detection chain (event → state → transition → guard eval → TTrace record) must be bulletproof because downstream tools (planner, verifier) depend on accurate root-cause attribution. Don't optimize for code brevity; prioritize traceability and testability.

## Common Pitfalls

### Pitfall 1: Replaying Events on Fresh Actor Per Event vs. Continuous Trace
**What goes wrong:** Current validate-traces.cjs creates a fresh actor per event (line 224). This works for single-step validation but fails to catch multi-event interaction bugs (e.g., polledCount set in event A, used in event B).

**Why it happens:** Single-step validation is simpler to implement and reason about. But real quorum rounds are multi-event sequences, and bugs surface only when context is carried across events.

**How to avoid:** DIAG-02's attribute-trace-divergence.cjs MUST replay the full event sequence up to the divergent event, maintaining context. Build xstate-trace-walker to accept a starting snapshot, then replayed snapshots will catch interaction bugs.

**Warning signs:** If divergence analysis shows "context field was never initialized," check if the initializing event was earlier in the trace. Validate-traces won't catch this.

### Pitfall 2: Guard Names in Error Output Don't Match XState Machine
**What goes wrong:** You output "minQuorumMet failed" but the hook searching for the fix can't find a guard named minQuorumMet in qgsd-workflow.machine.ts (it's defined in setup(), not as a named transition guard).

**Why it happens:** XState setup() syntax is unfamiliar; developers assume guard names appear in the machine spec the same way they're evaluated.

**How to avoid:** When iterating guards in attribute-trace-divergence.cjs, use machine.states[state].on[event][i].guard property directly. Don't manually hardcode guard names. Print both the guard property name AND the human-readable guard description from the machine definition.

**Warning signs:** "Guard X not found in machine" errors from users trying to fix reported issues.

### Pitfall 3: Observation Window Metadata Lost Between Tools
**What goes wrong:** validate-traces.cjs computes confidence tier and observation_window, but those are only written to check-result.ndjson. When attribute-trace-divergence.cjs runs later, it doesn't have confidence metadata and can't determine if divergence is "low confidence (1 round)" or "high confidence (10k rounds)."

**Why it happens:** check-result.ndjson is for CI/formal verification logs, not inter-tool communication. TTrace record format didn't include observation_window.

**How to avoid:** Include observation_window and confidence in TTrace export format. Both tools read from the same source (quorum-scoreboard.json), so recompute observation_window in both. Or, have validate-traces emit a metadata file (.divergence-metadata.json) that attribute-trace-divergence reads.

**Warning signs:** Recommendations changing between runs even though input didn't change — means confidence was lost.

### Pitfall 4: XState Guards Have Side Effects or Depend on Closure State
**What goes wrong:** You evaluate a guard multiple times during analysis and get different results because the guard's closure captured mutable state.

**Why it happens:** XState guards are plain functions; if they close over context or logger state that changes, evaluation is non-deterministic. Not an issue for production (context is immutable), but breaks tools that re-evaluate guards.

**How to avoid:** xstate-trace-walker must pass context as a fresh object each time it evaluates a guard. Never modify context during evaluation. Log guard results but don't let logs affect future evaluations.

**Warning signs:** Determinism test fails: "Same event, same state, guard evaluation result differs between runs."

## Open Questions

1. **Should xstate-trace-walker.cjs be a published module or internal bin/ tool?**
   - What we know: It's a core abstraction that validate-traces.cjs and attribute-trace-divergence.cjs both need.
   - What's unclear: Whether future phases (SPEC, PLAN) will also depend on it, making it public-facing.
   - Recommendation: Start as bin/xstate-trace-walker.cjs (internal); if SPEC or PLAN phases import it, move to src/lib/xstate-trace-walker.ts and export from package.json.

2. **How fine-grained should guard evaluation error messages be?**
   - What we know: Guards are context predicates; errors are "guard X evaluated to false when expected true" or vice versa.
   - What's unclear: Should we also capture the context values that caused failure (e.g., "slotsAvailable=11, successCount=5, minQuorum=6")?
   - Recommendation: Capture context snapshots in TTrace.guardEvaluations[].context; do not redact even if values look "wrong" — they might reveal initialization bugs.

3. **Should attribute-trace-divergence.cjs take multiple divergences or one at a time?**
   - What we know: Batch processing is more efficient; single-trace processing is easier to debug.
   - What's unclear: CI job time limits; how many divergences to analyze before stopping.
   - Recommendation: Accept a --batch-size N flag (default 10); analyze first N divergences per run, output summary, and indicate if more remain.

4. **How should diff-report.md handle SPEC-BUG vs IMPL-BUG uncertainty?**
   - What we know: Some divergences may be caused by either (e.g., guard logic vs. context initialization).
   - What's unclear: Should we recommend BOTH paths, or pick the most likely one?
   - Recommendation: Always list both with confidence scores ("90% likely impl-bug, 10% likely spec-bug"). Let human reviewer decide.

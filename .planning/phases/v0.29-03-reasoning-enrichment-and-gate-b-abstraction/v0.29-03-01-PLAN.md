  <action>
Create `bin/hazard-model.cjs` following the existing gate-a-grounding.cjs pattern (shebang, 'use strict', PROJECT_ROOT env var, --json flag, exit codes).

The script reads L2 observed-fsm.json (4 states, 16 state-event pairs) and applies FMEA scoring:

**Severity (1-10) -- domain judgment lookup table keyed by (fromState, event):**
- DELIBERATING -> DECIDE: 8 (wrong verdict = incorrect PASS/BLOCK)
- COLLECTING_VOTES -> VOTES_COLLECTED: 6 (incomplete votes = stalled quorum)
- Any -> CIRCUIT_BREAK: 6 (false positive break = workflow interruption)
- IDLE -> QUORUM_START: 4 (start failure = degraded, can retry)
- DECIDED -> any self-loop: 2 (already terminal, harmless)
- Default: 4 (moderate)

**Occurrence (1-10) -- computed from trace-corpus-stats.json:**
- ratio = transition.count / totalSessions (349)
- >80%: 10, >50%: 8, >20%: 6, >5%: 4, >0%: 2, 0: 1

**Detection (1-10) -- computed from failure-taxonomy.json + unit-test-coverage.json:**
- Has formalism AND tests: 2
- Has formalism OR tests: 4
- Neither (only conformance events): 8
- Nothing at all: 10

**RPN = S x O x D** (range 1-1000)

Each hazard entry MUST include a `derived_from` array with links to L2 sources (observed-fsm.json transition, relevant invariants from invariant-catalog.json). Use the exact pattern from RESEARCH.md Pattern 3.

Output hazard-model.json to `.planning/formal/reasoning/` with: schema_version, generated timestamp, methodology ("FMEA (IEC 60812)"), scoring scale documentation, hazards array sorted by RPN descending, and summary stats (total, max_rpn, critical_count for RPN>=200, high_count for 100<=RPN<200).

Export scoring functions for testing: computeSeverity, computeOccurrenceScore, computeDetectionScore, generateHazardModel.

Create `bin/hazard-model.test.cjs` using node:test + node:assert:
- Unit tests: computeSeverity returns correct scores for each severity tier (8 for DELIBERATING->DECIDE, 2 for DECIDED self-loops, etc.)
- Unit tests: computeOccurrenceScore returns correct tier for ratio thresholds
- Unit tests: computeDetectionScore returns correct tier based on formalism/test coverage
- Unit tests: RPN = S x O x D for known inputs
- Integration: script reads real observed-fsm.json and produces valid hazard-model.json with 16 entries
- Integration: all entries have non-empty derived_from arrays
- Integration: hazards sorted by RPN descending
- Regression guard: assert no RPN > 1000 and no RPN < 1

Watch out for: FMEA score inflation (pitfall 1 from research) -- verify that not all RPNs cluster above 200. The DECIDED self-loop transitions should have low RPNs (Severity 2).

Create `bin/failure-mode-catalog.cjs` following the same bin/ script pattern.

The script reads L2 observed-fsm.json and enumerates failure modes per state-event pair. For each of the 16 transitions, enumerate up to 3 failure modes:

1. **Omission** (always): transition does not fire when expected. Effect: system stays in current state.
2. **Commission** (conditional): transition fires when NOT modeled in XState. Only generate if the transition appears in observed-fsm.json `model_comparison.missing_in_model`. Effect: unmodeled state transition, behavior diverges from spec.
3. **Corruption** (conditional): transition fires but produces wrong target state. Only generate for transitions where severity >= 6 (from hazard-model.json). Effect: system enters incorrect state.

Each failure mode entry has: id (FM-{state}-{event}-{MODE}), failure_mode (omission/commission/corruption), description, effect, severity_class (critical/stalled/degraded/model_gap/cosmetic), and derived_from array linking to L2 sources.

Read hazard-model.json to get severity scores for the corruption threshold. Use mismatch-register.jsonl entries to enrich failure mode descriptions where mismatches exist for that state/event.

Target: 30-50 failure mode entries total (research warns against >60 -- failure mode explosion pitfall).

Output failure-mode-catalog.json to `.planning/formal/reasoning/` with: schema_version, generated timestamp, failure_modes array, summary (total count, by_mode counts, by_severity_class counts).

Export enumerateFailureModes function for testing.

Create `bin/failure-mode-catalog.test.cjs`:
- Unit: omission mode always generated for a transition

- Unit: corruption mode only generated for high-severity transitions
- Unit: severity_class mapping is correct
- Integration: reads real L2 artifacts, produces 30-50 entries

- Integration: no duplicate IDs

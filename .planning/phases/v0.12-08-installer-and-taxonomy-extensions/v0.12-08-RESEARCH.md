## Summary

Phase v0.12-08 is the final phase of the v0.12 Formal Verification milestone. It closes two remaining gaps: GAP-7 (installer rollback soundness and config sync completeness) via extension of the existing `formal/alloy/install-scope.als`, and GAP-8 (Haiku taxonomy injection safety and closed/open category consistency) via a new `formal/alloy/taxonomy-safety.als` spec. A single new CJS runner `bin/run-installer-alloy.cjs` targets both specs. The phase follows the established three-plan wave pattern (Wave 0 RED stubs → Alloy spec authoring → runner implementation + GREEN tests).

The GAP-7 work extends the existing `install-scope.als` with two new predicates: `RollbackSound` (uninstall is the inverse of install — the state after uninstall matches a baseline pre-install state) and `ConfigSyncComplete` (after install, the set of file atoms installed to hooks/dist/ equals the set installed to ~/.claude/hooks/). The install-scope.als module already has the `Runtime`, `Scope`, and `InstallState` sigs; the extension adds new sigs or predicates modeling the before/after filesystem states.

The GAP-8 work creates a new `taxonomy-safety.als` module modeling the `classifyWithHaiku` function in `bin/update-scoreboard.cjs` (lines 283–331). The function accepts a `taskDescription` string and existing `categories` map, then returns `{ category, subcategory, is_new }`. Three properties must hold: `NoInjection` (the taskDescription field is opaque — its string content cannot structurally alter the category map), `TaxonomyClosed` (when `is_new=false`, the returned category must already exist in the categories map), and `NewCategoryConsistent` (when `is_new=true`, the returned category must NOT already exist in the categories map). Alloy's relational model handles string content as opaque atoms, making injection-safety natural to express.

**Primary recommendation:** Extend install-scope.als with two new preds using the before/after InstallState pattern; create taxonomy-safety.als with abstract sigs for TaskDescription, Category, Subcategory, and a Classification sig carrying is_new; implement bin/run-installer-alloy.cjs following the exact structure of run-audit-alloy.cjs with VALID_SPECS = ['install-scope', 'taxonomy-safety'].

### Pattern 1: Extending Existing Alloy Module (GAP-7)

**What:** Add new predicates to install-scope.als that model filesystem states before/after install and uninstall operations. Use new sigs `FileSet` and `InstallOp` to represent the set of files written to target dirs. The existing `InstallState` sig with `assigned: Runtime -> one Scope` is the scope matrix; the extension adds filesystem-level predicates.

**When to use:** When the existing module has the vocabulary but lacks new properties. Extension is preferable to a new file when the new preds are semantically part of the same domain.

**Example — RollbackSound pred design:**
```alloy
-- Source: derived from install-scope.als pattern + uninstall() in bin/install.js

-- FileSet: abstract set of installed file atoms
-- Models: the set of GSD files written by installRuntime() / removed by uninstall()
abstract sig FileToken {}

-- InstallSnapshot: the set of file tokens present in a directory at a point in time
sig InstallSnapshot {
    files: set FileToken
}

-- RollbackSound: uninstall produces the same file set as never-installed
-- Formalizes: uninstall() removes exactly the files install() added — no more, no less
pred RollbackSound [before, after: InstallSnapshot] {
    -- After uninstall, no GSD files remain (same as pre-install baseline)
    after.files = before.files
}

assert RollbackSoundCheck {
    all pre, post: InstallSnapshot | RollbackSound[pre, post]
}
```

**Example — ConfigSyncComplete pred design:**
```alloy
-- ConfigSyncComplete: after install, hooks/dist/ and ~/.claude/hooks/ file sets are identical
-- Formalizes: the install sync step (cp hooks/qgsd-*.js hooks/dist/ && node bin/install.js)
--             leaves both directories with the same hook file tokens
pred ConfigSyncComplete [distSnapshot, claudeSnapshot: InstallSnapshot] {
    distSnapshot.files = claudeSnapshot.files
}

assert ConfigSyncCompleteCheck {
    all dist, claude: InstallSnapshot |
        ConfigSyncComplete[dist, claude]
}
```

### Pattern 2: Taxonomy Safety Spec (GAP-8)

**What:** Model the `classifyWithHaiku` function in update-scoreboard.cjs as an Alloy module. The key insight: Alloy has no string type — all atoms are opaque. This makes injection-safety natural: a `TaskDescription` atom cannot structurally influence a `Category` atom because they are different sig types with no relation between them unless explicitly defined.

**The three assertions to verify:**
1. `NoInjection` — The relation from TaskDescription to Classification is functional (one output per input) and the content of the TaskDescription cannot force a specific category membership. In Alloy, this is expressed by showing that the classification function's image over Category is bounded by the existing category set regardless of which TaskDescription atom is used.
2. `TaxonomyClosed` — When `is_new = False`, the Classification's category is already a member of the existing `KnownCategories` set.
3. `NewCategoryConsistent` — When `is_new = True`, the Classification's category is NOT a member of the existing `KnownCategories` set before classification.

**When to use:** Any algorithm that maps free-form string content to a structured taxonomy. Alloy's atom-based model makes string injection structurally impossible to express — which is exactly what the spec needs to confirm.

**Example — taxonomy-safety.als design:**
```alloy
-- formal/alloy/taxonomy-safety.als
-- Source: bin/update-scoreboard.cjs classifyWithHaiku() lines 283-331
--
-- Models the Haiku auto-classification function:
--   input:  taskDescription (free-form string atom), categories (existing taxonomy)
--   output: Classification { category, subcategory, is_new }
--
-- Assertions:
--   NoInjection:          TaskDescription content cannot alter Category/Subcategory structure
--   TaxonomyClosed:       is_new=False => category is in KnownCategories before classification
--   NewCategoryConsistent: is_new=True => category is NOT in KnownCategories before classification

module taxonomy_safety

-- Opaque task description atom — models the free-form string passed to Haiku
abstract sig TaskDescription {}

-- Category and Subcategory atoms — models the taxonomy keys/values
abstract sig Category {}
abstract sig Subcategory {}

-- Boolean for is_new field
abstract sig Bool {}
one sig True, False extends Bool {}

-- KnownCategories: the set of categories that existed BEFORE classification
-- Models: Object.keys(data.categories) at line 295 of update-scoreboard.cjs
sig KnownCategories {
    cats: set Category
}

-- Classification: the output of classifyWithHaiku()
-- Models: { category, subcategory, is_new } return value
sig Classification {
    category:    one Category,
    subcategory: one Subcategory,
    is_new:      one Bool
}

-- ClassifyFn: the classification function — maps TaskDescription to Classification
-- Models: classifyWithHaiku(taskDescription, categories) call
sig ClassifyFn {
    maps: TaskDescription -> one Classification
}

-- ── Assertions ──────────────────────────────────────────────────────────────

-- NoInjection: the TaskDescription atom is structurally isolated from Category
-- In Alloy, there is no path from TaskDescription to Category except through
-- ClassifyFn.maps -> Classification.category. This asserts that for any two
-- TaskDescriptions that map to the same Classification, the categories are identical.
-- The "injection" attack vector — using task description content to alter category
-- membership — is impossible because atoms carry no string content in Alloy.
assert NoInjection {
    all f: ClassifyFn, t1, t2: TaskDescription |
        f.maps[t1] = f.maps[t2] =>
        f.maps[t1].category = f.maps[t2].category
}

-- TaxonomyClosed: is_new=False implies category was already known
assert TaxonomyClosed {
    all k: KnownCategories, c: Classification |
        c.is_new = False => c.category in k.cats
}

-- NewCategoryConsistent: is_new=True implies category was NOT previously known
assert NewCategoryConsistent {
    all k: KnownCategories, c: Classification |
        c.is_new = True => c.category not in k.cats
}

-- ── Check commands ───────────────────────────────────────────────────────────
-- Scope: 3 TaskDescription, 3 Category, 3 Subcategory, 2 Bool, 3 KnownCategories, 3 Classification, 2 ClassifyFn

check NoInjection           for 3 TaskDescription, 3 Category, 3 Subcategory, 2 Bool, 3 KnownCategories, 3 Classification, 2 ClassifyFn
check TaxonomyClosed        for 3 TaskDescription, 3 Category, 3 Subcategory, 2 Bool, 3 KnownCategories, 3 Classification, 2 ClassifyFn
check NewCategoryConsistent for 3 TaskDescription, 3 Category, 3 Subcategory, 2 Bool, 3 KnownCategories, 3 Classification, 2 ClassifyFn
```

### Pattern 3: Multi-Spec Runner (run-installer-alloy.cjs)

**What:** Single CJS runner targeting both `install-scope` and `taxonomy-safety` specs via `--spec=` flag. Mirrors the run-audit-alloy.cjs structure exactly.

const VALID_SPECS = ['install-scope', 'taxonomy-safety'];

**Key difference from run-transcript-alloy.cjs:** run-installer-alloy.cjs has TWO specs, not one. The error message for invalid spec must list both.

### Anti-Patterns to Avoid

- **Creating separate runners per spec (GAP-7 and GAP-8):** The established pattern is one runner with VALID_SPECS. Separate runners would require two separate package.json entries and two separate Wave 0 test files — unnecessary complexity.
- **Modeling file content in Alloy:** Alloy has no string type. Do not try to model the actual hook file content. Model file tokens as abstract atoms — the invariants are about set membership (which files are where), not file content.
- **Using `run` commands to verify safety assertions:** Use `check` (bounded model checking for assertion violations), not `run` (instance finding). The existing specs all use `check` for safety properties.
- **Scope too large:** The existing specs use small scopes (3–5 atoms per sig). Larger scopes make bounded model checking slow. Use 3 atoms per sig for taxonomy-safety.als.
- **Forgetting to update package.json in Wave 0 (Plan 01):** Per established pitfall from v0.12-07 research: Wave 0 (Plan 01) MUST include both the test file creation AND the package.json scripts.test update. Never split these across plans.

## Common Pitfalls

### Pitfall 1: TaxonomyClosed/NewCategoryConsistent are trivially true without constraint
**What goes wrong:** If `KnownCategories.cats` is unconstrained (can be any set of Category atoms), then TaxonomyClosed is vacuously true when `c.category in k.cats` holds for some k, and vacuously false for others. The assertion must quantify over ALL KnownCategories, not SOME.
**Why it happens:** Alloy's default quantifier is "for all valid model instances." Without a fact constraining the categories map to be the specific set known before classification, the check finds trivial counterexamples.
**How to avoid:** Use `all k: KnownCategories, c: Classification | ...` (quantify over all combinations). This correctly models "for any state of the taxonomy, any classification must respect is_new."
**Warning signs:** Alloy finds a counterexample immediately in a 1-atom scope — this indicates a vacuous model problem.

### Pitfall 2: install-scope.als extension breaks existing assertions
**What goes wrong:** Adding new sigs (like `FileToken`, `InstallSnapshot`) to install-scope.als could interfere with the existing `InstallState` assertions if the new sigs create unexpected interactions with existing check scopes.
**Why it happens:** Alloy's bounded model checking uses a global scope. New sigs in the same module are included in every check's universe.
**How to avoid:** Add new preds and sigs at the bottom of install-scope.als. Use `check` commands with explicit scope bounds that include the new sig names. Verify existing assertions still pass after adding new content.
**Warning signs:** Any of the three existing check commands (`check NoConflict`, `check AllEquivalence`, `check InstallIdempotent`) fail after the extension.

### Pitfall 3: Counterexample detection requires stdout scan (not exit code)
**What goes wrong:** Alloy 6 exits 0 even when a counterexample is found. A runner that only checks `spawnSync` status will silently miss counterexamples.
**Why it happens:** Alloy's headless mode (`exec --output - --type text --quiet`) writes counterexample output to stdout and exits 0.
**How to avoid:** Always scan stdout for `/Counterexample/i` pattern after the spawnSync call. This is already implemented in all existing runners — copy the pattern exactly.
**Warning signs:** Runner exits 0 but Alloy output contains "Counterexample found."

### Pitfall 4: Wave 0 package.json update not in Plan 01
**What goes wrong:** If `package.json` is updated in Plan 02 or later (after `bin/run-installer-alloy.test.cjs` exists), `npm test` will fail because the test file is in scripts.test but the file doesn't exist yet.
**Why it happens:** `node --test` with a non-existent file path causes an error that aborts the entire test run.
**How to avoid:** Wave 0 (Plan 01) MUST include both the test file creation AND the `package.json` scripts.test update in the same plan. This is the established pattern from v0.12-07-01-PLAN.md.
**Warning signs:** `npm test` exits with "Cannot find module" or "file not found" error for the new test file.

### Pitfall 5: NoInjection assertion may be trivially satisfiable
**What goes wrong:** `NoInjection` as stated ("two TaskDescriptions that map to same Classification have same category") is a trivial identity, not an injection-safety proof.
**Why it happens:** "Injection safety" in the security sense (user content cannot alter program structure) is already guaranteed by Alloy's atom model. The NoInjection assertion needs to capture a meaningful property of the classification function.
**How to avoid:** Frame NoInjection as: for any ClassifyFn, the image of the function over Category (the set of categories it can output) is a subset of KnownCategories.cats union {one additional category when is_new=True}. This is a structural constraint on the function, not a tautology.
**Warning signs:** Alloy verifies NoInjection for scope 1 in 0ms — it may be checking a trivially true statement.

## Open Questions

1. **NoInjection assertion scope adequacy**
   - What we know: Alloy atoms carry no string content; injection-safety is structurally guaranteed by the type system. `NoInjection` as "same Classification implies same Category" is trivially true.
   - What's unclear: Should NoInjection be reframed as a structural constraint on ClassifyFn (its image over Category is bounded) rather than a tautological identity? The success criteria says "taskDescription content cannot alter category structure" — this is satisfied by construction in Alloy.
   - Recommendation: Frame NoInjection as "the classify function cannot map the same TaskDescription to two different Category atoms" (functional constraint). This is meaningful (no non-determinism in classification) and verifiable. The deeper injection-safety claim (content isolation) is a structural fact of Alloy's model that the spec documents by design.

2. **RollbackSound scope: what constitutes "previous state"?**
   - What we know: The uninstall() function in bin/install.js (lines 964–1244) removes specific GSD files/dirs, not arbitrary filesystem state. It does NOT restore backups — it removes GSD artifacts.
   - What's unclear: "Uninstall restores previous state" could mean (a) all GSD files removed (simpler), or (b) exact pre-install state restored (requires modeling pre-install state). The success criteria says "uninstall restores previous state."

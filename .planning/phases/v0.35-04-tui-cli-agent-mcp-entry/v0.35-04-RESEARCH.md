# Phase v0.35-04: TUI CLI Agent MCP Entry - Research

**Researched:** 2026-03-13
**Domain:** TUI workflow for generating working MCP configuration entries for native CLI agent slots
**Confidence:** HIGH

## Summary

The nForma TUI's "Add Agent → CLI Agent" flow currently writes incomplete MCP entries to `~/.claude.json` — it captures the bare CLI command name (e.g., "codex") but fails to resolve it to a full executable path. This breaks on any system where the CLI binary is not at the hardcoded `/opt/homebrew/bin/` location (Linux, WSL, non-standard Homebrew setups). The phase requires the TUI to call `resolveCli()` from the existing `bin/resolve-cli.cjs` utility (introduced in v0.35-03 and now integrated into unified-mcp-server.mjs and call-quorum-slot.cjs) at the moment the user submits the CLI agent form. The generated MCP entry must match the format that mcp-setup produces — specifically, the `command` field in `~/.claude.json` must contain the resolved absolute path (not a bare name), and the entry must ping successfully without requiring manual config edits.

**Primary recommendation:** Integrate `resolveCli()` into the TUI's "CLI Agent" form handler (nForma.cjs) to resolve the CLI path before writing the MCP entry. The resolved path should be written to `mcpServers[slotName].command`, and the generated entry should be validated by an immediate health ping that verifies the binary is executable.

## Architecture Patterns

### Current TUI CLI Agent Flow (nForma.cjs)
**What:** When user selects "Add Agent → CLI Agent" in the TUI, the flow prompts for CLI command name, main tool, model, and timeout, then writes directly to ~/.claude.json.

**Current bug:** The `command` field is written as a bare name (e.g., "codex") instead of a resolved path.

```javascript
// Current (BROKEN): nForma.cjs line ~1850
if (typeChoice.value === 'cli') {
  const command  = await promptInput({ title: 'Add Agent — CLI command', prompt: 'CLI command (e.g. codex, gemini, gh):' });
  // ...
  data.mcpServers = { ...servers, [slotName]: { type: 'stdio', command, args: [] } };
  writeClaudeJson(data);  // BUG: writes bare "codex", not "/opt/homebrew/bin/codex"
}
```

**When to use:** For any new CLI agent added via TUI; resolves paths at form submission time, not at MCP startup.

### Correct Pattern: Resolve Before Writing
**What:** Call `resolveCli(command)` immediately after the user enters the CLI command name, before writing to ~/.claude.json.

**Example (proposed fix):**
```javascript
// Proposed fix: nForma.cjs
const { resolveCli } = require('./resolve-cli.cjs');

if (typeChoice.value === 'cli') {
  const command  = await promptInput({ title: 'Add Agent — CLI command', prompt: 'CLI command (e.g. codex, gemini, gh):' });
  if (!command) { toast('Command is required', true); return; }

  // RESOLVE THE PATH (new step)
  const resolvedCommand = resolveCli(command);  // → '/opt/homebrew/bin/codex' or '/usr/bin/codex' or 'codex'

  // Verify path is executable (new validation)
  const { accessSync, constants } = require('fs');
  try {
    accessSync(resolvedCommand, constants.X_OK);
  } catch (e) {
    toast(`CLI not found or not executable: ${resolvedCommand}`, true);
    return;
  }

  // Write the RESOLVED path, not the bare name
  data.mcpServers = { ...servers, [slotName]: { type: 'stdio', command: resolvedCommand, args: [] } };
  writeClaudeJson(data);
}
```

### Integration Points

- Import resolveCli at the top of the file

- Validate the resolved path is executable (fs.accessSync with X_OK flag)

- After writing the MCP entry, optionally ping the CLI to verify it's executable

- If ping fails, display a warning but don't block (graceful degradation)

## Common Pitfalls

### Pitfall 1: Writing Bare CLI Name Instead of Resolved Path
**What goes wrong:** TUI accepts "codex" and writes `"command": "codex"` to ~/.claude.json. On Linux with /usr/bin/codex, Claude Code fails to find the binary because MCP expects an absolute path.

**Why it happens:** Developer assumes the OS will resolve "codex" via PATH when Claude Code spawns the process. In reality, MCP child_process spawn with non-absolute path depends on the cwd and environment at spawn time, which is fragile.

**How to avoid:** Always resolve the path at form submission time using resolveCli(). Write the absolute path, not the bare name. Validate with fs.accessSync() before writing.

**Warning signs:** User reports "agent not found" or "permission denied" on Linux after adding CLI agent via TUI; mcp-status shows agent in roster but quorum round fails with spawn errors.

### Pitfall 2: Forgetting to Import resolveCli in nForma.cjs
**What goes wrong:** Code calls resolveCli() but the function isn't imported. Runtime error when user adds a CLI agent.

**Why it happens:** Easy to miss when editing the TUI; resolve-cli.cjs is in a different directory.

**How to avoid:** Add the import at the top of nForma.cjs near other requires. Run the TUI after the change and test the "Add Agent" flow manually.

**Warning signs:** Quorum session or TUI test fails with "resolveCli is not defined" or "Cannot find module './resolve-cli.cjs'".

### Pitfall 3: Not Validating Resolved Path is Executable
**What goes wrong:** User enters "nonexistent-cli" and TUI writes a bare name. resolveCli() returns "nonexistent-cli" (bare fallback) and write succeeds. Later, MCP startup fails with unhelpful "spawn ENOENT" error.

**Why it happens:** resolveCli() is fail-open (never throws, returns bare name as fallback). This is correct for runtime code (let OS resolve it), but for TUI we should validate that the path actually exists and is executable.

**How to avoid:** After calling resolveCli(), validate with fs.accessSync(resolvedPath, fs.constants.X_OK) before writing. If validation fails, toast an error to the user and stay on the form (don't write the entry).

**Warning signs:** mcp-status shows agent as "UNAVAIL" or "UNKNOWN"; MCP logs show spawn errors for the agent.

### Pitfall 4: Mismatched Entry Format Between TUI and mcp-setup
**What goes wrong:** TUI writes a different structure than mcp-setup (e.g., includes extra env fields, different args format). User expects both flows to produce identical entries but they diverge.

**Why it happens:** TUI and mcp-setup.md evolve independently; no automated test comparing the two formats.

**How to avoid:** Document the exact expected format in RESEARCH.md (done). Create a unit test that compares a TUI-generated entry with a mcp-setup-generated entry for the same CLI type. Verify both have type='stdio', resolved command, and args=[].

**Warning signs:** TUI-added CLI agents work differently than mcp-setup-added agents; user reports confusion.

## Open Questions

1. **Should TUI also resolve args_template for each CLI type?**
   - What we know: providers.json contains args_template for each CLI (e.g., Codex: `["exec", "{prompt}"]`). TUI currently writes args: [] (empty).
   - What's unclear: Should TUI copy the correct args_template from providers.json based on the CLI name entered?
   - Recommendation: Keep args: [] for v0.35-04 scope. Document that custom CLI agents support empty args for now. Future phase (e.g., v0.36) can add args_template lookup and write.

2. **Should TUI perform a health ping to the CLI immediately after writing?**
   - What we know: mcp-setup doesn't do a health ping; it just writes the entry. Errors surface later when MCP starts.
   - What's unclear: Is immediate validation via spawn(..., ['--version']) beneficial, or does it add UX friction?
   - Recommendation: Optional health ping (show result but don't block). If ping succeeds, display "✓ Verified at <resolved-path>". If it fails, show a warning but still write the entry. Error feedback is valuable for immediate debugging.

3. **How should the TUI handle CLIs that don't exist at all?**
   - What we know: resolveCli() returns the bare name as a fallback (fail-open).
   - What's unclear: Should TUI reject completely nonexistent CLIs, or allow them to be written?
   - Recommendation: Validate with fs.accessSync() as described in the pitfalls section. If validation fails, toast an error and don't write the entry. This prevents broken entries while remaining helpful.

## Summary

Phase v0.15-01 is a pure regex surgery task. `qgsd-core/bin/gsd-tools.cjs` contains three health-check patterns written for GSD's legacy `NN-name` numeric phase naming that produce 64 false positives when run against the QGSD repo which uses `v0.X-YY-name` versioned phase directories.

The three bugs are independent and located in three distinct code sections:
1. **W005** (line 3833): Dir naming check uses `/^\d{2}(?:\.\d+)?-[\w-]+$/` ‚Äî rejects any directory not starting with two digits. All 41 `v0.X-YY-name` dirs fail this check.
2. **W007** (lines 3863, 3874): Two sub-bugs. (a) ROADMAP extractor `/#{2,4}\s*Phase\s+(\d+(?:\.\d+)?)\s*:/gi` cannot capture `v0.X-YY` phase IDs. (b) Disk dir extractor `/^(\d+(?:\.\d+)?)/` cannot extract IDs from `v0.X-YY-name` dirs. Together this produces 22 W007 warnings for phases that exist in both ROADMAP and disk.
3. **W002** (lines 3781, 3788): Same two-sub-bug pattern as W007 but for STATE.md validation. STATE.md contains `Phase v0.X-YY` references that the STATE extractor regex `/[Pp]hase\s+(\d+(?:\.\d+)?)/g` cannot capture.

The fix is replacing four regex literals and syncing the installed copy via `node bin/install.js --claude --global`. The file has 141 passing tests that must remain green.

**Primary recommendation:** Fix the four regex literals in `qgsd-core/bin/gsd-tools.cjs`, write three test cases in `gsd-tools.test.cjs`, sync install, verify with a before/after run of `validate health` on the QGSD repo.

## Common Pitfalls

### Pitfall 1: Name segment contains dots
**What goes wrong:** `v0.9-08-post-v0.9-install-sync` contains `.9` inside the name portion. The original W005 replacement regex `/^v\d+\.\d+-\d{2}(?:\.\d+)?-[\w-]+$/` fails because `[\w-]` does not match `.`.
**Why it happens:** `\w` = `[a-zA-Z0-9_]` ‚Äî no dot.
**How to avoid:** Use `[.\w-]+` (allow dots) in the name portion of the W005 pattern.
**Confirmed fix:** `/^(v\d+\.\d+-\d{2}(?:\.\d+)?-[.\w-]+|\d{2}(?:\.\d+)?-[\w-]+)$/` passes all 41 versioned dirs including `v0.9-08-post-v0.9-install-sync`.

### Pitfall 2: Two disk extractor sites, not one
**What goes wrong:** Developer fixes W007's disk extractor (line 3874) but misses W002's disk extractor (line 3788). W002 false positives persist.
**Why it happens:** The code is not DRY ‚Äî the same pattern appears in Check 4 and Check 8 independently.
**How to avoid:** Search for all occurrences of `/^(\d+(?:\.\d+)?)/` in the file ‚Äî there are 2 that need fixing (lines 3788 and 3874).

### Pitfall 3: Install sync omitted
**What goes wrong:** Fix applied to `qgsd-core/bin/gsd-tools.cjs`, tests pass, but `validate health` still shows false positives because the installed copy at `~/.claude/qgsd/bin/gsd-tools.cjs` is stale.
**Why it happens:** `/qgsd:` commands run from `~/.claude/qgsd/`, not from `qgsd-core/`.
**How to avoid:** Always include `node bin/install.js --claude --global` as a task step after editing `qgsd-core/`.

### Pitfall 4: W002 W002 false positive persists after fix
**What goes wrong:** STATE.md Check 4 regex is fixed, but disk extractor on line 3788 is not. The state ref "v0.15-01" is now extracted, but `diskPhases.has("v0.15-01")` returns false because `diskPhases` still only contains empty matches (no ID extracted from `v0.15-01-health-checker-regex-fix`).
**How to avoid:** Both the STATE extractor (line 3781) AND the disk extractor (line 3788) must be fixed together in Check 4.

### Pitfall 5: Existing tests break
**What goes wrong:** The existing decimal-phase test `FIX-04-TC-02` uses `Phase: 06.1` in STATE.md and `06.1-decimal-fix` dir. If the new STATE.md regex is accidentally made too narrow (e.g., only matches `v\d+...`), it stops matching `06.1` and the test fails.
**How to avoid:** The fixed STATE.md regex must be a union `/v\d+\.\d+-\d{2}...|d+(?:\.\d+)?/` ‚Äî both alternatives, not just the versioned one.

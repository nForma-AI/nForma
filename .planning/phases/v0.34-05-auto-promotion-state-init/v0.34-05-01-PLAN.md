Two changes to nf-solve.cjs to satisfy PROMO-02 and improve PROMO-03 diagnostics:

**Change 1: Explicit initialization in preflight (line ~3675-3677)**

After the existing `existingSolveState` try/catch block that reads solve-state.json, add an explicit initialization guard:

```javascript
// Ensure consecutive_clean_sessions is initialized (PROMO-02)
if (existingSolveState.consecutive_clean_sessions == null) {
  existingSolveState.consecutive_clean_sessions = 0;
}
```

**Intent clarification:** The existing `|| 0` fallback at line 3816 (`const prevClean = (existingSolveState && existingSolveState.consecutive_clean_sessions) || 0;`) already safely handles the missing field case. This explicit initialization guard is added for **code clarity only** — it makes the initialization contract visible at preflight time and prevents edge cases where other code reads `existingSolveState.consecutive_clean_sessions` before the persistence block. It is NOT functionally required given the `|| 0` fallback, but it satisfies PROMO-02's explicit initialization requirement and makes the code self-documenting. Add a code comment stating this intent.

**Change 2: Add diagnostic logging to checkCleanSession() (line ~3636-3660)**

Inside the `for` loop in checkCleanSession() where gate files are read, add a diagnostic log line that distinguishes between "loaded semantic_score from file" vs "defaulted to 0". This makes it visible in solve output whether semantic scoring is working:

After the line `semantic[label] = gateData.semantic_score != null ? gateData.semantic_score : 0;`, add:

```javascript
if (gateData.semantic_score == null) {
  process.stderr.write(TAG + ' checkCleanSession: gate ' + label + ' semantic_score defaulted to 0 (field missing)\n');
} else {
  process.stderr.write(TAG + ' checkCleanSession: gate ' + label + ' semantic_score=' + gateData.semantic_score + ' (from file)\n');
}
```

This logs BOTH cases: when semantic_score is MISSING (the failure case from Pitfall 1 in research) AND when it is present with its actual value. Logging successful reads validates that the v0.34-04 semantic scoring pipeline ran and populated gates — this is a PROMO-03 diagnostic requirement. Without it, a below-threshold semantic_score (e.g., 0.75) would be invisible, making it hard to debug why auto-promotion is not firing despite clean wiring scores.

Do NOT change:
- The checkCleanSession() threshold logic (wiring >= 1.0, semantic >= 0.8) — it is correct
- The state persistence block at line 3815-3822 — the `|| 0` fallback pattern is correct
- The autoPromote() call site — it already passes consecutive_clean_sessions correctly

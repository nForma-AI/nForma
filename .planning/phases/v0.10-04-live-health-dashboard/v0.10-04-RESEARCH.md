## Summary

Phase v0.10-04 implements a full-screen live health dashboard for `bin/manage-agents.cjs`. The primary technical challenge is the **mode-switch boundary** between inquirer (which holds the TTY in its own line-editing mode) and a raw keypress loop (which needs exclusive stdin control). These two cannot coexist — inquirer must be fully resolved (its Promise settled) before entering raw mode.

The required architecture is: `mainMenu()` calls `await liveDashboard()` → on return from inquirer prompt, `liveDashboard()` does NOT call inquirer again — instead it enters a raw stdin loop independently. On exit (`q` / Escape), the loop calls `setRawMode(false)`, `removeAllListeners('keypress')`, and then returns to `mainMenu()` which re-enters inquirer. This is the only safe pattern; any interleaving of inquirer and raw mode produces character-swallowing or frozen prompts.

The non-TTY fallback (CI/piped output) is a hard requirement. A `process.stdout.isTTY` guard must fire before any `setRawMode()` call. In non-TTY contexts the dashboard performs a single synchronous health print and returns immediately — no raw mode, no keypress loop.

**Primary recommendation:** Use `readline.emitKeypressEvents(process.stdin)` + `process.stdin.setRawMode(true)` inside `liveDashboard()` only, with a strict cleanup sequence on exit. Use `process.stdout.write('\x1b[2J\x1b[H')` for full-screen clear-and-home. Reuse the existing `probeProviderUrl()` function for health probing. All display logic extracted as pure functions for testability.

## Architecture Patterns

### Recommended Function Structure

```
bin/manage-agents.cjs additions:
├── renderDashboard(slots, healthMap, lastUpdated)   # pure: builds display string
├── probAllSlots(mcpServers, secretsLib)              # async: calls probeProviderUrl per slot
├── liveDashboard()                                   # async: orchestrates raw mode loop
└── mainMenu() — add 'dashboard' case to dispatch
```

### Pattern 1: Readline Mode-Switch Architecture

**What:** inquirer and raw mode are mutually exclusive — mode-switch requires inquirer to fully settle before entering raw mode.

**When to use:** Any time you transition from an inquirer prompt to a raw keypress loop (or vice versa).

**The invariant:** `liveDashboard()` is called from `mainMenu()` after inquirer resolves the `action` prompt. `liveDashboard()` never calls inquirer itself. On exit from the dashboard, `liveDashboard()` restores stdin and returns, then `mainMenu()` re-enters its `while (running)` loop which calls inquirer again.

```javascript
// Source: https://nodejs.org/api/readline.html + https://nodejs.org/api/tty.html

async function liveDashboard() {
  // TTY guard (DASH-01 SC4) — non-TTY fallback: single static print
  if (!process.stdout.isTTY) {
    await runStaticHealthPrint();
    return;
  }

  // Enter raw mode AFTER inquirer has fully resolved
  const readline = require('readline');
  readline.emitKeypressEvents(process.stdin);
  process.stdin.setRawMode(true);
  process.stdin.resume();

  // ... dashboard loop ...

  // Cleanup BEFORE returning to mainMenu() (Pattern 3 below)
}
```

**Why this works:** `readline.emitKeypressEvents()` tells Node to parse the raw bytes on stdin into structured `{ name, ctrl, meta, shift }` keypress objects. Without this call, `stdin.on('keypress', ...)` never fires. The call is idempotent if stdin is already emitting keypresses (safe to call multiple times).

### Pattern 2: Render Loop with Timestamp + Stale Detection

**What:** Dashboard renders on demand (keypress) rather than on a timer. Timer is used only for stale detection — it does NOT trigger renders while inquirer might hold the TTY.

**When to use:** Exactly as specified in SC2 — space/r triggers refresh; 60-second stale threshold.

```javascript
// Source: Node.js built-in Date API (no library needed)

async function liveDashboard() {
  // ...
  let healthMap = {};
  let lastUpdated = null;  // null = never refreshed

  async function doRefresh() {
    renderSpinner();  // show "Probing..." state immediately
    healthMap = await probeAllSlots(mcpServers, secretsLib);
    lastUpdated = Date.now();
    renderDashboard(slots, healthMap, lastUpdated);
  }

  // Initial render
  await doRefresh();

  // Stale check: use setInterval only for stale indicator update, not for probing
  // Interval is set AFTER raw mode is entered; cleared before raw mode exits
  const staleInterval = setInterval(() => {
    if (lastUpdated && Date.now() - lastUpdated > 60_000) {
      // Re-render just to update stale warning — no new HTTP probe
      renderDashboard(slots, healthMap, lastUpdated);
    }
  }, 5_000);  // Check every 5s is sufficient for 60s threshold

  // Keypress handler (see Pattern 3 for exit)
  function onKeypress(str, key) {
    if (key && (key.name === 'space' || key.name === 'r')) {
      doRefresh();  // fire-and-forget; renders when done
    }
    if (key && (key.name === 'q' || key.name === 'escape' ||
                (key.ctrl && key.name === 'c'))) {
      cleanup();
    }
  }

  process.stdin.on('keypress', onKeypress);

  // Wait for exit signal
  await new Promise((resolve) => { exitResolve = resolve; });

  // ... cleanup ...
}
```

**Stale warning rendering:**

```javascript
// Source: pattern from DASH-02 spec — 60s threshold, yellow ANSI

function formatTimestamp(ts) {
  if (!ts) return 'Never';
  const d = new Date(ts);
  return d.toTimeString().slice(0, 8);  // HH:MM:SS
}

function renderDashboard(slots, healthMap, lastUpdated) {
  const stale = lastUpdated && Date.now() - lastUpdated > 60_000;
  const tsLine = lastUpdated
    ? `Last updated: ${formatTimestamp(lastUpdated)}` +
      (stale ? '  \x1b[33m[stale]\x1b[0m' : '')
    : 'Last updated: —';
  // ... render rows ... then:
  process.stdout.write('\n' + tsLine + '\n');
  process.stdout.write('  [space/r] refresh   [q] exit\n');
}
```

### Pattern 3: Clean Exit Sequence

**What:** Strict ordering for stdin restoration before returning to inquirer.

**When to use:** Every exit path from the dashboard — `q`, Escape, Ctrl+C, and any error path.

```javascript
// Source: https://nodejs.org/api/tty.html (setRawMode docs)
//         https://nodejs.org/api/readline.html (emitKeypressEvents note)

function cleanup() {
  clearInterval(staleInterval);
  // Step 1: Remove listener BEFORE disabling raw mode
  //         (prevents any stale keypress event firing during teardown)
  process.stdin.removeAllListeners('keypress');
  // Step 2: Exit raw mode
  process.stdin.setRawMode(false);
  // Step 3: Pause stdin (prevents Node keeping process alive)
  process.stdin.pause();
  // Step 4: Restore cursor
  process.stdout.write('\x1b[?25h');
  // Step 5: Clear screen before returning to menu
  process.stdout.write('\x1b[2J\x1b[H');
  // Step 6: Signal the await Promise to resolve
  exitResolve();
}
```

**Critical order:** `removeAllListeners` BEFORE `setRawMode(false)`. If you call `setRawMode(false)` first, the TTY mode-change can emit a spurious event that fires the keypress handler during teardown.

### Pattern 4: Non-TTY Static Fallback

**What:** When stdout is not a TTY (CI, piped), skip all raw mode and render a one-shot table.

**When to use:** Always guard the live dashboard entry point with `process.stdout.isTTY`.

```javascript
// Source: https://nodejs.org/api/tty.html — isTTY property docs

async function liveDashboard() {
  if (!process.stdout.isTTY) {
    // Non-TTY: single health print, no keypress loop
    const healthMap = await probeAllSlots(mcpServers, secretsLib);
    renderStaticHealthTable(slots, healthMap);
    return;
  }
  // ... TTY path ...
}
```

**Why `process.stdout.isTTY` not `process.stdin.isTTY`:** The spec says "checks `process.stdout.isTTY` before entering dashboard mode" (SC4). Both will typically be the same value, but `process.stdout.isTTY` is the canonical check for whether the output can handle ANSI escapes — which is the right question for a visual dashboard.

### Anti-Patterns to Avoid

- **setInterval for probing while inquirer runs:** Never start a probe interval inside a function that will return to inquirer. The interval fires in the background and can write to stdout while inquirer is rendering — produces corrupted prompts.
- **Calling inquirer inside liveDashboard():** Cannot use inquirer inside the raw keypress loop. Inquirer takes over stdin in its own mode; nesting with raw mode is undefined behavior.
- **Not removing keypress listener on exit:** If `removeAllListeners('keypress')` is not called, the handler remains registered after `mainMenu()` re-enters inquirer. Any key pressed in the menu also triggers the stale dashboard handler — silent character-swallowing.
- **`process.stdin.on('data', ...)` instead of `'keypress'`:** Raw `data` events return Buffer/string bytes, not structured key objects. Space = `' '`, Escape = `'\x1b'`, arrow keys = multi-byte sequences. Always use `readline.emitKeypressEvents()` + `'keypress'` for named key detection.
- **Calling `process.stdin.destroy()` on exit:** This permanently closes stdin, preventing inquirer from reading it on return to mainMenu. Use `process.stdin.pause()` instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parsing raw key byte sequences | Custom byte parser for Escape, arrows, Ctrl combos | `readline.emitKeypressEvents(process.stdin)` | Multi-byte sequences (ESC+[+A for Up arrow) are complex; readline handles all platform variations |
| Screen clearing | Home-rolled cursor math | `\x1b[2J\x1b[H` ANSI sequence | One line; no library needed |
| Health probing | New HTTP client | Existing `probeProviderUrl()` | Already battle-tested with 7s timeout, healthy-status array, and error handling |
| Timestamp formatting | Custom date formatter | `new Date(ts).toTimeString().slice(0, 8)` | Built-in; already used in `writeKeyStatus` pattern |

**Key insight:** Node.js built-ins cover 100% of the raw mode / keypress domain. No npm package is needed or appropriate given the project's zero-new-deps constraint.

## Common Pitfalls

### Pitfall 1: inquirer + raw mode interleaving causes frozen prompt

**What goes wrong:** `liveDashboard()` calls inquirer for a "Press any key" confirmation while stdin is in raw mode. The prompt renders but never resolves — inquirer expects line-buffered input, not raw characters.

**Why it happens:** inquirer uses readline internally; readline's line-editing mode and raw mode are mutually exclusive.

**How to avoid:** Never call `inquirer.prompt()` inside `liveDashboard()`. All inquirer calls must be in `mainMenu()` only. The `liveDashboard()` function handles its own input via raw keypress listeners.

**Warning signs:** Prompt appears on screen but doesn't respond to input; `node` process hangs after closing.

### Pitfall 2: Stale keypress listener survives return to mainMenu

**What goes wrong:** User presses `q` to exit dashboard. Dashboard returns. User navigates the main menu. Suddenly a menu keypress triggers a re-render of the health table mid-prompt, corrupting the display.

**Why it happens:** `process.stdin.on('keypress', handler)` is still registered because cleanup didn't call `removeAllListeners('keypress')`.

**How to avoid:** Always call `process.stdin.removeAllListeners('keypress')` as the first step in the cleanup sequence, before `setRawMode(false)`.

**Warning signs:** Visible console noise after returning to main menu; characters typed at menu prompts trigger unexpected output.

### Pitfall 3: `setInterval` probe runs while inquirer holds TTY

**What goes wrong:** `setInterval(() => probeAllSlots(), 10_000)` is started in `liveDashboard()` but the interval fires after the user exits to mainMenu. The probe completes and calls `process.stdout.write(...)` while inquirer is rendering a prompt — visual corruption.

**Why it happens:** `setInterval` does not know about raw mode or inquirer state.

**How to avoid:** Call `clearInterval(staleInterval)` in the cleanup function before resolving the exit promise. Never use setInterval for probe scheduling; only use it for the stale-indicator re-render (low-frequency: 5s), and always clear it on exit.

**Warning signs:** Random blank lines or partial re-renders appearing in the main menu display.

### Pitfall 4: `process.stdin.destroy()` prevents re-entry into inquirer

**What goes wrong:** Cleanup calls `process.stdin.destroy()`. On return to `mainMenu()`, inquirer tries to read from stdin but gets `ERR_STREAM_DESTROYED`. The menu freezes or crashes.

**Why it happens:** Developers confuse `pause()` (temporarily suspends data flow) with `destroy()` (permanently closes stream).

**How to avoid:** Use `process.stdin.pause()` not `process.stdin.destroy()`. inquirer internally calls `process.stdin.resume()` when it starts a new prompt.

**Warning signs:** `ERR_STREAM_DESTROYED` or `ERR_STREAM_WRITE_AFTER_END` errors after returning from dashboard.

### Pitfall 5: `process.stdout.write` during async probeAllSlots races with keypress re-render

**What goes wrong:** User presses space. `doRefresh()` starts (async). While probing, user presses space again. Second `doRefresh()` starts. Both complete and write to stdout in unpredictable order — double-rendered rows or partial writes.

**Why it happens:** No concurrency guard on `doRefresh()`.

**How to avoid:** Add an `isRefreshing` boolean flag. If `isRefreshing` is true when keypress fires, ignore the keypress (or queue one deferred refresh). Reset flag on completion.

```javascript
let isRefreshing = false;
async function doRefresh() {
  if (isRefreshing) return;
  isRefreshing = true;
  try {
    // ... probe + render ...
  } finally {
    isRefreshing = false;
  }
}
```

### Pitfall 6: Ctrl+C not caught — raw mode suppresses SIGINT

**What goes wrong:** In raw mode, `setRawMode(true)` disables Ctrl+C → SIGINT. If the keypress handler doesn't handle `key.ctrl && key.name === 'c'`, the user cannot exit the dashboard with Ctrl+C and the process appears frozen.

**Why it happens:** This is documented TTY behavior — raw mode explicitly disables terminal signal processing.

**How to avoid:** Handle `key.ctrl && key.name === 'c'` in the keypress listener and treat it as an exit signal (same as `q`).

**Warning signs:** Ctrl+C in dashboard doesn't exit; user has to kill the process externally.

```javascript
// Source: https://nodejs.org/api/readline.html + https://nodejs.org/api/tty.html

async function liveDashboard() {
  const readline = require('readline');
  const data = readClaudeJson();
  const mcpServers = getGlobalMcpServers(data);
  const slots = Object.keys(mcpServers);

  let secretsLib = null;
  try { secretsLib = require('./secrets.cjs'); } catch (_) {}

  // Non-TTY fallback (DASH-01 SC4)
  if (!process.stdout.isTTY) {
    console.log('\n  Health check (non-TTY mode):\n');
    const healthMap = await probeAllSlots(mcpServers, slots, secretsLib);
    renderStaticHealthTable(slots, mcpServers, healthMap);
    return;
  }

  // Enter raw mode
  readline.emitKeypressEvents(process.stdin);
  process.stdin.setRawMode(true);
  process.stdin.resume();
  process.stdout.write('\x1b[?25l');  // hide cursor

  let healthMap = {};
  let lastUpdated = null;
  let isRefreshing = false;
  let exitResolve;

  async function doRefresh() {
    if (isRefreshing) return;
    isRefreshing = true;
    renderProbing(slots);
    try {
      healthMap = await probeAllSlots(mcpServers, slots, secretsLib);
      lastUpdated = Date.now();
    } finally {
      isRefreshing = false;
    }
    renderScreen(slots, mcpServers, healthMap, lastUpdated);
  }

  function cleanup() {
    clearInterval(staleCheck);
    process.stdin.removeAllListeners('keypress');
    process.stdin.setRawMode(false);
    process.stdin.pause();
    process.stdout.write('\x1b[?25h');  // restore cursor
    process.stdout.write('\x1b[2J\x1b[H');
    exitResolve();
  }

  const staleCheck = setInterval(() => {
    if (lastUpdated && Date.now() - lastUpdated > 60_000) {
      renderScreen(slots, mcpServers, healthMap, lastUpdated);
    }
  }, 5_000);

  function onKeypress(str, key) {
    if (!key) return;
    if (key.name === 'space' || key.name === 'r') { doRefresh(); return; }
    if (key.name === 'q' || key.name === 'escape' ||
        (key.ctrl && key.name === 'c')) { cleanup(); }
  }

  process.stdin.on('keypress', onKeypress);

  await doRefresh();  // initial load

  await new Promise((resolve) => { exitResolve = resolve; });
}
```

### probeAllSlots() — parallel probing with Promise.all

```javascript
// Note: probeAllSlots CAN use Promise.all because it only reads
// (no keychain writes, no qgsd.json writes). This is unlike batchRotateKeys
// which must use sequential for...of for write safety.

async function probeAllSlots(mcpServers, slots, secretsLib) {
  const results = await Promise.all(slots.map(async (slotName) => {
    const cfg = mcpServers[slotName];
    const env = cfg.env || {};
    if (!env.ANTHROPIC_BASE_URL) {
      return [slotName, { healthy: null, latencyMs: 0, statusCode: null, error: 'subprocess' }];
    }
    const account = 'ANTHROPIC_API_KEY_' + slotName.toUpperCase().replace(/-/g, '_');
    let apiKey = env.ANTHROPIC_API_KEY || '';
    if (secretsLib) {
      try { const k = await secretsLib.get('qgsd', account); if (k) apiKey = k; } catch (_) {}
    }
    const probe = await probeProviderUrl(env.ANTHROPIC_BASE_URL, apiKey);
    return [slotName, probe];
  }));
  return Object.fromEntries(results);
}
```

### renderScreen() — full-screen table

```javascript
// Source: existing listAgents() column pattern + ANSI clear sequence

function renderScreen(slots, mcpServers, healthMap, lastUpdated) {
  const lines = buildDashboardLines(slots, mcpServers, healthMap, lastUpdated);
  process.stdout.write('\x1b[2J\x1b[H');
  process.stdout.write(lines.join('\n') + '\n');
}

// Pure function — testable via _pure export
function buildDashboardLines(slots, mcpServers, healthMap, lastUpdated) {
  const lines = [];
  lines.push('  QGSD Live Health Dashboard');
  lines.push('  ' + '─'.repeat(60));
  lines.push('');

  for (const slotName of slots) {
    const cfg = mcpServers[slotName] || {};
    const env = cfg.env || {};
    const model = env.CLAUDE_DEFAULT_MODEL || '—';
    const provider = env.ANTHROPIC_BASE_URL
      ? env.ANTHROPIC_BASE_URL.replace(/^https?:\/\//, '').replace(/\/v\d+\/?$/, '').replace(/\/.*$/, '')
      : (cfg.command || '—');
    const probe = healthMap[slotName];
    let status;
    if (!probe) {
      status = '\x1b[90m—\x1b[0m';
    } else if (probe.error === 'subprocess') {
      status = '\x1b[90msubprocess\x1b[0m';
    } else if (probe.healthy) {
      status = `\x1b[32m✓ UP (${probe.latencyMs}ms)\x1b[0m`;
    } else {
      status = `\x1b[31m✗ DOWN\x1b[0m`;
    }
    lines.push(`  ${slotName.padEnd(14)} ${provider.slice(0, 24).padEnd(24)} ${model.slice(0, 30).padEnd(30)} ${status}`);
  }

  lines.push('');
  const stale = lastUpdated && Date.now() - lastUpdated > 60_000;
  const ts = lastUpdated ? new Date(lastUpdated).toTimeString().slice(0, 8) : '—';
  lines.push(`  Last updated: ${ts}` + (stale ? '  \x1b[33m[stale]\x1b[0m' : ''));
  lines.push('  [space/r] refresh   [q/Esc] exit');
  return lines;
}
```

- `readline.createInterface({ input: process.stdin })` for raw mode dashboards: Creates a readline interface that interferes with inquirer. For raw keypress-only use, call `readline.emitKeypressEvents(process.stdin)` directly without creating an interface.

## Open Questions

1. **Subprocess-provider slots in health display**
   - What we know: Subprocess slots have no `ANTHROPIC_BASE_URL` in their env; `probeProviderUrl()` would receive null and return `{ healthy: false, ... }`.
   - What's unclear: Should subprocess slots show as "subprocess" (not probeable) or be excluded from dashboard?
   - Recommendation: Show them in the table with `subprocess` status label — consistent with the existing `checkAgentHealth()` behavior which says "subprocess provider — no HTTP endpoint to probe." A `probe.error === 'subprocess'` sentinel value in `probeAllSlots()` handles this cleanly.

2. **Cursor position after probing animation**
   - What we know: `renderProbing()` writes a "Probing..." line. `renderScreen()` does `\x1b[2J\x1b[H` which clears everything including the probing line.
   - What's unclear: Should there be a visible "Probing N slots..." intermediate state or just silent refresh?
   - Recommendation: Show "Probing... (N slots)" by clearing screen and rendering a loading row for each slot before results arrive. This is UX polish — not a hard requirement from the spec.

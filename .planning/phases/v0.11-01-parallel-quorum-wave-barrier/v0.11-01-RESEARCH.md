## What We Are Building

Replace the sequential quorum slot-call loop with a wave-barrier pattern:
- **Round 1** fans out as parallel Task spawns (one `qgsd-quorum-worker` per active slot)
- A **barrier** collects all Round 1 results, checks consensus, builds the deliberation context
- **Round 2** fans out the same way with prior positions injected
- A **synthesizer** reads all worker outputs and emits the final verdict

Target: 10–12× wall-clock reduction (N×30s → ~30s for 10 slots).

## 2. Worker Agent Design

**Tools:** `Read, Bash, Glob, Grep` (no Write — worker never touches scoreboard)

**$ARGUMENTS schema (passed by orchestrator):**
```
slot: <slotName>
round: <integer>
timeout_ms: <integer>
repo_dir: <absolute path>
mode: A | B
question: <question text>
[artifact_path: <path>]
[prior_positions: ...]   # Round 2+ only
[traces: ...]            # Mode B only
```

**Execution flow:**
1. Parse $ARGUMENTS
2. Read CLAUDE.md, .planning/STATE.md, artifact file (if set), any relevant files via Grep/Glob
3. Build and run: `node "$HOME/.claude/qgsd-bin/call-quorum-slot.cjs" --slot <slot> --timeout <ms> --cwd <repo_dir>`
4. Return structured result

**Structured output schema:**
```
slot: <slotName>
round: <integer>
verdict: APPROVE | REJECT | FLAG | UNAVAIL | <mode-A position summary>
reasoning: <2-4 sentences>
raw: |
  <first 2000 chars of call-quorum-slot output>
[unavail_message: <first 500 chars, only when verdict=UNAVAIL>]
```

**Reference model:** `agents/qgsd-quorum-test-worker.md` — use same `<role>` / `<output_format>` / `<arguments>` structure.

## 3. Synthesizer Agent Design

The synthesizer is the barrier logic extracted into an agent. It:

1. Receives all worker result blocks (passed as $ARGUMENTS or via `artifact_path`)
2. Parses `slot/verdict/reasoning` fields from each result
3. For slots with `verdict: UNAVAIL` + non-empty `unavail_message`: returns hint text for orchestrator to call `set-availability`
4. Checks consensus (all available slots agree)
5. **If consensus:** emits `CONSENSUS REACHED` + final verdict + positions table
6. **If no consensus:** emits `DELIBERATION NEEDED` + cross-pollination context bundle (the full `Prior positions:` block for Round 2)

The orchestrator reads the synthesizer's output to decide whether to fire Wave 2.

**Tools:** `Read` (reads worker result files or parses $ARGUMENTS — no Bash needed)

## 4. Scoreboard — Two-Step Fix

### Step 1: Atomic rename (do now, 8 lines)
All 4 write sites in `update-scoreboard.cjs` currently use `fs.writeFileSync` directly. Replace with:
```javascript
const tmpPath = absPath + '.' + process.pid + '.tmp';
fs.writeFileSync(tmpPath, JSON.stringify(data, null, 2) + '\n', 'utf8');
fs.renameSync(tmpPath, absPath);
```
This eliminates torn-file corruption permanently.

### Step 2: `merge-wave` subcommand (for parallel dispatch)
```bash
node update-scoreboard.cjs merge-wave \
  --dir .planning/scoreboard-tmp \
  --task "quick-97" --round 1
```
Workers write per-slot temp files to `.planning/scoreboard-tmp/vote-<slot>-<task>-<round>-<pid>.json`. Orchestrator calls `merge-wave` once after the barrier to apply all votes in one transaction.

**Design invariant:** Workers NEVER call `update-scoreboard.cjs` themselves. They return `unavail_message` in their output; the orchestrator processes it sequentially at the barrier.

## 5. Orchestrator Rewrite — Key Changes

### SEQUENTIAL CALLS ONLY update
Add explicit carve-out:
> SEQUENTIAL CALLS ONLY — NO SIBLING TOOL CALLS — **except for the parallel worker wave.**
> Task spawns for the worker wave are explicitly parallel (one Task per slot in one message turn).
> All Bash calls remain sequential.

### Round 1 replacement
Current: sequential `Bash(call-quorum-slot --slot X)` loop
Target: single message turn with one `Task(subagent_type=qgsd-quorum-worker, ...)` per active slot

### Barrier step (after Wave 1)
1. Parse all worker results (slot/verdict/reasoning/raw/unavail_message)
2. Process UNAVAIL results: `Bash(update-scoreboard.cjs set-availability ...)` — sequential
3. Spawn `qgsd-quorum-synthesizer` with all results → get consensus decision or deliberation bundle
4. If consensus: skip Wave 2, proceed to scoreboard updates
5. If no consensus: build Wave 2 Task fan-out with deliberation prompts

### Timeout discovery
The orchestrator pre-resolves `quorum_timeout_ms` per slot from `providers.json` before the Task wave and passes `timeout_ms:` in each worker's $ARGUMENTS. Workers do not read `providers.json`.

## 6. Data Contract Between Rounds

Round 2 needs from Round 1 (collected at barrier, not from disk):

| Data | Source |
|------|--------|
| Each slot's position text | `reasoning` + `raw` from worker result |
| UNAVAIL status per slot | `verdict: UNAVAIL` in worker result |
| Rate-limit hint | `unavail_message` in worker result |

The synthesizer bundles this into a single `Prior positions:` block that all Round 2 workers receive identically.

---

## 7. Early Consensus Short-Circuit

Under sequential execution: orchestrator stops calling slots when first N agree.
Under wave-barrier: all slots in a wave are called unconditionally. Consensus is checked only at the barrier.

**This is acceptable:** The goal is wall-clock reduction, not inference-cost reduction. All slots run simultaneously regardless — there is no "early" position in the wave to short-circuit from.

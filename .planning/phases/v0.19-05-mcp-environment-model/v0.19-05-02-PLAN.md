  <action>
    Create the directory formal/spec/mcp-calls/ and write environment.md as a formal specification document. Base the content on Pattern 1 from the RESEARCH.md but tailored to the actual call-quorum-slot.cjs behavior.

    The document MUST contain these sections:

    **Section 1: MCP as Environment Process**
    - Explain that MCP servers (codex-1, gemini-1, opencode-1, copilot-1, claude-1..6) are nondeterministic environment processes
    - Describe the interaction model: QGSD issues a call-quorum-slot.cjs subprocess invocation, waits for outcome, applies consensus logic
    - State that this spec defines the ALLOWED behavior of the MCP environment (not QGSD's internal behavior)

    **Section 2: Response Set (4 types)**
    - SUCCESS: Response received within timeout, vote value parsed
    - FAILURE: Non-zero exit, CLI error, HTTP error response (non-200), or vote parsing failure
    - TIMEOUT: No response within configured timeout (default: 120s from `CLAUDE_MCP_TIMEOUT_MS` or 120000ms hardcoded in call-quorum-slot.cjs)
    - REORDER: Multiple concurrent calls; response arrives for a previously-cancelled or completed round (handled by roundCounter guard in qgsd-prompt.js)

    **Section 3: Timing Model**
    - Timeout defaults: provider.quorum_timeout_ms (typically 120000ms = 2 min); overridable via --timeout flag
    - Retry limits: call-quorum-slot.cjs uses runSubprocessWithRotation() with provider.rotation.retry_limit (typically 2-3 retries per provider)
    - Backoff: exponential with jitter in runSubprocessWithRotation() (approximately 1s, 2s, 4s between retries)
    - Fairness assumption: Weak fairness — enabled MCP processes eventually receive an outcome (either SUCCESS/FAILURE/TIMEOUT). No permanent suppression.

    **Section 4: Safety Invariants Quorum Must Preserve**
    Enumerate these four invariants:
    1. Consensus under partial availability: If ≥ floor(N/2)+1 slots respond SUCCESS, quorum reaches consensus
    2. Escalation under degradation: If > floor(N/2) slots become UNAVAILABLE or TIMEOUT, quorum escalates to user (R6 fail-open) rather than deadlocking
    3. No deadlock: Retry + backoff + timeout combination ensures quorum always makes finite-time progress
    4. Order independence: Unanimity check in qgsd-stop.js does not depend on response arrival order

    **Section 5: Nondeterminism Scope**
    - For K slots, TLC explores combinations: each slot can respond SUCCESS, FAILURE, or TIMEOUT
    - REORDER handled by quorum's roundCounter guard (not modeled as separate TLC branch; documented as out-of-scope for TLC)
    - Conservative exploration: failures more likely in TLC model (PRISM calibration adds empirical rates)
    - Symmetry reduction: slots treated as symmetric to reduce TLC state space

    Document length: 80-120 lines. Use Markdown headings. Reference call-quorum-slot.cjs and CLAUDE.md R6 where appropriate.

    Replace the Wave 0 stub in formal/tla/QGSDMCPEnv.tla with a full TLA+ specification modeling MCP server call behavior. Also update MCMCPEnv.cfg with proper TLC configuration.

    **QGSDMCPEnv.tla implementation:**

    EXTENDS Naturals, FiniteSets, TLC

    CONSTANTS NumSlots, MaxRetries, QuorumThreshold

    Key design decisions (from RESEARCH.md open questions + pitfall guidance):
    - Use {1..NumSlots} as the SLOTS set (symmetric)
    - Outcomes: {"SUCCESS", "FAIL", "TIMEOUT"} — not "REORDER" (handled by roundCounter guard, documented in environment.md as out-of-scope)
    - slotStatus: function [1..NumSlots -> {"AVAILABLE", "UNAVAILABLE"}]
    - callState: function [1..NumSlots -> {"PENDING", "SUCCESS", "FAIL", "TIMEOUT"}]
    - quorumPhase: "COLLECTING" | "DELIBERATING" | "DECIDED"

    Actions:
    1. Init: all slots AVAILABLE, all calls PENDING, quorumPhase = "COLLECTING"
    2. MCPEnvironmentStep: Nondeterministically assign each pending call an outcome. If slotStatus[s] = "UNAVAILABLE", outcome is forced to "TIMEOUT". If AVAILABLE, choose nondeterministically from {"SUCCESS", "FAIL", "TIMEOUT"}.
    3. QuorumProcessOutcomes: Count SUCCESS responses. If Cardinality({s: callState[s]="SUCCESS"}) >= QuorumThreshold, set quorumPhase = "DECIDED". If all non-PENDING calls leave < QuorumThreshold successes possible, set quorumPhase = "DECIDED" (escalation path).
    4. Timeout action: Explicit timeout action that advances callState[s] from "PENDING" to "TIMEOUT" for any slot (makes liveness provable without fairness).

    Safety property (invariant):
    ```
    SafeConsensus == quorumPhase = "DECIDED" => (
      Cardinality({s \in 1..NumSlots : callState[s] = "SUCCESS"}) >= QuorumThreshold \/
      \* Escalation: not enough successes, so quorum must escalate
      Cardinality({s \in 1..NumSlots : callState[s] = "SUCCESS"}) < QuorumThreshold
    )
    ```
    (This is trivially true but captures the structure — key safety is that DECIDED is only reached when quorum criteria are evaluated, not arbitrarily.)

    Stronger safety invariant — NoSpuriousConsensus:
    ```
    NoSpuriousConsensus == quorumPhase = "DECIDED" =>
      \/ Cardinality({s \in 1..NumSlots : callState[s] = "SUCCESS"}) >= QuorumThreshold
      \/ \A s \in 1..NumSlots : callState[s] \in {"FAIL", "TIMEOUT"}
    ```

    Liveness property:
    ```
    EventualDecision == <>(quorumPhase = "DECIDED")
    ```

    Spec uses WF_vars(QuorumProcessOutcomes \/ TimeoutAction) for liveness.

    PITFALL MITIGATION (state explosion): Use NumSlots=2 as default in config. Document in MCMCPEnv.cfg why symmetry is not used (TLA+ symmetry sets require SYMMETRY declaration with SymmetrySet; instead limit NumSlots).

    **MCMCPEnv.cfg update:**
    ```
    SPECIFICATION Spec
    CONSTANTS
      NumSlots = 2
      MaxRetries = 2
      QuorumThreshold = 1
    INVARIANT NoSpuriousConsensus
    PROPERTY EventualDecision
    CHECK_DEADLOCK TRUE
    ```

    The spec MUST compile and TLC must be runnable (even if not installed in CI, the syntax must be valid TLA+). Model all actions that change vars. Every variable in `vars` must be updated by Init and not inadvertently left stale.

    Reference QGSDQuorum_xstate.tla and QGSDDeliberation.tla for TLA+ syntax conventions already used in this project.

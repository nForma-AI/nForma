# Phase v0.19-05: MCP Environment Model - Research

**Researched:** 2026-02-27
**Domain:** Formal modeling of MCP servers as nondeterministic environment processes
**Confidence:** HIGH

## Summary

Phase v0.19-05 extends QGSD's formal verification infrastructure to model MCP servers as nondeterministic environment processes, enabling verification of quorum's fault-tolerance behavior under arbitrary MCP failures. The problem: MCP servers (quorum slot workers like codex-1, gemini-1, opencode-1, copilot-1, and the ccr/provider instances) fail in ways not captured by the existing TLA+ quorum model (QGSDQuorum_xstate.tla) — timeouts, unavailability, out-of-order responses, transient errors. These failure modes can break quorum's ability to reach consensus. Solution: (1) author a formal environment specification (`formal/spec/mcp-calls/environment.md`) documenting MCP's nondeterministic response set and timing model; (2) create a TLA+ spec (`formal/tla/QGSDMCPEnv.tla`) that models quorum's interaction with MCP processes under these failures; (3) extend the trace schema to capture MCP-specific metadata (request_id, slot name, outcome, attempt count); (4) calibrate PRISM's MCP availability model from scoreboard failure statistics using the existing `readScoreboardRates()` pattern.

This phase depends on v0.19-02 (CALIB policy infrastructure exists) and produces four artifacts aligned with the v0.19 FV pipeline:
- Formal spec: human-readable environment model defining MCP failure semantics
- TLA+ spec: machine-checkable quorum properties under MCP failures
- Trace schema extension: MCP-interaction events carry request_id, peer, outcome, attempt metadata
- PRISM model: MCP unavailability calibrated from empirical scoreboard data

**Primary recommendation:** Implement as four parallel workstreams:
1. **MCPENV-01:** Write `formal/spec/mcp-calls/environment.md` defining response set (success/failure/timeout/reorder), timing bounds (retry limits from call-quorum-slot.cjs: max retries, backoff strategy), and safety invariants that quorum must preserve
2. **MCPENV-02:** Create `formal/tla/QGSDMCPEnv.tla` modeling MCP servers as environment processes with nondeterministic response choices within declared bounds; check that quorum consensus/escalation properties hold under worst-case MCP failures
3. **MCPENV-03:** Extend `formal/trace/trace.schema.json` with `request_id`, `peer` (slot name), `outcome` (success/fail/timeout), and `attempt` (retry count) fields for MCP-interaction events; update `validate-traces.cjs` to validate these fields when present
4. **MCPENV-04:** Create `formal/prism/mcp-availability.pm` model with empirical failure rates from scoreboard; add `readScoreboardRates()` call to extract per-slot UNAVAIL rates; wire into `run-prism.cjs` to emit availability property check to `check-results.ndjson`

### Pattern 1: MCP Environment Specification (environment.md)

**What:** A formal document defining MCP servers as nondeterministic environment processes.

**Structure:**
```markdown
# formal/spec/mcp-calls/environment.md

## MCP as Environment Process

MCP servers (quorum slot workers) are nondeterministic environment processes.
QGSD's quorum layer interacts with MCP by:
1. Sending a call-quorum-slot.cjs subprocess invocation (promise pending)
2. Waiting for response (success, failure, timeout, reorder)
3. Proceeding with consensus logic based on outcome

## Response Set

Each MCP call can result in one of:
- **SUCCESS**: Response received within timeout, vote value parsed successfully
- **FAILURE**: Error response received (non-200 HTTP, CLI error, authentication failure)
- **TIMEOUT**: No response within configured timeout (call-quorum-slot.cjs: 120s default, configurable via --timeout)
- **REORDER**: Multiple outstanding calls; responses may arrive out-of-order (handled by `roundCounter` in qgsd-prompt.js)

## Timing Model

- **Retry limits:** call-quorum-slot.cjs uses `runSubprocessWithRotation()` for HTTP provider rotation; max retries = provider.rotation.retry_limit (typically 2–3)
- **Backoff:** exponential backoff with jitter in `runSubprocessWithRotation()` (wait 1s, then 2s, then 4s between retries)
- **Timeout defaults:** provider.quorum_timeout_ms (typically 120000ms = 2 min); overridable per-call via --timeout flag
- **Fairness assumption:** MCP processes do not permanently suppress responses; enabled calls eventually receive outcomes (weak fairness on I/O actions)

## Safety Invariants Quorum Must Preserve

1. **Consensus under partial availability:** If ≥N/2 + 1 slots are AVAILABLE and each responds, quorum reaches consensus
2. **Escalation under degradation:** If >N/2 slots become UNAVAILABLE or repeatedly TIMEOUT, quorum must escalate to user rather than halt
3. **No deadlock:** Quorum's retry + backoff prevents deadlock; timeout ensures finite wait
4. **Order independence:** Consensus logic (unanimity check) does not depend on response order

## Nondeterminism Scope

For a quorum of K slots, TLC explores all combinations:
- Each slot i can respond with SUCCESS, FAILURE, TIMEOUT, or REORDER (4^K possibilities)
- Within REORDER, all K! orderings are explored
- Conservative priors: failures more likely than production (PRISM calibration refines this)
```

**When to use:** MCPENV-01; referenced by MCPENV-02 when writing TLA+ nondeterminism; informs test case design in MCPENV-03 and MCPENV-04.

### Pattern 2: TLA+ MCP Environment Model (QGSDMCPEnv.tla)

**What:** Formal spec modeling quorum's interaction with MCP environment under nondeterministic failures.

**Structure (pseudocode):**
```tla
---- MODULE QGSDMCPEnv ----
(*
 * formal/tla/QGSDMCPEnv.tla
 * MCPENV-02: Models MCP call behavior and quorum fault-tolerance
 *)

EXTENDS Naturals, FiniteSets, TLC

CONSTANTS NumSlots, MaxRetries

VARIABLES
  slotStatus,      \* [slot -> {AVAILABLE, UNAVAILABLE}] — nondeterministic per round
  callState,       \* [call_id -> {PENDING, SUCCESS, FAIL, TIMEOUT}]
  responses,       \* set of { slot, call_id, result }
  quorumPhase      \* quorum state: COLLECTING, DELIBERATING, DECIDED

\* ── Environment: MCP failure nondeterminism ──
MCPFailure(call_id, slot) ==
  \* Nondeterministically choose outcome for this call
  CHOOSE outcome \in {SUCCESS, FAIL, TIMEOUT} :
    slotStatus[slot] = AVAILABLE => outcome = SUCCESS \/ outcome = FAIL  \* if available, may still fail
    slotStatus[slot] = UNAVAILABLE => outcome = TIMEOUT \/ outcome = FAIL \* if unavailable, no success

\* ── Safety property: Consensus is reachable ──
SafeConsensus ==
  <>[\exists result \in {CONSENSUS, ESCALATE} : quorumPhase = DECIDED /\ decision = result]

\* ── Liveness property: Quorum does not deadlock ──
EventualDecision ==
  <>(quorumPhase = DECIDED)  \* with weak fairness on response handling

====
```

**Key aspects:**
- MCP environment is a separate module defining slot availability and response outcomes
- Quorum actions conditioned on MCP response arrivals
- TLC explores all failure combinations (cross-product of per-slot outcomes)
- Liveness checked under weak fairness assumption

**When to use:** MCPENV-02; check: `tlc formal/tla/QGSDMCPEnv.tla MCMCPEnv.cfg` in CI to verify quorum safety/liveness under all MCP failures.

### Pattern 4: Empirical MCP Availability Model (PRISM)

**What:** Markov chain modeling per-slot MCP availability calibrated from scoreboard UNAVAIL rates.

**Structure (formal/prism/mcp-availability.pm):**
```prism
// formal/prism/mcp-availability.pm
// MCPENV-04: Per-slot MCP availability Markov chain

dtmc

// Per-slot availability rates (empirical from scoreboard)
const double codex_1_avail    = 0.95;    // derived from scoreboard TP count
const double gemini_1_avail   = 0.92;
const double opencode_1_avail = 0.96;
const double copilot_1_avail  = 0.90;

module mcp_slots
  \* Boolean for each slot: true = available, false = unavailable
  codex_available    : bool init true;
  gemini_available   : bool init true;
  opencode_available : bool init true;
  copilot_available  : bool init true;

  \* At each step, each slot independently transitions based on empirical availability
  [] codex_available -> codex_1_avail     : (codex_available'=true)
                     + (1-codex_1_avail)  : (codex_available'=false);
  [] gemini_available -> gemini_1_avail   : (gemini_available'=true)
                      + (1-gemini_1_avail) : (gemini_available'=false);
  \* ... similar for other slots
endmodule

// Property: At least one slot is always available
label "min_quorum_available" = codex_available | gemini_available | opencode_available | copilot_available;

// Properties in mcp-availability.props:
// P1: P=? [ F "min_quorum_available"=false ]   (probability of total outage)
// P2: P=? [ G "min_quorum_available"=true ]    (probability never loses quorum)
```

**When to use:** MCPENV-04; run via `node bin/run-prism.cjs --model mcp-availability` to check minimum quorum survival probability.

### Anti-Patterns to Avoid

- **Modeling MCP as always-available:** Environment nondeterminism is essential; skipping failure modes misses real quorum vulnerabilities
- **Hardcoding failure rates:** Always calibrate from scoreboard; empirical data >  guesses
- **Trace schema without validation:** New MCP fields must be validated; missing request_id or peer breaks correlation analysis
- **Ignoring retry/timeout bounds:** MCP timing model is formal part of environment spec; omitting retry limits breaks model fidelity
- **Confusing TLA+ and PRISM:** TLA+ checks safety under ALL failures; PRISM quantifies probability of specific failures. Both needed for complete assurance

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Nondeterministic failure modeling | Custom failure simulator in Node.js | TLA+ with TLC | TLC explores all combinations exhaustively; Node.js simulator requires manual case enumeration and is incomplete |
| Availability rate calculation | Custom slot-by-slot aggregator | `readScoreboardRates()` helper from run-prism.cjs baseline | Existing function handles TP/TN/UNAVAIL counting; reusing prevents double-counting and maintains consistency |
| MCP timing model | Custom backoff/retry logic | call-quorum-slot.cjs + runSubprocessWithRotation pattern | Existing code is production-proven; formal spec should document it, not reimplement it |
| Trace field validation | Ad-hoc regex checks | JSON Schema + ajv | Schema-driven validation is standard; prevents schema/validator skew (cf. DRIFT-01 from v0.19-04) |

**Key insight:** Formal modeling is specification and verification, not engineering — the focus is on documenting existing MCP behavior rigorously and proving quorum properties hold under that behavior, not reimplementing MCP or quorum logic.

## Common Pitfalls

### Pitfall 1: Environment Nondeterminism Explosion

**What goes wrong:** TLC model-checking QGSDMCPEnv.tla with K=4 slots exploring 4^4=256 failure combinations + K!=24 orderings explodes state space. Model checking can timeout or run out of memory.

**Why it happens:** Naive encoding of all outcome combinations without abstraction. TLC's state explosion problem is well-known for high-dimensionality models.

**How to avoid:**
- Use symmetry reduction: mark slots as symmetric in TLC config (`SYMMETRY Permutations(Slots)`)
- Restrict exploration: instead of all 256 combinations, check only representative failure patterns (all AVAILABLE, all UNAVAILABLE, N/2 AVAILABLE, best-case 1 UNAVAILABLE)
- Use deadlock detection instead of full model check: check only that quorum doesn't deadlock under adversarial MCP, not all orderings

**Warning signs:**
- TLC run time >60s for a simple property check
- State graph grows without bound as constant increases
- "Depth exceeded" or "heap space exhausted" errors

### Pitfall 2: Scoreboard Calibration Misalignment

**What goes wrong:** MCP availability rates are extracted from scoreboard TP/TN/UNAVAIL counts, but the mapping is inconsistent with how quorum-scoreboard.json actually records votes. Result: PRISM model uses wrong rates, availability property check produces invalid conclusions.

**Why it happens:** run-prism.cjs already reads scoreboard for calibration (CALIB phase); MCPENV-04 reuses that logic. If the logic changed between v0.19-02 and v0.19-05, or if slot naming is inconsistent (e.g., `claude-1` vs. `deepseek` in votes array), rates will be computed incorrectly.

**How to avoid:**
- Reuse `readScoreboardRates()` from run-prism.cjs as-is; don't rewrite it
- Verify slot names match: grep quorum-scoreboard.json for actual slot keys appearing in votes[]
- Add unit test: mock scoreboard with known TP/UNAVAIL counts, verify rates match expected values

**Warning signs:**
- PRISM availability rates sum to >1.0 per slot or <0.0
- run-prism.cjs emits "No scoreboard found" when quorum-scoreboard.json clearly exists
- Test that injects fake scoreboard data gets wrong calibration

### Pitfall 3: Trace Schema Drift (Revisited from v0.19-04)

**What goes wrong:** Trace schema is extended with request_id, peer, outcome, attempt, but validate-traces.cjs doesn't validate them. Result: traces lacking these fields pass validation, breaking MCP fault-tolerance analysis.

**Why it happens:** Schema extension happens in MCPENV-03, but validation code update (MCPENV-03 requirement) is deferred or skipped. Schema drift guard (from v0.19-04) only checks schema.json against validator/emitter co-modification, not that validators actually enforce new fields.

**How to avoid:**
- In same commit: update schema.json, update validate-traces.cjs to check new fields, add test cases for valid and invalid traces
- Add test: emit trace with missing request_id, verify validate-traces.cjs rejects it
- Run check-trace-schema-drift.cjs to verify atomic co-update

**Warning signs:**
- Unit tests in validate-traces.test.cjs don't include MCP field validation cases
- Trace conformance log includes mcp_call events lacking request_id
- JSON Schema validation passes but semantic checks fail

### Pitfall 4: TLA+ Fairness Assumptions Don't Match Execution

**What goes wrong:** TLA+ spec assumes weak fairness on MCP response handling (responses eventually arrive), but in production, slow/hung MCP servers can cause quorum to stall indefinitely. Formal verification says "must decide" but implementation hangs.

**Why it happens:** TLA+ fairness captures "if enabled, act"; MCP reality is "if enabled but slow, timeout and move on." Quorum's actual fault-tolerance relies on timeouts, not fairness; TLA+ spec must model both.

**How to avoid:**
- Include timeout as explicit action in TLA+ spec; don't rely solely on fairness
- Model two rounds of quorum: (1) with successful responses (checked under fairness), (2) with timeouts forcing move-on (checked without fairness)
- Verify liveness property holds under BOTH weak fairness (happy path) and timeout forcibly advancing state (degraded path)
- Document fairness assumption in formal/spec/mcp-calls/environment.md

**Warning signs:**
- TLA+ check says "DECIDE is reachable" but quorum gets stuck during execution (indicates fairness assumption not met)
- Liveness property checked only under WF_vars, no timeout transitions
- No test case for "all slots timeout simultaneously"

## Code Examples

### Example 1: Reading Scoreboard for MCP Rates (MCPENV-04)

```javascript
// Source: existing pattern in bin/run-prism.cjs (readScoreboardMeta)
// Reuse this function to extract per-slot availability rates

function readMCPAvailabilityRates(scoreboardPath) {
  try {
    const raw = fs.readFileSync(scoreboardPath, 'utf8');
    const sb = JSON.parse(raw);
    const rounds = Array.isArray(sb.rounds) ? sb.rounds : [];

    // Aggregate per-slot
    const slotStats = {};
    for (const round of rounds) {
      const votes = round.votes || {};
      for (const [slot, code] of Object.entries(votes)) {
        if (!slotStats[slot]) slotStats[slot] = { total: 0, unavail: 0 };
        slotStats[slot].total++;
        if (code === 'UNAVAIL') slotStats[slot].unavail++;
      }
    }

    // Compute availability rates
    const rates = {};
    for (const [slot, stats] of Object.entries(slotStats)) {
      rates[slot] = 1.0 - (stats.unavail / stats.total);
    }
    return rates;
  } catch (_) {
    return null;  // no scoreboard, fall back to priors
  }
}
```

**When to use:** MCPENV-04, in run-prism.cjs when generating mcp-availability.pm property checks. Returns `{ 'codex-1': 0.95, 'gemini-1': 0.92, ... }` for PRISM constant injection.

### Example 2: Validating MCP Trace Fields (MCPENV-03)

```javascript
// Source: existing validate-traces.cjs pattern (mapToXStateEvent)
// Extended to validate MCP metadata fields

function validateMCPMetadata(event) {
  // Check if this is an MCP call event
  if (event.action !== 'mcp_call') return true;  // not an MCP event, skip

  // MCP events MUST have: request_id, peer, outcome, attempt
  const errors = [];
  if (!event.request_id || typeof event.request_id !== 'string') {
    errors.push('mcp_call missing or invalid request_id');
  }
  if (!event.peer || typeof event.peer !== 'string') {
    errors.push('mcp_call missing or invalid peer (slot name)');
  }
  if (!['success', 'fail', 'timeout', 'reorder'].includes(event.outcome)) {
    errors.push(`mcp_call invalid outcome: ${event.outcome}`);
  }
  if (typeof event.attempt !== 'number' || event.attempt < 1) {
    errors.push(`mcp_call invalid attempt: ${event.attempt}`);
  }

  return errors.length === 0 ? true : errors;
}

// In main validation loop:
for (const line of lines) {
  let event = JSON.parse(line);
  const mcpErrors = validateMCPMetadata(event);
  if (mcpErrors !== true) {
    divergences.push({ event, reason: 'mcp_field_validation', errors: mcpErrors });
  }
}
```

**When to use:** MCPENV-03, in validate-traces.cjs check function after JSON parsing. Returns true if valid, array of error strings if invalid.

### Example 3: TLA+ MCP Environment Module Structure (MCPENV-02)

```tla
(*
 * Simplified MCP environment module for QGSDMCPEnv.tla
 *)

VARIABLE
  slotAvailable,   \* [slot -> BOOLEAN] — current availability state
  callResponses    \* set of [slot -> outcome] for current round

\* Nondeterministic environment step: each slot independently chooses outcome
MCPEnvironmentStep ==
  /\ slotAvailable' \in [SLOTS -> BOOLEAN]  \* each slot independently nondeterministic
  /\ callResponses' = { [slot |-> SlotOutcome(slotAvailable'[slot])] : slot \in SLOTS }

\* MCP outcome determined by slot availability (conservative model)
SlotOutcome(avail) ==
  IF avail THEN CHOOSE o \in {SUCCESS, FAILURE} : TRUE  \* available slots may still fail
  ELSE TIMEOUT                                            \* unavailable slots always timeout

\* Quorum must handle all possible outcome sets
QuorumProcessOutcomes ==
  LET availableResponses == { [s -> callResponses[s]] : s \in {ss \in SLOTS : callResponses[ss] = SUCCESS} }
      quorumSize == Cardinality(availableResponses)
  IN
    IF quorumSize >= QUORUM_THRESHOLD
    THEN QuorumConsensus
    ELSE QuorumEscalate

SafetyProperty ==
  \* Invariant: if consensus reached, all available slots had unanimous vote
  [][ QuorumPhase = DECIDED => UnanimityHeld ]
```

**When to use:** MCPENV-02, as the environment module in QGSDMCPEnv.tla. Replaces hand-coded failure cases with nondeterministic spec.

## Open Questions

1. **What is the right upper bound for TLC state explosion?**
   - What we know: K=4 slots, 4 outcomes each, K! orderings → large state space; symmetry reduction helps but degree unknown
   - What's unclear: Can TLC complete in <60s on representative laptop? Do we need stronger abstraction (abstract to just {SUCCESS, FAILURE} instead of 4 outcomes)?
   - Recommendation: Run experimental model checks during MCPENV-02 implementation with K=2, 3, 4 to find practical limits; document in MCMCPEnv.cfg comments

2. **Should MCPENV-04 model per-slot availability or aggregate quorum availability?**
   - What we know: run-prism.cjs currently models quorum as single Markov chain (s=COLLECTING_VOTES, DECIDED, DELIBERATING); CALIB-02 pattern exists
   - What's unclear: Does MCPENV-04 add separate `mcp-availability.pm` (per-slot model) or extend quorum.pm with MCP failure transitions?
   - Recommendation: Create separate `mcp-availability.pm` file (keeps separation of concerns) and wire both quorum.pm and mcp-availability.pm into run-prism.cjs; check both properties in CI

3. **How should trace request_id be generated and correlated?**
   - What we know: Each MCP call needs unique request_id to correlate with retry attempts; call-quorum-slot.cjs doesn't currently emit IDs
   - What's unclear: UUID vs. round_number+slot hash vs. incrementing counter? Where is ID generated (call-quorum-slot.cjs or qgsd-prompt.js)?
   - Recommendation: Use `round_number:slot_name:attempt` pattern (human-readable, deterministic); inject from qgsd-prompt.js into conformance-events.jsonl at trace time

4. **Is weak fairness the right assumption for MCP environment?**
   - What we know: WF_vars on quorum actions captures "enabled → eventually acts"; MCP reality includes timeouts forcing state advance
   - What's unclear: Should fairness be part of TLA+ spec or should timeout be explicit action making fairness unnecessary?
   - Recommendation: Model timeout as explicit action (MCPENV-02 implementation detail); verify liveness under both WF (happy path) and without WF (timeout forces progress)

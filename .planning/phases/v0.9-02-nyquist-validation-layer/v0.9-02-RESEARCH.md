## Architecture Patterns

### What "Nyquist Validation" means in this codebase

The name is a metaphor from signal processing (Nyquist sampling theorem): to faithfully reconstruct a signal, you must sample at at least 2x the highest frequency component. Applied to planning: to reliably detect regressions during a development phase, you need to test at least as frequently as you introduce changes.

In practice, "Nyquist Validation" in GSD/QGSD means: before creating plan artifacts, the researcher also produces a `VALIDATION.md` file that maps every planned task to a test command, specifies Wave 0 (pre-execution test scaffolding), and defines the sampling rate (how often to run tests). The plan-checker uses this to verify no implementation run of 3+ consecutive tasks lacks automated feedback.

### How the "Validation Architecture" trigger works

The plan-phase.md step 5.5 checks whether RESEARCH.md contains a `## Validation Architecture` section. If the researcher includes it, step 5.5 uses the VALIDATION.md template to generate a per-phase validation file. This means:

1. The QGSD researcher agent does NOT automatically produce this section ŌĆö it must be instructed to.
2. The researcher's prompt in step 5 (plan-phase.md) should mention that if Nyquist is enabled, the researcher should include a `## Validation Architecture` section in RESEARCH.md.
3. Currently, QGSD's plan-phase.md step 5 does NOT pass `nyquist_validation_enabled` to the researcher. This is a minor gap ŌĆö the researcher cannot know Nyquist is enabled unless the orchestrator tells it.

**Recommendation:** In QGSD's plan-phase.md step 5, add a conditional note to the research prompt: if `nyquist_validation_enabled` is true, instruct the researcher to include `## Validation Architecture` section covering test framework, Wave 0 requirements, and per-task test map.

## Common Pitfalls

### Pitfall 1: Path substitution mismatch
**What goes wrong:** Copying step 5.5 from GSD verbatim and forgetting to change `/Users/jonathanborduas/.claude/get-shit-done/` to `/Users/jonathanborduas/.claude/qgsd/` in the template path.
**How to avoid:** Search for `get-shit-done` in every copied text block before writing.

### Pitfall 2: Default value inversion
**What goes wrong:** Defaulting `nyquist_validation` to `false` (matching GSD) instead of `true` (required by NYQ-04).
**Warning signs:** `init plan-phase` JSON shows `nyquist_validation_enabled: false` when no config override is set.

### Pitfall 3: Missing researcher instruction
**What goes wrong:** Step 5.5 greps for `## Validation Architecture` in RESEARCH.md but the researcher never produces that section because nobody told it to. Result: Nyquist is enabled but VALIDATION.md is never generated; NYQ-05 halt fires every time.
**How to avoid:** Conditionally add researcher instruction in plan-phase.md step 5: if `nyquist_validation_enabled`, append "Include a `## Validation Architecture` section in your RESEARCH.md" to the research prompt.

### Pitfall 4: NYQ-05 is not in upstream GSD
**What goes wrong:** Assuming that porting the upstream GSD step 5.5 verbatim covers NYQ-05. It does not ŌĆö the upstream warning branch continues silently; QGSD requires a halt.
**How to avoid:** After copying step 5.5, add the halt condition explicitly.

### Pitfall 5: gsd-tools.cjs monolith vs GSD lib split
**What goes wrong:** Looking for `config.nyquist_validation` in `gsd-tools.cjs` and not finding it (it's not there yet), then looking in the wrong function.
**How to avoid:** `loadConfig()` is the single source of config defaults in QGSD's monolith. Both the defaults object and the return map need the new key.

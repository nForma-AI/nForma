# Phase v0.39-01: Foundation & Infrastructure - Research

**Researched:** 2026-03-18
**Domain:** Formal model verification mode tagging, trace parsing infrastructure, configurable iteration control
**Confidence:** HIGH (architecture + stack), MEDIUM-HIGH (pitfalls + validation patterns)

## Summary

Phase v0.39-01 establishes the foundational infrastructure for dual-cycle formal reasoning (Cycle 1: diagnostic feedback, Cycle 2: solution simulation). The phase makes three core architectural decisions:

1. **Verification Mode Tagging (FND-01):** All model checker invocations are tagged as `'diagnostic'` (Cycle 1, inverted semantics) or `'validation'` (Cycle 2, normal semantics) to prevent contradiction between inverted and normal verification during the same bug-fixing workflow.

2. **Trace Comparison Infrastructure (FND-02):** TLC counterexample traces are parsed from ITF JSON format into structured state-by-state objects, enabling field-level divergence detection and diff generation that will guide both diagnostic feedback (Cycle 1) and consequence model validation (Cycle 2).

3. **Configurable Iteration Limits (FND-03):** A `--max-iterations` CLI flag (default 3) is introduced, persisted to `.planning/config.json`, to control both Phase 3 refinement iteration (Cycle 1 diagnostic loop) and Phase 4.5 solution simulation attempts (Cycle 2 convergence loop).

This phase is **purely infrastructure and semantic setup** — no complex logic, no new model checkers, no mutations yet. The work is straightforward plumbing that unblocks both downstream cycles. Existing modules (refinement-loop.cjs, run-tlc.cjs, run-alloy.cjs) require **contract clarifications** (verification mode tagging at invocation) but no functional changes.

**Primary recommendation:** Implement verification mode tagging as a lightweight envelope (metadata tag + pass-through to all checker invocations), establish ITF trace parsing via native Node.js JSON, and add config.json iteration limit persistence. Avoid over-engineering; keep changes minimal and data structures simple.

### Pattern 1: Verification Mode Tagging

**What:** Every model checker invocation carries a metadata tag indicating verification semantics (diagnostic vs. validation).

**When to use:**
- Cycle 1 (Phase 2-3 refinement): `verification_mode: 'diagnostic'` — model MUST FAIL on bug trace (inverted semantics)
- Cycle 2 (Phase 4.5 convergence gates): `verification_mode: 'validation'` — model MUST PASS original invariants (normal semantics)

**Example:**

```javascript
// Source: Proposed run-tlc.cjs contract enhancement

// Cycle 1: Diagnostic run (inverted semantics)
const tlcResult = runTLC({
  modelPath: 'model.tla',
  config: 'MC.cfg',
  bugContext: 'timeout occurs after 5s',
  verification_mode: 'diagnostic',  // NEW: Tag indicates inverted semantics
  dumpTrace: true                    // Request JSON trace output
});

// Cycle 2: Validation run (normal semantics)
const consequenceResult = runTLC({
  modelPath: 'consequence-model.tla',
  config: 'MC.cfg',
  verification_mode: 'validation',   // NEW: Tag indicates normal gate semantics
  dumpTrace: false                   // No trace needed for consequence model
});

// Contract for interpret gate results:
// If (verification_mode === 'diagnostic' && hasViolation) → SUCCESS (model captures bug)
// If (verification_mode === 'diagnostic' && !hasViolation) → FAILURE (model incomplete)
// If (verification_mode === 'validation' && hasViolation) → FAILURE (invariant broken)
// If (verification_mode === 'validation' && !hasViolation) → SUCCESS (consequence model passes)
```

**Integration point:** `run-tlc.cjs` and `run-alloy.cjs` accept optional `verification_mode` parameter; pass through to result envelope. No logic changes to checker invocation itself — purely metadata for downstream gate interpretation.

**Safety rules:**
- Mode tag is **required** in Phase 1 (Foundation) contract for both cycles to work safely
- Gate logic (Phase 2 and Phase 4.5) MUST check mode before interpreting pass/fail
- Default mode (if unspecified) should be `'validation'` for backward compatibility

### Pattern 2: ITF Trace Parsing (Native JSON)

**What:** TLC outputs traces in ITF JSON format; parse directly to structured state sequences.

**When to use:** After TLC invocation with `-dumpTrace json` flag, when diagnostic diff or convergence validation needs state-by-state comparison.

**Example:**

```javascript
// Source: Proposed parse-tlc-counterexample.cjs

const fs = require('fs');

/**
 * Parse ITF JSON trace from TLC -dumpTrace json output.
 * @param {string} traceJsonPath - Path to .itf.json file written by TLC
 * @returns {{ states: Array, loopPoint: number?, violated_invariant: string? }}
 */
function parseITFTrace(traceJsonPath) {
  const content = fs.readFileSync(traceJsonPath, 'utf8');
  const trace = JSON.parse(content);

  // ITF structure: { states: [{var1: val1, ...}, ...], loop?: 5, violated?: "InvName" }
  return {
    states: trace.states || [],
    loopPoint: trace.loop,      // Optional: lasso cycle index
    violated_invariant: trace.violated_invariant || trace.violated,
    trace_length: (trace.states || []).length
  };
}

/**
 * Extract divergence between model's predicted trace and expected behavior.
 * @param {Array} modelStates - States from parseITFTrace
 * @param {Object} expectedBehavior - Expected behavior from bug context
 * @returns {Array<string>} Diffs like ["timeout: model=5000ms, bug=10000ms"]
 */
function extractTraceDivergence(modelStates, expectedBehavior) {
  const diffs = [];

  // Compare final state
  const finalState = modelStates[modelStates.length - 1] || {};
  for (const [key, expectedValue] of Object.entries(expectedBehavior)) {
    if (finalState[key] !== expectedValue) {
      diffs.push(`${key}: model=${finalState[key]}, expected=${expectedValue}`);
    }
  }

  return diffs;
}
```

**Integration point:** Called by `diagnostic-diff-generator.cjs` after TLC execution. Pure JSON parsing; no external dependencies beyond json-diff-ts for state comparison.

**Special handling:**
- ITF supports `{ "#bigint": "12345" }` for large integers — Node.js `JSON.parse()` handles natively
- ITF `loop` field indicates lasso trace (cycle point); used for oscillation detection
- `violated_invariant` field names differ between TLC and Alloy — normalize in parse step

### Pattern 3: Consequence Model Session Isolation

**What:** Store consequence models and diagnostic artifacts in session-scoped temporary directories, not in main registry.

**When to use:** Phase 4.5 (Cycle 2) will generate consequence models; Phase 1 establishes the storage pattern.

**Example:**

```javascript
// Source: Proposed config-update.cjs

const fs = require('fs');
const path = require('path');
const crypto = require('crypto');

/**
 * Initialize session-scoped artifact directory for consequence models.
 * @returns {string} Path to session directory (e.g., .planning/formal/.tmp/session-abc123/)
 */
function createSessionDirectory() {
  const sessionId = crypto.randomBytes(8).toString('hex');
  const sessionDir = path.join(
    __dirname, '..', '.planning', 'formal', '.tmp',
    `session-${sessionId}`
  );
  fs.mkdirSync(sessionDir, { recursive: true });
  return sessionDir;
}

/**
 * Update config.json with iteration limit and session metadata.
 * @param {number} maxIterations - Default 3; persisted for Cycles 1 & 2
 * @param {string} sessionId - Session identifier for artifact isolation
 */
function updateConfig(maxIterations = 3, sessionId = null) {
  const configPath = path.join(__dirname, '..', '.planning', 'config.json');
  const config = JSON.parse(fs.readFileSync(configPath, 'utf8'));

  config.max_iterations = maxIterations;
  if (sessionId) {
    config.current_session_id = sessionId;  // NEW
  }

  fs.writeFileSync(configPath, JSON.stringify(config, null, 2) + '\n', 'utf8');
}
```

**Rationale:** Session isolation prevents consequence model registry pollution; temporary directory can be cleaned up after 7 days or on explicit request. Main `.planning/formal/` registry stays clean for bug-to-model lookup.

### Anti-Patterns to Avoid

- **Modifying existing module logic:** Don't change refinement-loop.cjs or run-tlc.cjs internals; only add optional parameters and metadata envelopes
- **Inline mode detection:** Don't check `verification_mode` inside checker runners; move interpretation to gate logic (Phase 2/4.5)
- **Custom trace format:** Don't invent a new trace format; stick to standard ITF JSON
- **Unbounded iteration:** Always enforce a limit; default 3 prevents infinite loops

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| **State sequence comparison** | Custom tree-walk differ | json-diff-ts | Handles nested objects, arrays, type mismatches; proven in production; 100+ tests in library |
| **JSON trace parsing** | Regex-based ITF extractor | `JSON.parse()` directly | ITF is pure JSON; Node.js built-in handles all cases (bigints, special values); regex is fragile |
| **TLC output parsing** | Character-by-character state machine | Subprocess output capture + regex for summary lines | TLC and Alloy already output structured text; simple regex patterns work (proven in existing refinement-loop.cjs) |
| **Config file updates** | Direct file overwrite | fs.readFileSync → update → writeFileSync | Atomic updates prevent corruption; simple pattern reused throughout nForma |

**Key insight:** The domain is about orchestrating existing formal tools, not building new parsing libraries. Lean on Node.js built-ins and proven small libraries; complexity comes from gate logic (Phase 2/4.5), not infrastructure.

## Common Pitfalls

### Pitfall 1: Verification Mode Contradiction

**What goes wrong:** Refinement loop (Phase 3) uses inverted semantics; consequence model gates (Phase 4.5) use normal semantics. If mode isn't explicitly tagged, gate logic misinterprets results (model passes when it should fail, or vice versa).

**Why it happens:** Verification semantics are implicit; gate conditions look the same but mean opposite things depending on context.

**How to avoid:**
- FND-01 requirement: Tag ALL model checker invocations with `verification_mode` parameter
- Gate logic checks mode FIRST before interpreting violations
- Document semantics inversion clearly (model violation = bug captured, for diagnostic; model violation = invariant broken, for validation)

**Warning signs:**
- Gate logic applies same pass/fail condition to both Cycle 1 and Cycle 2
- Tests pass in isolation but fail when both cycles run together
- Convergence gate unexpectedly blocks valid fixes

### Pitfall 2: Trace Alignment Failures

**What goes wrong:** Bug description (coarse-grained, natural language) doesn't map cleanly to TLA+ trace (precise, state-level). Diagnostic diff generates spurious mismatches ("model assumes X but bug doesn't even mention X") that mislead refinement.

**Why it happens:** Bug context is user-provided, informal; TLA+ state space is formal and complete. Mapping is not mechanical.

**How to avoid:**
- FND-02: Keep trace parsing minimal (pure JSON extraction); don't try to interpret semantic equivalence
- Leave alignment/interpretation to downstream diagnostic diff generator (Phase 2 responsibility)
- Provide user feedback: "Could not align bug trace to model states" (fail-open)

**Warning signs:**
- Diagnostic output includes variables not mentioned in bug context
- User says "diagnostic says model wrong, but model looks correct"
- Phase 3 refinement ignores diagnostic guidance

### Pitfall 3: Iteration Limit Not Persisted

**What goes wrong:** User sets `--max-iterations=5` for one bug; fixes bug; tries a different bug with implicit default (3) — unexpected behavior surprise ("why did it stop after 3 tries?").

**Why it happens:** Iteration limit stored only in CLI args, not persisted to config.

**How to avoid:** FND-03 requirement: `--max-iterations` updates `.planning/config.json` (durable storage)
- Check config.json before checking CLI args
- CLI flag overrides config.json, but update config if changed
- Document persistence in help text

**Warning signs:**
- Different iteration limits for different bugs without user action
- No trace of where iteration limit came from

### Pitfall 4: Consequence Model Registry Pollution

**What goes wrong:** Consequence models accumulate in `.planning/formal/` without cleanup. Registry grows to 10K+ models; bug-to-model lookup becomes slow; false positives in semantic search.

**Why it happens:** Consequence models are temporary (created for testing), not permanent (results of analysis). Without explicit lifecycle, they stay forever.

**How to avoid:** FND-01 foundation work: Create `.planning/formal/.tmp/` directory; store consequence models there with session ID; cleanup after Cycle 2 completes or after 7 days.
- Don't add consequence models to main registry
- Document artifact lifecycle in schema

**Warning signs:**
- `.planning/formal/` directory has 100x more files than expected
- Model discovery scans take >5s
- Unrelated models appear in "related models" suggestions

### Pitfall 5: ITF Trace Format Variation

**What goes wrong:** Different formalism checkers (TLC, Alloy, PRISM) output different trace formats. Code written for TLC traces fails on Alloy traces.

**Why it happens:** While TLC uses standard ITF JSON, Alloy outputs XML; PRISM outputs text. Parser must handle variations.

**How to avoid:** FND-02: Separate parser modules per formalism
- `parse-tlc-counterexample.cjs` — ITF JSON (TLC standard)
- `parse-alloy-counterexample.cjs` — Alloy XML converter to ITF-like JSON
- `parse-prism-trace.cjs` — PRISM output to ITF-like JSON
- Normalize all to same `{ states: [...], loopPoint, violated_invariant }` interface

**Warning signs:**
- Traces from Alloy or PRISM cause "undefined" errors
- Diagnostic diffs are empty for non-TLC models

## Open Questions

1. **Should verification_mode default to 'validation' or fail explicitly?**
   - What we know: Backward compatibility suggests default to 'validation'
   - What's unclear: Will this hide bugs where old code (Phase 5b regression checks) gets wrong semantics?
   - Recommendation: Default to 'validation'; explicitly tag 'diagnostic' only in Phase 2-3; Phase 2 research will validate this assumption

2. **How to align coarse-grained bug traces to precise formal traces?**
   - What we know: Bug context is informal; TLA+ state space is formal
   - What's unclear: Can diagnostic diff be meaningful without explicit trace alignment?
   - Recommendation: Phase 2 research; Phase 1 just parses traces; Phase 2 decides alignment strategy

3. **Should session cleanup be automatic (7-day TTL) or manual?**
   - What we know: Consequence models are temporary; registry pollution is a real risk
   - What's unclear: Is 7-day TTL too aggressive for long-running bug investigations?
   - Recommendation: Implement automatic cleanup with flag to disable (--retain-consequence-models); Phase 2 can adjust TTL based on usage patterns

4. **How to detect when --max-iterations should be overridden?**
   - What we know: Default 3 iterations is reasonable for typical bugs
   - What's unclear: When should user know to increase limit? What signals?
   - Recommendation: When iteration N fails, suggest "try --max-iterations=5" if remaining budget; Phase 2 can implement this heuristic

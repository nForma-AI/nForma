## Implementation Details

### Task 1: ITF Trace Parser (b08e96a0)

**File:** `bin/parse-tlc-counterexample.cjs`

Exports three functions:

1. **`parseITFTrace(traceJsonPath)`** — Reads ITF JSON file from TLC `-dumpTrace json` output
   - Returns: `{ states: Array<Object>, loopPoint: number | null, violated_invariant: string | null, trace_length: number }`
   - Handles edge cases: empty files (throws), malformed JSON (throws), missing states array (returns empty)
   - Example: 3-state trace with final state violation parsed correctly in integration test

2. **`normalizeITFValue(value)`** — Normalizes ITF special value encodings
   - `{ "#bigint": "123" }` → `"123"` (string, JSON-serializable)
   - `{ "#set": [3,1,2] }` → `[1,2,3]` (sorted array)
   - `{ "#map": [["k1","v1"]] }` → `{ k1: "v1" }` (plain object)
   - Recursively handles nested structures
   - Passes primitives through unchanged

3. **`extractStateFields(states, fieldNames?)`** — Filters states to named fields
   - If `fieldNames` is null/undefined, returns all fields
   - If `fieldNames` provided, extracts only those fields per state
   - Gracefully handles missing fields (omitted from output)

- parseITFTrace: 6 tests (minimal 2-state trace, loopPoint extraction, violated_invariant extraction, missing states, empty file, malformed JSON)
- normalizeITFValue: 5 tests (bigint, set, map encoding, primitive pass-through, nested recursion)

### Task 2: Diagnostic Diff Generator (4c0012fa)

**File:** `bin/diagnostic-diff-generator.cjs`

Imports `parseITFTrace` and `extractStateFields` from `parse-tlc-counterexample.cjs` (verified via grep).

Exports two functions:

1. **`generateStateDiff(statesA, statesB, options?)`** — Compares two state sequences
   - Uses json-diff-ts `diff()` for per-state comparison
   - Returns:
     ```javascript
     {
       aligned_length: number,
       length_mismatch: boolean,
       per_state_diffs: Array<{ index, changes: Array<{ key, oldValue, newValue }> }>,
       summary: { total_changes, changed_fields: string[], first_divergence_index: number | null }
     }
     ```
   - Supports optional `fieldFilter` option (delegates to extractStateFields)
   - Detects first divergence index (used by downstream diagnostic logic)

2. **`formatDiffAsMarkdown(diffResult)`** — Formats diff result as human-readable markdown
   - Empty diff: "Traces are identical" message
   - Per-state changes: markdown table with Field | Trace A | Trace B columns
   - Length mismatch warning: "⚠️ Trace length mismatch" when trace lengths differ
   - Example output from integration test:
     ```markdown
     ## State Divergence Report
     **Traces diverge at state 2** (1 total changes across 1 fields)
     ### State 2
     | Field | Trace A | Trace B |
     |-------|---------|----------|
     | count | `2` | `3` |
     ```

**Tests (9 total):**
- generateStateDiff: 5 tests (identical states, single field change, multiple field changes, length mismatch, fieldFilter option)
- formatDiffAsMarkdown: 3 tests (markdown table output, identical message, length mismatch warning)
- Integration: 1 test (first divergence index detection)

## Contract and Interface Clarity

### For Phase v0.39-02 (Diagnostic Feedback)

- **Input:** TLC counterexample trace (ITF JSON)
- **Processing:**
  1. `parseITFTrace(path)` → structured state sequence
  2. Extract relevant fields via `extractStateFields(states, ['field1', 'field2'])`
  3. Compare with expected behavior using `generateStateDiff(modelStates, expectedStates)`
  4. Format diff using `formatDiffAsMarkdown(diffResult)` for human consumption
- **Output:** Markdown table showing state-by-state divergence with field names and values

### For Phase v0.39-03 (Consequence Model Validation)

- **Input:** Two trace sequences (model output vs. expected behavior)
- **Processing:** Same as above, with `verification_mode: 'validation'` context
- **Output:** Diff report used to validate model captures consequence of fix

## Known Limitations and Future Work

1. **ITF special value handling:** Currently normalizes to JSON-serializable forms (bigint → string). Actual BigInt arithmetic requires explicit conversion in downstream use.

2. **Markdown formatting:** Wraps all values in backticks for readability, but may not be ideal for large nested objects. Phase 2 can add summary view option if needed.

3. **Alloy/PRISM support:** Currently only tested with TLC ITF JSON. Alloy XML and PRISM output will require separate `parse-alloy-counterexample.cjs` and `parse-prism-trace.cjs` modules (noted in research as anti-pattern to avoid).

4. **Performance:** No optimization for very large traces (1000+ states). Current implementation is O(n*m) for state comparison; can be improved if needed in Phase 2+.

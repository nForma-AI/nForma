## Summary

The core problem is that the current batch-based fix-tests.md is architecturally blind to cross-batch test pollution. Because unfixed tests are always "expected to fail," a fix in batch 1 that changes HOW batch 2's test fails goes undetected — the post-batch full run still reports the same test as failing, so the batch 2 agent diagnoses the wrong root cause.

The solution is to replace batching entirely with a dependency-graph-aware pipeline: (1) run the full suite once with a pinned seed, (2) apply ddmin per failing test to build a directed dependency graph, (3) obtain quorum approval of the triage report before touching any code, (4) fix in topological order with per-fix quorum approval + regression test per commit, (5) re-run the full suite after each fix to detect regressions immediately, and (6) obtain final quorum sign-off on the terminal state.

The existing infrastructure covers most of the heavy lifting. `maintain-tests batch --seed N` already produces deterministic ordering. `maintain-tests ddmin` already implements ddmin2. `maintain-tests run-batch` already captures per-test results. The quorum orchestrator pattern (qgsd-quorum-orchestrator.md) works via `call-quorum-slot.cjs`. What is new: a whole-suite ddmin invocation with a raised run cap, the dependency graph data structure + cycle detection, the triage report format, the sequential fix loop with state diffing, and the new fix-tests.md workflow file that glues these together.

**Primary recommendation:** Rewrite fix-tests.md as a 4-phase workflow. Extend `maintain-tests ddmin` with a `--run-cap` flag (default stays 50, new default for whole-suite use: 200). Add `maintain-tests build-graph` subcommand that takes ddmin results and produces the dependency graph JSON. Everything else (quorum calls, run-batch, state management) reuses existing infrastructure verbatim.

### Pattern 2: Failure Signature Hashing

**What:** Normalize error output to a stable hash. Allows comparing "same failure" vs "new failure" after a fix.
**When to use:** After every full-suite run. Compare post-fix signatures against baseline.

Normalization rules (apply before hashing):
1. Strip line numbers from stack traces: `at Function (file.js:123:45)` → `at Function (file.js:LINE:COL)`
2. Strip absolute paths: replace project root prefix with `<ROOT>/`
3. Strip timestamps and session IDs (UUIDs, ISO dates)
4. Lowercase
5. SHA-256 of the normalized string

```javascript
// Source: derived from gsd-tools.cjs normalization conventions (inline in Python runner)
function normalizeError(errorText, projectRoot) {
  return errorText
    .replace(/:\d+:\d+\)/g, ':LINE:COL)')          // strip line numbers
    .replace(new RegExp(projectRoot, 'g'), '<ROOT>') // strip absolute paths
    .replace(/[0-9a-f-]{36}/gi, '<UUID>')            // strip UUIDs
    .replace(/\d{4}-\d{2}-\d{2}T[\d:.Z]+/g, '<TS>') // strip timestamps
    .toLowerCase()
    .trim();
}
```

**Signature comparison:** If `post_fix_error_hash !== baseline_error_hash` for a test that was failing before the fix and is still failing after — that is a "signature change" event, which requires re-ddmin on that test to understand if the failure root cause has shifted.

### Pattern 3: ddmin Whole-Suite Invocation

**What:** For each failing test, run ddmin with ALL OTHER failing tests + passing tests as candidates.
**Critical constraint:** Candidates must be in the SAME pinned order as the baseline run, minus the target test.

```bash
# Build candidates list: all tests except the target, in original pinned order
# (Python inline: filter ddmin-discover.json by removing failingTest, preserve order)

node /Users/jonathanborduas/.claude/qgsd/bin/gsd-tools.cjs maintain-tests ddmin \
  --failing-test path/to/target.test.js \
  --candidates-file .planning/ddmin-candidates-target.json \
  --run-cap 200 \
  --timeout 300 \
  --output-file .planning/ddmin-result-target.json
```

**Run cap guidance:**
- Current default: 50 — adequate for within-batch (small N)
- Whole-suite: needs 200+ for large suites. ddmin2 complexity is O(n log n) in best case.
- For 500 failing tests with 500 candidates each: 200 cap per test × 500 tests = 100,000 test runner invocations. This is expensive. Scope-bounding is required (see Pitfall 1).

### Pattern 6: Per-Fix Commit Structure

**What:** Every fix commit must include exactly 3 things: the code fix, a regression test, and a quorum approval record.
**How to embed approval record:**

Option A (preferred): Include approval record in commit message body:
```
fix(tests): fix shared-state pollution in auth.test.js

Root cause: auth.test.js leaves global `currentUser` set; subsequent tests fail
because they assume null initial state.

Fix: add afterEach cleanup in auth.test.js

Regression test: tests/__regression__/auth-cleanup.test.js added

Quorum approval: APPROVED (3 models: gemini-1, claude-1, claude-2)
Round: 1
Timestamp: 2026-02-23T12:34:56Z
```

Option B (alternative): Write approval record to `.planning/fix-approvals/TIMESTAMP-TESTFILE.json` and include that file in the commit.

**Recommendation:** Use Option A (commit message body) — no extra file, approval is co-located with the change, git log is the audit trail.

### Anti-Patterns to Avoid

- **Batching any part of the pipeline:** No batches. The whole point of this redesign is that batching hides cross-test state changes. Even grouping "independent" failures into a multi-fix commit is dangerous — fix one at a time.
- **Running ddmin with randomized order:** ddmin must use the SAME seed as the baseline run. Different orderings produce different dependency graphs. The polluter must be run in the same relative position it was during the polluting run.
- **Treating "test fails in isolation" as "independent failure" without verification:** The current ddmin implementation returns `ddmin_ran: false, reason: "test fails in isolation"` when the test fails with empty candidates. This is correct for independent failures. But confirm by running the test 3 times in isolation to distinguish genuine bugs from environment flakiness.
- **Fixing cycle groups with individual commits:** Cycle groups (A pollutes B, B pollutes A) must be treated atomically. Either fix the shared state in a single commit that resolves both, or quarantine the cycle and document it.
- **Skipping the post-fix full-suite run:** After EVERY fix commit, re-run the full suite and diff failure signatures. This is the mechanism that replaces batch-based iteration detection. If a new failure signature appears, ddmin on the delta immediately.

## Common Pitfalls

### Pitfall 1: Whole-Suite ddmin is Combinatorially Expensive
**What goes wrong:** Running ddmin for every failing test against the full candidate set (all other N tests) with a 200-run cap results in 200 × F test runner invocations (F = count of failing tests). For a suite with 50 failing tests and 500 total tests: 200 × 50 = 10,000 invocations. At 1 second per test: ~2.7 hours.
**Why it happens:** ddmin2 worst case is O(n²) per invocation.
**How to avoid:** Scope-bound ddmin by shared module/fixture first:
  1. Group failing tests by directory prefix (same 2-path-segment rule used in current dispatch)
  2. Run ddmin first within each group (small candidate set, fast)
  3. Only run cross-group ddmin for tests that passed step 2 with empty polluter_set
  This reduces candidate set size by 80-90% for most suites.
**Warning signs:** ddmin phase taking >30 minutes on suites with <100 tests.

### Pitfall 2: Signature Change False Positives After Unrelated Fixes
**What goes wrong:** Fixing test A changes the DB schema, which causes test B to fail with a different error message. The signature diff detects this as "new failure" and triggers unnecessary re-ddmin.
**Why it happens:** Failure signatures are sensitive to any shared environment state that the fix touches.
**How to avoid:** Normalize signatures aggressively (strip line numbers, absolute paths, timestamps, UUIDs). Accept that some false positives will occur — design the re-ddmin as a fast path that quickly confirms "no new polluter" rather than a full investigation.
**Warning signs:** Every fix triggers a new ddmin run on unrelated tests.

### Pitfall 3: Cycle Detection Deadlock
**What goes wrong:** Cycle group A→B→A is identified. The fix loop tries to fix A first (to remove the pollution), but fixing A requires knowing B's state, and vice versa. The loop deadlocks or produces an incorrect fix.
**Why it happens:** Circular state dependencies cannot be resolved by sequential fixing.
**How to avoid:** Cycle groups must be handled atomically:
  - Identify the shared state (global variable, DB table, file, singleton)
  - Write a single fix that resets that shared state in `beforeEach`/`afterEach` for ALL tests in the cycle group
  - Commit the atomic fix with quorum approval
  - If no single shared state can be identified: quarantine the cycle group, document it in the final report, and exit the fix loop for those tests.
**Warning signs:** ddmin result shows A pollutes B AND B pollutes A.

### Pitfall 4: ddmin "Test Fails Alone" Masking a Different Pollution
**What goes wrong:** Test B fails alone (ddmin returns `reason: "test fails in isolation"`). But actually B is a flaky test that sometimes passes and sometimes fails. The ddmin run happened to catch it during a fail. It gets classified as "independent failure" and a real-bug fix task is dispatched, wasting effort.
**Why it happens:** The baseline run captured B as "failed" but B is actually flaky (50/50 pass/fail).
**How to avoid:** Run the flakiness classification BEFORE ddmin, not after. For every test that fails in the baseline run, rerun it N=10 times in isolation. If fail_count/rerun_count < 0.8 → classify as flaky, skip ddmin. Only run ddmin on tests with fail_count/rerun_count >= 0.8 (consistently failing).
**Warning signs:** Tests that ddmin classifies as "independent failure" occasionally pass in CI.

### Pitfall 5: Run-Cap Reached = Incomplete Graph
**What goes wrong:** ddmin hits the run cap (200) before finding the minimal polluter set. It returns whatever `remaining` is at that point, which may be a superset of the true minimal polluter set.
**Why it happens:** Large candidate sets with complex interdependencies require more than O(n log n) best-case iterations.
**How to avoid:** The `reason: "run cap reached"` result is valid — use the returned `polluter_set` as an upper bound. Record `run_cap_reached: true` in the dependency graph edge. The triage report should flag these as "approximate chains" vs "confirmed chains." The fixer agent should fix the approximate chain in order and re-run ddmin post-fix to verify.
**Warning signs:** Many ddmin results have `reason: "run cap reached"` with large `polluter_set` arrays.

### Pitfall 6: Quorum Hanging During Fix Loop
**What goes wrong:** A quorum model hangs during a per-fix approval gate, stalling the entire pipeline for 23+ minutes before the timeout fires.
**Why it happens:** Known issue with DeepSeek/MiniMax providers (documented in MEMORY.md).
**How to avoid:** Always use `call-quorum-slot.cjs` with explicit `--timeout` (120s default from CLAUDE_MCP_TIMEOUT_MS). The orchestrator already handles timeout → UNAVAIL → skip. The fix loop must not use MCP tool calls directly — use the orchestrator pattern only.
**Warning signs:** Quorum approval gates taking >2 minutes per fix.

## Code Examples

### Example 1: Flakiness Classification Loop (inline in workflow)

```bash
# For each failing test in baseline_signatures, rerun N=10 in isolation
# Uses run-batch with a single-test "batch"
for TEST_FILE in $FAILING_TESTS; do
  echo '{"test_files":["'$TEST_FILE'"]}' > /tmp/single-test-discover.json
  node $GSD_TOOLS maintain-tests batch \
    --input-json '{"test_files":["'$TEST_FILE'"]}' \
    --seed $BASELINE_SEED \
    --size 1 \
    --manifest-file /tmp/single-test-manifest.json

  FAIL_COUNT=0
  for i in $(seq 1 10); do
    node $GSD_TOOLS maintain-tests run-batch \
      --batch-file /tmp/single-test-manifest.json \
      --batch-index 0 \
      --timeout 60 \
      --output-file /tmp/single-test-result.json
    STATUS=$(node -e "const r=require('/tmp/single-test-result.json');console.log(r.results[0]?.status)")
    if [ "$STATUS" = "failed" ]; then FAIL_COUNT=$((FAIL_COUNT + 1)); fi
  done

  # is_flaky = fail_count / 10 < 0.8
  RATIO=$(echo "scale=2; $FAIL_COUNT / 10" | bc)
  if (( $(echo "$RATIO < 0.8" | bc -l) )); then
    echo "FLAKY: $TEST_FILE (failed $FAIL_COUNT/10)"
    # Add to flaky list; skip ddmin
  fi
done
```

### Example 3: Post-Fix Signature Diff

```python
# After each fix commit, re-run full suite and compare signatures
def diff_signatures(baseline, post_fix):
    new_failures = []
    resolved = []
    signature_changed = []

    for test, sig in post_fix.items():
        if sig['status'] == 'failed':
            if test not in baseline or baseline[test]['status'] != 'failed':
                new_failures.append(test)  # regression
            elif sig['error_hash'] != baseline[test]['error_hash']:
                signature_changed.append(test)  # same test, different failure
        elif sig['status'] == 'passed' and baseline.get(test, {}).get('status') == 'failed':
            resolved.append(test)  # fixed

    return {
        'new_failures': new_failures,       # requires immediate ddmin
        'resolved': resolved,               # good
        'signature_changed': signature_changed,  # investigate
    }
```

## State of the Art

| Old Approach | New Approach | Impact |
|--------------|--------------|--------|
| Batch-based: fix N tests per batch | ddmin dependency graph: fix one test at a time in dependency order | Eliminates cross-batch pollution blind spot |
| Heuristic pre-classification (adapt/isolate/real-bug/fixture) | ddmin-based classification: independent failure vs pollution chain vs flaky | More precise root-cause routing |
| AI categorization then dispatch | Quorum triage approval then quorum per-fix approval | Adds two quorum gates that don't exist in current workflow |
| No regression detection within a run | Post-fix signature diff after every commit | Catches regressions immediately instead of at next batch boundary |
| Ddmin within a batch (50 candidates max) | Ddmin across full suite (scope-bounded, 200 cap) | Catches cross-module pollution that current within-batch ddmin misses |
| No cycle detection | Explicit SCC-based cycle detection + quarantine | Prevents deadlock in fix ordering |
| No flakiness classification before ddmin | N=10 rerun flakiness gate before ddmin | Prevents wasting ddmin budget on flaky tests |

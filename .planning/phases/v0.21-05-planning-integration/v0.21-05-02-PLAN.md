Create `bin/run-phase-tlc.cjs` following patterns from `bin/run-oscillation-tlc.cjs` (Java/TLC invocation, output parsing).

1. **`runPhaseTlc(specPath, cfgPath, options)`** function:
   - Locates Java binary (check `JAVA_HOME` env, then fall back to `java` on PATH). Accept optional `options.javaOverride` for testing.
   - **Explicit tla2tools.jar existence check:** Before invoking TLC, check `fs.existsSync(tla2toolsPath)` where tla2toolsPath is `path.join(__dirname, '..', 'formal', 'tla', 'tla2tools.jar')`. This matches the pattern in `run-oscillation-tlc.cjs` (line ~109) and prevents cryptic subprocess errors when the jar is missing. If the jar does not exist, return `{ passed: false, violations: ['tla2tools.jar not found at ' + tla2toolsPath + ' -- run: curl -L https://github.com/tlaplus/tlaplus/releases/download/v1.8.0/tla2tools.jar -o formal/tla/tla2tools.jar'], output: '', runtimeMs: 0 }` immediately (no subprocess spawned).
   - Invokes TLC via `spawnSync('java', ['-XX:+UseParallelGC', '-jar', tla2toolsPath, '-workers', '1', '-config', cfgPath, specPath])` with 60s timeout.
   - Parses stdout+stderr for violation patterns:
     - `Invariant (\w+) is violated` captures property name
     - `Property (\w+) is violated` captures property name
     - `Error:` lines captured as errors
     - `Model checking completed. No error has been found.` indicates pass
   - Returns `{ passed: boolean, violations: string[], output: string, runtimeMs: number }`.
   - If Java not found, returns `{ passed: false, violations: ['Java not found -- install JDK 17+'], output: '', runtimeMs: 0 }`.

2. **`formatTlcFeedback(attempt, maxAttempts, tlcResult, truthsList)`** function:
   - Takes attempt number (1-3), max attempts, TLC result, and original truths list.
   - For each violation, matches the property name (e.g., `Req01`) back to the truth index via the `Req` prefix and zero-padded number. The parsing regex should be `/Req(\d{2})/` to extract the NN suffix, then convert to 0-based array index: `parseInt(match, 10) - 1`. This aligns with PLAN-01's zero-padded `Req01`, `Req02`, ... naming scheme.
   - On failure, returns:
     ```
     ATTEMPT {N}/{max}: TLC verification FAILED.
     Violated: Req{NN} ({INVARIANT|PROPERTY}) -- "{truth text}"
     Suggestion: Revise the truth statement or adjust the plan to satisfy this constraint.
     ```
     When multiple violations exist, each gets its own `Violated:` line.
   - On pass, returns: `"ATTEMPT {N}/{max}: TLC verification PASSED. All {M} properties satisfied."`

3. **`iterativeVerify(planFilePath)`** function:
   - Designed to be called once per attempt by the orchestrator (plan-phase.md manages the loop and attempt counter).
   - Calls `generateProposedChanges(planFilePath)` from generate-proposed-changes.cjs.
   - If no truths (generated: false), returns `{ status: 'skipped', reason: 'no truths in plan' }`.
   - Calls `generateTlaCfg(specPath)` from generate-proposed-changes.cjs.
   - Runs `runPhaseTlc(specPath, cfgPath)`.
   - If passed: returns `{ status: 'passed', truthCount, specPath, runtimeMs }`.
   - If failed: returns `{ status: 'failed', violations, feedback: formatTlcFeedback(...), specPath }`.
   - The `feedback` field is the human-readable string that the orchestrator injects into the next planner revision prompt.

4. **CLI entrypoint** (guarded by `require.main === module`):
   - Usage: `node bin/run-phase-tlc.cjs <path-to-PLAN.md>`
   - Calls `iterativeVerify(planFilePath)`.
   - Prints result as JSON to stdout.
   - Exit 0 on pass or skip, exit 1 on fail.

5. **Exports:** `module.exports = { runPhaseTlc, iterativeVerify, formatTlcFeedback }`.

**Key pattern from run-oscillation-tlc.cjs:** Java discovery, spawnSync with timeout, stdout+stderr concatenation, violation regex parsing. Follow the same structure adapted for proposed-changes.tla context.

**Error handling:** Fail-open. If Java missing or TLC crashes, return a structured failure (not process.exit) so the orchestrator can document it and proceed.

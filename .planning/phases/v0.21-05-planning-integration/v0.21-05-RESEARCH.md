# Phase v0.21-05: Planning Integration - Research

**Researched:** 2026-03-01
**Domain:** Planning integration with formal verification (TLA+ delta synthesis, TLC verification gates, quorum context enrichment)
**Confidence:** HIGH

## Summary

Phase v0.21-05 closes the v0.16 deferral by making planning decisions formally verifiable before quorum sees them. Currently, `plan-phase.md` produces plans without any formal verification of the proposed changes. This phase requires three integrated components:

1. **PLAN-01**: Automatic TLA+ delta synthesis from each plan's `must_haves: truths:` block, saved as `formal/proposed-changes.tla` in the phase directory
2. **PLAN-02**: An iterative TLC verification loop inside plan-phase.md that gates the plan — if TLC fails, Claude iterates (max 3 attempts); plans reaching quorum are either TLC-verified or documented as failures
3. **PLAN-03**: Enrichment of quorum slot-worker prompts with `formal_spec_summary` (proposed TLA+ properties in plain language) and `verification_result` (TLC pass/fail/inconclusive) so agents vote with mathematical evidence attached

The path is clear: v0.21-01 (ARCH) and v0.21-04 (SPEC) completed the dependencies. v0.21-05 integrates the pieces — it wires the formal verification pipeline into the planning workflow itself, making it impossible to ship an unverified plan to quorum.

**Primary recommendation:**
Implement in order of dependency: (1) write a scratch TLA+ spec generator that reads `must_haves: truths:` from PLAN.md YAML frontmatter and produces a proposable TLA+ MODULE with INVARIANT/PROPERTY stubs; (2) wire TLC verification into plan-phase.md as a post-generation step with 3-attempt iteration and failure documentation; (3) extend the quorum slot-worker prompt template to inject formal_spec_summary + verification_result fields into the context when a formal spec exists.

## Architecture Patterns

### Pattern 1: TLA+ Delta Synthesis from Plan Frontmatter
**What:** Extract `must_haves: truths:` array from PLAN.md YAML frontmatter, translate each truth statement into a TLA+ INVARIANT (safety keywords: threshold, count, max, never, always) or PROPERTY (liveness keywords: eventually, deadline, async, progress).

**When to use:** Every plan that has `must_haves: truths:` entries. The generator runs post-plan, reads the PLAN.md file from disk, parses the YAML frontmatter (--- delimited block), and outputs a scratch TLA+ MODULE with PROPERTY stubs.

**Example:**
```
PLAN.md frontmatter truth:
"The circuit breaker enters RESET state after 3 consecutive failures"

Translation:
PROPERTY CircuitBreakerReset ==
  (<>failureCount = 3 /\ state = VOTING) => <>(state = RESET)
```

Data source: **PLAN.md YAML frontmatter** (NOT task-envelope.json; envelope truths are empty at planning time per v0.21-04 finding).

**Key integration point:** v0.21-04 created `bin/generate-phase-spec.cjs` that reads *-PLAN.md YAML frontmatter and generates scratch specs for verification. PLAN-01 reuses this pattern but runs during plan-phase.md, with per-plan `proposed-changes.tla` file saved to `.planning/phases/<phase>/formal/proposed-changes.tla`.

### Pattern 2: Iterative TLC Verification Loop Inside plan-phase.md
**What:** After planner generates a PLAN.md with `must_haves: truths:`, automatically:
1. Run TLA+ delta synthesis to create proposed-changes.tla
2. Run TLC on the spec
3. If violations found: capture the violation, inject it as feedback to planner, iterate (max 3 attempts)
4. If TLC passes after any attempt: document TLC PASS in verification summary, proceed to quorum
5. If 3 attempts exhausted: document all TLC failures, proceed to quorum with failure record

**When to use:** Always, when a plan contains `must_haves: truths:` entries. Skip silently if no truths (empty plan).

**Why this pattern:** Planning decisions become formally verifiable before quorum votes. Violations are caught early (cost ~5-10s per iteration). The iteration loop mirrors the RESEARCH-based improvement loop already in plan-phase.md (R3.6 improvement iterations). After 3 attempts, the planner has done its best; documented failures signal to quorum "this was attempted and found problematic" rather than silently shipping an unverified plan.

**Integration point:** The loop runs AFTER planner returns a PLAN.md but BEFORE quorum sees it. Sits between planning and quorum dispatch in plan-phase.md's improvement_iteration loop (existing structure from R3.6).

### Pattern 3: Quorum Slot-Worker Prompt Enrichment
**What:** When a formal spec exists (proposed-changes.tla for the phase), compute:
- `formal_spec_summary`: Proposed TLA+ properties in plain English (e.g., "Invariant: circuit breaker never enters RESET without at least 3 consecutive failures")
- `verification_result`: String field: "PASS" | "FAIL: [reason]" | "INCONCLUSIVE: [reason]"

Inject these fields into the quorum slot-worker prompt template (Mode A prompt) as optional context lines:

```
[If formal_spec_summary present:]
=== Formal Evidence ===
Proposed TLA+ properties (from plan's must_haves: truths:):
<formal_spec_summary>

Verification result: <verification_result>
==================
```

**When to use:** Whenever a plan's proposed-changes.tla exists. The orchestrator (plan-phase.md) checks for this file and includes it in the quorum dispatch context if present.

**Why this pattern:** Agents vote with mathematical evidence attached. They see not just the plan prose but the formal constraints the plan proposes to enforce and whether TLC verified them. This closes the informational gap between planning (what must be true) and quorum consensus (do we agree this is sound).

**Integration point:** qgsd-quorum-slot-worker agent's Step 3 (build prompt) — add optional injection after the artifact section but before prior_positions section.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML frontmatter parsing from PLAN.md | Hand-written regex with string splits | v0.21-04's existing `generate-phase-spec.cjs` logic (read entire file, split on `---`, parse with YAML or regex) | Frontmatter parsing has edge cases: empty files, no YAML, malformed dates; v0.21-04 already solved this; reuse pattern |
| TLA+ MODULE generation from truths | Template strings with string interpolation | Structured template with keyword classification (safety vs liveness) then `INVARIANT` vs `PROPERTY` assignment | Unstructured templates lead to invalid TLA+ syntax; keyword classification ensures semantic correctness |
| TLC invocation and output parsing | Inline spawnSync in plan-phase.md | Extracted runner script (e.g., `run-phase-tlc.cjs`) with dedicated output parser | Output parsing requires regex for "Violation", "Error", line numbers; separate script is testable, reusable, loggable |
| Quorum context injection | Modify qgsd-quorum-slot-worker.md directly | Create a new slot-worker argument (formal_spec_summary, verification_result) that the orchestrator injects conditionally | Direct modification mixes concerns; optional arguments are already supported (review_context pattern); keep agent generic |
| Caching TLC results between attempts | Custom JSON sidecar | Embed result in PLAN.md frontmatter as `verification: { result: "...", timestamp: "...", attempt: N }` field or as plain-text summary block | Sidecar files get lost; PLAN.md is the source of truth; keep result close to the data |

**Key insight:** Formal verification in planning is a pipeline: PLAN.md → TLA+ gen → TLC run → feedback → iterate → quorum. Each step has a clear responsibility. Resist the urge to inline everything into plan-phase.md; extract runners and let them compose.

## Common Pitfalls

### Pitfall 1: Attempting TLC on Empty or Malformed proposed-changes.tla
**What goes wrong:** Generator produces a spec that doesn't parse TLA+. TLC fails with syntax error. Plan iteration gets stuck because the error message is opaque to the planner.

**Why it happens:** YAML frontmatter parsing succeeds but produces truths that can't be translated to valid TLA+ (e.g., a truth with special characters, or one that uses keywords not recognized by the keyword classifier).

**How to avoid:**
1. The generator must validate truths BEFORE writing proposed-changes.tla. Add a schema check: each truth must match a set of allowed keywords (safety: threshold, count, max, never, always; liveness: eventually, deadline, async, progress, future).
2. If a truth doesn't match, the generator returns an error WITH THE BAD TRUTH QUOTED, not a silent skip.
3. Only write proposed-changes.tla if validation passes.

**Warning signs:** TLC runner returns "SPECIFICATION Spec not found" or "Parsing error" — indicates the .tla file is malformed, not that the spec violates invariants.

### Pitfall 2: TLC Iteration Feedback Loop Not Reaching Planner
**What goes wrong:** TLC finds a violation. The iteration loop captures it but doesn't inject it into the next planner call. Planner reruns with same PLAN.md, produces same violation, infinite loop or early termination.

**Why it happens:** The violation message is computed but not passed to the planner's "Improvement" context in the R3.6 iteration structure. The orchestrator's R3.6 loop (improvement_iteration) expects improvements from quorum, not from TLC. Wiring TLC feedback into the same improvement loop requires a new channel.

**How to avoid:**
1. Planner needs explicit feedback channel: `<tlc_iteration_context>` section in improvement revision mode (new, separate from quorum improvements).
2. Format: "ATTEMPT N/3: TLC violation detected. Invariant `X` violated because Y. Revision suggestions: [...]"
3. Parse violation output systematically: look for "Invariant X is violated" lines, extract line number in spec, translate back to corresponding truth in frontmatter.

**Warning signs:** Planner ignores TLC feedback and re-generates same broken plan. Check stderr logs for whether TLC output was passed to planner.

### Pitfall 3: Confusion Between proposed-changes.tla and Canonical Formal Models
**What goes wrong:** The proposed-changes.tla file created during planning becomes confused with the canonical models in `formal/tla/`. A planned change is promoted to the main model (ARCH-02). Later, another plan creates a proposed-changes.tla that conflicts with the promoted change.

**Why it happens:** proposed-changes.tla is a scratch, throwaway artifact scoped to one planning round. The path `formal/proposed-changes.tla` looks like it belongs in the canonical formal/ tree. After execution, the plan's truth implementations get merged into the real spec via promote-model.cjs (ARCH-02). But if proposed-changes.tla is left in formal/, it creates namespace confusion.

**How to avoid:**
1. **Scoped location:** Always write to `.planning/phases/<phase>/formal/proposed-changes.tla`, NOT `formal/tla/proposed-changes.tla`.
2. **Lifecycle:** proposed-changes.tla is ephemeral — scoped to planning of this phase, used by TLC in plan-phase.md, read by quorum. It is NOT committed to git by plan-phase.md (planning is a multi-attempt, streaming process). It IS only archived/visible if the plan passes and becomes PLAN.md in the repo.
3. **After execution:** When the planned changes are actually implemented and tests pass, VERIFICATION.md documents the truths as "verified". The spec fragments are promoted via promote-model.cjs into formal/tla/ with versioning in model-registry.json (ARCH-02 handles this separately).

**Warning signs:** Multiple proposed-changes.tla files in different locations. grep for "proposed-changes" finds multiple paths.

### Pitfall 4: Formal Spec Summary Too Technical for Quorum
**What goes wrong:** formal_spec_summary is a copy-paste of TLA+ properties in symbolic notation. Quorum reviewers don't read formal logic; they skip the evidence and vote based on the plan prose alone.

**Why it happens:** The generator outputs raw TLA+ (`INVARIANT SafetyInvariant1 == ...`). This is correct formally but opaque to non-logicians.

**How to avoid:**
1. formal_spec_summary must be a plain-English translation of the properties, NOT the TLA+ code.
2. Translator pattern: "TLA+ property X" → "In plain English: [one sentence capturing the constraint]"
3. Example:
   - TLA+: `PROPERTY EventuallyPass == HasQuorumEvidence => <>decision = PASS`
   - English: "If quorum evidence is provided, the system will eventually reach a PASS decision"
4. The translator runs AFTER TLC verification completes. It reads proposed-changes.tla, extracts INVARIANT/PROPERTY names and line content, and produces plain-English prose.

**Warning signs:** Quorum votes disregard formal evidence. Slot workers report "couldn't understand the formal section" in citations.

### Pitfall 5: Not Documenting Failures When TLC Exhausts 3 Attempts
**What goes wrong:** Plan iteration hits attempt 3 cap. TLC still fails. Quorum has no record that this was attempted. They approve an unverified plan thinking it wasn't checked.

**Why it happens:** The planner's final output is just the PLAN.md. The TLC failure logs are on disk or in stderr but not part of the hand-off to quorum.

**How to avoid:**
1. When attempt 3 TLC fails: update the PLAN.md frontmatter to add a `verification_failures` field (before quorum sees it):
   ```yaml
   verification_failures:
     - "Attempt 1: Invariant X violated"
     - "Attempt 2: Invariant Y violated"
     - "Attempt 3: Invariant Z violated (cap reached)"
   ```
2. Also write a summary line visible in the plan body:
   ```
   ⚠ NOTE: This plan underwent 3 TLC verification attempts. All attempts encountered violations. See verification_failures in frontmatter for details. Quorum is reviewing a plan with documented formal failures.
   ```
3. This ensures quorum cannot claim ignorance.

**Warning signs:** A plan ships to quorum with verification_failures but no summary note. Quorum reports "we didn't see any verification warnings."

## Code Examples

### TLA+ Keyword Classification for Truths
```typescript
// Source: v0.21-04 RESEARCH, pattern 2 — truth classification
function classifyTruth(truthStatement: string): 'INVARIANT' | 'PROPERTY' {
  const safetyKeywords = ['threshold', 'count', 'max', 'never', 'always', 'must'];
  const livenessKeywords = ['eventually', 'deadline', 'async', 'progress', 'future', 'will'];

  const statementLower = truthStatement.toLowerCase();

  if (safetyKeywords.some(kw => statementLower.includes(kw))) {
    return 'INVARIANT';
  }
  if (livenessKeywords.some(kw => statementLower.includes(kw))) {
    return 'PROPERTY';
  }

  // Default to PROPERTY (liveness is less risky than false-negative safety)
  return 'PROPERTY';
}

// Example usage:
const truth = "The circuit breaker never enters RESET without at least 3 failures";
const classification = classifyTruth(truth);  // returns 'INVARIANT' (keyword: "never")
```

### YAML Frontmatter Extraction from PLAN.md
```typescript
// Source: v0.21-04 bin/generate-phase-spec.cjs pattern
function extractMustHavesTruths(planFileContent: string): string[] {
  // Split on first --- to isolate YAML frontmatter
  const parts = planFileContent.split('---');
  if (parts.length < 3) {
    return [];  // No valid YAML frontmatter
  }

  const yamlBlock = parts[1];  // Content between --- delimiters

  // Parse YAML manually (avoid external dependency)
  // Look for "must_haves:" line, then "truths:" indented array
  const mustHavesMatch = yamlBlock.match(/must_haves:\s*\n\s+truths:\s*\n([\s\S]*?)(?=\n  \w|\n\w|$)/);
  if (!mustHavesMatch) {
    return [];
  }

  const truthLines = mustHavesMatch[1]
    .split('\n')
    .map(line => line.replace(/^\s*-\s*/, '').trim())  // Remove bullet and whitespace
    .filter(line => line.length > 0 && !line.startsWith('"'));  // Filter empty, keep quoted strings

  // Handle quoted strings: "..." format
  const truths: string[] = [];
  let currentTruth = '';
  let inQuote = false;

  for (const line of truthLines) {
    if (line.startsWith('"')) {
      inQuote = true;
      currentTruth = line.slice(1);  // Remove opening quote
    }
    if (inQuote) {
      currentTruth += ' ' + line;
    }
    if (line.endsWith('"')) {
      inQuote = false;
      truths.push(currentTruth.slice(0, -1));  // Remove closing quote
    }
  }

  return truths;
}

// Usage:
const planContent = fs.readFileSync('.planning/phases/v0.21-05-01-PLAN.md', 'utf8');
const truths = extractMustHavesTruths(planContent);
// Returns: ["Truth 1", "Truth 2", ...]
```

### TLA+ MODULE Generation from Truths
```typescript
// Source: v0.21-04 pattern
function generateTlaModule(phase: string, truths: string[], timestamp: string): string {
  const properties: string[] = [];

  truths.forEach((truth, idx) => {
    const classification = classifyTruth(truth);
    const propName = `Truth${idx + 1}`;

    properties.push(`  ${classification} ${propName} == TRUE  \\* "${truth}"`);
  });

  const moduleContent = `---- MODULE ProposedChanges ----
(* Proposed changes for phase ${phase}
   Generated: ${timestamp}
   Source: must_haves.truths from ${phase}-PLAN.md
*)

${properties.join('\n')}

Spec == INIT
====`;

  return moduleContent;
}

// Usage:
const tlaCode = generateTlaModule('v0.21-05', truths, new Date().toISOString());
fs.writeFileSync('.planning/phases/v0.21-05-planning-integration/formal/proposed-changes.tla', tlaCode);
```

### TLC Invocation and Violation Detection
```typescript
// Source: bin/run-oscillation-tlc.cjs pattern
function runTlcVerification(specPath: string, configPath: string): { passed: boolean; violations: string[] } {
  const { spawnSync } = require('child_process');

  const result = spawnSync('java', [
    '-XX:+UseParallelGC',
    '-jar', 'formal/tla/tla2tools.jar',
    '-workers', '1',
    '-config', configPath,
    specPath
  ], {
    cwd: process.cwd(),
    stdio: ['pipe', 'pipe', 'pipe'],
    timeout: 60000  // 60s timeout per attempt
  });

  const output = result.stdout.toString() + result.stderr.toString();
  const violations: string[] = [];

  // Parse for violation patterns
  const violationMatches = output.match(/(?:Invariant|Property) (\w+) is violated/g) || [];
  violations.push(...violationMatches.map(m => m.trim()));

  // Check for errors
  if (result.status !== 0 || output.includes('Error') || output.includes('Exception')) {
    return {
      passed: false,
      violations: violations.length > 0 ? violations : [output.slice(0, 200)]  // First 200 chars if no match
    };
  }

  return {
    passed: true,
    violations: []
  };
}

// Usage:
const tlcResult = runTlcVerification(
  '.planning/phases/v0.21-05-planning-integration/formal/proposed-changes.tla',
  '.planning/phases/v0.21-05-planning-integration/formal/MCProposedChanges.cfg'
);
if (!tlcResult.passed) {
  console.error('TLC violations:', tlcResult.violations);
}
```

## Open Questions

1. **How many TLA+ specs should a single proposed-changes.tla reference?**
   - What we know: The spec synthesizer can generate multiple PROPERTYs from a single PLAN.md's truths array. Each truth becomes one property. The .tla file has a single MODULE.
   - What's unclear: Should proposed-changes.tla directly reference existing canonical specs (e.g., `EXTENDS QGSDQuorum`) or be standalone with typed stubs?
   - Recommendation: Start standalone (stubs only). If a truth references an existing canonical spec's state variables, that indicates the plan changes the canonical spec — defer to ARCH-02 promote-model.cjs to handle merging. Keep proposed-changes.tla self-contained for the planning phase.

2. **How to handle truthstatements that don't contain TLA+-translatable keywords?**
   - What we know: The keyword classifier has safelist (threshold, count, max, never, always, eventually, deadline, async, progress). Truths outside this list fall back to PROPERTY (liveness, safer than false-negative safety).
   - What's unclear: Should the generator flag unclassified truths to the planner as "warning: this property may not parse TLA+ correctly"? Or silently classify and let TLC catch parse errors?
   - Recommendation: Silent classification with TLC as the validator. If a truth produces invalid TLA+, TLC reports a syntax error, which is fed back to planner as "Syntax error in proposed-changes.tla: [error line]". Planner can then revise the truth statement. This keeps feedback direct and actionable.

3. **Should formal_spec_summary be generated from proposed-changes.tla or from the original truths?**
   - What we know: proposed-changes.tla contains INVARIANT/PROPERTY stubs. The original truths are in PLAN.md frontmatter in plain English.
   - What's unclear: Generating from original truths is simpler (just reuse them). Generating from proposed-changes.tla requires parsing TLA+ and translating back to English (lossy). Which is safer?
   - Recommendation: Generate from original truths (lossless). Store a mapping from truth index to generated property name in the proposed-changes.tla file as a comment block. This preserves bidirectional traceability.

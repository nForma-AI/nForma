# Phase v0.38-04: B-to-F Solve Layer - Research

**Researched:** 2026-03-18
**Domain:** Solve pipeline layer integration — bug-to-formal model gap analysis
**Confidence:** HIGH

## Summary

Phase v0.38-04 adds the `b_to_f` layer to the nForma consistency solver. This layer classifies failing tests against the formal model registry to compute a residual score, then routes gaps to remediation: `close-formal-gaps` for missing models and `model-driven-fix` for blind spots. The implementation follows well-established patterns from the existing 19 layers.

The primary technical challenge is the classification algorithm: each failing test must be traced through the requirement/recipe/annotation lineage to determine whether a formal model covers its failure mode, and if so, whether the model reproduces the bug. The upstream dependency on `t_to_c` (test-to-code traceability) provides the test-to-requirement mapping needed for this lineage.

**Primary recommendation:** Follow the existing sweep function pattern (sweepHtoM/sweepFormalLint as templates), register `b_to_f` as the 20th layer key, and wire it into the wave DAG with `t_to_c` as its sole dependency.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- `covered` = failing test maps to >=1 formal model through requirement/recipe/annotation lineage AND that model was checked
- `covered_reproduced` = 0 residual (model reproduces the bug — no action needed)
- `covered_not_reproduced` = +1 residual (models exist but all pass — route to model refinement)
- `not_covered` = +1 residual (no linked formal model exists — route to model creation via close-formal-gaps)
- Route `covered_not_reproduced` to refinement by default; only route to creation if linked model set is empty or stale
- Defaults: max 3/cycle for `not_covered` gaps, max 2/cycle for blind spots (per success criteria)
- Config-backed limits matching existing per-layer caps pattern
- Overflow items remain in residual — never dropped, surfaced in `capped_layers`/detail reporting per CONV-03
- Carried forward to next solve cycle unchanged
- Use established unified table format (not custom one-off)
- Add `B -> F` row with aggregate residual score
- Include compact breakdown: `{covered_reproduced, covered_not_reproduced, not_covered}` plus top bug ID examples
- Both text and JSON detail output available
- Deterministic ordering matching existing `computeWaves()` bias
- Dispatch priority: `not_covered` before `covered_not_reproduced`
- Within each bucket: sort by repeated failure count, then requirement/test breadth
- Stable bug ID tiebreaker for deterministic output

### Pattern 1: Sweep Function Convention
**What:** Each layer transition has a `sweepXtoY()` function returning `{ residual: N, detail: {...} }`
**When to use:** Always — this is the mandatory interface for all layer sweeps
**Example:**
```javascript
function sweepBtoF() {
  if (fastMode) {
    return { residual: -1, detail: { skipped: true, reason: 'fast mode' } };
  }
  try {
    // Classification logic here
    return {
      residual: notCoveredCount + coveredNotReproducedCount,
      detail: {
        covered_reproduced: coveredReproducedCount,
        covered_not_reproduced: coveredNotReproducedCount,
        not_covered: notCoveredCount,
        total_failing: totalFailing,
        top_bugs: topBugIds,
      },
    };
  } catch (err) {
    return { residual: -1, detail: { error: true, stderr: 'sweepBtoF failed: ' + err.message } };
  }
}
```

### Anti-Patterns to Avoid
- **Custom table format:** Use the existing unified `renderRow()` pattern — do NOT create a one-off table for B->F
- **Blocking on model checker failures:** Treat checker failures as `covered_not_reproduced` (+1), not as errors. The sweep must be fail-open.
- **Modifying t_to_c output:** B->F reads t_to_c results — it must NOT mutate them

## Common Pitfalls

### Pitfall 1: Test Flakiness Noise
**What goes wrong:** Flaky tests are classified as `not_covered` or `covered_not_reproduced`, inflating the residual and triggering unnecessary remediation
**Why it happens:** The sweep classifies ALL failing tests, including intermittently-failing ones
**How to avoid:** STATE.md already flags this: "B-to-F layer noise from test flakiness needs stability filter in Phase 4." Implement a simple stability filter: only classify tests that failed in the most recent t_to_c sweep (not historical failures). If t_to_c.detail has skip/flaky annotations, exclude those tests.
**Warning signs:** Residual oscillating between solve cycles without remediation changes

### Pitfall 2: Circular Dependency with t_to_c
**What goes wrong:** sweepBtoF() calls sweepTtoC() recursively (t_to_c already has a recursive guard)
**Why it happens:** Trying to get fresh test results inside the B->F sweep
**How to avoid:** Read t_to_c results from the already-computed residual vector (passed to computeResidual or stored as module-level state). Never call sweepTtoC() from within sweepBtoF().
**Warning signs:** `recursive-guard: already running inside node --test` error

### Pitfall 3: Model Checker Timeout Cascade
**What goes wrong:** Running model checkers for every `covered` test during the sweep causes timeouts
**Why it happens:** Attempting to verify reproduction for many tests at once
**How to avoid:** The sweep should only CLASSIFY (model exists or not). Actual model checker execution for reproduction is done at remediation time, not sweep time. Classification uses model-registry.json and traceability-matrix — no checker execution needed.
**Warning signs:** Sweep taking >60 seconds when it should be <5 seconds

### Pitfall 4: Missing Requirement Lineage
**What goes wrong:** Failing test has no requirement mapping, making classification impossible
**Why it happens:** Tests written without requirement annotations or traceability recipes
**How to avoid:** Treat unmapped tests as `not_covered` (+1 residual) — this correctly identifies them as needing formal model coverage. The remediation step can then route them appropriately.
**Warning signs:** High `not_covered` count with many unmapped tests

## Code Examples

### Classification Algorithm (Core Logic)
```javascript
/**
 * Classify a failing test against formal model coverage.
 * @param {string} testId - Test file path or identifier
 * @param {Object} traceMatrix - Traceability matrix from generate-traceability-matrix.cjs
 * @param {Object} modelRegistry - Contents of model-registry.json
 * @returns {{ classification: string, models: string[], bug_id: string }}
 */
function classifyFailingTest(testId, traceMatrix, modelRegistry) {
  // 1. Find requirement IDs linked to this test via traceability
  const reqIds = findLinkedRequirements(testId, traceMatrix);
  if (reqIds.length === 0) {
    return { classification: 'not_covered', models: [], bug_id: computeBugId(testId) };
  }

  // 2. Find formal models covering those requirements
  const models = findModelsForRequirements(reqIds, modelRegistry);
  if (models.length === 0) {
    return { classification: 'not_covered', models: [], bug_id: computeBugId(testId) };
  }

  // 3. Check bug-model-gaps.json for prior reproduction status
  const gaps = loadBugModelGaps();
  const bugId = computeBugId(testId);
  const existing = gaps.entries.find(e => e.bug_id === bugId);
  if (existing && existing.status === 'reproduced') {
    return { classification: 'covered_reproduced', models, bug_id: bugId };
  }

  // 4. Models exist but no reproduction record — classify as covered_not_reproduced
  return { classification: 'covered_not_reproduced', models, bug_id: bugId };
}
```

### Remediation Dispatch Pattern (solve-remediate.md)
```markdown
## Step 3a-extra: B->F Gaps (Bug-to-Formal)

**Skip if:** `b_to_f` residual <= 0 or -1.

Read `b_to_f.detail` from the residual vector.

**Phase 1 — Route `not_covered` gaps (max 3/cycle):**
Dispatch `/nf:close-formal-gaps` for tests without linked formal models.
Priority: sort by repeated failure count desc, then requirement breadth desc, then bug_id asc.

**Phase 2 — Route `covered_not_reproduced` blind spots (max 2/cycle):**
Dispatch model refinement for tests where models exist but don't reproduce the bug.
Use `/nf:model-driven-fix --bug-context` with the failure description.
```

## Open Questions

1. **Test flakiness filter implementation**
   - What we know: STATE.md flags this as needed. CONTEXT.md does not specify the exact mechanism.
   - What's unclear: Whether to filter by t_to_c skip/flaky annotations only, or add a "failed N consecutive cycles" threshold
   - Recommendation: Start with the simplest approach — only classify tests that appear in the current cycle's t_to_c failure list. Add consecutive-failure counting in a future phase if noise persists.

2. **Residual bucket classification (automatable vs informational)**
   - What we know: Existing layers are split into automatable (dispatched), manual, and informational buckets
   - What's unclear: Whether b_to_f should be automatable (since it dispatches remediation) or informational (since it analyzes coverage)
   - Recommendation: Automatable — it directly dispatches close-formal-gaps and model-driven-fix, matching the pattern of r_to_f and other dispatching layers

# Phase v0.35-02: MCP Setup Slot Classification - Research

**Researched:** 2026-03-12
**Domain:** MCP agent slot configuration, authentication type classification
**Confidence:** HIGH

## Summary

This phase fixes a critical gap in slot type classification: the quorum system requires knowing whether each MCP slot uses subscription-based authentication ('sub') or API-key authentication ('api'), but this information is never populated during setup. Currently, the system defaults all unknown slots to 'api', which defeats the T1/T2 tiered fallback logic in FALLBACK-01.

The fix requires three coordinated changes:

1. **Add `auth_type` field to providers.json** — Each provider definition needs an explicit auth_type value ('sub' for CLI subscription slots like codex-1/gemini-1, 'api' for API-backed slots)
2. **Populate auth_type in nf.json during setup** — The `/nf:mcp-setup` workflow must read auth_type from providers.json and write it to `~/.claude/nf.json` under `agent_config[slotName].auth_type`
3. **Use auth_type for slot classification** — The setup wizard menu display and fallback logic must read from providers.json instead of inferring from slot name prefixes

**Primary recommendation:** Add `auth_type: 'sub' | 'api'` field to every provider entry in providers.json, then modify `/nf:mcp-setup` to write this value to nf.json during both first-run and re-run flows.

## Architecture Patterns

### Pattern 1: Slot Classification via providers.json Lookup

**What:** Instead of inferring auth_type from slot name patterns (e.g., `if (name.startsWith('codex') || name.startsWith('gemini'))`), read it from the provider definition.

**When to use:** Whenever slot type needs to be determined — in setup wizards, dashboard displays, quorum routing.

**Example:**
```javascript
// Source: bin/providers.json schema + manage-agents-core.cjs
const providersData = readProvidersJson();
const providerMap = {};
for (const p of (providersData.providers || [])) {
  providerMap[p.name] = p;
}

// Later, when processing a slot:
const slotName = 'codex-1';
const provider = providerMap[slotName];
const authType = provider ? provider.auth_type : 'api'; // explicit fallback

// Write to nf.json:
agentConfig[slotName] = { auth_type: authType, /* ...other fields... */ };
```

### Pattern 2: Config-Audit Validation

**What:** The existing `config-audit.cjs` checks that at least one slot has `auth_type: 'sub'` to prevent T1 tiered fallback from being permanently empty.

**When to use:** Post-setup validation, pre-quorum startup checks.

**Example:**
```javascript
// Source: bin/config-audit.cjs lines 54–65
const hasAnySub = quorumActive.some(slot => {
  const entry = agentConfig[slot];
  return entry && entry.auth_type === 'sub';
});

if (!hasAnySub && quorumActive.length > 0) {
  warnings.push(
    'All slots default to auth_type=api — T1 tiered fallback (FALLBACK-01) will be permanently empty. ' +
    'Set auth_type=\'sub\' for subscription-tier providers (codex-1, gemini-1, opencode-1, copilot-1) ' +
    'in ~/.claude/nf.json agent_config.'
  );
}
```

### Pattern 3: Build Fallover Rule with Tiered Slots

**What:** The `buildFalloverRule()` function in nf-prompt.js separates slots into T1 (sub) and T2 (api) based on auth_type, enabling the FALLBACK-01 dispatch sequence.

**When to use:** During quorum initialization, when constructing the multi-step fallback instruction.

**Example:**
```javascript
// Source: hooks/nf-prompt.js lines 671–674
const t1 = uniqueSlots
  .filter(s => s.authType === 'sub' && !cappedSlotNames.has(s.slot))
  .map(s => s.slot);

const t2 = uniqueSlots
  .filter(s => s.authType !== 'sub' && !cappedSlotNames.has(s.slot))
  .map(s => s.slot);

const rule = buildFalloverRule(capped, t1, t2, maxVotes);
```

### Anti-Patterns to Avoid

- **Name-based inference:** `if (slotName.startsWith('codex'))` — violates single source of truth
- **Hardcoding subscription slots:** Listing specific slots in code instead of reading from schema — breaks maintainability
- **Skipping auth_type during setup:** Leaving nf.json empty and defaulting in nf-prompt.js — causes config-audit warnings and defeats FALLBACK-01

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Determining slot auth type | Name-prefix regex patterns or hardcoded lists | `providers.json[slotName].auth_type` | Single source of truth; survives schema changes; testable |
| Validating auth_type configuration | Custom comparison logic | Existing `config-audit.cjs` | Already validates T1/T2 presence, checks for all-api anti-pattern |
| Reading/writing nf.json agent_config | Ad-hoc fs operations | `manage-agents-core.cjs readNfJson() / writeNfJson()` | Handles atomicity, tmp files, error recovery |
| Displaying slot type in menus | Conditional ANSI codes based on auth_type | Existing `buildKeyStatus(authType)` in manage-agents-core.cjs | Returns colored `[sub]` or `[key ✓]` badges |

**Key insight:** The auth_type field bridges two layers:
- **Schema layer** (`providers.json`) — declarative provider definitions
- **Runtime layer** (`nf.json agent_config`) — user-specific agent configuration

Skip hand-rolling either layer and you lose the validation guarantees that `config-audit.cjs` provides.

## Common Pitfalls

### Pitfall 1: Defaulting to 'api' for Subscription Slots

**What goes wrong:** When auth_type is missing from nf.json, the code defaults to 'api' (line 506 in nf-prompt.js). For subscription CLI slots, this causes them to be classified as T2 (api) instead of T1 (sub), which inverts the fallback priority and defeats FALLBACK-01.

**Why it happens:** Legacy code pre-dates auth_type population; setup never writes it.

**How to avoid:** Write auth_type to nf.json during both first-run (onboarding) and re-run (wizard) flows of `/nf:mcp-setup`. Verify with `config-audit.cjs` before quorum init.

**Warning signs:**
- `/nf:solve` shows "T1 (sub-CLI) slots: none" when codex-1 and gemini-1 are active
- `config-audit.cjs` reports "All slots default to auth_type=api"
- Quorum fallback skips to T2 immediately instead of trying T1 first

### Pitfall 2: Hardcoding Subscription Slots in Setup Wizard

**What goes wrong:** If `/nf:mcp-setup` uses hardcoded lists like `if (name === 'codex-1' || ...)` to assign auth_type, future slots (e.g., new CLI providers in v0.36) won't be classified correctly.

**Why it happens:** Tempting short-term fix; avoids schema changes.

**How to avoid:** Always read auth_type from `providers.json`. If a new provider is added, update providers.json schema only — setup logic stays generic.

**Warning signs:** Adding conditional logic like `codex.*startsWith()` or `gemini.*startsWith()` in setup.

### Pitfall 3: Inconsistency Between Initial Setup and Later Management

**What goes wrong:** First-run setup correctly sets auth_type, but the re-run menu (agent selector + key/provider management) doesn't display or preserve it, causing confusion.

**Why it happens:** Setup and agent management may be developed separately without shared validation.

**How to avoid:** Use `buildKeyStatus(authType)` consistently in both flows. Test both first-run and re-run with `manage-agents.test.cjs`.

**Warning signs:** Setup shows `[sub]` badge but agent menu shows `[no key]` for the same slot.

### Pitfall 4: Missing Fallback for Slots Not in providers.json

**What goes wrong:** If a slot is defined in `~/.claude.json` mcpServers but doesn't exist in `providers.json`, setup can't determine auth_type and skips configuration.

**Why it happens:** User may manually add custom slots to claude.json.

**How to avoid:** When auth_type is missing from providers.json, use a safe default ('api') and log a warning. Don't fail silently.

**Warning signs:** Custom slots appearing unconfigured in the agent menu.

## State of the Art

### Current (broken) Approach

| Step | Current Behavior | Problem |
|------|------------------|---------|
| 1. User runs `/nf:mcp-setup` | Slots selected, keys configured, slots written to `~/.claude.json` mcpServers | No `auth_type` written to `~/.claude/nf.json` |
| 2. User runs `/nf:solve` | nf-prompt.js reads agentCfg, auth_type missing, defaults to 'api' | All slots classified as T2 (api), T1 always empty |
| 3. Quorum initialization | buildFalloverRule() called with T1=[], T2=[all slots] | FALLBACK-01 degrades to "skip it", loses multi-step dispatch |

### New (correct) Approach

| Step | Fixed Behavior | Benefit |
|------|---|---|
| 1. User runs `/nf:mcp-setup` | Slots selected → auth_type read from providers.json → written to `~/.claude/nf.json` agent_config | Single source of truth (providers.json) |
| 2. User runs `/nf:solve` | nf-prompt.js reads agentCfg with populated auth_type → T1=[codex-1, gemini-1, ...], T2=[claude-1..6] | Correct tiering, T1/T2 split matches subscription vs API |
| 3. Quorum initialization | buildFalloverRule() called with populated T1/T2 → emits full FALLBACK-01 3-step sequence | Priority fallback works, higher success rate |

### Backward Compatibility

**Existing setups (without auth_type):**
- Code defaults to 'api' when auth_type is missing → no breaking changes
- config-audit.cjs warns about the anti-pattern → user aware of degradation
- After this phase, setup writes auth_type → warning resolves on re-run

**Slots added in future versions:**
- If new provider added to providers.json without auth_type, setup skips it with warning
- No crash, no silent misconfiguration

## Open Questions

1. **Should `auth_type` be stored in providers.json or derived at runtime?**
   - Evidence: Current code (manage-agents-core.cjs, nf-prompt.js) reads from nf.json agent_config, not providers.json
   - Recommendation: Add `auth_type` field to providers.json schema, then copy to nf.json during setup. Keeps durable config in one place (providers.json) but respects agent-level customization (nf.json)

2. **What if a user adds a custom slot to ~/.claude.json that isn't in providers.json?**
   - Evidence: No enforcement of providers.json membership in current code
   - Recommendation: Default to `auth_type: 'api'`, log warning. Custom slots are user-managed; don't prevent setup.

3. **Should auth_type be user-editable in the re-run menu?**
   - Evidence: config-audit warns if auth_type is wrong, but provides no UI to fix it
   - Recommendation: Out of scope for this phase (v0.35-02). Future TUI enhancement (v0.35-04) can expose auth_type as editable field.

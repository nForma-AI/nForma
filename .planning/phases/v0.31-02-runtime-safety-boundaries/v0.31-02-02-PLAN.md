<objective>
Add per-slot latency budget support to the quorum dispatch pipeline. Users set `latency_budget_ms` in providers.json per slot; the value acts as a hard ceiling that takes priority over the --timeout CLI arg and the existing quorum_timeout_ms field. Slots exceeding their budget are killed and recorded as TIMEOUT in telemetry.

Purpose: Satisfies LTCY-01 -- granular control over slot performance expectations so slow slots do not block the entire quorum pipeline. Maintains EventualConsensus invariant because existing majority voting logic handles TIMEOUT/UNAVAIL slots.
Output: Modified call-quorum-slot.cjs with latency_budget_ms support, updated providers.json with example fields.
</objective>

  <action>
Modify the effective timeout computation in `main()` of bin/call-quorum-slot.cjs (around lines 468-471).

Current logic:
```javascript
const providerCap = provider.quorum_timeout_ms ?? null;
const effectiveTimeout = (timeoutMs !== null && providerCap !== null)
  ? Math.min(timeoutMs, providerCap)
  : (timeoutMs ?? providerCap ?? provider.timeout_ms ?? 30000);
```

New logic — `latency_budget_ms` is the user-configured hard ceiling and always wins when present:
```javascript
const latencyBudget = provider.latency_budget_ms ?? null;
const providerCap = provider.quorum_timeout_ms ?? null;
let effectiveTimeout;
if (latencyBudget !== null && latencyBudget > 0) {
  // LTCY-01: latency_budget_ms is the user-configured hard ceiling
  effectiveTimeout = latencyBudget;
} else if (timeoutMs !== null && providerCap !== null) {
  effectiveTimeout = Math.min(timeoutMs, providerCap);
} else {
  effectiveTimeout = timeoutMs ?? providerCap ?? provider.timeout_ms ?? 30000;
}
```

Also update the error handler (around line 519) to include the timeout source in telemetry for debugging. Add a stderr log when latency_budget_ms is the active ceiling:
```javascript
if (latencyBudget !== null && latencyBudget > 0) {
  process.stderr.write(`[call-quorum-slot] Using latency_budget_ms=${latencyBudget} for slot ${slot}\n`);
}
```

Do NOT change any other timeout or retry logic. This is a targeted modification to the effective timeout computation only.

# Phase v0.31-02: Runtime Safety Boundaries - Research

**Researched:** 2026-03-08
**Domain:** Circuit breaker persistence, quorum latency budgets, tool-access sandboxing
**Confidence:** HIGH

## Summary

This phase adds three runtime safety boundaries to the nForma hook system: (1) oscillation signature persistence so future sessions can preemptively detect recurring patterns, (2) per-slot latency budgets in providers.json that terminate slow slots mid-dispatch, and (3) restricted tool access for review-only quorum slots. All three are modifications to existing, well-understood code paths -- no new frameworks or external libraries are needed.

The circuit breaker already writes state to `.claude/circuit-breaker-state.json` and an oscillation log at `.planning/oscillation-log.json`. BRKR-01 extends this to write a structured evidence file in `.planning/formal/evidence/` that persists oscillation signatures across sessions. The latency budget (LTCY-01) plugs into the existing `quorum_timeout_ms` field in providers.json and the `call-quorum-slot.cjs` timeout mechanism. The restricted tool access (EXEC-01) leverages the `--allowedTools` flag already supported by the `ccr` CLI for `claude-*` slots and adds instruction-level restrictions for non-CCR slots (codex, gemini, opencode, copilot).

**Primary recommendation:** Implement as three independent workstreams (one per requirement), each modifying 2-3 files max, with existing test patterns for each hook.

### Pattern 2: Latency Budget Enforcement (LTCY-01)
**What:** Per-slot timeout value in providers.json, enforced by the subprocess dispatch
**When to use:** Every quorum slot dispatch via call-quorum-slot.cjs
**Example:**
```javascript
// In providers.json, per provider entry:
{
  "name": "claude-5",
  "latency_budget_ms": 15000,  // NEW FIELD
  "quorum_timeout_ms": 10000,  // Existing field (used as fallback)
  // ...
}

// In call-quorum-slot.cjs dispatch logic:
// Priority: latency_budget_ms > --timeout arg > quorum_timeout_ms > timeout_ms
const effectiveTimeout = provider.latency_budget_ms
  || timeoutMs
  || provider.quorum_timeout_ms
  || provider.timeout_ms
  || 30000;
```

### Pattern 3: Review-Only Tool Restriction (EXEC-01)
**What:** Restrict tool access for review-only quorum dispatches
**When to use:** When a slot is dispatched for verification or code review tasks
**Example:**
```javascript
// For ccr-based slots (claude-1 through claude-6):
// args_template already includes {prompt}, add --allowedTools before prompt
const reviewArgs = [
  ...baseArgs,
  '--allowedTools', 'Read,Grep,Glob,Bash(read-only)',
];

// For non-ccr slots (codex, gemini, opencode, copilot):
// Prepend instruction to the prompt (these CLIs don't support --allowedTools)
const reviewPrompt = `IMPORTANT: This is a READ-ONLY review task. You MUST NOT modify any files. ` +
  `Only use read operations (read files, search code, list files). ` +
  `Do NOT use Write, Edit, or destructive Bash commands.\n\n${originalPrompt}`;
```

### Pattern 4: Monitoring State Recovery (MonitoringReachable)
**What:** Circuit breaker returns to monitoring state after oscillation resolution
**When to use:** When Haiku convergence check confirms resolution, or manual reset
**Current mechanism:** Already implemented in PostToolUse handler of nf-circuit-breaker.js:
  - Haiku checks each Bash output for resolution signals
  - On YES verdict: clears state file, marks oscillation log as resolved
  - Manual: `npx nforma --reset-breaker` clears state
**New requirement:** Evidence file signature must also be marked `resolved_at` when breaker clears

### Anti-Patterns to Avoid
- **Blocking quorum on evidence file reads:** Evidence preemptive check must be fail-open. If the evidence file is missing or malformed, continue with normal detection (never block a tool call because evidence is unavailable).
- **Coupling latency_budget_ms to quorum_timeout_ms:** These are independent values. `latency_budget_ms` is the user-configured hard ceiling; `quorum_timeout_ms` is the existing per-provider default. Do not merge them into one field.
- **Relying solely on prompt-level restrictions for EXEC-01:** Prompt-level instructions can be ignored by the LLM. For ccr slots, always use `--allowedTools` CLI flag for structural enforcement. Prompt-level is a best-effort fallback for non-ccr CLIs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Subprocess timeout/kill | Custom signal handlers | Existing `killGroup()` in call-quorum-slot.cjs | Already handles SIGTERM/SIGKILL cascading, process group cleanup |
| Evidence file locking | flock/lockfile | Atomic writeFileSync with schema_version | Single-writer (circuit breaker hook), no concurrent writes |
| Tool access restriction | Custom hook interceptor | `--allowedTools` CLI flag for ccr slots | Structural enforcement at the Claude Code level, not interceptable by the LLM |
| Telemetry recording | New telemetry system | Existing `recordTelemetry()` in call-quorum-slot.cjs | Already writes structured JSONL with slot, latency_ms, error_type fields |

**Key insight:** Every new capability in this phase extends existing mechanisms. The circuit breaker already detects oscillation and writes state. The slot dispatch already has timeout logic. The ccr CLI already supports `--allowedTools`. The phase is additive modifications to proven code paths.

## Common Pitfalls

### Pitfall 1: Evidence File Growing Unbounded
**What goes wrong:** Every oscillation detection appends to the evidence file without cleanup, eventually consuming disk space or slowing reads.
**Why it happens:** No TTL or max-entry limit on persisted signatures.
**How to avoid:** Cap the signatures array at 50 entries. On write, sort by `last_seen` descending, truncate to 50. Signatures older than 30 days can be pruned on read.
**Warning signs:** Evidence file > 100KB.

### Pitfall 2: Latency Budget vs. Quorum Timeout Confusion
**What goes wrong:** `latency_budget_ms` and `quorum_timeout_ms` are both timeout values in providers.json. Users set one expecting both to apply, or set conflicting values.
**Why it happens:** Two timeout fields with similar semantics.
**How to avoid:** Document clearly: `latency_budget_ms` is the user-configurable hard cap (LTCY-01). `quorum_timeout_ms` is the provider-default timeout used when no budget is set. `latency_budget_ms` always wins when present.
**Warning signs:** Slot killed at unexpected time; user confusion about which timeout applies.

### Pitfall 3: Review-Only Restriction Bypassed by Non-CCR CLIs
**What goes wrong:** codex, gemini, opencode, and copilot CLIs don't support `--allowedTools`, so the restriction is prompt-level only.
**Why it happens:** These are third-party CLIs without tool-restriction flags.
**How to avoid:** Accept this as a known limitation. Document that structural enforcement only applies to ccr-based slots (claude-1 through claude-6). For non-ccr slots, prompt-level restriction is best-effort. The quorum vote still catches unauthorized modifications.
**Warning signs:** Non-ccr review slot writes files during a review-only task.

### Pitfall 4: Preemptive Detection False Positives
**What goes wrong:** A resolved oscillation signature still matches new work on the same files, preemptively triggering the breaker unnecessarily.
**Why it happens:** Matching only on file set hash without checking if the pattern is actually oscillating again.
**How to avoid:** Preemptive detection should WARN (stderr) but not BLOCK. Only the full run-group analysis (existing `detectOscillation()`) should activate the breaker. The evidence file provides context to the Haiku reviewer, not direct blocking.
**Warning signs:** Circuit breaker fires immediately on session start for files that were previously oscillating but are now being legitimately modified.

### Pitfall 5: Install Sync Forgotten
**What goes wrong:** Hook source files are edited in `hooks/` but not synced to `hooks/dist/` and then installed via `node bin/install.js --claude --global`.
**Why it happens:** Developer forgets the two-step sync process.
**How to avoid:** Every plan that edits hook source files MUST include an install sync task as a final step: `cp hooks/nf-circuit-breaker.js hooks/dist/ && node bin/install.js --claude --global`.
**Warning signs:** Tests pass on source but production hook behavior differs.

## Code Examples

### Writing Evidence File on Oscillation Detection
```javascript
// In nf-circuit-breaker.js, after oscillation is confirmed:
function writeEvidenceSignature(gitRoot, fileSet, fileSets, fileSetHash, patternHash) {
  try {
    const evidencePath = path.join(gitRoot, '.planning', 'formal', 'evidence', 'oscillation-signatures.json');
    let evidence = { schema_version: 1, signatures: [] };
    if (fs.existsSync(evidencePath)) {
      try { evidence = JSON.parse(fs.readFileSync(evidencePath, 'utf8')); } catch { /* reset */ }
    }
    if (!Array.isArray(evidence.signatures)) evidence.signatures = [];

    const existing = evidence.signatures.find(s => s.file_set_hash === fileSetHash);
    const now = new Date().toISOString();
    if (existing) {
      existing.last_seen = now;
      existing.alternation_count = (existing.alternation_count || 0) + 1;
    } else {
      evidence.signatures.push({
        id: `sig_${fileSetHash}`,
        file_set_hash: fileSetHash,
        pattern_hash: patternHash,
        files: fileSet.slice().sort(),
        alternation_count: 1,
        time_window: { first_seen: now, last_seen: now },
        resolved_at: null,
        resolved_by_commit: null,
        session_id: process.env.CLAUDE_SESSION_ID || null,
      });
    }

    // Cap at 50 entries, sorted by recency
    evidence.signatures.sort((a, b) =>
      new Date(b.time_window.last_seen) - new Date(a.time_window.last_seen));
    evidence.signatures = evidence.signatures.slice(0, 50);

    fs.mkdirSync(path.dirname(evidencePath), { recursive: true });
    fs.writeFileSync(evidencePath, JSON.stringify(evidence, null, 2), 'utf8');
  } catch (e) {
    process.stderr.write(`[nf] WARNING: Could not write evidence: ${e.message}\n`);
    // Fail-open: evidence write failure never blocks tool calls
  }
}
```

### Reading Evidence for Preemptive Detection
```javascript
// In nf-circuit-breaker.js, early in PreToolUse handler:
function checkPreemptiveEvidence(gitRoot, fileSets) {
  try {
    const evidencePath = path.join(gitRoot, '.planning', 'formal', 'evidence', 'oscillation-signatures.json');
    if (!fs.existsSync(evidencePath)) return null;
    const evidence = JSON.parse(fs.readFileSync(evidencePath, 'utf8'));
    if (!Array.isArray(evidence.signatures)) return null;

    // Check current file sets against known unresolved signatures
    for (const fileSetEntry of fileSets) {
      const hash = makeFileSetHash(fileSetEntry);
      const match = evidence.signatures.find(s =>
        s.file_set_hash === hash && !s.resolved_at
      );
      if (match) return match; // Known unresolved oscillation pattern
    }
    return null;
  } catch {
    return null; // Fail-open
  }
}
```

## Open Questions

1. **Should preemptive evidence detection BLOCK or WARN?**
   - What we know: The existing breaker blocks after full run-group analysis. Preemptive detection based on file set hash alone has higher false-positive risk.
   - What's unclear: Whether users prefer aggressive preemptive blocking or just warnings.
   - Recommendation: WARN only (stderr message + Haiku context). Let full `detectOscillation()` handle blocking. This avoids false positives while still providing useful context.

2. **How to signal review-only mode from nf-prompt.js to the slot worker?**
   - What we know: The slot worker receives arguments via prompt text (slot, round, mode, etc.). Mode B is already used for "review an artifact" dispatches.
   - What's unclear: Whether Mode B is always review-only, or if a separate `review_only: true` argument is needed.
   - Recommendation: Add explicit `review_only: true` argument to slot worker dispatch. Mode B is strongly correlated but not identical to review-only (Mode B could theoretically request improvements that involve writes).

3. **What about codex/gemini/copilot/opencode not supporting --allowedTools?**
   - What we know: These are third-party CLIs. Only ccr (Claude Code Router) supports `--allowedTools`.
   - What's unclear: Whether prompt-level restriction is sufficient for compliance.
   - Recommendation: Accept prompt-level as best-effort for non-ccr slots. Document the limitation. The quorum vote mechanism already catches unauthorized modifications (a review slot that writes files would be flagged by other quorum participants).

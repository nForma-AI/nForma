<objective>
Implement file-based execution progress tracking so sub-task state survives compaction.

Purpose: Currently, the agent tracks execution state purely in conversation context. When compaction fires mid-plan, the agent loses which tasks it has completed and either restarts from the beginning (wasting tokens) or guesses incorrectly. This plan creates a JSON progress file that records task completion state, extends nf-precompact.js to inject that state into compaction continuation context, adds termination guards (iteration cap + stuck detection), and wires progress tracking into the execute-plan workflow.
Output: `bin/execution-progress.cjs` (progress module), `bin/execution-progress.test.cjs` (tests), updated `hooks/nf-precompact.js` (progress injection), updated `core/workflows/execute-plan.md` (progress calls at task boundaries), `.gitignore` update.
</objective>

  <action>
Create `bin/execution-progress.cjs` as a CommonJS module with `'use strict'` at top. Zero new dependencies -- uses only `fs` and `path` built-ins.

**Constants:**
- `PROGRESS_FILE = '.planning/execution-progress.json'`
- `DEFAULT_MAX_ITERATIONS = 5`
- `STUCK_THRESHOLD = 3` (resume attempts on same task before declaring stuck)

**getProgressPath(cwd) function:**
Returns `path.join(cwd, PROGRESS_FILE)`.

**initProgress(cwd, { planFile, totalTasks, taskNames }) function:**
Creates `.planning/execution-progress.json` with schema:
```json
{
  "version": 1,
  "phase": "<extracted from planFile via regex>",
  "plan": "<extracted from planFile via regex>",
  "plan_file": "<planFile>",
  "total_tasks": totalTasks,
  "status": "in_progress",
  "started_at": "<ISO timestamp>",
  "updated_at": "<ISO timestamp>",
  "iteration_count": 0,
  "max_iterations": 5,
  "last_known_commit": null,
  "tasks": [{ number, name, status: "pending", commit_hash: null, completed_at: null, resume_attempts: 0 }]
}
```
Always overwrites any existing file (idempotent). Creates `.planning/` directory if missing via `fs.mkdirSync({ recursive: true })`. Uses `fs.writeFileSync` for atomic single-writer writes.

**completeTask(cwd, { taskNumber, commitHash }) function:**
Reads progress file, finds task by number, sets `status: "complete"`, `commit_hash`, `completed_at` (ISO timestamp). Updates `last_known_commit` and `updated_at` on the root object. If ALL tasks are `complete` or `skipped`, sets root `status` to `"complete"`. Returns updated progress object or null if file/task not found.

**getStatus(cwd) function:**
Read-only. Returns parsed JSON from progress file, or `{ status: "no_progress_file" }` if file does not exist. NEVER modifies the file.

**incrementIteration(cwd) function:**
Called ONLY by nf-precompact.js on compaction. Reads progress file, increments `iteration_count`, sets `updated_at`. Then checks:
1. If `iteration_count >= max_iterations`: set `status: "failed"`, `failure_reason: "iteration_cap_exhausted"`.
2. Find any task with `status: "in_progress"`. If found, increment its `resume_attempts`. If `resume_attempts >= STUCK_THRESHOLD` (3): set `status: "failed"`, `failure_reason: "stuck_on_task"`, `stuck_task: <task number>`.
Writes file back. Returns updated progress object or null if file not found.

**clearProgress(cwd) function:**
Removes `.planning/execution-progress.json` if it exists. No-op if already absent.

**CLI interface:**
When run as a script (`require.main === module`), parse `process.argv` for subcommands:
- `init --plan <planFile> --tasks <count> --names <comma-separated>`: calls `initProgress`, prints JSON result to stdout.
- `complete-task --number <N> --commit <hash>`: calls `completeTask`, prints JSON result.
- `get-status`: calls `getStatus`, prints JSON result.
- `increment-iteration`: calls `incrementIteration`, prints JSON result.
- `clear`: calls `clearProgress`, prints `{"cleared": true}`.

All CLI commands use `process.cwd()` as cwd. Wrap everything in try/catch with fail-open (`process.exit(0)` on error, error message to stderr).

Export: `{ getProgressPath, initProgress, completeTask, getStatus, incrementIteration, clearProgress, PROGRESS_FILE, DEFAULT_MAX_ITERATIONS, STUCK_THRESHOLD }`

**Tests** in `bin/execution-progress.test.cjs` using vitest with `describe`/`it`/`expect`:
- Use `os.tmpdir()` + `fs.mkdtempSync` for isolated temp directories in `beforeEach`; `fs.rmSync` in `afterEach`.
- `initProgress` creates valid JSON file matching schema (all fields present, correct types)
- `initProgress` extracts phase/plan from planFile string (e.g., `"v0.30-02-01-PLAN.md"` -> phase `"v0.30-02"`, plan `"01"`)
- `initProgress` overwrites stale progress from a different plan (idempotent)
- `completeTask` marks task complete with commit hash and timestamp
- `completeTask` sets root status to `"complete"` when ALL tasks are done
- `completeTask` returns null when file does not exist
- `completeTask` returns null when task number not found
- `getStatus` returns `{ status: "no_progress_file" }` when file missing
- `getStatus` returns full progress object when file exists
- `getStatus` does NOT modify the file (read file before and after, assert identical)
- `incrementIteration` increments counter and writes updated file
- `incrementIteration` sets status to `"failed"` with reason `"iteration_cap_exhausted"` when counter reaches max_iterations
- `incrementIteration` detects stuck-on-task: same task `in_progress` across 3 increments -> status `"failed"`, reason `"stuck_on_task"`
- `incrementIteration` returns null when file does not exist
- `clearProgress` removes the file
- `clearProgress` is a no-op when file does not exist (no throw)

**Step 1: Add require at top (fail-open):**

```javascript
  try { return require(path.join(__dirname, '..', 'bin', 'execution-progress.cjs')); }
  catch { return null; }

```

**Step 2: Add readExecutionProgress(cwd) function:**
- If `executionProgress` is null (module not found), return null.
- Call `executionProgress.getStatus(cwd)`. If result has `status === "no_progress_file"`, return null.
- If result has `status !== "in_progress"`, return null (plan already complete or failed -- no injection needed).
- Call `executionProgress.incrementIteration(cwd)` to bump the counter and check caps/stuck detection.
- Return the updated progress object from incrementIteration.

**Step 3: Add formatProgressInjection(progress) function:**
- If progress is null, return null.
- Build a string with these lines:
  ```
  ## Execution Progress (auto-injected at compaction)

  Plan: {progress.plan_file}
  Status: {progress.status} ({completed_count}/{progress.total_tasks} tasks complete, iteration {progress.iteration_count} of {progress.max_iterations})
  ```
- If `progress.status === "failed"`:
  Append: `\nEXECUTION FAILED: {progress.failure_reason}\nReport this failure to the user. Do NOT continue execution.`
  Return immediately.
- If completed tasks > 5: summarize as `Tasks 1-{N} complete. Resume at Task {next.number}.`
- If completed tasks 1-5: list each as `  [x] {name} (commit {hash})`
- Find next task (first `pending` or `in_progress`). If found:
  ```
  Resume at:
    [ ] {next.name}

  IMPORTANT: Read {progress.plan_file} and continue from Task {next.number}.
  Do NOT re-execute Tasks 1-{completed_count} -- they are already committed.
  ```
- Cap total output to 3200 characters (800 token estimate). If longer, truncate completed task list and keep the resume instruction.

**Step 4: Integrate into the main stdin handler:**
After building `additionalContext` from STATE.md position and pending tasks (after the `lines.push('## Resume Instructions')` block), add:
```javascript
// Execution progress injection (VERF-01)
const execProgress = readExecutionProgress(cwd);
const progressBlock = formatProgressInjection(execProgress);
if (progressBlock) {
  lines.push('');
  lines.push(progressBlock);
}
```

**Step 5: Export new functions for testing:**
Add to the existing `module.exports` block:
```javascript
module.exports.readExecutionProgress = readExecutionProgress;
module.exports.formatProgressInjection = formatProgressInjection;
```

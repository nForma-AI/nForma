## Summary

Phase v0.15-02 adds a content-length safety gate to the `regenerateState` repair action inside `cmdValidateHealth`. Currently, when `--repair` is passed and `regenerateState` is in the repairs queue, the code immediately creates a timestamped backup and then overwrites STATE.md with a 15-line blank template — regardless of how large the existing file is. This caused a real data loss incident where the full QGSD STATE.md (160 lines) was silently replaced.

The fix requires three coordinated changes: (1) parse a new `--force` flag in the `validate health` argument handler, (2) pass `force` into `cmdValidateHealth` options, and (3) inside the `regenerateState` case, read the existing STATE.md line count before overwriting — if the count exceeds 50 and `force` is false, print a warning and skip the overwrite. If `force` is true, or if the file is 50 lines or fewer, proceed normally.

The change is small, targeted, and isolated to three call sites. No new dependencies are required. The existing test framework (`node:test` + `node:assert`) with its `createTempProject()` / `runGsdTools()` helpers handles all test scenarios cleanly. The Nyquist pattern requires failing tests (Wave 0) before implementation begins.

**Primary recommendation:** Extend `cmdValidateHealth` with a `force` option check inside the `regenerateState` case. Parse `--force` in the `validate health` arg handler. Write three TDD tests (wave 0 must be red before the fix, green after) and sync the install.

### Anti-Patterns to Avoid

- **Modifying backupState logic:** The backup creation path is correct and must remain unchanged for the `--force` path.
- **Using `process.stderr.write` or `console.warn`:** The rest of the function outputs structured JSON via `output()`. Stay consistent — put the warning inside `repairs_performed`.
- **Changing the threshold to a runtime flag:** 50 lines is the defined threshold (per SAFE-01). Hardcode it as a named constant inside the case block.
- **Blocking non-STATE repairs:** The safety gate must only affect the `regenerateState` repair. `createConfig` and `resetConfig` are unaffected and should still proceed without `--force`.
- **Exiting non-zero on the skipped case:** `cmdValidateHealth` always exits 0 (repairs_performed captures the result). The skipped case must not change the exit code.

## Common Pitfalls

### Pitfall 1: Both `--repair` and `--force` needed simultaneously
**What goes wrong:** `--force` alone should not trigger repair. `--repair --force` is the correct combined form. If `--force` is parsed globally and interpreted as "run repair + force", callers get unexpected behavior.
**Why it happens:** Misreading the CLI design — `--force` is only meaningful when `--repair` is also present.
**How to avoid:** The `--force` flag only affects behavior inside `cmdValidateHealth` when `repairs.includes('regenerateState')` is true. `--repair` is still required to activate the repair loop.

### Pitfall 2: `--force` accidentally applies to config repairs
**What goes wrong:** `options.force` is checked inside the `regenerateState` case but also bleeds into `createConfig`/`resetConfig` cases — changing behavior for those repairs.
**Why it happens:** Putting the force check at the outer `if (options.repair && repairs.length > 0)` level instead of inside the `regenerateState` case.
**How to avoid:** The force check belongs inside the `case 'regenerateState':` block only.

### Pitfall 3: `split('\n').length` off-by-one
**What goes wrong:** A 51-line file that ends in a newline produces `split('\n').length === 52` (empty string after final `\n`). A 50-line file with final newline produces 51.
**Why it happens:** `\n`.split gives one extra empty element for trailing newlines.
**How to avoid:** The threshold is >50 (strictly greater than), so the off-by-one does not matter for the specified behavior: "more than 50 lines." A 50-line file with trailing newline gives `.length === 51`, which is `> 50`. If this is undesirable, use `content.split('\n').filter(l => l.length > 0).length` for a non-empty-line count. However, the requirement says "more than 50 lines" without specifying empty-line exclusion — use simple `split('\n').length` and verify behavior against the test cases.
**Recommendation:** Use `content.split('\n').length` (simple). Write the test with a 51-character-line file (no trailing newline) for the "above threshold" case to avoid ambiguity. The current QGSD STATE.md has 160 lines — well above 50 regardless of method.

### Pitfall 4: Install sync omitted
**What goes wrong:** Tests pass against `qgsd-core/bin/gsd-tools.cjs`, but `node ~/.claude/qgsd/bin/gsd-tools.cjs validate health --repair` on the real QGSD STATE.md still silently overwrites it.
**Why it happens:** The installed copy at `~/.claude/qgsd/bin/gsd-tools.cjs` is not updated until `node bin/install.js --claude --global` is run.
**How to avoid:** Always include install sync as the final task before the smoke test.

### Pitfall 5: Test scaffold tests pass before implementation (wrong direction)
**What goes wrong:** Wave 0 tests are written but immediately pass because the success condition is "no overwrite" — and the tests don't actually trigger overwrite behavior.
**Why it happens:** Test scenario does not reproduce the failing condition: the test must create a STATE.md > 50 lines AND pass `--repair` AND check that the file was NOT overwritten.
**How to avoid:** In each Wave 0 test, verify the pre-condition (file content unchanged) by reading back the file after the command and asserting the original content is still there.

### Triggering `regenerateState` reliably in tests

The `regenerateState` repair is queued when either:
1. `E004` fires: STATE.md does not exist at all
2. `W002` fires: STATE.md references a phase that does not exist on disk (and the phases dir has content)

For Wave 0 tests, use the **E004 path** (no STATE.md) to trigger `regenerateState` when verifying that the guard blocks the overwrite, then write the rich STATE.md back before running the command. But this is awkward.

**Simpler approach for tests:** Use the **W002 path** — write a STATE.md that mentions a phantom phase, while only a different phase dir exists on disk. Example:

```javascript
// STATE.md mentions "Phase v0.phantom-01" but only v0.99-01-test dir exists on disk
const richState = '# State\n\nPhase v0.phantom-01\n' +
  Array.from({ length: 52 }, (_, i) => `Padding line ${i}`).join('\n');
fs.writeFileSync(statePath, richState);
// phases/ only has v0.99-01-test (not v0.phantom-01) → W002 fires → regenerateState queued
```

**Alternative (cleanest):** Delete STATE.md after writing it — the `E004` path unconditionally queues `regenerateState`. Then check that the file was NOT recreated (since the file didn't exist, the guard has nothing to count — need to write STATE.md BEFORE running the tool, so the test must write the file, then let the tool see it via the W002 path).

**Recommended test setup** (avoids fragile scenarios):
- Write a STATE.md with 55 lines that references `Phase v0.phantom-01` (phantom phase)
- Create only `v0.99-01-test` on disk
- The phase reference mismatch triggers W002 → `regenerateState` queued
- Tool reads STATE.md line count, finds 55 > 50, skips without `--force`

## Open Questions

1. **Should the warning also appear on stderr or only in JSON?**
   - What we know: The rest of `gsd-tools.cjs` outputs only structured JSON via `output()`. There is no `console.warn` or `process.stderr.write` anywhere in the validate paths.
   - What's unclear: The requirement says "prints a warning showing the current line count." This could mean stdout JSON or stderr text.
   - Recommendation: Keep the warning in the JSON `repairs_performed` array (consistent with file). The `reason` field contains the line count. If `/qgsd:health` needs to surface this as human-readable text, the workflow layer can read `repairs_performed[*].reason`. Do NOT add `console.warn` unless the planner decides the requirement explicitly demands stderr output.

2. **Does E004 (STATE.md missing entirely) need the same guard?**
   - What we know: E004 fires when `!fs.existsSync(statePath)`. In this case there is no existing content to protect. The guard checks `fs.existsSync(statePath)` before reading line count.
   - What's unclear: The requirement only mentions "overwriting a STATE.md with more than 50 lines." Missing STATE.md has 0 lines.
   - Recommendation: The guard naturally handles this — `if (fs.existsSync(statePath))` is false, so no line count check occurs, and the code proceeds to create a new STATE.md. This is correct behavior and requires no special handling.

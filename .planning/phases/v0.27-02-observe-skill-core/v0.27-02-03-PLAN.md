**bin/observe-debt-writer.cjs** — exporting `writeObservationsToDebt(observations, ledgerPath)`:

1. Import `readDebtLedger`, `writeDebtLedger` from `./debt-ledger.cjs` (v0.27-01)
2. Import `fingerprintIssue` from `./fingerprint-issue.cjs` and `fingerprintDrift` from `./fingerprint-drift.cjs` (v0.27-01)
3. Import `validateDebtEntry` from `./validate-debt-entry.cjs` (v0.27-01)
4. Read current ledger via `readDebtLedger(ledgerPath || '.formal/debt.json')`
5. For each observation in the observations array:
   a. Compute fingerprint: if `issue_type === "drift"`, call `fingerprintDrift({ formal_parameter_key: obs.formal_parameter_key })`. Otherwise call `fingerprintIssue({ exception_type: obs.exception_type, function_name: obs.function_name, message: obs.title })`
   b. Search existing ledger entries for matching fingerprint
   c. If match found: increment `occurrences`, update `last_seen` to current ISO timestamp, append to `source_entries` array
   d. If no match: create new debt entry with: `id` (crypto.randomUUID()), `fingerprint`, `title: obs.title`, `occurrences: 1`, `first_seen` and `last_seen` (current ISO), `environments: ["production"]`, `status: "open"`, `formal_ref: obs.formal_parameter_key || null`, `source_entries: [{ source_type: obs.source_type, source_id: obs.id, observed_at: obs.created_at || new Date().toISOString() }]`
   e. Validate the entry with `validateDebtEntry` before adding. If validation fails, log warning and skip (fail-open).
6. Write updated ledger via `writeDebtLedger(ledgerPath, ledger)`
7. Return `{ written: count, updated: count, errors: count }`

CRITICAL per Pitfall 3 in research: NEVER compute fingerprints inline. ALWAYS use the imported v0.27-01 functions.

**bin/observe-render.cjs** — exporting `renderObserveOutput(results)`:

1. Accept array of handler results (from dispatchAll, after MCP bridge resolution — all results are standard schema at this point)
2. Separate successful results (status: "ok") from errors (status: "error")
3. Collect all issues from successful results
4. Split into two groups by `issue_type`: "issue" items and "drift" items
5. Sort each group by severity (error > bug > warning > info), then by age (newest first)
6. Render header: `QGSD > OBSERVE: N issues, M drifts across K source(s)`
7. Render ISSUES table with columns: #, Title, Source, Sev, Age (per research dual-table format)
8. Render DRIFTS table (if any) with columns: #, Parameter, Formal, Actual, Sev
9. Render errors section at bottom: `Errors from sources (N failed, did not block others):` with each error listed
10. Return the complete output string (don't console.log — let caller decide output destination)

Also export `classifySeverity(severity)` helper that maps severity strings to sort order and `formatAge(isoDate)` helper.

Create `commands/qgsd/observe.md` — the command spec that replaces `/qgsd:triage` with `/qgsd:observe`. Adapt directly from `commands/qgsd/triage.md` structure (same YAML frontmatter format, same allowed-tools).

**YAML frontmatter:**
```yaml
name: qgsd:observe
description: Fetch issues and drifts from configured sources, render dual-table output, and write to debt ledger. Replaces /qgsd:triage.
argument-hint: "[--source github|sentry|sentry-feedback|bash] [--since 24h|7d] [--limit N]"
allowed-tools:
  - Read
  - Bash
  - Glob
  - Task
  - AskUserQuestion
```

**Process steps (adapted from triage.md, adding observe-specific behavior):**

**Step 1: Parse arguments** — Same as triage (--source, --since, --limit)

**Step 2: Load source configuration** — Use `loadObserveConfig()` from `bin/observe-config.cjs`. Falls back to triage-sources.md. Apply source filter if --source is set.

**Step 3: Register handlers** — Import handler functions from `bin/observe-handlers.cjs`. Register each with `registerHandler()` from `bin/observe-registry.cjs`: github, sentry, sentry-feedback, bash.

**Step 4: Dispatch parallel fetch** — Call `dispatchAll(sources, { sinceOverride, limitOverride })` from `bin/observe-registry.cjs`. This dispatches ALL sources uniformly through the registry with per-source timeout. No special-casing per source type at this stage.

**Step 4b: MCP bridge for pending_mcp results** — After `dispatchAll` returns, iterate over results. For any result with `status: "pending_mcp"`:
1. Read the `_mcp_instruction` object from the result: `{ type: "mcp", tool, params, mapper }`
2. Execute the MCP tool call directly (the observe command runs in Claude context where MCP tools are available): call the tool named in `_mcp_instruction.tool` with `_mcp_instruction.params`
3. Import the mapper function from `bin/observe-handlers.cjs` (the mapper name is in `_mcp_instruction.mapper`, e.g., `mapSentryIssuesToSchema` or `mapSentryFeedbackToSchema`)
4. Call the mapper with the MCP result and the original sourceConfig to produce a standard schema result `{ source_label, source_type, status: "ok", issues: [...] }`
5. Replace the `pending_mcp` result in the results array with the mapped standard schema result
6. If the MCP call fails, replace with `{ source_label, source_type, status: "error", error: "MCP call failed: {message}", issues: [] }`

This pattern keeps handlers as pure testable CJS functions while the observe command (running in Claude context) handles MCP execution. The registry's `dispatchAll` works uniformly — the MCP bridge is a post-processing step in the orchestrator.

**Step 5: Collect and render** — Call `renderObserveOutput(results)` from `bin/observe-render.cjs`. At this point ALL results are standard schema (pending_mcp results were resolved in Step 4b). Renders dual table: Issues first, Drifts second, errors at bottom.

**Step 6a: Write to debt ledger** — Call `writeObservationsToDebt(allIssues, '.formal/debt.json')` from `bin/observe-debt-writer.cjs`. This upserts all observations by fingerprint. Display: `Wrote N new, updated M existing debt entries.`

**Step 6b: Show debt summary** — Display count of open/acknowledged/resolving/resolved entries in ledger.

**Step 7: Route to action** — Same as triage Step 7: prompt user to select issue number, suggest /qgsd:debug for errors/bugs, /qgsd:quick for warnings/info. Add option to acknowledge selected issues (sets status to "acknowledged" via debt state machine).

Ensure the command spec references the utility modules by path so executors know which files to import. Include the exact module paths in code blocks within each step. Document the MCP bridge pattern clearly in Step 4b so future handlers know the convention.

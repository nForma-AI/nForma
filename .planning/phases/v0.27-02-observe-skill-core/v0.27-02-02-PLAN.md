Create `bin/observe-handlers.cjs` exporting four handler functions plus two mapper helpers.

**CRITICAL DESIGN RULE — Uniform Return Type:**
ALL handlers return the SAME schema: `{ source_label, source_type, status: "ok"|"error"|"pending_mcp", issues: [...], error?: string, _mcp_instruction?: object }`

- GitHub and bash handlers do their work directly (CLI calls) and return `status: "ok"` with populated `issues[]`.
- Sentry and sentry-feedback handlers are PURE FUNCTIONS that cannot call MCP tools (they are CJS modules, not Claude prompts). They return `status: "pending_mcp"` with `issues: []` and an `_mcp_instruction` object describing what MCP tool to call. The observe command (Plan 03) detects `pending_mcp`, executes the MCP call, then uses exported mapper functions to convert raw MCP results to standard schema.
- On failure, ALL handlers return `status: "error"` (never throw).

Each handler accepts `(sourceConfig, options)` where options has `{ sinceOverride, limitOverride }`.
Each issue in issues[] has: `id, title, severity, url, age, created_at, meta, source_type, issue_type`.

**handleGitHub(sourceConfig, options):**
- Use `child_process.execFileSync` (NOT exec/execSync with string interpolation — shell injection risk). Build args array: `['issue', 'list', '--repo', repo, '--state', state, '--limit', limit, '--json', 'number,title,url,labels,createdAt,assignees']`. Add `--label` args for each label.
- Parses JSON output, applies `since` filter (from sourceConfig.filter.since or options.sinceOverride)
- Maps to standard schema: id=`gh-{number}`, severity from label matching (bug|error|critical|warning|enhancement|info, default info)
- Detects repo from `git remote get-url origin` if sourceConfig.repo is not set
- Returns `status: "ok"` with populated issues[], or `status: "error"` on failure

**handleSentry(sourceConfig, options):**
- Returns `status: "pending_mcp"` with `issues: []` and `_mcp_instruction`:
  ```js
  {
    type: "mcp",
    tool: "list_project_issues",
    params: { organization_slug, project_slug, query: `is:unresolved firstSeen:>${since}` },
    mapper: "mapSentryIssuesToSchema"  // name of exported mapper function
  }
  ```
- The `_mcp_instruction` tells the observe command WHAT to call and HOW to map the result.
- On failure constructing params, returns `status: "error"`.

**Also export `mapSentryIssuesToSchema(mcpResult, sourceConfig)`:**
- Takes raw MCP tool response and maps each issue to standard schema
- id=`sentry-{id}`, severity from level mapping (fatal->error, error->error, warning->warning, info->info)
- Returns `{ source_label, source_type, status: "ok", issues: [...], error: undefined }`

**handleSentryFeedback(sourceConfig, options):**
- Returns `status: "pending_mcp"` with `_mcp_instruction`:
  ```js
  {
    type: "mcp",
    tool: "list_user_feedback",
    params: { organization_slug, project_slug },
    mapper: "mapSentryFeedbackToSchema"
  }
  ```
- On failure constructing params, returns `status: "error"`.

**Also export `mapSentryFeedbackToSchema(mcpResult, sourceConfig)`:**
- Maps each feedback: id=`feedback-{id}`, title=`[Feedback] {comment truncated to 80 chars}`, severity=info
- Returns standard schema object

**handleBash(sourceConfig, options):**
- Use `child_process.execFileSync` with `['sh', '-c', sourceConfig.command]` for shell features. The command string comes from the user's own config file (trusted input from observe-sources.md).
- If parser="json", parses output as JSON array of `{title, url?, severity?}` objects
- If parser="lines" (default), splits output into non-empty lines, each becomes an issue with severity=info
- Maps to standard schema: id=`bash-{index}`
- Returns `status: "ok"` with populated issues[], or `status: "error"` on failure (non-zero exit)

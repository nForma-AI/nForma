  <action>
Create `bin/observe-handlers.cjs` exporting four handler functions. Each handler MUST:
- Accept `(sourceConfig, options)` where options has `{ sinceOverride, limitOverride }`
- Return `{ source_label, source_type, status: "ok"|"error", issues: [...], error?: string }`
- NEVER throw — wrap all logic in try-catch, return `status: "error"` on failure (fail-open per OBS-08)
- Each issue in issues[] has: `id, title, severity, url, age, created_at, meta, source_type, issue_type`

**handleGitHub(sourceConfig, options):**
- Use `execFileNoThrow` pattern from `src/utils/execFileNoThrow.ts` or `child_process.execFileSync` (NOT exec/execSync with string interpolation — shell injection risk). Build args array: `['issue', 'list', '--repo', repo, '--state', state, '--limit', limit, '--json', 'number,title,url,labels,createdAt,assignees']`. Add `--label` args for each label.
- Parses JSON output, applies `since` filter (from sourceConfig.filter.since or options.sinceOverride)
- Maps to standard schema: id=`gh-{number}`, severity from label matching (bug|error|critical|warning|enhancement|info, default info)
- Detects repo from `git remote get-url origin` if sourceConfig.repo is not set
- On failure, returns status: "error" with error message

**handleSentry(sourceConfig, options):**
- This handler generates INSTRUCTIONS for a Task subagent (since Sentry uses MCP tools that must run in a subagent context). Returns a structured object with `instructions` field that the observe command will pass to Task().
- Alternative: if MCP tools are directly available, call them. If not, return instructions for subagent dispatch.
- For testability, the handler function itself constructs the MCP tool call parameters and returns them. The actual MCP call happens in the observe command's dispatch step.
- Maps Sentry issue levels: fatal->error, error->error, warning->warning, info->info
- On failure, returns status: "error"

**handleSentryFeedback(sourceConfig, options):**
- Similar to Sentry handler — prepares MCP tool call parameters for `list_user_feedback`
- Maps each feedback: id=`feedback-{id}`, title=`[Feedback] {comment truncated to 80 chars}`, severity=info
- On failure, returns status: "error"

**handleBash(sourceConfig, options):**
- Use `child_process.execFileSync` with `['sh', '-c', sourceConfig.command]` for shell features, or `execFileSync` directly for simple commands. The command string comes from the user's own config file (trusted input from observe-sources.md), not from external user input.
- If parser="json", parses output as JSON array of `{title, url?, severity?}` objects
- If parser="lines" (default), splits output into non-empty lines, each becomes an issue with severity=info
- Maps to standard schema: id=`bash-{index}`
- On failure (non-zero exit), returns status: "error" with stderr

Create `bin/observe-handlers.test.cjs` with node:test covering:

- handleBash: mock with JSON output, verify JSON parsing
- handleBash: mock failure, verify status: "error"
- All handlers: verify standard schema fields present in every returned issue

For mocking: use node:test's `mock` API to stub child_process functions. Alternatively, inject the exec function as a dependency for testability.

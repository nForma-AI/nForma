# Phase v0.20-03: Planning Gate — Research

**Researched:** 2026-02-28
**Domain:** Formal verification integration with planning workflow, fail-open error handling, NDJSON result synthesis
**Confidence:** HIGH

## Summary

Phase v0.20-03 implements a planning-time formal verification gate that makes TLA+ results visible to the planner before quorum, without blocking plan creation when formal verification is flaky or unavailable. The phase transforms a background verification step into an **active planning input** that surfaces concrete verification failures as hypotheses the planner should address.

The gate is **fail-open by design**: TLC failures surface as warnings, not blockers. This prevents formal verification flakiness (TLC timeouts, intermittent crashes) from breaking the planning workflow. The planner can address flagged hypotheses or proceed with planning even if FV is unavailable.

**Core workflow change:** `plan-phase.md` adds a pre-quorum formal verification step that:
1. Runs `run-formal-verify --only=tla` (TLA+ checks only, fast ~60s)
2. Reads `check-results.ndjson` and extracts `result=fail` entries
3. Formats failures as hypotheses in structured text for planner consumption
4. If TLC is unavailable (non-zero exit, no NDJSON), logs a warning and proceeds to quorum

**Primary recommendation:** Add a pre-quorum FV step to `plan-phase.md` (step 8.3) that:
- Executes `run-formal-verify --only=tla` (synchronously, failure ignored)
- Reads `check-results.ndjson` with two-pass parsing (v1 5-field lines from prior runs + v2.1 fields from current run)
- Extracts entries with `result=fail` and formats them as a structured hypothesis block
- Injects the hypothesis block into the planner's additional context via environment variable for injection into quorum context

This integrates v0.20-01 (enriched schema) and v0.20-02 (liveness lint infrastructure) into the planning decision loop.

### Formal Verification Gate Implementation Pattern

**Where:** New step 8.3 in `/Users/jonathanborduas/.claude/qgsd/workflows/plan-phase.md` (or installed copy at `~/.claude/qgsd/workflows/plan-phase.md`)

**What it does:**
```javascript
1. Execute formal verify (child process):
   spawnSync('node', ['bin/run-formal-verify.cjs', '--only=tla'])
   → logs exit code, does NOT throw if exit !== 0 (fail-open)

2. Read check-results.ndjson with dual-format parsing:
   - Line format v1: { tool, formalism, result, timestamp, metadata }
   - Line format v2.1: { tool, surface, check_id, property, result, summary, ... }
   - Parse both; extract result=fail entries from both

3. Format failures as hypotheses:
   For each fail entry:
     - Extract check_id (v2.1) or generate from tool+formalism (v1)
     - Extract summary (v2.1) or metadata.reason (v1)
     - Extract property (v2.1) or "(no property)" (v1)
     - Build markdown hypothesis block

4. Set environment variable FV_HYPOTHESES for step 8.5 quorum context

5. Return status (via FV_STATUS env var):
   - SUCCESS: 1+ hypotheses found and injected
   - SUCCESS: 0 hypotheses (no FV failures; planning proceeds cleanly)
   - SUCCESS: TLC unavailable (exit !== 0, no NDJSON; warning logged, no hypotheses)
   Always exit 0 (fail-open)
```

### NDJSON Backwards Compatibility (v1 + v2.1 parsing)

**Scenario:** check-results.ndjson contains both formats:
- Older runs (v0.20-01 pre-completion or earlier): 5-field v1 format
- Current run (v0.20-01 complete): 10+ field v2.1 format
- Parser must handle both in a single file

**Pattern:**
```javascript
const parseCheckResultLine = (line) => {
  const record = JSON.parse(line);

  // v2.1 fields (present if written by writeCheckResult v2.1)
  const check_id = record.check_id;
  const summary = record.summary;
  const property = record.property;
  const surface = record.surface;

  // v1 fallback (for older NDJSON entries)
  const tool = record.tool;  // present in both v1 + v2.1
  const formalism = record.formalism;
  const result = record.result;

  return {
    check_id: check_id || `${tool}:${formalism}`,  // synthetic ID if missing
    surface: surface || tool,  // synthetic surface if missing
    result,
    property: property || '(unknown)',  // fallback for v1
    summary: summary || record.metadata?.reason || '(no summary)',  // fallback for v1
  };
};
```

### NDJSON Reading Robustness

**Pitfalls to avoid:**
1. **Partial NDJSON writes:** If TLC crashes mid-write, last line may be truncated. Parser should skip silently (not throw).
2. **Missing file:** If run-formal-verify fails completely, no NDJSON written. Parser should check file existence first.
3. **Empty result set:** If all TLC runs pass, no fail entries. Parser should handle empty set gracefully (0 hypotheses).

**Robustness pattern:**
```javascript
const parseFVResults = (ndjsonPath) => {
  const failResults = [];

  // Check file exists; if not, return empty (FV unavailable)
  if (!fs.existsSync(ndjsonPath)) {
    return failResults;  // FV did not run; no hypotheses
  }

  const lines = fs.readFileSync(ndjsonPath, 'utf-8').split('\n');
  for (const line of lines) {
    if (!line.trim()) continue;  // skip blank lines (trailing newline)
    try {
      const record = JSON.parse(line);
      if (record.result === 'fail') {
        failResults.push(parseCheckResultLine(record));
      }
    } catch (e) {
      // Skip unparseable lines (partial writes, corruption)
      // Log warning but continue
      console.warn(`[plan-gate-fv] Skipping unparseable NDJSON line`);
    }
  }

  return failResults;
};
```

## Common Pitfalls

### Pitfall 1: Exit Code Blocking
**What goes wrong:** Developer treats non-zero exit from run-formal-verify.cjs as a fatal error and prevents quorum from running.
**Why it happens:** Natural instinct to check exit codes.
**How to avoid:** Always use `|| FV_EXIT=$?` pattern, never `set -e` or `&&` chaining. Non-zero exit means SOME steps failed, but NDJSON may still have useful data.
**Warning signs:** Any `process.exit(1)` or bash `exit 1` in the FV gate step.

### Pitfall 2: Reading Stale NDJSON
**What goes wrong:** Reading check-results.ndjson before run-formal-verify.cjs truncates and rewrites it.
**Why it happens:** run-formal-verify.cjs truncates the file at the TOP of runOnce() — before any steps run. But the gate runs AFTER planner completes (step 8), so the file is from the PREVIOUS run.
**How to avoid:** The gate RUNS the TLA+ steps first, then reads. The file is always fresh after the gate's `node bin/run-formal-verify.cjs --only=tla` call completes.
**Verification:** Check that NDJSON timestamps are recent (within last 5 minutes) relative to gate execution.

### Pitfall 3: Surfacing Inconclusive as Failures
**What goes wrong:** Developer surfaces `result=inconclusive` entries as hypotheses, creating false urgency.
**Why it happens:** Inconclusive looks like a problem.
**How to avoid:** Only filter for `result === 'fail'`. Inconclusive (e.g., liveness properties without fairness) is expected and normal.

### Pitfall 4: FV_HYPOTHESES in quorum_context when empty
**What goes wrong:** Appending empty FV_HYPOTHESES to every quorum call, adding noise.
**Why it happens:** Forgetting the conditional.
**How to avoid:** Only inject FV context when `FV_STATUS === "fail"`. The condition `if [ -n "$FV_HYPOTHESES" ]` gate.

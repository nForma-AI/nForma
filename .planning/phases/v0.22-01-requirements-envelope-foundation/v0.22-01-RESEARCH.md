## Summary

Phase v0.22-01 builds two CLI tools: `bin/aggregate-requirements.cjs` (parses `.planning/REQUIREMENTS.md` into `formal/requirements.json`) and `bin/validate-requirements-haiku.cjs` (runs claude-haiku-4-5 semantic analysis for duplicates, contradictions, and ambiguity). The aggregated JSON is validated against a JSON Schema, and after user approval the envelope gains a `frozen_at` timestamp making it immutable.

The codebase already has strong precedents: `bin/task-envelope.cjs` for structured JSON sidecar files with schema validation, `bin/update-scoreboard.cjs` for Haiku API calls with fail-open patterns, and `formal/*.schema.json` files using JSON Schema draft-07. The project uses Node.js built-in test runner (`node:test`) with CommonJS (`.cjs`) modules and no external test framework.

**Primary recommendation:** Follow existing codebase patterns exactly. Use inline JSON Schema validation (no ajv dependency -- project has zero schema validation libraries). Parse REQUIREMENTS.md with regex matching the existing markdown structure. Call Haiku via `@anthropic-ai/sdk` following the `update-scoreboard.cjs` pattern.

## Architecture Patterns

### Recommended Project Structure
```

  aggregate-requirements.test.cjs  # Unit tests for aggregation

```

### Pattern 1: REQUIREMENTS.md Parsing
**What:** Regex-based extraction of requirement entries from markdown bullet list
**When to use:** Converting the markdown requirement definitions to structured JSON

The REQUIREMENTS.md format is consistent:
```markdown
- [ ] **ENV-01**: Requirements are aggregated into `formal/requirements.json` -- during...
```

Regex pattern: `^- \[[ x]\] \*\*([A-Z]+-\d+)\*\*:\s*(.+)$`
- Group 1: REQ-ID (e.g., `ENV-01`)
- Group 2: Full text description

Category is derived from the `### Category Name -- PREFIX` section header preceding each group of requirements. Phase assignment comes from the Traceability table at the bottom.

### Pattern 2: JSON Schema Inline Validation (from task-envelope.cjs)
**What:** Manual schema validation function without external library
**When to use:** Validating `formal/requirements.json` structure

Following `bin/task-envelope.cjs:validateEnvelope()` pattern: define a `validateRequirementsEnvelope(obj)` function that checks each field type/format and returns `{ valid, errors }`.

Additionally, write a standalone `formal/requirements.schema.json` (JSON Schema draft-07) for external tooling and documentation -- but runtime validation uses the inline function.

### Pattern 3: Haiku API Call (from update-scoreboard.cjs)
**What:** Fail-open Haiku call with SDK availability guard
**When to use:** Semantic validation in `validate-requirements-haiku.cjs`

Key elements:
- SDK availability guard with `require.resolve('@anthropic-ai/sdk')`
- `new Anthropic.default({ apiKey: process.env.ANTHROPIC_API_KEY })`
- `client.messages.create({ model: 'claude-haiku-4-5-20251001', max_tokens: 4096, messages: [...] })`
- JSON response parsing with error handling

### Pattern 4: Deterministic Output
**What:** Sort requirements by REQ-ID, use stable JSON serialization
**When to use:** Ensuring re-running aggregation produces identical output (Success Criterion 4)

- Sort requirements array by `id` field (lexicographic)
- Use `JSON.stringify(obj, null, 2)` with sorted keys
- Include `schema_version` in document root for versioning
- Content hash in metadata enables quick drift checks

### Anti-Patterns to Avoid
- **Non-deterministic JSON output:** Object key order is not guaranteed in JS. Use sorted keys explicitly or a stable serializer.
- **Haiku validation without user approval:** ENV-02 requires user resolution. Never auto-accept or auto-reject findings.
- **Freezing without explicit user consent:** `frozen_at` must only be set after user approves validation results.
- **Parsing markdown with assumptions about whitespace:** Use flexible regex that handles leading/trailing spaces.

## Common Pitfalls

### Pitfall 1: Markdown Parsing Edge Cases
**What goes wrong:** Requirements text may contain markdown formatting (backticks, bold, links) that breaks naive regex
**Why it happens:** The REQUIREMENTS.md contains inline code like `formal/requirements.json` and formatting like `claude-haiku-4-5`
**How to avoid:** Match the bullet pattern prefix only, capture everything after the colon as raw text. Preserve markdown in the text field -- don't strip formatting.
**Warning signs:** Unit test with backtick-heavy requirement text fails

### Pitfall 2: Haiku Non-Determinism
**What goes wrong:** Running validation twice produces different findings
**Why it happens:** LLM outputs are inherently non-deterministic
**How to avoid:** Use explicit rubrics in the prompt (define exactly what "duplicate", "contradiction", and "ambiguity" mean). Consider running 3 passes and aggregating (as noted in STATE.md decisions). Present raw findings to user -- don't filter.
**Warning signs:** Same requirement set gets different duplicate/contradiction counts on re-run

### Pitfall 3: Traceability Table Parsing
**What goes wrong:** Phase assignment extracted from wrong location
**Why it happens:** REQUIREMENTS.md has both inline `**Requirements**: ENV-01, ENV-02` in phase descriptions AND a Traceability table at the bottom
**How to avoid:** Parse the Traceability table exclusively for phase assignment -- it's the canonical source. The inline mentions in phase descriptions are informational only.
**Warning signs:** Requirement shows up with wrong phase in JSON output

### Pitfall 4: Frozen Envelope Re-aggregation
**What goes wrong:** Re-running aggregation on a frozen envelope overwrites `frozen_at`
**Why it happens:** No guard checking if envelope is already frozen
**How to avoid:** Check for existing `frozen_at` before writing. If frozen, error with "Envelope is frozen -- use amendment workflow (ENV-04)"
**Warning signs:** `frozen_at` timestamp changes after re-aggregation

### Pitfall 5: Category Detection Across Milestones
**What goes wrong:** Category extraction fails when REQUIREMENTS.md contains multiple milestone categories
**Why it happens:** The `### Category Name -- PREFIX` headers organize requirements by functional area
**How to avoid:** Parse section headers to build category map. Each `### Category Name -- PREFIX` header creates a new category context for subsequent bullet items.
**Warning signs:** Requirements get "unknown" category

## Code Examples

### REQUIREMENTS.md Parsing
```javascript
// Parse requirement bullets from REQUIREMENTS.md
function parseRequirements(content) {
  const requirements = [];
  let currentCategory = null;
  const lines = content.split('\n');

  for (const line of lines) {
    // Detect category headers: ### Category Name -- PREFIX
    const catMatch = line.match(/^### (.+?) \u2014 ([A-Z]+)$/);
    if (catMatch) {
      currentCategory = catMatch[1].trim();
      continue;
    }

    // Detect requirement bullets: - [ ] **ID**: Description
    const reqMatch = line.match(/^- \[[ x]\] \*\*([A-Z]+-\d+)\*\*:\s*(.+)$/);
    if (reqMatch) {
      requirements.push({
        id: reqMatch[1],
        text: reqMatch[2].trim(),
        category: currentCategory || 'uncategorized',
        completed: line.includes('[x]'),
      });
    }
  }
  return requirements;
}
```

### Traceability Table Parsing
```javascript
// Parse traceability table for phase assignments
function parseTraceability(content) {
  const map = {};
  const tableRegex = /^\|\s*([A-Z]+-\d+)\s*\|\s*(v[\d.]+-\d+)\s*\|\s*(\w+)\s*\|/gm;
  let match;
  while ((match = tableRegex.exec(content)) !== null) {
    map[match[1]] = { phase: match[2], status: match[3] };
  }
  return map;
}
```

## Open Questions

1. **Multi-milestone aggregation scope**
   - What we know: Current REQUIREMENTS.md is scoped to v0.22 only
   - What's unclear: Should `aggregate-requirements.cjs` handle multiple milestones in a single REQUIREMENTS.md, or is one-milestone-per-file the pattern?
   - Recommendation: Parse all requirements in the file regardless of milestone. The Traceability table provides phase assignment, which implicitly encodes milestone scope. This future-proofs the tool.

2. **Haiku validation pass count**
   - What we know: STATE.md notes "mitigated by aggregation of 3+ passes" for non-determinism
   - What's unclear: Should the tool run 3 Haiku passes by default or let the user configure?
   - Recommendation: Default to 3 passes with `--passes N` flag override. Aggregate by intersection (only report findings that appear in 2+ of 3 passes).

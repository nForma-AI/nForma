## Summary

This phase adds a formal TLA+ specification of QGSD's quorum workflow alongside a Node.js script that invokes the TLC model checker. The core artifact is `formal/tla/QGSDQuorum.tla` — a hand-authored TLA+ module whose state names (`IDLE`, `COLLECTING_VOTES`, `DELIBERATING`, `DECIDED`) are intentionally mirrored from the XState machine built in v0.12-01. Two separate configuration files are required: `MCsafety.cfg` uses TLC's symmetry reduction (valid for safety-only checks) with N=5 agents, while `MCliveness.cfg` omits symmetry and uses N=3 agents to keep liveness checking tractable. This two-config pattern is an established TLA+ best practice because TLC cannot apply symmetry optimizations when checking liveness properties.

The `bin/run-tlc.cjs` wrapper script follows the same pattern as existing QGSD CLI scripts (`validate-traces.cjs`, `check-provider-health.cjs`): zero production dependencies, Node.js built-ins only, `child_process.spawnSync` for JVM invocation. The script must detect Java availability via `JAVA_HOME` and `java --version`, provide a clear error if Java is absent, and download or locate `tla2tools.jar` from `formal/tla/`. The requirement that `npm test` passes without Java installed means the test for `run-tlc.cjs` must mock or skip the actual JVM invocation.

The primary risk flagged in STATE.md is the fairness assumption for `EventualConsensus`: whether `WF_vars(Decide)` (weak fairness) or `SF_vars(Decide)` (strong fairness) is the correct choice. Research confirms that WF is sufficient when the `Decide` action stays continuously enabled once votes are collected — which is true in the QGSD quorum model. Use `WF_vars(Decide)` in the Spec definition.

**Primary recommendation:** Author `QGSDQuorum.tla` first with safety invariants only, verify MCsafety.cfg passes TLC clean, then add the Spec fairness conjunct and verify MCliveness.cfg. Write `run-tlc.cjs` after both cfg files are validated.

### Pattern 1: TLA+ Module Structure (QGSDQuorum.tla)

**What:** A TLA+ module that models QGSD's quorum workflow as a state machine with named safety invariants and a liveness property.
**When to use:** State names MUST mirror the XState machine from v0.12-01 (`IDLE`, `COLLECTING_VOTES`, `DELIBERATING`, `DECIDED`) per the success criteria.

```tla
---- MODULE QGSDQuorum ----
(* TLA+ specification of QGSD quorum workflow.
   State names mirror the XState machine in src/machines/qgsd-workflow.machine.ts.
*)
EXTENDS Naturals, FiniteSets

CONSTANTS
    Agents,          \* Set of quorum model slots (e.g., {a1, a2, a3, a4, a5})
    MaxDeliberation  \* Maximum deliberation rounds before forced DECIDED

ASSUME MaxDeliberation \in Nat /\ MaxDeliberation > 0

\* N is the cardinality of Agents, used for majority calculation
N == Cardinality(Agents)

\* Symmetry set for TLC (referenced by MCsafety.cfg as SYMMETRY AgentSymmetry)
AgentSymmetry == Permutations(Agents)

VARIABLES
    phase,             \* One of: "IDLE", "COLLECTING_VOTES", "DELIBERATING", "DECIDED"
    successCount,      \* Number of APPROVE votes collected
    deliberationRounds \* Number of deliberation rounds completed

vars == <<phase, successCount, deliberationRounds>>

\* ── Type invariant ──────────────────────────────────────────────────────────
TypeOK ==
    /\ phase \in {"IDLE", "COLLECTING_VOTES", "DELIBERATING", "DECIDED"}
    /\ successCount \in 0..N
    /\ deliberationRounds \in 0..MaxDeliberation

\* ── Initial state ───────────────────────────────────────────────────────────
Init ==
    /\ phase = "IDLE"
    /\ successCount = 0
    /\ deliberationRounds = 0

\* ── Actions ─────────────────────────────────────────────────────────────────
StartQuorum ==
    /\ phase = "IDLE"
    /\ phase' = "COLLECTING_VOTES"
    /\ UNCHANGED <<successCount, deliberationRounds>>

CollectVotes(n) ==  \* n = number of APPROVE votes received
    /\ phase = "COLLECTING_VOTES"
    /\ successCount' = n
    /\ IF n * 2 >= N
       THEN phase' = "DECIDED"
       ELSE phase' = "DELIBERATING" /\ deliberationRounds' = deliberationRounds + 1
    /\ IF n * 2 >= N
       THEN UNCHANGED deliberationRounds
       ELSE TRUE

Deliberate(n) ==  \* n = votes after deliberation round
    /\ phase = "DELIBERATING"
    /\ successCount' = n
    /\ IF n * 2 >= N \/ deliberationRounds >= MaxDeliberation
       THEN phase' = "DECIDED" /\ UNCHANGED deliberationRounds
       ELSE phase' = "DELIBERATING" /\ deliberationRounds' = deliberationRounds + 1

Decide ==
    /\ phase = "DELIBERATING"
    /\ deliberationRounds >= MaxDeliberation
    /\ phase' = "DECIDED"
    /\ UNCHANGED <<successCount, deliberationRounds>>

Next ==
    \/ StartQuorum
    \/ \E n \in 0..N : CollectVotes(n)
    \/ \E n \in 0..N : Deliberate(n)
    \/ Decide

\* ── Safety invariants (TLA-01, TLA-02) ─────────────────────────────────────
\* MinQuorumMet: once DECIDED via approval, successCount * 2 >= N
MinQuorumMet ==
    phase = "DECIDED" =>
        (successCount * 2 >= N \/ deliberationRounds >= MaxDeliberation)

\* NoInvalidTransition: IDLE can only move to COLLECTING_VOTES (not skip states)
NoInvalidTransition ==
    [](phase = "IDLE" => phase' /= "DELIBERATING" /\ phase' /= "DECIDED")

\* ── Liveness (TLA-03) ───────────────────────────────────────────────────────
\* EventualConsensus: the system eventually reaches DECIDED in every behavior
EventualConsensus == <>(phase = "DECIDED")

\* ── Full spec with fairness ──────────────────────────────────────────────────
Spec == Init /\ [][Next]_vars /\ WF_vars(Decide) /\ WF_vars(StartQuorum)

====
```

**Key structural decisions:**
- `MinQuorumMet`: A safety invariant (checked by INVARIANT in cfg) — not a temporal property
- `NoInvalidTransition`: A temporal property expressed as `[]P` — can be checked as INVARIANT since `[]P` = invariant on P, or as PROPERTY
- `EventualConsensus`: A liveness property (`<>(phase = "DECIDED")`) — must be checked as PROPERTY
- `WF_vars(Decide)` ensures that once `deliberationRounds >= MaxDeliberation`, the `Decide` action is eventually taken

### Pattern 2: Safety Config (MCsafety.cfg, TLA-02)

**What:** TLC configuration for safety checking only. Uses symmetry reduction and N=5 agents.
**When to use:** Safety checks — can run with many workers and symmetry for speed.

```
\* MCsafety.cfg
SPECIFICATION Spec
CONSTANTS
    Agents <- AgentsSet5
    MaxDeliberation = 4
SYMMETRY AgentSymmetry
INVARIANT TypeOK
INVARIANT MinQuorumMet
PROPERTY NoInvalidTransition
CHECK_DEADLOCK FALSE
```

**Companion constant override in QGSDQuorum.tla (or separate MC model file):**
```tla
\* In the .tla file or a MCsafety.tla override module:
AgentsSet5 == {"a1", "a2", "a3", "a4", "a5"}
```

**Important:** SYMMETRY requires a named identifier from the spec. `AgentSymmetry == Permutations(Agents)` must be defined in `QGSDQuorum.tla` and referenced as `SYMMETRY AgentSymmetry` in the cfg.

**TLC invocation:**
```bash
java -jar formal/tla/tla2tools.jar \
  -config formal/tla/MCsafety.cfg \
  -workers auto \
  formal/tla/QGSDQuorum.tla
```

### Pattern 3: Liveness Config (MCliveness.cfg, TLA-03)

**What:** TLC configuration for liveness checking. No symmetry (incompatible), N=3 for tractability.
**When to use:** Liveness checks — PROPERTY only, no SYMMETRY.

```
\* MCliveness.cfg
SPECIFICATION Spec
CONSTANTS
    Agents <- AgentsSet3
    MaxDeliberation = 4
PROPERTY EventualConsensus
CHECK_DEADLOCK FALSE
```

**Companion constant:**
```tla
AgentsSet3 == {"a1", "a2", "a3"}
```

**TLC invocation:**
```bash
java -jar formal/tla/tla2tools.jar \
  -config formal/tla/MCliveness.cfg \
  -workers 1 \
  formal/tla/QGSDQuorum.tla
```

Note: Use `-workers 1` for liveness — multiple workers have had known bugs in some TLC versions (v1.7.4 fixed a liveness checking issue with multiple workers per release notes; confirm v1.8.0 is safe).

### Pattern 4: Node.js run-tlc.cjs Wrapper (TLA-04)

**What:** CLI script that checks Java ≥17, locates the JAR, runs TLC via spawnSync, exits with clear error if Java is absent.
**When to use:** `node bin/run-tlc.cjs --config MCsafety` or `node bin/run-tlc.cjs --config MCliveness`

```javascript
#!/usr/bin/env node
'use strict';
// bin/run-tlc.cjs
// Invokes TLC model checker for QGSD quorum formal specification.
// Requirements: TLA-04

const { spawnSync } = require('child_process');
const fs   = require('fs');
const path = require('path');

const args       = process.argv.slice(2);
const configArg  = args.find(a => a.startsWith('--config=')) || null;
const configName = configArg ? configArg.split('=')[1] : (args[0] || 'MCsafety');

const VALID_CONFIGS = ['MCsafety', 'MCliveness'];
if (!VALID_CONFIGS.includes(configName)) {
  process.stderr.write('[run-tlc] Unknown config: ' + configName + '. Valid: ' + VALID_CONFIGS.join(', ') + '\n');
  process.exit(1);
}

// ── 1. Locate Java ──────────────────────────────────────────────────────────
const JAVA_HOME = process.env.JAVA_HOME;
let javaExe;

if (JAVA_HOME) {
  javaExe = path.join(JAVA_HOME, 'bin', 'java');
  if (!fs.existsSync(javaExe)) {
    process.stderr.write('[run-tlc] JAVA_HOME is set but java binary not found at: ' + javaExe + '\n');
    process.stderr.write('[run-tlc] Unset JAVA_HOME or fix the path.\n');
    process.exit(1);
  }
} else {
  // Fall back to PATH lookup
  const probe = spawnSync('java', ['--version'], { encoding: 'utf8' });
  if (probe.error || probe.status !== 0) {
    process.stderr.write('[run-tlc] Java not found. Install Java ≥17 and set JAVA_HOME.\n');
    process.stderr.write('[run-tlc] Download: https://adoptium.net/\n');
    process.exit(1);
  }
  javaExe = 'java';
}

// ── 2. Check Java version ≥17 ───────────────────────────────────────────────
const versionResult = spawnSync(javaExe, ['--version'], { encoding: 'utf8' });
if (versionResult.error || versionResult.status !== 0) {
  process.stderr.write('[run-tlc] Failed to run: ' + javaExe + ' --version\n');
  process.exit(1);
}
const versionOutput = versionResult.stdout + versionResult.stderr;
// Java version string: "openjdk 17.0.1 ..." or "java version \"17.0.1\""
const versionMatch = versionOutput.match(/(?:openjdk|java version ["\s]*)(\d+)/i);
const javaMajor = versionMatch ? parseInt(versionMatch[1], 10) : 0;
if (javaMajor < 17) {
  process.stderr.write('[run-tlc] Java ≥17 required. Found: ' + versionOutput.split('\n')[0] + '\n');
  process.exit(1);
}

// ── 3. Locate tla2tools.jar ─────────────────────────────────────────────────
const jarPath = path.join(__dirname, '..', 'formal', 'tla', 'tla2tools.jar');
if (!fs.existsSync(jarPath)) {
  process.stderr.write('[run-tlc] tla2tools.jar not found at: ' + jarPath + '\n');
  process.stderr.write('[run-tlc] Download v1.8.0: curl -L https://github.com/tlaplus/tlaplus/releases/download/v1.8.0/tla2tools.jar -o formal/tla/tla2tools.jar\n');
  process.exit(1);
}

// ── 4. Invoke TLC ───────────────────────────────────────────────────────────
const specPath   = path.join(__dirname, '..', 'formal', 'tla', 'QGSDQuorum.tla');
const cfgPath    = path.join(__dirname, '..', 'formal', 'tla', configName + '.cfg');
const workers    = configName === 'MCliveness' ? '1' : 'auto';

process.stdout.write('[run-tlc] Running TLC with config: ' + configName + '\n');

const tlcResult = spawnSync(javaExe, [
  '-jar', jarPath,
  '-config', cfgPath,
  '-workers', workers,
  specPath,
], { encoding: 'utf8', stdio: 'inherit' });

if (tlcResult.error) {
  process.stderr.write('[run-tlc] TLC invocation error: ' + tlcResult.error.message + '\n');
  process.exit(1);
}

process.exit(tlcResult.status || 0);
```

### Anti-Patterns to Avoid

- **SYMMETRY in liveness config:** Never put `SYMMETRY` in `MCliveness.cfg` — TLC will refuse to check liveness properties when symmetry is active (documented incompatibility).
- **Expressing NoInvalidTransition as INVARIANT instead of PROPERTY:** `NoInvalidTransition == [](...)` is a temporal property; it should go under PROPERTY in the cfg, not INVARIANT. An INVARIANT checks a state predicate; PROPERTY checks a temporal formula.
- **Using `interpret()` from XState v4 in tests:** Not applicable here (no XState in this phase), but if test helpers reference the machine, use `createActor` (v5 API).
- **Putting `MaxDeliberation = 4` directly in CONSTANTS without model file:** Works, but coupling N to the cfg means changing it requires editing cfg files. Use named constants for testability.
- **Not using `\* ` for TLA+ comments:** TLA+ uses `\*` for line comments and `(* ... *)` for block comments — not `//` or `#`.
- **Running TLC with multiple workers for liveness (older TLC versions):** TLC v1.7.4 fixed a liveness-with-multiple-workers bug. This phase uses v1.8.0 which should be safe, but `MCliveness.cfg` uses `-workers 1` defensively.
- **Forgetting CHECK_DEADLOCK FALSE:** The QGSD workflow has a `final` state (`DECIDED`) — TLC will report a deadlock if it finds `DECIDED` with no outgoing transitions. Add `CHECK_DEADLOCK FALSE` to both cfg files.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State space exploration | Custom BFS over QGSD states | TLC model checker | TLC handles symmetry reduction, counterexample generation, liveness checking — all impossible to replicate correctly by hand |
| Symmetry reduction | Custom canonical form computation | `Permutations(Agents)` + `SYMMETRY AgentSymmetry` in cfg | TLC's symmetry is a sound reduction; hand-rolled canonicalization produces false negatives |
| Java version parsing | Ad-hoc string splitting | Regex against `java --version` output | Java version string format varies by JDK vendor (OpenJDK vs Oracle); regex is more robust |
| JAR download in run-tlc.cjs | `node-fetch` or `https.get` download | Pre-committed JAR at `formal/tla/tla2tools.jar` | Network access in a CLI wrapper adds failure modes; JAR is stable and versioned |
| TLA+ config from code | Dynamic cfg generation at test time | Static `.cfg` files committed to `formal/tla/` | Static files are versionable, diffable, and reviewable by tool users |

**Key insight:** TLC is a peer-reviewed model checker with 30+ years of development. The `run-tlc.cjs` wrapper's job is invocation and error reporting only — not model checking logic.

## Common Pitfalls

### Pitfall 1: SYMMETRY incompatibility with liveness
**What goes wrong:** Developer puts `SYMMETRY AgentSymmetry` in `MCliveness.cfg` and TLC refuses to check the `EventualConsensus` PROPERTY or produces incorrect results.
**Why it happens:** TLC's liveness algorithm (B\u00FCchi automaton construction) is not compatible with symmetry reduction because permuting states can create spurious counterexamples.
**How to avoid:** Never combine SYMMETRY and PROPERTY in the same cfg. Use two separate configs — `MCsafety.cfg` (SYMMETRY + INVARIANT) and `MCliveness.cfg` (PROPERTY only, smaller N).
**Warning signs:** TLC prints "Warning: Liveness checking is not compatible with symmetry reduction."

### Pitfall 2: SYMMETRY requires a named identifier, not an inline expression
**What goes wrong:** `SYMMETRY Permutations(Agents)` in cfg causes "Expected keyword but did not find it" parse error.
**Why it happens:** TLC's cfg parser requires SYMMETRY to reference a single operator name defined in the .tla file.
**How to avoid:** Define `AgentSymmetry == Permutations(Agents)` in `QGSDQuorum.tla` and use `SYMMETRY AgentSymmetry` in the cfg.
**Warning signs:** TLC fails on cfg parse before checking any states.

### Pitfall 3: NoInvalidTransition requires temporal action syntax
**What goes wrong:** Writing `NoInvalidTransition == phase = "IDLE" => phase' /= "DELIBERATING"` as a state invariant — this is a transition predicate (references `phase'`) and cannot be checked as a plain INVARIANT.
**Why it happens:** TLC INVARIANT checks stateless predicates; action predicates (with prime notation) are temporal properties.
**How to avoid:** Express as an action property: `NoInvalidTransition == [][phase = "IDLE" => phase' /= "DELIBERATING"]_vars` and check as PROPERTY (not INVARIANT). Or simplify: just check TypeOK as INVARIANT and express transition safety via the Next action definition itself.
**Warning signs:** TLC throws "Invariant references a primed variable" error.

### Pitfall 4: WF vs SF choice for EventualConsensus
**What goes wrong:** Using `SF_vars(Decide)` when `WF_vars(Decide)` is sufficient causes TLC to require stronger assumptions that may not hold in the model, producing spurious liveness violations.
**Why it happens:** SF (strong fairness) requires action to fire even if it's intermittently disabled; WF (weak fairness) requires firing only if continuously enabled. The `Decide` action in QGSD is enabled continuously once `deliberationRounds >= MaxDeliberation` — so WF suffices.
**Recommendation from STATE.md flag:** Use `WF_vars(Decide)` — confirmed by research as correct for this model.
**Warning signs:** TLC liveness check takes excessive time or reports unexpected counterexamples.

### Pitfall 5: TLA+ syntax for majority (integer division)
**What goes wrong:** Using `successCount >= N / 2` where TLA+ integer division truncates, making majority wrong for odd N (e.g., N=5: `5/2 = 2` instead of `3`).
**Why it happens:** TLA+ `Div` is integer division; `N / 2` in TLA+ is floor division.
**How to avoid:** Use `successCount * 2 >= N` — avoids division entirely and works for all N values. This mirrors the XState guard: `Math.ceil(context.slotsAvailable / 2)`.
**Warning signs:** TLC finds no violation of MinQuorumMet even when successCount = 2 of 5 agents approved.

### Pitfall 6: Missing CHECK_DEADLOCK FALSE causes false deadlock errors
**What goes wrong:** TLC reports "Deadlock reached" when the spec enters `DECIDED` state because `DECIDED` has no outgoing transitions.
**Why it happens:** TLC's default behavior treats terminal states as deadlocks. `DECIDED` is intentionally terminal.
**How to avoid:** Add `CHECK_DEADLOCK FALSE` to both cfg files. This is correct here because `DECIDED` is the intended terminal state.
**Warning signs:** TLC output shows "Error: Deadlock reached" with states leading to DECIDED.

### Pitfall 7: npm test fails when Java is not installed
**What goes wrong:** The test for `run-tlc.cjs` actually invokes TLC, failing in CI or on developer machines without Java.
**Why it happens:** Test invokes `node bin/run-tlc.cjs` directly without stubbing Java detection.
**How to avoid:** The test suite for `run-tlc.cjs` must use `spawnSync` with a stub that sets `JAVA_HOME` to a nonexistent path — testing the error exit path — rather than testing actual TLC execution. The "TLC completes with no violations" success criterion is validated manually or in a separate `npm run verify:tla` script that is NOT part of `npm test`.
**Warning signs:** `npm test` fails in CI with "Java not found" or hangs on TLC execution.

```javascript
const major = versionMatch ? parseInt(versionMatch[1], 10) : 0;
if (major < 17) { process.exit(1); }
```

## Open Questions

1. **Constant override syntax: `<-` vs `=` in cfg CONSTANTS**
   - What we know: TLC cfg CONSTANTS supports both `Agents = {"a1","a2","a3"}` (direct assignment) and `Agents <- AgentsSet5` (model value override from .tla operator). The override arrow `<-` allows the .tla file to define named sets, keeping cfg files cleaner.
   - What's unclear: Whether TLC v1.8.0 requires the `<-` form when using model values for symmetry, or if direct set assignment in cfg works with `SYMMETRY`.
   - Recommendation: Use `Agents <- AgentsSet5` with the operator defined in `QGSDQuorum.tla`. This is the pattern shown in official toolbox-generated cfg files. Direct set assignment (`Agents = {"a1","a2","a3","a4","a5"}`) in the cfg may also work — verify during implementation.

2. **NoInvalidTransition as INVARIANT vs PROPERTY**
   - What we know: `NoInvalidTransition` as defined above (`[](phase = "IDLE" => ...next state)`) is a temporal property requiring prime variables — it cannot be a plain INVARIANT.
   - What's unclear: Whether the simplest formulation (just check TypeOK as INVARIANT and rely on the `Next` action to encode valid transitions) is sufficient for the requirement, or if the planner should express it as an explicit action constraint.
   - Recommendation: Express `NoInvalidTransition` as `[][phase = "IDLE" => phase' \in {"IDLE", "COLLECTING_VOTES"}]_vars` — a temporal formula checked via PROPERTY. This satisfies TLA-02 without requiring the primed-variable workaround.

3. **tla2tools.jar in git vs gitignored**
   - What we know: The JAR is ~50MB. REQUIREMENTS.md says Java ≥17 is a shared prerequisite documented in VERIFICATION_TOOLS.md.
   - What's unclear: Whether the project convention prefers committing the JAR or requiring a download step.
   - Recommendation: Add `formal/tla/tla2tools.jar` to `.gitignore`. Document the download command in `formal/tla/README.md` and print it as the error message in `run-tlc.cjs` when JAR is missing. This avoids 50MB binary in git history.

4. **Whether N=5 safety check completes in reasonable time**
   - What we know: N=5 with MaxDeliberation=4 means each agent votes in 0..5 — state space is bounded. Symmetry reduces it by 5! = 120.
   - What's unclear: Actual TLC runtime on this model.
   - Recommendation: `-workers auto` for safety. If runtime exceeds 5 minutes, reduce N to 4 for safety cfg. Flag this for the planner.

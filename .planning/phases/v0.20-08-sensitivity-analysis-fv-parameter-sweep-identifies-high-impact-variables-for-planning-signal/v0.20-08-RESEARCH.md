## Summary

Phase v0.20-08 adds a sensitivity sweep tool (`run-sensitivity-sweep.cjs`) that varies key model parameters across defined ranges, runs TLA+/PRISM checks per configuration (using existing runners), and records outcome deltas (pass→fail, pass→inconclusive) in `formal/sensitivity-report.ndjson`. A second tool (`sensitivity-report.cjs`) renders a human-readable ranked report. A workflow extension inserts step 8.3 into `plan-phase.md` to inject the top-3 high-sensitivity parameters as `SENSITIVITY_CONTEXT` into the quorum review prompt.

## Domain Analysis

### Sensitivity Analysis in Formal Verification

Sensitivity analysis answers: "which model parameters, when varied, flip verification outcomes?" In QGSD's context:

- **Parameter**: a configuration value passed to TLC or PRISM (e.g., N=3 vs N=5 agents, tp_rate=0.70 vs 0.95, MaxDeliberation=3 vs 10)
- **Outcome flip**: a check that was `pass` at one parameter value becomes `fail` or `inconclusive` at another value
- **High-sensitivity**: a parameter where small changes cause outcome flips — these are the variables most worth monitoring, testing at boundaries, and surfacing to planners

The result is not about correctness of the *model* — it's about *which operational parameters the system is most brittle to*.

### Parameters to Sweep

From codebase analysis, the following parameters exist and are candidates for sensitivity sweeping:

**TLA+ parameters (formal/tla/MCsafety.cfg, MCliveness.cfg)**:
- `MaxSize` (quorum size N): currently 3. Range: 1, 2, 3, 4, 5. Flipping from 3→1 removes consensus requirement.
- `MaxDeliberation`: currently 7. Range: 1, 3, 5, 7, 10. Lower values force earlier timeout.
- `Agents` set size (cardinality N): currently 5. Range: 3, 4, 5. Affects state space and quorum ratio.

**PRISM parameters (formal/prism/quorum.pm)**:
- `tp_rate`: currently 0.85. Range: 0.50, 0.70, 0.85, 0.90, 0.95. Approval probability per slot.
- `unavail`: currently 0.15. Range: 0.05, 0.10, 0.15, 0.25, 0.40. Unavailability probability.

**UPPAAL timing parameters (formal/uppaal/quorum-races.xml)**:
- `MIN_SLOT_MS`, `MAX_SLOT_MS`, `TIMEOUT_MS` from run-uppaal.cjs defaults. UPPAAL is not yet installed so these sweeps will produce `inconclusive` — they are secondary priority.

**Minimum viable sweep** (SENS-01 requires ≥2 parameters, ≥3 values each):
- Parameter A: `MaxSize` (quorum size N): values [1, 2, 3] → 3 TLC model runs
- Parameter B: `tp_rate` (PRISM slot approval rate): values [0.5, 0.75, 0.95] → 3 PRISM runs
- Total: 6 model runs per sweep

### How to Parameterize Existing Runners

Both `run-tlc.cjs` and `run-prism.cjs` follow the pattern: spawn external model checker with args. For sensitivity sweeps:

**TLC parameterization** — dynamically generate a cfg file variant:
- Write a temp cfg file with overridden `MaxSize` (change `MaxSize = 3` to `MaxSize = 1`)
- Run TLC with `--config=<temp_cfg>` — run-tlc.cjs supports `--config=<name>` flag
- Alternative: directly spawn `java -jar tla2tools.jar -config temp.cfg MCsafety.tla`

**PRISM parameterization** — use `-const` CLI flag:
- PRISM supports `prism quorum.pm -pf "P=? [ F s=1 ]" -const tp_rate=0.5`
- Export `PRISM_BIN` for the check; if PRISM not found → result is `inconclusive` (fail-open)

The sweep tool does NOT go through `run-tlc.cjs` or `run-prism.cjs` directly — it invokes model checkers directly using the same patterns those runners use, but with parameter overrides. This avoids coupling to their file-level check_id mapping.

### v2.1 NDJSON Output Schema

Each sweep data point writes one NDJSON record to `formal/sensitivity-report.ndjson` (separate file from `check-results.ndjson`):

```json
{
  "tool": "run-sensitivity-sweep",
  "formalism": "tla",
  "result": "pass" | "fail" | "inconclusive",
  "check_id": "sens:tla-quorum-safety",
  "surface": "sensitivity",
  "property": "TLA+ safety: MaxSize sweep",
  "runtime_ms": 1234,
  "summary": "pass: MaxSize=3 — safety satisfied",
  "triage_tags": [],
  "timestamp": "...",
  "metadata": {
    "parameter": "MaxSize",
    "value": 3,
    "baseline_result": "pass",
    "delta": "stable",
    "model": "MCsafety"
  }
}
```

Key fields:
- `metadata.parameter`: which parameter was varied
- `metadata.value`: the value used in this run
- `metadata.baseline_result`: the outcome at the default value (for delta comparison)
- `metadata.delta`: `"stable"` | `"flip-to-fail"` | `"flip-to-inconclusive"` | `"flip-to-pass"`

IMPORTANT: `sensitivity-report.ndjson` is a SEPARATE file from `check-results.ndjson`. The sweep produces its own output file; it does NOT append to the main check-results.

### SENSITIVITY_CONTEXT Injection (SENS-02)

SENS-02 requires modifying `~/.claude/qgsd/workflows/plan-phase.md` to add step 8.3 (between step 8.4 envelope update and step 8.5 quorum dispatch).

Step 8.3 runs `run-sensitivity-sweep.cjs` (fail-open) and formats the top-3 high-sensitivity parameters for injection into the quorum `review_context`:

```
SENSITIVITY_CONTEXT:
  Parameter N (quorum size): flips safety at N<3 — code path: hooks/qgsd-prompt.js:FAN_OUT_COUNT
  Parameter tp_rate: flips consensus probability at tp_rate<0.6 — code path: export-prism-constants.cjs:TP_PRIOR
  Parameter MaxDeliberation: liveness timeout at MaxDeliberation<3 — code path: formal/tla/MCsafety.cfg
```

This context is prepended to the quorum `review_context` string so reviewers know which parameters are boundary-sensitive before evaluating the plan.

**Critical implementation detail**: plan-phase.md is a USER-OWNED workflow file at `~/.claude/qgsd/workflows/plan-phase.md`. Editing it directly would modify a system file. The correct approach: create a new step script `bin/run-sensitivity-sweep.cjs` (Plan 01), and then modify the workflow file (Plan 03) with a targeted insertion of step 8.3. This is a user-space file edit — it's acceptable per the workflow pattern used in v0.20-03 (PLAN-01 modified plan-phase.md step 8.1).

### Sensitivity Report Generation (SENS-03)

`bin/sensitivity-report.cjs` reads `formal/sensitivity-report.ndjson`, groups by parameter, counts outcome flips, ranks parameters by flip count (descending), and generates `formal/sensitivity-report.md`.

Report format:
```markdown
# Sensitivity Report — QGSD v0.20

## High-Sensitivity Parameters (ranked by outcome-flip count)

### 1. MaxSize (quorum size N) — 3 flips
- Flip at: N=1 (pass→fail), N=2 (pass→inconclusive)
- Code path: hooks/qgsd-prompt.js FAN_OUT_COUNT, formal/tla/MCsafety.cfg MaxSize
- Recommended test cases: test at N=2 boundary (FAN_OUT_COUNT=2), test at N=1 (no quorum)
- Recommended monitoring: track FAN_OUT_COUNT per planning session

### 2. tp_rate (slot approval probability) — 2 flips
- Flip at: tp_rate=0.5 (pass→fail)
- Code path: export-prism-constants.cjs TP_PRIOR (0.85), quorum-scoreboard.json tp_rate
- Recommended test cases: test quorum with 2/4 slots APPROVE (tp_rate≈0.5)
- Recommended monitoring: track per-slot APPROVE rate in quorum-scoreboard
```

### Existing Pattern: run-prism.cjs / run-uppaal.cjs

Both these runners follow a 4-section pattern: requires+constants → helper functions → main() → module.exports (if needed). The sweep runner should follow the same pattern.

### Fail-Open Requirements

- If TLC not installed: record `result=inconclusive`, `triage_tags=['no-tlc']`
- If PRISM not installed: record `result=inconclusive`, `triage_tags=['no-prism']`
- Exit 0 always (graceful degradation — same as run-uppaal.cjs, run-prism.cjs)
- The sensitivity context injection (step 8.3) is also fail-open: if sweep fails/produces empty results, inject empty SENSITIVITY_CONTEXT and continue

### VALID_FORMALISMS Check

`write-check-result.cjs` does NOT need to be updated — `sensitivity-report.ndjson` does NOT use `writeCheckResult()`. The sweep writes its own NDJSON format directly (to avoid the formalism validation). The records are v2.1-compatible but stored in a separate file. This is the same approach used by `generate-triage-bundle.cjs` (no writeCheckResult dependency).

Alternative: add `'sensitivity'` to VALID_FORMALISMS. But since the report is purely informational and stored separately, avoiding writeCheckResult() keeps the implementation simpler and avoids coupling.

## Risk Assessment

**risk_level: medium**

Risks:
1. **TLC and PRISM not installed** — Highly likely in this environment. Sweep will produce all-inconclusive results. The report and SENSITIVITY_CONTEXT injection must handle this gracefully (show "no data available" instead of crashing).
2. **plan-phase.md modification** — This is a system workflow file. The modification must be surgical (insert step 8.3 between ENV-02 and quorum). A botched edit could break the planning workflow. Mitigation: write the insertion as an exact string match + targeted edit.
3. **Slow TLC runs** — TLC on N=5 agents takes ~30s. Sweeping across N=1..5 × MaxDeliberation=1..10 would be ~150 model runs × 30s = 75 minutes. Mitigation: limit sweep to N=3 values per parameter, use MCsafety (faster) not MCliveness for the sweep.
4. **Circular dependency** — plan-phase.md runs run-sensitivity-sweep.cjs at step 8.3; this plan's execution uses plan-phase.md. No actual circularity since we're BUILDING the sweep tool here, not using it during this phase's planning.

# Phase v0.40-03: Scope Guard Hook - Research

**Researched:** 2026-03-19
**Domain:** PreToolUse Hook Integration & File Scope Matching
**Confidence:** HIGH

## Summary

Phase v0.40-03 extends nForma's session intelligence by adding a scope guard hook that warns Claude when Edit, Write, or MultiEdit tool calls target files outside the declared task scope. This phase builds directly on Phase v0.40-02, which writes a scope contract to `.claude/scope-contract.json` for each quick task. The scope guard is a warn-only advisory in v0.40 — it never blocks tool calls, only emits additionalContext warnings when out-of-scope files are targeted.

The implementation follows an established pattern in the codebase (nf-destructive-git-guard.js, nf-console-guard.js, nf-node-eval-guard.js). It requires four types of changes: a new hook file, installation/uninstallation registration, config-loader profile mapping, and comprehensive unit tests. All changes are pure Node.js — no new npm dependencies.

**Primary recommendation:** Follow the nf-destructive-git-guard.js pattern exactly. The hook pattern is mature, fail-open semantics are proven, and registration logic is consistent across existing PreToolUse guards. Focus implementation effort on scope matching logic and warning message clarity.

### Pattern 1: PreToolUse Hook Structure (Warn-Only Advisory)

**What:** A PreToolUse hook that evaluates file edits against declared scope, emits non-blocking warnings, exits 0 regardless of outcome.

**When to use:** When you need to audit/advise on tool calls without preventing them. Scope guard is warn-only specifically because false positives are possible at v1 (scope patterns may be too broad or too narrow).

**Example:**
```javascript
// Source: hooks/nf-destructive-git-guard.js (reference pattern)
'use strict';
const fs = require('fs');
const path = require('path');
const { spawnSync } = require('child_process');
const { loadConfig, shouldRunHook, validateHookInput } = require('./config-loader');

function main() {
  let raw = '';
  process.stdin.setEncoding('utf8');
  process.stdin.on('data', chunk => { raw += chunk; });
  process.stdin.on('end', () => {
    try {
      if (!raw || raw.trim() === '') {
        process.exit(0); // fail-open: empty stdin
      }

      const input = JSON.parse(raw);

      // Validate hook input
      const eventType = input.hook_event_name || input.hookEventName || 'PreToolUse';
      const validation = validateHookInput(eventType, input);
      if (!validation.valid) {
        process.stderr.write('[nf] WARNING: invalid input: ' + JSON.stringify(validation.errors) + '\n');
        process.exit(0); // fail-open
      }

      const cwd = input.cwd || process.cwd();
      const gitRoot = getGitRoot(cwd);
      if (!gitRoot) {
        process.exit(0); // not a git repo
      }

      // Profile guard
      const config = loadConfig(gitRoot);
      const profile = config.hook_profile || 'standard';
      if (!shouldRunHook('nf-scope-guard', profile)) {
        process.exit(0);
      }

      // Only act on specific tool names (Edit, Write, MultiEdit)
      const toolName = input.tool_name || input.toolName || '';
      if (!['Edit', 'Write', 'MultiEdit'].includes(toolName)) {
        process.exit(0);
      }

      // Extract target file path
      const targetPath = extractTargetPath(input.tool_input);
      if (!targetPath) {
        process.exit(0);
      }

      // Get current branch and read scope contract
      const branch = getCurrentBranch(cwd);
      const scopeContract = readScopeContract(gitRoot);

      // If no contract exists, no-op
      if (!scopeContract || !scopeContract[branch]) {
        process.exit(0);
      }

      // Check if target file is in scope
      const branchEntry = scopeContract[branch];
      const isInScope = isFileInScope(targetPath, branchEntry.scope_patterns || []);

      if (!isInScope) {
        const warningText =
          `[nf-scope-guard] WARNING: Editing file '${targetPath}' is outside declared scope. ` +
          `Declared scope: ${(branchEntry.scope_patterns || []).join(', ')}. ` +
          `Task: "${branchEntry.approach}".`;

        // Surface via additionalContext if nForma active
        const activityFile = path.join(gitRoot, '.planning', 'current-activity.json');
        if (fs.existsSync(activityFile)) {
          process.stdout.write(JSON.stringify({
            hookSpecificOutput: {
              hookEventName: 'PreToolUse',
              additionalContext: warningText,
            },
          }));
        } else {
          process.stderr.write(warningText + '\n');
        }
      }

      // Always exit 0 (warn-only, never block)
      process.exit(0);
    } catch (e) {
      if (e instanceof SyntaxError) {
        process.stderr.write('[nf] WARNING: malformed JSON: ' + e.message + '\n');
      }
      process.exit(0); // fail-open on any error
    }
  });
}

if (require.main === module) main();
module.exports = { getGitRoot, getCurrentBranch, readScopeContract, isFileInScope, extractTargetPath };
```

### Pattern 2: Scope Contract Reading & Matching

**What:** Read `.claude/scope-contract.json` (written by v0.40-02), extract scope patterns for the current branch, and match the target file path against those patterns.

**Scope contract format** (from Phase v0.40-02 quick.md Step 2.7):
```json
{
  "quick-333-example-branch": {
    "task_id": 333,
    "task_description": "...",
    "approach": "...",
    "out_of_scope": [...],
    "branches_affected": ["quick-333-example-branch"],
    "created_at": "2026-03-19T...",
    "planner_model": "opus",
    "created_by": "quick-orchestrator"
  }
}
```

**Matching logic:**
- Extract `scope_patterns` field (if present; v0.40-02 may differ from final schema)
- For target file path, check if it starts with any pattern in `scope_patterns`
- Trailing slashes in patterns are optional (normalize both)
- No glob expansion — patterns are literal string prefixes

### Pattern 3: Installation & Uninstallation (Idempotent)

**What:** Hook registration in bin/install.js follows a three-step pattern: check if hook exists by searching command string, create entry if missing, print status.

**When to use:** Every time a new PreToolUse hook is added.

**Registration code pattern** (from bin/install.js):
```javascript
// Register nForma scope guard hook (PreToolUse — warn on out-of-scope edits)
const hasScopeGuardHook = settings.hooks.PreToolUse.some(entry =>
  entry.hooks && entry.hooks.some(h => h.command && h.command.includes('nf-scope-guard'))
);
if (!hasScopeGuardHook) {
  settings.hooks.PreToolUse.push({
    hooks: [{ type: 'command', command: buildHookCommand(targetDir, 'nf-scope-guard.js'), timeout: 10 }]
  });
  console.log(`  ${green}✓${reset} Configured nForma scope guard hook (PreToolUse)`);
}
```

**Uninstallation code pattern** (from bin/install.js uninstall section):
```javascript
// Remove nf-scope-guard hook (uninstall path)
if (settings.hooks && settings.hooks.PreToolUse) {
  const before = settings.hooks.PreToolUse.length;
  settings.hooks.PreToolUse = settings.hooks.PreToolUse.filter(entry =>
    !(entry.hooks && entry.hooks.some(h => h.command && h.command.includes('nf-scope-guard')))
  );
  if (settings.hooks.PreToolUse.length < before) {
    settingsModified = true;
    console.log(`  ${green}✓${reset} Removed nForma scope guard hook`);
  }
  if (settings.hooks.PreToolUse.length === 0) delete settings.hooks.PreToolUse;
}
```

### Anti-Patterns to Avoid

- **Blocking on out-of-scope (v0.40):** The spec requires warn-only. Blocking introduces false-positive risk and must await post-v0.40 data validation. Exception: v1+ may change this if data supports escalation.
- **Global scope contract:** The contract is branch-specific (keyed by git branch). A global scope would conflict with concurrent quick tasks on different branches.
- **Ignoring the config-loader hook profile:** Every hook must respect `shouldRunHook()` and exit immediately if the profile is disabled. Skipping this breaks user configuration intent.
- **Outputting warnings to stdout:** PreToolUse hooks use stdout as the decision channel. Warnings must go to stderr or hookSpecificOutput.additionalContext. Mixing breaks the hook protocol.
- **Missing fail-open semantics:** If scope contract is missing, unreadable, or JSON parsing fails, the hook must silently no-op (exit 0). This is not an error case — it's the expected state before v0.40-02 completes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Input validation | Custom per-field validators | `validateHookInput()` from config-loader.js | Already handles all PreToolUse fields, fails gracefully, tested across 10+ hooks |
| Config loading | Bespoke config merge logic | `loadConfig()` from config-loader.js | Handles global + project layering, global overrides, null checks, permission errors |
| Hook registration | Ad-hoc install.js edits | Follow the `hasSomeHook` pattern from existing guards | Idempotency guard prevents duplicate entries on repeated installs; filter pattern ensures clean uninstalls |
| Git operations | Direct file reads (.git/HEAD) | `spawnSync('git', ['rev-parse', '--abbrev-ref', 'HEAD'])` | `.git/HEAD` is raw format (refs/heads/...), spawnSync abstracts this; matches project convention |
| Scope matching | Custom path logic | Literal string prefix matching with trailing-slash normalization | Glob matching would require scope contract schema expansion (deferred to v1); simple prefix match is sufficient for v0.40 |

**Key insight:** The hook ecosystem in this project is highly mature. Every common operation (input validation, config loading, git queries, installation registration) has a tested, documented pattern. Hand-rolling any of these invites subtle bugs (missed error cases, incorrect validation, installation collisions) and fragments the codebase.

## Common Pitfalls

### Pitfall 1: Scope Contract Missing or Malformed
**What goes wrong:** Hook fires before Phase v0.40-02 has written the scope contract, or the contract is corrupted/incomplete JSON.
**Why it happens:** The scope contract is only written by `/nf:quick` tasks in v0.40-02. If no quick task has run, there is no contract yet. If a write fails (disk full, permission), the contract may be truncated.
**How to avoid:** Always implement fail-open semantics: if `.claude/scope-contract.json` is missing, unreadable, or not valid JSON, silently exit 0 with no output. This is not an error — it's the expected state when no scope contract exists.
**Warning signs:** Hook logs or stderr messages appear when the contract is missing; warnings trigger before v0.40-02 completes. Verify with `cat ~/.claude/scope-contract.json` — if file doesn't exist or is empty, the contract is not yet written.

### Pitfall 2: Tool Name Matcher Too Narrow or Too Broad
**What goes wrong:** Hook fires on tools it shouldn't (e.g., Read, Bash), or fails to catch actual edits (e.g., a tool called "EditFile" instead of "Edit").
**Why it happens:** Tool names are strings in the hook input. Edit tool is named "Edit" (not "Edits", "edit-file", etc.). MultiEdit is distinct. Bash and Read are different tools and should not trigger scope checking.
**How to avoid:** Hard-code the tool name check: `if (!['Edit', 'Write', 'MultiEdit'].includes(toolName)) { process.exit(0); }`. Verify against the [Claude Code PreToolUse hook documentation](https://claude.ai/docs) and test with actual Edit/Write/MultiEdit calls.
**Warning signs:** Hook fires on non-edit tools (Bash, Read); or scope warnings don't appear for actual edits. Test by running Edit/Write/MultiEdit directly and checking stderr/additionalContext.

### Pitfall 3: Git Branch Query Fails Silently
**What goes wrong:** Hook can't determine the current git branch (e.g., in detached HEAD state), defaults to wrong branch, or mismatches scope contract key.
**Why it happens:** `spawnSync('git', ['rev-parse', '--abbrev-ref', 'HEAD'])` returns "HEAD" in detached HEAD state, not a branch name. If the scope contract was written with a branch name, detached HEAD won't match.
**How to avoid:** Check the exit status of spawnSync. If status !== 0 or result is empty, treat as "no branch found" and silently no-op (exit 0). Document the detached HEAD limitation: "Scope guard is branch-specific; detached HEAD state is not matched against any scope contract."
**Warning signs:** Hook warnings appear inconsistently; scope contract keys don't match branch names in logs. Verify with `git rev-parse --abbrev-ref HEAD` in your current state.

### Pitfall 4: Target File Path Extraction Wrong
**What goes wrong:** Hook extracts the wrong path from tool_input (e.g., gets directory instead of file, or undefined).
**Why it happens:** Edit/Write/MultiEdit tools pass file_path or files differently. Edit takes `file_path`, Write takes `file_path`, MultiEdit takes `files` (array). Mixing these up or not handling all cases means the hook doesn't see the target file.
**How to avoid:** Write a helper `extractTargetPath(toolInput)` that handles all three tool types. Test each: `Edit` → `tool_input.file_path`, `Write` → `tool_input.file_path`, `MultiEdit` → `tool_input.files[0]` (first file or warn if multiple). Return null if path is missing and silently no-op.
**Warning signs:** Hook never fires even when editing files; or warnings appear for the wrong file. Add debug logging: `process.stderr.write('[nf-scope-guard] DEBUG: extracted path = ' + targetPath + '\n');` to verify extraction.

### Pitfall 5: Scope Contract Key Format Mismatch
**What goes wrong:** Hook looks for `scopeContract[branch]` but the v0.40-02 implementation uses a different key format (e.g., with namespace prefix, or a different branch identifier).
**Why it happens:** The scope contract schema is defined in v0.40-02. If v0.40-02 changes the key format after Phase research, the hook's key lookup will fail.
**How to avoid:** Don't hard-code scope contract schema assumptions. Instead, verify the schema in v0.40-02's PLAN.md before implementation. Keep the extraction logic flexible: if `scopeContract[branch]` is missing, try alternative keys (e.g., `scopeContract['branch-' + branch]`). Log the actual contract structure for debugging.
**Warning signs:** Scope contract file exists but hook never triggers warnings. Verify contract structure: `cat ~/.claude/scope-contract.json | jq 'keys'` and match against current branch name.

### Pitfall 6: Warning Message Clarity
**What goes wrong:** Warning text is vague ("File out of scope") and Claude doesn't understand what to do.
**Why it happens:** Warn-only advisories must be crystal clear: which file, which scope, what to do next. Vague warnings get ignored.
**How to avoid:** Warning must include: (1) the file being edited, (2) the declared scope patterns, (3) the task approach for context. Example: `[nf-scope-guard] WARNING: Editing 'bin/install.js' is outside declared scope. Declared scope: ['hooks/', 'config-loader.js']. Task: "Add scope guard hook". Consider: is this file within the task scope?`
**Warning signs:** Claude ignores the warning and commits out-of-scope changes anyway. Review the warning text and test with actual out-of-scope edits.

## Code Examples

Verified patterns from official sources:

### Reading Scope Contract and Matching Paths
```javascript
// Source: Phase v0.40-02 quick.md (scope contract format)
function readScopeContract(gitRoot) {
  const contractPath = path.join(gitRoot, '.claude', 'scope-contract.json');
  if (!fs.existsSync(contractPath)) {
    return null;
  }
  try {
    const content = fs.readFileSync(contractPath, 'utf8');
    return JSON.parse(content);
  } catch (e) {
    // Malformed JSON or read error — fail-open
    return null;
  }
}

function isFileInScope(targetPath, patterns) {
  if (!patterns || !Array.isArray(patterns)) {
    return true; // If no patterns, all paths are in scope
  }

  // Normalize paths: remove trailing slashes
  const normalized = targetPath.replace(/\/$/, '');

  for (const pattern of patterns) {
    const normalizedPattern = pattern.replace(/\/$/, '');
    // Literal prefix match (not glob)
    if (normalized.startsWith(normalizedPattern)) {
      return true;
    }
    // Also check if normalized matches pattern exactly
    if (normalized === normalizedPattern) {
      return true;
    }
  }

  return false;
}
```

### Extracting Target Path from Edit/Write/MultiEdit
```javascript
// Source: Claude Code PreToolUse hook input schema
function extractTargetPath(toolInput) {
  if (!toolInput || typeof toolInput !== 'object') {
    return null;
  }

  // Edit and Write both use file_path
  if (toolInput.file_path) {
    return toolInput.file_path;
  }

  // MultiEdit uses files (array of {file_path, action, ...})
  if (toolInput.files && Array.isArray(toolInput.files) && toolInput.files.length > 0) {
    const first = toolInput.files[0];
    if (first && first.file_path) {
      return first.file_path;
    }
  }

  return null;
}
```

### Getting Current Branch Name
```javascript
// Source: nf-destructive-git-guard.js pattern (adapted for scope guard)
function getCurrentBranch(cwd) {
  const result = spawnSync('git', ['rev-parse', '--abbrev-ref', 'HEAD'], {
    cwd,
    encoding: 'utf8',
    timeout: 5000,
  });
  if (result.status !== 0 || result.error) {
    return null; // Not a git repo or detached HEAD
  }
  const branch = (result.stdout || '').trim();
  return branch || null;
}
```

## Open Questions

1. **Scope pattern format (literal prefix vs glob)?**
   - What we know: Phase v0.40-02 ROADMAP says "scope_patterns" but doesn't specify format. The quick.md Step 2.7 example uses `["hooks/", "core/workflows/quick.md"]` — plain strings.
   - What's unclear: Does v0.40-02 allow glob patterns (e.g., `*.md`)? Should the matcher support them?
   - Recommendation: For v0.40, implement literal prefix matching only (simplest, least room for bugs). If v0.40-02 writes glob patterns, the hook will treat them as literals and still match correctly (e.g., `*.md` would only match files literally named `*.md`, not markdown files). Glob support can be added in v1 if data shows it's needed.

2. **Scope contract schema disagreement between v0.40-02 ROADMAP and quick.md?**
   - What we know: ROADMAP says `"scope_patterns": [...]` but quick.md Step 2.7 doesn't explicitly list scope_patterns in the contract structure (it lists approach, out_of_scope, etc.).
   - What's unclear: Does the contract include a `scope_patterns` field, or should the scope guard derive patterns from `out_of_scope`?
   - Recommendation: Treat `scope_patterns` as the primary field (from ROADMAP). If v0.40-02 implementation writes something different, the implementation task can adjust. For now, assume `scopeContract[branch].scope_patterns` exists and is an array.

3. **MultiEdit handling: warn on first out-of-scope or every out-of-scope file?**
   - What we know: MultiEdit passes multiple files. The spec says "warn on out-of-scope edits" but doesn't clarify if one warning or per-file.
   - What's unclear: Should the hook warn once (better UX) or per file (more verbosity)?
   - Recommendation: Warn once on first out-of-scope file found. This matches the destructive-git-guard pattern (one advisory per tool call, not per file). If multiple files are out of scope, Claude can see the first warning and decide.

4. **Scope contract location: project-root/.claude/scope-contract.json or ~/.claude/?**
   - What we know: ROADMAP says ".claude/scope-contract.json" (project-local). Phase v0.40-02 quick.md Step 2.7 also says ".claude/scope-contract.json" (project-local).
   - What's unclear: Is this at project root or user home?
   - Recommendation: Project-local (same directory as `.claude/nf.json`). This isolates scope contracts per project, prevents one project's scope from affecting another.

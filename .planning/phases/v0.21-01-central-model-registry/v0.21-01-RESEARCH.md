# Phase v0.21-01: Central Model Registry - Research

**Researched:** 2026-03-01
**Domain:** Formal model registry and versioning architecture; atomic spec promotion
**Confidence:** HIGH

## Summary

Phase v0.21-01 establishes the foundational infrastructure for QGSD's formal verification system by centralizing all model artifacts in a single canonical `formal/` directory with provenance tracking. Currently, formal models are generated artifacts with no central registry, and update flows (manual edits, XState generation, debug sessions, per-phase proposals) have no coordination path. This phase creates three critical pieces: (1) a `formal/model-registry.json` that tracks each model's metadata (last_updated, update_source, version, session_id), (2) an atomic promotion script (`promote-model.cjs`) that merges per-phase proposed specs into central specs with atomic rename-based writes, and (3) the debug-to-canonical write path that allows debug sessions to update formal specs directly.

The phase is first in the v0.21 sequence by design — all downstream phases (v0.21-02 through v0.21-06) depend on ARCH-01/02/03 to write to `formal/` directly, never to per-phase scratch only. High confidence stems from established patterns in the codebase: generate-formal-specs.cjs uses atomic writes via tmp+rename; the XState machine is already the single source of truth; and per-phase proposed specs (PLAN-01 requirement) are already expected in the REQUIREMENTS.md.

**Primary recommendation:** Build model-registry.json as a flat inventory keyed by relative path (registry[`tla/QGSDQuorum_xstate.tla`] = {version, last_updated, update_source, session_id}); implement promote-model.cjs as a simple merge-and-atomically-rename operation with validation that all invariants in proposed-changes.tla exist as PROPERTY statements; add debug session write path via explicit model update calls in qgsd-debug orchestrator before returning results.

## Architecture Patterns

### Registry Schema: model-registry.json

The registry uses a flat, single-level JSON structure keyed by relative file paths:

```json
{
  "version": "1.0",
  "last_sync": "2026-03-01T12:34:56.789Z",
  "models": {
    "tla/QGSDQuorum_xstate.tla": {
      "version": 2,
      "last_updated": "2026-03-01T12:34:56.789Z",
      "update_source": "generate",
      "source_id": "generate:tla-from-xstate",
      "session_id": null,
      "description": "XState → TLA+ quorum state machine (generated)"
    },
    "tla/QGSDOscillation.tla": {
      "version": 1,
      "last_updated": "2026-02-27T08:15:30.000Z",
      "update_source": "manual",
      "source_id": null,
      "session_id": null,
      "description": "Run-collapse oscillation detection (hand-verified)"
    }
  }
}
```

**Registry key design:**
- **Keys:** Relative paths from project root (e.g., `tla/QGSDQuorum_xstate.tla`, `alloy/quorum-votes.als`)
- **Flat structure:** Single `models` object, no nested directories
- **Metadata fields:**
  - `version`: Integer, incremented on each update
  - `last_updated`: ISO 8601 timestamp (sortable, diff-friendly)
  - `update_source`: One of `["generate", "debug", "plan-promote", "manual"]`
  - `source_id`: Tool/phase identifier (e.g., `v0.21-01-001`, null for manual/debug)
  - `session_id`: Debug session ID (format: `debug-sess-<unix-timestamp>-<uuid4-first-8>`, null otherwise)
  - `description`: Human-readable annotation

### Pattern 1: Atomic Promotion via Tmp+Rename

**What:** Merge proposed invariants into canonical spec and atomically update both spec and registry in sequence.

**When to use:** Every time a plan's verified `proposed-changes.tla` is accepted; debug session acceptance.

**Key algorithm:**
1. Read proposed-changes.tla, extract PROPERTY definitions
2. Load target spec, extract existing PROPERTY names
3. Validate no name conflicts (fail if duplicate)
4. Write merged spec to tmp file
5. Update registry entry in tmp file
6. Atomic rename both files (sequence: spec first, registry second)

### Pattern 2: Registry Initialization

**What:** On first run, scan `formal/` and populate registry with current metadata.

**When to use:** Single initialization step in Phase v0.21-01 (one-time, idempotent).

**Key algorithm:**
1. Check if registry exists (idempotent: exit if yes)
2. Walk formal/tla, formal/alloy, formal/prism subdirectories
3. For each .tla, .als, .pm file:
   - Use mtime as `last_updated`
   - Set `update_source: "generate"` if filename contains "xstate", else "manual"
   - Create entry with version=1
4. Write registry as JSON (sorted keys for consistent diffs)

### Pattern 3: Debug Session Write Path

**What:** Record invariant accepted from debug session directly into canonical spec with session ID.

**When to use:** During debug session completion (LOOP-04 integration).

**Key algorithm:**
1. Validate property doesn't already exist in target spec
2. Append PROPERTY definition with debug session provenance comment
3. Atomic write: tmp file, then rename
4. Update registry with update_source="debug" and session_id

### Anti-Patterns to Avoid
- **Sidecar update logs:** Registry MUST be updated immediately, never "sync later"
- **Partial registry updates:** All fields must be atomically consistent (version + last_updated + update_source together)
- **Direct edits bypassing promote:** Manual edits must still update registry or they desync

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic multi-file updates | Custom file-locking logic | POSIX rename (atomic) + tmp+rename pattern | Rename is atomic; custom locking adds complexity |
| Merging TLA+ specifications | String concatenation | Dedicated promote-model.cjs with deduplication | TLA+ has strict syntax rules; naive merge creates invalid specs |
| Registry versioning | Manual timestamp tracking | Explicit `version` field (increment on update) | Simple, diff-friendly, enables change detection |
| Session ID generation | Random UUIDs alone | Format `debug-sess-<unix-timestamp>-<uuid4-first-8>` | Enables sorting by session time + uniqueness |

## Common Pitfalls

### Pitfall 1: Registry Desync Due to Partial Writes

**What goes wrong:** Spec updated but registry not (spec write succeeds, registry fails mid-operation).

**Why it happens:** Treating spec and registry updates as independent operations.

**How to avoid:** Always tmp+rename both atomically. If either rename fails, next operation detects inconsistency via comparing mtime.

**Warning signs:**
- Registry last_updated older than spec file mtime
- Duplicate versions in consecutive runs
- Debug invariants disappear after next generate run

### Pitfall 2: Duplicate Property Names During Promotion

**What goes wrong:** Proposed property has same name as existing property; merge succeeds syntactically but creates invalid TLA+.

**Why it happens:** No validation before merge.

**How to avoid:** Extract PROPERTY names from both files, intersect, fail if not empty.

**Warning signs:**
- TLC fails with "already defined" or "duplicate definition" errors
- Specs parse in IDE but fail in TLC

### Pitfall 3: Registry Path Key Inconsistency

**What goes wrong:** Same file registered with different keys (e.g., `formal/tla/QGSDQuorum.tla` vs `./formal/tla/QGSDQuorum.tla`).

**Why it happens:** Not normalizing paths when reading/writing registry.

**How to avoid:** Always use `path.relative(process.cwd(), absolutePath)` for registry keys. Validate no `./` or `/` prefixes.

**Warning signs:**
- Registry grows unexpectedly (duplicate entries)
- Promotion updates one key but not another

### Pitfall 4: Missing Session ID for Debug Updates

**What goes wrong:** Debug invariant written with `update_source: "debug"` but null session_id. Impossible to trace which session added what.

**Why it happens:** Not tracking session context when calling debug update functions.

**How to avoid:** Require session_id as parameter. Fail if null for debug updates. Generate format: `debug-sess-<unix-timestamp-seconds>-<uuid4-first-8>`.

**Warning signs:**
- Multiple debug properties with null session_id
- Can't reconstruct debug history

## Code Examples

### Initialize Registry (Idempotent)

Function signature for `initialize-model-registry.cjs`:
- Input: none (scans formal/ directory)
- Output: creates `formal/model-registry.json`
- Idempotent: exits silently if registry exists
- Scans formal/tla, formal/alloy, formal/prism for .tla, .als, .pm files
- Uses file mtime as initial last_updated timestamp
- Sets update_source="generate" for xstate files, "manual" for others

### Promote Model with Validation

Function signature for `promote-model.cjs`:
- Input: proposed-changes.tla path, target spec path, source_id
- Output: updated target spec and registry.json
- Validates no duplicate PROPERTY names before merge
- Writes to tmp file, then atomically renames
- Updates registry: version++, last_updated=now, update_source="plan-promote", source_id=input

### Record Debug Invariant

Function signature for `accept-debug-invariant.cjs`:
- Input: spec path, property name, property body, session_id
- Output: updated spec and registry
- Validates property doesn't already exist
- Appends PROPERTY with debug session provenance comment
- Atomic write (tmp+rename)
- Updates registry: update_source="debug", session_id=input

## Open Questions

1. **Registry initialization from git history**
   - What we know: Registry scans current files and guesses update_source from filename
   - What's unclear: Backtrack to git history for unmarked specs, or accept "manual" with "pre-v0.21-01 origin unknown"?
   - Recommendation: Accept unmarked origin. Simplicity over perfect historical tracking.

2. **Collision avoidance for concurrent updates**
   - What we know: Tmp+rename is atomic within single script
   - What's unclear: Will promote-model.cjs run in parallel with generate-formal-specs.cjs?
   - Recommendation: Use `tmp.${Date.now()}.${Math.random().toString(36).slice(2)}` for collision-free tmp files. Assume rename atomicity.

3. **Registry retention policy**
   - What we know: Registry is flat JSON, O(1) lookup
   - What's unclear: Will registry grow unbounded, or archive old specs?
   - Recommendation: No action in v0.21-01; add policy in v0.22 if needed. Git provides full audit trail anyway.

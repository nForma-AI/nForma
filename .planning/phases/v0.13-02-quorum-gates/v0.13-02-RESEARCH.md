## Architecture Patterns

### Plan 01: plan-milestone-gaps.md Step 5 Replacement (QUORUM-01)

**Current Step 5 structure (lines 75-112):**
```
## 5. Present Gap Closure Plan

[markdown presentation block — shows proposed phases]

Create these {X} phases? (yes / adjust / defer all optional)

Wait for user confirmation.
```

**New Step 5 structure:**
```
## 5. Present Gap Closure Plan and Run Quorum Approval (QUORUM-01)

[markdown presentation block — unchanged, shows proposed phases]

Form your own position first: are the proposed phases correctly scoped to close all identified gaps?
State your vote: APPROVE or BLOCK with 1-2 sentence rationale.

Run R3 quorum inline (dispatch_pattern from `commands/qgsd/quorum.md`):
- Mode A — pure question
- Question: "Are these gap closure phases correctly scoped and complete to close the identified milestone gaps? Review: {summary of proposed phases, 1 line each}"
- Dispatch all active slots as sibling `qgsd-quorum-slot-worker` Tasks (one per slot)
- Synthesize results inline, deliberate up to 10 rounds per R3.3
- Update scoreboard via `node bin/update-scoreboard.cjs` after each round completes — BEFORE proceeding to Step 6

Route on quorum_result:
- **APPROVED**: Display `<!-- GSD_DECISION --> Quorum approved {N} gap closure phases.` then proceed to Step 6.
- **BLOCKED**: Report the blocker — specific model objection + proposed resolution path. Halt. Do not update ROADMAP.md until resolved.
- **ESCALATED**: Escalate to user per R3.4. Halt.
```

### Plan 03: discuss-phase.md for_user[] Second Quorum Pass (QUORUM-03)

**Current present_gray_areas structure (line 234+):**
```
<step name="present_gray_areas">
[display auto-resolved assumptions]
[AskUserQuestion multiSelect for for_user[] items]
```

**New present_gray_areas structure (when in auto mode):**

Add a new sub-step BEFORE the AskUserQuestion:

```
**Auto mode check (before presenting to user):**

Read auto-advance config:
AUTO_CFG=$(node ~/.claude/qgsd/bin/gsd-tools.cjs config-get workflow.auto_advance 2>/dev/null || echo "true")

**If `--auto` flag present OR `AUTO_CFG` is true AND `for_user[]` is non-empty:**

Run second quorum pass on each item in `for_user[]`:

For each question in `for_user[]`:
1. Form Claude's position: can quorum decide this, or does it genuinely need user vision?
2. Run R3 quorum inline (Mode A):
   - Question: "Should '[question text]' be decided by quorum now (removing it from user presentation), or does it genuinely require the user's vision/preference? If quorum can decide: provide the recommended answer biased toward the long-term solution."
   - Include phase name and goal as context
   - Update scoreboard after each round — BEFORE proceeding to next question
3. If APPROVED with answer: move to `auto_resolved[]`, remove from `for_user[]`
4. If BLOCKED: keep in `for_user[]`

**If `for_user[]` is now empty:** Skip to `write_context` directly (same as existing R4 pre-filter empty path).

**If `for_user[]` still has items (or in interactive mode):** Present to user via AskUserQuestion as before.
```

**Key design note:** This is distinct from the R4 pre-filter (which runs at analyze_phase / r4_pre_filter steps). The R4 pre-filter operates on candidate questions identified by analyze_phase. This second pass operates on survivors that R4 already tried once and couldn't resolve. It's a second attempt at auto-resolution, not a duplication.

## Common Pitfalls

### Pitfall 1: Modifying Only Source, Not Installed Copy

**What goes wrong:** qgsd-core/workflows/ changes do not affect runtime — ~/.claude/qgsd/workflows/ is what executes.
**How to avoid:** Each plan must edit BOTH paths. Verify with diff (expect only tilde-vs-absolute path differences after edit).

### Pitfall 2: Blowing execute-phase Orchestrator Context Budget

**What goes wrong:** Plan 02's quorum prompt in the orchestrator context (which is ~10-15% when running execute-phase) uses too much of the remaining context window.
**How to avoid:** Keep the gaps_found quorum prompt compact — 1-sentence-per-gap summary, not full VERIFICATION.md text. The question template: "Phase {X} verification found {N} gaps: {1-sentence-per-gap}. Auto-resolvable or needs human?"

### Pitfall 3: Duplicating R4 Pre-filter Logic in discuss-phase

**What goes wrong:** Plan 03 accidentally replaces the R4 pre-filter instead of adding a second pass after it.
**How to avoid:** The insertion point is at present_gray_areas step (AFTER r4_pre_filter runs). The r4_pre_filter step (lines 194-232) is unchanged. The second quorum pass fires on for_user[] survivors at present_gray_areas entry.

### Pitfall 4: Scoreboard Update After Task Spawn (Ordering Violation)

**What goes wrong:** Scoreboard update fires after the plan-phase Task is already spawned.
**How to avoid:** Explicitly state in each plan: `update-scoreboard.cjs` call completes BEFORE any downstream Task spawn. This is an ordering guarantee required by R8.

### Pitfall 5: discuss-phase Second Pass in Interactive Mode

**What goes wrong:** Running quorum on for_user[] items even when not in auto mode, eliminating user agency.
**How to avoid:** The second quorum pass is strictly gated on `--auto` flag OR `workflow.auto_advance=true`. Interactive mode (neither present) skips the second pass and goes straight to AskUserQuestion.

### Pitfall 6: offer_next Exception in execute-phase Not Updated

**What goes wrong:** The offer_next step has an exception clause that says "gaps_found already presents path, skip auto-advance." After QUORUM-02, the gaps_found branch now uses quorum, not a presentation. The exception text must be updated to reflect the new behavior.
**How to avoid:** Explicitly include the offer_next exception update in Plan 02.

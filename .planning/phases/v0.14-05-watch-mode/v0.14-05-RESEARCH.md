## Summary

Watch mode is a self-contained enhancement to `bin/run-formal-verify.cjs`. The implementation requires no new dependencies — Node.js built-in `fs.watch` on the machine file's **parent directory** is the correct and robust approach. Watching the file directly fails on macOS after the first rename-based save (vim/neovim pattern), while directory watching survives all save styles.

The core design is: extract the existing IIFE body into a `runOnce()` async function; add a `--watch` branch that calls `runOnce()` immediately, then sets up a 300ms debounce loop on `fs.watch(machineDir)`, registered with `SIGINT → watcher.close() + process.exit(0)`. In watch mode, `runOnce()` failures do not terminate the process. In non-watch mode, current behavior (exit code) is unchanged.

Tests belong in the existing `bin/run-formal-verify.test.cjs` file, use `spawn()` (not `spawnSync()`), and require no Java/TLC/Alloy/PRISM. Three integration tests cover: startup-and-persist, change-triggers-rerun, and clean-SIGINT-exit. All tests verified to run within 15s.

**Primary recommendation:** Watch the parent directory `src/machines/`, filter events to `qgsd-workflow.machine.ts`, debounce 300ms, call extracted `runOnce()`. Register SIGINT handler. No external dependencies.

## Architecture Patterns

### Recommended Change: Extract `runOnce()` from IIFE

**Current structure (`run-formal-verify.cjs`):**
```javascript
// Current: everything inside a single async IIFE that exits on completion
(async () => {
  // ... 60 lines of setup + step execution + summary printing ...
  process.exit(failed > 0 ? 1 : 0);
})();
```

**New structure:**
```javascript
// Step 1: Extract body into runOnce()
async function runOnce() {
  const startMs = Date.now();
  // ... existing setup + step execution + summary printing ...
  // CHANGE: return failed count instead of process.exit()
  return failed;
}

// Step 2: Watch mode wrapper
const watchArg = argv.includes('--watch');

if (watchArg) {
  // Initial run
  runOnce().catch(err => process.stderr.write(TAG + ' Error: ' + err.message + '\n'));

  // Watch setup
  const machineDir  = path.resolve(__dirname, '..', 'src', 'machines');
  const machineName = 'qgsd-workflow.machine.ts';
  let debounceTimer = null;

  process.stdout.write(TAG + ' Watching: ' + path.join(machineDir, machineName) + '\n');
  process.stdout.write(TAG + ' Press Ctrl+C to stop.\n');

  const watcher = fs.watch(machineDir, (eventType, filename) => {
    if (filename === machineName) {
      if (debounceTimer) clearTimeout(debounceTimer);
      debounceTimer = setTimeout(() => {
        process.stdout.write('\n' + TAG + ' ' + HR + '\n');
        process.stdout.write(TAG + ' Change detected — re-running verification\n');
        process.stdout.write(TAG + ' ' + HR + '\n\n');
        runOnce().catch(err =>
          process.stderr.write(TAG + ' Error: ' + err.message + '\n')
        );
      }, 300);
    }
  });

  process.on('SIGINT', () => {
    if (debounceTimer) clearTimeout(debounceTimer);
    watcher.close();
    process.stdout.write('\n' + TAG + ' Exiting watch mode.\n');
    process.exit(0);
  });

} else {
  // Non-watch: existing behavior
  runOnce().then(failed => {
    if (failed > 0) {
      process.stderr.write(TAG + ' ' + failed + ' step(s) failed.\n');
      process.exit(1);
    }
    process.exit(0);
  });
}
```

### Pattern: Directory Watch, Not File Watch

**What:** Watch `path.dirname(machineFile)` and filter on `filename === basename`.

**Why:** On macOS, `fs.watch()` on a **file** stops receiving events after the first rename event. Editors like vim/neovim write via `write-to-temp + rename`; after the rename, the original inode is gone and the watcher becomes deaf.

**Verified behavior (macOS, Node v25):**
- `fs.watch(filePath)` — fires on first rename, then silent forever (tested)
- `fs.watch(dirPath)` — fires on every rename and write, survives all save styles (tested)

**Example:**
```javascript
// Source: verified experimentally on macOS Node v25.6.1
const machineDir  = path.resolve(__dirname, '..', 'src', 'machines');
const machineName = 'qgsd-workflow.machine.ts';

const watcher = fs.watch(machineDir, (eventType, filename) => {
  if (filename === machineName) {
    // handle change
  }
});
```

### Pattern: 300ms Debounce

**What:** A `setTimeout` that resets on each new event; verification fires only once after the burst settles.

**Why:** Directory watch fires multiple events per save (tested: rename-save generates 1–3 events). Without debounce, verification would trigger multiple times per keystroke.

**Verified:** 3 rapid saves in 20ms produced 2 verification runs without debounce, 2 properly collapsed runs with 300ms debounce.

```javascript
// Source: verified experimentally
let debounceTimer = null;

const watcher = fs.watch(machineDir, (eventType, filename) => {
  if (filename === machineName) {
    if (debounceTimer) clearTimeout(debounceTimer);
    debounceTimer = setTimeout(triggerRun, 300);
  }
});
```

### Pattern: Composable with `--only`

Watch mode re-uses the existing `steps` filter (already applied from `--only=X`). No additional logic needed — `runOnce()` reads `steps` from the outer scope, which is already filtered.

**Result:** `--watch --only=generate` watches and re-runs only the 2 generate steps on change. Useful for fast TLA+ generation feedback.

### Anti-Patterns to Avoid

- **`fs.watch(machineFilePath)` directly:** Dies after first rename (vim/neovim pattern). Do NOT do this.
- **`spawnSync` replaced with `spawn` for async:** Unnecessary complexity. `spawnSync` blocks the JS event loop, but OS still queues file system events during the block. After `spawnSync` returns, the next debounced event fires correctly. Tested and confirmed.
- **Exiting on verification failure in watch mode:** In watch mode, a failing run should print the summary and keep watching. Only exit on SIGINT. Do NOT call `process.exit(1)` inside `runOnce()` — return the failure count instead.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| File change detection | Custom polling loop with `fs.statSync` | `fs.watch(directory)` | OS-native, zero CPU overhead, no polling interval to tune |
| Debouncing | Stateful counter or queue | Simple `setTimeout` clear-and-reset | Sufficient for this use case; tested against rapid saves |
| Process cleanup | Finalize() patterns, `process.on('beforeExit')` | `process.on('SIGINT', () => { watcher.close(); process.exit(0); })` | Ctrl+C is the primary exit signal for CLI watch modes |

**Key insight:** File watching is deceptively simple at the API surface but has platform-specific edge cases (rename behavior on macOS vs Linux). The directory-watch pattern sidesteps the macOS inode issue without adding any external dependency.

## Common Pitfalls

### Pitfall 1: Watching the File Directly (macOS Inode Death)

**What goes wrong:** `fs.watch('src/machines/qgsd-workflow.machine.ts')` fires for the first save, then goes silent. Subsequent saves produce no events.

**Why it happens:** vim, neovim, and many editors write to a temp file then `rename()` it over the target. The rename replaces the inode. The `fs.watch` listener was watching the old inode, which no longer exists.

**How to avoid:** Watch the parent **directory** and filter `filename === 'qgsd-workflow.machine.ts'`.

**Warning signs:** Watch mode fires once, then nothing on subsequent saves. Test by saving twice consecutively.

### Pitfall 2: Not Clearing Debounce Timer on SIGINT

**What goes wrong:** User hits Ctrl+C; a debounce timer fires 300ms later after the watcher is closed, attempting to call `runOnce()` on a closed watcher or a dead process.

**How to avoid:**
```javascript
process.on('SIGINT', () => {
  if (debounceTimer) clearTimeout(debounceTimer);  // CRITICAL
  watcher.close();
  process.exit(0);
});
```

### Pitfall 3: Calling `process.exit()` Inside `runOnce()` in Watch Mode

**What goes wrong:** `runOnce()` currently ends with `process.exit(1)` on failure. In watch mode this terminates the whole process on the first failing run.

**How to avoid:** Return the `failed` count from `runOnce()`. Let the caller decide whether to `process.exit()` (non-watch) or print and continue (watch).

### Pitfall 4: Not Printing Timestamp/Separator on Re-runs

**What goes wrong:** Multiple runs produce interleaved output with no visual separation. Developer can't tell where one run ends and the next begins.

**How to avoid:** Print a clear separator with timestamp at the top of each watch-triggered run:
```javascript
process.stdout.write('\n' + TAG + ' ' + HR + '\n');
process.stdout.write(TAG + ' Change detected — re-running verification\n');
process.stdout.write(TAG + ' ' + new Date().toISOString() + '\n');
process.stdout.write(TAG + ' ' + HR + '\n\n');
```

### Pitfall 5: `--watch` Flag Parsed Before `--only` in argv

**What goes wrong:** `argv.find(a => a.startsWith('--only='))` still works correctly; `--watch` is a simple includes check. No interaction. But if the watch flag is detected with `argv.find` (which returns value), it could match `--only=watch` accidentally.

**How to avoid:** Parse `--watch` with `argv.includes('--watch')` (exact match), not `startsWith`.

### Directory Watch with Debounce (Complete Pattern)
```javascript
// Source: verified experimentally on macOS Node v25.6.1
const fs   = require('fs');
const path = require('path');

const machineDir  = path.resolve(__dirname, '..', 'src', 'machines');

// Source: verified experimentally (node:test + spawn)
const { test }   = require('node:test');
const assert     = require('node:assert');
const { spawn }  = require('child_process');
const fs         = require('node:fs');
const os         = require('os');
const path       = require('path');

test('watch mode: starts and does not exit (DX-01)', { timeout: 30000 }, async () => {
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'rfv-watch-'));
  try {
    // Copy the real machine file to tmpdir for isolation
    const srcMachine = path.resolve(__dirname, '..', 'src', 'machines', 'qgsd-workflow.machine.ts');
    const dstMachine = path.join(tmpDir, 'machines', 'qgsd-workflow.machine.ts');
    fs.mkdirSync(path.dirname(dstMachine), { recursive: true });
    fs.copyFileSync(srcMachine, dstMachine);

    const child = spawn(process.execPath, [RUN_FV, '--watch', '--only=generate'], {
      cwd: tmpDir,
      stdio: ['ignore', 'pipe', 'pipe'],
      env: { ...process.env },
    });

    let stdout = '';
    child.stdout.on('data', d => { stdout += d.toString(); });
    child.stderr.on('data', d => { stdout += d.toString(); });

    // Wait for 'Watch mode enabled' message
    await new Promise((resolve, reject) => {
      const deadline = setTimeout(() => reject(new Error('timeout waiting for watch start')), 10000);
      const check = setInterval(() => {
        if (stdout.includes('Watch mode enabled') || stdout.includes('Watching:')) {
          clearInterval(check); clearTimeout(deadline); resolve();
        }
      }, 100);
    });

    // Send SIGINT — verify clean exit
    child.kill('SIGINT');
    const { code } = await new Promise(resolve =>
      child.on('close', (code, signal) => resolve({ code, signal }))
    );

    assert.strictEqual(code, 0, 'watch mode must exit 0 on SIGINT');
    assert.match(stdout, /Watch mode|Watching/i, 'must print watching message');
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

### Test Pattern: Change Detection
```javascript
// Source: verified experimentally (macOS, Node v25.6.1)
test('watch mode: re-runs on machine file change (DX-01)', { timeout: 30000 }, async () => {
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'rfv-watch-change-'));
  try {
    const machinesDir = path.join(tmpDir, 'src', 'machines');
    fs.mkdirSync(machinesDir, { recursive: true });
    const machine = path.join(machinesDir, 'qgsd-workflow.machine.ts');
    fs.copyFileSync(
      path.resolve(__dirname, '..', 'src', 'machines', 'qgsd-workflow.machine.ts'),
      machine
    );

    const child = spawn(process.execPath, [RUN_FV, '--watch', '--only=generate'], {
      cwd: tmpDir,
      stdio: ['ignore', 'pipe', 'pipe'],
    });

    let stdout = '';
    child.stdout.on('data', d => { stdout += d.toString(); });
    child.stderr.on('data', d => { stdout += d.toString(); });

    // Wait for first run to print its summary
    await new Promise((resolve, reject) => {
      const deadline = setTimeout(() => reject(new Error('timeout waiting for initial run')), 15000);
      const check = setInterval(() => {
        if (stdout.includes('SUMMARY')) { clearInterval(check); clearTimeout(deadline); resolve(); }
      }, 100);
    });

    // Touch the machine file (simulate save)
    const snapLen = stdout.length;
    fs.writeFileSync(machine, fs.readFileSync(machine, 'utf8') + '\n// watch-trigger');

    // Wait for second 'Change detected' message
    await new Promise((resolve, reject) => {
      const deadline = setTimeout(() => reject(new Error('timeout waiting for change detection')), 10000);
      const check = setInterval(() => {
        if (stdout.length > snapLen && stdout.includes('Change detected')) {
          clearInterval(check); clearTimeout(deadline); resolve();
        }
      }, 100);
    });

    child.kill('SIGINT');
    await new Promise(resolve => child.on('close', resolve));

    assert.match(stdout, /Change detected/i);
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

## Open Questions

1. **Machine path: hardcoded vs configurable**
   - What we know: The machine file path is currently hardcoded in STEPS[0] as `src/machines/qgsd-workflow.machine.ts`
   - What's unclear: Should `--watch` accept a `--watch-path=<file>` override for future flexibility?
   - Recommendation: Hardcode for DX-01. DX-02 (deferred) can add `--watch-path` when incremental results are added. Keep it simple now.

2. **`--only` + `--watch`: default behavior when no `--only`**
   - What we know: Full 21-step pipeline takes ~10 min; watch mode with full pipeline means 10min per save
   - What's unclear: Should watch mode default to `--only=generate` for speed? Or document "use --watch --only=generate for fast feedback"?
   - Recommendation: No default; let user choose. Print a note: "Tip: use --watch --only=generate for faster feedback." DX-01 says "re-runs verification" without qualification.

3. **Concurrent run guard**
   - What we know: spawnSync blocks the event loop during execution; events queue in OS buffer
   - What's unclear: If `runOnce()` becomes async in the future (e.g., if spawnSync is replaced with spawn), a re-entrant call could be triggered
   - Recommendation: Add a `let running = false` guard even now as defensive code. No behavior change today; prevents bugs in future refactors.

<decisions>
## Implementation Decisions

### Source authentication & configuration
- Endpoint URL configured in .planning/observe-sources.md YAML frontmatter (same as existing sources)
- Auth tokens referenced via environment variable names (e.g., `auth_env: PROMETHEUS_TOKEN`), never inline in config
- Pattern matches existing conventions: gh CLI uses GH_TOKEN, Sentry MCP uses SENTRY_AUTH_TOKEN
- Each source config includes: type, label, endpoint, auth_env, query/filter template, timeout, fail_open

### Query/alert mapping to issue schema
- **Prometheus:** Alert name = title, severity label = severity (critical/warning/info), alertstate = status context, labels = meta. PromQL query result → each firing alert becomes one issue.
- **Grafana:** Alert rule name = title, alert state maps to severity (alerting=error, pending=warning, ok=info), dashboard URL = url, annotations = meta.
- **Logstash/Elasticsearch:** Message field = title, level field = severity (error/warn/info/debug), @timestamp = created_at, full hit _source = meta. Query uses Elasticsearch DSL.
- All handlers return standard schema: { id, title, severity, url, age, created_at, meta, source_type, issue_type }

### Drift vs Issue classification
- Prometheus sources → issue_type: "drift" by default (metrics are quantitative divergences from expected values)
- Grafana sources → issue_type: "issue" by default (alerts are discrete events)
- Logstash sources → issue_type: "issue" by default (log entries are discrete events)
- Already handled by v0.27-02 config loader issue_type inference — no new inference logic needed
- Users can override issue_type per-source in config if needed

### Testing without live instances
- Mock HTTP responses using node:test mock API (consistent with v0.27-02 handler test pattern)
- Each handler test provides sample API responses as fixtures
- Test coverage: successful fetch + parse, auth failure (401/403), timeout, malformed response, empty results
- No live API calls in tests — all external HTTP mocked

### Claude's Discretion
- HTTP client choice (node:https, fetch, or child_process curl)
- Exact PromQL/Elasticsearch query templates for default configs
- How to handle pagination for large result sets
- Rate limiting / retry logic within handlers

</decisions>

# Phase v0.27-04: Production Source Types — Research

## Summary

Add three production monitoring source handlers (Prometheus, Grafana, Logstash) to the existing observe handler registry. Each handler fetches from its respective API via HTTP, maps results to the standard issue schema, and registers itself through `observe-registry.cjs`. The existing architecture (pluggable registry, parallel dispatch, fail-open, config loader with issue_type inference) already supports these types — only handler implementations and config samples are needed.

## Existing Architecture

### Handler Registry (`bin/observe-registry.cjs`)
- `registerHandler(sourceType, handlerFn)` — register a handler
- `dispatchSource(sourceConfig, options, timeoutSeconds)` — dispatch with timeout + fail-open
- `dispatchAll(sources, options)` — Promise.allSettled parallel dispatch
- Standard return schema: `{ source_label, source_type, status: "ok"|"error", issues: [...], error? }`

### Config Loader (`bin/observe-config.cjs`)
- Already defines `DRIFT_TYPES = ['prometheus', 'grafana', 'logstash']` — issue_type inference is pre-wired
- Loads from `.planning/observe-sources.md` YAML frontmatter
- Applies defaults for timeout, fail_open

### Existing Handlers (`bin/observe-handlers.cjs`)
- `handleGitHub` — uses `execFileSync('gh', ...)`, direct CLI
- `handleSentry` / `handleSentryFeedback` — returns `pending_mcp` with MCP instruction
- `handleBash` — uses `execFileSync('sh', ['-c', command])`
- All follow try-catch fail-open pattern, return standard schema

### Issue Schema (per handler return)
```js
{
  id: string,        // e.g., "prom-alert-0"
  title: string,     // alert name or log message
  severity: string,  // "critical"|"error"|"warning"|"info"
  url: string,       // link to source dashboard
  age: string,       // human-readable age
  created_at: string,// ISO8601
  meta: string,      // extra context
  source_type: string,
  issue_type: string // "issue" or "drift"
}
```

### Config Pattern (`.planning/observe-sources.md`)
Each source entry: `{ type, label, endpoint, auth_env, query, timeout, fail_open, issue_type }`

## API Research

### Prometheus API
- **Alerts endpoint:** `GET /api/v1/alerts` — returns all active alerts
  - Response: `{ status: "success", data: { alerts: [{ labels: {alertname, severity, ...}, annotations: {summary, description}, state: "firing"|"pending", activeAt, value }] } }`
- **Query endpoint:** `GET /api/v1/query?query=<PromQL>` — instant query
  - Response: `{ status: "success", data: { resultType: "vector", result: [{ metric: {...}, value: [timestamp, "value"] }] } }`
- **Auth:** Bearer token via `Authorization: Bearer $TOKEN` header, or no auth for local instances
- **Mapping strategy (from CONTEXT.md):**
  - Alert name → title
  - severity label → severity (critical/warning/info)
  - alertstate → status context in meta
  - labels → meta
  - Each firing alert → one issue with issue_type: "drift"

### Grafana API
- **Alerting endpoint:** `GET /api/v1/provisioning/alert-rules` — all alert rules (Grafana 9+)
  - Alternative: `GET /api/alerting/grafana/state/history` for alert state history
- **Simpler approach:** `GET /api/alerts` — legacy alerting endpoint, returns `[{ id, dashboardId, panelId, name, state, newStateDate, url }]`
  - State values: "ok", "paused", "no_data", "pending", "alerting"
- **Unified alerting (Grafana 9+):** `GET /api/prometheus/grafana/api/v1/alerts` — returns Prometheus-compatible alert format
- **Auth:** `Authorization: Bearer $TOKEN` (API key or service account token)
- **Mapping strategy (from CONTEXT.md):**
  - Alert rule name → title
  - Alert state → severity (alerting=error, pending=warning, ok=info)
  - Dashboard URL → url
  - Annotations → meta

### Logstash / Elasticsearch API
- **Search endpoint:** `POST /<index>/_search` with Elasticsearch DSL body
- **Query:** `{ query: { bool: { filter: [{ range: { "@timestamp": { gte: "now-1h" } } }, { terms: { level: ["error", "warn"] } }] } }, size: 50, sort: [{ "@timestamp": "desc" }] }`
- **Auth:** Basic auth or API key via `Authorization: ApiKey $KEY`
- **Mapping strategy (from CONTEXT.md):**
  - message field → title
  - level field → severity (error/warn/info/debug)
  - @timestamp → created_at
  - Full hit _source → meta

## Implementation Approach

### HTTP Client
Use Node.js built-in `fetch` (available since Node 18+, project uses Node 25 per TAP test fix context). No external dependencies needed. Each handler:
1. Reads `auth_env` from config, resolves `process.env[auth_env]`
2. Makes HTTP request with appropriate headers
3. Parses JSON response
4. Maps to standard issue schema
5. Returns fail-open result on any error

### Handler File Structure
Add three new handler functions to a new file `bin/observe-handlers-production.cjs` (keeping existing handlers separate for clarity), or add directly to `bin/observe-handlers.cjs`. Given CONTEXT.md says "no changes to the observe orchestrator — pure handler + config additions," adding to the existing handlers file is cleaner.

**Recommended:** Add to `bin/observe-handlers.cjs` since:
- Single import point for all handlers
- Consistent with existing pattern
- observe-registry.cjs just calls registerHandler() — agnostic to file layout

### Registration
The observe command spec (`commands/qgsd/observe.md`) must register the new handlers. Follow existing pattern where handlers are imported and registered at command init.

### Config Samples
Add commented-out sample entries in `.planning/observe-sources.md` for each new source type (Prometheus sample already partially exists).

## Risk Assessment

### Low Risk
- Handler implementation is straightforward HTTP fetch + JSON parse
- Existing registry, config loader, and dispatch already support the new types
- No changes to observe orchestrator, renderer, or debt writer
- All external calls mocked in tests

### Medium Risk
- Prometheus/Grafana API differences between versions (Grafana 8 vs 9+ alerting API)
  - **Mitigation:** Support the most common endpoint, document version requirements
- Elasticsearch query DSL complexity
  - **Mitigation:** Provide a sensible default query template, let users customize

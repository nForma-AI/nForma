**Researched:** 2026-03-09

## Summary

This phase updates the User Guide (`docs/USER-GUIDE.md`) to satisfy two requirements: GUIDE-01 (cross-reference TUI screenshots to feature descriptions) and GUIDE-02 (step-by-step Getting Started walkthrough with screenshots at each stage). The User Guide is currently a 532-line pure-text Markdown document with zero images or screenshots. It has strong content -- workflow diagrams, command reference, configuration reference, usage examples, troubleshooting -- but no visual anchors for a new user to confirm they are on the right track.

The project already has 11 TUI screenshots in `docs/assets/` captured via VHS tape automation. These screenshots cover all five TUI modules (Agents, Reqs, Config, Sessions, Solve) plus sub-views and modals. The README already references three of these screenshots. The User Guide needs to selectively embed the most relevant ones alongside feature descriptions and add new contextual screenshots to the Getting Started walkthrough showing install output and first command results.

The work is pure Markdown editing in a single file (`docs/USER-GUIDE.md`) with image references to existing assets in `docs/assets/`. No new screenshots need to be generated in this phase -- that is Phase v0.32-04 (VIS-01/VIS-02). This phase uses the existing screenshot inventory as-is.

**Primary recommendation:** Add inline `<img>` tags with width=600 to embed TUI screenshots in feature sections; restructure the User Guide with a new "Getting Started" section at the top that walks through install -> quorum setup -> first command with screenshots at each stage; use relative paths (`assets/tui-*.png`) since the User Guide lives in `docs/`.

### Anti-Patterns to Avoid
- **Screenshot wall:** Do not add every screenshot to every section. Each screenshot should appear once, in the most contextually relevant location.
- **Screenshots without context:** Never place a screenshot without preceding text explaining what it shows and why the user is looking at it.
- **Full-width screenshots:** Always use `width="600"` to prevent 3600px-wide screenshots from dominating the page.
- **Duplicate screenshots between README and User Guide:** The README already has 3 screenshots. The User Guide should reference screenshots the README does NOT use prominently (prioritize tui-config, tui-sessions, tui-reqs-coverage, tui-modal-add-agent, tui-solve).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Screenshot sizing | CSS stylesheets or custom HTML | `<img width="600">` | GitHub strips most CSS; width attribute is the reliable approach |
| Image captions | Custom `<figure>/<figcaption>` HTML | Italic text line after image | GitHub GFM renders `<figure>` inconsistently; italic line is the standard pattern |
| Table of contents | Manual anchor links | Match existing ToC pattern in the file | The User Guide already has a ToC; extend it, don't replace it |

**Key insight:** The User Guide is a single Markdown file rendered by GitHub. All formatting must use GitHub-compatible HTML subset and GFM features. No external tooling or build steps.

## Common Pitfalls

### Pitfall 1: Broken Image Paths
**What goes wrong:** Images don't render on GitHub because the path is wrong relative to the Markdown file.
**Why it happens:** The User Guide is in `docs/` while screenshots are in `docs/assets/`. Using `docs/assets/` as the path (relative to repo root) would work in README but not in User Guide.
**How to avoid:** Use `assets/tui-*.png` (relative to `docs/USER-GUIDE.md`). Test by checking that the path from `docs/` resolves correctly.
**Warning signs:** Images render locally but not on GitHub, or vice versa.

### Pitfall 2: Screenshot-Text Mismatch
**What goes wrong:** A screenshot shows UI state that doesn't match the surrounding text description.
**Why it happens:** Screenshots were captured at a different time than the documentation was written. TUI modules may have been renamed or reorganized.
**How to avoid:** Verify each screenshot's content matches what the text describes. The VHS tape (`scripts/tui-screenshot.tape`) documents exactly what each screenshot captures.
**Warning signs:** Text says "click the Health tab" but the screenshot shows the Agents list view.

### Pitfall 3: Getting Started Assumes Too Much
**What goes wrong:** The walkthrough skips a step that a new user needs, or uses jargon ("quorum", "slots") without introduction.
**Why it happens:** Documentation writers are domain experts who forget what new users don't know.
**How to avoid:** The Getting Started walkthrough should define "quorum" on first use, explain what MCP servers are in one sentence, and never assume the user has read the README first. Each step should be self-contained.
**Warning signs:** Steps reference concepts introduced in later sections.

### Pitfall 4: ToC Not Updated
**What goes wrong:** New sections added but Table of Contents not updated to match.
**Why it happens:** Easy to forget when adding content.
**How to avoid:** Include ToC update as an explicit task step. Test by verifying every `##` heading has a ToC entry.
**Warning signs:** Clicking a ToC link leads nowhere.

### Test Scaffolding Content

```javascript
// test/user-guide-overhaul.test.cjs
const { describe, it } = require('node:test');
const { strict: assert } = require('node:assert');
const fs = require('node:fs');
const path = require('node:path');

const GUIDE = fs.readFileSync(path.join(__dirname, '..', 'docs', 'USER-GUIDE.md'), 'utf8');

describe('GUIDE-01: TUI screenshots cross-referenced to features', () => {
  it('contains at least 4 TUI screenshot references', () => {
    const matches = GUIDE.match(/<img\s+src="assets\/tui-[^"]+"/g) || [];
    assert.ok(matches.length >= 4, `Expected >= 4 TUI screenshots, found ${matches.length}`);
  });
  it('all referenced images exist on disk', () => {
    const imgRefs = [...GUIDE.matchAll(/src="(assets\/[^"]+)"/g)].map(m => m[1]);
    for (const ref of imgRefs) {
      const fullPath = path.join(__dirname, '..', 'docs', ref);
      assert.ok(fs.existsSync(fullPath), `Referenced image missing: ${ref}`);
    }
  });
  it('screenshots are distributed across sections, not clustered', () => {
    const lines = GUIDE.split('\n');
    const imgLines = lines.reduce((acc, line, idx) => {
      if (/<img\s+src="assets\/tui-/.test(line)) acc.push(idx);
      return acc;
    }, []);
    if (imgLines.length >= 2) {
      const spread = imgLines[imgLines.length - 1] - imgLines[0];
      assert.ok(spread > 50, `Screenshots should be spread across the document (spread: ${spread} lines)`);
    }
  });
});

describe('GUIDE-02: Getting Started walkthrough', () => {
  it('has a Getting Started section', () => {
    assert.ok(/^## Getting Started/m.test(GUIDE), 'User Guide should have ## Getting Started');
  });
  it('Getting Started appears before Workflow Diagrams', () => {
    const gsIdx = GUIDE.indexOf('## Getting Started');
    const wdIdx = GUIDE.indexOf('## Workflow Diagrams');
    assert.ok(gsIdx > -1 && wdIdx > -1, 'Both sections must exist');
    assert.ok(gsIdx < wdIdx, 'Getting Started must appear before Workflow Diagrams');
  });
  it('contains step-by-step structure', () => {
    assert.ok(/### Step 1/i.test(GUIDE), 'Should have Step 1');
    assert.ok(/### Step 2/i.test(GUIDE), 'Should have Step 2');
  });
  it('includes screenshots in Getting Started', () => {
    const gsIdx = GUIDE.indexOf('## Getting Started');
    const nextSection = GUIDE.indexOf('\n## ', gsIdx + 1);
    const gsContent = GUIDE.substring(gsIdx, nextSection > -1 ? nextSection : undefined);
    const imgCount = (gsContent.match(/<img\s/g) || []).length;
    assert.ok(imgCount >= 2, `Getting Started should have >= 2 screenshots, found ${imgCount}`);
  });
  it('walkthrough mentions install and quorum setup', () => {
    const gsIdx = GUIDE.indexOf('## Getting Started');
    const nextSection = GUIDE.indexOf('\n## ', gsIdx + 1);
    const gsContent = GUIDE.substring(gsIdx, nextSection > -1 ? nextSection : undefined).toLowerCase();
    assert.ok(gsContent.includes('install') || gsContent.includes('npx'), 'Should mention installation');
    assert.ok(gsContent.includes('quorum') || gsContent.includes('mcp-setup'), 'Should mention quorum setup');
  });
});
```

2. **Should the Getting Started section replace or precede existing content?**

   - What's unclear: Whether to insert Getting Started before Workflow Diagrams or restructure the entire document.

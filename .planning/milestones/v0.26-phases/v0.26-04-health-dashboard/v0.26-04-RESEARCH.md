### Pattern 1: Pure Function Unit Testing via `_pure` Namespace
**What:** All testable functions are exported via `module.exports._pure` in manage-agents-core.cjs. Tests import from `_pure` and call functions with controlled inputs.
**When to use:** For `buildDashboardLines` and `formatTimestamp` tests.
**Confidence:** HIGH — this is the established project pattern used by all 181 existing tests.

```javascript

const { buildDashboardLines, formatTimestamp } = _pure;

test('buildDashboardLines: renders slot with healthy probe', () => {
  const slots = ['claude-1'];
  const mcpServers = { 'claude-1': { env: { ANTHROPIC_BASE_URL: 'https://api.akashml.com/v1', CLAUDE_DEFAULT_MODEL: 'deepseek-v3' } } };
  const healthMap = { 'claude-1': { healthy: true, latencyMs: 42, statusCode: 200 } };
  const lines = buildDashboardLines(slots, mcpServers, healthMap, Date.now());
  // Assert specific content

```

### Pattern 2: Non-TTY Smoke Test for liveDashboard
**What:** When `process.stdout.isTTY` is falsy (CI/test runner), `liveDashboard` takes the non-interactive path: probe once, render once, return. No raw mode, no blocking.
**When to use:** For integration smoke test of liveDashboard without raw-mode complications.
**Confidence:** HIGH — existing test at line 1287 already uses this pattern.

```javascript
test('liveDashboard smoke: non-TTY path resolves without throwing', async () => {
  const { liveDashboard } = _pure;
  if (process.stdout.isTTY) return; // Skip in real TTY
  await assert.doesNotReject(liveDashboard());
});
```

### Anti-Patterns to Avoid
- **Raw mode in test runner:** Never call `setRawMode(true)` inside node:test — it will hang the test runner waiting for keypress input.
- **ANSI comparison with `===`:** Use `.includes()` for ANSI-colored strings; exact byte sequences vary across terminals.
- **Timezone-dependent assertions:** `formatTimestamp` uses local timezone via `toTimeString()` — test format (length, colon positions) not exact values.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ANSI color testing | Custom ANSI parser | String `.includes()` checks for known escape codes | ANSI sequences are standard; `\x1b[32m` = green, `\x1b[31m` = red, `\x1b[33m` = yellow, `\x1b[90m` = dim |
| TTY mocking | Full TTY emulator | Non-TTY branch + subprocess spawn for TTY tests | liveDashboard already has non-TTY fallback path |
| Health probe mocking | Mock HTTP server | Direct function call with fixture data | buildDashboardLines takes healthMap object — no HTTP needed |

**Key insight:** The pure-function architecture of buildDashboardLines means most DASH requirements can be verified by calling the function with fixture data and checking the output strings. No HTTP mocking, no TTY simulation, no stdin manipulation needed for the core tests.

## Common Pitfalls

### Pitfall 1: Test Runner Raw Mode Hang
**What goes wrong:** Tests that trigger `setRawMode(true)` hang indefinitely because node:test doesn't provide keypress input.
**Why it happens:** liveDashboard enters raw mode when `process.stdout.isTTY` is true.
**How to avoid:** Guard with `if (process.stdout.isTTY) return;` or test only the non-TTY path. For TTY testing, use `child_process.spawn` in a subprocess.
**Warning signs:** Test suite hangs after "liveDashboard" test name appears.

### Pitfall 2: Stale Boundary Race Condition
**What goes wrong:** Stale threshold test fails intermittently because `Date.now()` drifts between test setup and function call.
**Why it happens:** `buildDashboardLines` calls `Date.now()` internally to compare against `lastUpdated`.
**How to avoid:** Use extreme values (70_000ms past for stale, 10_000ms for fresh) rather than boundary values in most tests. Existing boundary test at line 898 uses 60_001ms and 59_999ms margins which are safe.
**Warning signs:** Flaky test failures on CI with "expected stale but got fresh" or vice versa.

### Pitfall 3: ANSI Escape Code Assertion Brittleness
**What goes wrong:** String equality assertions fail when ANSI reset sequences change.
**Why it happens:** `\x1b[0m` (reset) appears at end of colored strings; different terminal contexts may add wrapping.
**How to avoid:** Use `.includes('\x1b[32m')` (contains green) and `.includes('\u2713')` (contains checkmark) rather than exact string match.
**Warning signs:** Tests pass locally but fail in CI with string mismatch on ANSI sequences.

# Phase v0.26-02: Credential Management - Research

**Researched:** 2026-03-03
**Domain:** API key rotation, credential health persistence, async batch operations

## Summary

This phase adds credential management capabilities on top of the v0.26-01 policy foundation. The core requirement is two-fold:
1. **CRED-01:** Batch API key rotation from main menu (currently a sequential TUI flow already exists, but needs async/non-blocking validation)
2. **CRED-02:** Key validity status persists to `qgsd.json` after health probes (infrastructure 90% in place; needs wiring to health check flows and batch rotation)

The codebase has nearly all infrastructure in place: `writeKeyStatus()` exists, `probeProviderUrl()` works, `probeAllSlots()` runs in parallel, and the batch rotation TUI flow is implemented. The work is primarily **wiring existing functions together** and ensuring status persists and survives across sessions without re-probing.

**Primary recommendation:** Focus on integrating existing `writeKeyStatus()` calls into the health probe flow (`probeAllSlots` and during batch rotation), and ensure the dashboard can read persisted status from `qgsd.json` to display without re-probing.

### CRED-01: Batch Rotation Flow (Mostly Complete)

**Status:** Core flow exists but needs async safety validation.

**Current flow logic:**

**Gaps for CRED-01 requirements:**
1. **No per-slot validation:** After entering a key, the flow doesn't verify the key is valid (401 check) before persisting. This is acceptable UX (key validation is async and can happen later), but if we want immediate feedback, we need `probeProviderUrl()` call post-entry.

   - Option A: Keep sync but validate keys async in background (fire-and-forget after rotation completes)
   - Option B: Add a "validate keys now?" prompt at end of batch, then run async validation
   - Option C: Do nothing — current sync flow is fast enough (keytar write ~10ms per slot)

**Evidence:** Line 1272 shows write happens immediately if `!secrets` (env-based storage), suggesting sync is expected. Keytar writes are negligible latency.

**Recommendation:** CRED-01 as-is meets requirement. No blocking detected. If validation is needed, implement post-flow async validation (doesn't block rotation completion).

### Pattern 1: Health Probe with Persistence

**What:** Call probeProviderUrl or probeAllSlots, then write status to qgsd.json for persistence.

**When to use:** Any health check endpoint (dashboard, batch rotation, manual "check health" action, auto-check during quorum dispatch)

**Example (sync single-slot probe with persistence):**
```javascript
// bin/manage-agents-core.cjs — add function (or enhance probeProviderUrl):
async function probeAndPersistKey(slotName, baseUrl, apiKey) {
  const probe = await probeProviderUrl(baseUrl, apiKey);
  if (probe.healthy || probe.statusCode) {  // Skip 'unreachable' (provider down, not key issue)
    const status = classifyProbeResult(probe);
    writeKeyStatus(slotName, status);  // CRED-02: Persist status
  }
  return probe;
}
```

**Example (async batch probe with persistence):**
```javascript
// Enhance probeAllSlots to write status:
async function probeAllSlots(mcpServers, slots, secretsLib) {
  const results = await Promise.all(slots.map(async (slotName) => {
    const cfg = mcpServers[slotName] || {};

      return [slotName, { healthy: null, ... }];
```

### Pattern 3: Reading Persisted Status (No Re-probe)

**What:** Display key status from qgsd.json without calling health endpoints (use this for fast startup).

**When to use:** Dashboard initial load, agent list display, anywhere you want cached status without network latency.

**Example:**
```javascript
function getPersistedKeyStatus(slotName, qgsdData) {
  const slotCfg = (qgsdData.agent_config || {})[slotName] || {};
  const ks = slotCfg.key_status;
  if (!ks) return null;  // No cached status

  const age = Date.now() - new Date(ks.checkedAt).getTime();
  const ageMinutes = Math.floor(age / 60000);

  return {
    status: ks.status,      // 'ok' | 'invalid' | 'unreachable'
    checkedAt: ks.checkedAt,
    ageMinutes,
  };
}
```

### Anti-Patterns to Avoid

- **Don't probe on every dashboard render:** Probing 12 slots sequentially takes ~84 seconds (7s timeout × 12). Cache results and only re-probe on user request (space / r keypress). ✅ `renderHealth()` already does this correctly.

- **Don't write status for 'unreachable' verdict:** If a provider endpoint is down (timeout), don't write status. It's not a key validity issue. Only write if provider responded (healthy OR statusCode present). ✅ Invariant to maintain in all new writes.

- **Don't block rotation on validation:** CRED-01 says "Batch rotation does not block quorum dispatch." If we add validation, make it fire-and-forget (async background task), not part of the rotation completion wait.

- **Don't truncate qgsd.json when writing status:** `writeKeyStatus()` uses the correct pattern: read full qgsd.json, update one key, write full qgsd.json back. ✅ Already correct in existing code.

## Common Pitfalls

### Pitfall 1: Writing Status for Unreachable Providers

**What goes wrong:** After probing a slot, you write `writeKeyStatus(slotName, 'unreachable')`. Later, the provider recovers, but the cached status still says 'unreachable'. User sees stale error.

**Why it happens:** Trying to be exhaustive — "capture all states." But 'unreachable' is not a key validity issue; it's a provider/network issue.

**How to avoid:** Only call `writeKeyStatus()` if `probe.healthy === true OR probe.statusCode !== null`. Filter out timeouts/network errors (where statusCode is null).

**Invariant:** In the code:
```javascript
if (probe.healthy || probe.statusCode) {  // statusCode present = provider responded
  const status = classifyProbeResult(probe);
  writeKeyStatus(slotName, status);
}
// Don't write if timeout/network error (statusCode is null)
```

**Warning signs:** Test shows status persisted as 'unreachable' after probe timeout; user manually checks again next session and status is stale.

### Pitfall 2: Blocking Batch Rotation with Validation

**What goes wrong:** User rotates 5 keys. Code waits for validation on each one (7s timeout per slot = 35s total). Meanwhile, quorum dispatch waits for rotation to finish. Quorum times out.

**Why it happens:** Wanting immediate feedback — "user rotates key, should know if it's valid right away."

**How to avoid:** If validation is needed, make it **fire-and-forget** after rotation completes. Don't await the validation inside the rotation loop. Rotation finishes fast (keytar write ~10ms), validation happens async in background.

**Code pattern:**
```javascript
// ✅ GOOD: Async validation doesn't block rotation completion
await secretsLib.set('qgsd', account, newKey);
rotated.push(picked.value);
// Validation happens later, doesn't block:
validateRotatedKeys(rotated, ...).catch(() => {});

// ❌ BAD: Validation blocks rotation
await secretsLib.set('qgsd', account, newKey);
const probe = await probeProviderUrl(...);  // 7s wait per slot!
writeKeyStatus(...);
rotated.push(picked.value);
```

**Warning signs:** Batch rotation takes >30 seconds for 5 keys; quorum dispatch waits and times out.

### Pitfall 3: Not Preserving Existing qgsd.json Keys

**What goes wrong:** You call `writeKeyStatus()` but accidentally pass an empty object instead of the existing qgsd.json data. Now `agent_config`, `orchestrator`, and other keys are gone.

**Why it happens:** Forgetting that `writeQgsdJson()` overwrites the entire file. You must read the full file first.

**How to avoid:** Always use the pattern:
```javascript
const qgsd = readQgsdJson(filePath);  // Read existing data
// Mutate qgsd (add/update keys)
qgsd.agent_config[slotName].key_status = { status, checkedAt };
writeQgsdJson(qgsd, filePath);  // Write full file
```

✅ `writeKeyStatus()` already does this correctly (see line 190-198).

**Warning signs:** After a key status update, user's qgsd.json loses orchestrator config or other top-level keys.

### Pitfall 4: Calling writeKeyStatus Without Async Await

**What goes wrong:** You call `probeProviderUrl()` (async), then immediately call `writeKeyStatus()` before the probe completes. Status written is undefined or stale.

**Why it happens:** Forgetting that `probeProviderUrl()` returns a Promise.

**How to avoid:** Always await probes before writing status:
```javascript
// ❌ BAD
const probe = probeProviderUrl(...);  // Returns Promise, not awaited
writeKeyStatus(slotName, classifyProbeResult(probe));  // probe is still a Promise!

// ✅ GOOD
const probe = await probeProviderUrl(...);  // Wait for result
writeKeyStatus(slotName, classifyProbeResult(probe));
```

**Warning signs:** `classifyProbeResult()` receives a Promise object; returns undefined status.

### Pitfall 5: Dashboard Doesn't Update When Status Changes

**What goes wrong:** User rotates a key and validates it (status now 'ok'). Dashboard still shows old status (e.g., 'invalid'). User doesn't know the validation worked.

**Why it happens:** Dashboard cached the status and doesn't re-fetch after batch rotation completes.

**How to avoid:** If batch rotation validates keys in the background, dashboard should either:
1. Re-fetch status after rotation completes (call `probeAllSlots()` again)
2. Or mark as "status pending validation" until background task finishes
3. Or just accept that status will update on next dashboard open (fire-and-forget, eventual consistency)

**Code pattern:**
```javascript
if (rotated.length) {
  toast(`✓ ${rotated.length} slot(s) rotated`);
  // Option: Fire async validation
  validateRotatedKeys(rotated, servers, secrets)
    .then(() => {
      // Status now written; if user opens dashboard soon, they'll see updated status
    })
    .catch(() => {});
}
```

**Warning signs:** User rotates key, validation passes, but dashboard still shows old status until they manually refresh (space key).

### Reading and Writing to qgsd.json

```javascript
// Read qgsd.json safely (returns {} if missing)
function readQgsdJson(filePath) {
  const p = filePath || QGSD_JSON_PATH;
  if (!fs.existsSync(p)) return {};
  try {
    return JSON.parse(fs.readFileSync(p, 'utf8'));
  } catch (_) {
    return {};
  }
}

// Write qgsd.json with atomic rename (temp file → rename)
function writeQgsdJson(data, filePath) {
  const p = filePath || QGSD_JSON_PATH;
  const tmp = p + '.tmp';
  fs.writeFileSync(tmp, JSON.stringify(data, null, 2), 'utf8');
  fs.renameSync(tmp, p);
}

// Example: Update a single slot's config without truncating other keys
function writeUpdatePolicy(slotName, policy, filePath) {
  const qgsd = readQgsdJson(filePath);
  if (!qgsd.agent_config) qgsd.agent_config = {};
  if (!qgsd.agent_config[slotName]) qgsd.agent_config[slotName] = {};
  qgsd.agent_config[slotName].update_policy = policy;
  writeQgsdJson(qgsd, filePath);
}
```

### Probing a Provider and Classifying Result

**Source:** manage-agents-core.cjs lines 125-185

```javascript
// Probe a provider endpoint (7s timeout, returns structured result)
async function probeProviderUrl(baseUrl, apiKey) {
  return new Promise((resolve) => {
    if (!baseUrl) return resolve({ healthy: false, latencyMs: 0, statusCode: null, error: 'No URL provided' });
    let url;
    try {
      const base = baseUrl.endsWith('/') ? baseUrl.slice(0, -1) : baseUrl;
      url = new URL(base + '/models');
    } catch {
      return resolve({ healthy: false, latencyMs: 0, statusCode: null, error: `Invalid URL: ${baseUrl}` });
    }
    const lib = url.protocol === 'https:' ? https : http;
    const start = Date.now();
    const req = lib.request(
      {
        hostname: url.hostname,
        port: url.port || (url.protocol === 'https:' ? 443 : 80),
        path: url.pathname + url.search,
        method: 'GET',
        headers: {
          'Authorization': apiKey ? `Bearer ${apiKey}` : '',
          'Accept': 'application/json',
          'User-Agent': 'qgsd-manage-agents/1.0',
        },
        timeout: 7000,
      },
      (res) => {
        const latencyMs = Date.now() - start;
        res.resume();
        res.on('end', () => {
          const healthy = [200, 401, 403, 404, 422].includes(res.statusCode);
          resolve({ healthy, latencyMs, statusCode: res.statusCode, error: null });
        });
      }
    );
    req.on('error', (e) => {
      resolve({ healthy: false, latencyMs: Date.now() - start, statusCode: null, error: e.message });
    });
    req.on('timeout', () => {
      req.destroy();
      resolve({ healthy: false, latencyMs: Date.now() - start, statusCode: null, error: 'Timed out after 7000ms' });
    });
    req.end();
  });
}

function classifyProbeResult(probeResult) {

    return 'unreachable';
  }
  if (probeResult.statusCode === 401) {
```

```javascript
async function probeAllSlots(mcpServers, slots, secretsLib) {
  // Fire all probes in parallel using Promise.all
  const results = await Promise.all(slots.map(async (slotName) => {
    const cfg = mcpServers[slotName] || {};
    const env = cfg.env || {};
    if (!env.ANTHROPIC_BASE_URL) {
      return [slotName, { healthy: null, latencyMs: 0, statusCode: null, error: 'subprocess' }];
    }
    const account = deriveKeytarAccount(slotName);
    let apiKey = env.ANTHROPIC_API_KEY || '';
    if (secretsLib) {
      try {
        const k = await secretsLib.get('qgsd', account);
        if (k) apiKey = k;
      } catch (_) {}
    }
    const probe = await probeProviderUrl(env.ANTHROPIC_BASE_URL, apiKey);
    return [slotName, probe];
  }));
  return Object.fromEntries(results);
}
```

```javascript
async function validateRotatedKeys(rotatedSlots, mcpServers, secretsLib) {

  }
```

## Open Questions

1. **Should post-rotation validation be mandatory or optional?**
   - What we know: Validation adds 7s per slot; batch rotation is typically 2-5 keys; user experience is better with immediate feedback.
   - What's unclear: Should validation be automatic (all users wait 7-35s) or opt-in (checkbox at end of batch)?
   - Recommendation: Make it opt-in (default OFF) with a checkbox "Validate keys after rotation?" Allows power users to enable it; casual users get fast rotation without wait.

2. **Should persisted status expire (e.g., >24h old becomes stale)?**
   - What we know: Status is stored with `checkedAt` timestamp; could implement expiry logic.
   - What's unclear: Is 24h stale status acceptable? Or should status stay fresh for entire session?
   - Recommendation: Show "last checked Xm ago" badge on dashboard. Don't expire status; let users manually refresh (space key).

3. **Should batch rotation block if any slot fails validation?**
   - What we know: Current implementation rotates all slots regardless of validation outcome.
   - What's unclear: Should one failed validation prevent other rotations?
   - Recommendation: Continue rotating regardless of validation outcome (fail-open). Validation is informational (fire-and-forget).

4. **Integration with quorum dispatch timeout?**
   - What we know: Quorum dispatch has CLAUDE_MCP_TIMEOUT_MS (default 120s); batch rotation is ~100ms per key (keytar write).
   - What's unclear: Is async validation timeout possible? (Validation could take 35s for 5 keys; if quorum dispatch runs after, timeout risk?)
   - Recommendation: Ensure validation is truly fire-and-forget (don't await in any caller). Validation updates qgsd.json; quorum dispatch reads it on next cycle.

## Summary

## Technical Design

### 1. Pair Detection

**Requirement prefix pairing:** Extract the prefix from each requirement ID (everything before the last `-` and digits, e.g., `DETECT-01` -> `DETECT`, `CRED-07` -> `CRED`). Two models are candidates if they share at least one requirement prefix.

**Source file pairing:** Two models are candidates if their `source_files` arrays have a non-empty intersection. Currently all arrays are empty, so this produces zero pairs — but the code must handle it for future use.

**Algorithm:**
```
for each unique pair (modelA, modelB) where A < B (alphabetical):
  sharedPrefixes = intersection(prefixes(A.requirements), prefixes(B.requirements))
  sharedSourceFiles = intersection(A.source_files, B.source_files)
  if sharedPrefixes.length > 0 OR sharedSourceFiles.length > 0:
    candidatePair(A, B, sharedPrefixes, sharedSourceFiles)
```

### 2. Merged State-Space Estimation

For a candidate pair (A, B):
- If both have `estimated_states` (non-null): `merged = A.estimated_states * B.estimated_states`
- If either is null (unbounded/unresolvable): `merged = null`

### 3. TLC Runtime Budget

The 5-minute (300s) budget maps to a state-space threshold via TLC throughput:
- Conservative estimate: ~10,000 states/second for typical TLA+ models
- 300s * 10,000 = 3,000,000 states threshold
- This constant (`MERGE_BUDGET_STATES`) should be configurable at the top of the file

Decision rule:
- `merged <= MERGE_BUDGET_STATES` -> recommend **merge**
- `merged > MERGE_BUDGET_STATES` or `merged === null` -> flag **interface-contract**

### 4. Interface Contract Flagging

When merge is not recommended, the output should include:
- Shared requirement IDs (the intersection of requirement arrays)
- Shared requirement prefixes
- Shared source files (if any)
- Estimated merged states (or "unbounded")
- Recommendation: "interface-contract" with rationale

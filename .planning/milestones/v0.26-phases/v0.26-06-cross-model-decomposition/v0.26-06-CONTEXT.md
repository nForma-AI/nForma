# Phase v0.26-06: Cross-Model Decomposition - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend `analyze-state-space.cjs` to identify TLA+ model pairs that share source files or requirement prefixes, estimate merged state-space, and recommend merge when combined TLC runtime stays under 5 minutes — or flag the interface contract needed when merge would exceed the time budget. Requirement: DECOMP-05.

</domain>

<decisions>
## Implementation Decisions

### Pair detection criteria
- Cross-reference `model-registry.json` entries: two models are candidates if they share one or more source files in their `source_files` arrays
- Also pair models whose requirement IDs share a common prefix (e.g., `DECOMP-01` and `DECOMP-02` share `DECOMP`)
- Both heuristics contribute independently — either match produces a candidate pair

### Output format
- Extend existing `state-space-report.json` with a new `cross_model` top-level key (sibling to `models` and `summary`)
- `cross_model` contains an array of pair analysis objects, each with: model pair, shared files, shared requirements, estimated merged states, recommendation (merge/interface-contract), and rationale
- No separate output file — keeps the single-report pattern

### TLC runtime estimation
- Use state-space product of the two models as proxy for merged TLC runtime
- The 5-minute budget maps to a state-space threshold (configurable constant, derived from empirical TLC throughput)
- If either model has unbounded state-space (null estimate), recommend interface contract by default

### Interface contract flagging
- When merged state-space exceeds the budget threshold, output: shared variables (intersection of VARIABLES blocks), shared requirement IDs, estimated merged states, and a recommendation to define an interface contract
- When merged state-space fits within budget, output: merge recommendation with estimated states and shared context

### Claude's Discretion
- Exact state-space-to-runtime conversion factor
- Output ordering and sorting of candidate pairs
- Whether to add a `--cross-model` CLI flag or always include cross-model analysis

</decisions>

<specifics>
## Specific Ideas

- The existing tool already parses `model-registry.json`, extracts VARIABLES, and estimates per-model state-space — cross-model analysis reuses all of this infrastructure
- The `requirements` field in `model-registry.json` entries provides the requirement-prefix pairing data
- Follow the existing pattern: pure functions for analysis, `main()` for CLI orchestration, `--json` and `--quiet` flags

</specifics>

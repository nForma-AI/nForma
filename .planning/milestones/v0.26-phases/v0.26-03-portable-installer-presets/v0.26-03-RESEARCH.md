# Phase v0.26-03: Portable Installer & Presets - Research

**Researched:** 2026-03-03
**Domain:** Configuration management, portable data export/import, CLI provisioning
**Confidence:** HIGH

## Summary

Phase v0.26-03 adds portability and presets to the QGSD roster management system. It requires implementing three major features: (1) exporting and importing full roster configs with redacted API keys and validation, (2) provider preset selection for faster agent provisioning, and (3) slot cloning. The codebase already has 80% of the foundation in place — export/import helpers exist in `manage-agents-core.cjs`, provider presets are hardcoded, and clone logic is partially implemented. The main work is integrating these into the TUI flows, removing hardcoded `get-shit-done/` path references (REN-03), and adding comprehensive validation/testing.

**Primary recommendation:** Leverage existing pure functions in `manage-agents-core.cjs._pure` namespace; integrate export/import into blessed TUI via menu actions; create test suite that validates schema, backup creation, and redaction patterns before implementation.

## Architecture Patterns

### Recommended Project Structure

Export/import operations follow this existing pattern:

```
bin/manage-agents-core.cjs
├── readClaudeJson()              # Read ~/.claude.json
├── writeClaudeJson(data)         # Write ~/.claude.json with atomic rename
├── readQgsdJson(filePath)        # Read ~/.claude/qgsd.json (agent config metadata)
├── writeQgsdJson(data, filePath) # Write qgsd.json atomically
└── _pure = {
    buildExportData()             # Strip API keys → __redacted__
    validateImportSchema()        # Schema validation before apply
    buildBackupPath()             # Generate timestamped backup path
    buildRedactedEnv()           # Redact single env object
    buildCloneEntry()            # Clone slot configuration
    findPresetForUrl()           # Detect preset match for provider URL
}

qgsd.cjs (TUI entry point)
├── MENU_ITEMS: action='export' | action='import' | action='clone'
└── Interactive flows calling core functions
```

### Pattern 1: Atomic File Write
**What:** Write to temp file, then rename atomically to target. Prevents partial corruption if process crashes mid-write.

**When to use:** All config file updates (`.claude.json`, `qgsd.json`, backups)

**Example:**
```javascript
// Source: manage-agents-core.cjs lines 69-72
function writeClaudeJson(data) {
  const tmp = CLAUDE_JSON_PATH + '.tmp';
  fs.writeFileSync(tmp, JSON.stringify(data, null, 2), 'utf8');
  fs.renameSync(tmp, CLAUDE_JSON_PATH);  // atomic
}
```

### Pattern 2: Pure Transformation Functions (Namespace: `_pure`)
**What:** All data transformations exported in `module.exports._pure` namespace for testability.

**When to use:** Export/import logic, validation, schema building, backup path generation

**Why:** Tests can call pure functions directly without mocking file I/O; keeps side effects out of transformation logic.

**Example:**
```javascript
// Source: manage-agents-core.cjs lines 573-580
function buildExportData(claudeJsonData) {
  const out = JSON.parse(JSON.stringify(claudeJsonData));  // deep clone
  const servers = out.mcpServers || {};
  for (const cfg of Object.values(servers)) {
    if (cfg.env) cfg.env = buildRedactedEnv(cfg.env);
  }
  return out;
}
```

### Pattern 3: Redaction via Regex Matching
**What:** Identify sensitive keys by naming convention, replace values with `__redacted__` placeholder.

**When to use:** Export operation; keeps exported JSON portable without leaking credentials

**Regex pattern:**
```javascript
// Source: manage-agents-core.cjs lines 567-570
const SENSITIVE_RE = /(_KEY|_SECRET|_TOKEN|_PASSWORD)$/i;
// Matches: ANTHROPIC_API_KEY, TOGETHER_API_KEY, OPENAI_API_KEY, etc.
```

### Pattern 4: Schema Validation with Error Collection
**What:** Collect all schema violations into an errors array, return all at once (not fail-fast).

**When to use:** Import validation; allows user to see all problems and fix in one pass

**Example:**
```javascript
// Source: manage-agents-core.cjs lines 582-607
function validateImportSchema(parsed) {
  const errors = [];
  // ... collect all violations ...
  return errors;  // empty = valid
}
```

### Pattern 5: Timestamped Backup Path Generation
**What:** Append `.pre-import.ISO_TIMESTAMP` to original path. Keeps backups sortable by time.

**When to use:** Before applying import, create restore point

**Example:**
```javascript
// Source: manage-agents-core.cjs lines 561-563
function buildBackupPath(claudeJsonPath, isoTimestamp) {
  return claudeJsonPath + '.pre-import.' + isoTimestamp;
}
// Result: ~/.claude.json.pre-import.2026-03-03T14:30:45.123Z
```

### Anti-Patterns to Avoid

- **Don't inline validation logic in TUI:** Validation belongs in pure functions; TUI calls them. (Testing becomes impossible otherwise.)
- **Don't skip atomic writes:** Direct fs.writeFileSync() without temp file = corruption risk.
- **Don't hardcode home paths in config:** Template files should use `~` or `${HOME}`; at runtime, expand with `os.homedir()`.
- **Don't mix env keys (ANTHROPIC_API_KEY) with provider-map keys:** env vars store values; provider records store metadata. Keep them separate.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema validation | Custom recursion | `validateImportSchema()` existing function + Regex tests | Existing function handles edge cases (home paths, command whitelist); tested |
| Secure credential storage | Env vars for API keys | `secrets.cjs` (keytar wrapper) for keychain + env fallback | Keychain survives restarts; env for CI/subprocess |
| Config file atomicity | Direct writes | writeClaudeJson() temp+rename pattern | Process crash mid-write = corrupted config; temp+rename is atomic at OS level |
| Provider preset mapping | Hardcoded if-else | `findPresetForUrl()` function | Single source of truth; easier to extend when adding new providers |

**Key insight:** Portable configs must handle three critical scenarios: (1) process crash during write (atomicity), (2) user moving files across machines (no absolute paths), (3) credentials exposed in export (redaction). All three are solved by existing patterns in the codebase.

## Common Pitfalls

### Pitfall 1: Exporting Unredacted Keys
**What goes wrong:** User exports config, shares it in GitHub issue or Slack, accidentally leaks API key.

**Why it happens:** Forgetting to redact before stringifying, or exporting from wrong source.

**How to avoid:**
- Always call `buildExportData(claudeJsonData)` before JSON.stringify()
- That function explicitly redacts via regex before returning
- Add test case: export → reimport → verify redacted keys show as `__redacted__` string literal

**Warning signs:**
- Test output contains actual API key values in exported JSON
- User reports "key exposed" after export

### Pitfall 2: Import Applies Changes Without Backup
**What goes wrong:** User imports bad config, loses previous working setup.

**Why it happens:** Importing directly to ~/.claude.json without creating restore point first.

**How to avoid:**
1. Call `buildBackupPath(CLAUDE_JSON_PATH, new Date().toISOString())` to generate path
2. Copy existing ~/.claude.json to backup path BEFORE applying import
3. Inform user: "Backup created at ~/.claude.json.pre-import.2026-03-03T..."
4. Add test: import flow → verify backup file exists with original content

**Warning signs:**
- No .pre-import files in user's home after imports
- User asks "how do I restore my previous config?"

### Pitfall 3: Schema Validation Allows Dangerous Paths
**What goes wrong:** User imports config containing `/Users/jonathan/path/to/get-shit-done/bin/agent.cjs`, machine breaks on next run.

**Why it happens:** Not validating args arrays for absolute home paths.

**How to avoid:**
- `validateImportSchema()` already blocks `^/(Users|home)/` in args (line 585)
- Test case: try importing config with home path in args → expect error
- Error message should guide user: "Remove absolute paths; use relative paths or $HOME expansion"

**Warning signs:**
- User reports broken slots after import
- Config validation passes but slots fail at runtime

### Pitfall 4: Clone Creates Slot But Doesn't Copy All Config
**What goes wrong:** User clones slot, new slot is missing timeout settings or update policy.

**Why it happens:** `buildCloneEntry()` copies env vars but not qgsd.json metadata (policies, timeouts).

**How to avoid:**
- Clone two places:
  1. Use `buildCloneEntry()` to copy mcpServers config (env vars, command, args)
  2. After user confirms name, also copy qgsd.json agent_config[newName] from source
- Test case: clone slot → verify timeout_ms, update_policy, key_status all copied

**Warning signs:**
- Cloned slot loses timeout setting or policy
- User has to re-configure cloned slot's settings

### Pitfall 5: Provider Preset Doesn't Update When New Provider Launches
**What goes wrong:** User expects AkashML2 provider in preset menu, it's not there.

**Why it happens:** PROVIDER_PRESETS hardcoded in manage-agents-core.cjs (lines 34-38).

**How to avoid:**
- This is deferred design (v0.27+): user-extensible presets
- For v0.26: document that presets are locked; custom URL option available
- Landmark: PROVIDER_PRESETS list in source = "known providers"; if new provider needed, edit here + test
- Future: load from providers.json instead of hardcoding

**Warning signs:**
- User asks "why can't I add provider X from preset menu?"
- Need to cut new release to add provider

### Clone Slot
```javascript
// Source: manage-agents-core.cjs lines 465-478
function buildCloneEntry(sourceCfg, newName) {
  const sourceEnv = (sourceCfg && sourceCfg.env) || {};
  const newEnv = {};
  if (sourceEnv.ANTHROPIC_BASE_URL)    newEnv.ANTHROPIC_BASE_URL    = sourceEnv.ANTHROPIC_BASE_URL;
  if (sourceEnv.CLAUDE_DEFAULT_MODEL)  newEnv.CLAUDE_DEFAULT_MODEL  = sourceEnv.CLAUDE_DEFAULT_MODEL;
  if (sourceEnv.CLAUDE_MCP_TIMEOUT_MS) newEnv.CLAUDE_MCP_TIMEOUT_MS = sourceEnv.CLAUDE_MCP_TIMEOUT_MS;
  newEnv.PROVIDER_SLOT = newName;
  return {
    type: (sourceCfg && sourceCfg.type) || 'stdio',
    command: (sourceCfg && sourceCfg.command) || 'node',
    args: (sourceCfg && sourceCfg.args) ? [...sourceCfg.args] : [],
    env: newEnv,
  };
}
```

### Provider Preset Selection
```javascript
// Source: manage-agents-core.cjs lines 34-38
const PROVIDER_PRESETS = [
  { name: 'AkashML',       value: 'https://api.akashml.com/v1',             label: 'AkashML       (api.akashml.com/v1)' },
  { name: 'Together.xyz',  value: 'https://api.together.xyz/v1',            label: 'Together.xyz  (api.together.xyz/v1)' },

];

// Find preset for URL:
function findPresetForUrl(url) {
  if (!url) return '__custom__';
  const found = PROVIDER_PRESETS.find((p) => p.value === url);
  return found ? found.value : '__custom__';
}
```

## Open Questions

1. **Should import prompt for each redacted key individually, or show a list?**
   - What we know: Current code has no interactive key re-entry flow
   - What's unclear: UX preference for batch vs. individual prompts
   - Recommendation: Show list of redacted keys, let user choose which to re-enter (faster than asking for all). Defer batch-prompt pattern to v0.27 if needed.

2. **Should cloned slots inherit the source slot's auth_type (sub vs. api)?**
   - What we know: buildCloneEntry() copies env vars but not qgsd.json metadata
   - What's unclear: Whether auth_type should follow provider or be independent
   - Recommendation: Clone should copy source slot's auth_type from qgsd.json agent_config[source]; user can override if different provider. Implement as: copy metadata AFTER user confirms new name.

3. **What path should backup files be created at?**
   - What we know: buildBackupPath() generates ~/.claude.json.pre-import.ISO_TIMESTAMP
   - What's unclear: Whether to create backups in same directory or .claude/backups/
   - Recommendation: Same directory (~/) for v0.26 — simpler UX, easier to discover. User sees backup in home. Future: auto-cleanup old backups (>7 days) could use dedicated directory.

4. **Should import reject configs with unrecognized slot names that conflict with existing slots?**
   - What we know: validateImportSchema() checks structure, not uniqueness
   - What's unclear: Conflict resolution strategy (skip, overwrite, rename)
   - Recommendation: For v0.26, reject import if any slot name already exists. Show user: "Slots x, y already exist; either remove them first or edit your import file." Defer merge/overwrite patterns to v0.27.

5. **How should hardcoded paths in REN-03 be handled across workflow files?**
   - What we know: Grep found zero `get-shit-done/` in bin/*.cjs
   - What's unclear: Whether qgsd-core/workflows/* or .github/workflows/* contain hardcoded paths
   - Recommendation: Grep those directories now during research; if found, add removal task to plan. If none found, add smoke test to verify zero matches in future builds.

# Phase v0.26-05: Architecture Enforcement - Research

**Researched:** 2026-03-03
**Domain:** Static analysis, CI enforcement, import scanning
**Confidence:** HIGH

## Summary

Phase v0.26-05 enforces QGSD's architectural constraint: the plugin MUST NOT bundle LLM SDKs. Instead, all Haiku/Sonnet/Opus calls use the Agent tool's `model` parameter to delegate to Claude Code's quorum dispatch layer. The codebase currently has **two direct SDK imports** that must be refactored: (1) `update-scoreboard.cjs` uses `@anthropic-ai/sdk` for Haiku classification (lines 454-455, 482), and (2) `validate-requirements-haiku.cjs` imports the Anthropic SDK at module load (line 30). Both use fail-open patterns (return null if SDK unavailable), but direct imports are architecturally unsound for a plugin. The phase work: (1) refactor both files to use subprocess-based Agent tool dispatch instead, (2) implement `check-bundled-sdks.cjs` CI linter to prevent future violations, (3) add comprehensive test coverage for SDK detection and whitelist logic.

**Primary recommendation:** Leverage subprocess dispatch pattern already used in `call-quorum-slot.cjs`; create Agent tool wrappers for Haiku calls; implement linter with realistic false-positive scoping (test/ and docs/ excluded); baseline false positives on clean codebase before enabling CI gate.

## Architecture Patterns

### Recommended Project Structure

SDK enforcement follows a **scoped static analysis** pattern:

```
bin/check-bundled-sdks.cjs          # Main linter executable
├── scanSourceFiles()               # Glob + read files in scope
├── detectSdkImports()              # Regex match patterns
├── validateWhitelist()             # Known exceptions (provider integrations)
└── reportViolations()              # JSONL output format

bin/check-bundled-sdks.test.cjs     # Test suite
├── Test fixtures (.test.cjs files)
└── Whitelist validation tests

.formal/architecture-baseline.jsonl  # Baseline violations on clean repo
└── Persisted after first run (v0.26-05 start)

.formal/architecture-linter-config.json  # Scoping rules
├── scope: ["bin/**", "hooks/**"]
├── exclude: ["**/*.test.cjs", "docs/**"]
└── whitelist: [patterns for provider integrations]
```

### Pattern 1: Scoped Scanning with Glob Matching

**What:** Restrict linter to core plugin code; exclude test files, docs, vendored code

**When to use:** Architecture linting where false positives in test/doc are expected and acceptable

**Example:**
```javascript
// Source: to be implemented in check-bundled-sdks.cjs
const scope = {
  include: ["bin/**/*.{js,cjs}", "hooks/**/*.js"],
  exclude: ["**/*.test.cjs", "docs/**", "node_modules/**", "**/*.example.js"]
};

function scanSourceFiles() {
  // Use tinyglobby to match include patterns
  const files = glob.sync(scope.include, { ignore: scope.exclude });
  return files;
}
```

**Why:** Prevents linter from flagging SDK references in test fixtures, documentation examples, or vendored dependencies. Keeps signal-to-noise ratio high.

### Pattern 2: Regex-Based Import Detection

**What:** Match `require()` and `import` statements against forbidden SDK names

**When to use:** Fast, lightweight import detection; no need for AST parsing

**Patterns to detect:**
```javascript
// Anthropic SDK
/require\s*\(\s*['"]@anthropic-ai\/sdk['\"]\s*\)/
/require\s*\(\s*['"]@anthropic-ai\/sdk.*['\"]\s*\)/
/import\s+.*\s+from\s+['"]@anthropic-ai\/sdk['"]/

// OpenAI SDK
/require\s*\(\s*['"]openai['\"]\s*\)/
/import\s+.*\s+from\s+['"]openai['"]/

// Google Generative AI
/require\s*\(\s*['"]@google\/generative-ai['\"]\s*\)/
/import\s+.*\s+from\s+['"]@google\/generative-ai['"]/

// Forbidden list (comprehensive)
const FORBIDDEN_SDKS = [
  '@anthropic-ai/sdk',
  'anthropic',
  'openai',
  '@google/generative-ai',
  'google-generativeai',
  'cohere-ai',
  '@azure/openai',
  'huggingface-js',
  'aws-sdk',  // if LLM calls ever use Bedrock
];
```

**Why:** Regex avoids AST complexity; catches most common patterns without false negatives.

### Pattern 3: Baseline & Grandfathering

**What:** Record known violations on clean codebase; flag only NEW violations on subsequent runs

**When to use:** Linting rules deployed to existing codebases with legacy violations

**Implementation:**
```javascript
// On first run (v0.26-05):
function generateBaseline(violations) {
  const baseline = violations.map(v => ({
    file: v.file,
    line: v.line,
    import: v.import,
    reason: 'grandfathered-v0.26-05-baseline',
    timestamp: new Date().toISOString()
  }));
  fs.writeFileSync('.formal/architecture-baseline.jsonl', baseline.map(b => JSON.stringify(b)).join('\n'));
}

// On subsequent runs:
function checkViolations(violations) {
  const baseline = loadBaseline();  // read .formal/architecture-baseline.jsonl
  const newViolations = violations.filter(v =>
    !baseline.some(b => b.file === v.file && b.line === v.line)
  );
  return newViolations;  // only report NEW violations
}
```

**Why:** Allows phased enforcement. Phase v0.26-05 refactors existing violations; CI gate only blocks NEW violations. This prevents backlog of legacy code from blocking merges while still preventing future regressions.

### Pattern 4: Fail-Open with Detailed Warnings

**What:** Linter runs in CI but only ERRORs on new violations; grandfathered violations show as WARNINGs

**When to use:** Transitioning existing violation linters to production

**Output format:**
```
[LINT] Architecture: SDK Bundling Check
[WARN] Grandfathered violation: bin/update-scoreboard.cjs:454 (refactored in v0.26-05)
[ERROR] New violation: bin/some-new-file.cjs:42 requires 'openai'
[FAIL] 1 new violations found. CI gate blocking.
```

**Why:** Gives visibility to legacy violations while still enforcing no NEW violations.

### Anti-Patterns to Avoid

- **Don't scan node_modules:** Linter falsely flags vendored dependencies; always exclude node_modules/
- **Don't require SDK to be uninstalled:** Linter checks source files only, not npm packages. SDK might remain a transitive dependency (fine); direct imports are what matter.
- **Don't hardcode file paths in linter:** Use glob patterns; supports future refactoring without linter changes.
- **Don't fail on first violation:** Collect all violations and report together; lets user fix all in one pass.
- **Don't blanket-block all SDK references:** Test fixtures and documentation sometimes mention SDK names (e.g., "The Anthropic SDK is deprecated"). Whitelist expected patterns.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Import detection logic | Custom string parsing | Regex patterns against `require()` and `import` statements | Handles escaped quotes, multiline imports, comments |
| AST-based import analysis | Write your own parser | Regex only (sufficient; AST overkill for this use case) | Regex catches 99% of cases; AST complexity not justified |
| File I/O for baseline management | Manual file hacks | `.formal/architecture-baseline.jsonl` with standard JSONL format | JSONL is append-only, sortable, audit trail friendly |
| Glob matching for scoping | Custom file walk | tinyglobby (already in package.json) | Already used; consistent with rest of codebase |

**Key insight:** Architecture linting for SDK bundling is simpler than most linters because it's purely syntactic (detect import statements) not semantic (understand what code does). Regex is sufficient; avoid over-engineering with AST parsing or custom file walkers.

## Common Pitfalls

### Pitfall 1: False Positives from Comments and Strings

**What goes wrong:** Linter flags "anthropic-sdk" in a comment or error message and blocks the merge.

Example:
```javascript
// Deprecated: use Agent tool instead of @anthropic-ai/sdk
const message = 'The Anthropic SDK is no longer supported';
```

Regex naively matches and reports violation.

**Why it happens:** Regex pattern doesn't distinguish between code and comments/strings. Overly broad matching.

**How to avoid:**

1. **Scope scanning to source files only:** Exclude `*.md`, `*.test.cjs`, comments
2. **Use two-pass detection:**
   - First pass: Regex on raw file
   - Second pass: Filter by AST context (is this a `require()` statement or just a string?)
3. **Whitelist known safe patterns:** SDK names in comments, strings, documentation

**Warning signs:**
- Linter flags 50+ violations on clean codebase
- All violations are in test files or comments
- Users add `.architecture-lintignore` files to silence warnings

**Validation in this phase:**
- Run `node bin/check-bundled-sdks.cjs` on clean codebase (v0.26-05 initial state)
- Verify < 5 violations (only the 2 known: update-scoreboard.cjs and validate-requirements-haiku.cjs)
- After refactoring both files, re-run and verify 0 violations
- Add test case for false positive patterns (SDK name in comment → should NOT flag)

### Pitfall 2: Whitelist Too Permissive

**What goes wrong:** Linter's whitelist allows SDK imports in vendor integrations, but a developer uses it in core plugin logic by mistake.

Example:
```javascript
// bin/my-new-feature.cjs
const { Anthropic } = require('@anthropic-ai/sdk');  // WRONG — uses SDK directly
```

Linter whitelist allows it because `bin/**` is a provider integration directory (supposedly). No violation reported. Bug ships to production.

**Why it happens:** Whitelist is too coarse-grained. Can't distinguish between legitimate provider integrations and misuse.

**How to avoid:**

1. **Never whitelist entire directories.** Whitelist specific files only.
   ```javascript
   const WHITELISTED_FILES = [
     'bin/providers.json',          // metadata only
     'bin/provider-mapping.test.cjs' // test fixtures
   ];
   // Reject: whitelist: ["bin/**"]  (too broad)
   ```
2. **Require whitelist entry to list specific SDK + reason.** Comments in whitelist explain WHY each entry is needed.
   ```javascript
   // Examples of whitelist entries (if needed in future):
   // 'bin/call-provider-health.cjs': { sdk: '@anthropic-ai/sdk', reason: 'Direct SDK call for health checks (provider integration)', approved_by: 'Architecture Decision v0.27' }
   ```
3. **Review whitelist quarterly.** Remove entries for code that no longer exists.

**Warning signs:**
- Whitelist grows over time; entries are never removed
- New SDK imports appear in core code without adding whitelist entries (broken enforcement)
- Developer says "It's in bin/, so it's allowed" without checking whitelist

**Validation in this phase:**
- Start with empty whitelist (v0.26-05)
- If whitelist needed for provider integrations, document each entry with approval chain
- Add test: try to import SDK in core file → should FAIL even if directory is whitelisted

### Pitfall 3: Linter Breaks on Invalid Syntax

**What goes wrong:** Source file has a syntax error (unmatched quote, missing semicolon). Linter crashes trying to parse it.

```javascript
// bin/broken-syntax.cjs
const x = "unclosed string
const y = require('@anthropic-ai/sdk');
```

Linter crashes on parsing; CI gate is blocked.

**Why it happens:** Linter doesn't validate syntax first; assumes files are valid JavaScript.

**How to avoid:**

1. **Wrap file reads in try-catch:** Gracefully handle unparseable files
   ```javascript
   function scanFile(filePath) {
     try {
       const content = fs.readFileSync(filePath, 'utf8');
       return detectSdkImports(content);
     } catch (err) {
       // Log and continue; don't crash
       console.error(`[WARN] Could not read ${filePath}: ${err.message}`);
       return [];
     }
   }
   ```
2. **Don't rely on AST parsing.** Use regex only. Regex is forgiving of syntax errors.
3. **Add error handling for violations list:** Collect errors and report at end, not during scan.

**Warning signs:**
- Linter exits with non-zero code on first file read error
- CI job logs show "Cannot parse file" instead of listing violations
- Merges are blocked by linter crash on unrelated syntax error

**Validation in this phase:**
- Add test case: scan file with unclosed quotes → should log warning, not crash
- Verify linter continues scanning other files after error

### Pitfall 4: False Negatives from String Concatenation

**What goes wrong:** SDK import is obfuscated via string concatenation. Linter doesn't detect it.

```javascript
const sdk = require('@anthropic' + '-ai/sdk');  // Bypasses regex check
const lib = require(['@anthropic-ai/sdk'][0]); // Array indexing
```

Regex pattern `/@anthropic-ai\/sdk/` doesn't match. Violation undetected.

**Why it happens:** Regex looks for literal strings; dynamic string construction bypasses it.

**How to avoid:**

1. **Accept that regex has limits.** Dynamic imports are harder to detect statically.
2. **Focus on common patterns:** Catch 99% of violations (literal require/import) and accept false negatives for obfuscated cases. Architectural review catches the rest.
3. **Document the limitation:** Code review policy states "SDK imports are forbidden; no obfuscation tricks."

**Warning signs:**
- Code review finds SDK import that linter didn't flag
- Multiple developers independently try string concatenation tricks

**Validation in this phase:**
- Add test case: literal require() → should detect
- Add test case: string concatenation → EXPECTED to NOT detect (document limitation)
- Rely on code review for obfuscation cases

### Pitfall 5: Integration with CI Gate Fails Silently

**What goes wrong:** Linter runs in CI but its output isn't integrated with the gate. Violations are logged but don't block merge.

```bash
# .github/workflows/lint.yml
- name: Check bundled SDKs
  run: node bin/check-bundled-sdks.cjs
  # No: if: failure()
  # No: exit code check
  # Violations logged but not fatal
```

Developer sees linter output, ignores it (since it didn't fail CI), and merges anyway.

**Why it happens:** Linter is added to CI but developer forgets to connect it to the gate. Or gate logic is wrong.

**How to avoid:**

1. **Explicitly fail on violations:**
   ```bash
   - name: Check bundled SDKs
     run: node bin/check-bundled-sdks.cjs || exit 1
   ```
2. **Test the gate:** Create a test PR with known violation; verify it blocks.
3. **Document the gate in project README:** "Architecture Linter checks for SDK bundling; merge blocked if violations found."

**Warning signs:**
- Linter runs in CI but developers don't see output (not printed to stdout)
- Violations appear in logs but merge still succeeds
- Developer says "I didn't know there was an architecture linter"

**Validation in this phase:**
- After CI integration: create test branch with SDK import, verify merge is blocked
- Verify success criteria: violation prevents merge; refactoring allows merge

### Example 3: Linter Implementation — `check-bundled-sdks.cjs`

```javascript
#!/usr/bin/env node
'use strict';

const fs = require('fs');
const path = require('path');

// Configuration
const FORBIDDEN_SDKS = [
  '@anthropic-ai/sdk',
  'anthropic',
  'openai',
  '@google/generative-ai',
  'google-generativeai',
  'cohere-ai',
  '@azure/openai',
];

// Regex patterns for imports
function buildImportPatterns() {
  return FORBIDDEN_SDKS.map(sdk => {
    const escaped = sdk.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    return new RegExp(
      `require\\s*\\(\\s*['"]${escaped}['"]\\s*\\)|import\\s+.*\\s+from\\s+['"]${escaped}['"]/`,
      'g'
    );
  });
}

// Scoping configuration
const SCOPE = {
  include: ['bin/**/*.{js,cjs}', 'hooks/**/*.js'],
  exclude: ['**/*.test.cjs', 'docs/**', 'node_modules/**', 'examples/**'],
};

function isInScope(filePath) {
  // Simplified: exclude test files and node_modules
  if (filePath.includes('.test.') || filePath.includes('node_modules')) return false;
  if (filePath.includes('/docs/') || filePath.includes('/examples/')) return false;
  return filePath.startsWith('bin/') || filePath.startsWith('hooks/');
}

function scanFile(filePath) {
  const violations = [];
  try {
    const content = fs.readFileSync(filePath, 'utf8');
    const lines = content.split('\n');
    const patterns = buildImportPatterns();

    lines.forEach((line, idx) => {
      patterns.forEach(pattern => {
        if (pattern.test(line)) {
          violations.push({
            file: filePath,
            line: idx + 1,
            content: line.trim(),
            violation: 'SDK_IMPORT',
          });
        }
      });
    });
  } catch (err) {
    console.error(`[WARN] Could not read ${filePath}: ${err.message}`);
  }

  return violations;
}

function main() {
  const allFiles = getAllSourceFiles();
  const violations = [];

  for (const file of allFiles) {
    if (!isInScope(file)) continue;
    violations.push(...scanFile(file));
  }

  // Load baseline
  const baseline = loadBaseline();
  const newViolations = violations.filter(v =>
    !baseline.some(b => b.file === v.file && b.line === v.line)
  );

  // Report
  console.log(`[LINT] Architecture: SDK Bundling Check`);
  console.log(`Scanned ${allFiles.length} files`);
  console.log(`Found ${violations.length} total violations, ${newViolations.length} new`);

  if (newViolations.length > 0) {
    console.error('[ERROR] New SDK imports detected:');
    newViolations.forEach(v => {
      console.error(`  ${v.file}:${v.line}: ${v.content}`);
    });
    process.exit(1);
  }

  // Grandfathered violations as warnings
  if (violations.length > newViolations.length) {
    console.warn(
      `[WARN] ${violations.length - newViolations.length} grandfathered violations (to be refactored)`
    );
  }

  process.exit(0);
}

function getAllSourceFiles() {
  const files = [];
  walkDir('.', files);
  return files;
}

function walkDir(dir, files) {
  try {
    const entries = fs.readdirSync(dir);
    for (const entry of entries) {
      const full = path.join(dir, entry);
      const stat = fs.statSync(full);
      if (stat.isDirectory()) {
        if (!['node_modules', '.git', 'dist'].includes(entry)) {
          walkDir(full, files);
        }
      } else if (entry.endsWith('.js') || entry.endsWith('.cjs')) {
        files.push(full);
      }
    }
  } catch (_) {
    // Ignore errors
  }
}

function loadBaseline() {
  const baselinePath = '.formal/architecture-baseline.jsonl';
  if (!fs.existsSync(baselinePath)) return [];
  try {
    const lines = fs
      .readFileSync(baselinePath, 'utf8')
      .split('\n')
      .filter(l => l.trim());
    return lines.map(l => JSON.parse(l));
  } catch (_) {
    return [];
  }
}

main();
```

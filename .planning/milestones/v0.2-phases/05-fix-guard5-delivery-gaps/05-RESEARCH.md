# Phase 5: Fix GUARD 5 Delivery Gaps - Research

**Researched:** 2026-02-21
**Domain:** QGSD hook delivery — installer config generation, dist build pipeline, CHANGELOG maintenance
**Confidence:** HIGH

## Summary

Phase 4 shipped GUARD 5 (decision-turn scoping) into `hooks/qgsd-prompt.js` and `hooks/qgsd-stop.js` in the source tree, but the changes never reached two of the three delivery paths that real users touch:

1. **Source installs** (developers cloning the repo): `hooks/dist/` was never rebuilt after Phase 4. Both `qgsd-prompt.js` and `qgsd-stop.js` in `hooks/dist/` are the Phase 3 versions — missing GUARD 5 entirely from the stop hook, and missing step 5 (the `<!-- GSD_DECISION -->` marker instruction) from the prompt hook.

2. **Installer users** (everyone running `npx qgsd@latest`): The installer writes `~/.claude/qgsd.json` with a `quorum_instructions` field generated by `buildQuorumInstructions()` in `bin/install.js`. That function generates numbered steps dynamically but stops at step N+1 ("Present all model responses...") — it does not append the step-5 marker instruction that Phase 4 added to `DEFAULT_QUORUM_INSTRUCTIONS_FALLBACK`. This means `hasDecisionMarker()` in the stop hook can never fire for any user whose config was written by the installer.

3. **The templates/qgsd.json fallback**: The template file used as a human-readable example and documentation anchor also lacks step 5. Since `templates/qgsd.json` is the canonical reference for what the installer writes (see SYNC-02 maintenance note in CHANGELOG), it needs to be kept in sync.

4. **CHANGELOG gap**: The `--redetect-mcps` flag was implemented in Phase 3 but never documented in CHANGELOG.md. The SYNC-02 maintenance note in the `[0.1.0]` entry tells maintainers where to update `quorum_commands` but says nothing about `--redetect-mcps`.

**Primary recommendation:** Three targeted edits plus one rebuild: (1) add `<!-- GSD_DECISION -->` step to `buildQuorumInstructions()` in `bin/install.js`, (2) add the same step to `quorum_instructions` in `templates/qgsd.json`, (3) run `npm run build:hooks` to propagate Phase 4 source changes into `hooks/dist/`, (4) add `--redetect-mcps` documentation to CHANGELOG. All changes are mechanical — no new logic, no new tests required.

### Pattern 1: Three-surface quorum_instructions sync

**What:** `quorum_instructions` text lives in three places that must stay identical:
1. `DEFAULT_QUORUM_INSTRUCTIONS_FALLBACK` in `hooks/qgsd-prompt.js` — fallback for users without a config or with a config that has no `quorum_instructions` field
2. `buildQuorumInstructions()` in `bin/install.js` — dynamic generation for installer-written config
3. `quorum_instructions` in `templates/qgsd.json` — static reference example

The CHANGELOG SYNC-02 maintenance note explicitly calls this out for `quorum_commands`, but not for `quorum_instructions`. Phase 5 must add step 5 to surfaces 2 and 3.

**When to use:** Any time quorum instructions change — all three surfaces update.

### Anti-Patterns to Avoid

- **Editing `hooks/dist/` directly:** dist is build output. Always edit source, then rebuild. Direct dist edits will be overwritten by the next `npm run build:hooks`.
- **Adding a hardcoded step 5 string to `buildQuorumInstructions()`:** The function dynamically numbers steps based on `required.length`. Step 5 is the marker step and must be appended AFTER the dynamically numbered "present responses" step, not interleaved.
- **Changing the DECISION_MARKER token:** The token `<!-- GSD_DECISION -->` is a constant in `hooks/qgsd-stop.js` and referenced in `hooks/qgsd-prompt.js`. Any change in one surface must propagate everywhere. Do not paraphrase.

## Common Pitfalls

### Pitfall 1: Incorrect step numbering in buildQuorumInstructions()

**What goes wrong:** The dynamic step numbering means "step 5" is only correct when there are 3 required models (steps 1-3 are model calls, step 4 is "present responses", step 5 is the marker). With a different number of models, the marker step number changes.

**Why it happens:** The function generates step N+1 as "present responses" where N = number of required models. The marker step is always N+2.

**How to avoid:** Append the marker step using `required.length + 2` (not hardcoded `5`), with the same clarifying language as in DEFAULT_QUORUM_INSTRUCTIONS_FALLBACK.

**Warning signs:** If a user has added a 4th quorum model, step 5 would be "present responses" and step 6 would be the marker. Hardcoding "step 5" would be wrong.

### Pitfall 2: Forgetting templates/qgsd.json

**What goes wrong:** Updating `buildQuorumInstructions()` but not `templates/qgsd.json` leaves the template as documentation for the old behavior.

**Why it happens:** `templates/qgsd.json` is not in the build pipeline — it is a static file. There is no automated sync.

**How to avoid:** Treat templates/qgsd.json as the third surface of the three-surface sync (Pattern 1 above).

**Warning signs:** `grep "GSD_DECISION" templates/qgsd.json` returns 0 matches after the fix → templates/qgsd.json missed.

### Pitfall 3: Not rebuilding dist after source edits

**What goes wrong:** Edit source hooks, verify locally using source files, but forget to run `npm run build:hooks`. Ship package with stale dist.

**Why it happens:** `prepublishOnly` only runs on publish — not on save. Local testing uses source files directly (the hooks installed at `~/.claude/hooks/` point to the dist directory, not the source directory).

**How to avoid:** Run `npm run build:hooks` as an explicit task before verification. Verify by diffing source vs dist again after build.

**Warning signs:** `diff hooks/qgsd-stop.js hooks/dist/qgsd-stop.js` still shows differences after supposed rebuild.

### Pitfall 4: Users with existing ~/.claude/qgsd.json don't get step 5

**What goes wrong:** Fixing `buildQuorumInstructions()` only affects NEW installs (when `~/.claude/qgsd.json` doesn't exist). Existing users with a `qgsd.json` written before Phase 5 will still have the old instructions.

**Why it happens:** The installer skips config write if the file already exists (INST-06 idempotency). Only `--redetect-mcps` forces a config refresh.

**How to handle:** This is acceptable per the fail-open philosophy (documented in Phase 4 Plan 02 context: "Users who have customized their config will need to add the decision marker instruction manually — this is acceptable"). The CHANGELOG entry should document that existing users should run `npx qgsd@latest --redetect-mcps` to refresh their config, which is exactly why `--redetect-mcps` documentation is part of this phase.

## Open Questions

1. **Should CHANGELOG move [0.1.0] to [0.1.1] for this fix?**
   - What we know: All changes to Phase 4 GUARD 5 features were merged into the repo but never packaged for release (dist stale, installer broken)
   - What's unclear: Whether to version-bump or keep in [Unreleased] until intentionally published
   - Recommendation: Use `[Unreleased]` section only (the plan says "document --redetect-mcps in CHANGELOG" without specifying a version bump). Version bumping is a separate release decision not in scope for this plan.

2. **Should `--redetect-mcps` also update quorum_instructions when the config already has a `quorum_instructions` field?**
   - What we know: Currently `--redetect-mcps` deletes the entire config and regenerates it (including quorum_instructions). This means it DOES update quorum_instructions.
   - What's unclear: Users with custom quorum_instructions would lose their customization on `--redetect-mcps`.
   - Recommendation: This is pre-existing behavior, not introduced in Phase 5. Document it accurately in CHANGELOG but do not change the behavior — that would be out of scope.

# Phase v0.28-03: Runtime Intelligence -- Budget Downgrade + Stall Detection + Smart Compact - Research

**Researched:** 2026-03-06
**Domain:** Claude Code hook system / runtime harness monitoring
**Confidence:** HIGH

## Summary

This phase adds three runtime intelligence features to the nForma harness: (1) token budget monitoring with automatic model profile downgrade, (2) stall detection for quorum slots that stop responding, and (3) workflow-aware compaction suggestions at clean boundaries. All three features integrate into the existing hook infrastructure -- no new hook files are needed. The budget system reads from `nf.json` config and writes to the existing `token-usage.jsonl` telemetry pipeline. Stall detection leverages the existing `quorum-failures.json` and slot correlator data. Smart compact suggestions extend the existing `gsd-context-monitor.js` PostToolUse hook.

The critical constraint is that all token values in `token-usage.jsonl` are currently zero -- Claude Code subagent transcripts do not reliably expose `message.usage` fields. Budget tracking must therefore use the `context_window` field from PostToolUse events (which provides `remaining_percentage`) as the primary signal, supplemented by cumulative token records when available. The `model_profile` in `.planning/config.json` (currently `"balanced"`) is the existing lever for model tier selection.

**Primary recommendation:** Build budget tracking on top of `gsd-context-monitor.js` (PostToolUse hook, already has context_window access), stall detection as a new `bin/` script invoked from `nf-stop.js`, and smart compact as an enhancement to the existing context monitor output.

### Pattern 2: Budget Tracking via Context Window (Primary Signal)
**What:** Use PostToolUse `context_window.remaining_percentage` as the primary budget signal since token-usage.jsonl values are unreliable (all zeros)
**When to use:** Always -- this is the only reliable runtime signal for context consumption
**Example:**
```javascript
// In gsd-context-monitor.js, after computing usedPct:
const budgetCfg = config.budget || {};
if (budgetCfg.session_limit_tokens) {
  // Estimate tokens from context_window percentage
  // Claude context windows are typically 200K tokens
  const estimatedTokens = Math.round((usedPct / 100) * 200000);
  const limitTokens = budgetCfg.session_limit_tokens;
  const budgetUsedPct = Math.round((estimatedTokens / limitTokens) * 100);

  if (budgetUsedPct >= (budgetCfg.downgrade_pct || 85)) {
    // Trigger profile downgrade
    triggerProfileDowngrade(cwd);
  } else if (budgetUsedPct >= (budgetCfg.warn_pct || 60)) {
    // Inject budget warning into context
  }
}
```

### Pattern 3: Profile Downgrade Chain
**What:** Automatic model profile downgrade: quality -> balanced -> budget
**When to use:** When budget threshold is crossed
**Example:**
```javascript
// Model profile downgrade chain
const DOWNGRADE_CHAIN = {
  quality: 'balanced',
  balanced: 'budget',
  budget: null,  // cannot downgrade further
};

function triggerProfileDowngrade(cwd) {
  const configPath = path.join(cwd, '.planning', 'config.json');
  const config = JSON.parse(fs.readFileSync(configPath, 'utf8'));
  const current = config.model_profile || 'balanced';
  const next = DOWNGRADE_CHAIN[current];
  if (!next) return false; // already at minimum
  config.model_profile = next;
  fs.writeFileSync(configPath, JSON.stringify(config, null, 2), 'utf8');
  return true;
}
```

### Pattern 4: Stall Detection via Quorum Failures
**What:** Monitor quorum-failures.json for consecutive TIMEOUT entries per slot
**When to use:** At stop-hook time, after quorum completes
**Example:**
```javascript
// stall-detector.cjs
function detectStalledSlots(cwd, config) {
  const stallCfg = config.stall_detection || {};
  const threshold = stallCfg.consecutive_threshold || 2;
  const pp = require('./planning-paths.cjs');
  const failPath = pp.resolveWithFallback(cwd, 'quorum-failures');
  if (!fs.existsSync(failPath)) return [];

  const records = JSON.parse(fs.readFileSync(failPath, 'utf8'));
  // Group by slot, check for consecutive TIMEOUT entries
  const bySlot = {};
  for (const r of records) {
    if (r.error_type !== 'TIMEOUT') continue;
    if (!bySlot[r.slot]) bySlot[r.slot] = 0;
    bySlot[r.slot]++;
  }
  return Object.entries(bySlot)
    .filter(([, count]) => count >= threshold)
    .map(([slot, count]) => ({ slot, consecutive_timeouts: count }));
}
```

### Pattern 5: Smart Compact Boundary Detection
**What:** Detect clean workflow boundaries where compaction is safe
**When to use:** At PostToolUse time, when context is above warn threshold
**Example:**
```javascript
// Workflow boundaries where /compact is safe:
// 1. Phase completion: gsd-tools.cjs phase-complete was just called
// 2. Verification done: VERIFICATION.md was just committed
// 3. Commit just made: gsd-tools.cjs commit was the last tool call
function isCleanBoundary(toolName, toolInput) {
  if (toolName === 'Bash') {
    const cmd = JSON.stringify(toolInput || '');
    if (cmd.includes('gsd-tools.cjs phase-complete')) return 'phase_complete';
    if (cmd.includes('gsd-tools.cjs commit') && /VERIFICATION\.md/.test(cmd)) return 'verification_done';
    if (cmd.includes('gsd-tools.cjs commit')) return 'commit';
  }
  return null;
}
```

### Anti-Patterns to Avoid
- **Reading token-usage.jsonl for budget math:** All values are currently 0 (Claude Code limitation). Use context_window percentage as primary signal.
- **Creating new hook files:** The roadmap states "no new hook files are needed." Extend existing hooks and create bin/ scripts instead.
- **Blocking on stall detection:** Stall detection is informational, not a gate. Never block the user's session due to stalled slots.
- **Modifying model_tier_planner/model_tier_worker directly:** Use `.planning/config.json` `model_profile` field which is the existing profile lever. The tier fields in nf.json are user-set defaults.
- **Triggering compact automatically:** Only suggest `/compact` -- never auto-compact. Users must choose when to compact.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config validation | Custom validation per feature | Extend config-loader.js validateConfig() | Consistent patterns, shared warnings, all config in one place |
| Path resolution | Hard-coded .planning/ paths | planning-paths.cjs resolve/resolveWithFallback | Handles legacy/canonical migration, creates dirs |
| Conformance logging | Custom event writing | appendConformanceEvent() pattern from nf-prompt.js | Atomic writes, fail-open, consistent schema |
| Context window data | Parsing transcript for usage | PostToolUse event's context_window field | Already provided by Claude Code runtime |
| Slot auth_type lookup | Parsing providers.json | config.agent_config[slot].auth_type | Already loaded and validated by config-loader |

**Key insight:** The harness already has all the data sources needed -- context_window in PostToolUse events, quorum-failures.json for timeouts, agent_config for auth_type classification, config.json for model_profile. The work is wiring these together with threshold logic, not building new data collection.

## Common Pitfalls

### Pitfall 1: Shallow Config Merge Destroys Nested Objects
**What goes wrong:** Adding `budget: { ... }` to DEFAULT_CONFIG and then user sets `budget: { session_limit_tokens: 100000 }` in nf.json -- shallow merge `{ ...DEFAULT, ...user }` replaces entire budget object, losing warn_pct and downgrade_pct defaults.
**Why it happens:** config-loader.js uses shallow spread merge (line 336).
**How to avoid:** Follow the same pattern as `circuit_breaker` validation (lines 162-197): validate the sub-object and fill missing sub-keys with defaults individually.
**Warning signs:** Tests pass with full config objects but fail with partial configs.

### Pitfall 2: Double loadConfig() Call in Context Monitor
**What goes wrong:** gsd-context-monitor.js currently has a bug: it calls `loadConfig()` twice (lines 25 and 39), with the second call shadowing the first via `const config = ...`. The first call's profile guard works but the second `const config` causes a redeclaration error in strict mode.
**Why it happens:** The two responsibilities (profile guard + threshold logic) were added at different times.
**How to avoid:** When modifying gsd-context-monitor.js, fix this by removing the duplicate `const config` declaration. Use a single loadConfig() call.
**Warning signs:** Context monitor fails silently (fail-open) on every invocation.

### Pitfall 3: Token Values Are All Zeros
**What goes wrong:** Building budget math on cumulative token sums from token-usage.jsonl gives 0 forever.
**Why it happens:** Claude Code subagent transcripts don't expose `message.usage` fields reliably. All 1124 records have `input_tokens: 0, output_tokens: 0`.
**How to avoid:** Use `context_window.remaining_percentage` from PostToolUse events as the primary budget signal. It is always populated by Claude Code runtime.
**Warning signs:** Budget warnings never fire despite heavy usage.

### Pitfall 4: Profile Downgrade Persistence
**What goes wrong:** Writing model_profile to .planning/config.json but the downgrade doesn't take effect because the model is selected at session start, not per-request.
**Why it happens:** Model selection happens when Claude Code starts, not when hooks run.
**How to avoid:** Profile downgrade should be treated as a recommendation injected into context, not an automatic switch. Inject a message like "BUDGET: Context at 87% of limit. Model profile downgraded to 'budget'. Consider running /compact."
**Warning signs:** Downgrade appears in config but model behavior doesn't change.

### Pitfall 5: Stall Detection False Positives on Cached Quorums
**What goes wrong:** Cache hits bypass slot dispatch entirely, but stall detector still checks for recent timeouts and reports them.
**Why it happens:** The NF_CACHE_HIT path in nf-prompt.js exits before reaching any stall check.
**How to avoid:** Stall detection should only run when live quorum dispatch actually occurred. Check for cache_hit in the conformance event stream.
**Warning signs:** Stall reports appear for slots that haven't been dispatched in hours.

### Pitfall 6: Install Sync Forgotten
**What goes wrong:** Editing hooks/* source files but not syncing to hooks/dist/ and re-running install.
**Why it happens:** The installed hooks at ~/.claude/hooks/ are the ones that actually run; they are copied from hooks/dist/.
**How to avoid:** Every plan that modifies hook source files MUST include: `cp hooks/<file> hooks/dist/ && node bin/install.js --claude --global`
**Warning signs:** Tests pass but live hook behavior doesn't change.

### Budget Config Validation Pattern
```javascript
// In config-loader.js validateConfig():
// Validate budget sub-object (following circuit_breaker pattern)
if (typeof config.budget !== 'object' || config.budget === null) {
  config.budget = { ...DEFAULT_CONFIG.budget };
} else {
  if (config.budget.session_limit_tokens !== null &&
      config.budget.session_limit_tokens !== undefined) {
    if (!Number.isInteger(config.budget.session_limit_tokens) ||
        config.budget.session_limit_tokens < 1000) {
      process.stderr.write('[nf] WARNING: nf.json: budget.session_limit_tokens must be integer >= 1000 or null; disabling\n');
      config.budget.session_limit_tokens = null;
    }
  }
  // Fill missing sub-keys with defaults
  if (config.budget.warn_pct === undefined) {
    config.budget.warn_pct = DEFAULT_CONFIG.budget.warn_pct;
  }
  if (config.budget.downgrade_pct === undefined) {
    config.budget.downgrade_pct = DEFAULT_CONFIG.budget.downgrade_pct;
  }
}
```

## Open Questions

1. **Context window total size assumption**
   - What we know: PostToolUse provides `remaining_percentage` but not absolute token counts.
   - What's unclear: Whether Claude Code exposes the total context window size (200K? 128K?). The percentage-based threshold already works for warnings, but `budget.session_limit_tokens` is an absolute number.
   - Recommendation: Support both modes: (a) percentage-based thresholds (always reliable, use context_window data), (b) absolute token limits (when token-usage.jsonl becomes reliable in the future). For now, percentage mode is primary.

2. **Profile downgrade runtime effect**
   - What we know: `.planning/config.json` `model_profile` is read by planning workflows.
   - What's unclear: Whether changing it mid-session actually affects model selection for subsequent operations, or only takes effect on next session start.
   - Recommendation: Treat as "best effort" -- write the downgrade and inject a notification. Even if the current session's model doesn't change, the notification itself helps the user decide to /compact or adjust.

3. **Stall checkpoint definition**
   - What we know: quorum-failures.json tracks TIMEOUT events per slot with timestamps.
   - What's unclear: What constitutes a "checkpoint" -- each quorum dispatch, each PostToolUse event, or a time-based interval?
   - Recommendation: Define a checkpoint as each quorum dispatch round (quorum_start conformance event). Two consecutive dispatches where the same slot times out = stalled.

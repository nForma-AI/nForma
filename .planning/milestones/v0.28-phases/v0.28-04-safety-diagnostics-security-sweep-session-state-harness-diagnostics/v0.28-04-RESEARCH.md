# Phase v0.28-04: Safety & Diagnostics -- Security Sweep + Session State + Harness Diagnostics - Research

**Researched:** 2026-03-06
**Domain:** Claude Code hook system / verify-phase integration / diagnostic tooling
**Confidence:** HIGH

## Summary

This phase adds three safety and diagnostic features to the nForma harness: (1) a security sweep script that runs at verify-phase boundary and reports secrets/debug artifacts in VERIFICATION.md, (2) a session state reminder injected on new sessions when work is in progress, and (3) a unified harness diagnostic tool that aggregates slot availability, pass@k, token spend, stall events, and recommendations.

All three features are informational only and follow the fail-open pattern. The security sweep is a standalone bin/ script (SEC-03 requirement), the session state reminder extends the existing nf-session-start.js hook (STATE-01, STATE-02), and the diagnostic tool is a new bin/ script callable via the existing `/nf:health` command or directly.

The critical constraint is preserving stop hook BLOCK/PASS behavior (LivenessProperty2, LivenessProperty3). None of these features modify nf-stop.js -- they are purely additive.

**Primary recommendation:** Build security sweep as `bin/security-sweep.cjs` (standalone, called by verify-phase workflow), session state as an addition to `nf-session-start.js`, and harness diagnostic as `bin/harness-diagnostic.cjs` with `--json` flag support.

### Pattern 1: Security Sweep Regex Scanning
**What:** Scan tracked files for secret patterns using regex
**When to use:** At verify-phase boundary only
**Example:**
```javascript
const SECRET_PATTERNS = [
  { name: 'AWS Access Key', pattern: /AKIA[A-Z0-9]{16}/, severity: 'high' },
  { name: 'GitHub Token', pattern: /gh[ps]_[A-Za-z0-9_]{36,}/, severity: 'high' },
  { name: 'GitHub PAT', pattern: /github_pat_[A-Za-z0-9_]{22,}/, severity: 'high' },
  { name: 'Stripe Secret', pattern: /[sr]k_live_[A-Za-z0-9]{20,}/, severity: 'high' },
  { name: 'Generic API Key', pattern: /(?:api[_-]?key|apikey)\s*[:=]\s*['"][^'"]{10,}['"]/i, severity: 'medium' },
  { name: 'Generic Secret', pattern: /(?:secret|password)\s*[:=]\s*['"][^'"]{8,}['"]/i, severity: 'medium' },
  { name: 'OpenAI Key', pattern: /sk-[A-Za-z0-9]{32,}/, severity: 'high' },
  { name: 'Debugger Statement', pattern: /\bdebugger\b/, severity: 'low' },
  { name: 'Console.log with sensitive', pattern: /console\.log\(.*(?:password|secret|token|key|api_key)/i, severity: 'low' },
];
```

### Pattern 2: Session State Reminder from STATE.md
**What:** Parse STATE.md for in-progress phase and inject a terse reminder
**When to use:** On SessionStart, when STATE.md shows an in-progress phase
**Example:**
```javascript
function parseStateForReminder(stateContent) {
  const phaseMatch = stateContent.match(/Phase:\s*(.+)/);
  const statusMatch = stateContent.match(/Status:\s*(.+)/);
  const lastMatch = stateContent.match(/Last activity:\s*(.+)/);
  const planMatch = stateContent.match(/Plan:\s*(.+)/);

  if (!phaseMatch || !statusMatch) return null;
  const status = statusMatch[1].trim();
  if (status === 'Complete' || status === 'Not started') return null;

  return `SESSION STATE REMINDER: Phase ${phaseMatch[1].trim()} -- ${planMatch ? planMatch[1].trim() : 'unknown plan'} -- ${status} (last: ${lastMatch ? lastMatch[1].trim() : 'unknown'})`;
}
```

### Anti-Patterns to Avoid
- **Scanning .gitignore'd files for secrets:** Only scan tracked files (`git ls-files`). Scanning node_modules or .quorum-cache would be noisy and slow.
- **Modifying nf-stop.js for any of these features:** Stop hook BLOCK/PASS behavior must be unchanged. All three features are informational only.
- **Session state reminder on compaction:** STATE-02 explicitly requires no reminder on compaction. nf-precompact.js already handles compaction context.
- **Token spend sums as reliable budget data:** token-usage.jsonl values are all zeros (known limitation from v0.28-03 research). Report the data but note it's unreliable.
- **Forgetting install sync:** nf-session-start.js modifications require cp to hooks/dist/ and re-running install.

## Common Pitfalls

### Pitfall 1: Security Sweep False Positives in Test Fixtures
**What goes wrong:** Test files intentionally contain patterns like `sk-test123` or `AKIA...` as test fixtures, producing false positive findings.
**Why it happens:** Regex patterns are broad enough to match test data.
**How to avoid:** Exclude common test fixture patterns (lines containing `test`, `mock`, `fake`, `example`, `dummy`) or add an allowlist of known-safe patterns. Also exclude the security-sweep script itself.
**Warning signs:** Security sweep produces findings in test files that are obviously test fixtures.

### Pitfall 2: STATE.md Parsing Brittleness
**What goes wrong:** STATE.md format changes break the regex parsing for state reminder.
**Why it happens:** STATE.md is a markdown file written by gsd-tools.cjs with a semi-structured format.
**How to avoid:** Use the extractCurrentPosition() function already proven in nf-precompact.js. Parse the specific fields needed with generous fallbacks. Fail-open: if parsing fails, skip the reminder.
**Warning signs:** State reminder appears as "Phase undefined -- unknown plan" or never fires.

### Pitfall 3: Diagnostic Tool Crashes on Missing Data Sources
**What goes wrong:** harness-diagnostic.cjs throws when scoreboard, token-usage, or conformance-events don't exist.
**Why it happens:** Fresh installs or repos that haven't run quorum rounds yet lack these files.
**How to avoid:** Every data source read is wrapped in try/catch with a meaningful "no data available" fallback. Each section of the report is independent.
**Warning signs:** `node bin/harness-diagnostic.cjs` crashes on a fresh project.

### Pitfall 4: Install Sync Forgotten for Session Start Hook
**What goes wrong:** Editing nf-session-start.js but not syncing to hooks/dist/ and re-running install.
**Why it happens:** The installed hooks at ~/.claude/hooks/ are the ones that actually run; they are copied from hooks/dist/.
**How to avoid:** Plan that modifies nf-session-start.js MUST include: `cp hooks/nf-session-start.js hooks/dist/ && node bin/install.js --claude --global`
**Warning signs:** Tests pass but live hook behavior doesn't change.

### Pitfall 5: Diagnostic Report Tries to Read Scoreboard Round-Level Data
**What goes wrong:** Attempting to parse scoreboard.rounds[] for per-slot availability, but rounds may be empty or have different structures.
**Why it happens:** Scoreboard format has evolved; the .slots section is more reliable than .rounds.
**How to avoid:** Use scoreboard.slots (keyed by "slotName:modelId") for availability data. Use conformance-events.jsonl for event counts. Don't parse rounds[].
**Warning signs:** Diagnostic report shows 0% availability for all slots.

## Open Questions

1. **Security sweep exclusion patterns**
   - What we know: Test files intentionally contain secret-like patterns.
   - What's unclear: Whether to exclude by path pattern (e.g., `*.test.*`) or by content heuristic (line contains `test`/`mock`).
   - Recommendation: Exclude files matching `*.test.*` and `*.spec.*` patterns. Also exclude the security-sweep.cjs file itself. This is simpler and more reliable than content heuristics.

2. **Diagnostic auto-run on SessionStart**
   - What we know: DIAG-03 says diagnostics can run automatically on SessionStart when the nForma dev repo is detected.
   - What's unclear: Whether running the full diagnostic on every session start adds unacceptable latency.
   - Recommendation: Run a lightweight health check only (slot availability from provider cache, circuit breaker status). Full diagnostic is on-demand only.

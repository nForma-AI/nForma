## Summary

This phase adds two complementary data pipeline features to the nForma quorum system: (1) SHA-256 response caching that avoids redundant API calls for identical quorum dispatches, and (2) pass@k metrics that track how efficiently the quorum reaches consensus (first round vs. requiring deliberation rounds).

Both features integrate into existing infrastructure without new dependencies. The cache lives in nf-prompt.js (the UserPromptSubmit hook that constructs quorum dispatch instructions). The pass@k metric is recorded in nf-stop.js (the Stop hook that validates quorum completion) and reported by verify-quorum-health.cjs. The conformance-schema.cjs gains one new action (`cache_hit`) and a new optional field (`pass_at_k`) on `quorum_complete` events.

The v0.29 research decision of "zero new npm dependencies" applies here equally. Node.js `crypto` module provides SHA-256. File I/O uses `fs` built-ins. All new code follows the existing fail-open pattern: cache failures never block quorum dispatch.

**Primary recommendation:** Build cache as a transparent layer in nf-prompt.js that short-circuits slot dispatch on hit, and add pass@k as a field on existing quorum_complete conformance events.

### Pattern 1: Transparent Cache Layer in nf-prompt.js
**What:** Before building the quorum dispatch instructions, compute a cache key from the inputs that determine the quorum outcome. If a valid cached response exists, inject the cached results as additionalContext instead of dispatch instructions.
**When to use:** Every quorum dispatch (planning command detection in nf-prompt.js).
**Example:**
```javascript
// In nf-prompt.js, after computing orderedSlots but before building instructions:
const crypto = require('crypto');
const cacheKey = computeCacheKey(prompt, contextYaml, cappedSlots, config);

function computeCacheKey(prompt, context, slots, config) {
  const gitHead = getGitHead();
  const slotNames = slots.map(s => s.slot).sort().join(',');
  const configHash = crypto.createHash('sha256')
    .update(JSON.stringify(config.quorum_active || []))
    .digest('hex')
    .slice(0, 16);
  return crypto.createHash('sha256')
    .update([prompt, context, slotNames, configHash, gitHead].join('\x00'))
    .digest('hex');
}

function getGitHead() {
  try {
    const { spawnSync } = require('child_process');
    const r = spawnSync('git', ['rev-parse', 'HEAD'], { encoding: 'utf8', timeout: 3000 });
    return r.status === 0 ? r.stdout.trim() : '';
  } catch { return ''; }
}
```

### Pattern 2: Cache Entry Format
**What:** Each cache file is a self-contained JSON document with metadata for invalidation and the cached slot responses.
**When to use:** Reading/writing cache entries.
**Example:**
```javascript
// Cache entry schema
{
  "version": 1,
  "key": "<sha256-hex>",
  "created": "2026-03-06T12:00:00.000Z",
  "ttl_ms": 3600000,  // 1 hour default
  "git_head": "<commit-sha>",
  "quorum_active": ["codex-1", "gemini-1", "opencode-1"],
  "instructions": "<the full quorum instruction string that was injected>"
}
```

**IMPORTANT DESIGN NOTE:** The cache does NOT store slot responses. It stores the *dispatch instructions*. Why: The quorum dispatch in nf-prompt.js builds instruction text that Claude then executes. The actual slot responses happen inside Claude's execution, not in the hook. The cache's purpose is to avoid re-dispatching when the same question has already been asked within TTL -- but since the hook only controls the *instructions*, not the responses, the cache must work at the instruction level.

**REVISION:** After closer analysis, caching at the instruction level provides no benefit -- the instructions are deterministic given the same inputs, and the expensive work (API calls to slots) happens inside Claude's execution, not in the hook. The cache needs to operate at a higher level.

### Pattern 3: Cache Architecture -- Hook-Level Marker + Claude-Level Replay
**What:** The cache operates in two phases: (1) nf-prompt.js checks cache and, on hit, injects a `<!-- NF_CACHE_HIT -->` marker with cached vote summary into additionalContext, telling Claude to skip slot dispatch and use cached results; (2) nf-stop.js recognizes the cache hit marker and accepts cached quorum evidence.
**When to use:** This is the only viable approach given that hooks cannot execute slot dispatches directly.
**Example:**
```javascript
// nf-prompt.js: on cache hit
const cacheEntry = readCache(cacheKey);
if (cacheEntry && isCacheValid(cacheEntry)) {
  appendConformanceEvent({
    ts: new Date().toISOString(),
    phase: 'DECIDING',
    action: 'cache_hit',
    cache_key: cacheKey.slice(0, 12),
    slots_available: cacheEntry.slot_count,
    vote_result: cacheEntry.vote_result,
    outcome: 'APPROVE',
    schema_version,
  });
  const instructions = `<!-- NF_CACHE_HIT -->\nQUORUM CACHE HIT: Identical dispatch was completed ${timeAgo(cacheEntry.created)} ago.\n` +
    `Cached result: ${cacheEntry.vote_result} of ${cacheEntry.slot_count} slots approved.\n` +
    `Decision: ${cacheEntry.outcome}\n` +
    `Skip all slot-worker Task dispatches. Use this cached quorum result.\n` +
    `Include <!-- GSD_DECISION --> in your final output.`;
  process.stdout.write(JSON.stringify({
    hookSpecificOutput: {
      hookEventName: 'UserPromptSubmit',
      additionalContext: instructions,
    }
  }));
  process.exit(0);
}

// nf-prompt.js: on cache miss, after building normal instructions,
// write a cache entry with the dispatch metadata for future hits.
writeCache(cacheKey, {
  version: 1,
  key: cacheKey,
  created: new Date().toISOString(),
  ttl_ms: config.cache_ttl_ms || 3600000,
  git_head: getGitHead(),
  quorum_active: (config.quorum_active || []).slice(),
  slot_count: cappedSlots.length,
  // vote_result and outcome are UNKNOWN at dispatch time --
  // they are backfilled by nf-stop.js after quorum completes.
});
```

```javascript
// nf-stop.js: recognize cache hit marker
function hasCacheHitMarker(currentTurnLines) {
  for (const line of currentTurnLines) {
    try {
      const entry = JSON.parse(line);
      if (entry.type === 'user') {
        const content = entry.message?.content;
        const text = typeof content === 'string' ? content :
          (Array.isArray(content) ? content.map(c => c?.text || '').join('') : '');
        if (text.includes('<!-- NF_CACHE_HIT -->')) return true;
      }
    } catch {}
  }
  return false;
}

// In main(), after GUARD 6 (solo mode):
if (hasCacheHitMarker(currentTurnLines)) {
  appendConformanceEvent({
    ts: new Date().toISOString(),
    phase: 'DECIDING',
    action: 'quorum_complete',
    slots_available: 0,
    vote_result: 0,
    outcome: 'APPROVE',
    cache_hit: true,
    schema_version,
  });
  process.exit(0); // Cache hit: quorum already verified
}
```

### Pattern 4: Cache Backfill After Quorum Completion
**What:** When nf-stop.js approves a quorum (non-cached), it backfills the cache entry with the actual vote_result and outcome, making it available for future cache hits.
**When to use:** Every successful quorum completion on a cache miss.
**Example:**
```javascript
// nf-stop.js: after successful quorum (successCount >= maxSize), backfill cache
function backfillCache(cacheKey, voteResult, slotCount) {
  try {
    const cachePath = path.join(process.cwd(), '.planning', '.quorum-cache', cacheKey + '.json');
    if (!fs.existsSync(cachePath)) return;
    const entry = JSON.parse(fs.readFileSync(cachePath, 'utf8'));
    entry.vote_result = voteResult;
    entry.outcome = 'APPROVE';
    entry.completed = new Date().toISOString();
    fs.writeFileSync(cachePath, JSON.stringify(entry, null, 2), 'utf8');
  } catch {} // fail-open
}
```

**Challenge:** nf-stop.js does not know the cache key. Options:
1. nf-prompt.js writes the cache key to a sidecar file (`.planning/.quorum-cache/.pending-key`)
2. nf-stop.js recomputes the cache key from the same inputs
3. nf-prompt.js includes the cache key in the injected instructions (detectable by nf-stop.js)

**Recommendation:** Option 3 -- embed cache key in instructions as `<!-- NF_CACHE_KEY:abc123... -->`. nf-stop.js extracts it from the transcript. This is consistent with how `<!-- GSD_DECISION -->` and `<!-- NF_SOLO_MODE -->` markers already work.

### Pattern 5: pass@k Tracking
**What:** Count deliberation rounds in the current turn to determine k (round at which consensus was reached). Record as `pass_at_k` field on the `quorum_complete` conformance event.
**When to use:** Every quorum completion in nf-stop.js.
**Example:**
```javascript
// nf-stop.js: count deliberation rounds before final approval
function countDeliberationRounds(currentTurnLines) {
  let rounds = 0;
  for (const line of currentTurnLines) {
    try {
      const entry = JSON.parse(line);
      if (entry.type !== 'assistant') continue;
      const content = entry.message?.content;
      if (!Array.isArray(content)) continue;
      for (const block of content) {
        if (block.type !== 'tool_use') continue;
        // Count slot-worker dispatches as round markers
        if (block.name === 'Task' && JSON.stringify(block.input || {}).includes('nf-quorum-slot-worker')) {
          rounds++;
          break; // one dispatch group = one round
        }
      }
    } catch {}
  }
  return Math.max(1, rounds); // at least round 1
}

// In the quorum_complete event:
appendConformanceEvent({
  ts: new Date().toISOString(),
  phase: 'DECIDING',
  action: 'quorum_complete',
  slots_available: agentPool.length,
  vote_result: successCount,
  outcome: 'APPROVE',
  pass_at_k: countDeliberationRounds(currentTurnLines),
  schema_version,
});
```

### Anti-Patterns to Avoid
- **Caching inside Claude's execution flow:** The hook system controls instructions injected before Claude acts, not Claude's runtime behavior. Cache must work at the hook level.
- **Storing raw API responses:** Hooks never see slot responses directly. Only store dispatch metadata + outcome.
- **Blocking on cache failures:** All cache operations are wrapped in try/catch with fail-open semantics. A corrupted cache file must never prevent quorum dispatch.
- **Mutating conformance-schema.cjs VALID_ACTIONS without updating validate-traces.cjs:** The schema is the source of truth; trace validation will reject unknown actions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SHA-256 hashing | Custom hash function | `crypto.createHash('sha256')` | Collision resistance, standard, fast |
| TTL expiration | Custom timer/scheduler | Compare `Date.now() - created_ms > ttl_ms` | Simple arithmetic, no timers needed |
| Git HEAD detection | Parse .git/HEAD manually | `git rev-parse HEAD` via spawnSync | Handles detached HEAD, worktrees, submodules |
| Cache directory creation | Manual mkdir -p | `fs.mkdirSync(dir, { recursive: true })` | Idempotent, handles nested paths |
| JSONL parsing | Line-by-line regex | `JSON.parse(line)` per line | Already used throughout hooks and bin/ |

**Key insight:** The entire cache system uses primitives already present in the codebase. No new patterns or libraries are needed.

## Common Pitfalls

### Pitfall 1: Cache Key Instability
**What goes wrong:** Cache key includes volatile data (timestamps, random IDs) causing 0% hit rate.
**Why it happens:** Including too much context in the hash input.
**How to avoid:** Hash only: prompt text, sorted slot names, config hash, git HEAD. Exclude timestamps, session IDs, stderr noise.
**Warning signs:** Cache directory fills with unique files, no cache_hit conformance events.

### Pitfall 2: Stale Cache Serving Wrong Results
**What goes wrong:** Cache returns results for a prompt that is semantically different after a code change.
**Why it happens:** Git HEAD is the same (no new commits between dispatches), but the working tree has changed.
**How to avoid:** Include git HEAD in the cache key. Accept that working-tree-only changes will reuse cache (this is intentional -- the quorum reviews the same prompt, not the code).
**Warning signs:** Quorum approving plans based on stale review context.

### Pitfall 3: pass@k Miscount from Tool-Result Noise
**What goes wrong:** Counting all tool_use blocks as separate rounds when they are part of the same dispatch batch.
**Why it happens:** Claude dispatches multiple slot-workers in one message turn, each as a separate tool_use block.
**How to avoid:** Count unique dispatch *groups* (batches within a single assistant message), not individual tool_use blocks. Or count by the number of `merge-wave` scoreboard updates (one per round).
**Warning signs:** pass@k shows k=4 when only 1 round of 4 parallel dispatches occurred.

### Pitfall 4: Cache Path Not Gitignored
**What goes wrong:** `.planning/.quorum-cache/` gets committed, leaking prompt text and slot responses into git history.
**Why it happens:** Forgetting to add the gitignore entry.
**How to avoid:** Add `.planning/.quorum-cache/` to .gitignore in the same commit that creates the cache infrastructure.
**Warning signs:** `git status` shows untracked files under `.planning/.quorum-cache/`.

### Pitfall 5: nf-stop.js Not Recognizing Cache Hits
**What goes wrong:** Stop hook blocks the response because it cannot find slot-worker evidence in the transcript.
**Why it happens:** Cache hit path bypasses slot dispatch, so the usual evidence (Task tool_use blocks) is absent.
**How to avoid:** nf-stop.js must explicitly recognize `<!-- NF_CACHE_HIT -->` marker and approve without slot evidence.
**Warning signs:** Blocked responses after cache hits; conformance events show `quorum_block` immediately after `cache_hit`.

### Reading/Writing Cache
```javascript
function readCache(cacheKey, cacheDir) {
  try {
    const filePath = path.join(cacheDir, cacheKey + '.json');
    if (!fs.existsSync(filePath)) return null;
    const entry = JSON.parse(fs.readFileSync(filePath, 'utf8'));
    // Validate version
    if (entry.version !== 1) return null;
    // Check TTL
    const age = Date.now() - new Date(entry.created).getTime();
    if (age > (entry.ttl_ms || 3600000)) return null;
    // Check completion (only serve completed entries)
    if (!entry.completed) return null;
    return entry;
  } catch { return null; } // fail-open
}

function writeCache(cacheKey, entry, cacheDir) {
  try {
    fs.mkdirSync(cacheDir, { recursive: true });
    const filePath = path.join(cacheDir, cacheKey + '.json');
    fs.writeFileSync(filePath, JSON.stringify(entry, null, 2), 'utf8');
  } catch {} // fail-open
}
```

### pass@k Reporting in verify-quorum-health.cjs
```javascript
function computePassAtKRates(conformanceEventsPath) {
  const lines = fs.readFileSync(conformanceEventsPath, 'utf8')
    .split('\n')
    .filter(l => l.trim());

  let total = 0, pass1 = 0, pass3 = 0;
  for (const line of lines) {
    try {
      const ev = JSON.parse(line);
      if (ev.action !== 'quorum_complete' || ev.outcome !== 'APPROVE') continue;
      if (ev.pass_at_k === undefined) continue; // pre-PASSK events
      total++;
      if (ev.pass_at_k === 1) pass1++;
      if (ev.pass_at_k <= 3) pass3++;
    } catch {}
  }

  return {
    total,
    pass_at_1: total > 0 ? pass1 / total : 0,
    pass_at_3: total > 0 ? pass3 / total : 0,
    avg_k: total > 0 ? lines.reduce((sum, l) => {
      try {
        const ev = JSON.parse(l);
        return ev.action === 'quorum_complete' && ev.outcome === 'APPROVE' && ev.pass_at_k
          ? sum + ev.pass_at_k : sum;
      } catch { return sum; }
    }, 0) / total : 0,
  };
}
```

## Open Questions

1. **Cache key granularity: should working tree state be included?**
   - What we know: Git HEAD captures committed state. Working tree changes are not reflected.
   - What's unclear: Whether a prompt about uncommitted changes should bust the cache.
   - Recommendation: Exclude working tree from cache key. The quorum reviews the *prompt*, not the diff. If the prompt is identical, the review is identical. Document this as a known limitation.

2. **Cache TTL configuration key name**
   - What we know: Requirement says "configurable, default 1 hour". Config uses flat keys (shallow merge).
   - What's unclear: Key naming convention -- `cache_ttl_ms` or `quorum_cache_ttl_ms`.
   - Recommendation: Use `cache_ttl_ms` at the top level of nf.json. Consistent with existing flat-key pattern (e.g., `hook_profile`, `task_envelope_enabled`).

3. **How to pass cache key from nf-prompt.js to nf-stop.js**
   - What we know: Hooks communicate through Claude's transcript (additionalContext -> user message). Marker pattern established by `<!-- GSD_DECISION -->` and `<!-- NF_SOLO_MODE -->`.
   - What's unclear: Whether cache key should be in the marker or computed independently.
   - Recommendation: Embed in marker: `<!-- NF_CACHE_KEY:abc123... -->`. nf-stop.js extracts it for backfill.

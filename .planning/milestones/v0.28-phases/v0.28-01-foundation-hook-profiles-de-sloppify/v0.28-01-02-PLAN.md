  <name>Task 1: Create cleanup-review.md workflow for Haiku-based code review</name>
  <files>core/workflows/cleanup-review.md</files>
  <action>
Create `core/workflows/cleanup-review.md` -- the prompt template for the cleanup subagent.

The workflow should:

1. **Accept context:** phase directory path, list of files modified in the phase (from git diff or plan summaries), phase name.

2. **Instructions for the subagent:**
   - Read only the files modified in the current phase (passed as list). Cap at 20 files. If more than 20, prioritize newly created files over modified files.
   - Scan each file for three categories:
     - **Redundancy:** Duplicate logic across files (same function/pattern reimplemented), copy-pasted blocks
     - **Dead code:** Unused exports, unreachable branches, functions with no callers within the modified set
     - **Over-defensive patterns:** Unnecessary try/catch wrapping exit(0), triple-nested error handling where a single outer catch suffices, redundant null checks after guaranteed initialization
   - For each finding, record: file path, line number, category, description

3. **Output format:** Write a structured markdown report to `{phase_dir}/CLEANUP-REPORT.md` with this format:
```markdown
# Cleanup Report: Phase {phase_name}

**Generated:** {date}
**Model:** {model_id}
**Files reviewed:** {count}

## Findings

### Redundancy
| File | Line | Description |
|------|------|-------------|
| path/to/file.js | 45 | Description of redundancy |

### Dead Code
| File | Line | Description |
|------|------|-------------|

### Over-Defensive Patterns
| File | Line | Description |
|------|------|-------------|

## Summary
- Redundancy: N findings
- Dead code: N findings
- Over-defensive: N findings
- **Total: N findings**
```

4. **Model tier:** Use Haiku (claude-haiku-4-5-20251001) -- cheap and fast. The workflow should specify this is a worker-tier task.

5. **Error handling:** If the subagent fails or produces no output, log a warning but do NOT fail the phase. The cleanup report is informational only.

Follow the style and structure of existing workflow files in `core/workflows/`. Use markdown with clear sections. Include a note that this workflow is spawned by execute-phase.md after successful verification.

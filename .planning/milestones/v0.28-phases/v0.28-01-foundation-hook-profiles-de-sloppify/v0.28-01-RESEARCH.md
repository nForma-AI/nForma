# Phase v0.28-01: Foundation -- Hook Profiles + De-Sloppify - Research

**Researched:** 2026-03-06
**Domain:** Claude Code hook infrastructure / config-driven hook dispatch / cleanup subagent
**Confidence:** HIGH

## Summary

This phase adds two capabilities to nForma's hook infrastructure: (1) a `hook_profile` config key (`minimal`, `standard`, `strict`) that controls which hooks actively enforce vs exit(0) immediately, and (2) a post-verification cleanup subagent that reviews code produced during execute-phase for redundancy, dead code, and over-defensive patterns.

The hook profile system is architecturally straightforward because every hook already loads config via `loadConfig()` from `hooks/config-loader.js`, and every hook already follows the fail-open pattern (exit 0 on error). Adding a profile check at the top of each hook is a ~5-line guard. The key design decision is WHERE to put the profile-to-hook mapping: it belongs in `config-loader.js` as a shared constant (not duplicated across hooks), with a `shouldRunHook(hookName, profile)` helper that each hook calls on startup. The cleanup subagent is a new workflow file (`core/workflows/cleanup-review.md`) spawned as a Task subagent from the execute-phase workflow after verification succeeds, using the Haiku model tier.

The critical constraint is PROF-04: profile changes must take effect on the next tool call without reinstall. This is inherently satisfied by the current architecture -- hooks read `nf.json` from disk on every invocation (no caching). The only risk would be introducing a module-level cache in config-loader.js, which must NOT happen.

**Primary recommendation:** Add a `HOOK_PROFILE_MAP` constant and `shouldRunHook()` helper to `config-loader.js`, add a 5-line early-exit guard to each hook, create a cleanup workflow for Haiku-based code review, and extend `config-loader.js` validation to handle the new `hook_profile` field.

### Pattern 1: Profile-to-Hook Mapping
**What:** A constant mapping each profile to the set of hooks that should actively run.
**When to use:** Every hook checks this on startup before doing any work.
**Design:**

```javascript
// In config-loader.js

// Maps hook_profile → set of hook basenames that should actively execute.
// Hooks NOT in the set for the active profile exit(0) immediately.
// Circuit breaker is ALWAYS active (all profiles) per success criterion 5.
const HOOK_PROFILE_MAP = {
  minimal: new Set([
    'nf-circuit-breaker',  // Always active (formal: MonitoringReachable)
    'nf-precompact',       // Always active (session continuity)
  ]),
  standard: new Set([
    'nf-circuit-breaker',
    'nf-precompact',
    'nf-prompt',
    'nf-stop',
    'gsd-context-monitor',
    'nf-spec-regen',
    'nf-token-collector',
    'nf-slot-correlator',
    'nf-session-start',
    'nf-statusline',
  ]),
  strict: new Set([
    'nf-circuit-breaker',
    'nf-precompact',
    'nf-prompt',
    'nf-stop',
    'gsd-context-monitor',
    'nf-spec-regen',
    'nf-token-collector',
    'nf-slot-correlator',
    'nf-session-start',
    'nf-statusline',
  ]),
};

// Returns true if the hook should run for the given profile.
// hookBasename: e.g. 'nf-circuit-breaker' (no .js extension)
function shouldRunHook(hookBasename, profile) {
  const validProfile = HOOK_PROFILE_MAP[profile] ? profile : 'standard';
  return HOOK_PROFILE_MAP[validProfile].has(hookBasename);
}
```

**Key insight:** `standard` and `strict` have the SAME hook set. The difference is behavioral: in `strict` mode, `nf-prompt.js` and `nf-stop.js` treat ALL `/nf:` commands as quorum commands (not just the `quorum_commands` list). This is handled inside those two hooks, not by the profile map.

### Pattern 2: Hook Early-Exit Guard
**What:** A 5-line guard at the top of each hook's main function.
**When to use:** Every hook except circuit breaker and precompact (which are always active).

```javascript
// In each hook's main() or top-level execution:
const { loadConfig, shouldRunHook } = require('./config-loader');

// ... after reading stdin and determining cwd/gitRoot ...
const config = loadConfig(gitRoot || process.cwd());
const profile = config.hook_profile || 'standard';

if (!shouldRunHook('nf-prompt', profile)) {
  // Profile disables this hook — exit cleanly (fail-open)
  process.stdout.write(JSON.stringify({ decision: 'allow' }) + '\n');
  process.exit(0);
}
```

**Important:** The guard must come AFTER stdin reading (hooks that don't read stdin will hang) but BEFORE any expensive operations (git log, file reads, etc.).

### Pattern 3: Strict Mode Quorum Extension (PROF-03)
**What:** In `strict` profile, quorum enforcement applies to ALL `/nf:` commands.
**When to use:** Only in `nf-prompt.js` and `nf-stop.js`.
**Design:**

```javascript
// In nf-prompt.js and nf-stop.js:
const profile = config.hook_profile || 'standard';

let effectiveQuorumCommands;
if (profile === 'strict') {
  // PROF-03: In strict mode, ALL /nf: commands require quorum
  // Use a catch-all pattern instead of the quorum_commands list
  effectiveQuorumCommands = null; // signals "match any /nf: command"
} else {
  effectiveQuorumCommands = config.quorum_commands;
}
```

In nf-stop.js, the `buildCommandPattern()` function constructs a regex from `quorum_commands`. For strict mode, replace with a broader regex: `\/(nf|q?gsd):\w+` (matches any command, not just listed ones).

### Pattern 4: Cleanup Subagent (CLEAN-01, CLEAN-02)
**What:** A Task subagent spawned from execute-phase after verification succeeds.
**When to use:** After successful verification in execute-phase workflow.
**Design:**

The cleanup subagent is spawned as a `Task` with `subagent_type="nf-cleanup-reviewer"`:
- Uses Haiku model tier (cheap, fast)
- Reads the list of files modified in the phase (from plan summaries or git diff)
- Scans for: redundancy (duplicate logic), dead code (unused exports/functions), over-defensive patterns (unnecessary try/catch wrapping exit(0))
- Produces a structured JSON report saved to `{phase_dir}/CLEANUP-REPORT.md`

The workflow file (`core/workflows/cleanup-review.md`) provides the cleanup prompt template. The execute-phase workflow spawns it ONLY when verification passes.

### Pattern 5: Config Validation Extension
**What:** Add `hook_profile` to config-loader.js validation.
**When to use:** In `validateConfig()`.

```javascript
// In validateConfig():
const VALID_PROFILES = ['minimal', 'standard', 'strict'];
if (config.hook_profile !== undefined) {
  if (typeof config.hook_profile !== 'string' || !VALID_PROFILES.includes(config.hook_profile)) {
    process.stderr.write('[nf] WARNING: nf.json: hook_profile must be "minimal", "standard", or "strict"; defaulting to "standard"\n');
    config.hook_profile = 'standard';
  }
}
```

And in DEFAULT_CONFIG:
```javascript
const DEFAULT_CONFIG = {
  // ... existing fields ...
  hook_profile: 'standard',  // NEW: minimal | standard | strict
};
```

### Anti-Patterns to Avoid
- **Module-level config caching:** NEVER cache loadConfig() result at module scope. Each hook invocation must re-read nf.json to satisfy PROF-04 (instant profile switching).
- **Per-hook profile logic:** Don't put profile-specific behavior (what hooks run in minimal vs standard) inside each hook. Centralize in config-loader.js HOOK_PROFILE_MAP.
- **Modifying installed hooks:** Profile changes must NOT require re-running `bin/install.js`. All hooks stay installed; the profile gate makes them no-ops.
- **Cleanup subagent blocking execution:** The cleanup report is informational only -- it must not fail the phase or block any subsequent work.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config validation for hook_profile | Custom validation scattered across hooks | Extend existing `validateConfig()` in config-loader.js | Consistent with 15+ existing validated fields |
| Hook-to-profile mapping | Per-hook conditionals checking profile | Shared `HOOK_PROFILE_MAP` + `shouldRunHook()` | Single source of truth, testable in isolation |
| Cleanup report format | Custom output format | Structured markdown with `## Findings` sections and `file:line` format | Consistent with VERIFICATION.md pattern |
| Model tier for cleanup | Hardcoded model string | Use `config.model_tier_worker` or fallback to 'haiku' | Respects user's model tier preferences |

**Key insight:** The hook infrastructure already has all the extension points needed. This phase is about wiring new config into existing patterns, not building new infrastructure.

## Common Pitfalls

### Pitfall 1: Stdin Deadlock on Early Exit
**What goes wrong:** Hook exits before reading stdin, causing the parent process (Claude Code) to hang or error.
**Why it happens:** The profile guard exits before consuming stdin. Some hooks read stdin synchronously, others async.
**How to avoid:** The profile guard MUST come after stdin is fully consumed. For hooks that accumulate stdin with `process.stdin.on('data')` + `process.stdin.on('end')`, the guard goes inside the 'end' handler.
**Warning signs:** Hook hangs in test when given a profile that disables it.

### Pitfall 2: Stdout Corruption on Profile Skip
**What goes wrong:** A hook that's supposed to be disabled still writes partial output to stdout.
**Why it happens:** The hook's early exit path writes a different JSON format than what Claude Code expects.
**How to avoid:** Each hook type has a specific "allow" response format. PreToolUse uses `{}` or no output. Stop uses `{}`. UserPromptSubmit uses `{}`. Check each hook's existing pass-through path and replicate it exactly.
**Warning signs:** Claude Code reports "malformed hook output" in stderr.

### Pitfall 3: Strict Mode Regex Over-Matching
**What goes wrong:** Strict mode's catch-all regex matches non-nForma slash commands (e.g., `/help`, `/compact`).
**Why it happens:** The regex `\/\w+:\w+` is too broad.
**How to avoid:** Strict regex must still scope to nForma prefixes: `\/(nf|q?gsd):\w+`. It just doesn't limit which commands after the prefix.
**Warning signs:** Non-nForma commands get blocked by the stop hook in strict mode.

### Pitfall 4: Cleanup Subagent Context Explosion
**What goes wrong:** Cleanup subagent reads too many files, exceeds context window, produces incomplete report.
**Why it happens:** Phase touches many files; subagent tries to read them all.
**How to avoid:** Limit cleanup to files modified in the current phase (from git diff or plan summaries). Cap at ~20 files. For larger changesets, scan only new/modified files, not all project files.
**Warning signs:** Cleanup subagent response is truncated or takes excessively long.

### Pitfall 5: Config-Loader Shallow Merge Erasing hook_profile
**What goes wrong:** Project-level nf.json doesn't set `hook_profile`, but global does; merge overwrites global with project defaults.
**Why it happens:** Shallow merge `{ ...DEFAULT_CONFIG, ...global, ...project }` -- if project config is `{}`, it doesn't overwrite global. But if project config explicitly sets another field, the shallow merge is fine. The real risk is if someone adds `hook_profile: undefined` to project config.
**How to avoid:** `validateConfig()` already handles this by checking `!== undefined` before validating. Follow the same pattern as `model_tier_planner`.
**Warning signs:** Profile setting in global nf.json is ignored.

## Code Examples

### Hook Profile Guard (for nf-prompt.js pattern -- stdin-accumulating hooks)
```javascript
// hooks/nf-prompt.js -- add after stdin accumulation, inside the 'end' handler
process.stdin.on('end', () => {
  try {
    const payload = JSON.parse(_stdinRaw);
    const cwd = payload.cwd || process.cwd();
    const config = loadConfig(cwd);
    const profile = config.hook_profile || 'standard';

    // Profile gate: exit early if this hook is disabled
    if (!shouldRunHook('nf-prompt', profile)) {
      process.stdout.write(JSON.stringify({}) + '\n');
      process.exit(0);
    }

    // ... rest of existing logic ...
  } catch (err) {
    // Fail-open
    process.exit(0);
  }
});
```

### Hook Profile Guard (for nf-circuit-breaker.js pattern -- sync stdin hooks)
```javascript
// Circuit breaker is ALWAYS active regardless of profile,
// but the guard pattern for other sync-stdin hooks would be:
const input = JSON.parse(stdinContent);
const cwd = input.cwd || process.cwd();
const gitRoot = getGitRoot(cwd);
const config = loadConfig(gitRoot || cwd);

if (!shouldRunHook('nf-stop', config.hook_profile || 'standard')) {
  // Output empty JSON -- Claude Code interprets as "allow"
  process.stdout.write('{}');
  process.exit(0);
}
```

## Open Questions

1. **Should nf-statusline.js be active in minimal profile?**
   - What we know: Statusline is display-only (no enforcement), zero overhead in practice
   - What's unclear: Whether users choosing "minimal" want the status bar too
   - Recommendation: Keep it inactive in minimal (consistent with "minimal = least interference"), but this is a discretionary decision

2. **Should nf-session-start.js (keychain sync) be active in minimal profile?**
   - What we know: It syncs secrets to ~/.claude.json -- important for MCP functionality
   - What's unclear: If a user on "minimal" still uses MCP servers that need secrets
   - Recommendation: Keep it inactive in minimal. Users on minimal likely want bare-bones experience. If MCP breaks, they can switch to standard.

3. **Cleanup subagent file count cap**
   - What we know: Haiku context window is smaller than Opus/Sonnet
   - What's unclear: Exact file count that fits comfortably
   - Recommendation: Start with 20 files max, prioritize new/modified files. Can adjust based on experience.

## Summary

Phase v0.19-08 closes two critical requirements (MCPENV-02, MCPENV-04) from the v0.19 FV Pipeline Hardening milestone by integrating two existing but partially-integrated formal models into the verification pipeline:

1. **QGSDMCPEnv.tla** (MCPENV-02) — TLA+ model of MCP call environment semantics, already created in v0.19-05 but missing from orchestrator and TLC runners
2. **mcp-availability.pm** (MCPENV-04) — PRISM model of per-slot MCP availability, exists but has three defects: dead module.exports export, missing composite-key PRISM constant filtering, and missing from `run-formal-verify.cjs` STEPS

The phase also creates the missing formal specification document `formal/spec/mcp-calls/invariants.md`, which is **required** by LIVE-02 enforcement (emits `result=inconclusive` when fairness declaration is absent).

The integration spans:
- TLA+ runner updates (SURFACE_MAP in run-tlc.cjs)
- PRISM pipeline wiring (run-formal-verify.cjs STEPS array + CI workflow)
- Composite key guard (run-prism.cjs constant name sanitization)
- Formal specification documentation (fairness declarations for EventualDecision)

**Primary recommendation:** Approach as two sequential efforts: (1) documentation + TLA+ runner integration (creates the formal spec, registers model in TLC runner), (2) PRISM model fixes + orchestrator wiring (defect fixes + pipeline integration). This separates the documentation requirement (blocks any liveness property in MCMCPEnv.cfg) from the PRISM implementation work.

## Architecture Patterns

### TLA+ Model Registration Pattern

TLA+ models are discovered and run via the SURFACE_MAP in `bin/run-tlc.cjs` (lines 24–33):

```javascript
// Source: bin/run-tlc.cjs
const SURFACE_MAP = {
  'MCliveness':            'quorum',
  'MCdeliberation':        'deliberation',
  'MCprefilter':           'prefilter',
  'MCrecruiting-liveness': 'recruiting',
  'MCbreaker':             'breaker',
  'MCconvergence':         'convergence',
  'MCoscillation':         'oscillation',
  'MCaccount-manager':     'account-manager',
  // MCPENV-02: Add MCMCPEnv mapping
};
```

**Purpose:** Maps TLC config filename (e.g., `MCMCPEnv.cfg`) to spec subdirectory (e.g., `formal/spec/mcp-calls/`) for liveness property fairness validation. When a config file contains `PROPERTY` lines, `detectLivenessProperties()` looks up the surface, finds the corresponding `invariants.md`, and verifies that each property has a fairness declaration.

**Add entry:** `'MCMCPEnv': 'mcp-calls'` — this tells run-tlc.cjs to look for fairness declarations in `formal/spec/mcp-calls/invariants.md`.

### Liveness Property Fairness Declaration Pattern

Each TLA+ liveness property **must** have a declaration in the corresponding `formal/spec/<surface>/invariants.md`. Pattern from `formal/spec/quorum/invariants.md`:

```markdown
# Liveness Fairness Declarations: quorum

**Spec source:** `formal/tla/QGSDQuorum.tla`
**Config:** `formal/tla/MCliveness.cfg`

## EventualConsensus

**Property:** `EventualConsensus == <>(phase = "DECIDED")`
**Config line:** `PROPERTY EventualConsensus` (MCliveness.cfg)
**Fairness assumption:** WF_vars on 4 actions: Decide, StartQuorum, AnyCollectVotes, AnyDeliberate
**Realism rationale:** [explanation of why fairness assumption holds in deployed system]

**Source:** `formal/tla/QGSDQuorum.tla`, lines 141, 148–152
```

**For MCPENV-02:** The spec defines `EventualDecision == <>(quorumPhase = "DECIDED")` with fairness `WF_vars(QuorumProcessOutcomes), WF_vars(QuorumDecide), WF_vars(TimeoutAction)` in the Spec definition (lines 110–113 of QGSDMCPEnv.tla). The invariants.md must declare this rationale: weak fairness is justified because call-quorum-slot.cjs has finite timeout (120s default), and the event loop in qgsd-prompt.js guarantees enabled actions fire eventually (IO-bounded system).

### Formal Verification Pipeline Integration Pattern

The `run-formal-verify.cjs` STEPS array orchestrates all formal verification tools. STEPS entries for MCP models follow the pattern:

```javascript
// TLA+ model
{ tool: 'tla', id: 'tla:mcp-environment',
  label: 'TLA+ QGSDMCPEnv — MCMCPEnv',
  type: 'node', script: 'run-tlc.cjs', args: ['MCMCPEnv'],
}

// PRISM model
{ tool: 'prism', id: 'prism:mcp-availability',
  label: 'PRISM mcp-availability — per-slot MCP availability rates',
  type: 'node', script: 'run-prism.cjs', args: ['--model=mcp-availability'],
}
```

**Integration points:**
1. TLA+ step calls `run-tlc.cjs MCMCPEnv`, which reads `formal/tla/MCMCPEnv.cfg` and invokes TLC
2. PRISM step calls `run-prism.cjs --model=mcp-availability`, which reads `formal/prism/mcp-availability.props` and injects per-slot rates from quorum-scoreboard.json

### PRISM Constant Naming and Slot Key Filtering

MCP slot names can contain hyphens (e.g., `codex-1`, `claude-1`) and potentially colons if composite keys are used (e.g., `claude-1:deepseek-ai/DeepSeek-V3.2`).

**Current transformation** (run-prism.cjs line 272):
```javascript
const constName = slot.replace(/-/g, '_') + '_avail';  // 'codex-1' -> 'codex_1_avail'
```

**Issue:** If slot name contains colon or slash, the constant name becomes invalid for PRISM:
- `claude-1:deepseek-ai/DeepSeek-V3.2` → `claude_1:deepseek_ai/DeepSeek_V3_2_avail` (INVALID PRISM identifier)

**Fix pattern:** Filter slots by checking for composite keys before constant generation. This prevents passing invalid constant names to PRISM.

### readMCPAvailabilityRates Export Pattern

The function is exported for testing:

```javascript
// Source: bin/run-prism.cjs, end of file
if (typeof module !== 'undefined') {
  module.exports = { readMCPAvailabilityRates };
}
```

**Pattern rationale:** Allows test files to import and unit-test the rate-reading logic without requiring PRISM installation. Tests can `spawnSync()` run-prism.cjs in a temporary directory and inject fixtures via environment.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TLA+ state-space exploration | Custom state-space traversal | TLC model checker (tla2tools.jar) | TLC implements optimized symmetry reduction, deadlock detection, and cone-of-influence reduction; hand-rolled traversal would miss critical performance optimizations |
| Fairness assumption validation | Ad-hoc fairness checkers | TLC with WF_/SF_ declarations + detectLivenessProperties() in run-tlc.cjs | TLC is the authority on fairness; manual fairness checking is error-prone. detectLivenessProperties() is the QGSD-specific structural enforcement layer |
| Markov chain steady-state analysis | Custom matrix computation | PRISM's built-in steady-state solver | PRISM uses optimized algorithms (power iteration, Jacobi); hand-rolled solvers are numerically unstable |
| PRISM constant injection | String concatenation | Structured args array (spawnSync) | String concatenation is error-prone; structured args prevent injection issues and ensure proper quoting |

**Key insight:** QGSD's formal verification pipeline relies on **external tools as authorities**. TLC and PRISM are the only correct sources of truth for state-space correctness and probability quantification. The orchestrator's role is to wire models into these tools and structure the output (check-results.ndjson), not to replicate their algorithms.

## Common Pitfalls

### Pitfall 1: Missing Fairness Declaration for Liveness Properties

**What goes wrong:** A TLC config file contains `PROPERTY EventualDecision` but `formal/spec/mcp-calls/invariants.md` has no `## EventualDecision` section. TLC runs the property, but `detectLivenessProperties()` in run-tlc.cjs emits `result=inconclusive` (LIVE-02), and the property result is discounted until the fairness assumption is documented.

**Why it happens:** Liveness properties are **meaningless without fairness assumptions**. If the TLC config says "check that DECIDED is reached eventually" but doesn't specify fairness (WF, SF), the property is vacuously true in systems with guaranteed divergence or stutter. LIVE-02 enforces this by requiring explicit fairness declarations before a liveness result counts.

**How to avoid:** Before committing a .cfg file with `PROPERTY` lines, create the corresponding invariants.md section declaring fairness. Verification: check that `detectLivenessProperties()` returns empty array (meaning fairness declarations were found).

**Warning signs:**
- `result=inconclusive` in check-results.ndjson for a property you expected to pass/fail
- `detectLivenessProperties()` returning non-empty array in test

### Pitfall 2: Invalid PRISM Constant Names from Composite Slot Keys

**What goes wrong:** Scoreboard contains a composite key like `claude-1:deepseek-ai/DeepSeek-V3.2` (base slot + provider). run-prism.cjs transforms it to `claude_1:deepseek_ai/DeepSeek_V3_2_avail` and passes it to PRISM. PRISM rejects it as an invalid identifier (colons and slashes not allowed in constant names).

**Why it happens:** Slot name composition may use `:` (colon) to separate base slot from provider, or `/` (slash) to denote model path. The code replaces only `-` (hyphen), assuming slot names are simple. If composite keys enter the scoreboard, the transformation fails silently (or causes PRISM parse error).

**How to avoid:** Add explicit filtering for composite keys before constant generation. Skip any slot name containing `:` or `/`.

**Warning signs:**
- PRISM error: `"Unexpected character" or "Invalid constant name"`
- Test output shows slot names with colons/slashes in the PRISM Args line

### Pitfall 3: SURFACE_MAP Entry Missing for New TLA+ Model

**What goes wrong:** You add a new TLA+ config (e.g., `MCMCPEnv.cfg`) with `PROPERTY EventualDecision`. You run `run-tlc.cjs MCMCPEnv`, but `detectLivenessProperties()` silently returns `[]`. Actually, properties **were** detected, but the SURFACE_MAP lookup failed (no entry for 'MCMCPEnv'), so the surface is `null` and the function returns early without checking invariants.md.

**Why it happens:** detectLivenessProperties() does SURFACE_MAP[configName] lookup and returns `[]` if `surface === null` (line 67–69 of run-tlc.cjs). This is a deliberate no-op for unknown configs (fail-open). But it means a new model won't be validated until the SURFACE_MAP entry is added.

**How to avoid:** Add SURFACE_MAP entry **before** creating a new .cfg file with liveness properties. The pattern is `'MCXyz': 'xyz'` (config name → spec subdirectory name).

**Warning signs:**
- TLC runs but no `result=inconclusive` in check-results.ndjson (liveness properties were silently ignored)
- Test stub for liveness property passes unexpectedly

### Pitfall 4: Scoreboard Unavailable in CI, Priors Used Instead of Empirical Rates

**What goes wrong:** Local run: `run-prism.cjs --model=mcp-availability` reads quorum-scoreboard.json and injects empirical rates (e.g., `codex_1_avail=0.94`). CI run: scoreboard doesn't exist (CI has no quorum history), so readMCPAvailabilityRates() returns null, and priors are used (`codex_1_avail=0.85`). Property passes locally but results differ in CI.

**Why it happens:** CI jobs are ephemeral; they don't have a `quorum-scoreboard.json` from previous runs. CALIB-02 specifies that PRISM checks should emit `warn` (not `fail`) during cold-start. But if the implementation doesn't handle this gracefully, tests become flaky.

**How to avoid:**
1. run-prism.cjs already handles null case gracefully (line 91: `catch (_) return null`)
2. Empirical injection is conditional (line 270: `if (activeMcpRates) { ... }`)
3. Priors are only used when activeMcpRates is null
4. CALIB-02 enforcement (warn vs fail) is in readPolicy() + coldStartState check

Tests should verify both paths: one with fixture scoreboard (empirical rates) and one without (priors).

**Warning signs:**
- Property result differs between local and CI runs
- Observation window in check-results.ndjson shows `null` metadata or `"priors"`

### Declaring Fairness for an EventualDecision Property

```markdown
# Liveness Fairness Declarations: mcp-calls

**Spec source:** `formal/tla/QGSDMCPEnv.tla`
**Config:** `formal/tla/MCMCPEnv.cfg`

## EventualDecision

**Property:** `EventualDecision == <>(quorumPhase = "DECIDED")`
**Config line:** `PROPERTY EventualDecision` (MCMCPEnv.cfg)
**Fairness assumption:** WF_vars(QuorumProcessOutcomes), WF_vars(QuorumDecide), WF_vars(TimeoutAction)
**Realism rationale:** The TLA+ spec models MCP servers as nondeterministic environment processes.
Quorum collects responses from multiple MCP slots in parallel via call-quorum-slot.cjs subprocesses.
Each subprocess has a finite timeout (120s default, configurable via CLAUDE_MCP_TIMEOUT_MS).
Once a timeout expires or a response is received, the subprocess terminates and the response is
queued. The qgsd-prompt.js event loop processes the response queue and enables either QuorumDecide
(if threshold reached) or TimeoutAction (if all pending calls expired). By weak fairness, enabled
actions must eventually fire — the event loop does not permanently skip an enabled action. Therefore,
quorum must eventually reach DECIDED.

**Source:** `formal/tla/QGSDMCPEnv.tla`, lines 110–113 (Spec with fairness), lines 141–144 (EventualDecision property)
```

### Filtering Composite Keys in Constant Name Generation

```javascript
// Source pattern: bin/run-prism.cjs, lines 268–275 with filtering
// Only process base slot names (no colons or slashes)
if (useMCPAvailabilityModel) {
  if (activeMcpRates) {
    for (const [slot, rate] of Object.entries(activeMcpRates)) {
      // Filter: skip composite keys (contain ':' or '/')
      if (slot.includes(':') || slot.includes('/')) {
        process.stderr.write(`[run-prism] Skipping composite key: ${slot}\n`);
        continue;
      }
      const constName = slot.replace(/-/g, '_') + '_avail';
      prismArgs.push('-const', constName + '=' + rate);
    }
  }
}
```

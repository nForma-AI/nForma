## Summary

Phase v0.19-11 closes two items of documented tech debt from the v0.19 milestone audit.

**Item 1 — UNIF-03 (the only real implementation work):** `run-formal-verify.cjs` executes its triage summary read of `check-results.ndjson` at the end of `runOnce()`. However, two CI steps in `.github/workflows/formal-verify.yml` — `check-trace-redaction.cjs` and `check-trace-schema-drift.cjs` — run as separate workflow steps AFTER the `run-formal-verify.cjs` step completes. These two scripts call `writeCheckResult()` which appends entries to `check-results.ndjson`. Because the orchestrator reads the NDJSON file inside its own `runOnce()` (before the post-orchestrator CI steps run), it systematically under-counts: the summary shows N entries when N+2 actually exist by CI end. The fix is to add both scripts as `STEPS` entries inside the orchestrator's STEPS array so they run inside `runOnce()`, which moves their writes before the summary read. This closes the timing gap without moving the summary-read position.

## Bug Analysis: The UNIF-03 Timing Problem

### Root Cause

The `run-formal-verify.cjs` STEPS array (line 48–176) does NOT include `check-trace-redaction.cjs` or `check-trace-schema-drift.cjs`. These two scripts are only referenced as separate workflow steps in the GitHub Actions CI YAML, run after the master orchestrator step.

**CI step order in `.github/workflows/formal-verify.yml`:**
```
Step 1-9:   setup + generate + TLC + Alloy + PRISM (tooling steps)
Step 10:    node bin/run-formal-verify.cjs          ← orchestrator (includes NDJSON summary read)
Step 11:    node bin/check-trace-redaction.cjs       ← appends redaction entry AFTER summary
Step 12:    node bin/check-trace-schema-drift.cjs    ← appends drift entry AFTER summary
Step 13:    node bin/check-results-exit.cjs          ← reads full NDJSON (correct, sees all entries)
```

The NDJSON summary in the orchestrator `runOnce()` (lines 335–350 in `run-formal-verify.cjs`) reads `check-results.ndjson` at Step 10 time. Steps 11 and 12 append 2 more entries after that read. So the orchestrator triage summary is always missing 2 entries compared to what `check-results-exit.cjs` reports at Step 13.

### Fix Strategy

Add `check-trace-redaction.cjs` and `check-trace-schema-drift.cjs` to the `STEPS` array in `run-formal-verify.cjs`. This brings them inside `runOnce()` so their writes happen before the NDJSON summary read at the end of `runOnce()`.

**Proposed STEPS entries:**
```javascript
// ─ CI enforcement — redaction + schema drift ──────────────────────────────
{
  tool: 'ci', id: 'ci:trace-redaction',
  label: 'Trace redaction enforcement (check-trace-redaction.cjs)',
  type: 'node', script: 'check-trace-redaction.cjs', args: [],
},
{
  tool: 'ci', id: 'ci:trace-schema-drift',
  label: 'Trace schema drift guard (check-trace-schema-drift.cjs)',
  type: 'node', script: 'check-trace-schema-drift.cjs', args: [],
},
```

**Impact on orchestrator step count:** The file header comment says "Total: 21 steps" — after adding 2 CI steps it becomes "Total: 23 steps". The header comment needs updating.

**Impact on STEPS filter:** `--only=ci` would become a valid filter value. The existing filter logic handles this correctly without modification (it filters `s.tool === only`).

**Impact on existing tests:** The `parallelization smoke (PERF-01)` test only checks TLA+ step IDs — unaffected. The `timing` test uses `--only=generate` — unaffected. The `--only=tla` test checks 8 TLA+ step IDs — currently correct. No existing tests break.

**Note:** The redaction and drift steps were already running in CI (as separate workflow steps) so their behavior is not changing — only whether they run inside or outside the orchestrator. The GitHub Actions YAML steps for these should remain as-is (they are still needed as standalone CI steps to provide explicit job failure messages). The NDJSON truncation at the start of `runOnce()` (line 283: `fs.writeFileSync(ndjsonPath, '', 'utf8')`) means each orchestrator run starts fresh — this is correct behavior.

## Architecture Patterns

### Current STEPS Structure (23 steps after fix)

```
STEPS = [
  // generate (2) — phase 1 (sequential prerequisite)
  // petri    (2) — phase 2 (concurrent)
  // tla      (9) — phase 2 (concurrent) [was 8, now 9 with tla:mcp-environment]
  // alloy    (7) — phase 2 (concurrent)
  // prism    (3) — phase 2 (concurrent) [was 2, now 3 with prism:mcp-availability]
  // ci       (2) — NEW: phase 2 (concurrent or sequential — see pitfall below)
]
```

### Pattern: Adding New Tool Group

Existing groups: `generate`, `petri`, `tla`, `alloy`, `prism`. Adding `ci` follows the same pattern — no changes to `runGroup()` or `Promise.all()` dispatch logic needed.

The `toolGroupNames` computation on line 306 (`[...new Set(toolSteps.map(s => s.tool))]`) will automatically pick up `ci` as a group name.

### Pattern: Step Type = 'node'

Both new STEPS use `type: 'node'` (same as `prism`, `tla`, etc.) which calls `runNodeStep()`. Both scripts are `.cjs` files in `bin/`. This is consistent with existing steps.

### Anti-Patterns to Avoid

- **Do not put ci steps in the generate group.** The generate group is a sequential prerequisite. The ci steps have no ordering dependency relative to TLC/Alloy/PRISM steps — they run on `formal/trace/` artifacts that are static during a verification run.
- **Do not change the NDJSON summary position.** The summary read at the end of `runOnce()` is the correct position — the fix is to move the appending tools INTO `runOnce()`, not to move the summary read outside.
- **Do not remove the GitHub Actions workflow steps.** The individual `check-trace-redaction.cjs` and `check-trace-schema-drift.cjs` steps in `formal-verify.yml` should remain — they provide explicit CI step names and failure messages. Running them twice (inside orchestrator + standalone) is safe because both scripts are idempotent: they write exactly 1 NDJSON entry per invocation, and `check-results.ndjson` is truncated at the START of the orchestrator run (so no double-counting occurs within a single `runOnce()` call). In CI, the standalone steps run after the orchestrator has already truncated and rewritten the file, so they append their entries to the fresh file again — this is the existing behavior and remains correct.

## Common Pitfalls

### Pitfall 1: Step Count Mismatch in Header Comment
**What goes wrong:** The header comment says "Total: 21 steps" — if not updated to 23 it becomes misleading.
**Why it happens:** Header comment is manually maintained, not computed from STEPS.
**How to avoid:** Update the header comment AND the `Usage:` section at the top of `run-formal-verify.cjs`.
**Warning signs:** Comment says "21 steps" but `STEPS.length` returns 23.

### Pitfall 2: Breaking Existing Test — parallelization smoke
**What goes wrong:** The `parallelization smoke (PERF-01)` test lists exactly 8 TLA+ step IDs. If `tla:mcp-environment` was added earlier (v0.19-08), the test may already be checking 8 IDs and NOT including `tla:mcp-environment`. If so, that test should be checked.
**Why it happens:** Tests were written before the STEPS array grew.
**How to avoid:** Run existing tests before and after, verify they still pass.
**Verification:** `node --test bin/run-formal-verify.test.cjs` — all 8 tests must still pass.

### Pitfall 3: ci steps group and --only filter collision
**What goes wrong:** If a user passes `--only=ci`, they get just the 2 redaction+drift steps. This is new behavior (previously `--only=ci` would exit with "Unknown --only value").
**Why it happens:** The filter code accepts any `tool` value that exists in STEPS.
**How to avoid:** This is correct, not a bug. Just document that `--only=ci` is now valid.

### Pitfall 4: NDJSON truncation + double-run
**What goes wrong:** If CI runs the orchestrator AND the standalone drift/redaction steps in sequence, the NDJSON gets entries from both the orchestrator's internal `ci:trace-redaction` step AND the standalone workflow step — potentially causing double entries.
**Why it happens:** The standalone CI step writes to the same `check-results.ndjson` that the orchestrator already wrote to.
**How to avoid:** This is acceptable because: (1) the orchestrator truncates NDJSON at the start of each run, (2) the standalone CI steps run after the orchestrator, so they append to the file that already has orchestrator entries. The `check-results-exit.cjs` reads all entries including duplicates, but having 2 redaction entries (both pass or both fail) does not cause false CI failures. If this is considered undesirable, the standalone CI steps can be removed from `formal-verify.yml` once they are inside the orchestrator — but that is a scope decision for the planner.

### Pitfall 5: REQUIREMENTS.md checkbox task is already done
**What goes wrong:** The phase description says to update UNIF-01..04 checkboxes from `[ ]` to `[x]`. But current `REQUIREMENTS.md` already shows all 4 as `[x]` with zero unchecked items remaining.
**Why it happens:** v0.19-09 resolved all checkbox debt. The milestone audit's tech_debt list was written before v0.19-09 executed.
**How to avoid:** The planner should create a single grep-verification task that confirms the already-correct state and marks the task done. No file edits needed.

## Open Questions

1. **Should the standalone CI workflow steps be removed after adding them to STEPS?**
   - What we know: Keeping them in YAML means they run twice per CI job (once inside orchestrator, once standalone). Both invocations write to NDJSON.
   - What's unclear: Whether double entries cause any downstream issue for `check-results-exit.cjs` (it counts all entries, including duplicates — but 2 pass entries vs 1 doesn't affect exit code).
   - Recommendation: Keep the standalone CI steps in `formal-verify.yml` for now. They provide explicit CI step labels and independent failure reporting. This is consistent with other steps like PRISM, which also runs both inside `run-formal-verify.cjs` AND as standalone CI steps.

2. **Should `--only=ci` be documented in the Usage block at the top of run-formal-verify.cjs?**
   - What we know: Other tool categories (`tla`, `alloy`, `prism`, `petri`, `generate`) are listed in the Usage comment.
   - Recommendation: Yes, add `--only=ci` to the Usage comment for completeness.

### Why This Fixes UNIF-03

**Before:** `.github/workflows/formal-verify.yml` ran:
1. `run-formal-verify.cjs` (step 10) — reads NDJSON, summary shows N entries
2. `check-trace-redaction.cjs` (step 11) — appends 1 entry → NDJSON now N+1
3. `check-trace-schema-drift.cjs` (step 12) — appends 1 entry → NDJSON now N+2
4. `check-results-exit.cjs` (step 13) — reads all N+2 entries

**After:** `run-formal-verify.cjs` now:
1. Runs all steps including ci:trace-redaction and ci:trace-schema-drift inside `runOnce()`
2. Both CI scripts write their entries BEFORE the NDJSON summary read at the end
3. Summary sees all N+2 entries
4. `check-results-exit.cjs` still sees N+2 (or N+4 if standalone steps also run, but both invocations are idempotent)

- **STEPS array:** Now 23 steps (was 21)

- **Parallel execution:** Both CI steps run concurrently with TLA+/Alloy/PRISM (no sequential constraint)

## Key Decisions

1. **Keep standalone CI workflow steps:** Even though the scripts now run inside the orchestrator, the `.github/workflows/formal-verify.yml` steps are kept intact. This provides explicit CI step naming and failure reporting. Double-entry behavior is safe because `check-results.ndjson` is truncated at the start of each `runOnce()`.

2. **New tool category ('ci'):** Added 'ci' as a new tool group in STEPS rather than combining with existing categories. This maintains tool isolation and documentation clarity.

3. **Idempotent script pattern:** Confirmed both `check-trace-redaction.cjs` and `check-trace-schema-drift.cjs` are idempotent (write exactly 1 entry per invocation), so double execution is safe.

# Phase v0.19-04: Enforcement Layer - Research

**Researched:** 2026-02-27
**Domain:** Redaction enforcement, evidence confidence qualification, and trace schema drift detection
**Confidence:** HIGH

## Summary

Phase v0.19-04 hardens the formal verification pipeline with three enforcement mechanisms:

1. **Redaction Enforcement (REDACT-01..03):** PII/secret redaction is structurally enforced in CI via `check-trace-redaction.cjs`, which validates all trace event files against a `formal/trace/redaction.yaml` policy. Forbidden keys and value patterns (regex) are detected before traces leave the repository.

2. **Evidence Confidence Qualification (EVID-01..02):** The `validate-traces.cjs` tool (from v0.19-01 baseline) must emit `never_observed` evidence with time-qualified confidence tiers (low/medium/high), eliminating bare "absence of evidence" claims. Confidence thresholds map to quorum rounds and calendar days accumulated during observation.

3. **Trace Schema Drift Guard (DRIFT-01..02):** `check-trace-schema-drift.cjs` detects when `formal/trace/trace.schema.json` is modified without atomically co-updating both the validator (`validate-traces.cjs`) and trace emitter files in the same commit. CI fails on non-atomic schema changes to prevent validator/emitter skew.

This phase depends on v0.19-01 (unified NDJSON output) and adds three new tools that conform to the `check-results.ndjson` schema introduced in v0.19-01. All three tools append a single line per check run with `formalism=redaction` or `formalism=trace` and follow the shared `writeCheckResult()` pattern established in v0.19-01.

**Primary recommendation:** Implement in three parallel workstreams:
- **Stream 1 (Redaction):** Publish `formal/trace/redaction.yaml`, create `bin/check-trace-redaction.cjs` with banned key/pattern detection, wire into CI.
- **Stream 2 (Confidence):** Extend `validate-traces.cjs` to emit `n_rounds`, `window_days`, `confidence` tier in `never_observed` metadata; define thresholds in code or config.
- **Stream 3 (Drift):** Create `bin/check-trace-schema-drift.cjs` to detect schema file changes without co-modification of validator/emitter; wire into CI.

### Pattern 1: Redaction Policy (redaction.yaml)

**What:** A YAML configuration file defining PII and secret redaction rules.

**Structure:**
```yaml
# formal/trace/redaction.yaml
# Redaction policy: forbidden keys and value patterns in trace events

forbidden_keys:
  - "api_key"
  - "secret"
  - "password"
  - "token"
  - "authorization"
  - "bearer"

forbidden_patterns:
  - name: "aws_secret_access_key"
    regex: "^AKIA[0-9A-Z]{16}$"
  - name: "github_pat"
    regex: "^ghp_[A-Za-z0-9_]{36}$"
  - name: "email"
    regex: "^[\\w\\.-]+@[\\w\\.-]+\\.\\w+$"
```

**When to use:** Load once at `check-trace-redaction.cjs` startup. Reject any trace event file containing a forbidden key name or matching a forbidden pattern.

**Example pattern validation:**
```javascript
// Source: pattern-based PII detection (standard in data masking pipelines)
const patterns = redactionPolicy.forbidden_patterns;
for (const event of traceEvents) {
  for (const key in event) {
    for (const pattern of patterns) {
      if (new RegExp(pattern.regex).test(String(event[key]))) {
        violations.push({ event, key, pattern: pattern.name, value: event[key] });
      }
    }
  }
}
```

### Pattern 2: Redaction Checker (check-trace-redaction.cjs)

**What:** A standalone tool that validates all trace event files in a directory against `redaction.yaml`.

**When to use:** Run in CI after trace events are collected. Exit 0 if all events pass; exit 1 if violations found. Write one NDJSON line to `check-results.ndjson`.

**Execution flow:**
```
1. Load redaction.yaml
2. Find all .json or .jsonl files in formal/trace/ (or env-configurable directory)
3. For each file: parse lines, test each line's keys and values
4. Collect violations (forbidden key found OR forbidden pattern matched)
5. Call writeCheckResult({ tool: 'check-trace-redaction', formalism: 'redaction', result: pass/fail, metadata: { violations: [...] } })
6. Exit 0 (success/failure handled by CI via check-results-exit.cjs)
```

**Key implementation notes:**
- Parse redaction.yaml via simple line-by-line regex (no external YAML lib)
- For each trace event JSON: iterate keys and values
- Test value against all forbidden_patterns regex
- Accumulate violations array with event location, key name, pattern matched
- Do NOT fail fast — collect all violations before writing NDJSON
- Return both count and sample violations in metadata for debugging

### Pattern 3: Evidence Confidence Qualification (validate-traces.cjs update)

**What:** Extend the existing trace validator to emit confidence tiers (`low`/`medium`/`high`) for `never_observed` evidence.

**Confidence thresholds:**
```javascript
const CONFIDENCE_TIERS = {
  low:    { min_rounds: 0,    min_days: 0,   label: 'low' },    // < 50 rounds OR < 3 days
  medium: { min_rounds: 500,  min_days: 14,  label: 'medium' }, // ≥ 500 rounds AND ≥ 14 days
  high:   { min_rounds: 10000, min_days: 90, label: 'high' }    // ≥ 10k rounds AND ≥ 90 days
};

function assignConfidence(n_rounds, window_days) {
  // Exact thresholds per EVID-02
  if (n_rounds >= 10000 && window_days >= 90) return 'high';
  if (n_rounds >= 500 && window_days >= 14) return 'medium';
  return 'low';
}
```

**When to use:** Every time `validate-traces.cjs` emits `never_observed` evidence, include `n_rounds`, `window_days`, and `confidence` tier in the metadata.

**Example metadata:**
```javascript
{
  n_rounds: 215,
  window_days: 5,
  confidence: 'low',
  reason: 'path_never_observed'
}
```

**Data source:** `n_rounds` comes from quorum-scoreboard.json (count of deliberation rounds); `window_days` is computed from first/last timestamp in scoreboard.

### Pattern 4: Schema Drift Guard (check-trace-schema-drift.cjs)

**What:** A tool that detects when `formal/trace/trace.schema.json` is modified without atomically co-updating the validator and emitter in the same commit.

**When to use:** Run in CI as a pre-flight check. If schema.json was modified in the current commit, verify that `validate-traces.cjs` and one or more emitter files were also modified.

**Execution flow:**
```
1. Get the current commit hash (git rev-parse HEAD)
2. Run git diff --name-only HEAD~1 to get modified files
3. If formal/trace/trace.schema.json is NOT in the list, exit 0 (no schema change)
4. If schema.json IS modified:
   a. Check if bin/validate-traces.cjs was also modified
   b. Check if any emitter files were modified (grep for common emitter patterns)
   c. If validator was NOT modified OR no emitters were modified: FAIL
   d. Otherwise: PASS
5. Write NDJSON line with result=pass/fail
```

**Emitter detection:** Search git commit for files that export or call trace-emitting functions. Common patterns:
- Files importing `conformance-events.jsonl` or trace-writing modules
- Files with `emit()`, `writeTrace()`, `appendTraceEvent()` function calls

**Example check (using safe subprocess execution):**
```javascript
// Detect schema change in this commit
// Source: v0.19-02 policy.yaml reading pattern, adapted for git inspection
const { execFileSync } = require('child_process');
const fs = require('fs');

try {
  const changedFiles = execFileSync('git', ['diff', '--name-only', 'HEAD~1'], { encoding: 'utf8' })
    .split('\n')
    .filter(f => f.trim().length > 0);

  const hasSchemaChange = changedFiles.includes('formal/trace/trace.schema.json');

  if (!hasSchemaChange) {
    writeCheckResult({ tool: 'check-trace-schema-drift', formalism: 'trace', result: 'pass', metadata: { reason: 'no-schema-change' } });
    process.exit(0);
  }

  // If schema changed, require co-update
  const hasValidatorChange = changedFiles.includes('bin/validate-traces.cjs');
  const hasEmitterChange = changedFiles.some(f => f.includes('conformance') || f.includes('trace-emit'));

  if (!hasValidatorChange || !hasEmitterChange) {
    writeCheckResult({ tool: 'check-trace-schema-drift', formalism: 'trace', result: 'fail', metadata: {
      reason: 'schema-drift-detected',
      schema_changed: true,
      validator_updated: hasValidatorChange,
      emitter_updated: hasEmitterChange
    } });
    process.exit(1);
  }

  writeCheckResult({ tool: 'check-trace-schema-drift', formalism: 'trace', result: 'pass', metadata: {
    reason: 'schema-change-atomic',
    files_changed: changedFiles.length
  } });
} catch (err) {
  // git command failed
  writeCheckResult({ tool: 'check-trace-schema-drift', formalism: 'trace', result: 'fail', metadata: {
    reason: 'git-error',
    error: err.message
  } });
  process.exit(1);
}
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing for redaction policy | Custom recursive YAML parser | Manual regex/split parsing (key: value lines) | YAML is simple key-value here; full parser is overkill; regex is sufficient and has zero deps |
| Regex-based PII detection | Custom detector from scratch | Redaction.yaml policy + standard regex engine | Policy-as-code is maintainable; regex is built-in; policy updates don't require code changes |
| Git history inspection for schema drift | Shell scripting or spawning multiple git commands | Single `git diff --name-only HEAD~1` + `execFileSync` | Built-in Node.js; one command; no external git library needed; safe subprocess execution |
| Confidence scoring algorithm | Manual threshold logic per check | Centralized CONFIDENCE_TIERS constant with helper function | Single source of truth; easy to update thresholds; reusable across multiple validators |
| Evidence collection from conformance log | Ad-hoc timestamp parsing | Use quorum-scoreboard.json as authoritative source of rounds + dates | Scoreboard is already computed; eliminates double-parsing of conformance-events.jsonl |

**Key insight:** These three enforcement layers succeed because they are policy-driven (redaction.yaml, CONFIDENCE_TIERS, schema drift detection rules), not custom logic. Policy changes don't require code changes; new rules are added to YAML or threshold constants.

## Common Pitfalls

### Pitfall 1: Regex Performance on Large Trace Files

**What goes wrong:** Forbidden patterns with complex regex (e.g., `.*secret.*`) are tested against every value in every trace event. On a 10MB conformance-events.jsonl, this becomes O(n*m) where n=event count and m=pattern count.

**Why it happens:** Naive implementation iterates all events × all values × all patterns without optimization.

**How to avoid:**
- Pre-compile all forbidden pattern regexes at startup (cache `new RegExp()` objects)
- Use anchored patterns (`^pattern$` instead of `.*pattern.*`) to fail fast
- Consider early exit: if a violation is found, either fail fast or buffer first N violations and stop

**Warning signs:**
- `check-trace-redaction.cjs` takes >30s on a fresh checkout
- Tool is slow on first run but fast on subsequent runs (caching works)

### Pitfall 2: Confidence Tier Boundary Errors

**What goes wrong:** Off-by-one errors in threshold logic: `>=500 rounds` vs. `>499 rounds` or `>=14 days` vs. `>13 days`. Edge cases (exactly 500 rounds, exactly 14 days) are misclassified.

**Why it happens:** Thresholds are defined in REQUIREMENTS.md using `>=` and `<`, but implementation uses `>` and `<=` inconsistently.

**How to avoid:**
- Define thresholds in code as constants with clear names: `LOW_THRESHOLD_ROUNDS = 50`, `MEDIUM_THRESHOLD_ROUNDS = 500`, etc.
- Write unit tests for exact boundary values: test 49/50/51 rounds, 3/14/15 days
- Document the tier assignment in comments: "medium = rounds >= 500 AND days >= 14"

**Warning signs:**
- Test case `{ n_rounds: 500, window_days: 14 }` is classified `low` instead of `medium`
- Confidence tier changes unexpectedly when scoreboard crosses a threshold

### Pitfall 3: Schema Drift Detection False Negatives

**What goes wrong:** Schema.json is modified but the drift checker doesn't catch it because:
- Emitter file detection is too strict (only looks for one specific filename, missing multiple emitters)
- `git diff` is run against wrong base (assumes HEAD~1 is parent, but in rebase/merge scenarios it's not)
- Validator is excluded from modification check because it's in a subdirectory and glob pattern doesn't match

**Why it happens:** Hardcoded file patterns, fragile git assumptions, or incomplete emitter registry.

**How to avoid:**
- Maintain a registry of known emitter files (e.g., `KNOWN_EMITTERS = ['qgsd-stop.js', 'qgsd-prompt.js', ...]`)
- Use `git diff --merge-base` or assume CI always runs on a linear history (rebased before merge)
- Include directory prefixes in all file checks: `formal/trace/trace.schema.json` not just `trace.schema.json`
- Test with multiple git scenarios: normal commit, rebase, merge commit, force-push recovery

**Warning signs:**
- Schema change is silently allowed in CI but caught locally
- Emitter file is in a bin/ subdirectory and schema drift checker doesn't see it
- Test passes when run on one branch but fails on another (git history difference)

### Pitfall 4: Missing Trace Event Directory or Empty Files

**What goes wrong:** `check-trace-redaction.cjs` assumes `formal/trace/` directory exists and contains event files. If the directory doesn't exist or is empty:
- Tool crashes with "Cannot read directory" error
- Tool silently passes because no files to validate (false negative)

**Why it happens:** Trace event collection may not run on all CI jobs; directory creation may be deferred to a later phase.

**How to avoid:**
- Check for directory existence at startup; if missing, emit `result=pass` with metadata `{ reason: 'no-trace-directory', directory: '...' }`
- If directory exists but is empty, also emit `pass` with `{ reason: 'no-trace-events' }`
- Document the assumption: "If formal/trace/ does not exist or is empty, the check passes (no PII possible if no traces exist)"

**Warning signs:**
- Tool fails in a job that doesn't generate traces
- Different CI jobs have different exit codes for the same tool

### Pitfall 5: Confidence Tiers Not Updated When Scoreboard Changes

**What goes wrong:** A path transitions from `confidence: low` to `confidence: high` as the scoreboard accumulates rounds/days, but old cached results or static documentation still references the old tier.

**Why it happens:** Confidence is computed at runtime from the scoreboard, but test fixtures or documentation have hardcoded tiers. When scoreboard grows, code behavior changes but tests/docs become stale.

**How to avoid:**
- Compute confidence at validation time (not at plan time) from live scoreboard
- Test with multiple scoreboard states: `n_rounds: 10`, `n_rounds: 500`, `n_rounds: 10000` — verify tier assignment changes
- Never hardcode "this path is high confidence" in documentation; instead, compute it live

**Warning signs:**
- Test passes with a mock scoreboard but fails in CI where the real scoreboard has more rounds
- Confidence tier doesn't change even though CI has run 1000+ times

## Code Examples

Verified patterns from existing v0.19 phases:

### Example 1: Redaction Policy Parsing

```javascript
// Source: v0.19-02 policy.yaml parsing pattern
const fs = require('fs');
const path = require('path');

function parseRedactionPolicy(policyPath) {
  const content = fs.readFileSync(policyPath, 'utf8');
  const policy = { forbidden_keys: [], forbidden_patterns: [] };

  let section = null;
  for (const line of content.split('\n')) {
    const trimmed = line.trim();
    if (trimmed.startsWith('#') || trimmed.length === 0) continue;

    if (trimmed === 'forbidden_keys:') {
      section = 'keys';
      continue;
    }
    if (trimmed === 'forbidden_patterns:') {
      section = 'patterns';
      continue;
    }

    if (section === 'keys' && trimmed.startsWith('- ')) {
      policy.forbidden_keys.push(trimmed.substring(2).replace(/["']/g, ''));
    }

    if (section === 'patterns' && trimmed.startsWith('- name:')) {
      const nameMatch = trimmed.match(/name:\s*["']([^"']+)["']/);
      if (nameMatch) {
        const nextLine = content.split('\n')[content.split('\n').indexOf(line) + 1];
        const regexMatch = nextLine.match(/regex:\s*["']([^"']+)["']/);
        if (regexMatch) {
          policy.forbidden_patterns.push({
            name: nameMatch[1],
            regex: regexMatch[1]
          });
        }
      }
    }
  }

  return policy;
}
```

### Example 2: Confidence Tier Assignment

```javascript
// Source: Evidence confidence computation
const CONFIDENCE_THRESHOLDS = {
  low:    { min_rounds: 0,     min_days: 0,  label: 'low' },
  medium: { min_rounds: 500,   min_days: 14, label: 'medium' },
  high:   { min_rounds: 10000, min_days: 90, label: 'high' }

function computeConfidenceTier(n_rounds, window_days) {
  // Exact thresholds per EVID-02
  if (n_rounds >= 10000 && window_days >= 90) return 'high';
  if (n_rounds >= 500 && window_days >= 14) return 'medium';
  return 'low';

// Usage in validate-traces.cjs
function emitNeverObservedEvidence(path, n_rounds, window_days) {
  const confidence = computeConfidenceTier(n_rounds, window_days);
  return {
    path,
    status: 'never_observed',
    n_rounds,
    window_days,
    confidence,
    metadata: { computed_at_rounds: n_rounds, computed_at_days: window_days }
  };
}
```

### Example 3: Git-Based Schema Drift Detection

```javascript
// Source: Atomic commit verification pattern (safe subprocess execution)
const { execFileSync } = require('child_process');
const fs = require('fs');
const path = require('path');

const KNOWN_EMITTERS = [
  'bin/validate-traces.cjs',
  'hooks/qgsd-stop.js',
  'hooks/qgsd-prompt.js',
  // Add other trace-emitting files here
];

function checkSchemaDrift() {
  try {
    // Get files changed in current commit (HEAD) vs parent
    // Using execFileSync with array args prevents shell injection
    const changedFiles = execFileSync('git', ['diff', '--name-only', 'HEAD~1'], { encoding: 'utf8' })
      .split('\n')
      .filter(f => f.trim().length > 0);

    const hasSchemaChange = changedFiles.some(f =>
      f.includes('formal/trace/trace.schema.json')
    );

    if (!hasSchemaChange) {
      return { status: 'pass', reason: 'no-schema-change' };
    }

    // Schema changed — require co-update
    const validatorUpdated = changedFiles.some(f => f.includes('validate-traces.cjs'));
    const emitterUpdated = changedFiles.some(f =>
      KNOWN_EMITTERS.some(emitter => f.includes(emitter))
    );

    if (validatorUpdated && emitterUpdated) {
      return { status: 'pass', reason: 'schema-change-atomic', files: changedFiles.length };
    } else {
      return {
        status: 'fail',
        reason: 'schema-drift-detected',
        schema_changed: true,
        validator_updated: validatorUpdated,
        emitter_updated: emitterUpdated,
        changed_files: changedFiles
      };
    }
  } catch (err) {
    // git diff failed (e.g., not in a git repo or commit history unavailable)
    return { status: 'fail', reason: 'git-error', error: err.message };
  }
}

module.exports = { checkSchemaDrift, KNOWN_EMITTERS };
```

## Open Questions

1. **Trace event directory location:** Current research assumes `formal/trace/` is the directory holding trace events and schema. Should `.planning/conformance-events.jsonl` be moved there, or should `check-trace-redaction.cjs` scan `.planning/`?
   - What we know: v0.19-01 creates `formal/check-results.ndjson`; conformance log is at `.planning/conformance-events.jsonl`
   - What's unclear: Canonical location for trace event storage (formal vs. planning)
   - Recommendation: Assume `formal/trace/` is the formal artifacts directory; tool scans there. If conformance log is elsewhere, tool can accept a configurable path or symlink.

2. **Emitter file registry:** The `check-trace-schema-drift.cjs` tool needs to know which files "emit" trace events. Currently proposed: hardcoded `KNOWN_EMITTERS` array.
   - What we know: v0.19-03 added explicit fairness declarations in invariants.md; similar approach could document trace emitters
   - What's unclear: Should emitter list be dynamic (read from a config file) or static (hardcoded in tool)?
   - Recommendation: Start with hardcoded `KNOWN_EMITTERS` in the tool; if emitter set grows, migrate to `formal/trace/emitters.json` registry.

3. **Confidence tier revision cadence:** Thresholds are defined as `low`/`medium`/`high` with round/day counts. Should these be revisable without a full phase?
   - What we know: Thresholds are defined in REQUIREMENTS.md and will be embedded in `validate-traces.cjs`
   - What's unclear: Who updates thresholds when empirical evidence shows a tier boundary is wrong?
   - Recommendation: Define thresholds as constants at the top of `validate-traces.cjs` with clear comments. Update via quick tasks if needed; no phase required unless major revision.

4. **Redaction pattern maintenance:** Forbidden patterns are defined in `redaction.yaml`. How often should new patterns be added, and what's the review process?
   - What we know: Policy-as-code; new patterns don't require code changes
   - What's unclear: Who maintains the pattern list? Is there a security review gate?
   - Recommendation: Plan a separate "security rules update" process later. For v0.19-04, populate `redaction.yaml` with common patterns (AWS keys, GitHub PATs, email). Treat as baseline.

# Phase v0.19-10: CALIB-04 Policy Wire-Up - Research

**Researched:** 2026-02-28
**Domain:** PRISM runtime calibration configuration wiring (policy.yaml → run-prism.cjs constants)
**Confidence:** HIGH

## Summary

Phase v0.19-10 closes a medium-priority tech-debt item from the v0.19 audit: `policy.conservative_priors` values are read and exposed by `read-policy.cjs` but then **never consumed** by `run-prism.cjs`. Instead, `run-prism.cjs` uses hardcoded constants `PRISM_PRIOR_TP=0.85` and `PRISM_PRIOR_UNAVAIL=0.15` (lines 108–109). This means changing `formal/policy.yaml` has zero runtime effect — the priors are baked into the code.

The fix is straightforward: replace the two hardcoded constants with values read from the policy object at runtime. When `readPolicy()` returns `policy.conservative_priors.tp_rate` and `policy.conservative_priors.unavail`, use those instead of the hardcoded `PRISM_PRIOR_*` values.

**Primary recommendation:** Single task: replace hardcoded constants with policy-driven values; verify existing tests still pass and add one test confirming policy values are used.

### Pattern 1: Replace Hardcoded Constants with Policy Values

**What:** After `readPolicy()` returns the policy object, use `policy.conservative_priors.tp_rate` and `policy.conservative_priors.unavail` instead of hardcoded `PRISM_PRIOR_TP` and `PRISM_PRIOR_UNAVAIL`.

**Current code (lines 108–109 of run-prism.cjs):**
```javascript
const PRISM_PRIOR_TP     = 0.85;
const PRISM_PRIOR_UNAVAIL = 0.15;
```

**Current usage (lines 158–162):**
```javascript
if (liveTPRate === null) {
  liveTPRate    = PRISM_PRIOR_TP;
  liveUnavail   = PRISM_PRIOR_UNAVAIL;
  process.stderr.write(
    '[run-prism] No scoreboard found — using conservative priors: ' +
    'tp_rate=' + PRISM_PRIOR_TP + ' unavail=' + PRISM_PRIOR_UNAVAIL + '\n'
  );
}
```

**Refactored approach:**
1. Keep the two const lines (or remove them — no longer used)
2. After `policy = readPolicy(policyPath)` succeeds (line 118), policy object contains `policy.conservative_priors.tp_rate` and `policy.conservative_priors.unavail`
3. Replace `PRISM_PRIOR_TP` with `policy.conservative_priors.tp_rate`
4. Replace `PRISM_PRIOR_UNAVAIL` with `policy.conservative_priors.unavail`
5. All downstream uses automatically read from policy

**Example (lines 158–162 refactored):**
```javascript
if (liveTPRate === null) {
  liveTPRate    = policy.conservative_priors.tp_rate;
  liveUnavail   = policy.conservative_priors.unavail;
  process.stderr.write(
    '[run-prism] No scoreboard found — using conservative priors: ' +
    'tp_rate=' + policy.conservative_priors.tp_rate + ' unavail=' + policy.conservative_priors.unavail + '\n'
  );
}
```

**Key observation:** `policy` is already a module-level variable (line 116) after `readPolicy()` succeeds. It is guaranteed to exist before the code block that needs the constants (lines 158–162 come after policy read at line 118). No refactoring of the code flow is needed — just replace literal references.

### Pattern 2: Verification via Existing Test Suite

**What:** Existing test suite in `bin/run-prism.test.cjs` already exercises the prior-injection codepath (lines 36–59: "conservative priors used and logged when no scoreboard exists").

**Validation mechanism:** The test verifies that `Args:` log line contains `-const tp_rate=0.85` and `-const unavail=0.15`. After refactoring, if policy.yaml contains different values, the test should verify those new values are used.

**Proposed new test (simple check):**
```javascript
test('conservative priors from policy.yaml are injected when no scoreboard', () => {
  const { readPolicy } = require('./read-policy.cjs');
  const policy = readPolicy(path.join(__dirname, '..', 'formal', 'policy.yaml'));

  // Verify policy has expected structure
  assert.strictEqual(typeof policy.conservative_priors.tp_rate, 'number');
  assert.strictEqual(typeof policy.conservative_priors.unavail, 'number');

  // Verify existing test still captures the policy values
  const tmpDir = fs.mkdtempSync(path.join(os.tmpdir(), 'run-prism-test-'));
  try {
    const result = spawnSync(process.execPath, [RUN_PRISM], {
      encoding: 'utf8',
      cwd:      tmpDir,
      env:      { ...process.env, PRISM_BIN: 'prism' },
    });
    // The Args line should contain the values from policy.yaml, not hardcoded 0.85/0.15
    assert.match(result.stdout, new RegExp('-const tp_rate=' + policy.conservative_priors.tp_rate),
      'Args line should have -const tp_rate from policy.yaml');
    assert.match(result.stdout, new RegExp('-const unavail=' + policy.conservative_priors.unavail),
      'Args line should have -const unavail from policy.yaml');
  } finally {
    fs.rmSync(tmpDir, { recursive: true, force: true });
  }
});
```

### Anti-Patterns to Avoid

- **Keeping both hardcoded constants and policy values:** Confusing and error-prone. Remove or deprecate the hardcoded constants.
- **Hardcoding policy fallback values in run-prism.cjs:** If policy.yaml is missing or malformed, `readPolicy()` throws an error (line 119 catch block). This is correct — make policy required, don't create a fallback.
- **Treating policy values as "defaults" if policy.yaml changes:** Policy values ARE the priors; changing policy.yaml should change runtime behavior. No additional defaults or overrides should interfere.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Where to get prior values | Custom prior calculation or separate policy object | `policy.conservative_priors` from readPolicy() | Already extracted by read-policy.cjs; single source of truth |
| Validation of prior values | Custom min/max checks | Existing JSON Schema in `formal/policy.schema.json` | Already validates tp_rate and unavail are in [0, 1] |
| Testing prior values are used | Manual verification of constants | Existing integration test suite + simple new test | Tests already capture Args line output; just verify values match policy |

**Key insight:** This is a pure constant replacement, not a refactoring of logic. The complexity is zero — read the policy once (already done), use the values from the policy object (replace two references). No new patterns, no new state machines, no new data structures.

## Common Pitfalls

### Pitfall 1: Forgetting to Replace All References

**What goes wrong:** Code replaces `PRISM_PRIOR_TP` in one place but not another, leading to inconsistent values. For example, replacing line 158 but not line 162 (the stderr message). Tests pass because the constants are still defined, but the message doesn't match the actual behavior.

**Why it happens:** Grep-and-replace without understanding all usages. The constants `PRISM_PRIOR_TP` and `PRISM_PRIOR_UNAVAIL` appear 5 times (lines 108, 109, 158, 159, 162) — easy to miss one.

**How to avoid:** Grep for all occurrences of `PRISM_PRIOR_TP` and `PRISM_PRIOR_UNAVAIL` before starting. Verify each one should be replaced (it should, they're all fallback values). Run existing tests to confirm behavior unchanged. New test should verify all values are from policy.

**Warning signs:** Grep output shows 5+ matches; code replaces only 3; test runs but stderr messages don't match actual behavior.

### Pitfall 2: Using `policy.conservative_priors` Before Policy Is Read

**What goes wrong:** Code tries to reference `policy.conservative_priors.tp_rate` before the `readPolicy()` call at line 118. If that code path is taken (early return or error before policy read), code crashes with "cannot read property of undefined".

**Why it happens:** Refactoring assumes `policy` is always defined. But if there's an error path before the readPolicy() call, or a code path that doesn't reach it, `policy` is still undefined.

**How to avoid:** Verify that all uses of `policy.conservative_priors` come AFTER line 118 (`policy = readPolicy(policyPath)`). Check if there are any early returns or error paths that skip the policy read.

**Warning signs:** Code paths that reference `policy.conservative_priors` but can execute before the readPolicy() call; runtime errors about "policy is not defined" or "cannot read conservative_priors of undefined".

### Pitfall 3: Constants Removed But Referenced Elsewhere

**What goes wrong:** Code removes the `const PRISM_PRIOR_TP = ...` lines entirely (cleanup), but other files or tests still import/reference them. Breakage is silent until those code paths are executed.

**Why it happens:** The constants are module-level in run-prism.cjs; they're not exported, so external references are unlikely. But if tests or other utilities import them, the module will fail to load.

**How to avoid:** Check if the constants are exported: `module.exports = { PRISM_PRIOR_TP, ... }`? (They're not in v0.19-02.) If not exported, it's safe to remove. If they are, migrate all external references first.

**Warning signs:** `npm test` fails with "Cannot find module"; grep finds references to `PRISM_PRIOR_TP` outside run-prism.cjs.

### Pitfall 4: Test Still Uses Hardcoded Values Instead of Policy

**What goes wrong:** Existing test at line 46 checks `assert.match(result.stdout, /Args:.*-const tp_rate=0\.85/, ...)`. After refactoring, if policy.yaml has `tp_rate: 0.95`, the test still expects 0.85 and fails.

**Why it happens:** Test was written for the hardcoded values. When the code switches to policy-driven values, the test expectations become stale.

**How to avoid:** Update the test to read the policy and check for the actual policy values, not hardcoded numbers. Use `policy.conservative_priors.tp_rate` in the test assertion.

**Warning signs:** Test fails after refactoring with message "expected 0.85 but got 0.95" (or vice versa); test was written assuming constants, not policy.

### Pitfall 5: Policy File Not Found or Malformed

**What goes wrong:** If `readPolicy()` throws (missing file or invalid YAML), the catch block at line 119 exits the process. If tests don't verify this error path still works, silent breakage can occur in CI.

**Why it happens:** Refactoring assumes policy.yaml always exists and is valid. But fresh clones or corrupted files could cause issues.

**How to avoid:** Existing code already handles this (lines 119–128: catch block exits with error). No change needed, but verify the test suite includes a scenario where policy.yaml is missing or malformed.

**Warning signs:** Fresh clone of repo fails on `npm test` with unclear error about policy.yaml; integration test doesn't cover the error path.

```javascript
test('policy.yaml conservative_priors values are used as PRISM constants', () => {

      encoding: 'utf8',
      cwd:      tmpDir,
      env:      { ...process.env, PRISM_BIN: 'prism' },

    const expectedTPStr = expectedTP.toString();
    const expectedUnavailStr = expectedUnavail.toString();

```

### Human Verification Required

#### 1. Fairness Rationale Quality Review

**Test:** Read each `formal/spec/<surface>/invariants.md` entry. For each WF_vars declaration, verify the rationale explains (a) which QGSD component creates the enabling condition, and (b) why that component guarantees the action eventually fires in practice.
**Expected:** Each rationale references a specific QGSD execution context (e.g., sequential dispatcher loop, bounded round cap, HTTP timeout) — not generic language like "the system will eventually proceed."
**Why human:** Requires judgment of operational soundness in QGSD's deployment context; semantic quality of prose cannot be automated via grep.

#### 2. Spec-Invariants Drift Check

**Test:** For each surface, open both the `.tla` file and the corresponding `invariants.md`. Confirm the action names in the `Fairness assumption` line of `invariants.md` exactly match the `WF_vars(...)` calls in the `Spec` definition of the `.tla` file.
**Expected:** Action names are identical — e.g., `invariants.md` says "WF_vars on 4 actions: Decide, StartQuorum, AnyCollectVotes, AnyDeliberate" and `QGSDQuorum.tla` Spec shows `WF_vars(Decide) /\ WF_vars(StartQuorum) /\ WF_vars(AnyCollectVotes) /\ WF_vars(AnyDeliberate)`.
**Why human:** Semantic matching of action name lists against TLA+ operator definitions requires reading formal spec; cannot automate without a TLA+ parser.

# Phase v0.19-03: Liveness Fairness - Research

**Researched:** 2026-02-27
**Domain:** TLA+ liveness properties with explicit fairness declarations and operational documentation
**Confidence:** HIGH

## Summary

Phase v0.19-03 makes liveness properties operationally meaningful by requiring explicit fairness assumption documentation. Currently, all liveness properties in TLA+ specs (QGSDQuorum, QGSDDeliberation, QGSDPreFilter, QGSDRecruiting) already declare weak fairness (`WF_vars`) or strong fairness (`SF_vars`) in their `Spec` definition. However, there is no human-readable record of what fairness assumption is required for each liveness property to be sound, or why that assumption holds in the deployment context.

The phase creates `formal/spec/<surface>/invariants.md` files for each TLA+ surface (quorum, deliberation, prefilter, recruiting) that document:
1. For each liveness property: which fairness assumption is applied (WF_vars, SF_vars, or SF_actions)
2. A one-sentence realism rationale explaining why the assumption holds in QGSD's operational context

Additionally, `run-tlc.cjs` is updated to detect when a liveness `.cfg` file is present but the corresponding `invariants.md` is missing the fairness declaration entry. In this case, the checker emits `result=inconclusive` to `check-results.ndjson` instead of `result=pass`, signaling that the liveness property is unverified (the fairness assumption is untested).

**Primary recommendation:** Implement a two-task approach: (1) Author `formal/spec/<surface>/invariants.md` files with fairness declarations for all existing liveness specs (quorum, deliberation, prefilter, recruiting); (2) Update `run-tlc.cjs` to parse `invariants.md` and emit `result=inconclusive` when a property's fairness declaration is missing.

### Pattern 1: Invariants.md Structure

**What:** A human-readable markdown document listing all liveness properties and their fairness assumptions.

**When to use:** Create one per TLA+ surface that has liveness properties. Each surface must list its fairness declarations.

**Example structure:**

```markdown
# Liveness Fairness Declarations

## EventualConsensus
**Property:** `EventualConsensus == <>(phase = "DECIDED")`
**Fairness assumption:** WF_vars (weak fairness)
**Realism rationale:** Quorum workflow eventually either reaches consensus or exhausts deliberation rounds; both terminal actions (Decide and StartQuorum) must not be permanently disabled. In a deployed system with sequential message processing, this is guaranteed by the message dispatcher loop.

## ProtocolTerminates
**Property:** `ProtocolTerminates == <>(voteState = "CONSENSUS" \/ voteState = "ESCALATED")`
**Fairness assumption:** WF_vars (weak fairness)
**Realism rationale:** Deliberation protocol must eventually terminate. Actions leading to termination (Escalate, ReceiveApprove) are executed sequentially by the model voting dispatcher; WF ensures that a continuously enabled termination action will eventually fire.
```

**Why this format:**
- Markdown is human-readable and git-friendly
- Property name appears in the cfg file and in the .tla file — exact match makes it grep-able
- Fairness assumption is explicit and checkable
- Rationale justifies the assumption in QGSD's operational context

### Pattern 2: Run-TLC Detection Logic

**What:** Update `bin/run-tlc.cjs` to:
1. Detect if a liveness config (e.g., `MCliveness.cfg`, `MCdeliberation.cfg`) is being run
2. Parse the corresponding `invariants.md` file to extract fairness declarations
3. Cross-check: for each PROPERTY line in the .cfg, verify that a matching entry exists in `invariants.md`
4. If any liveness property lacks a fairness declaration in `invariants.md`, emit `result=inconclusive` instead of `result=pass`

**When to use:** Integrated into the main TLC execution path in `run-tlc.cjs`.

**Example detection:**

```javascript
// After TLC completes successfully (status === 0):
// 1. Identify if this is a liveness config
const isLivenessConfig = configName.includes('liveness') ||
  fs.readFileSync(cfgPath, 'utf8').includes('PROPERTY');

if (isLivenessConfig) {
  // 2. Determine surface and invariants.md path
  const surface = determineSurface(configName); // 'quorum' | 'deliberation' | 'prefilter' | 'recruiting'
  const invariantsPath = path.join(__dirname, '..', 'formal', 'spec', surface, 'invariants.md');

  // 3. Parse .cfg to extract PROPERTY names
  const cfgText = fs.readFileSync(cfgPath, 'utf8');
  const properties = cfgText.match(/^PROPERTY\s+(\w+)/gm) || [];

  // 4. Check invariants.md for matching entries
  const invariantsText = fs.readFileSync(invariantsPath, 'utf8');
  for (const prop of properties) {
    if (!invariantsText.includes(`## ${propName}`)) {
      // Missing fairness declaration
      writeCheckResult({
        tool: 'run-tlc',
        formalism: 'tla',
        result: 'inconclusive',
        metadata: {
          config: configName,
          reason: `Fairness declaration missing for property: ${propName}`
        }
      });
      process.exit(0); // inconclusive is not a failure; don't exit(1)
    }
  }
}
```

### Anti-Patterns to Avoid

- **Don't hardcode fairness assumptions in code:** Always document in `invariants.md`, not as comments. Markdown is queryable and reviewable.
- **Don't skip liveness properties:** If a `.cfg` has a PROPERTY line but no fairness declaration exists, emit inconclusive. Don't default to pass.
- **Don't modify Spec definitions during this phase:** Fairness is already correct in the .tla files. This phase documents, not changes.
- **Don't create surface-specific directories without invariants.md:** Every new surface must have an invariants.md from day one.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Custom fairness validation | Regex matching for fairness types | Structured `invariants.md` with clear property headers | Regex brittle; markdown is parseable and git-friendly; avoids ad-hoc format drift |
| Per-property documentation in comments | Embedded comments in .tla files | Separate `invariants.md` per surface | Comments get out of sync with code; separate doc is reviewable and versionable |
| Dynamic fairness assumption inference | Regex parsing of Spec lines | Explicit declarations in invariants.md | Inference brittle; explicit declarations are verifiable and reviewable |

**Key insight:** Fairness declarations are operational knowledge, not code logic. Markdown documentation is the right medium — it's human-verifiable, machine-parseable, and stays synchronized with formal specs via PR reviews.

## Common Pitfalls

### Pitfall 1: Mismatched Property Names

**What goes wrong:** The property name in the `.cfg` file doesn't exactly match the header in `invariants.md`, causing detection to fail.

**Why it happens:** Manual transcription between files; case sensitivity; use of underscores vs camelCase.

**How to avoid:** Use a script to extract property names from `.cfg` files and validate they exist in `invariants.md`. Consider a test that runs `run-tlc.cjs` in dry-run mode and checks for all required declarations.

**Warning signs:** Detection logic reports "property not found" despite the property existing in the markdown.

### Pitfall 2: Insufficient Fairness Rationale

**What goes wrong:** The realism rationale is vague (e.g., "actions eventually fire") without explaining why in QGSD's context.

**Why it happens:** Treating rationale as boilerplate rather than operational evidence.

**How to avoid:** Rationale must reference QGSD's specific execution model: "message dispatcher loop", "sequential slot-trying", "quorum worker dispatch", etc. Weak fairness (WF) is justified by sequential execution; strong fairness (SF) requires additional justification about contention or fairness scheduling.

**Warning signs:** A rationale that could apply to any TLA+ model (not specific to QGSD).

### Pitfall 3: Liveness Config Without Fairness Declaration

**What goes wrong:** A new liveness `.cfg` is added (e.g., for a new surface) but `invariants.md` is not created, causing CI to emit inconclusive indefinitely.

**Why it happens:** Developer focuses on the .tla spec and .cfg and forgets the documentation step.

**How to avoid:** Make `invariants.md` creation a pre-requisite for any liveness `.cfg`. Update the TLA+ author checklist: "Before adding a PROPERTY line to a .cfg, create or update `formal/spec/<surface>/invariants.md`".

**Warning signs:** `run-tlc.cjs` output shows `result=inconclusive` with reason "fairness declaration missing".

### Pitfall 4: Conflicting Fairness Assumptions Between Versions

**What goes wrong:** The spec uses `WF_vars(Action)` but `invariants.md` documents `SF_vars(Action)`, or vice versa.

**Why it happens:** The spec changed but documentation wasn't updated.

**How to avoid:** Add a unit test that parses both the `.tla` file and `invariants.md` and verifies the fairness type matches (not the exact action, but the fairness flavor: WF vs SF).

**Warning signs:** A test comparing spec definitions to documentation flags a mismatch.

## Code Examples

Verified patterns from formal verification literature and QGSD's existing specs:

### Example 1: Weak Fairness (WF_vars) Declaration

```markdown
## EventualConsensus
**Property:** `EventualConsensus == <>(phase = "DECIDED")`
**Fairness assumption:** WF_vars (weak fairness)
**Realism rationale:** The quorum workflow's main loop in run-quorum.cjs processes vote responses sequentially. If a vote is received (or the deliberation timeout triggers), the corresponding action in the spec becomes enabled. By weak fairness, once enabled, the action must eventually fire. This models the fact that a message in the queue will not be dropped indefinitely.

**Source:** formal/tla/QGSDQuorum.tla, line 148-150
```

### Example 2: Strong Fairness (SF_vars) Declaration

```markdown
## PreFilterTerminates
**Property:** `PreFilterTerminates == <>(filterPhase = "AUTO_RESOLVED" \/ filterPhase = "USER_NEEDED")`
**Fairness assumption:** SF_vars (strong fairness on composite action `EvaluateAndResolve`)
**Realism rationale:** The pre-filter loop in qgsd-prompt.js runs the R4 voting round in a loop until consensus or escalation. If an outcome (agreement or disagreement) is infinitely often enabled, it must eventually fire. This models the fact that the voting dispatcher will not indefinitely allow a consensus outcome to be skipped.

**Source:** formal/tla/QGSDPreFilter.tla, line 105-107
```

### Example 3: Run-TLC Detection Integration

```javascript
// Simplified pseudocode for run-tlc.cjs liveness detection
function detectLivenessProperties(configName, cfgPath) {
  const cfgContent = fs.readFileSync(cfgPath, 'utf8');
  const propertyLines = cfgContent.match(/^PROPERTY\s+(\w+)/gm) || [];

  if (propertyLines.length === 0) {
    return []; // No liveness properties declared
  }

  // Extract surface name from config (e.g., MCliveness → quorum, MCdeliberation → deliberation)
  const surfaceMap = {
    'MCliveness': 'quorum',
    'MCdeliberation': 'deliberation',
    'MCprefilter': 'prefilter',
    'MCrecruiting-liveness': 'recruiting'
  };

  const surface = surfaceMap[configName] || null;
  if (!surface) {
    return propertyLines; // Unknown surface — can't validate
  }

  const invariantsPath = path.join(__dirname, '..', 'formal', 'spec', surface, 'invariants.md');
  if (!fs.existsSync(invariantsPath)) {
    // Missing invariants.md entirely
    return propertyLines; // All properties lack declarations
  }

  const invariantsContent = fs.readFileSync(invariantsPath, 'utf8');
  const missingDeclarations = [];

  for (const line of propertyLines) {
    const match = line.match(/^PROPERTY\s+(\w+)/);
    if (match) {
      const propName = match[1];
      if (!invariantsContent.includes(`## ${propName}`)) {
        missingDeclarations.push(propName);
      }
    }
  }

  return missingDeclarations;
}

// In the main run-tlc.cjs flow:
if (passed) {
  const missingDeclarations = detectLivenessProperties(configName, cfgPath);
  if (missingDeclarations.length > 0) {
    try {
      writeCheckResult({
        tool: 'run-tlc',
        formalism: 'tla',
        result: 'inconclusive',
        metadata: {
          config: configName,
          reason: `Fairness declaration missing for: ${missingDeclarations.join(', ')}`
        }
      });
    } catch (e) {
      process.stderr.write('[run-tlc] Warning: failed to write inconclusive result: ' + e.message + '\n');
    }
    process.exit(0); // inconclusive is not a failure
  } else {
    // All properties have declarations — emit pass
    try {
      writeCheckResult({
        tool: 'run-tlc',
        formalism: 'tla',
        result: 'pass',
        metadata: { config: configName }
      });
    } catch (e) {
      process.stderr.write('[run-tlc] Warning: failed to write check result: ' + e.message + '\n');
    }
    process.exit(0);
  }
}
```

## Open Questions

1. **Should all spec actions be documented, or only those in PROPERTY lines?**
   - **What we know:** LIVE-01 requires documenting "each liveness property" — specifically properties declared via PROPERTY in .cfg files
   - **What's unclear:** Specs like QGSDAccountManager have fairness but no explicit PROPERTY. Should `invariants.md` list all fairness assumptions even for non-property specs?
   - **Recommendation:** Start with properties (PROPERTY lines in .cfg). If a spec has fairness but no property, create a minimal `invariants.md` that documents the fairness anyway. Consistency wins over ambiguity.

2. **What if a spec has multiple liveness properties but only one is checked?**
   - **What we know:** A single .tla file can define multiple liveness properties; a .cfg file selects which ones to check
   - **What's unclear:** Should `invariants.md` document all properties in the spec, or only those in the active .cfg?
   - **Recommendation:** Document all properties defined in the spec, not just the checked ones. Future phases may add new .cfg files that check other properties. Having documentation in place avoids work duplication.

3. **How should fairness be documented for composite actions like `AnyCollectVotes`?**
   - **What we know:** Some specs use `\E ... : Action` syntax to create composite actions for fairness
   - **What's unclear:** How to document this without duplicating the TLA+ syntax?
   - **Recommendation:** Document the composite action name (e.g., "WF_vars(AnyCollectVotes)") and explain in the rationale that it covers the existentially quantified set of actions.

4. **Should the fairness type be validated against the spec definition?**
   - **What we know:** `invariants.md` will document WF_vars/SF_vars; the .tla file has the actual definition
   - **What's unclear:** Is there a need for a test that cross-validates the documented fairness type against the spec?
   - **Recommendation:** Add a unit test in `bin/run-tlc.test.cjs` that parses both files and ensures the documented fairness type matches the spec definition (at least for simple cases like `WF_vars(Action)` parsing).

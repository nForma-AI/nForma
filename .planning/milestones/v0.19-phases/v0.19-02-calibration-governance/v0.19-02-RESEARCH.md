# Phase v0.19-02: Calibration Governance - Research

**Researched:** 2026-02-27
**Domain:** PRISM cold-start policy governance and evidence-driven calibration checks
**Confidence:** HIGH

## Summary

Phase v0.19-02 prevents day-one calibration thrash by implementing a cold-start mode in PRISM. The problem: on a fresh installation (with zero quorum history), PRISM has no empirical evidence of check pass rates, so it falls back to conservative priors (tp_rate=0.85, unavail=0.15). These priors are pessimistic by design and cause false alarms in the first hours of operation. Solution: implement a `formal/policy.yaml` file that defines cold-start thresholds (`min_ci_runs`, `min_quorum_rounds`, `min_days`), below which all PRISM calibration checks emit `result=warn` instead of `result=fail`. Once the system accumulates sufficient evidence (crosses all three thresholds), it switches to steady-state mode and enforces the configured calibration mode (`warn` or `fail`). Additionally, evidence-driven checks must include `observation_window` metadata in their NDJSON entries, documenting the time window and event count the check observed.

This phase builds directly on v0.19-01 (unified NDJSON output format). The NDJSON schema already supports `warn` and `inconclusive` results; v0.19-02 adds the policy logic to `run-prism.cjs` and extends the metadata structure for observation window tracking.

**Primary recommendation:** Implement two tasks:
1. **CALIB-01 + CALIB-04:** Author `formal/policy.yaml` with JSON Schema validation, cold-start thresholds, and conservative-priors fallthrough documentation.
2. **CALIB-02 + CALIB-03:** Update `run-prism.cjs` to read policy.yaml, compute cold-start state, emit `result=warn` during cold-start, and add `observation_window` metadata to all check results.

### Pattern 1: Policy YAML Structure and Schema Validation

**What:** A configuration file defining when to leave cold-start mode and transition to steady-state calibration enforcement.

**policy.yaml structure:**
```yaml
# formal/policy.yaml
# Calibration governance policy for PRISM evidence-driven checks

cold_start:
  # Thresholds — all three must be met to leave cold-start mode
  min_ci_runs:       5          # Number of CI runs (check-results.ndjson entries from run-formal-verify)
  min_quorum_rounds: 10         # Number of quorum rounds in scoreboard (rounds[].round count)
  min_days:          1          # Minimum calendar days since first run

steady_state:
  # Calibration mode after cold-start thresholds are met
  mode: "warn"                  # "warn" (never fail) or "fail" (use empirical rates); default "warn"

conservative_priors:
  # Documentation: fallthrough strategy
  # When no scoreboard or below cold-start thresholds:
  #   - tp_rate: 0.85 (85% of external models vote TP; 15% unavailable/FP/FN)
  #   - unavail:  0.15 (15% of votes are UNAVAIL, treated as absent)
  #
  # Rationale: Fresh installs have no evidence, so assume worst-case:
  #   external models are less reliable than in steady-state.
  #   These are intentionally conservative (lower tp_rate than steady-state empirical ~0.95)
  #   to avoid false-passing checks before evidence accumulates.
  tp_rate: 0.85
  unavail: 0.15

  # Fallthrough to empirical rates happens when:
  #   cold_start.min_ci_runs   <= ci_run_count   AND
  #   cold_start.min_quorum_rounds <= quorum_round_count AND
  #   cold_start.min_days      <= days_since_first_run
  #
  # Once all three are true, run-prism.cjs switches from conservative_priors to
  # empirical tp_rate/unavail computed from quorum-scoreboard.json.
```

**policy.schema.json (similar to check-result.schema.json):**
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["cold_start", "steady_state"],
  "properties": {
    "cold_start": {
      "type": "object",
      "required": ["min_ci_runs", "min_quorum_rounds", "min_days"],
      "properties": {
        "min_ci_runs":       { "type": "integer", "minimum": 1 },
        "min_quorum_rounds": { "type": "integer", "minimum": 1 },
        "min_days":          { "type": "number", "minimum": 0 }
      }
    },
    "steady_state": {
      "type": "object",
      "required": ["mode"],
      "properties": {
        "mode": { "type": "string", "enum": ["warn", "fail"] }
      }
    },
    "conservative_priors": {
      "type": "object",
      "properties": {
        "tp_rate": { "type": "number", "minimum": 0, "maximum": 1 },
        "unavail":  { "type": "number", "minimum": 0, "maximum": 1 }
      }
    }
  }
}
```

### Pattern 2: Cold-Start State Detection in run-prism.cjs

**What:** Logic to determine if the system is in cold-start mode and conditionally emit `result=warn`.

**Algorithm (pseudocode):**
```javascript
// 1. Read policy.yaml
const policy = readAndValidatePolicy('formal/policy.yaml');

// 2. Count evidence from scoreboard (existing code)
const scoreboardPath = path.join(process.cwd(), '.planning', 'quorum-scoreboard.json');
let ciRunCount = 0;
let quorumRoundCount = 0;
let firstRunTimestamp = null;

if (fs.existsSync(scoreboardPath)) {
  const sb = JSON.parse(fs.readFileSync(scoreboardPath, 'utf8'));
  quorumRoundCount = (sb.rounds || []).length;
  firstRunTimestamp = extractFirstRoundDate(sb.rounds[0]);  // ISO 8601 or custom format
  // ci_run_count = number of entries in check-results.ndjson (rough proxy: quorum_round_count * avg_runs_per_round)
}
// Simplification: count entries in check-results.ndjson for exact CI run count
if (fs.existsSync(checkResultsPath)) {
  const lines = fs.readFileSync(checkResultsPath, 'utf8').trim().split('\n').filter(l => l);
  ciRunCount = lines.length;
}

// 3. Determine if cold-start
const daysSinceFirst = firstRunTimestamp ? (Date.now() - firstRunTimestamp) / (1000 * 60 * 60 * 24) : 0;
const inColdStart =
  ciRunCount < policy.cold_start.min_ci_runs ||
  quorumRoundCount < policy.cold_start.min_quorum_rounds ||
  daysSinceFirst < policy.cold_start.min_days;

// 4. Before calling writeCheckResult, conditionally override result
let finalResult = passed ? 'pass' : 'fail';
if (!passed && inColdStart) {
  finalResult = 'warn';  // Never 'fail' during cold-start, only 'warn'
}

// 5. Add observation_window metadata
const metadata = {
  observation_window: {
    window_start: firstRunTimestamp || new Date().toISOString(),
    window_end:   new Date().toISOString(),
    n_rounds:     quorumRoundCount,
    n_events:     ciRunCount
  }
};

writeCheckResult({
  tool: 'run-prism',
  formalism: 'prism',
  result: finalResult,
  metadata
});
```

**Key points:**
- `inColdStart` is a boolean: true if ANY of the three thresholds is not met
- `observation_window` is added to ALL check results (not just failures), for transparency
- The priors (tp_rate=0.85, unavail=0.15) remain hardcoded; only the `result` override changes

### Pattern 3: YAML Reading (Lightweight, No External Deps)

**What:** Safe, synchronous YAML parsing without adding a dependency.

**Options:**

**Option A: Manual Key-Value Parsing (Simple, Minimal Risk)**
```javascript
function readPolicy(path) {
  const yaml = fs.readFileSync(path, 'utf8');

  const extract = (key) => {
    const match = yaml.match(new RegExp(`^${key}:\\s*(\\S+)`, 'm'));
    return match ? match[1] : null;
  };

  return {
    cold_start: {
      min_ci_runs: parseInt(extract('min_ci_runs'), 10),
      min_quorum_rounds: parseInt(extract('min_quorum_rounds'), 10),
      min_days: parseFloat(extract('min_days'))
    },
    steady_state: {
      mode: extract('mode') || 'warn'
    }
  };
}
```

**Option B: Convert to JSON First (More Robust)**
Many YAML files are valid JSON. If policy.yaml uses simple structure (no anchors, no special YAML syntax), it can be parsed as JSON after minor preprocessing.

**Recommendation:** Option A (manual parsing) for this phase. It's simple, clear, and robust against schema changes. Use helper function `validatePolicy(policy)` post-parse to check required fields exist.

### Anti-Patterns to Avoid

- **Hardcoding thresholds in run-prism.cjs:** Policy thresholds MUST come from `formal/policy.yaml`, never from code constants. This allows ops teams to tune without code changes.
- **Silently defaulting on policy read failure:** If policy.yaml is missing or invalid, emit stderr warning and fail the check. Do not default to "in steady-state" or "use priors anyway" — make policy drift explicit.
- **Confusing "cold-start mode" with "no scoreboard":** Cold-start is based on evidence thresholds, not just file existence. A system with a scoreboard but <5 CI runs is still in cold-start.
- **Adding observation_window only on failures:** Include it on ALL results for consistency. It documents the evidence base for every check, not just failures.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing library | Custom regex-based state machine | Manual key-value extraction via `fs.readFileSync + regex` | Regex is simple, auditable, no deps; complex parsers are overkill |
| Policy validation | Manual if-else chains | JSON Schema validator (optional: `ajv` or post-hoc validation function) | Schema declarative, reusable; if-else is error-prone |
| Evidence aggregation from scoreboard | Custom counting loops | Existing pattern: parse `quorum-scoreboard.json`, iterate rounds, count slots | Already in run-prism.cjs; reuse code |
| Timestamp calculations | Manual date arithmetic | ISO 8601 strings + `Date.parse()` + `Date.now()` | Built-in, reliable, no deps |
| Result override logic | Multiple conditional branches | Single boolean `inColdStart = ...` flag; one if-statement overriding result | Keeps logic clear and testable |

**Key insight:** The critical complexity here is NOT parsing or validation — it's the cold-start state machine itself. The policy.yaml is deliberately simple (no nested structures, no advanced YAML features) so manual parsing is safe and transparent. The actual work is updating `run-prism.cjs` to compute and apply the cold-start logic correctly.

## Common Pitfalls

### Pitfall 1: Off-by-One in Threshold Comparison

**What goes wrong:** Policy says `min_ci_runs: 5`, but code checks `ciRunCount <= 5` (leaving cold-start one run too late) or `ciRunCount < 5` (leaving too early). Test suite passes because the boundary is never exercised.

**Why it happens:** Threshold comparisons are easy to get backwards. Off-by-one errors are a classic bug.

**How to avoid:** Write explicit test cases for the boundary. E.g., `ciRunCount=4` must be IN cold-start; `ciRunCount=5` must be OUT.

**Warning signs:** Unit tests that only check `ciRunCount=100` or `ciRunCount=0`; no tests at exactly `min_ci_runs`.

### Pitfall 2: Using AND Instead of OR in Cold-Start Condition

**What goes wrong:** Code checks `ciRunCount >= min_ci_runs AND quorumRounds >= min AND days >= min`. If any ONE threshold is not met, cold-start should be true. The AND logic is correct WITHIN the parentheses but the overall condition is backwards. Should be `NOT (all thresholds met)` = `NOT AND` = `OR` of the negations.

**Why it happens:** Logical negation is easy to mess up. Writing the condition directly as "is it cold-start?" is less error-prone than "is it NOT cold-start?" and then negating.

**How to avoid:** Write it as: `const allThresholdsMet = ciRunCount >= min_ci_runs && ...; const inColdStart = !allThresholdsMet;` so the negation is explicit.

**Warning signs:** Tests that assume cold-start ends when only one threshold is met; failure to test all combinations of threshold states (e.g., high CI runs + low days should still be cold-start).

### Pitfall 3: Floating-Point Days Calculation Loses Precision

**What goes wrong:** `daysSinceFirst = (Date.now() - firstRunTime) / (1000 * 60 * 60 * 24)` returns a float. Comparing `1.5 < 1` is false, but the check means "is it less than 1 calendar day?" — boundary cases (e.g., exactly 24 hours later) behave unexpectedly.

**Why it happens:** Mixing floating-point days with integer thresholds. The comparison logic assumes exact equality is impossible, but in tests with fixed timestamps, it can occur.

**How to avoid:** Either convert both to the same unit (e.g., milliseconds) before comparison, or explicitly round/floor the days value for comparison: `Math.floor(daysSinceFirst) >= policy.cold_start.min_days`.

**Warning signs:** Tests that set `min_days: 1` and verify the boundary at exactly 24 hours later fail or pass unpredictably.

### Pitfall 4: Metadata Object Mutation Causes NDJSON Corruption

**What goes wrong:** Code builds `metadata = { observation_window: {...} }` and passes it to multiple `writeCheckResult()` calls, then later mutates it. Each entry in NDJSON contains a reference, not a copy. Later calls see the mutated values.

**Why it happens:** JavaScript object references; if the same object is reused across multiple writes, mutations affect all previous entries (if the caller later mutates it).

**How to avoid:** Always create a fresh metadata object per call, or deep-copy before passing. Pass inline: `writeCheckResult({ ..., metadata: { observation_window: {...} } })` — fresh object per call.

**Warning signs:** NDJSON entries show different observation_window values than expected for earlier checks; metadata timestamps don't match the actual check run order.

### Pitfall 5: Policy File Not Found Silently Uses Defaults

**What goes wrong:** `run-prism.cjs` checks `if (fs.existsSync(policy.yaml))` and only reads if true. If false, code silently assumes default thresholds (or infinite thresholds, making it never cold-start). Fresh install has no policy.yaml, so the user gets unexpected behavior.

**Why it happens:** Defensive programming assumes file might not exist. But policy.yaml is a required artifact; its absence should be an error, not a silent fallback.

**How to avoid:** Make policy.yaml required. Throw an error if not found. Provide a default template in the repo (e.g., committed to git) so it always exists.

**Warning signs:** Fresh install produces unexpected PRISM results; user forgets to create policy.yaml and wonders why cold-start logic isn't working; no clear error message.

// Source: This research; based on Node.js built-in fs
const fs = require('fs');
const path = require('path');

function readPolicy(policyPath) {
  if (!fs.existsSync(policyPath)) {
    throw new Error(`[run-prism] Policy file not found: ${policyPath}`);
  }

  const yaml = fs.readFileSync(policyPath, 'utf8');

  // Manual key extraction via regex
  const extract = (key) => {
    const match = yaml.match(new RegExp(`^\\s*${key}:\\s*(\\S+)`, 'm'));
    if (!match) throw new Error(`[run-prism] Policy missing key: ${key}`);
    return match[1];
  };

  return {
    cold_start: {
      min_ci_runs:       parseInt(extract('min_ci_runs'), 10),
      min_quorum_rounds: parseInt(extract('min_quorum_rounds'), 10),
      min_days:          parseFloat(extract('min_days'))
    },
    steady_state: {
      mode: extract('mode')
    }
  };
}

// Usage
const policy = readPolicy(path.join(__dirname, '..', 'formal', 'policy.yaml'));

### Cold-Start State Detection

```javascript
// Source: This research; builds on existing run-prism.cjs patterns

function computeColdStartState(policy, scoreboardPath, checkResultsPath) {
  let ciRunCount = 0;
  let quorumRoundCount = 0;
  let firstRunTimestamp = null;

  // Count CI runs from check-results.ndjson
  if (fs.existsSync(checkResultsPath)) {
    const lines = fs.readFileSync(checkResultsPath, 'utf8')
      .trim()
      .split('\n')
      .filter(l => l.length > 0);
    ciRunCount = lines.length;
  }

  // Read quorum rounds from scoreboard
  if (fs.existsSync(scoreboardPath)) {
    try {
      const sb = JSON.parse(fs.readFileSync(scoreboardPath, 'utf8'));
      const rounds = sb.rounds || [];
      quorumRoundCount = rounds.length;

      // Extract first run timestamp from oldest round
      if (rounds.length > 0) {
        const firstRound = rounds[0];
        // Assume rounds have a 'date' field in ISO 8601 or "MM-DD" format
        // For this implementation, use a timestamp if available, else compute from date string
        firstRunTimestamp = firstRound.timestamp
          ? new Date(firstRound.timestamp).getTime()
          : new Date(firstRound.date).getTime();  // Assumes "MM-DD" can be parsed
      }
    } catch (e) {
      process.stderr.write(`[run-prism] Warning: failed to parse scoreboard: ${e.message}\n`);
    }
  }

  // Compute days since first run
  const daysSinceFirst = firstRunTimestamp
    ? (Date.now() - firstRunTimestamp) / (1000 * 60 * 60 * 24)
    : 0;

  // Determine cold-start state: true if ANY threshold not met
  const inColdStart =
    ciRunCount < policy.cold_start.min_ci_runs ||
    quorumRoundCount < policy.cold_start.min_quorum_rounds ||
    daysSinceFirst < policy.cold_start.min_days;

  return {
    inColdStart,
    ciRunCount,
    quorumRoundCount,
    daysSinceFirst,
    firstRunTimestamp
  };
}
```

### Result Override and Metadata Addition

```javascript
// Source: This research; extends v0.19-01 writeCheckResult pattern

const { writeCheckResult } = require('./write-check-result.cjs');

// After PRISM runs and determines pass/fail:
const passed = (result.status || 0) === 0;
let finalResult = passed ? 'pass' : 'fail';

// Apply cold-start override: never fail during cold-start, warn instead
const coldStartState = computeColdStartState(policy, scoreboardPath, checkResultsPath);
if (!passed && coldStartState.inColdStart) {
  finalResult = 'warn';
  process.stderr.write(
    '[run-prism] Cold-start mode: suppressing fail, emitting warn instead\n' +
    '[run-prism]   CI runs: ' + coldStartState.ciRunCount + ' / ' + policy.cold_start.min_ci_runs + '\n' +
    '[run-prism]   Quorum rounds: ' + coldStartState.quorumRoundCount + ' / ' + policy.cold_start.min_quorum_rounds + '\n' +
    '[run-prism]   Days: ' + coldStartState.daysSinceFirst.toFixed(2) + ' / ' + policy.cold_start.min_days + '\n'
  );
}

// Build metadata with observation window
const metadata = {
  observation_window: {
    window_start:  coldStartState.firstRunTimestamp ? new Date(coldStartState.firstRunTimestamp).toISOString() : new Date().toISOString(),
    window_end:    new Date().toISOString(),
    n_rounds:      coldStartState.quorumRoundCount,
    n_events:      coldStartState.ciRunCount
  }
};

// Write result with metadata
try {
  writeCheckResult({
    tool: 'run-prism',
    formalism: 'prism',
    result: finalResult,
    metadata: metadata
  });
} catch (e) {
  process.stderr.write(`[run-prism] Warning: failed to write check result: ${e.message}\n`);
}

process.exit(passed ? 0 : (finalResult === 'warn' ? 0 : 1));  // warn does not exit 1
```

## Open Questions

1. **Timestamp Format in quorum-scoreboard.json**
   - What we know: `quorum-scoreboard.json` has a `rounds[]` array; each round may have a `date` field (format unclear).
   - What's unclear: Is `date` ISO 8601 ("2026-02-27T14:00:00Z"), Unix timestamp (ms), or simple string ("02-27")?
   - Recommendation: Check existing scoreboard fixtures and format accordingly. If `date` is "MM-DD" format (as seen in run-prism.test.cjs), parsing is trickier; recommend adding an explicit `timestamp: ISO8601` field to future scoreboard entries, or parse date+year context.

2. **CI Run Count Source**
   - What we know: need to count "CI runs" for `min_ci_runs` threshold.
   - What's unclear: Is this "number of check-results.ndjson entries" or "number of formal-verify.cjs orchestrator invocations" or "number of quorum rounds"?
   - Recommendation: Use `check-results.ndjson` line count as the ground truth for CI runs. This is the artifact CI actually produces. Scoreboard rounds are quorum-specific and not all CI runs have quorum.

3. **Policy Validation Strictness**
   - What we know: policy.yaml is required; invalid YAML should error.
   - What's unclear: Should missing optional fields (e.g., `conservative_priors`) default or error?
   - Recommendation: All four top-level sections (cold_start, steady_state, conservative_priors, and their fields) are required. Fail fast if any are missing. This keeps policy explicit and auditable.

4. **Cold-Start to Steady-State Transition Behavior**
   - What we know: Once thresholds are met, `inColdStart` becomes false.
   - What's unclear: Does a single past-threshold run stay false forever, or do we re-evaluate on each check?
   - Recommendation: Re-evaluate on each `run-prism.cjs` invocation. Scoreboard might be cleared or reset; use current state, not cached state. This is simpler and more robust.

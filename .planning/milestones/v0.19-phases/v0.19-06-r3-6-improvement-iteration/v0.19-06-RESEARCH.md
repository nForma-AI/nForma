## Architecture Patterns

### Pattern 1: Improvements Field Parsing (Mode A, when request_improvements: true)

**What:** When `request_improvements: true` is set in YAML arguments, slot-worker parses the model output for an `Improvements:` section (YAML list format) and extracts each `- suggestion: ... rationale: ...` entry into a structured `improvements:` field in the result block.

**When to use:** Every call to qgsd-quorum-slot-worker in Mode A with `request_improvements: true` (plan-phase and quick workflows after implementation).

**Existing code structure (qgsd-quorum-slot-worker.md):**
```markdown
# Lines 251-260: Mode A improvements parsing instructions
**Mode A, when `request_improvements: true`:** After extracting verdict/reasoning, also scan `$RAW_OUTPUT` for an `Improvements:` section. Parse each `- suggestion: ... rationale: ...` entry.

improvements:
  - suggestion: "..."
    rationale: "..."
```

**What must be implemented:**
- Extract the `Improvements:` section from $RAW_OUTPUT (if present)
- Parse YAML list format: `- suggestion: <text>\n  rationale: <text>`
- Validate each entry has both fields
- Return in result block ONLY when entries found; omit field entirely if absent/empty/malformed
- Never fail the result on parse errors — improvements are additive, not required

**Source:** agents/qgsd-quorum-slot-worker.md lines 251-260

### Pattern 2: Improvements Signal Emission (Quorum Final Round)

**What:** After quorum reaches consensus in Mode A, the quorum orchestrator collects `improvements:` fields from worker result blocks of the final consensus round, de-duplicates by suggestion text, and emits an HTML comment signal parseable by the calling workflow.

**When to use:** End of Mode A quorum run when `request_improvements: true` was set in initial dispatch.

**Signal format:**
```html
<!-- QUORUM_IMPROVEMENTS_START
[{"model":"<slotName>","suggestion":"...","rationale":"..."},...]
QUORUM_IMPROVEMENTS_END -->
```

**De-duplication rule:** If multiple models propose identical or semantically equivalent suggestions, keep only the first occurrence (by dispatch order).

**Fail-open:** If no improvements collected → emit `<!-- QUORUM_IMPROVEMENTS_START [] QUORUM_IMPROVEMENTS_END -->`

**Signal placement:** Emit in main quorum output after consensus verdict section, before the "Supporting positions" table. This makes it visible to the calling workflow (plan-phase, quick) which scans the response for the delimiters.

**Source:** commands/qgsd/quorum.md lines 449-471 (Post-consensus improvements extraction section)

### Pattern 3: R3.6 Outer Loop (Plan-Phase)

**What:** After quorum returns, plan-phase parses the QUORUM_IMPROVEMENTS signal and conditionally loops: if improvements non-empty AND iteration < 10, spawn planner in improvement-revision mode to incorporate them; if no improvements or iteration >= 10, break loop and proceed to next step.

**Loop state:**
- `improvement_iteration = 0` initially (not counted in 10-iteration cap)
- Increment to 1 when first planner revision spawned
- Cap at 10 total iterations
- Count only planner revision calls, not the initial quorum run

**When to use:** After every quorum run in plan-phase when `request_improvements: true` (currently set in step 8.5).

**Planner revision prompt fields:**
```yaml
<revision_context>
Phase: ${PHASE_NUMBER}
Mode: improvement-revision (R3.6 iteration ${improvement_iteration}/10)

<files_to_read>
- ${PHASE_DIR}/*-PLAN.md (current plans to revise)
- ${CONTEXT_PATH} (USER DECISIONS — must not contradict)
</files_to_read>

<quorum_improvements>
${QUORUM_IMPROVEMENTS formatted as readable bullet list}
</quorum_improvements>

<instructions>
Incorporate the quorum improvements above into the plans.
Make targeted updates only. Do NOT replan from scratch.
If an improvement conflicts with user decisions in CONTEXT.md, skip it and note why.
Return a summary of what changed.
</instructions>
</revision_context>
```

**Conflict detection:** If two improvements are mutually incompatible (one undoes the other), escalate to user BEFORE spawning planner for user resolution and filtering.

**Break conditions:**
1. BLOCKED verdict → report blocker, do not proceed
2. ESCALATED verdict → present escalation, await user resolution
3. APPROVED AND ($QUORUM_IMPROVEMENTS empty OR improvement_iteration >= 10) → include `<!-- GSD_DECISION -->` summary, break loop, proceed to next step
4. APPROVED AND $QUORUM_IMPROVEMENTS non-empty AND improvement_iteration < 10 → increment iteration, spawn planner revision, continue loop

**Planner return handling:**
- `## PLANNING COMPLETE` → re-read plan files, continue loop
- `## PLANNING INCONCLUSIVE` → do NOT loop again, note "planner could not incorporate improvements", proceed with current plan

**Source:** qgsd-core/workflows/plan-phase.md lines 301-425

### Anti-Patterns to Avoid

- **Pre-filtering improvements:** Do NOT discard improvements deemed "minor" before passing to planner. Pass full `$QUORUM_IMPROVEMENTS` array.
- **Counting initial quorum as iteration 1:** Initial quorum run is iteration 0; iteration counter starts at 1 when first planner revision spawns.
- **Early `<!-- GSD_DECISION -->` emission:** Only emit when breaking the loop, not after each quorum run.
- **Parallel improvement planner:** R3.6 planner revision must be sequential after quorum, not parallel with other tasks.
- **Assuming IMPROVEMENTS signal always present:** Signal may be absent if `request_improvements` not set or if no improvements proposed; fail-open to empty array.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parse YAML improvements list from model output | Custom regex/parser for all YAML edge cases | Simple line-by-line scan for `- suggestion:` and `- rationale:` pairs with basic validation | Edge cases: quoted strings, multiline suggestions, nested YAML all require robust YAML parser; better to validate exact format expected and fail gracefully |
| De-duplicate improvements across models | Custom object diffing algorithm | Exact string matching on `suggestion` field; if identical strings → keep first occurrence | Semantic equivalence detection is non-trivial; lexical matching is deterministic and reproducible |
| Track improvement loop state across quorum rounds | Custom state machine | Simple counter variable `improvement_iteration`; increment in one place (planner spawn), reset never | State machines over-complicate linear counting; single counter sufficient for 10-iteration cap |
| Parse HTML comment signal from quorum output | Custom HTML parser | Grep for delimiters, substring extract between QUORUM_IMPROVEMENTS_START/END, JSON.parse the content | Full HTML parsing is overkill for fixed-format signal; substring matching is fast and deterministic |

**Key insight:** Improvements infrastructure is mostly string-matching and array operations. YAML parsing should be defensive (fail gracefully on malformed input) rather than comprehensive.

## Common Pitfalls

### Pitfall 1: Improvements Signal Not Emitted in All Cases

**What goes wrong:** Quorum completes successfully but no improvements signal appears in output (either missing delimiters or empty array). Calling workflow fails to detect improvements and doesn't loop.

**Why it happens:** `request_improvements: true` not set in initial dispatch, or improvements parsing in slot-worker failed silently (graceful degradation per spec), or quorum signal emission code was conditional on some intermediate state that wasn't met.

**How to avoid:**
- Always set `request_improvements: true` in YAML dispatch block for plan-phase and quick (lines in plan-phase.md and quick.md)
- Emit signal even if empty: `<!-- QUORUM_IMPROVEMENTS_START [] QUORUM_IMPROVEMENTS_END -->` must appear in output
- Test signal emission with mock improvements data

**Warning signs:**
- Plan-phase/quick workflow completes without looping despite improvements being present
- Signal block appears in debate file but not in main quorum output
- Calling workflow greps for signal and gets no match

### Pitfall 2: Iteration Counter Confusion (Counting Initial Quorum as Iteration 1)

**What goes wrong:** Loop breaks prematurely because iteration counter was incremented during initial quorum run, leaving only 9 iterations for actual improvements.

**Why it happens:** Confusion about when to increment: immediately after parsing signal vs. only when spawning planner revision.

**How to avoid:**
- Initialize `improvement_iteration = 0` before first quorum
- Increment ONLY inside the planner revision spawn block (lines ~367-414 in plan-phase.md)
- Do NOT increment after quorum parsing
- Comment clearly: "// iteration_0 = initial quorum, iteration_1..10 = planner revisions"

**Warning signs:**
- Quorum loop exits at iteration 9 or lower despite improvements remaining
- Logs show "iteration_0" being treated as a revision iteration
- Total improvements incorporated = fewer than expected

### Pitfall 3: CLAUDE.md References Not Fully Removed

**What goes wrong:** Some workflow/agent files still contain direct CLAUDE.md file-read instructions. If CLAUDE.md is later deleted or project structure changes, these reads fail silently (per fail-open policy) but guidance is lost.

**Why it happens:** Incomplete audit; missed a reference in one of the 4 agents + 4 workflows.

**How to avoid:**
- Grep all references before starting: `grep -rn "CLAUDE.md" qgsd-core/workflows agents commands` (yields ~10 matches)
- For each match, identify the specific guidance being referenced
- Embed guidance inline as comments or workflow-specific examples
- Re-grep after changes to confirm all removed
- Add verification step to check removal

**Warning signs:**
- Workflow file imports CLAUDE.md in prompt
- Agent file has "Read CLAUDE.md if exists" instruction
- Test suite fails if CLAUDE.md is temporarily removed

### Pitfall 4: Improvement Conflicts Not Detected

**What goes wrong:** Two improvements suggest contradictory changes (e.g., "use async pattern" vs. "simplify to sync"). Planner receives both and produces incorrect/conflicting output.

**Why it happens:** Conflict detection code missing or incomplete; assumed improvements are independent.

**How to avoid:**
- Implement conflict detection before spawning planner: compare each improvement's suggestion text for direct contradictions
- Define "contradiction": one improvement explicitly undoes or contradicts the other (e.g., "add logging" vs. "remove logging")
- If detected, escalate to user BEFORE spawning planner: "R3.6 conflict: improvements from [model A] and [model B] are incompatible. [describe conflict]. Which should take precedence?"
- Await user choice, filter improvements accordingly
- Document conflict resolution in GSD_DECISION comment

**Warning signs:**
- Plan revisions contain contradictory changes in same section
- Planner notes "conflicting improvements" in response
- Improvements loop makes negative progress (earlier improvements undone by later ones)

### Example 1: Parsing Improvements from Raw Output (Slot-Worker)

**Source:** agents/qgsd-quorum-slot-worker.md lines 251-260

The slot-worker receives raw model output and must extract an `Improvements:` section if `request_improvements: true`. Expected format:

```yaml
Improvements:
- suggestion: "Reorder tasks for better dependency resolution"
  rationale: "Tasks 2 and 3 have implicit dependency on task 1; explicit ordering prevents race conditions"
- suggestion: "Add verification step for file existence before read"
  rationale: "Prevents silent failures if artifact paths are invalid during quorum"
```

**Extraction logic (pseudo-code):**
```javascript
function parseImprovements(rawOutput) {
  const improvements = [];
  const lines = rawOutput.split('\n');
  let inImprovementsSection = false;
  let currentEntry = null;

  for (let i = 0; i < lines.length; i++) {
    const line = lines[i];

    if (line.startsWith('Improvements:')) {
      inImprovementsSection = true;
      continue;
    }

    if (inImprovementsSection) {
      if (line.match(/^- suggestion:/)) {
        if (currentEntry && currentEntry.suggestion && currentEntry.rationale) {
          improvements.push(currentEntry);
        }
        currentEntry = { suggestion: line.replace(/^- suggestion:\s*/, '').trim() };
      } else if (line.match(/^\s+rationale:/)) {
        if (currentEntry) {
          currentEntry.rationale = line.replace(/^\s+rationale:\s*/, '').trim();
        }
      } else if (line.trim() === '') {
        // blank line might signal end of improvements section
        if (currentEntry && currentEntry.suggestion && currentEntry.rationale) {
          improvements.push(currentEntry);
          currentEntry = null;
        }
      } else if (!line.match(/^\s/)) {
        // non-indented, non-empty line signals end
        break;
      }
    }
  }

  if (currentEntry && currentEntry.suggestion && currentEntry.rationale) {
    improvements.push(currentEntry);
  }

  return improvements;
}
```

**Integration point:** Call this after extracting `verdict` and `reasoning` from $RAW_OUTPUT. Only add `improvements:` field to result block if array is non-empty.

### Example 2: Emitting Improvements Signal from Quorum

**Source:** commands/qgsd/quorum.md lines 449-471

After consensus is reached, collect improvements from final-round worker blocks:

```bash
# Collect improvements from final consensus round workers
IMPROVEMENTS_JSON="[]"
if [ "$request_improvements" = "true" ]; then
  # For each worker result block from the final round:
  #   - If verdict is not UNAVAIL and improvements field is present and non-empty
  #   - Extract suggestion + rationale
  #   - Add to array

  # De-duplicate by suggestion text (keep first occurrence)
  IMPROVEMENTS_JSON=$(echo "$FINAL_ROUND_RESULTS" | jq -s '
    reduce .[] as $result (
      {};
      if ($result.verdict != "UNAVAIL" and ($result.improvements | length) > 0) then
        reduce $result.improvements[] as $imp (
          .;
          if (.seen[$imp.suggestion] | not) then
            .improvements += [$imp] |
            .seen[$imp.suggestion] = true
          else . end
        )
      else . end
    ) | .improvements // []
  ')
fi

# Emit signal in quorum output
cat <<EOF
<!-- QUORUM_IMPROVEMENTS_START
${IMPROVEMENTS_JSON}
QUORUM_IMPROVEMENTS_END -->
EOF
```

**De-duplication rule:** If two models propose `suggestion: "Add validation step"` with different rationales, keep only the first (by dispatch order).

### Example 3: R3.6 Loop in Plan-Phase Workflow

**Source:** qgsd-core/workflows/plan-phase.md lines 301-425 (condensed)

```bash
# Initialize loop
improvement_iteration=0

# LOOP START
while [ $improvement_iteration -le 10 ]; do
  # Run quorum with request_improvements: true
  # (existing quorum call at lines 328-335)

  # Parse improvements signal
  QUORUM_IMPROVEMENTS=$(echo "$QUORUM_OUTPUT" | \
    sed -n '/<!-- QUORUM_IMPROVEMENTS_START/,/QUORUM_IMPROVEMENTS_END -->/p' | \
    sed 's/^<!-- QUORUM_IMPROVEMENTS_START//' | \
    sed 's/QUORUM_IMPROVEMENTS_END -->$//' | \
    tr -d '\n' | xargs)

  # Route based on verdict
  if [ "$QUORUM_VERDICT" = "BLOCKED" ]; then
    # Report blocker, break loop
    break
  elif [ "$QUORUM_VERDICT" = "ESCALATED" ]; then
    # Report escalation, await user, break loop
    break
  elif [ "$QUORUM_VERDICT" = "APPROVED" ]; then
    # Check improvements
    if [ -z "$QUORUM_IMPROVEMENTS" ] || [ "$QUORUM_IMPROVEMENTS" = "[]" ] || [ $improvement_iteration -ge 10 ]; then
      # No improvements or max iterations reached
      echo "<!-- GSD_DECISION: Quorum approved. R3.6 improvements: $improvement_iteration iterations. -->"
      break
    else
      # Improvements present and iterations remaining
      improvement_iteration=$((improvement_iteration + 1))

      # Spawn planner revision (Task call)
      Task(
        prompt="...<revision_context>...",
        subagent_type="general-purpose",
        model="{planner_model}",
        description="R3.6 improvements (iteration ${improvement_iteration}/10)"
      )

      # Re-read plan files after planner returns
      # Continue loop for next quorum run
    fi
  fi
done
# LOOP END
```

## Open Questions

1. **Improvement conflict detection scope:** How deep to detect conflicts? Are we looking for direct contradictions (e.g., "add X" vs. "remove X") or semantic conflicts (e.g., "use async" vs. "simplify with sync")?
   - **What we know:** Spec requires escalation for incompatible improvements before planner spawn
   - **What's unclear:** Definition of "incompatible" — lexical vs. semantic?
   - **Recommendation:** Start with lexical detection (opposite suggestions: "add"→"remove", "yes"→"no"). Semantic detection can be added if conflicts arise in practice.

2. **Empty improvements signal vs. missing signal:** What if quorum signal block is missing entirely (e.g., `request_improvements` not set)? Should calling workflow treat as empty array or error?
   - **What we know:** Signal emission is conditional on `request_improvements: true` (quorum.md line 471)
   - **What's unclear:** Calling workflow's handling when signal absent
   - **Recommendation:** Treat missing signal as empty array (fail-open). Loop break condition: `if [ -z "$QUORUM_IMPROVEMENTS" ] || [ "$QUORUM_IMPROVEMENTS" = "[]" ]`

3. **De-duplication across multiple rounds:** R3.6 loop runs multiple times. Should improvements be de-duplicated across rounds (e.g., if round 1 quorum proposes improvement A, and round 2 re-proposes it, skip round 2)?
   - **What we know:** De-duplication spec applies within a single quorum run (final consensus round)
   - **What's unclear:** Whether to track improvements across R3.6 iterations to avoid redundant revisions
   - **Recommendation:** Start without cross-round de-duplication (each R3.6 iteration considers fresh improvements from quorum). If performance or convergence issues arise, add tracking.

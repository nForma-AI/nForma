<objective>
Replace the sequential 13-layer remediation dispatch in solve-remediate.md with a dependency-ordered wave-parallel system. Independent layers within each wave run in parallel (up to 3 concurrent agents per RAM budget). Total remediation wall-clock time is measurably shorter than the previous sequential approach.
</objective>

<must_haves>
1. Solve remediation dispatches layers in dependency-ordered waves (max 6 waves) — independent layers within a wave run in parallel, and total remediation wall-clock time is measurably shorter than the previous 13 sequential steps
2. A dependency DAG module (bin/solve-wave-dag.cjs) computes the wave grouping and can be tested independently
</must_haves>

<tasks>
<task type="auto">
  <name>Task 1: Create bin/solve-wave-dag.cjs — dependency DAG and wave computation</name>
  <files>bin/solve-wave-dag.cjs</files>
  <action>
Create a new module bin/solve-wave-dag.cjs that:

1. Defines the layer dependency DAG as an adjacency list. The dependencies are:
   - r_to_f: no dependencies (root)
   - r_to_d: no dependencies (root)
   - t_to_c: no dependencies (root)
   - f_to_t: depends on r_to_f
   - c_to_f: depends on t_to_c
   - f_to_c: depends on f_to_t, c_to_f
   - d_to_c: no cross-layer dependencies (can run with wave 4)
   - git_heatmap: no cross-layer dependencies (can run with wave 4)
   - p_to_f: no dependencies (can run early)
   - c_to_r, t_to_r, d_to_r (reverse discovery): depends on ALL forward layers being complete
   - hazard_model: depends on reverse discovery
   - l1_to_l3: depends on hazard_model (Gate A needs fresh L3 data)
   - l3_to_tc: depends on l1_to_l3 (Gate C needs Gate A results)
   - per_model_gates: depends on l1_to_l3 and l3_to_tc

Note: Use l1_to_l3 (not l1_to_l2/l2_to_l3) per STRUCT-01 collapse.

2. Implements `computeWaves(residualVector)` that:
   - Takes the residual vector and filters to layers with residual > 0
   - Topologically sorts the DAG
   - Groups into waves: layers whose dependencies are all in earlier waves
   - Respects RAM budget: max 3 concurrent agents per wave (split large waves into sub-waves)
   - Returns an array of wave objects: `[{ wave: 1, layers: ['r_to_f', 'r_to_d', 't_to_c'] }, ...]`

3. Implements `getLayerDeps(layerKey)` for individual dependency lookups.

4. When run as CLI (`node bin/solve-wave-dag.cjs`), prints the full DAG and sample wave grouping.

Expected wave grouping (with all layers active):
- Wave 1: r_to_f, r_to_d, t_to_c (3 parallel — no deps)
- Wave 2: f_to_t, c_to_f, p_to_f (3 parallel — deps met by wave 1)
- Wave 3: f_to_c (needs f_to_t + c_to_f from wave 2)
- Wave 4: d_to_c, git_heatmap (independent of wave 3, 2 parallel)
- Wave 5: c_to_r + t_to_r + d_to_r (reverse discovery — needs all forward layers)
- Wave 6: hazard_model → l1_to_l3 → l3_to_tc → per_model_gates (sequential gate chain)

The exact grouping may vary when some layers have residual=0 (skipped).
  </action>
  <verify>node bin/solve-wave-dag.cjs 2>&1 | head -20 && node -e "const {computeWaves} = require('./bin/solve-wave-dag.cjs'); const waves = computeWaves({r_to_f:{residual:1},r_to_d:{residual:1},t_to_c:{residual:1},f_to_t:{residual:1},c_to_f:{residual:1},f_to_c:{residual:1},d_to_c:{residual:1},git_heatmap:{residual:1},c_to_r:{residual:1},t_to_r:{residual:1},d_to_r:{residual:1},hazard_model:{residual:1},l1_to_l3:{residual:1},l3_to_tc:{residual:1},per_model_gates:{residual:1},p_to_f:{residual:1}}); console.log('Waves:', waves.length); console.assert(waves.length <= 6, 'max 6 waves'); console.assert(waves[0].layers.length <= 3, 'max 3 per wave'); console.log('OK')"</verify>
  <done>bin/solve-wave-dag.cjs exists, exports computeWaves, produces at most 6 waves with max 3 layers per wave.</done>
</task>

  <action>
Rewrite the dispatch section of commands/nf/solve-remediate.md to use wave-parallel execution:

1. Replace the "Dispatch order" table and sequential dispatch logic with a wave-based system:

   ```
   ## Step 3: Wave-Parallel Remediation Dispatch

   Load the wave DAG:
   ```javascript
   const { computeWaves } = require('./bin/solve-wave-dag.cjs');
   const waves = computeWaves(residualVector);
   ```

   For each wave:
   a. Display wave banner: `"Wave {N}/{total}: dispatching {layers} in parallel"`
   b. Record wave start time
   c. Dispatch all layers in the wave as parallel Agent calls (up to 3 concurrent per RAM budget)
   d. Wait for ALL agents in the wave to complete
   e. Record wave end time and log: `"Wave {N} complete: {duration}ms"`
   f. Move to next wave
   ```

3. Add wave timing to the output contract:
   ```json
   "remediation_report": {
     "wave_timing": [
       { "wave": 1, "layers": ["r_to_f", "r_to_d", "t_to_c"], "start_ms": 0, "duration_ms": 12000 },
       { "wave": 2, "layers": ["f_to_t", "c_to_f"], "start_ms": 12000, "duration_ms": 8000 }
     ],
     "total_wall_ms": 45000,
     "sequential_estimate_ms": 120000,
     ...existing fields...
   }
   ```

4. Update the RAM constraint note to clarify that waves respect the 3-agent limit: waves with >3 layers are split into sub-waves of 3.

5. The inline layers (3g D->C display-only and 3j hazard model refresh) remain inline — they execute at the start of their wave before dispatching parallel agents.

6. The gate chain (l1_to_l3 → l3_to_tc → per_model_gates) remains sequential within its wave since each depends on the previous.

7. Update constraint #4 (ordering) to reference wave-parallel: "Remediation order is enforced by the dependency DAG in bin/solve-wave-dag.cjs. Within each wave, layers run in parallel. Cross-wave dependencies are respected."

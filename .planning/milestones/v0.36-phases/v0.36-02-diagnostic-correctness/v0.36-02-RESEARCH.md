# Phase v0.36-02 Research: Diagnostic Correctness

## Summary

The solve diagnostic engine (`bin/nf-solve.cjs`) has three correctness gaps: focus filter propagation is incomplete (only 2 of 19 layers check `focusSet`), missing-file scenarios return `residual: 0` instead of `-1` (hiding unknowns as false convergence), and D-to-C classification cache keys are line-number-based rather than content-hash-based (causing stale cache hits when doc content changes at the same line).

## DIAG-01: Focus Filter Propagation

### Current State

`focusSet` is a module-level `Set<string>` in `bin/nf-solve.cjs` (line ~93), populated from `filterRequirementsByFocus()` in `bin/solve-focus-filter.cjs`. Currently only two layers check it:

1. **R->F** (line 659): `uncoveredReqs = uncoveredReqs.filter(r => focusSet.has(r.id || r.requirement_id || ''))`
2. **R->D** (line 1326): `requirements = requirements.filter(r => focusSet.has(r.id || r.requirement_id || ''))`

### Layer Classification

**Forward layers that reference requirements (can filter):**
- F->T (line 726): Has `gapsList` with `requirement_id` ‚Äî can filter gaps to only focus-set requirements
- C->F (line 756): Has `mismatches` with `constant`/`source` ‚Äî no direct req mapping; mark `scoped: false`
- T->C (line 831): Test runner output ‚Äî no req-level granularity; mark `scoped: false`
- F->C (line 1088): Formal verification checks ‚Äî no req-level filtering; mark `scoped: false`
- D->C (line 1379): Broken claims from doc files ‚Äî no direct req mapping; mark `scoped: false`
- P->F (external `sweepPtoF.cjs`): Debt entries with optional `formal_ref` ‚Äî no req mapping; mark `scoped: false`

**Reverse layers (discovery ‚Äî mark as unscoped):**
- C->R (line 1644): Scans source modules for tracing ‚Äî not req-filterable; mark `scoped: false`
- T->R (line 1769): Orphan test discovery ‚Äî not req-filterable; mark `scoped: false`
- D->R (line 1857): Unbacked doc claims ‚Äî could filter req keyword matching but result quality degrades; mark `scoped: false`

**Layer alignment (informational ‚Äî mark as unscoped):**
- L1->L2 (line 2247): Gate A score ‚Äî aggregate; mark `scoped: false`
- L2->L3 (line 2280): Gate B score ‚Äî aggregate; mark `scoped: false`
- L3->TC (line 2311): Gate C score ‚Äî aggregate; mark `scoped: false`
- per_model_gates (line 2363): Model maturity ‚Äî aggregate; mark `scoped: false`

**Informational (mark as unscoped):**
- git_heatmap (line 2412): File-level churn ‚Äî not req-filterable; mark `scoped: false`
- git_history (line 2455): TLA+ drift ‚Äî not req-filterable; mark `scoped: false`
- formal_lint (line 2495): Model lint ‚Äî not req-filterable; mark `scoped: false`
- hazard_model (line 2531): FMEA hazards ‚Äî not req-filterable; mark `scoped: false`

### Implementation Pattern

For filterable layers (F->T), add focus filtering logic similar to R->F:
```javascript
if (focusSet) {
  gapsList = gapsList.filter(g => focusSet.has(g.requirement_id));
}
```

For non-filterable layers, add to the return detail object:
```javascript
return {
  residual: N,
  detail: {
    ...,
    scoped: focusSet ? false : undefined,  // Only present when focus is active
  },
};
```

The `scoped: false` flag tells downstream consumers (TUI, trend) that this layer's residual was NOT filtered to the focus set.

4. **sweepRtoD** (line 1292-1296): no doc files found ‚Üí `residual: 0`
5. **sweepDtoC** (line 1381-1385): no doc files found ‚Üí `residual: 0`
6. **sweepCtoR** (line 1647): `requirements.json` not found ‚Üí `residual: 0`
7. **sweepCtoR** (line 1654): `requirements.json` parse error ‚Üí `residual: 0`
8. **sweepDtoR** (line 1860): `requirements.json` not found ‚Üí `residual: 0`
9. **sweepDtoR** (line 1867): `requirements.json` parse error ‚Üí `residual: 0`
10. **sweepDtoR** (line 1892): no doc files found ‚Üí `residual: 0`
11. **sweepTtoR** (line 1797): no test files ‚Üí `residual: 0` (this is debatable ‚Äî no tests = nothing to trace, but should be -1 for consistency)
12. **sweepTtoC** (line 844): `runner=none` ‚Üí `residual: 0` (intentional skip, NOT a missing file ‚Äî leave as 0)

### Pattern

All missing-file cases should return:
```javascript
return { residual: -1, detail: { skipped: true, reason: 'missing: <filename>' } };
```

Exception: `runner=none` in T->C config is an intentional skip (user configured), not a missing file. Leave as `residual: 0`.

### Impact on Total Computation

The `computeResidual()` function (line 2596) already handles -1: `(r_to_f.residual >= 0 ? r_to_f.residual : 0)`. So changing 0 to -1 for missing files will correctly exclude these from the total while making the unknown state visible in the detail.

## CLASS-01: Content-Hash Cache Keys

### Current Implementation

`itemKey()` in `bin/solve-tui.cjs` (line 1393):
```javascript
function itemKey(catKey, item) {
  if (catKey === 'dtoc') return `${item.doc_file}:${item.value}`;
  if (catKey === 'ctor') return item.file;
  if (catKey === 'ttor') return item.file;
  if (catKey === 'dtor') return `${item.doc_file}:${item.line}`;
  return JSON.stringify(item).slice(0, 100);
}
```

For `dtoc`, the key is `doc_file:value` where `value` is the file path or dependency name being checked. This is already somewhat content-based (the claim value). However, the spec says `dtor` uses `doc_file:line` which IS line-based and needs content hashing.

Wait ‚Äî re-reading the design context: "Currently for dtoc: returns `${item.doc_file}:${item.value}` (file:line-based)" ‚Äî the concern is actually about the `dtor` case where `item.line` is used. For `dtoc`, `item.value` is the claim value (e.g., a file path like "bin/foo.cjs"), which is content-derived. Let me check what `dtor` items look like.

Looking at sweepDtoR (line 1961-1965):
```javascript
unbacked.push({
  doc_file: relativePath,
  line: i + 1,
  claim_text: line.trim().slice(0, 120),
});
```

So for `dtor`, `itemKey` returns `${item.doc_file}:${item.line}` ‚Äî that's the line number, not the content. If a line is inserted above, the line number shifts and the cache key becomes stale.

For `dtoc`, `item.value` is the structural claim value (file path, command, dependency name) ‚Äî not content-hash but also not line-based. Per the design context, we should use SHA-256 of the claim text content.

### Implementation

Replace line-based keys with content hashes:
```javascript
const crypto = require('crypto');

function itemKey(catKey, item) {
  if (catKey === 'dtoc') {
    const content = item.reason || item.value || '';
    return crypto.createHash('sha256').update(content).digest('hex').slice(0, 16);
  }
  if (catKey === 'ctor') return item.file;
  if (catKey === 'ttor') return item.file;
  if (catKey === 'dtor') {
    const content = item.claim_text || item.reason || item.value || '';
    return crypto.createHash('sha256').update(content).digest('hex').slice(0, 16);
  }
  return JSON.stringify(item).slice(0, 100);
}
```

### Cache Migration

Old keys (format: `doc_file:value` or `doc_file:line`) won't match new SHA-256 keys. This is the desired behavior: items get reclassified with fresh Haiku calls. No explicit migration needed ‚Äî old keys become cache misses automatically.

The classification cache file is `.planning/formal/solve-classifications.json`.

# Plan v0.36-02-02: Content-Hash Classification Cache Keys

## Goal

Replace line-number-based and value-based D-to-C classification cache keys with SHA-256 content hashes so that changing doc content at the same line number invalidates the cached verdict and triggers reclassification.

## must_haves

1. D-to-C classification cache keys are SHA-256 content hashes of claim text — changing doc content at the same line number invalidates the cached verdict and triggers reclassification

## Tasks

<task id="01">
<title>Replace itemKey with content-hash implementation</title>
<context>
`itemKey()` in `bin/solve-tui.cjs` (line ~1393) generates cache keys for the Haiku classification cache (`.planning/formal/solve-classifications.json`). Currently:
- `dtoc`: `${item.doc_file}:${item.value}` — file path + claim value (partially content-based but doesn't capture claim context)
- `dtor`: `${item.doc_file}:${item.line}` — file path + line number (line-based, breaks when lines shift)
- `ctor`: `item.file` — file path (already stable, no change needed)
- `ttor`: `item.file` — file path (already stable, no change needed)

The design calls for SHA-256 hashing of the claim text content for `dtoc` and `dtor` categories.
</context>
<action>
In `bin/solve-tui.cjs`:

1. Add `const crypto = require('crypto');` at the top of the file (near other requires).

2. Replace the `itemKey()` function (~line 1393) with:

```javascript
/** Generate a stable content-hash key for an item to use in classification cache */
function itemKey(catKey, item) {
  if (catKey === 'dtoc') {
    const content = item.reason || item.value || '';
    return crypto.createHash('sha256').update(content).digest('hex').slice(0, 16);
  }
  if (catKey === 'ctor') return item.file;
  if (catKey === 'ttor') return item.file;
  if (catKey === 'dtor') {
    const content = item.claim_text || item.reason || item.value || '';
    return crypto.createHash('sha256').update(content).digest('hex').slice(0, 16);
  }
  return JSON.stringify(item).slice(0, 100);
}
```

3. Also check the `archiveItem` function and the archive key derivation (~line 1363):
```javascript
const archKey = catKey === 'dtoc' ? `${item.doc_file}:${item.value}`
  : catKey === 'dtor' ? `${item.doc_file}:${item.line}`
```
This archive key is separate from the classification cache key and uses a different format for the archive file (`archived-solve-items.json`). The archive uses its own key format — do NOT change archive keys, only classification cache keys.
</action>
<done>
`itemKey()` uses `crypto.createHash('sha256')` for `dtoc` and `dtor` categories. The `crypto` module is imported. Archive keys are unchanged.
</done>

<task id="02">
<title>Verify cache migration is automatic (old keys become misses)</title>
<context>
The existing classification cache at `.planning/formal/solve-classifications.json` has keys in the old format (e.g., `README.md:bin/foo.cjs` for dtoc, `README.md:42` for dtor). After the itemKey change, these old keys won't match the new SHA-256 format. This is intentional: items get reclassified with fresh Haiku calls.

No explicit migration code is needed — old keys naturally become cache misses. But we should verify this works correctly and that the cache file doesn't grow unboundedly with orphaned old-format keys.
</context>
<action>
1. In `bin/solve-tui.cjs`, in the `classifyWithHaiku()` function, find where it reads the existing cache (~line 1230). After classification completes and before writing the cache back, add a cleanup step that removes keys that don't match the current SHA-256 format (16 hex chars):

```javascript
// Clean up old-format keys that are no longer valid (cache migration)
for (const catKey of ['dtoc', 'dtor']) {
  const catCache = existingCache[catKey] || {};
  for (const key of Object.keys(catCache)) {
    // New keys are 16-char hex strings; old keys contain ':' or '/'
    if (key.includes(':') || key.includes('/')) {
      delete catCache[key];
    }
  }
}
```

This runs on every classification pass and gradually cleans up old keys.

2. Add a comment near `itemKey()` noting the format change:
```javascript
// NOTE: Keys changed from path:value to SHA-256 content hashes in v0.36-02 (CLASS-01).
// Old-format keys (containing ':' or '/') are treated as cache misses and cleaned up
// during the next classifyWithHaiku() run.
```
</action>
<done>
Old-format classification cache keys are automatically cleaned up during reclassification. No manual migration step needed. Comment documents the format change.
</done>

# Phase v0.24-02: Dispatch Reliability - Research

**Researched:** 2026-03-02
**Domain:** Quorum dispatch reliability — provider health checking, slot availability windows, success rate ordering
**Confidence:** HIGH

## Summary

Phase v0.24-02 implements three core reliability features for the quorum dispatch system:

1. **Provider health probing** (DISP-01): Before building the dispatch list, `qgsd-prompt.js` runs a fast HTTP health probe (< 3 seconds) per provider. Any provider that fails the probe is excluded, and ALL slots backed by that provider are removed from the dispatch list.

2. **Slot availability windows** (DISP-02): The dispatch list excludes slots whose `available_at` timestamp in the quorum scoreboard is in the future. This prevents dispatching to slots that are in a cooling-down period after recent failures.

3. **Success rate ordering** (DISP-03): Rather than using a static tier sequence, the dispatch list is ordered by descending recent success rate from scoreboard slot stats. The most reliable slots (highest recent success rate) are dispatched first.

This phase depends on v0.24-01 (Provider Infrastructure) which established the provider field in providers.json, the provider-skip pattern in qgsd-prompt.js, and the retry/backoff logic in call-quorum-slot.cjs. DISP-02 and DISP-03 require reading and parsing the quorum-scoreboard.json data structure populated by earlier phases.

**Primary recommendation:** Implement in order: (1) slot-availability window filtering, (2) success-rate ordering by reading scoreboard slot stats, (3) finalize provider health integration (already mostly done in v0.24-01, just needs integration with the ordering system).

## Architecture Patterns

### Existing Pattern: Provider Health Caching

**What:** `check-provider-health.cjs` probes LLM provider endpoints (GET /models) and caches results to `~/.claude/qgsd-provider-cache.json` with TTL:
- UP providers: 180s TTL (3 minutes)
- DOWN providers: 300s TTL (5 minutes)

**Why:** Fast health checks avoid blocking the critical path. Cache avoids repeated probes for the same provider within a short window.

**Location:** `bin/check-provider-health.cjs` (implements probing and caching); results read by `qgsd-prompt.js` via `getDownProviderSlots()` function.

### Existing Pattern: Hostname Normalization for Provider Matching

**What:** To match cache entries (keyed by base URL like `https://api.akashml.com/v1`) to provider fields in providers.json (values like `akashml`), normalize the hostname:
1. Extract hostname from URL: `api.akashml.com`
2. Strip `api.` prefix: `akashml.com`
3. Strip TLD (.com, .ai, .xyz, .io): `akashml`
4. Match against provider field using substring containment: provider field includes normalized hostname OR normalized hostname includes provider field

**Example matching:**
- Cache key: `https://api.akashml.com/v1` → hostname: `api.akashml.com` → normalized: `akashml`
- Provider field: `akashml` → exact match → skip all slots with provider="akashml"

**Why:** Providers may vary their hostname formats; substring matching is flexible and fail-safe.

**Location:** Already implemented in `qgsd-prompt.js` lines 176-210 (getDownProviderSlots function).

### New Pattern: Slot Availability Window Filtering (DISP-02)

**What:** Before adding a slot to the dispatch list, check if `available_at` timestamp in quorum-scoreboard.json is in the future. If yes, exclude it (skip the slot).

**When to use:** After reading quorum-scoreboard.json and before building the cappedSlots list for dispatch.

**Example:**
```javascript
// Read scoreboard
const scoreboard = JSON.parse(fs.readFileSync(scoreboardPath, 'utf8'));

// Filter out slots with available_at in the future
const availableSlots = cappedSlots.filter(slot => {
  const slotData = scoreboard.slots?.[slot.slot];
  if (!slotData || !slotData.available_at) return true; // Default: include (no availability window data)
  const availableTime = new Date(slotData.available_at).getTime();
  return availableTime <= Date.now(); // Exclude if available_at is in the future
});
```

**Fail-open:** If scoreboard is missing, malformed, or slot data absent, default to including the slot (not excluding it). This prevents cascading failures.

### New Pattern: Success Rate Ordering (DISP-03)

**What:** Sort the dispatch list by descending recent success rate computed from scoreboard slot stats.

**When to use:** After filtering out DOWN providers and unavailable slots, reorder the remaining slots by success rate.

**Formula:** `success_rate = tp / (tp + fn)` where `tp` = true positives (APPROVE decisions), `fn` = false negatives (slots that should have APPROVEd but BLOCKED instead).

**Example:**
```javascript
// Compute success rate for each slot
const slotsWithRates = availableSlots.map(slot => {
  const slotData = scoreboard.slots?.[slot.slot] || { tp: 0, fn: 0 };
  const rate = (slotData.tp + slotData.fn) > 0
    ? slotData.tp / (slotData.tp + slotData.fn)
    : 0.5; // Default: neutral rate if no history
  return { ...slot, successRate: rate };
});

// Sort by descending success rate
const ordered = slotsWithRates.sort((a, b) => b.successRate - a.successRate);
```

**Fail-open:** If scoreboard missing or slot stats absent, use a neutral success rate (0.5) so the slot is tried at normal priority.

### Anti-Patterns to Avoid

- **Blocking on health probes:** Don't wait for check-provider-health.cjs to complete synchronously. Use the pre-cached results from qgsd-provider-cache.json instead. If cache is stale, that's acceptable — better than blocking the prompt hook (< 100ms overhead target).
- **Hardcoding success rates:** Don't compute success rate on-the-fly in qgsd-prompt.js. Read it from the pre-computed scoreboard. Scoreboard is maintained by update-scoreboard.cjs after each quorum round.
- **Overwriting the static FALLBACK-01 pattern:** DISP-03 reorders the dispatch list by success rate, but it doesn't change the T1/T2 fallback tiers. The ordered list is still used sequentially; UNAVAIL slots still skip to the next tier. Don't merge the ordering with tier logic — keep them orthogonal.
- **Excluding all slots when all providers down:** Don't return an empty dispatch list. Fall back gracefully (success criterion 4: dispatch has >= 1 candidate slot). If all primary providers are DOWN, dispatch to whatever T1/T2 fallback slots remain.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Provider health checking | Custom HTTP client with retry logic | `check-provider-health.cjs` (v0.24-01) | Handles timeout, TTL caching, error classification; already battle-tested |
| Slot availability window logic | Custom time-window parser | Native `Date` and timestamp comparison (source: quorum-scoreboard.json) | Simple comparison; no library needed. Fail-open if data missing. |
| Success rate computation | Custom statistical analysis | Simple arithmetic: `tp / (tp + fn)` from scoreboard | No library needed; scoreboard stats already collected by prior phases |
| Hostname normalization | Regex-based URL parsing | Native `URL` constructor + string methods (source: v0.24-01 pattern) | URL parsing is error-prone; native constructor handles edge cases |
| Sorting algorithms | Custom comparator logic | Native `Array.sort()` with a simple comparator function | JavaScript sort is O(n log n) and proven; don't implement custom sort |

**Key insight:** All the hard problems (provider probing, scoreboard aggregation, state management) are already solved. This phase just combines existing pieces: filter by availability window, sort by success rate, ensure graceful fallback.

## Common Pitfalls

### Pitfall 1: Stale Cache Data Leading to Dispatch of Dead Providers

**What goes wrong:** Cache entry shows provider UP, but provider actually DOWN since cache was written. Dispatch fails when task tries to use that slot.

**Why it happens:** TTL is too long, or the cache was not refreshed before prompt hook runs. Provider can go DOWN between cache writes.

**How to avoid:**
- Accept that a small percentage of dispatches will hit recently-failed providers
- Rely on retry logic in call-quorum-slot.cjs (FAIL-01: 2 retries with backoff)
- Monitor quorum-failures.json for recurring DOWN events
- Don't try to detect DOWN in real-time; use the cache as a heuristic, not ground truth

**Warning signs:**
- Logs show "slot timed out" repeatedly for the same provider
- Multiple failures on the same provider within 5 minutes
- quorum-failures.json has many TIMEOUT entries for a single provider

### Pitfall 2: Missing Scoreboard Data During Early Runs

**What goes wrong:** First quorum run happens before scoreboard is populated. Success rate is undefined. Code crashes or dispatch list is empty.

**Why it happens:** Scoreboard is built incrementally by update-scoreboard.cjs after each deliberation round. Early runs have no historical data.

**How to avoid:**
- Default to neutral success rate (0.5) if scoreboard missing or slot stats absent
- Fail-open: assume all slots are equally reliable if no history
- Accept that early runs are slightly less optimized (no success rate bias)
- Don't skip dispatch if scoreboard is missing; dispatch anyway with default ordering

**Warning signs:**
- quorum-scoreboard.json does not exist on first run
- Slot entries in scoreboard have missing tp/fn fields
- Code throws "Cannot read property 'tp' of undefined"

### Pitfall 3: Availability Window Check Uses Wrong Timezone

**What goes wrong:** `available_at` timestamp is stored in UTC, but the filter checks it against local system time. Slots may be incorrectly excluded or included based on timezone offset.

**Why it happens:** JavaScript `Date.now()` returns milliseconds in UTC, but developer assumes local timezone.

**How to avoid:**
- Always use UTC timestamps in JSON (ISO 8601 format: "2026-03-02T14:30:00Z")
- Always compare using `Date.now()` (UTC milliseconds)
- Never use `new Date().getHours()` or similar local-time methods
- Document that `available_at` is always UTC

**Warning signs:**
- Slots excluded for 1 hour at a time
- Discrepancy between "now" on the server and actual time
- Tests pass locally but fail in CI (different timezone)

### Pitfall 4: Dispatch List Empty When All Providers Down (No Graceful Fallback)

**What goes wrong:** All primary providers are DOWN. Code returns an empty dispatch list. Quorum fails with "no slots to dispatch."

**Why it happens:** Filter logic is too aggressive: removes all DOWN-provider slots without checking if any slots remain.

**How to avoid:**
- Always verify that at least one slot remains after filtering (success criterion 4)
- If all providers are DOWN, still dispatch to whatever slots remain (even if unreliable)
- Use the T1/T2 fallback tiers: if all primary (capped) slots are DOWN, fall back to unused T1 subs, then T2 ccr/api slots
- Document the graceful degradation: "Worst case: dispatch to a single T2 slot if all primaries fail"

**Warning signs:**
- DISPATCH_LIST has 0 slots when providers are DOWN
- Quorum instructions reference slots that were all filtered out
- Log shows "no available slots for dispatch"

### Pitfall 5: Success Rate Never Transitions Slot from Low to High

**What goes wrong:** A slot has low success rate historically, gets improved, but the ordered dispatch list still has it at low priority.

**Why it happens:** Success rate is computed from cumulative tp/fn across all time. Recent improvements don't quickly change the aggregate rate.

**How to avoid:**
- Accept that success rate ordering is a heuristic, not a ground truth
- Don't try to use a "recent window" (e.g., last 10 runs) for rate — use cumulative stats
- Rely on human monitoring and eventual-consensus: even if a slot is low-priority, it will eventually get dispatched if higher-priority slots fail
- If a provider's slots are historically bad, deprioritize them and monitor for recovery

**Warning signs:**
- A slot with 1000+ dispatch attempts has low success rate; new improvements don't move it up
- Operator sees new provider added with high reliability but dispatch doesn't prioritize it
- Deliberation takes many rounds because low-rate slots are tried first

## Code Examples

### Example 1: Slot Availability Window Filtering (DISP-02)

Verified pattern from codebase analysis:

```javascript
// From qgsd-prompt.js (new pattern, DISP-02)
// Read quorum-scoreboard.json and filter out slots whose available_at is in the future
function getAvailableSlots(cappedSlots, cwd) {
  try {
    const scoreboardPath = path.join(cwd, '.planning', 'quorum-scoreboard.json');
    if (!fs.existsSync(scoreboardPath)) return cappedSlots; // Fail-open: no scoreboard yet

    const scoreboard = JSON.parse(fs.readFileSync(scoreboardPath, 'utf8'));
    if (!scoreboard.slots || typeof scoreboard.slots !== 'object') {
      return cappedSlots; // Fail-open: malformed scoreboard
    }

    const now = Date.now();
    return cappedSlots.filter(slot => {
      const slotData = scoreboard.slots[slot.slot];
      if (!slotData || !slotData.available_at) {
        return true; // No availability window → include by default
      }

      try {
        const availableTime = new Date(slotData.available_at).getTime();
        return availableTime <= now; // Exclude if available_at is in the future
      } catch (_) {
        return true; // Invalid timestamp → fail-open, include
      }
    });
  } catch (_) {
    return cappedSlots; // Fail-open: any error reading scoreboard → include all
  }
}
```

### Example 2: Success Rate Ordering (DISP-03)

Verified pattern from codebase analysis:

```javascript
// From qgsd-prompt.js (new pattern, DISP-03)
// Sort slots by descending recent success rate
function sortBySuccessRate(availableSlots, cwd) {
  try {
    const scoreboardPath = path.join(cwd, '.planning', 'quorum-scoreboard.json');
    if (!fs.existsSync(scoreboardPath)) {
      return availableSlots; // Fail-open: no scoreboard yet, return in original order
    }

    const scoreboard = JSON.parse(fs.readFileSync(scoreboardPath, 'utf8'));
    if (!scoreboard.slots || typeof scoreboard.slots !== 'object') {
      return availableSlots; // Fail-open: malformed scoreboard
    }

    // Compute success rate and attach to each slot
    const slotsWithRates = availableSlots.map(slot => {
      const slotData = scoreboard.slots[slot.slot] || {};
      const tp = slotData.tp || 0;
      const fn = slotData.fn || 0;

      // Success rate = true positives / (true positives + false negatives)
      // Default to 0.5 (neutral) if no history
      const successRate = (tp + fn > 0) ? tp / (tp + fn) : 0.5;

      return { ...slot, successRate };
    });

    // Sort by descending success rate
    return slotsWithRates.sort((a, b) => b.successRate - a.successRate);
  } catch (_) {
    return availableSlots; // Fail-open: any error, return in original order
  }
}
```

## Open Questions

1. **Success rate denominator: tp+fn vs. tp+fn+fp?**
   - What we know: QGSD uses tp (true positives = APPROVE) and fn (false negatives = should have APPROVED). tp+fn is the total of "correct decisions."
   - What's unclear: Should success rate include false positives (FP = incorrectly BLOCKED)? Current formula: `tp / (tp + fn)` ignores FP.
   - Recommendation: Use `tp / (tp + fn)` for now. FP are rare (slots mostly APPROVE unless spec is wrong). If FP become common, revisit denominator to `tp / (tp + fn + fp)`.

2. **Availability window default: neutral or optimistic?**
   - What we know: If a slot has no `available_at` data, should we include or exclude it?
   - What's unclear: Is "no window" = "always available" or "unknown availability"?
   - Recommendation: Default to include (optimistic). If a slot has no availability window, it's probably always available. This aligns with fail-open principle.

3. **How often is quorum-scoreboard.json updated during a session?**
   - What we know: update-scoreboard.cjs is called after each deliberation round (per qgsd-prompt.js lines 436-437).
   - What's unclear: Is it called after every round, or only when consensus is reached?
   - Recommendation: Verify in update-scoreboard.cjs implementation (v0.24-03). For DISP-02/DISP-03, assume the file is updated and read on each prompt hook invocation.

4. **Should DISP-02 and DISP-03 be in qgsd-prompt.js or a separate module?**
   - What we know: Both are used to filter/reorder cappedSlots before building dynamic steps.
   - What's unclear: At what point does the code become too complex for a single hook file?
   - Recommendation: Keep in qgsd-prompt.js for now (lines ~334-360). If logic grows > 50 lines, refactor to `bin/quorum-slot-ordering.cjs` as a pure module. Hook remains simple: call the module, get ordered slots.

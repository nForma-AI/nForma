# Plan v0.24-02-02 Summary: Dispatch Reliability Implementation

## What was built

Three dispatch reliability functions added to hooks/qgsd-prompt.js and integrated into the quorum dispatch flow:

1. **triggerHealthProbe()** (DISP-01): Runs `check-provider-health.cjs --json` via spawnSync with 3s timeout before reading the provider cache. Fail-open: if probe script not found or spawn times out, dispatch continues.

2. **getAvailableSlots(slots, cwd)** (DISP-02): Reads `.planning/quorum-scoreboard.json` availability section. Filters out slots whose `available_at_iso` timestamp is in the future (cooling down). Handles NaN dates via explicit `isNaN(ts)` check. Emits `console.error` with `AVAILABILITY EXCLUDE` message for each filtered slot. Adds `SKIP (COOLING DOWN)` skip note.

3. **sortBySuccessRate(slots, cwd)** (DISP-03): Reads `.planning/quorum-scoreboard.json` slots section. Aggregates tp/fn across all model entries per slot using `key.split(':')[0]` composite key extraction. Sorts by descending success rate with 0.5 default for unknown slots. Emits per-slot `SUCCESS RATE` and final `DISPATCH ORDER` debug logs.

4. **Graceful fallback** (SC-4): After all filtering, if cappedSlots is empty, relaxes filters to include any non-DOWN-provider slot. Last resort: uses first original slot regardless.

## Integration point

```
triggerHealthProbe()                          // DISP-01
cappedSlots = filter by skipSet               // existing
cappedSlots = getAvailableSlots(cappedSlots)  // DISP-02
cappedSlots = sortBySuccessRate(cappedSlots)  // DISP-03
if (cappedSlots.length === 0) { fallback }    // SC-4
```

# Phase v0.24-01 Plan 02: Retry with Exponential Backoff

**One-liner:** Implement retry-with-exponential-backoff (1s, 3s, max 2 retries) for transient slot call failures before recording UNAVAIL in call-quorum-slot.cjs (FAIL-01).

## Summary

Completed the retry-with-exponential-backoff implementation in `bin/call-quorum-slot.cjs` as specified in requirement FAIL-01. The implementation:

- Adds `retryWithBackoff()` function that wraps slot dispatch calls (both subprocess and HTTP)
- Implements transient error classification via `isRetryable()` function
- Applies exponential backoff delays: 1000ms (first retry), 3000ms (second retry)
- Limits retries to 2 (3 total attempts)
- Non-retryable errors (CLI_SYNTAX, spawn errors) fail immediately without retry
- Retryable errors (TIMEOUT, network errors) trigger automatic retry
- Fail-open principle: unknown errors are retryable by default
- Preserves OAuth rotation loop: retry wrapping is applied inside the rotation loop, so each OAuth rotation attempt gets retry protection
- Logs retry attempts to stderr in format: `[call-quorum-slot] retry N/M for slot <name> after <delay>ms`
- Failure recording (writeFailureLog) only occurs after all retries are exhausted

### Task 1: Implement Retry-with-Backoff in call-quorum-slot.cjs

**Status:** COMPLETE

**Changes:**
1. Added `sleep(ms)` utility function for async delays
2. Added `isRetryable(error)` function:
   - Returns `false` for CLI_SYNTAX patterns (usage, unknown flag, etc.) and spawn errors
   - Returns `true` for TIMEOUT, network errors (ECONNREFUSED, ENOTFOUND, ECONNRESET, ETIMEDOUT)
   - Returns `true` for unknown errors (fail-open)
3. Added `retryWithBackoff(fn, slotName, maxRetries = 2, delays = [1000, 3000])` function:
   - Executes function up to maxRetries times (3 total attempts)
   - Fails immediately if error is non-retryable
   - Fails immediately on last attempt if error is retryable
   - Logs retry attempts to stderr with delay information
   - Sleeps with exponential backoff before each retry
4. Updated `main()` dispatch logic:
   - Non-OAuth subprocess slots: `await retryWithBackoff(() => runSubprocess(...), slot)`
   - HTTP slots: `await retryWithBackoff(() => runHttp(...), slot)`
   - OAuth subprocess slots: Wrapped inner `runSubprocess()` call inside `runSubprocessWithRotation()` loop with `retryWithBackoff()`
5. Existing error handling (writeFailureLog, appendTokenSentinel) remains unchanged — retry is transparent

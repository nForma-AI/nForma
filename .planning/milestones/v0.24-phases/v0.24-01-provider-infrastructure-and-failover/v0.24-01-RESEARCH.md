# Research: Phase v0.24-01 — Provider Infrastructure and Failover

## Summary

Phase v0.24-01 implements the foundational failover and provider mapping layer for quorum dispatch. The two requirements (FAIL-01, FAIL-02) address: (1) retry logic with exponential backoff in `call-quorum-slot.cjs` before recording UNAVAIL, and (2) explicit slot-to-provider mapping in `providers.json` so that a single provider health probe can skip all slots backed by a dead provider in one dispatch decision.

## Requirements Addressed

- **FAIL-01**: `call-quorum-slot.cjs` retries a failed slot call up to 2 times with exponential backoff (1s, 3s) before recording UNAVAIL in `quorum-failures.json`
- **FAIL-02**: `providers.json` contains explicit slot-to-provider mapping; when a provider probe returns DOWN, all slots on that provider are skipped in a single dispatch decision

## Current State Analysis

### call-quorum-slot.cjs (FAIL-01 target)

**Location:** `bin/call-quorum-slot.cjs`

Current behavior:
- On failure (timeout or error), immediately writes to `quorum-failures.json` via `writeFailureLog()` and exits with code 1
- No retry logic exists — a single transient failure immediately records UNAVAIL
- OAuth rotation retry exists (`runSubprocessWithRotation`) but only for auth/quota patterns, not general failover
- Timeout logic correctly handles `--timeout` flag with provider cap
- Process group kill (`killGroup`) already handles cleanup correctly

What needs to change:
- Add a retry wrapper around `runSubprocess()` / `runHttp()` calls (not the OAuth rotation path)
- Retry up to 2 times: delays of 1s then 3s (exponential backoff)
- Only retry on transient errors (TIMEOUT, network errors) — NOT on CLI_SYNTAX or spawn errors
- Only record UNAVAIL in `quorum-failures.json` after all retries exhausted
- Log retry attempts to stderr for observability

Key constraint: OAuth rotation already has its own retry loop. The FAIL-01 retry wraps the *inner* call, not the OAuth loop. For slots with `oauth_rotation.enabled`, the flow is: oauth-rotation-loop → retry-with-backoff → runSubprocess. This means the retry is applied per-attempt inside the oauth loop.

### providers.json (FAIL-02 target)

**Location:** `bin/providers.json`

Current state:
- Each provider entry has a `display_provider` field (e.g., "AkashML", "Together.xyz", "Fireworks", "OpenAI", "Google", "GitHub", "OpenCode")
- However `display_provider` is used for display only — no code uses it for dispatch decisions
- `check-provider-health.cjs` reads `ANTHROPIC_BASE_URL` from `~/.claude.json` mcpServers env to group by provider — but this only covers HTTP-type MCP servers (claude-1 through claude-6), not subprocess CLI slots (codex-1, gemini-1, opencode-1, copilot-1)
- Subprocess CLI slots have no provider mapping that `check-provider-health.cjs` can use

What needs to change:
- Add a `provider` field to each entry in `providers.json` that maps to a canonical provider name
- Provider names should match across slot types: e.g., codex-1 → "openai", codex-2 → "openai", gemini-1 → "google", claude-1 → "akashml", claude-3 → "together-xyz", etc.
- `probe-quorum-slots.cjs` should be extended (or a new integration added to `qgsd-prompt.js`) to read provider-level health status and skip all slots for a DOWN provider in one decision
- `check-provider-health.cjs` already has the HTTP probing logic — the subprocess CLI slots need a health check mechanism too (version check or fast probe)

### qgsd-prompt.js (dispatch integration)

**Location:** `hooks/qgsd-prompt.js`

Current dispatch behavior:
- Reads `quorum-failures.json` to find recently-timed-out slots (30min TTL) via `getRecentlyTimedOutSlots()`
- Emits `SKIP (TIMEOUT < 30min ago): [slot-list]` in the quorum instructions
- Does NOT use provider-level health information for dispatch decisions

What needs to change for FAIL-02:
- Before building the dispatch list, read provider health status (from `check-provider-health.cjs` cache at `~/.claude/qgsd-provider-cache.json`)
- If a provider is DOWN, skip ALL slots backed by that provider — not just individual slots
- This is a single dispatch decision (one provider probe → skip N slots), not per-slot probing

### Provider Health Cache

**Location:** `~/.claude/qgsd-provider-cache.json`

Current state:
- `check-provider-health.cjs` maintains a TTL cache (DOWN: 5min, UP: 3min)
- Cache entries keyed by `baseUrl` (ANTHROPIC_BASE_URL)
- Only covers HTTP/MCP-based providers (AkashML, Together.xyz, Fireworks)
- Subprocess CLI providers (OpenAI/Codex, Google/Gemini, OpenCode, GitHub/Copilot) are not cached

For FAIL-02, the dispatch integration in `qgsd-prompt.js` can read this cache directly. For subprocess CLI providers, health can be inferred from `quorum-failures.json` (recent failures indicate provider issues).

### quorum-failures.json (failure patterns)

**Location:** `.planning/quorum-failures.json`

Current data shows heavy timeout patterns:
- claude-1 (AkashML/DeepSeek): 28 TIMEOUT failures — provider is frequently overloaded
- claude-4 (Fireworks/Kimi): 15 TIMEOUT failures
- claude-2 (AkashML/MiniMax): 13 TIMEOUT failures
- codex-1: 7 TIMEOUT failures (but old, "TIMEOUT after 1ms" suggests zero-timeout bug)
- copilot-1: 5 TIMEOUT failures

This data validates the need for both retry (FAIL-01) and provider-level skip (FAIL-02).

## Implementation Strategy

### Task 1: Retry with Backoff in call-quorum-slot.cjs (FAIL-01)

**Approach:** Add a `retryWithBackoff()` wrapper function that:
1. Attempts the slot call
2. On transient failure (TIMEOUT, network error, non-zero exit with stderr matching transient patterns), waits and retries
3. Delays: 1000ms (attempt 1), 3000ms (attempt 2)
4. On final failure, falls through to existing `writeFailureLog()` + exit(1) path
5. Non-retryable errors (CLI_SYNTAX, spawn errors) fail immediately without retry

**Files modified:** `bin/call-quorum-slot.cjs`

**Test approach:** Unit test with mocked subprocess/HTTP calls that fail transiently then succeed. Test file: `bin/call-quorum-slot.test.cjs` (new).

### Task 2: Provider Field in providers.json (FAIL-02 data layer)

**Approach:** Add a `provider` field to each entry in `providers.json`:
- codex-1, codex-2 → `"openai"`
- gemini-1, gemini-2 → `"google"`
- opencode-1 → `"opencode"` (xAI/Grok backend)
- copilot-1 → `"github"`
- claude-1, claude-2 → `"akashml"`
- claude-3, claude-5 → `"together-xyz"`
- claude-4, claude-6 → `"fireworks"`

Also add a `provider_health_url` field for subprocess slots where applicable (optional — enables future probing).

**Files modified:** `bin/providers.json`

**Test approach:** Structural test that every entry in providers.json has a `provider` field. Use `probe-quorum-slots.cjs` to verify each slot resolves to its provider.

### Task 3: Provider-Aware Dispatch in qgsd-prompt.js (FAIL-02 dispatch layer)

**Approach:** In `qgsd-prompt.js`, before building the dispatch list:
1. Read `~/.claude/qgsd-provider-cache.json` for HTTP provider health
2. Read `bin/providers.json` to map slots → providers
3. If a provider is DOWN, add ALL its slots to the skip list (alongside the existing timeout-based skips)
4. Emit `SKIP (PROVIDER DOWN): [slot-list]` in the quorum instructions

**Files modified:** `hooks/qgsd-prompt.js`, `hooks/dist/qgsd-prompt.js` (install sync)

**Test approach:** Unit test in `hooks/qgsd-prompt.test.js` that mocks provider cache and verifies provider-level skip logic. Integration verified by `probe-quorum-slots.cjs --slots <all>` confirming slot-to-provider resolution.

### Task 4: Acceptance Tests + Integration Verification

**Approach:** End-to-end test that:
1. Runs `node bin/probe-quorum-slots.cjs --slots codex-1,gemini-1 --timeout 1` to verify slot-to-provider resolution
2. Runs `node bin/call-quorum-slot.cjs` with a known-failing slot to verify retry behavior (observable via stderr retry messages and timing)
3. Structural test that `EventualConsensus` is preserved — when at least one provider with responsive slots remains, quorum reaches DECIDED

**Files modified:** New test files

## Formal Considerations

### EventualConsensus Invariant

The quorum formal spec (`formal/tla/QGSDQuorum.tla`) declares `EventualConsensus == <>(phase = "DECIDED")` under weak fairness. FAIL-01 retry does not change the state machine — it only delays the UNAVAIL recording, making responsive slots more likely to succeed. FAIL-02 provider-skip is a dispatch optimization that removes unreachable slots from the candidate set, which does not violate EventualConsensus as long as at least one responsive slot remains (success criterion #4 already covers this).

**Conclusion:** No formal artifact changes needed. Both FAIL-01 and FAIL-02 are implementation changes that preserve the existing formal invariants.

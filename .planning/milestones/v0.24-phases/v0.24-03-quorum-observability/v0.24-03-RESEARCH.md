# Phase v0.24-03: Quorum Observability - Research

**Researched:** 2026-03-03
**Domain:** Quorum dispatch telemetry, slot reliability tracking, structured logging
**Confidence:** HIGH

## Summary

Phase v0.24-03 implements observability for quorum dispatch rounds to enable diagnosis of reliability problems. The phase depends on v0.24-01 (provider infrastructure: providers.json, mapping, health checks) and must deliver three capabilities:

1. **Per-round telemetry logging** — After each quorum round completes, a session-scoped log file records structured data for every slot dispatched: slot name, round number, verdict (APPROVE/BLOCK/FLAG), latency in milliseconds, provider status.

2. **Quorum delivery rate tracking** — The scoreboard (.planning/quorum-scoreboard.json) tracks what percentage of quorum rounds achieved the target vote count (e.g., "3 slots voted: 87%", "2 slots voted: 13%"). This metric surfaces degradation immediately.

3. **Per-slot flakiness scoring** — Each slot gets a computed flakiness score from recent UNAVAIL/timeout frequency (e.g., slot with 3 timeouts in 10 rounds has visibly higher flakiness). High-flakiness slots are deprioritized in dispatch ordering for the next round.

These three pieces form the observability foundation for v0.24-04 (self-healing), which will consume telemetry to compute P(consensus) and auto-escalate when needed.

**Primary recommendation:** Implement telemetry logging in call-quorum-slot.cjs (per-slot data capture), extend quorum-scoreboard.json schema to include per-round delivery stats and per-slot flakiness arrays, and add flakiness-aware dispatch ordering in qgsd-quorum-slot-worker.md or prompt injection.

## Architecture Patterns

### Pattern 1: Session-Scoped Telemetry Logging

**What:** Each quorum dispatch session gets a unique session ID (from Claude Code environment or qgsd-prompt.js). Telemetry for all slots in all rounds is appended to `.planning/quorum-rounds-<SESSION_ID>.jsonl`.

**When to use:** OBS-01 requires structured per-slot, per-round telemetry. Session scoping prevents cross-session log mixing and enables per-session analysis.

**Data structure:**
```json
{
  "ts": "2026-03-03T15:42:13.123Z",
  "session_id": "sess_abc123",
  "round": 1,
  "slot": "gemini-1",
  "verdict": "APPROVE",
  "latency_ms": 2341,
  "provider": "gemini",
  "provider_status": "available",
  "error_type": null,
  "retry_count": 0
}
```

**Implementation location:** call-quorum-slot.cjs main() or wrapper function. After subprocess completes (success or failure), before returning to orchestrator, append telemetry record.

### Pattern 3: Flakiness-Aware Dispatch Ordering

**What:** When building DISPATCH_LIST in qgsd-prompt.js or quorum.md dispatch step, sort slots by flakiness (ascending, i.e., most reliable first).

**Before (DISP-03 — success rate only):**
```javascript
slots.sort((a, b) => scoreboardRates[b] - scoreboardRates[a]); // highest rate first
```

**After (OBS-03):**
```javascript
slots.sort((a, b) => {
  // Primary: flakiness (ascending — lower = more reliable)
  const flakDiff = (slotFlakiness[a] ?? 0) - (slotFlakiness[b] ?? 0);
  if (flakDiff !== 0) return flakDiff;
  // Secondary: success rate (descending — higher = better)
  return scoreboardRates[b] - scoreboardRates[a];
});
```

**When to use:** After each quorum round computes flakiness scores, the next round's dispatch order reflects which slots have been most reliable recently.

### Anti-Patterns to Avoid

- **Logging without fail-open guards:** Telemetry failures must never block quorum dispatch. Always wrap telemetry writes in try/catch and continue.
- **Mutable scoreboard during rounds:** Avoid concurrent reads/writes. Use atomic JSON.parse/stringify with file locks or sequential update calls.
- **Flakiness window too narrow:** 10 rounds is the minimum; less and noise dominates. More than 30 rounds and recent failures have less impact. Use 10 as the canonical window.
- **Missing session ID:** If session ID is not available from environment, generate one via UUID or use a stable timestamp-based ID. Never drop telemetry because session ID is missing.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session ID generation | Custom UUID logic | Use `process.env.CLAUDE_SESSION_ID` (injected by Claude Code hook) or crypto.randomUUID() | CLAUDE_SESSION_ID is reliable when available; fallback is simple |
| Atomic JSON updates | Custom file locking | JSON.parse → modify → JSON.stringify → fs.writeFileSync (atomic for < 4KB per POSIX) | QGSD pattern is established; already used in quorum-scoreboard.json updates |
| Per-slot metrics aggregation | Custom reduce() logic | Use verify-quorum-health.cjs pattern (lines 59-78): filter rounds by slot, count TP/UNAVAIL, compute rate | Existing logic already tested; reuse saves bugs |
| Telemetry routing decisions | Custom filtering rules | Use structured slot status from providers.json (provider field) + call-quorum-slot.cjs subprocess result | Provider metadata is canonical; subprocess result is ground truth |

**Key insight:** Telemetry infrastructure at this scale (< 100 rounds/session, < 10 slots) doesn't need databases or streaming libraries. Append-only JSONL + atomic JSON.parse/stringify scales perfectly and keeps the system testable.

## Common Pitfalls

### Pitfall 1: Session ID Not Available at Dispatch Time

**What goes wrong:** call-quorum-slot.cjs tries to read process.env.CLAUDE_SESSION_ID, gets null, and loses session correlation.

**Why it happens:** Claude Code injects CLAUDE_SESSION_ID as an environment variable only when running inside the Claude Code context. If testing locally or in CI, the variable may not exist.

**How to avoid:** Implement a fallback in call-quorum-slot.cjs:
```javascript
const sessionId = process.env.CLAUDE_SESSION_ID || 'session-' + Date.now();
```

**Warning signs:** Telemetry logs appear in quorum-rounds-null.jsonl or quorum-rounds-undefined.jsonl. Check logs immediately.

### Pitfall 2: Telemetry Latency Measured Incorrectly

**What goes wrong:** Latency includes retry delays, so a 3-second retry + 1-second actual call appears as 4 seconds total. Then flakiness scoring treats it as "slow" not "retried".

**Why it happens:** If telemetry is appended after the entire retry loop, the latency field conflates retries with true dispatch time.

**How to avoid:** Measure latency for the final successful attempt only. Record `retry_count` separately so flakiness scoring can distinguish "retried 2x but succeeded" from "timed out".

**Warning signs:** Slots with 2 retries consistently show 3-4x normal latency. Flakiness scores spike after patterns of retries.

### Pitfall 3: Flakiness Score Computed from Wrong Window

**What goes wrong:** Flakiness array includes more than 10 rounds, or fewer than 5 rounds. Either way, it's not representative of "recent reliability".

**Why it happens:** No explicit window size validation when building the flakiness array.

**How to avoid:** Always use a trailing 10-round window. When appending a new round verdict:
```javascript
slot.flakiness.push({ round, verdict });
if (slot.flakiness.length > 10) slot.flakiness.shift(); // keep exactly 10
```

**Warning signs:** Flakiness arrays of length > 12 or < 8 after round completion.

### Pitfall 4: Delivery Rate Computed Before All Slots Complete

**What goes wrong:** compute-delivery-stats runs before merge-wave finishes, so it reports 2/3 slots when 3 were intended.

**Why it happens:** No dependency check before calculating percentage.

**How to avoid:** Only run compute-delivery-stats after update-scoreboard.cjs merge-wave completes and all votes are recorded. Make it a separate step in qgsd-prompt.js quorum sequence.

**Warning signs:** Delivery stats show 2 votes when all 3 slots actually voted; percentage < expected after retries.

### OBS-01: Telemetry Record Append (in call-quorum-slot.cjs)

```javascript
// Source: bin/call-quorum-slot.cjs pattern (lines 37-53, extended for OBS-01)
async function recordTelemetry(slotName, round, verdict, latencyMs, provider, providerStatus, retryCount, errorType) {
  try {
    const sessionId = process.env.CLAUDE_SESSION_ID || 'session-' + Date.now();
    const record = JSON.stringify({
      ts:              new Date().toISOString(),
      session_id:      sessionId,
      round:           round,
      slot:            slotName,
      verdict:         verdict,        // APPROVE, BLOCK, FLAG
      latency_ms:      latencyMs,      // final successful attempt only
      provider:        provider,        // from providers.json
      provider_status: providerStatus, // available, down, degraded
      retry_count:     retryCount,
      error_type:      errorType,      // null, TIMEOUT, AUTH, CLI_SYNTAX, etc.
    });
    const logPath = path.join(
      findProjectRoot(),
      '.planning',
      `quorum-rounds-${sessionId}.jsonl`
    );
    fs.mkdirSync(path.dirname(logPath), { recursive: true });
    fs.appendFileSync(logPath, record + '\n', 'utf8');
  } catch (err) {
    // Observational — never fails quorum dispatch
    process.stderr.write(`[telemetry] append failed: ${err.message}\n`);
  }
}

// Usage (in main after subprocess completes):
```

### OBS-02: Scoreboard Schema Extension (in update-scoreboard.cjs)

```javascript
// Source: bin/update-scoreboard.cjs pattern (lines 145-150, extended for OBS-02)
function emptyData() {
  return {
    models: { /* existing */ },
    rounds: [ /* existing */ ],
    delivery_stats: {
      total_rounds: 0,
      target_vote_count: 3,
      achieved_by_outcome: {
        '3_votes': { count: 0, pct: 0 },
        '2_votes': { count: 0, pct: 0 },
      }
    },
    slots: {}  // Populated per-slot below
  };
}

// New function: compute delivery stats after merge-wave
function computeDeliveryStats(data) {
  const rounds = data.rounds || [];
  const targetCount = data.delivery_stats?.target_vote_count || 3;

  let counts = {};
  for (const round of rounds) {
    if (!round.votes) continue;
    const voteCount = Object.values(round.votes).filter(v => v && v !== 'UNAVAIL' && v !== '').length;
    const key = voteCount + '_votes';
    counts[key] = (counts[key] || 0) + 1;
  }

  const total = rounds.length;
  data.delivery_stats = {
    total_rounds: total,
    target_vote_count: targetCount,
    achieved_by_outcome: {
      '3_votes': { count: counts['3_votes'] || 0, pct: total ? ((counts['3_votes'] || 0) / total * 100).toFixed(1) : 0 },
      '2_votes': { count: counts['2_votes'] || 0, pct: total ? ((counts['2_votes'] || 0) / total * 100).toFixed(1) : 0 },
    }
  };

  return data;
}
```

### OBS-03: Flakiness Scoring (in update-scoreboard.cjs or new module)

```javascript
// Source: verify-quorum-health.cjs pattern (lines 59-78, extended for flakiness)
function computeFlakiness(slot, rounds, windowSize = 10) {
  // Get the last `windowSize` rounds for this slot
  const relevant = rounds
    .filter(r => r.votes && r.votes[slot] !== undefined)
    .slice(-windowSize);

  if (relevant.length === 0) {
    return { flakiness_score: 0, recent_verdicts: [] };
  }

  // Count failures in window
  const failures = relevant.filter(r =>
    r.votes[slot] === 'UNAVAIL' || r.votes[slot] === 'TIMEOUT'
  ).length;

  const flakiness_score = failures / relevant.length; // 0.0 = reliable, 1.0 = always fails

  return {
    flakiness_score: parseFloat(flakiness_score.toFixed(2)),
    recent_verdicts: relevant.map(r => ({
      round: r.round_num,
      verdict: r.votes[slot]
    }))
  };
}

// Usage in update-scoreboard.cjs after computing delivery stats:
for (const [slotName] of Object.entries(providers)) {
  const flakData = computeFlakiness(slotName, data.rounds);
  if (!data.slots[slotName]) data.slots[slotName] = {};
  data.slots[slotName].flakiness_score = flakData.flakiness_score;
  data.slots[slotName].recent_verdicts = flakData.recent_verdicts;
}
```

### Dispatch List Ordering with Flakiness (in qgsd-prompt.js or quorum.md)

```javascript
// Source: DISP-03 logic extended with OBS-03
function buildDispatchList(scoreboard, riskLevel) {
  const providerSlots = readProvidersJson();  // from v0.24-01
  const fanOut = computeFanOut(riskLevel);     // from DISP-03

  // Sort by (1) flakiness ascending, (2) success rate descending
  const sorted = providerSlots.sort((a, b) => {
    const aFlak = scoreboard.slots?.[a.name]?.flakiness_score ?? 0;
    const bFlak = scoreboard.slots?.[b.name]?.flakiness_score ?? 0;

    if (aFlak !== bFlak) {
      return aFlak - bFlak; // lower flakiness first (more reliable)
    }

    // Tiebreaker: success rate (from scoreboard.slots or verify-quorum-health data)
    const aRate = computeSuccessRate(scoreboard, a.name);
    const bRate = computeSuccessRate(scoreboard, b.name);
    return bRate - aRate; // higher rate first
  });

  return sorted.slice(0, fanOut);
}
```

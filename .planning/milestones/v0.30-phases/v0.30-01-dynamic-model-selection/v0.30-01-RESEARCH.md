# Phase v0.30-01: Dynamic Model Selection - Research

**Researched:** 2026-03-07
**Domain:** Token optimization -- dynamic model routing, thinking budget scaling, compaction, and cost dashboarding
**Confidence:** HIGH

## Summary

This phase implements four token optimization requirements: automatic compaction at clean workflow boundaries (TOKN-01), configurable extended thinking budget scaling per task type (TOKN-02), token usage dashboard (TOKN-03), and task-complexity-aware model routing (TOKN-04). All four capabilities build on existing infrastructure -- `gsd-context-monitor.js` (PostToolUse hook with boundary detection), `bin/budget-tracker.cjs` (profile downgrade chain), `hooks/nf-token-collector.js` (per-slot token recording to `token-usage.jsonl`), and `hooks/config-loader.js` (two-layer config merge). Zero new npm dependencies are required.

The critical risk is **downgrade loop oscillation** (Pitfall 1 from milestone research): the existing budget tracker and new model selector create dual control loops that can fight each other. The unified decision function with cooldown designed into this phase prevents this.

**Primary recommendation:** Build a single `bin/task-classifier.cjs` script that classifies task complexity AND selects model tier in one function, then integrate it into `nf-prompt.js` dispatch and `gsd-context-monitor.js` boundary detection. This avoids the dual-loop pitfall by having one decision authority.

### Pattern 1: Task Complexity Classifier

**What:** A pure function that takes a task envelope (from `nf-prompt.js` context) and returns a complexity classification + recommended model tier + thinking budget.

**When to use:** Called by `nf-prompt.js` during quorum dispatch setup, before building the slot dispatch list.

**Design:**

```javascript
// bin/task-classifier.cjs
'use strict';
const fs = require('fs');
const path = require('path');

// Complexity levels map to model tiers and thinking budgets
const COMPLEXITY_MAP = {
  trivial:  { tier: 'haiku',  thinking_budget: 0,     description: 'formatting, typos, simple renames' },
  simple:   { tier: 'sonnet', thinking_budget: 8000,   description: 'single-file edits, config changes, test fixes' },
  moderate: { tier: 'sonnet', thinking_budget: 16000,  description: 'multi-file changes, refactoring, feature additions' },
  complex:  { tier: 'opus',   thinking_budget: 31999,  description: 'architecture decisions, new subsystems, formal verification' },
};

// Heuristic classifier based on task envelope signals
function classifyTask(taskEnvelope) {
  const signals = {
    fileCount: (taskEnvelope.files || []).length,
    hasArchitecture: /architect|design|system|interface|api/i.test(taskEnvelope.objective || ''),
    hasTest: /test|verify|check|validate/i.test(taskEnvelope.objective || ''),
    riskLevel: taskEnvelope.risk_level || 'medium',
    planTaskCount: taskEnvelope.task_count || 1,
    isExploration: /explore|research|investigate|discover/i.test(taskEnvelope.objective || ''),
    isReview: /review|check|audit|inspect/i.test(taskEnvelope.objective || ''),
  };

  // Classification rules (ordered by specificity)
  if (signals.isExploration) return 'trivial';  // exploration = no thinking needed
  if (signals.isReview && signals.fileCount <= 3) return 'simple';
  if (signals.riskLevel === 'high' || signals.hasArchitecture) return 'complex';
  if (signals.fileCount > 5 || signals.planTaskCount > 3) return 'moderate';
  if (signals.riskLevel === 'low' || signals.riskLevel === 'routine') return 'simple';
  return 'moderate'; // fail-open default
}

function getModelRecommendation(complexity, config) {
  const entry = COMPLEXITY_MAP[complexity] || COMPLEXITY_MAP.moderate;
  // Respect user overrides from config
  const tierOverride = config.model_routing && config.model_routing[complexity];
  return {
    complexity,
    tier: tierOverride || entry.tier,
    thinking_budget: entry.thinking_budget,
    description: entry.description,
  };
}

module.exports = { classifyTask, getModelRecommendation, COMPLEXITY_MAP };
```

**Confidence:** HIGH -- heuristic approach is simple, testable, and extensible. No ML needed.

### Pattern 2: Unified Budget + Model Decision Function

**What:** Single function that coordinates budget status, model selection, and compaction triggering. Prevents the dual-loop oscillation pitfall.

**When to use:** Called from `gsd-context-monitor.js` PostToolUse handler.

**Key design principle:** One decision authority, not two independent loops.

```javascript
// In gsd-context-monitor.js -- unified decision
function computeUnifiedDecision(usedPct, config, scoreboard) {
  const budgetStatus = budgetTracker.computeBudgetStatus(usedPct, config.budget || {}, config.agent_config || {});
  const complexity = taskClassifier.getCurrentTaskComplexity(); // reads from .planning/task-envelope.json
  const recommendation = taskClassifier.getModelRecommendation(complexity, config);

  // Cooldown check: don't downgrade if last downgrade was < 3 quorum rounds ago
  const lastDowngrade = readLastDowngradeTimestamp();
  const cooldownActive = lastDowngrade && (Date.now() - lastDowngrade < COOLDOWN_MS);

  return {
    shouldDowngrade: budgetStatus.shouldDowngrade && !cooldownActive,
    shouldCompact: usedPct >= (config.smart_compact?.context_warn_pct || 60),
    modelRecommendation: recommendation,
    budgetStatus,
  };
}
```

### Pattern 5: Enhanced Compaction Triggering

**What:** Extend `gsd-context-monitor.js` to trigger compaction proactively at clean boundaries when context exceeds configurable threshold (default 60-70%).

**Key constraint:** Must NOT trigger compaction mid-quorum. The `<!-- NF_CACHE_KEY -->` marker or open slot workers indicate active quorum.

```javascript
// Enhanced detectCleanBoundary with compaction auto-trigger
if (usedPct >= compactThreshold && boundary) {
  // Check for active quorum (quorum-in-progress flag)
  const quorumActive = fs.existsSync(path.join(cwd, '.claude', 'quorum-in-progress'));
  if (!quorumActive) {
    compactMessage = formatCompactSuggestion(usedPct, boundary);
    // Optionally: write checkpoint before suggesting compact
    writeCompactionCheckpoint(cwd, { usedPct, boundary, timestamp: new Date().toISOString() });
  }
}
```

### Anti-Patterns to Avoid

- **Two independent control loops:** Budget tracker and model selector MUST share a single decision function. Never let one downgrade while the other is trying to maintain quality.
- **Fixed compaction threshold without boundary awareness:** Lowering the threshold to 50% without requiring a clean boundary causes compaction storms (see Pitfalls).
- **Hardcoded model tier in dispatch:** Model tier must be computed per-task, not set globally. The existing `model_tier_planner`/`model_tier_worker` are static; the new system is dynamic.
- **Nested config objects for new keys:** Shallow merge in `config-loader.js` replaces entire nested objects. Use flat keys: `thinking_budget_exploration`, not `thinking_budget: { exploration: 0 }`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token cost estimation | Real-time API pricing lookup | Hardcoded cost table in config, updated per release | API pricing rarely changes; lookup adds latency and dependency |
| Task complexity ML classifier | Neural network / embedding-based | Keyword + structural heuristics on task envelope | Simple tasks are identifiable by signal count (file count, risk_level, command type) |
| JSONL database | SQLite or indexed storage | Line-by-line `fs.readFileSync` + filter | Token-usage.jsonl grows ~1 line per quorum slot per invocation; even 10K lines parses in <100ms |
| Thinking budget enforcement | Custom token counter | Advisory directive in additionalContext | Claude Code has no API to programmatically cap thinking tokens; instruction-following is the available mechanism |

## Common Pitfalls

### Pitfall 1: Downgrade Loop Oscillation (CRITICAL)

**What goes wrong:** Budget tracker downgrades model tier, which produces lower-quality votes, which triggers more deliberation rounds, which consumes more tokens, which triggers further downgrades. Two independent control loops fight each other.

**Why it happens:** `budget-tracker.cjs` `triggerProfileDowngrade()` writes to `.planning/config.json` without checking whether the downgrade will help. No feedback loop measures quality impact.

**How to avoid:**
1. Implement a cooldown period: minimum 3 quorum rounds (or 5 minutes) between downgrades
2. Add a `downgrade_history` array with timestamps to `.planning/config.json`
3. Detect oscillation: 3+ alternating upgrades/downgrades within 10 rounds = freeze at current level
4. Before downgrading, check scoreboard consensus rate. If already below 70%, inject compaction suggestion instead of downgrading further.

**Warning signs:** Scoreboard shows declining consensus rates correlating with profile changes; rapid alternation of `budget_downgrade` and `quality_warning` conformance events.

### Pitfall 2: Compaction Storm from Aggressive Triggering

**What goes wrong:** Lowering compaction threshold to 60% triggers frequent compactions. Each compaction forces re-reading files, consuming tokens, pushing toward next compaction.

**How to avoid:** Only trigger at clean boundaries (`phase_complete`, `verification_done`, `commit`). Never trigger mid-quorum. Use workflow-aware thresholds, not a fixed percentage.

**Warning signs:** More than 2 compactions per phase execution; compactions closer than 5 minutes apart.

### Pitfall 3: Quorum Evidence Lost to Compaction

**What goes wrong:** Earlier compaction fires between slot dispatch and consensus synthesis. Stop hook can't find quorum evidence in transcript. Response is blocked.

**How to avoid:**
1. Set a `quorum-in-progress` flag file when dispatch starts, clear on decision delivery
2. `gsd-context-monitor.js` checks this flag before suggesting compaction
3. Checkpoint quorum evidence to `.claude/quorum-evidence.json` as fallback for Stop hook

**Warning signs:** Stop hook blocking responses immediately after compaction.

### Pitfall 4: Thinking Budget Directive Ignored

**What goes wrong:** Claude may not reliably honor advisory thinking budget instructions, especially when task complexity changes mid-conversation.

**How to avoid:** Reinject the directive at every UserPromptSubmit, not just the first. Include explicit numeric cap. Accept that enforcement is best-effort.

### Pitfall 5: Hook Execution Order with New PostToolUse Logic

**What goes wrong:** All new logic (model selection feedback, compaction trigger enhancement) goes into `gsd-context-monitor.js` PostToolUse. If any branch throws, entire hook fails.

**How to avoid:** Each new branch wrapped in its own try/catch within the existing hook. Never add a separate PostToolUse hook -- consolidate into `gsd-context-monitor.js`.

## Open Questions

1. **Thinking budget enforcement reliability**
   - What we know: Claude Code has no API to programmatically set thinking token budget. The `--model` flag sets the model but not thinking limits.
   - What's unclear: How reliably Claude honors "use maximum N thinking tokens" as an instruction
   - Recommendation: Implement as advisory directive. Monitor actual thinking token usage in JSONL. If compliance is poor, explore `CLAUDE_THINKING_BUDGET` env var (may exist but undocumented).

2. **Cost-per-token rates for non-Anthropic models**
   - What we know: Anthropic model pricing is published. ccr-routed models (DeepSeek, Qwen, Kimi, etc.) go through third-party providers with different pricing.
   - What's unclear: Exact $/1M token rates for AkashML, Together.xyz, Fireworks models
   - Recommendation: Use approximate rates from provider websites. Mark dashboard costs as "estimated". Allow user override via config.

3. **Task envelope availability during quorum dispatch**
   - What we know: `task_envelope_enabled: true` writes task-envelope.json, but it's written by the planning workflow, not always present during ad-hoc `/nf:quick` commands
   - What's unclear: Whether task envelope exists for all command types that trigger quorum
   - Recommendation: Classifier falls back to `moderate` when no task envelope is available. Parse prompt text as secondary signal source.

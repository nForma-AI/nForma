   ```
   | Checkpoints | Independence | Pattern | Execution |
   |-------------|-------------|---------|-----------|
   | None | Yes, --parallel | D (parallel) | Parallel worktree executors per task, merge orchestration |
   | None | No or no --parallel | A (autonomous) | Single subagent: full plan |
   | Verify-only | - | B (segmented) | Segments between checkpoints |

2. Add a new step `detect_parallel_tasks` (after `parse_segments`, before dispatch):
   - Only runs when Pattern D is selected (no checkpoints AND --parallel flag detected)
   - Parse all `<task>` elements and extract `<files>` lists
   - Apply independence detection:
     a. Tasks with overlapping files must run sequentially
     b. Tasks with EMPTY `<files>` lists should be treated as non-overlapping and parallelizable (Q-R3.6-1: documentation-only or verification tasks with no file output can safely run in parallel)
     c. SERIAL_FILES list forces sequential execution. Include: STATE.md, ROADMAP.md, REQUIREMENTS.md, package.json, package-lock.json, .gitignore, .claude/settings.json, and any config files matching `*.config.{js,ts,cjs,mjs}` (Q-R3.6-6: expanded serial files list to prevent parallel execution of shared config)
     d. Tasks with install sync (`install.js --claude --global`) force sequential
   - Output: `parallel_groups` (tasks that can run simultaneously) and `sequential_tasks` (must run in order)
   - If no parallel groups found (all tasks overlap), fall back to Pattern A

3. Add step `parallel_dispatch` (Pattern D execution):
   - For each task in parallel_groups:
     ```
     Spawn Task(subagent_type="nf-worktree-executor",
                description="Execute task {N}: {task_name}",
                prompt="Execute task {N} from plan at {plan_path}.
                        IMPORTANT: Skip state_updates and final_commit steps -- the orchestrator handles those after merge. (Q-R3.6-6)
                        Report completion with a fenced JSON block at the end of your output:
                        ```json
                        {\"worktree_branch\": \"<branch>\", \"task_number\": <N>, \"status\": \"complete|failed\", \"files_modified\": [...]}
                        ```",
                maxTurns=50)
     ```
   - **Wall-clock timeout** (Q-R3.6-5): Set a PARALLEL_BATCH_TIMEOUT of 15 minutes for the entire parallel batch. If any executor has not completed within this window, treat it as failed and proceed with completed executors only.
   - **maxTurns** (Q-R3.6-5): Set maxTurns=50 for worktree executors (prevents runaway execution). This is the same default used for nf-executor tasks.
   - Wait for ALL parallel tasks to complete (or timeout)
   - **Partial failure handling** (Q-R3.6-5): When one or more executors fail but others succeed:
     a. Log failed executors to stderr with task numbers and error details
     b. Proceed with merging ONLY the successful branches (do not block on failures)
     c. Record failed tasks in the SUMMARY.md with status "failed-parallel" and the error
     d. After merge of successful branches, report which tasks need re-execution (fall back to Pattern A for those specific tasks)
   - Collect branch names from each executor's output by parsing the fenced JSON block (Q-R3.6-4: reliable parsing). Parse using regex: extract content between ```json and ``` markers, then JSON.parse. Do NOT use grep for free-form worktree_branch text.
   - Run sequential_tasks in order (same as Pattern A, in main worktree)
   - After all tasks complete:
     a. Run `node bin/worktree-merge.cjs ensure-clean` to verify pre-merge state (Q-R3.6-2)
     b. Run `node bin/worktree-merge.cjs merge <branch1> <branch2> ...` to merge parallel branches
     c. Check merge results -- if any conflict, report to user and fall back to manual resolution
     d. Run `node bin/worktree-merge.cjs verify` to validate merged state (includes automatic rollback on failure per Q-R3.6-3)
     e. Run `node bin/worktree-merge.cjs cleanup <branch1> <branch2> ...` to clean up branches
     f. Aggregate execution progress from worktrees: `node bin/execution-progress.cjs aggregate-parallel`
   - Create SUMMARY.md aggregating all task results (parallel + sequential)
   - Commit metadata

4. Add a `--parallel` flag detection note at the top of the workflow:
   ```
   If the user invokes /nf:execute-phase with --parallel flag, Pattern D routing is enabled.
   Without --parallel, existing Pattern A/B/C behavior is unchanged.
   ```

5. Add safety notes:
   - If worktree creation fails (Claude Code error), fall back to Pattern A with stderr warning
   - maxTurns=50 on worktree executors to prevent runaway execution (Q-R3.6-5)
   - Worktrees branch from origin/main -- ensure current changes are committed before parallel dispatch
   - After merge, verify no orphaned worktrees: `git worktree list` should show only main worktree
   - On partial executor failure: merge successful branches, report failures, suggest Pattern A re-run for failed tasks (Q-R3.6-5)
   - Pre-merge state is validated by ensureCleanState; post-merge failures trigger automatic rollback (Q-R3.6-2, Q-R3.6-3)

CRITICAL: The --parallel flag is opt-in. Without it, execute-plan.md behaves exactly as before. This is a non-breaking additive change.

1. Add `aggregateParallelProgress(mainCwd, worktreeProgressFiles)` function to `bin/execution-progress.cjs`:
   - Takes the main worktree cwd and an array of `{ worktreePath, progressData }` objects
   - Each progressData is the JSON from a worktree's `.planning/execution-progress.json`
   - Merges task statuses from all worktrees into the main progress file:
     a. For each task in each worktree progress, find matching task by number in main progress
     b. Update status, commit_hash, completed_at from the worktree's version
     c. If task is complete in worktree, mark complete in main
   - Writes merged progress to main worktree's execution-progress.json
   - Returns the merged progress object
   - Fail-open: if any worktree progress file is missing or malformed, skip it with stderr warning

2. Add CLI command `aggregate-parallel` to the switch statement:
   ```
   case 'aggregate-parallel': {
     // Reads worktree paths from args, loads their progress files, aggregates
     const worktreePaths = args.slice(1);
     const worktreeData = worktreePaths.map(p => {
       try {
         const data = JSON.parse(fs.readFileSync(path.join(p, PROGRESS_FILE), 'utf8'));
         return { worktreePath: p, progressData: data };
       } catch (_) { return null; }
     }).filter(Boolean);
     const result = aggregateParallelProgress(cwd, worktreeData);
     process.stdout.write(JSON.stringify(result) + '\n');
     break;
   }
   ```

3. Export `aggregateParallelProgress` in module.exports.

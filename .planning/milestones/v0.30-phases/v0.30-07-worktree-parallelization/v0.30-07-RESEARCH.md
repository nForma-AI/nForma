# Phase v0.30-07: Worktree Parallelization - Research

**Researched:** 2026-03-08 (updated)
**Domain:** Git worktree isolation, parallel subagent execution, merge orchestration
**Confidence:** HIGH

## Summary

Phase v0.30-07 enables independent plan tasks to execute simultaneously in isolated git worktrees instead of sequentially. Claude Code natively supports `isolation: worktree` in subagent frontmatter -- when set, each subagent spawned with that agent type automatically gets its own git worktree at `<repo>/.claude/worktrees/<name>/`. The worktree branches from the default remote branch and is automatically cleaned up when the subagent finishes without changes.

The implementation requires three components: (1) a worktree-aware executor agent definition (`nf-worktree-executor.md`) with `isolation: worktree` in frontmatter, (2) a parallel dispatch orchestrator in `execute-plan.md` that identifies independent tasks and spawns them as parallel `Task()` calls, and (3) merge orchestration logic that collects results from parallel worktree executors and merges their branches back into the working branch. The existing `execute-plan.md` workflow already has Pattern A (single subagent), Pattern B (segmented), and Pattern C (main context) -- this phase adds Pattern D (parallel worktree) for plans with independent tasks.

The zero-dependency constraint continues. All implementation uses Node.js built-ins, git CLI commands, and Claude Code's native `isolation: worktree` feature. The `execution-progress.cjs` module needs extension to track parallel task status across multiple worktrees. The formal invariant `EventuallyTerminates` must hold for each parallel executor independently -- the iteration cap and stuck detection already in `execution-progress.cjs` provide this guarantee per-worktree.

**Primary recommendation:** Create `agents/nf-worktree-executor.md` with `isolation: worktree` frontmatter, add a `bin/worktree-merge.cjs` module for merge orchestration, extend `execute-plan.md` with Pattern D parallel dispatch, and extend `execution-progress.cjs` to track per-worktree task state.

### Pattern 1: Worktree Executor Agent (PARA-01)

**What:** A specialized executor agent definition that adds `isolation: worktree` to the existing `nf-executor` pattern. Each spawned instance gets its own git worktree automatically via Claude Code's native feature.

**When to use:** Every parallel task dispatch in Pattern D.

**Design:** The `nf-worktree-executor.md` agent is nearly identical to `nf-executor.md` but with `isolation: worktree` in frontmatter. It inherits all executor behavior (deviation rules, commit protocol, verification) but operates in an isolated worktree.

**Key behaviors from Claude Code official docs (verified 2026-03-08):**
- Worktree created at `<repo>/.claude/worktrees/<auto-generated-name>/`
- Branches from default remote branch (typically `origin/main`)
- Branch is named `worktree-<name>`
- Automatic cleanup when subagent finishes without changes
- If subagent makes commits, worktree and branch persist for merge
- Subagents cannot spawn other subagents (confirmed in official docs)

**Mutable state scoping:** All mutable state files that `execution-progress.cjs` writes must be scoped to the worktree's `.planning/` directory. Since each worktree is a full repo copy, the `.planning/execution-progress.json` file is naturally isolated -- each worktree has its own copy.

**Available frontmatter fields (verified from official docs):**

| Field | Value | Purpose |
|-------|-------|---------|
| `isolation` | `worktree` | Git worktree isolation |
| `model` | `inherit` | Use same model as parent |
| `maxTurns` | number | Cap agentic turns to prevent runaway |
| `permissionMode` | `bypassPermissions` or `acceptEdits` | Executor needs write access |
| `tools` | list | Restrict tool access |
| `background` | boolean | Run as background task (NOT recommended for executors -- see Alternatives Considered) |

**Example frontmatter:**
```yaml
---
name: nf-worktree-executor
description: Executes plan tasks in an isolated git worktree for parallel execution. Spawned by execute-plan Pattern D.
tools: Read, Write, Edit, Bash, Grep, Glob
isolation: worktree
color: green
---
```

**Critical constraint:** The worktree branches from the default remote branch, NOT from the current working branch. This means the executor starts from a clean base. The merge orchestrator must handle integrating changes from a clean-base worktree back into the current working branch.

### Pattern 2: Parallel Dispatch in execute-plan.md (PARA-02)

**What:** A new Pattern D in `execute-plan.md` that identifies independent tasks within a plan and dispatches them simultaneously as parallel `Task()` calls, each using `nf-worktree-executor`.

**When to use:** Plans where multiple tasks have no `depends_on` relationships and modify non-overlapping files.

**Design:** Pattern D adds to the existing Pattern A/B/C routing:

```
| Checkpoints | Independence | Pattern | Execution |
|-------------|-------------|---------|-----------|
| None, all independent | Yes | D (parallel) | Parallel worktree executors per task, merge orchestration |
| Any checkpoints | - | A/B/C | Existing patterns unchanged |
| None, dependencies exist | - | A | Single subagent sequential |
```

**Independence detection:** Two tasks are independent if:
1. Their `<files>` lists do not overlap
2. Neither depends on output from the other (no cross-task references in `<action>`)
3. They are in the same wave (wave system already captures dependency ordering)

**Dispatch:**
```
For each independent task group:
  Spawn Task(subagent_type="nf-worktree-executor", description="Execute task N: {task_name}")
  with prompt containing: plan path, specific task to execute, deviation rules
Wait for all parallel Task() calls to complete
Collect results (commit hashes, branch names)
Run merge orchestration (worktree-merge.cjs)
Create SUMMARY.md aggregating all task results
```

**Parallelism cap:** Claude Code does not document an explicit concurrent subagent limit in the frontmatter reference. Web sources suggest a cap of 10 concurrent tasks but this is LOW confidence. For most plans with 2-4 tasks, this is not a concern. If a plan has many independent tasks, batch them conservatively (4-5 at a time).

### Pattern 3: Merge Orchestration (PARA-02)

**What:** After parallel worktree executors complete, their branches must be merged back into the working branch. The `worktree-merge.cjs` module handles this.

**When to use:** After all parallel Task() calls in Pattern D return.

**Design:** The merge orchestrator:
1. Collects branch names from all completed worktree executors (branches named `worktree-<name>`)
2. For each branch, attempts `git merge --no-ff <branch>` into the current branch
3. If merge succeeds: continue to next branch
4. If merge conflicts: attempt automatic resolution (accept both changes for non-overlapping hunks), or flag for manual resolution
5. After all merges: verify combined state (run tests, lint)
6. Clean up worktree branches: `git branch -d <branch>` for merged branches

**Merge order matters:** If tasks are truly independent (non-overlapping files), merge order does not affect the result. The independence check in Pattern 2 ensures this.

**Fallback:** If merge fails (unexpected file conflicts despite independence check), fall back to sequential re-execution of conflicting tasks in the main worktree.

### Anti-Patterns to Avoid

- **Parallelizing tasks with shared files:** If two tasks modify the same file, they MUST run sequentially. The independence check prevents this, but the planner should be aware.
- **Relying on worktree branch base:** Worktrees branch from `origin/main`, not the current branch. Any uncommitted or branch-local changes in the main worktree will NOT be visible in parallel worktrees. The merge orchestrator must account for this.
- **Skipping merge verification:** After merging all branches, the combined state must be verified (tests, lint). Individual task verification is necessary but not sufficient.
- **Manual worktree management:** Do NOT use `git worktree add/remove` manually. Claude Code's `isolation: worktree` handles the full lifecycle. Manual management creates orphaned worktrees.
- **Breaking fail-open:** If worktree creation fails (disk space, git lock), fall back to sequential Pattern A execution. Never block on parallelization failures.
- **Using `background: true`:** Background subagents auto-deny unpre-approved permissions and cannot ask clarifying questions. Foreground parallel Task() calls are more appropriate for executors that may need to interact with tools.
- **Forgetting `node_modules` in worktrees:** Fresh worktrees do NOT share `node_modules/`. However, for this project (nForma), there are no build steps required -- all scripts are standalone CJS files that use Node.js built-ins. This is a non-issue for nForma but would be critical for projects with build dependencies.
- **Spawning nested subagents:** Subagents cannot spawn other subagents (confirmed in official docs). The worktree executor must execute tasks directly, not delegate.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Git worktree creation/cleanup | Custom `git worktree add/remove` wrapper | Claude Code `isolation: worktree` frontmatter | Native feature handles naming, branching, cleanup automatically |
| Task independence detection | Complex dependency graph analysis | File overlap check on `<files>` elements + wave system | Plans already encode dependencies in `wave` and `depends_on`; file overlap is the remaining check |
| Per-worktree execution state | New state tracking system | `execution-progress.cjs` (naturally isolated per worktree) | Each worktree has its own `.planning/` directory |
| Merge conflict resolution | Custom three-way merge logic | `git merge` with `--no-ff` | Git's merge engine handles non-overlapping changes correctly; true conflicts indicate tasks were not independent |
| Subagent result collection | Custom IPC mechanism | Task() return values | Claude Code Task() tool returns the subagent's output as a string |

**Key insight:** Claude Code's native `isolation: worktree` eliminates 80% of the complexity. The remaining work is dispatch logic (which tasks to parallelize) and merge orchestration (collecting results). Both are straightforward when tasks are truly independent.

## Common Pitfalls

### Pitfall 1: Worktree Branches from origin/main, Not Current Branch
**What goes wrong:** Parallel worktree executors start from a clean `origin/main` base, not from the current branch with all prior phase commits. If prior phases created files or modified code that the current tasks depend on, the worktree executor will not have those changes.
**Why it happens:** Claude Code's worktree feature always branches from the default remote branch for a clean starting point. The official docs state: "Worktrees are created at `<repo>/.claude/worktrees/<name>` and branch from the default remote branch."
**How to avoid:** Before dispatching parallel tasks, ensure all prior phase changes are pushed to the remote branch (or at minimum committed to the branch that `origin/main` tracks). Alternatively, the merge orchestrator could cherry-pick the current branch's commits into each worktree before task execution begins.
**Warning signs:** Worktree executors failing with "file not found" or "module not found" errors for files created in prior phases.

### Pitfall 2: Merge Conflicts Despite Independence Check
**What goes wrong:** Two tasks that appeared independent produce merge conflicts because they both modify a shared transitive dependency (e.g., both add entries to `package.json` scripts or both modify `STATE.md`).
**Why it happens:** Independence check only looks at `<files>` elements in the plan. Implicit shared files (STATE.md updates, package.json, config files) are not listed.
**How to avoid:** (1) Exclude metadata files from independence check (STATE.md, ROADMAP.md, REQUIREMENTS.md, package.json are always serial). (2) Move metadata updates to the merge orchestrator phase, not the per-task executor. (3) Accept that some merge conflicts are acceptable and can be auto-resolved (both tasks adding different lines to the same file).
**Warning signs:** Merge failures on files not listed in any task's `<files>`.

### Pitfall 3: Parallel Executors Blocking Each Other
**What goes wrong:** One worktree executor stalls (e.g., waiting for user input at a checkpoint, or hitting an API rate limit) and blocks the merge orchestration from proceeding.
**Why it happens:** Pattern D waits for ALL parallel tasks before merging.
**How to avoid:** (1) Pattern D should only be used for checkpoint-free plans (all `type="auto"` tasks). (2) Each executor has its own iteration cap from `execution-progress.cjs` (5 iterations max). (3) Set a wall-clock timeout for parallel dispatch (e.g., 10 minutes). (4) If any executor exceeds timeout, mark it as failed and proceed with successful ones. (5) Consider using `maxTurns` frontmatter field to cap executor turns.
**Warning signs:** Merge orchestration waiting indefinitely; `EventuallyTerminates` violation for the overall plan.

### Pitfall 4: Install Sync in Parallel Worktrees
**What goes wrong:** If a parallel task needs to run `node bin/install.js --claude --global`, multiple worktrees could race on writing to `~/.claude/hooks/` simultaneously.
**Why it happens:** The global install directory is shared across all worktrees.
**How to avoid:** Install sync tasks should NEVER be parallelized. Any task containing install sync must run in the main worktree sequentially. The independence check should flag tasks with install sync as non-parallelizable.
**Warning signs:** Corrupted hook files in `~/.claude/hooks/`; race condition errors during `install.js`.

### Pitfall 5: additionalContext Contention Across Parallel Executors
**What goes wrong:** Each parallel executor receives the same `additionalContext` injection from hooks, which may reference the main worktree's state rather than the worktree-specific state.
**Why it happens:** Hooks run in the context of the spawning session, not the worktree executor's session.
**How to avoid:** The `isolation: worktree` feature gives each executor its own working directory. Hook state that reads from `cwd` will naturally read the worktree's files. However, hooks that read from absolute paths (like `~/.claude/nf.json`) will share state, which is expected and correct for global config.
**Warning signs:** Parallel executors receiving stale or incorrect context injections.

### Pitfall 6: Worktree Cleanup With Changes
**What goes wrong:** If a worktree executor makes commits but the merge fails, the worktree and branch persist. Multiple failed runs accumulate orphaned worktrees in `.claude/worktrees/`.
**Why it happens:** Claude Code only auto-cleans worktrees with NO changes. With changes or commits, cleanup is manual or prompted.
**How to avoid:** The merge orchestrator should explicitly clean up worktree branches after merge (success or failure) using `git branch -d <branch>`. Add `.claude/worktrees/` to `.gitignore`.
**Warning signs:** Growing `.claude/worktrees/` directory; `git worktree list` showing stale entries.

### worktree-merge.cjs Module Structure
```javascript
// bin/worktree-merge.cjs
'use strict';

const { execFileSync } = require('child_process');
const path = require('path');

const MERGE_TIMEOUT_MS = 30000; // 30s per merge

function runGit(cwd, args) {
  return execFileSync('git', args, {
    cwd,
    timeout: MERGE_TIMEOUT_MS,
    encoding: 'utf8',
  });
}

function mergeBranches(cwd, branches) {
  const results = [];

  for (const branch of branches) {
    try {
      runGit(cwd, ['merge', '--no-ff', branch, '-m', `merge: parallel task from ${branch}`]);
      results.push({ branch, status: 'merged' });

      // Clean up branch after successful merge
      try {
        runGit(cwd, ['branch', '-d', branch]);
      } catch (_) { /* branch cleanup is best-effort */ }
    } catch (err) {
      const stderr = err.stderr ? err.stderr.toString() : '';
      if (stderr.includes('CONFLICT')) {
        // Abort and mark conflict -- do not auto-resolve
        try { runGit(cwd, ['merge', '--abort']); } catch (_) { /* best-effort */ }
        results.push({ branch, status: 'conflict', error: stderr.slice(0, 500) });
      } else {
        results.push({ branch, status: 'failed', error: stderr.slice(0, 500) });
      }
    }
  }

  return results;
}

function verifyMergedState(cwd) {
  try {
    execFileSync('npm', ['test'], { cwd, timeout: 120000, encoding: 'utf8' });
    return { pass: true };
  } catch (err) {
    return { pass: false, error: err.stderr ? err.stderr.toString().slice(0, 500) : 'test failure' };
  }
}

module.exports = { mergeBranches, verifyMergedState, MERGE_TIMEOUT_MS };

// CLI interface
if (require.main === module) {
  try {
    const args = process.argv.slice(2);
    const command = args[0];
    const cwd = process.cwd();

    switch (command) {
      case 'merge': {
        const branches = args.slice(1);
        const result = mergeBranches(cwd, branches);
        process.stdout.write(JSON.stringify(result) + '\n');
        break;
      }
      case 'verify': {
        const result = verifyMergedState(cwd);
        process.stdout.write(JSON.stringify(result) + '\n');
        break;
      }
      default:
        process.stderr.write('Usage: worktree-merge.cjs <merge|verify> [branches...]\n');
        process.exit(0);
    }
  } catch (e) {
    process.stderr.write('[worktree-merge] Error: ' + e.message + '\n');
    process.exit(0); // Fail open
  }
}
```

### Independence Detection Logic
```javascript
// Logic for determining which tasks can run in parallel
function detectIndependentTasks(tasks) {
  // tasks: array of { number, name, files: string[] }

  // Metadata files that should never be parallelized
  const SERIAL_FILES = [
    'STATE.md', 'ROADMAP.md', 'REQUIREMENTS.md',
    'package.json', 'package-lock.json',
  ];

  // Check if any task modifies serial files
  const serialTasks = tasks.filter(t =>
    t.files.some(f => SERIAL_FILES.some(sf => f.endsWith(sf)))
  );

  // Tasks modifying serial files must run sequentially
  const parallelCandidates = tasks.filter(t => !serialTasks.includes(t));

  // Among candidates, group by file overlap
  const independent = [];
  const usedFiles = new Set();

  for (const task of parallelCandidates) {
    const hasOverlap = task.files.some(f => usedFiles.has(f));
    if (!hasOverlap) {
      independent.push(task.number);
      task.files.forEach(f => usedFiles.add(f));
    } else {
      // This task overlaps with a prior independent task; must be sequential
      serialTasks.push(task);
    }
  }

  return {
    parallel: independent,   // Can run simultaneously
    sequential: serialTasks.map(t => t.number), // Must run in order
  };
}
```

## Open Questions

1. **How does the worktree executor get prior phase changes?**
   - What we know: Claude Code worktrees branch from `origin/main` by default, not the current branch. Official docs confirm: "Worktrees are created at `<repo>/.claude/worktrees/<name>` and branch from the default remote branch."
   - What's unclear: Whether Claude Code offers a way to specify the base branch for worktree creation, or if we must push all changes to origin/main first. The official docs do not document a `baseBranch` frontmatter field.
   - Recommendation: Before dispatching parallel tasks, ensure the current branch is pushed to origin. Alternatively, the merge orchestrator could cherry-pick the current branch's commits into each worktree before task execution. This needs validation during implementation.

2. **Should independence detection be in the planner or the executor?**
   - What we know: The planner already assigns `wave` numbers and `depends_on` arrays. The executor already parses `<files>` elements.
   - What's unclear: Whether the planner should explicitly mark tasks as `parallel: true` in the plan, or the executor should infer parallelizability.
   - Recommendation: The planner marks tasks as parallelizable in the plan frontmatter (e.g., `parallel_groups: [[1,2], [3,4]]`). The executor trusts this annotation. This keeps independence logic in the planning phase where it belongs, and the executor simply dispatches.

3. **What happens to execution-progress.cjs in parallel mode?**
   - What we know: Each worktree has its own `.planning/execution-progress.json`. After merging, the main worktree needs a combined progress state.
   - What's unclear: How to aggregate per-worktree progress into a single coherent state after merge.
   - Recommendation: The merge orchestrator reads each worktree's progress file before cleanup, aggregates task statuses, and writes a combined progress file in the main worktree. Alternatively, the orchestrator tracks progress externally (in memory) and writes a single progress file after all merges complete.

4. **Should we support partial parallelization?**
   - What we know: Some plans may have tasks 1-2 independent but task 3 depending on both.
   - What's unclear: Whether to support mixed parallel+sequential within a single plan dispatch.
   - Recommendation: Yes, support partial parallelization. Dispatch independent tasks in parallel, wait for completion, merge results, then dispatch dependent tasks sequentially. This maximizes parallelism while respecting dependencies.

5. **What is the actual concurrent subagent limit?**
   - What we know: Official docs do not specify a hard limit in the frontmatter fields table. Web sources suggest 10.
   - What's unclear: Whether there is a hard cap enforced by the runtime.
   - Recommendation: Batch conservatively at 4-5 concurrent tasks. Validate actual limits during implementation with a test dispatch of multiple worktree subagents.

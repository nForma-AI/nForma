# Phase v0.30-03: Memory Persistence - Research

**Researched:** 2026-03-07
**Domain:** Structured memory persistence across compaction and sessions for decisions, errors, and quorum rationale
**Confidence:** HIGH

## Summary

This phase implements four memory persistence requirements (MEMP-01 through MEMP-04) that together ensure accumulated knowledge survives both context compaction events and session boundaries. Currently, nForma stores decisions in STATE.md's "Accumulated Context" section (~3.7KB, manually maintained) and MEMORY.md (Claude's auto-memory, ~4KB, user-controlled). Neither is structured for machine consumption, neither survives compaction reliably (only a summary is injected via nf-precompact.js), and error resolution patterns are lost entirely between sessions.

The solution is a set of JSONL-based memory stores under `.planning/memory/` that record three categories of persistent knowledge: (1) decisions and rejected approaches, (2) error resolution patterns (symptom/root-cause/fix triples), and (3) quorum debate rationale. These stores are written at natural boundaries (task completion, error resolution, quorum conclusion) and read by two injection points: `nf-session-start.js` for proactive session reminders (MEMP-02) and `nf-precompact.js` for compaction survival (MEMP-01). A single `bin/memory-store.cjs` script provides the CRUD interface. The quorum debate pattern already exists in `.planning/quorum/debates/` as markdown files -- MEMP-04 extends this by extracting structured rationale into a JSONL store that is cheaper to query than parsing full debate markdown.

**Primary recommendation:** Build `bin/memory-store.cjs` with append/query/prune commands operating on three JSONL files under `.planning/memory/`. Extend `nf-session-start.js` to inject a session reminder from the last 3 decisions and any active blockers. Extend `nf-precompact.js` to inject a compact memory summary alongside the existing execution progress injection. The total additionalContext budget consumed by memory injection MUST stay under 1200 characters to leave room for execution progress (3200 chars) and STATE.md position (~800 chars).

### Pattern 1: JSONL Memory Store

**What:** Three append-only JSONL files, each storing a different category of persistent memory. Each line is a self-contained JSON object with a consistent envelope schema.

**When to use:** All memory writes use `bin/memory-store.cjs` commands via the workflow (not direct file writes from hooks).

**Envelope schema:**

```javascript
// .planning/memory/decisions.jsonl — one entry per line
{
  "type": "decision",                    // discriminator
  "ts": "2026-03-07T14:30:00Z",         // ISO timestamp
  "phase": "v0.30-03",                  // phase context
  "summary": "Use JSONL not SQLite for memory stores",  // one-line summary
  "detail": "Chose JSONL because...",    // optional longer explanation
  "rejected": ["SQLite", "LevelDB"],    // what was NOT chosen
  "tags": ["architecture", "storage"],   // searchable tags
  "source": "quorum"                    // "quorum" | "user" | "research" | "error"
}

// .planning/memory/errors.jsonl — one entry per line
{
  "type": "error_resolution",
  "ts": "2026-03-07T15:00:00Z",
  "symptom": "EACCES permission denied on ~/.claude/hooks/nf-stop.js",
  "root_cause": "install.js did not set executable bit after copy",
  "fix": "Added chmod +x in installHook() after fs.copyFileSync",
  "files": ["bin/install.js"],           // affected files
  "tags": ["install", "permissions"],
  "confidence": "high"                   // "high" | "medium" | "low"
}

// .planning/memory/quorum-decisions.jsonl — one entry per line
{
  "type": "quorum_decision",
  "ts": "2026-03-07T16:00:00Z",
  "question": "Option A vs Option B for slot worker optimization",
  "chosen": "Option A -- thin passthrough",
  "rationale": "~85% overhead reduction, 30min effort, preserves Task-based parallelism",
  "rejected_options": ["Option B -- eliminate Claude subagent entirely"],
  "consensus": "APPROVE",
  "rounds": 1,
  "debate_file": ".planning/quorum/debates/2026-03-02-option-a-vs-b-slot-worker-optimization.md",
  "phase": "v0.24"
}
```

### Pattern 2: Memory Store CLI Interface

**What:** A single `bin/memory-store.cjs` with subcommands for each memory operation.

**Commands:**

```bash
# Append a decision
node bin/memory-store.cjs append-decision \
  --summary "Use JSONL for memory" \
  --detail "Chose JSONL because zero deps and proven pattern" \
  --rejected "SQLite,LevelDB" \
  --tags "architecture,storage" \
  --source "quorum" \
  --phase "v0.30-03"

# Append an error resolution
node bin/memory-store.cjs append-error \
  --symptom "EACCES permission denied" \
  --root-cause "Missing chmod after copy" \
  --fix "Added chmod +x in installHook()" \
  --files "bin/install.js" \
  --tags "install,permissions"

# Append a quorum decision
node bin/memory-store.cjs append-quorum \
  --question "Option A vs B" \
  --chosen "Option A" \
  --rationale "85% overhead reduction" \
  --rejected "Option B" \
  --consensus "APPROVE" \
  --rounds 1 \
  --debate-file ".planning/quorum/debates/2026-03-02-option-a-vs-b.md"

# Query recent decisions (last N, default 5)
node bin/memory-store.cjs query-decisions --last 3

# Query errors by symptom (grep match)
node bin/memory-store.cjs query-errors --symptom "EACCES"

# Query quorum decisions (last N)
node bin/memory-store.cjs query-quorum --last 3

# Get session reminder (combined view for SessionStart injection)
node bin/memory-store.cjs session-reminder

# Prune entries older than N days (default 90)
node bin/memory-store.cjs prune --days 90
```

### Anti-Patterns to Avoid

- **Storing memory in STATE.md directly:** STATE.md is already 3.7KB and is human-curated. Embedding machine-written memory entries creates maintenance burden and parsing fragility. Keep STATE.md for high-level project position, JSONL for machine memory.
- **Modifying MEMORY.md programmatically:** MEMORY.md is user-controlled auto-memory per REQUIREMENTS.md out-of-scope ("Automatic CLAUDE.md updates from memory" is explicitly excluded, and MEMORY.md has the same constraint). Write to separate JSONL files.
- **Unbounded memory accumulation:** REQUIREMENTS.md explicitly marks "Full conversation history persistence" as out-of-scope due to unbounded growth. JSONL files MUST have a prune mechanism (default 90 days).
- **Embedding-based semantic search:** Explicitly out-of-scope per REQUIREMENTS.md ("Over-engineering for CLI plugin; file-based grep over JSONL sufficient").
- **Writing memory from hooks directly:** Hooks must remain fast (< 100ms). Memory writes happen via workflow calls to `bin/memory-store.cjs`, not inline in hooks. Hooks only READ memory for injection.
- **Injecting full error details at session start:** Keep it to counts and one-liner summaries. Full error details are retrieved on-demand via `query-errors`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Append-only log files | Custom log rotator | JSONL with appendFileSync + prune command | Proven in conformance-events.jsonl, atomic for single-writer |
| Path resolution | Hardcoded paths | planning-paths.cjs `resolve()` / `resolveWithFallback()` | Handles canonical vs legacy layout automatically |
| Session context injection | New hook | Extend existing nf-session-start.js _contextPieces array | Hook already collects multiple context pieces |
| Compaction context injection | New hook | Extend existing nf-precompact.js lines array | Hook already builds multi-section continuation context |
| Deduplication | Hash-based dedup library | Check last N entries for matching summary/symptom before append | Simple, no dependencies, adequate for append-mostly workload |

**Key insight:** The memory system is an append-mostly log with two read patterns (session start reads last-N, compaction reads last-N, error query does grep). This is exactly the same pattern as conformance-events.jsonl and token-usage.jsonl. Do not over-engineer with databases or search indices.

## Common Pitfalls

### Pitfall 1: additionalContext Budget Exhaustion

**What goes wrong:** Memory injection at compaction consumes too many characters, crowding out execution progress (3200 chars) and STATE.md position that are critical for correct agent resumption.

**Why it happens:** Memory stores grow over time. Injecting all decisions and errors into compaction context exceeds the practical ~5000 character ceiling.

**How to avoid:**
- Memory compaction injection is capped at 1200 characters (hard limit in code, not configurable).
- Session start injection is capped at 800 characters.
- Both use count-only references ("4 error patterns available") for detail-heavy categories.
- Execution progress (3200 chars) always has priority over memory injection. If combined output exceeds 5000 chars, truncate memory first.

**Warning signs:** Agent ignoring quorum instructions after compaction; agent losing execution position after compaction.

### Pitfall 2: JSONL File Growth Without Bounds

**What goes wrong:** After months of use, decisions.jsonl grows to thousands of entries. Reading the full file for each session start or compaction becomes slow.

**Why it happens:** No automatic pruning. Append-only by default.

**How to avoid:**
- `prune` command removes entries older than 90 days by default (configurable via `memory_prune_days` in nf.json).
- Session-start hook runs prune as a side effect (best-effort, fail-open) every 7 days (tracked via a `.planning/memory/.last-prune` timestamp file).
- Query commands read from tail of file (most recent entries) using readline in reverse, not by parsing the entire file.

**Warning signs:** Session start latency > 200ms; memory JSONL files > 100KB.

### Pitfall 3: Duplicate Memory Entries

**What goes wrong:** The same decision or error fix is recorded multiple times (e.g., after compaction, the agent re-discovers and re-records the same decision).

**Why it happens:** Agent context is lost at compaction. The injected memory summary may not prevent the agent from re-recording discoveries.

**How to avoid:**
- `append-decision` and `append-error` check the last 10 entries for matching summary/symptom. If a match exists (case-insensitive substring), skip the append and return the existing entry.
- Dedup is best-effort, not guaranteed. A small number of duplicates is acceptable -- prune handles long-term cleanup.

**Warning signs:** Consecutive entries with identical summaries in the JSONL file.

### Pitfall 4: Session Reminder Injection Conflicts with Existing Context Pieces

**What goes wrong:** nf-session-start.js already injects telemetry alerts and STATE.md reminders. Adding memory reminders to `_contextPieces` may produce an additionalContext payload that is too long or confusing.

**Why it happens:** Multiple systems competing for the same injection channel.

**How to avoid:**
- Memory reminder is appended to `_contextPieces` with lower priority than telemetry alerts.
- If telemetry alert is present (priority >= 50), memory reminder is shortened to a one-liner: "Memory: 3 decisions, 4 errors, 7 quorum decisions available. Query with: node bin/memory-store.cjs ..."
- Combined _contextPieces output tested to stay under 2000 characters total.

**Warning signs:** Session start output exceeding 2000 characters; agent receiving multiple competing directives.

### Pitfall 5: Error Pattern Matching Too Broad or Too Narrow

**What goes wrong:** `query-errors --symptom` either returns too many irrelevant matches (broad grep) or misses relevant errors (exact match too strict).

**Why it happens:** Symptom descriptions are natural language, not structured identifiers. Grep matching on free text is inherently fuzzy.

**How to avoid:**
- Primary match: case-insensitive substring on `symptom` field.
- Secondary match: also check `tags` array for any keyword match.
- Results sorted by recency (newest first), limited to 5 matches by default.
- Include a `--limit N` flag for controlling result count.

**Warning signs:** Agent reporting "no matching error patterns" when a relevant one exists; agent receiving 20+ matches for a common term.

## Code Examples

### Memory Store Core (bin/memory-store.cjs)

```javascript
'use strict';

const fs = require('fs');
const path = require('path');

const MEMORY_DIR = '.planning/memory';
const FILES = {
  decisions: 'decisions.jsonl',
  errors: 'errors.jsonl',
  quorum: 'quorum-decisions.jsonl',
};

function getMemoryPath(cwd, category) {
  const dir = path.join(cwd, MEMORY_DIR);
  fs.mkdirSync(dir, { recursive: true });
  return path.join(dir, FILES[category]);
}

function appendEntry(cwd, category, entry) {
  const filePath = getMemoryPath(cwd, category);
  const line = JSON.stringify({ ...entry, ts: new Date().toISOString() }) + '\n';
  fs.appendFileSync(filePath, line, 'utf8');
  return entry;
}

function readLastN(cwd, category, n = 5) {
  const filePath = getMemoryPath(cwd, category);
  if (!fs.existsSync(filePath)) return [];
  const lines = fs.readFileSync(filePath, 'utf8').trim().split('\n').filter(Boolean);
  return lines.slice(-n).map(l => {
    try { return JSON.parse(l); } catch { return null; }
  }).filter(Boolean);
}

function queryByField(cwd, category, field, keyword, limit = 5) {
  const filePath = getMemoryPath(cwd, category);
  if (!fs.existsSync(filePath)) return [];
  const lines = fs.readFileSync(filePath, 'utf8').trim().split('\n').filter(Boolean);
  const kw = keyword.toLowerCase();
  const matches = [];
  // Search from end (most recent first)
  for (let i = lines.length - 1; i >= 0 && matches.length < limit; i--) {
    try {
      const entry = JSON.parse(lines[i]);
      const fieldVal = (entry[field] || '').toLowerCase();
      const tagsVal = (entry.tags || []).join(' ').toLowerCase();
      if (fieldVal.includes(kw) || tagsVal.includes(kw)) {
        matches.push(entry);
      }
    } catch { /* skip malformed lines */ }
  }
  return matches;
}

// Dedup check: returns true if a similar entry exists in last N entries
function isDuplicate(cwd, category, field, value, checkLast = 10) {
  const recent = readLastN(cwd, category, checkLast);
  const needle = value.toLowerCase();
  return recent.some(e => {
    const hay = ((e[field]) || '').toLowerCase();
    return hay.includes(needle) || needle.includes(hay);
  });
}
```

### Session Reminder Generation

```javascript
function generateSessionReminder(cwd) {
  const decisions = readLastN(cwd, 'decisions', 3);
  const errorCount = countEntries(cwd, 'errors');
  const quorumCount = countEntries(cwd, 'quorum');

  if (decisions.length === 0 && errorCount === 0 && quorumCount === 0) return null;

  const lines = ['SESSION MEMORY REMINDER:'];

  if (decisions.length > 0) {
    lines.push('');
    lines.push('Recent decisions:');
    for (const d of decisions) {
      const phase = d.phase ? `[${d.phase}]` : '';
      lines.push(`  - ${phase} ${d.summary} (${d.source || 'unknown'})`);
    }
  }

  if (errorCount > 0) {
    lines.push('');
    lines.push(`Error patterns: ${errorCount} entries (query: node bin/memory-store.cjs query-errors --symptom "<keyword>")`);
  }

  if (quorumCount > 0) {
    lines.push('');
    lines.push(`Quorum decisions: ${quorumCount} entries (query: node bin/memory-store.cjs query-quorum --last 5)`);
  }

  const result = lines.join('\n');
  // Cap at 800 characters
  return result.length > 800 ? result.slice(0, 797) + '...' : result;
}
```

### PreCompact Memory Injection

```javascript
function formatMemoryInjection(cwd) {
  // Fail-open require of memory-store
  let memoryStore;
  try {
    memoryStore = require(path.join(__dirname, '..', 'bin', 'memory-store.cjs'));
  } catch { return null; }

  const decisions = memoryStore.readLastN(cwd, 'decisions', 3);
  const recentErrors = memoryStore.readLastN(cwd, 'errors', 2);

  if (decisions.length === 0 && recentErrors.length === 0) return null;

  const lines = ['## Memory Snapshot (auto-injected at compaction)', ''];

  if (decisions.length > 0) {
    lines.push('Last decisions:');
    for (const d of decisions) {
      const phase = d.phase ? `[${d.phase}]` : '';
      lines.push(`  - ${phase} ${d.summary}`);
    }
  }

  if (recentErrors.length > 0) {
    lines.push('');
    lines.push('Recent error fixes:');
    for (const e of recentErrors) {
      // Truncate symptom to 60 chars
      const symptom = e.symptom.length > 60 ? e.symptom.slice(0, 57) + '...' : e.symptom;
      lines.push(`  - ${symptom} -> ${e.fix.slice(0, 60)}`);
    }
  }

  lines.push('');
  lines.push('Query more: node bin/memory-store.cjs query-decisions --last 10');

  const result = lines.join('\n');
  // Hard cap at 1200 characters
  return result.length > 1200 ? result.slice(0, 1197) + '...' : result;
}
```

## Open Questions

1. **Should memory JSONL files be gitignored?**

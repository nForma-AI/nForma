# Phase v0.30-06: Subagent Orchestration - Research

**Researched:** 2026-03-08
**Domain:** Quorum subagent context retrieval, phase context accumulation, specialized retrieval agents
**Confidence:** HIGH

## Summary

Phase v0.30-06 transforms quorum slot workers from passive context consumers (receiving fixed prompt dumps) into active context retrievers that can iteratively request additional context. It also introduces a context accumulation stack that carries architecture decisions, test results, and API contracts across completed phases into subsequent planning. Finally, it adds specialized retrieval agent definitions (test-retriever, architecture-retriever, formal-model-retriever) with preloaded domain skills.

The codebase already has all necessary infrastructure: `quorum-slot-dispatch.cjs` builds prompts and parses output, `call-quorum-slot.cjs` handles subprocess dispatch, `memory-store.cjs` provides JSONL-based persistence with query capabilities, and `execution-progress.cjs` tracks file-based state. The key architectural decision is that quorum slot workers are spawned as `Task(subagent_type="nf-quorum-slot-worker")` with `tools: Bash` only -- they cannot use Read/Grep/Glob directly. Iterative retrieval must therefore work through the existing Bash tool, using `node` commands to invoke retrieval scripts that return structured results. The per-slot token budget (8k max) and max 2 retrieval rounds are hard constraints from the success criteria.

The zero-dependency constraint from prior phases continues. All implementation uses Node.js built-ins, existing JSONL storage patterns, and the established hook injection mechanism (`additionalContext`). Context accumulation builds on the existing `memory-store.cjs` categories pattern, adding a new `context-stack` category for phase-level architectural context.

**Primary recommendation:** Build a `bin/context-retriever.cjs` module with domain-specific retrieval functions, a `bin/context-stack.cjs` module for phase accumulation, and extend `quorum-slot-dispatch.cjs` to support a two-round retrieval protocol before the main quorum prompt is sent to the external CLI.

### Pattern 1: Iterative Retrieval Protocol (ORCH-01)

**What:** Quorum slot workers can request additional context before forming their verdict. The `quorum-slot-dispatch.cjs` module is extended to support a pre-query phase where the worker first asks for context, receives it, then forms its position.

**When to use:** Every quorum dispatch that involves code review, plan assessment, or architecture decisions.

**Key constraint:** Slot workers are `Task(subagent_type="nf-quorum-slot-worker")` with `tools: Bash` only. They CANNOT call Read/Grep/Glob directly. The retrieval must happen INSIDE `quorum-slot-dispatch.cjs` (which runs in Node.js) before the external CLI is called. The external CLI (Gemini, Codex, etc.) receives an enriched prompt with retrieved context already inlined.

**Design:** Two-phase prompt construction in `quorum-slot-dispatch.cjs`:
1. **Analysis phase:** Parse the question and artifact to identify what additional context is needed (file patterns, test results, architecture decisions). This is deterministic code -- not an LLM call.
2. **Retrieval phase:** Call `context-retriever.cjs` functions to fetch relevant context within the 8k token budget. Max 2 retrieval rounds (first round: primary context, second round: cross-references found in first round).
3. **Enriched prompt:** Build the final prompt with retrieved context sections appended, then dispatch to the external CLI as before.

**Token budget enforcement:** Each retrieval round produces context measured in characters. The budget is 8000 tokens ~ 32000 characters (4 chars/token estimate). Track accumulated context size and stop retrieval when approaching the budget.

**Example:**
```javascript
// In quorum-slot-dispatch.cjs, new function
function buildEnrichedPrompt(basePrompt, question, artifactPath, cwd) {
  const retriever = require('./context-retriever.cjs');
  const TOKEN_BUDGET = 8000; // tokens
  const CHAR_BUDGET = TOKEN_BUDGET * 4; // ~4 chars per token
  const MAX_ROUNDS = 2;

  let additionalContext = '';
  let charCount = 0;

  for (let round = 0; round < MAX_ROUNDS && charCount < CHAR_BUDGET; round++) {
    const needs = retriever.analyzeContextNeeds(question, artifactPath, additionalContext);
    if (needs.length === 0) break;

    const retrieved = retriever.fetchContext(cwd, needs, CHAR_BUDGET - charCount);
    if (!retrieved || retrieved.length === 0) break;

    additionalContext += retrieved;
    charCount = additionalContext.length;
  }

  if (additionalContext) {
    return basePrompt + '\n\n=== RETRIEVED CONTEXT ===\n' + additionalContext + '\n========================\n';
  }
  return basePrompt;
}
```

### Pattern 2: Context Stack (ORCH-02)

**What:** Architecture decisions, test results, and API contracts from completed phases are persisted in a phase-indexed context stack. When a new phase begins, relevant entries from the stack are injected into the planning context.

**When to use:** Phase transitions and planning command injection.

**Design:** A new `bin/context-stack.cjs` module with:
- **append(cwd, { phase, type, content, tags })** -- add a context entry after phase completion
- **queryByPhase(cwd, phase)** -- get all entries for a phase
- **queryByType(cwd, type, limit)** -- get entries by type (decision, test_result, api_contract)
- **formatInjection(cwd, currentPhase)** -- format accumulated context for injection into planning

Storage: `.planning/memory/context-stack.jsonl` -- follows existing JSONL pattern from memory-store.cjs.

Entry types:
- `architecture_decision`: Design choices that affect downstream phases
- `test_result`: Test suite outcomes relevant to integration points
- `api_contract`: Interface definitions between modules
- `constraint`: Discovered constraints that must carry forward

**Injection point:** Extend `nf-prompt.js` to include context stack entries in planning command `additionalContext`. Also extend `nf-precompact.js` to carry forward context stack summary on compaction.

**Example:**
```javascript
// bin/context-stack.cjs
const CONTEXT_STACK_FILE = 'context-stack.jsonl';
```

### Pattern 3: Specialized Retrieval Agents (ORCH-03)

**What:** Domain-specific agent definitions that know how to fetch targeted context more efficiently than generic file reading. These are NOT separate Claude Code subagent types -- they are retrieval FUNCTIONS within `context-retriever.cjs` with domain-specific search strategies.

**Architectural decision:** Claude Code subagents cannot spawn subagents (explicit out-of-scope constraint). Therefore "specialized retrieval agents" are implemented as domain-specific functions within `context-retriever.cjs`, NOT as separate agent .md files. However, agent definition files CAN be created as documentation that describes the retrieval strategy for each domain, which the functions implement.

**Three domains:**
1. **Test Retriever:** Searches test files, test results, coverage data. Knows to look in `hooks/dist/*.test.*`, `bin/*.test.cjs`, `.planning/formal/unit-test-coverage.json`.
2. **Architecture Retriever:** Searches architecture decisions, module interfaces, hook contracts. Knows to look in `.planning/memory/decisions.jsonl`, `.planning/STATE.md`, `CLAUDE.md`, agent definitions.
3. **Formal Model Retriever:** Searches formal verification artifacts. Knows to look in `.planning/formal/` (TLA+, Alloy, PRISM specs), `.planning/formal/solve-state.json`, `.planning/formal/traceability-matrix.json`.

**Example:**
```javascript
// bin/context-retriever.cjs
const RETRIEVERS = {
  test: {
    patterns: ['*.test.cjs', '*.test.js', 'unit-test-coverage.json'],
    searchDirs: ['bin/', 'hooks/dist/', '.planning/formal/'],
    maxChars: 8000,
  },
  architecture: {
    patterns: ['STATE.md', 'decisions.jsonl', 'CLAUDE.md'],
    searchDirs: ['.planning/', '.planning/memory/', ''],
    maxChars: 8000,
  },
  formal: {
    patterns: ['*.tla', '*.als', '*.pm', 'traceability-matrix.json'],
    searchDirs: ['.planning/formal/'],
    maxChars: 8000,
  },
};

function analyzeContextNeeds(question, artifactPath, existingContext) {
  const needs = [];
  const q = question.toLowerCase();

  if (q.includes('test') || q.includes('coverage') || q.includes('verify')) {
    needs.push({ domain: 'test', query: question });
  }
  if (q.includes('architecture') || q.includes('design') || q.includes('pattern') || q.includes('decision')) {
    needs.push({ domain: 'architecture', query: question });
  }
  if (q.includes('formal') || q.includes('invariant') || q.includes('spec') || q.includes('tla') || q.includes('alloy')) {
    needs.push({ domain: 'formal', query: question });
  }

  // Also analyze artifact path for domain hints
  if (artifactPath) {
    if (artifactPath.includes('test')) needs.push({ domain: 'test', query: artifactPath });
    if (artifactPath.includes('formal')) needs.push({ domain: 'formal', query: artifactPath });
  }

  return needs;
}
```

### Anti-Patterns to Avoid

- **Trying to give slot workers Read/Grep/Glob tools:** Claude Code architectural constraint -- `nf-quorum-slot-worker` has `tools: Bash` only. All retrieval must happen BEFORE the prompt reaches the external CLI, inside the Node.js dispatch layer.
- **Unbounded context retrieval:** Without the 8k token budget cap, retrieved context could exceed the external CLI's context window. Always enforce the char budget.
- **Storing full file contents in context stack:** Context stack entries should be summaries/decisions, not full file contents. Full files can be retrieved on demand via the retrieval functions.
- **Modifying the slot-worker agent definition:** The thin passthrough pattern (`nf-quorum-slot-worker.md`) is intentional and battle-tested. Add retrieval logic to `quorum-slot-dispatch.cjs` instead.
- **Breaking the fail-open pattern:** All new modules MUST follow try/catch with fail-open. If retrieval fails, the quorum dispatch proceeds with the original un-enriched prompt.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSONL persistence | Custom file I/O | `memory-store.cjs` patterns (appendEntry, readLastN, queryByField) | Handles dedup, malformed lines, prune, and is battle-tested |
| Path resolution | Hardcoded paths | `planning-paths.cjs` resolve/resolveWithFallback | Handles legacy vs canonical layout, creates dirs |
| Token counting | Character heuristic only | 4 chars/token estimate with hard cap | Good enough for budget enforcement; exact counting requires tokenizer dependency |
| Hook injection | New hook events | Extend existing `additionalContext` in nf-prompt.js and nf-precompact.js | additionalContext is the established injection channel |
| Test file discovery | Manual glob | `fs.readdirSync` + extension filter | Matches existing patterns in learning-extractor.cjs |

**Key insight:** The entire retrieval system is an enhancement layer on top of the existing prompt construction pipeline. It should be surgically inserted into `quorum-slot-dispatch.cjs`'s `buildModeAPrompt`/`buildModeBPrompt` flow without changing the external interface.

## Common Pitfalls

### Pitfall 1: additionalContext Token Budget Contention
**What goes wrong:** Multiple injection sources (quorum instructions + memory + verification + context stack) compete for the ~4000 token additionalContext ceiling. Adding context stack injection could push past the limit.
**Why it happens:** STATE.md already notes "additionalContext token budget contention: multiple injection sources must share ~4000 token ceiling."
**How to avoid:** Context stack injection into additionalContext should be capped at 800 chars (matching memory-store's generateSessionReminder cap). For quorum dispatch enrichment, the 8k budget is separate -- it goes INTO the prompt text, not into additionalContext.
**Warning signs:** additionalContext strings exceeding 4000 chars; Claude receiving truncated or garbled context.

### Pitfall 2: Two-Round Retrieval Becoming a Performance Bottleneck
**What goes wrong:** If retrieval rounds involve expensive file reads or recursive directory scans, the quorum dispatch latency doubles or triples.
**Why it happens:** Each retrieval round reads files synchronously before the external CLI call.
**How to avoid:** Retrieval functions should be fast: `fs.readFileSync` of known small files, not recursive globbing. Pre-compute retrieval targets from question keywords rather than scanning the entire repo. Cap each retrieval at 100ms with a timeout.
**Warning signs:** Quorum round times increasing by more than 2 seconds.

### Pitfall 3: Context Stack Growing Unbounded
**What goes wrong:** After many phases, the context stack accumulates hundreds of entries, and injection becomes bloated.
**Why it happens:** No automatic pruning or relevance filtering.
**How to avoid:** (1) Cap entries per phase at 10. (2) Format injection shows only entries from the last 3 completed phases. (3) Include a prune command that removes entries older than the current milestone.
**Warning signs:** context-stack.jsonl exceeding 100KB; injection strings exceeding 2000 chars.

### Pitfall 4: Retrieval Missing the Correct Domain
**What goes wrong:** The `analyzeContextNeeds` function mis-classifies the question domain, retrieving test context when architecture context was needed.
**Why it happens:** Keyword-based domain detection is imprecise.
**How to avoid:** Use multiple signal sources: question keywords, artifact path segments (using existing `PATH_CATEGORY_MAP` from `quorum-slot-dispatch.cjs`), and the `review_context` string. Default to multi-domain retrieval rather than single-domain.
**Warning signs:** Quorum slots giving low-quality verdicts on architecture questions despite retrieval being active.

### Pitfall 5: Breaking the Install Sync Chain
**What goes wrong:** New bin/ files created but not registered in install.js, so they are not available at `~/.claude/nf-bin/` at runtime.
**Why it happens:** Forgetting the install sync step that is required for all bin/ scripts.
**How to avoid:** Any new bin/*.cjs file MUST be added to install.js BIN_FILES list. Test by running `node bin/install.js --claude --global` and verifying the file appears in `~/.claude/nf-bin/`.
**Warning signs:** `require()` failures in production with "Cannot find module" errors.

```javascript
// bin/context-retriever.cjs
'use strict';

const fs = require('fs');
const path = require('path');

const TOKEN_BUDGET_CHARS = 32000; // 8k tokens * 4 chars/token
const MAX_ROUNDS = 2;

// Domain-specific retrieval configurations
const DOMAIN_CONFIG = {
  test: {
    files: [
      '.planning/formal/unit-test-coverage.json',
    ],
    searchPatterns: [/\.test\.(cjs|js)$/],
    searchDirs: ['bin/', 'hooks/'],
  },
  architecture: {
    files: [
      '.planning/STATE.md',
      '.planning/memory/decisions.jsonl',
    ],
    searchDirs: ['.planning/'],
  },
  formal: {
    files: [
      '.planning/formal/traceability-matrix.json',
      '.planning/formal/solve-state.json',
    ],
    searchPatterns: [/\.(tla|als|pm)$/],
    searchDirs: ['.planning/formal/'],
  },
};

function analyzeContextNeeds(question, artifactPath, existingContext) {
  // Returns array of { domain, query } objects
  // Uses keyword detection + artifact path signals
}

function fetchContext(cwd, needs, charBudget) {
  // For each need, read relevant files up to charBudget
  // Returns concatenated context string
  let result = '';
  let remaining = charBudget;

  for (const need of needs) {
    if (remaining <= 0) break;
    const config = DOMAIN_CONFIG[need.domain];
    if (!config) continue;

    // Read known files first (fast path)
    for (const relPath of (config.files || [])) {
      if (remaining <= 0) break;
      try {
        const fullPath = path.join(cwd, relPath);
        if (!fs.existsSync(fullPath)) continue;
        const content = fs.readFileSync(fullPath, 'utf8');
        const chunk = content.slice(0, remaining);
        result += '\n--- ' + relPath + ' ---\n' + chunk + '\n';
        remaining -= chunk.length + relPath.length + 20;
      } catch (_) { /* fail-open */ }
    }
  }

  return result;
}

module.exports = {
  analyzeContextNeeds,
  fetchContext,
  DOMAIN_CONFIG,
  TOKEN_BUDGET_CHARS,
  MAX_ROUNDS,
};
```

### Context Stack Module Structure
```javascript
// bin/context-stack.cjs
'use strict';

const fs = require('fs');
const path = require('path');

const STACK_FILE = 'context-stack.jsonl';
const MAX_ENTRIES_PER_PHASE = 10;
const INJECTION_CAP_CHARS = 2000;

function getStackPath(cwd) {
  const dir = path.join(cwd, '.planning', 'memory');
  fs.mkdirSync(dir, { recursive: true });
  return path.join(dir, STACK_FILE);
}

function append(cwd, { phase, type, content, tags }) {
  const stackPath = getStackPath(cwd);
  const entry = {
    phase,
    type, // architecture_decision | test_result | api_contract | constraint
    content,
    tags: tags || [],
    ts: new Date().toISOString(),
  };
  fs.appendFileSync(stackPath, JSON.stringify(entry) + '\n', 'utf8');
  return entry;
}

function queryRecentPhases(cwd, currentPhase, maxPhases = 3) {
  const stackPath = getStackPath(cwd);
  if (!fs.existsSync(stackPath)) return [];
  const lines = fs.readFileSync(stackPath, 'utf8').trim().split('\n').filter(Boolean);
  const entries = lines.map(l => { try { return JSON.parse(l); } catch { return null; } }).filter(Boolean);

  // Get unique phases, sorted by recency, exclude current
  const phases = [...new Set(entries.map(e => e.phase))].filter(p => p !== currentPhase);
  const recentPhases = phases.slice(-maxPhases);

  return entries.filter(e => recentPhases.includes(e.phase));
}

function formatInjection(cwd, currentPhase) {
  const entries = queryRecentPhases(cwd, currentPhase);
  if (entries.length === 0) return null;

  const lines = ['## Context Stack (accumulated from prior phases)', ''];
  let chars = 0;

  for (const e of entries) {
    const line = '- [' + e.phase + '] ' + e.type + ': ' + e.content;
    if (chars + line.length > INJECTION_CAP_CHARS) break;
    lines.push(line);
    chars += line.length;
  }

  return lines.join('\n');
}

module.exports = { append, queryRecentPhases, formatInjection, getStackPath };
```

### Extending quorum-slot-dispatch.cjs for Retrieval
```javascript
// Addition to quorum-slot-dispatch.cjs main() function
// After prompt is built, before spawning call-quorum-slot.cjs:

function enrichPromptWithRetrieval(prompt, question, artifactPath, cwd) {
  try {
    const retriever = require('./context-retriever.cjs');
    const contextStack = require('./context-stack.cjs');

    // Phase 1: Domain-aware retrieval
    let additionalContext = '';
    let charCount = 0;
    const charBudget = retriever.TOKEN_BUDGET_CHARS;

    for (let round = 0; round < retriever.MAX_ROUNDS && charCount < charBudget; round++) {
      const needs = retriever.analyzeContextNeeds(question, artifactPath, additionalContext);
      if (needs.length === 0) break;

      const retrieved = retriever.fetchContext(cwd, needs, charBudget - charCount);
      if (!retrieved || retrieved.length === 0) break;

      additionalContext += retrieved;
      charCount = additionalContext.length;
    }

    // Phase 2: Context stack injection
    const stackContext = contextStack.formatInjection(cwd, 'current');
    if (stackContext && charCount + stackContext.length < charBudget) {
      additionalContext += '\n' + stackContext;
    }

    if (additionalContext) {
      return prompt + '\n\n=== RETRIEVED CONTEXT ===\n' + additionalContext + '\n========================\n';
    }
  } catch (_) {
    // Fail-open: retrieval errors never prevent quorum dispatch
  }
  return prompt;
}
```

## Open Questions

1. **How should context stack entries be populated?**
   - What we know: Entries should be appended after phase completion (verification pass)
   - What's unclear: Should the planner auto-extract decisions, or should the executor explicitly call context-stack append?
   - Recommendation: Auto-extract from VERIFICATION.md and SUMMARY.md at phase completion in the execute-phase workflow. Also support manual CLI: `node bin/context-stack.cjs append --phase v0.30-05 --type architecture_decision --content "..."`

2. **Should retrieval be opt-in or always-on?**
   - What we know: The success criteria say "workers CAN request" implying optional
   - What's unclear: Whether the performance overhead is acceptable for all quorum dispatches
   - Recommendation: Always-on with a config kill switch (`context_retrieval_enabled: false` in nf.json). The retrieval is fast (file reads, not network calls) and fail-open, so the overhead is minimal (~50-100ms per dispatch).

3. **How do agent definition .md files for specialized retrievers interact with the system?**
   - What we know: Claude Code subagents cannot spawn subagents. The retrieval happens in Node.js code, not via agent spawning.
   - What's unclear: Whether to create agent .md files at all, or just implement as code modules
   - Recommendation: Create agent .md files as documentation/strategy definitions only. The actual retrieval is in `context-retriever.cjs` functions. The .md files describe the retrieval strategy per domain and can be referenced by the context-retriever code comments.

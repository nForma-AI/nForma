# Phase v0.30-02: File-Based Execution State - Research

**Researched:** 2026-03-07
**Domain:** File-based sub-task progress tracking for compaction-resilient agent execution
**Confidence:** HIGH

## Summary

This phase implements VERF-01: sub-task progress within a plan is tracked in a file so that mid-execution compaction does not lose position. Currently, the agent tracks execution state purely in conversation context -- when compaction fires, the agent loses which tasks it has completed within a multi-step plan and either restarts from the beginning (wasting tokens) or guesses incorrectly (producing incomplete work). The `nf-precompact.js` hook preserves STATE.md's Current Position section (phase/plan level) but not sub-task progress within a plan.

The solution is a file-based execution progress tracker: a JSON file (`.planning/execution-progress.json`) that records which tasks within the current plan have been completed, along with their commit hashes and completion timestamps. The `nf-precompact.js` hook injects this progress into the compaction continuation context so the agent resumes at the correct sub-task. The execute-plan workflow updates the progress file after each task commit. A termination guard ensures the agent loop always reaches a terminal state -- the progress file includes an iteration counter and a hard cap, preventing infinite loops or stalls.

**Primary recommendation:** Build a single `bin/execution-progress.cjs` script with three commands (`init`, `complete-task`, `get-status`) that the execute-plan workflow calls at task boundaries. Extend `nf-precompact.js` to read and inject this progress. Add a termination guard that detects stuck-in-loop patterns (same task attempted 3+ times without advancing).

### Pattern 1: Execution Progress File

**What:** A JSON file that tracks sub-task completion state within the currently executing plan.

**When created:** At plan execution start, before the first task begins.

**When updated:** After each task commit succeeds.

**When cleared:** When plan SUMMARY.md is written (plan complete) or on `execution-progress clear`.

**Schema:**

```javascript
// .planning/execution-progress.json
{
  "version": 1,
  "phase": "v0.30-02",
  "plan": "01",
  "plan_file": "v0.30-02-01-PLAN.md",
  "total_tasks": 4,
  "status": "in_progress",  // "not_started" | "in_progress" | "complete" | "failed"
  "started_at": "2026-03-07T14:30:00Z",
  "updated_at": "2026-03-07T14:35:00Z",
  "iteration_count": 0,       // incremented on each resume-after-compaction
  "max_iterations": 5,        // hard cap -- status -> "failed" if exceeded
  "tasks": [
    {
      "number": 1,
      "name": "Task 1: Create execution-progress.cjs",
      "status": "complete",    // "pending" | "in_progress" | "complete" | "skipped"
      "commit_hash": "abc1234",
      "completed_at": "2026-03-07T14:32:00Z"
    },
    {
      "number": 2,
      "name": "Task 2: Extend nf-precompact.js",
      "status": "in_progress",
      "commit_hash": null,
      "completed_at": null
    },
    {
      "number": 3,
      "name": "Task 3: Update execute-plan workflow",
      "status": "pending",
      "commit_hash": null,
      "completed_at": null
    }
  ]
```

### Pattern 2: Progress Injection at Compaction

**What:** Extend `nf-precompact.js` to read `execution-progress.json` and inject sub-task state into the compaction continuation context.

**When:** Only when an execution is in progress (file exists with status `in_progress`).

**Injection format:**

```
## Execution Progress (auto-injected at compaction)

Plan: v0.30-02-01-PLAN.md
Status: in_progress (2/4 tasks complete, iteration 1 of 5)

Completed tasks:
  [x] Task 1: Create execution-progress.cjs (commit abc1234)
  [x] Task 2: Extend nf-precompact.js (commit def5678)

Resume at:
  [ ] Task 3: Update execute-plan workflow

IMPORTANT: Read the plan file above and continue from Task 3.
Do NOT re-execute Tasks 1-2 -- they are already committed.
```

### Pattern 3: Termination Guard

**What:** A mechanism that prevents the agent loop from stalling or looping infinitely when file-based state tracking is in use.

**How it works:**
1. `iteration_count` in the progress file is incremented each time the agent resumes after compaction (detected by PreCompact hook writing `"compaction_occurred": true` flag, cleared on first read).
2. If `iteration_count >= max_iterations` (default 5), the progress file's status is set to `"failed"` and the injected context says "Execution cap exhausted -- report to user."
3. A stuck-detection heuristic checks if the same task has been marked `in_progress` across 3+ consecutive iteration increments without advancing. If detected, status is set to `"failed"` with reason `"stuck_on_task"`.

**Formal property mapping:**
- `iteration_count < max_iterations` ensures bounded execution (maps to TLA+ `NeverExceedsCap`)
- Status always reaches terminal (`"complete"` or `"failed"`) (maps to TLA+ `EventuallyTerminates`)
- Once complete/failed, status does not revert (maps to TLA+ `TerminalIsFinal`)

### Pattern 4: Workflow Integration Points

**Where execute-plan.md changes:**

1. **Before first task:** Call `node bin/execution-progress.cjs init --plan <plan_file> --tasks <count>` to create progress file.
2. **After each task commit:** Call `node bin/execution-progress.cjs complete-task --number <N> --commit <hash>` to mark task done.
3. **After SUMMARY.md creation:** Call `node bin/execution-progress.cjs clear` to remove the progress file.
4. **On resumption (after compaction):** Call `node bin/execution-progress.cjs get-status` to determine where to resume. If status is `"failed"`, report to user instead of continuing.

### Anti-Patterns to Avoid

- **Storing progress in STATE.md directly:** STATE.md is for human-readable project state. Embedding JSON task arrays in it creates parsing fragility and bloats the file.
- **Using conversation context as state:** This is exactly what VERF-01 prohibits -- conversation context is lost at compaction.
- **Tracking progress per-task in separate files:** One file per task creates filesystem clutter and makes "where am I?" queries expensive.
- **Complex state machine with many transitions:** The progress tracker has exactly 4 states and simple transitions. Do not over-engineer with XState or formal FSM libraries at runtime.
- **Writing progress file on every tool call:** Only write on task-boundary events (commit, SUMMARY creation, init). Not on every Read/Write/Bash.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic file writes | Custom lock manager | `fs.writeFileSync` (atomic for single writer) | nForma is single-agent by design; no concurrent writers for progress file |
| Task parsing from PLAN.md | Custom PLAN parser | Count `<task type=` markers via regex | PLAN format is stable; task elements follow XML-like pattern |
| Compaction detection | Custom compaction detector | PreCompact hook event (already fires before compaction) | Claude Code guarantees this hook fires |
| Iteration cap enforcement | Watchdog timer | Counter in progress file + check on resume | Simpler, no background process needed |

**Key insight:** The progress file is a simple record, not a complex state machine. The state transitions are: not_started -> in_progress -> complete (happy path) or not_started -> in_progress -> failed (cap exhausted or stuck). There is no concurrent access concern because nForma runs a single agent per plan execution.

## Common Pitfalls

### Pitfall 1: Progress File Survives Into Next Plan

**What goes wrong:** If the progress file from plan 01 is not cleared before plan 02 starts, the agent resumes from plan 01's last position instead of starting plan 02 fresh.

**Why it happens:** Plan execution can be interrupted (compaction, crash, user abort) before SUMMARY.md is created, leaving the progress file behind. The next plan execution reads the stale file.

**How to avoid:**
- On `init`, always overwrite any existing progress file. The init command is idempotent.
- Validate that the `plan_file` field matches the current plan before using progress data. If it does not match, treat as "not started" and overwrite.
- Add a `clear` step to the SUMMARY.md creation, not just to the end of the plan.

**Warning signs:** Agent skipping tasks in a new plan; SUMMARY.md referencing commits from a previous plan.

### Pitfall 2: Compaction During Task (Mid-Task State)

**What goes wrong:** Compaction fires while a task is in progress (marked `in_progress` but not yet committed). After compaction, the agent sees the task as `in_progress` and re-attempts it from scratch, potentially creating duplicate code or conflicting commits.

**Why it happens:** Tasks can involve multiple tool calls (read file, write code, run tests, commit). Compaction can fire between any of these steps. The progress file shows `in_progress` but does not record intermediate work.

**How to avoid:**
- On resume, check git log for commits matching the task pattern (`{phase}-{plan}` prefix in commit message). If a commit exists for the "in_progress" task, mark it complete and advance.
- Include a `last_known_commit` field in the progress file that records the HEAD at the last task completion. On resume, diff `last_known_commit..HEAD` to detect any work done after the last recorded completion.
- The resume instructions should say "verify git state before continuing" not "repeat the in_progress task."

**Warning signs:** Duplicate commits for the same task; "file already exists" errors on resume.

### Pitfall 3: Progress Injection Exceeding additionalContext Budget

**What goes wrong:** The progress injection adds too many tokens to `nf-precompact.js`'s additionalContext output, crowding out the STATE.md position and resume instructions that already exist.

**Why it happens:** Progress files for plans with many tasks (10+) can produce long task lists. Combined with STATE.md injection and pending task injection, the total exceeds the practical ~4000 token ceiling.

**How to avoid:**
- Cap progress injection at 800 tokens (measured as characters / 4 rough estimate).
- For plans with 6+ tasks, summarize: "Tasks 1-5 complete, resume at Task 6" instead of listing all.
- Progress injection has lower priority than STATE.md position and pending tasks. If budget is tight, truncate progress first.

**Warning signs:** Agent ignoring quorum instructions after compaction; additionalContext total exceeding 3000 characters.

### Pitfall 4: Iteration Counter Incrementing on Non-Compaction Restarts

**What goes wrong:** The iteration counter increments every time the agent reads the progress file, not just on compaction. If the agent reads progress multiple times during normal execution (e.g., to check status), the counter hits `max_iterations` prematurely.

**Why it happens:** No distinction between "reading for status check" and "reading for post-compaction resume."

**How to avoid:**
- Only increment `iteration_count` in the `nf-precompact.js` hook (which fires exactly once per compaction). Add a `"last_compaction_at"` timestamp field.
- The `get-status` command is a read-only operation that never modifies the file.
- The `complete-task` command updates tasks but does not touch `iteration_count`.

**Warning signs:** `iteration_count` reaching 3+ during a single uncompacted session.

### Pitfall 5: Formal EventuallyTerminates Violation via Stuck Detection Race

**What goes wrong:** The stuck-detection heuristic (same task `in_progress` across 3+ iterations) can be defeated if the agent makes partial progress (e.g., edits a file but does not commit) that does not advance the progress file. The agent appears stuck but is actually working.

**Why it happens:** The progress file only advances on commit. Work done between commits is invisible to the progress tracker.

**How to avoid:**
- Use git state as a secondary signal: if `git diff --name-only` shows changed files since `last_known_commit`, the agent is making progress even if no task is "complete."
- The stuck heuristic should check BOTH progress file AND git state. Only declare "stuck" if both show no advancement across 3+ iterations.
- Add a `"resumed_task_attempts"` counter per task, not just a global iteration counter.

**Warning signs:** False "stuck" declarations when the agent is mid-task; agent reporting "failed" when partial work exists uncommitted.

## Code Examples

### Execution Progress Script Interface

```javascript
// bin/execution-progress.cjs
'use strict';

const fs = require('fs');
const path = require('path');

const PROGRESS_FILE = '.planning/execution-progress.json';

function getProgressPath(cwd) {
  return path.join(cwd, PROGRESS_FILE);
}

// Initialize progress tracking for a plan
function initProgress(cwd, { planFile, totalTasks, taskNames }) {
  const progress = {
    version: 1,
    phase: planFile.match(/v[\d.]+-\d+/)?.[0] || 'unknown',
    plan: planFile.match(/-(\d+)-PLAN/)?.[1] || '01',
    plan_file: planFile,
    total_tasks: totalTasks,
    status: 'in_progress',
    started_at: new Date().toISOString(),
    updated_at: new Date().toISOString(),
    iteration_count: 0,
    max_iterations: 5,
    last_known_commit: null,
    tasks: taskNames.map((name, i) => ({
      number: i + 1,
      name,
      status: 'pending',
      commit_hash: null,
      completed_at: null,
      resume_attempts: 0,
    })),
  };
  const progressPath = getProgressPath(cwd);
  fs.mkdirSync(path.dirname(progressPath), { recursive: true });
  fs.writeFileSync(progressPath, JSON.stringify(progress, null, 2), 'utf8');
  return progress;
}

// Mark a task as complete
function completeTask(cwd, { taskNumber, commitHash }) {
  const progressPath = getProgressPath(cwd);
  if (!fs.existsSync(progressPath)) return null;

  const progress = JSON.parse(fs.readFileSync(progressPath, 'utf8'));
  const task = progress.tasks.find(t => t.number === taskNumber);
  if (!task) return null;

  task.status = 'complete';
  task.commit_hash = commitHash;
  task.completed_at = new Date().toISOString();
  progress.updated_at = new Date().toISOString();
  progress.last_known_commit = commitHash;

  // Check if all tasks complete
  if (progress.tasks.every(t => t.status === 'complete' || t.status === 'skipped')) {
    progress.status = 'complete';
  }

  fs.writeFileSync(progressPath, JSON.stringify(progress, null, 2), 'utf8');
  return progress;
}

// Get current status (read-only, never modifies file)
function getStatus(cwd) {
  const progressPath = getProgressPath(cwd);
  if (!fs.existsSync(progressPath)) return { status: 'no_progress_file' };
  return JSON.parse(fs.readFileSync(progressPath, 'utf8'));
}

// Increment iteration count (called ONLY by nf-precompact.js on compaction)
function incrementIteration(cwd) {
  const progressPath = getProgressPath(cwd);
  if (!fs.existsSync(progressPath)) return null;

  const progress = JSON.parse(fs.readFileSync(progressPath, 'utf8'));
  progress.iteration_count += 1;
  progress.updated_at = new Date().toISOString();

  // Check iteration cap
  if (progress.iteration_count >= progress.max_iterations) {
    progress.status = 'failed';
    progress.failure_reason = 'iteration_cap_exhausted';
  }

  // Check stuck detection: same task in_progress for 3+ iterations
  const inProgressTask = progress.tasks.find(t => t.status === 'in_progress');
  if (inProgressTask) {
    inProgressTask.resume_attempts = (inProgressTask.resume_attempts || 0) + 1;
    if (inProgressTask.resume_attempts >= 3) {
      progress.status = 'failed';
      progress.failure_reason = 'stuck_on_task';
      progress.stuck_task = inProgressTask.number;
    }
  }

  fs.writeFileSync(progressPath, JSON.stringify(progress, null, 2), 'utf8');
  return progress;
}

// Clear progress file (plan complete or abandoned)
function clearProgress(cwd) {
  const progressPath = getProgressPath(cwd);
  if (fs.existsSync(progressPath)) {
    fs.unlinkSync(progressPath);
  }
}

module.exports = {
  getProgressPath,
  initProgress,
  completeTask,
  getStatus,
  incrementIteration,
  clearProgress,
};
```

### PreCompact Hook Extension

```javascript
// In nf-precompact.js, after extractCurrentPosition() and readPendingTasks():

function readExecutionProgress(cwd) {
  const progressPath = path.join(cwd, '.planning', 'execution-progress.json');
  if (!fs.existsSync(progressPath)) return null;
  try {
    const progress = JSON.parse(fs.readFileSync(progressPath, 'utf8'));
    if (progress.status !== 'in_progress') return null;

    // Increment iteration count on compaction
    progress.iteration_count = (progress.iteration_count || 0) + 1;
    progress.updated_at = new Date().toISOString();

    // Check caps
    if (progress.iteration_count >= (progress.max_iterations || 5)) {
      progress.status = 'failed';
      progress.failure_reason = 'iteration_cap_exhausted';
    }

    fs.writeFileSync(progressPath, JSON.stringify(progress, null, 2), 'utf8');
    return progress;
  } catch (e) {
    process.stderr.write('[nf-precompact] Could not read execution-progress.json: ' + e.message + '\n');
    return null;
  }
}

function formatProgressInjection(progress) {
  if (!progress) return null;

  const completed = progress.tasks.filter(t => t.status === 'complete');
  const next = progress.tasks.find(t => t.status === 'pending' || t.status === 'in_progress');

  const lines = [
    '## Execution Progress (auto-injected at compaction)',
    '',
    `Plan: ${progress.plan_file}`,
    `Status: ${progress.status} (${completed.length}/${progress.total_tasks} tasks complete, iteration ${progress.iteration_count} of ${progress.max_iterations})`,
  ];

  if (progress.status === 'failed') {
    lines.push('');
    lines.push(`EXECUTION FAILED: ${progress.failure_reason}`);
    lines.push('Report this failure to the user. Do NOT continue execution.');
    return lines.join('\n');
  }

  // For plans with 6+ tasks, summarize instead of listing all
  if (completed.length > 5) {
    lines.push('');
    lines.push(`Tasks 1-${completed.length} complete. Resume at Task ${next ? next.number : 'N/A'}.`);
  } else if (completed.length > 0) {
    lines.push('');
    lines.push('Completed tasks:');
    for (const t of completed) {
      lines.push(`  [x] ${t.name} (commit ${t.commit_hash || 'unknown'})`);
    }
  }

  if (next) {
    lines.push('');
    lines.push('Resume at:');
    lines.push(`  [ ] ${next.name}`);
    lines.push('');
    lines.push(`IMPORTANT: Read ${progress.plan_file} and continue from Task ${next.number}.`);
    lines.push(`Do NOT re-execute Tasks 1-${completed.length} -- they are already committed.`);
  }

  return lines.join('\n');
}
```

## Open Questions

1. **What is the right max_iterations default?**
   - What we know: The TLA+ spec uses `MaxIterations` = 5 in MCAgentLoop.cfg. Phase v0.30-01 plans have 3-4 tasks each. The longest plan in v0.29 had 7 tasks.
   - What's unclear: Whether 5 iterations (compaction cycles) is too conservative for plans with many tasks.
   - Recommendation: Default to 5, configurable via `execution_progress.max_iterations` in nf.json. 5 compactions per plan is generous -- if the agent needs 5+ compactions, something is wrong.

2. **Should the progress file be gitignored?**
   - What we know: The file is transient execution state, not project state. It is cleared on plan completion.
   - What's unclear: Whether git-tracking it provides audit value or just noise.
   - Recommendation: Gitignore it. It is ephemeral runtime state like `circuit-breaker-state.json` and `quorum-in-progress`. Add `.planning/execution-progress.json` to `.gitignore`.

3. **How does this interact with subagent executors (Pattern A/B in execute-plan.md)?**
   - What we know: Pattern A spawns a fresh subagent for the full plan. Pattern B spawns per-segment. Both get fresh 200k context.
   - What's unclear: Should the subagent update the progress file, or should the orchestrator update it after the subagent returns?
   - Recommendation: Orchestrator updates progress. Subagents execute tasks and return results. The orchestrator (execute-phase.md) calls `complete-task` after each subagent returns successfully. This keeps the progress file owned by a single writer (the orchestrator).

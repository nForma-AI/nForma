# Phase v0.30-05: Continuous Verification - Research

**Researched:** 2026-03-08
**Domain:** Claude Code PostToolUse hooks, incremental verification, machine-verifiable completion conditions
**Confidence:** HIGH

## Summary

This phase adds two capabilities: (1) boundary-batched test/lint verification triggered by PostToolUse hooks during plan execution, and (2) machine-verifiable completion conditions that replace LLM judgment with programmatic checks. Both integrate into the existing hook infrastructure with zero new npm dependencies.

The PostToolUse hook event fires after every successful tool call and receives `tool_name`, `tool_input`, and `tool_response` on stdin. Three existing PostToolUse hooks (`gsd-context-monitor.js`, `nf-post-edit-format.js`, `nf-spec-regen.js`) demonstrate the exact pattern needed. The new `bin/continuous-verify.cjs` module runs checks via `spawnSync` with a 5-second timeout, injecting results as advisory `additionalContext`. A verification budget (max 3 runs per phase) prevents performance overhead, and a boundary-detection strategy batches checks instead of running them on every file write.

**Primary recommendation:** Create `bin/continuous-verify.cjs` as a standalone check runner, extend `gsd-context-monitor.js` to detect file-write accumulation boundaries, and add a `done_conditions` field to PLAN.md task schema that `execution-progress.cjs` evaluates programmatically.

### Pattern 1: PostToolUse File-Write Detection and Boundary Batching
**What:** Detect when Claude writes/edits code files via PostToolUse, accumulate a list of changed files, and trigger verification only at defined boundaries.
**When to use:** Every PostToolUse event where `tool_name` is `Write` or `Edit`.

**Boundary conditions (trigger verification):**
- (a) 5+ file writes accumulated since last verification run
- (b) A test file was just modified (`*.test.*`, `*.spec.*`)
- (c) A config file was modified (`package.json`, `tsconfig.json`, etc.)
- (d) A clean workflow boundary was detected (already implemented in `detectCleanBoundary`)

**Budget enforcement:** Maximum 3 verification runs per phase execution. After budget exhausted, skip silently.

**Example (extending gsd-context-monitor.js):**
```javascript
// Source: Synthesized from existing patterns in gsd-context-monitor.js + nf-post-edit-format.js
const WRITE_TOOLS = new Set(['Write', 'Edit']);
const TEST_FILE_RE = /\.(test|spec)\.(js|ts|cjs|mjs|jsx|tsx)$/;
const CONFIG_FILES = new Set(['package.json', 'tsconfig.json', '.eslintrc.json', 'vitest.config.ts']);

function shouldTriggerVerification(toolName, toolInput, verifyState) {
  if (verifyState.runs_used >= verifyState.max_runs) return false; // budget exhausted
  if (!WRITE_TOOLS.has(toolName)) return false;

  const filePath = (toolInput && toolInput.file_path) || '';
  const basename = path.basename(filePath);

  // Accumulate the file
  if (!verifyState.accumulated_files.includes(filePath)) {
    verifyState.accumulated_files.push(filePath);
  }

  // Check boundary conditions
  if (verifyState.accumulated_files.length >= 5) return true;
  if (TEST_FILE_RE.test(basename)) return true;
  if (CONFIG_FILES.has(basename)) return true;

  return false; // No boundary reached yet
}
```

### Pattern 2: Standalone Check Runner (bin/continuous-verify.cjs)
**What:** A CommonJS script that runs configurable verification checks (lint, typecheck, test subset) on accumulated files with a hard timeout.
**When to use:** Called by gsd-context-monitor.js via spawnSync when a boundary is reached.

**Check types:**
- `lint` -- run eslint or biome on accumulated JS/TS files
- `typecheck` -- run `tsc --noEmit` (project-wide, not file-scoped)
- `test` -- run vitest on test files in the accumulated set

**Execution model:**
```
gsd-context-monitor.js (PostToolUse)
  |-> detects file-write boundary
  |-> reads .planning/continuous-verify.json
  |-> calls: spawnSync('node', ['bin/continuous-verify.cjs', '--files', ...], { timeout: 5000 })
  |-> parses stdout JSON result
  |-> injects warning into additionalContext if violations found
  |-> saves updated state to .planning/continuous-verify.json
```

### Pattern 3: Machine-Verifiable Completion Conditions
**What:** Define "done" as a set of programmatic checks (tests pass, linter clean, type-check passes) evaluated by code, not LLM judgment.
**When to use:** At task completion in execution-progress.cjs, and at plan completion.

**Condition format in execution-progress.json:**
```json
{
  "tasks": [
    {
      "number": 1,
      "name": "Create continuous-verify.cjs",
      "status": "complete",
      "done_conditions": [
        { "type": "test_pass", "pattern": "continuous-verify.test.cjs" },
        { "type": "file_exists", "path": "bin/continuous-verify.cjs" },
        { "type": "lint_clean", "files": ["bin/continuous-verify.cjs"] }
      ]
    }
  ]
}
```

**Condition types:**
| Type | What it checks | How |
|------|---------------|-----|
| `test_pass` | Named test file(s) pass | `spawnSync('npx', ['vitest', 'run', pattern])` |
| `file_exists` | File exists on disk | `fs.existsSync(path)` |
| `lint_clean` | No lint errors on files | `spawnSync(linter, [...files])` exit code 0 |
| `typecheck_pass` | TypeScript compiles | `spawnSync('tsc', ['--noEmit'])` exit code 0 |
| `command_pass` | Arbitrary command succeeds | `spawnSync(cmd, args)` exit code 0 |

### Pattern 4: Verification State File
**What:** A JSON file tracking verification budget usage, accumulated files, and last run results.
**Where:** `.planning/continuous-verify.json` (ephemeral per execution)

```json
{
  "version": 1,
  "phase": "v0.30-05",
  "max_runs": 3,
  "runs_used": 0,
  "timeout_ms": 5000,
  "accumulated_files": [],
  "last_run": null,
  "runs": []
}
```

### Anti-Patterns to Avoid
- **Running verification on every tool call:** PostToolUse fires on Read, Grep, Glob, Bash -- not just file writes. MUST filter on `tool_name === 'Write' || tool_name === 'Edit'`.
- **Blocking execution on verification failure:** Continuous verification is ADVISORY ONLY. Never use `decision: "block"` or exit code 2. Inject warnings via `additionalContext`.
- **Running full test suite:** Only run tests related to changed files. Full suite is for phase-end gates.
- **Exceeding the 5s timeout:** If a check takes >5s, it gets killed. Design checks to be fast (single-file lint, targeted test run).
- **Creating a separate PostToolUse hook:** Consolidate into gsd-context-monitor.js to maintain single-writer semantics for `additionalContext`. Multiple hooks writing additionalContext may conflict.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Finding related test files | Custom test-file-finder | vitest's `--run` flag with glob patterns | vitest already resolves test file relationships |
| Linting specific files | Custom lint wrapper | eslint/biome CLI with file args | Both accept file lists directly; handle config resolution |
| Type checking | Custom tsc wrapper | `tsc --noEmit` | Standard TypeScript check; no custom logic needed |
| Tracking verification state | In-memory object | `.planning/continuous-verify.json` file | Must survive compaction (same reason as execution-progress.json) |
| Detecting file writes | Parsing transcript | PostToolUse `tool_name` field | Claude Code already tells us `Write`/`Edit` in the event payload |

**Key insight:** The PostToolUse hook infrastructure already provides the exact trigger point needed. The challenge is NOT detection but throttling -- preventing verification from running too often.

## Common Pitfalls

### Pitfall 1: Verification Performance Overhead
**What goes wrong:** Every Write/Edit triggers verification, adding 2-30s per tool call. A 40-tool-call phase goes from 5 to 25 minutes.
**Why it happens:** "Shift left" instinct applied uniformly without batching.
**How to avoid:** Boundary batching (5+ files OR test/config change) with max 3 runs per phase. Advisory-only (never blocking).
**Warning signs:** Phase execution time more than doubles; PostToolUse exceeds 5s.

### Pitfall 2: PostToolUse Hook Ordering Conflicts
**What goes wrong:** Multiple PostToolUse hooks inject conflicting additionalContext. Earlier research (PITFALLS.md) warns that gsd-context-monitor.js and a new continuous-verify hook would race.
**Why it happens:** Claude Code runs all matching PostToolUse hooks, but additionalContext from multiple hooks may not merge cleanly.
**How to avoid:** Consolidate verification logic INTO gsd-context-monitor.js as a new code path. Do NOT create a separate hook. Import continuous-verify.cjs as a module.
**Warning signs:** Budget tracker and verifier both injecting warnings simultaneously.

### Pitfall 3: Stale Verification State After Compaction
**What goes wrong:** Accumulated file list and budget counter are lost on compaction, causing the budget to reset and re-run all checks.
**Why it happens:** State stored only in memory.
**How to avoid:** Write verification state to `.planning/continuous-verify.json`. The nf-precompact hook already reads execution-progress.json -- same pattern.
**Warning signs:** Verification running 6+ times in a phase (3 pre-compaction + 3 post).

### Pitfall 4: False Sense of Security from Advisory Warnings
**What goes wrong:** Claude sees "lint warning" in additionalContext but ignores it and continues. User thinks verification is enforced but it's only advisory.
**Why it happens:** Advisory warnings are easily dismissed by the LLM.
**How to avoid:** Machine-verifiable completion conditions (VERF-03) act as the hard gate. Continuous verification (VERF-02) is early-warning only. The done_conditions at task completion are the enforceable check. Clear wording in additionalContext: "VERIFICATION WARNING: 2 lint errors in foo.js. Fix before completing this task."
**Warning signs:** SUMMARY.md shows "verification clean" but done_conditions evaluation finds failures.

### Pitfall 5: EventuallyTerminates Violation
**What goes wrong:** A verification check hangs (e.g., tsc on a massive codebase), the 5s timeout is not enforced, and the hook blocks the agent loop.
**Why it happens:** spawnSync timeout not set, or set too high, or the spawned process ignores SIGTERM.
**How to avoid:** Hard 5000ms timeout on spawnSync. On timeout, result = `{ status: 'timeout', check: 'typecheck' }` and verification is skipped. Fail-open: process.exit(0) in catch block.
**Warning signs:** Hook execution times >5s in conformance events.

### Verification State Read/Write
```javascript
// Source: Pattern from execution-progress.cjs (file-based state)
const VERIFY_STATE_FILE = '.planning/continuous-verify.json';

function getVerifyState(cwd) {
  const statePath = path.join(cwd, VERIFY_STATE_FILE);
  if (!fs.existsSync(statePath)) return null;
  try { return JSON.parse(fs.readFileSync(statePath, 'utf8')); }
  catch { return null; }
}

function saveVerifyState(cwd, state) {
  const statePath = path.join(cwd, VERIFY_STATE_FILE);
  fs.mkdirSync(path.dirname(statePath), { recursive: true });
  fs.writeFileSync(statePath, JSON.stringify(state, null, 2), 'utf8');
}
```

### Done Conditions Evaluator
```javascript
// Source: Synthesized from execution-progress.cjs pattern + project conventions
function evaluateCondition(cwd, condition, timeoutMs) {
  const { spawnSync } = require('child_process');
  try {
    switch (condition.type) {
      case 'file_exists':
        return { pass: fs.existsSync(path.join(cwd, condition.path)), type: condition.type };

      case 'test_pass': {
        const result = spawnSync('npx', ['vitest', 'run', condition.pattern], {
          cwd, encoding: 'utf8', timeout: timeoutMs || 30000,
        });
        return { pass: result.status === 0, type: condition.type, output: (result.stdout || '').slice(-500) };
      }

      case 'lint_clean': {
        const eslint = path.join(cwd, 'node_modules', '.bin', 'eslint');
        if (!fs.existsSync(eslint)) return { pass: true, type: condition.type, reason: 'no linter' };
        const result = spawnSync(eslint, condition.files, { cwd, encoding: 'utf8', timeout: timeoutMs || 10000 });
        return { pass: result.status === 0, type: condition.type };
      }

      case 'typecheck_pass': {
        const tsc = path.join(cwd, 'node_modules', '.bin', 'tsc');
        if (!fs.existsSync(tsc)) return { pass: true, type: condition.type, reason: 'no tsc' };
        const result = spawnSync(tsc, ['--noEmit'], { cwd, encoding: 'utf8', timeout: timeoutMs || 15000 });
        return { pass: result.status === 0, type: condition.type };
      }

      case 'command_pass': {
        const [cmd, ...args] = (condition.command || '').split(' ');
        const result = spawnSync(cmd, args, {
          cwd, encoding: 'utf8', timeout: timeoutMs || 10000,
        });
        return { pass: result.status === 0, type: condition.type };
      }

      default:
        return { pass: true, type: condition.type, reason: 'unknown condition type' };
    }
  } catch (e) {
    return { pass: true, type: condition.type, reason: 'error: ' + e.message }; // fail-open
  }
}
```

## Open Questions

1. **Config key naming: flat vs nested**
   - What we know: Project uses flat keys for config (e.g., `smart_compact_threshold_pct`, not `smart_compact.threshold_pct`) due to shallow merge semantics.
   - What's unclear: Exact config key names.
   - Recommendation: Use flat keys: `continuous_verify_enabled` (boolean), `continuous_verify_timeout_ms` (integer), `continuous_verify_max_runs` (integer), `continuous_verify_checks` (array of strings).

2. **Where to persist done_conditions**
   - What we know: execution-progress.json tracks task status. PLAN.md defines tasks.
   - What's unclear: Should done_conditions be defined in PLAN.md frontmatter and parsed at init, or defined at runtime?
   - Recommendation: Define in PLAN.md task XML (e.g., `<done_conditions>`) and parse into execution-progress.json at init time. This makes them plan-authored and version-controlled.

3. **Hook profile gating**
   - What we know: `HOOK_PROFILE_MAP` in config-loader.js controls which hooks run per profile. `gsd-context-monitor` is in `standard` and `strict` profiles.
   - What's unclear: Should continuous verification be gated by profile or by config flag?
   - Recommendation: Both. The hook logic lives in gsd-context-monitor.js (already profile-gated). The `continuous_verify_enabled` config flag provides an additional kill switch.

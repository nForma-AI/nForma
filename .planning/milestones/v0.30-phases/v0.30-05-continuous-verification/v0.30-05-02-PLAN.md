  <name>Task 1: Extend gsd-context-monitor.js with continuous verification</name>
  <files>hooks/gsd-context-monitor.js, hooks/gsd-context-monitor.test.js</files>
  <action>
Extend hooks/gsd-context-monitor.js to integrate with bin/continuous-verify.cjs for boundary-batched verification during execution. All new code follows fail-open pattern.

**Import continuous-verify.cjs (dual-path require, fail-open):**
Add near the top of the file, after existing requires:
```javascript
const continuousVerify = (() => {
  try { return require(path.join(__dirname, '..', 'bin', 'continuous-verify.cjs')); }
  catch { return null; }
})();
```

**Add config flag check:**
Read `continuous_verify_enabled` from config (default true). If false, skip all verification logic. This provides a kill switch.

**Add verification logic inside the stdin 'end' handler, AFTER the existing budget tracking block and BEFORE the "Combine all messages" section:**

1. Read verification state: `continuousVerify.getVerifyState(cwd)`. If null (first call), initialize with `continuousVerify.initVerifyState(progress?.phase || 'unknown')` and save it.

2. Call `continuousVerify.shouldTriggerVerification(input.tool_name, input.tool_input, verifyState)`.

3. If trigger returns true:
   - Call `continuousVerify.runChecks(cwd, verifyState.accumulated_files, verifyState.timeout_ms || 5000)`.
   - Increment `verifyState.runs_used`.
   - Record run result in `verifyState.runs` array.
   - Clear `verifyState.accumulated_files` (reset for next batch).
   - Set `verifyState.last_run` to ISO timestamp.
   - Save state via `continuousVerify.saveVerifyState(cwd, verifyState)`.
   - If any check failed (pass=false), build a verification message:
     ```
     VERIFICATION WARNING: {N} issue(s) found in recent code changes.
     {For each failed check: "- {type}: {truncated output (200 chars max)}"}
     Fix before completing this task. Budget: {runs_used}/{max_runs} runs used.
     ```
   - Set `verifyMessage` variable (null if all checks passed or budget exhausted).

4. If shouldTriggerVerification returned false but accumulated files changed, still save updated state (accumulated_files may have grown).

5. Add `verifyMessage` to the `messages` array alongside contextMessage, budgetMessage, compactMessage.

**Critical constraints:**
- Never use `decision: "block"` -- all verification is advisory via additionalContext only.
- Wrap entire verification block in try/catch with fail-open (set verifyMessage = null on any error).
- The verification spawnSync calls happen INSIDE continuous-verify.cjs which already has 5s timeout. The hook itself should not add additional timeout.
- Do NOT change the existing context monitor, budget, or compact logic at all.

**Extend hooks/gsd-context-monitor.test.js with new tests (using node:test):**

Add a new section `// --- Continuous verification tests ---` at the end:

1. **Verification triggers on test file write:**
   - Create tmpDir with nf.json config (`continuous_verify_enabled: true`, high warn_pct to suppress context warnings).
   - Create `.planning/continuous-verify.json` with `runs_used: 0, max_runs: 3, accumulated_files: []`.
   - Send PostToolUse payload with `tool_name: 'Write'`, `tool_input: { file_path: '/tmp/foo.test.js' }`, context_window with remaining_percentage: 90.
   - Assert exit code 0. If parsed output exists AND contains VERIFICATION, that's expected. The test primarily verifies the hook doesn't crash.

2. **Verification skipped when budget exhausted:**
   - Create `.planning/continuous-verify.json` with `runs_used: 3, max_runs: 3`.
   - Send Write tool payload with test file path.
   - Assert exit code 0 and no VERIFICATION in output (budget exhausted, should skip silently).

3. **Verification skipped when disabled via config:**
   - Create nf.json with `continuous_verify_enabled: false`.
   - Send Write tool payload.
   - Assert exit code 0 and no VERIFICATION in output.

4. **Verification skipped for non-write tools:**
   - Send payload with `tool_name: 'Read'`.
   - Assert exit code 0 and no VERIFICATION output.

5. **Fail-open when continuous-verify.cjs not loadable:**
   - This is implicitly tested when running in tmpdir without the module. The dual-path require returns null, and verification is skipped. Verify hook still exits 0 for all existing tests (regression check).

   Add a note that done_conditions are optional -- if a task has no `<done_conditions>` element, it defaults to an empty array.

2. In the task completion flow (after each task succeeds), add a new instruction block:
   ```
   ## Machine-Verifiable Completion Check

   After each task completes successfully, if the task has done_conditions defined:

   1. Run: `node bin/continuous-verify.cjs evaluate --cwd . --conditions '<json-array>'`
   2. Parse the JSON result. If `all_pass` is false:
      - Report which conditions failed
      - Attempt to fix the failing conditions (e.g., run lint fix, fix test)
      - Re-evaluate. If still failing after one retry, note in SUMMARY.md and continue (advisory, not blocking).
   3. If `all_pass` is true, proceed to mark task complete via execution-progress.cjs.

   Note: done_conditions evaluation is advisory. A task can still be marked complete even if conditions fail,
   but the executor should make a good-faith effort to satisfy them first.
   ```

3. In the `record_start_time` step, extend the execution-progress init command to support done_conditions:
   ```bash
   # If PLAN.md tasks have <done_conditions> elements, extract them:
   DONE_CONDITIONS=$(node -e "
     const fs = require('fs');
     const plan = fs.readFileSync('$PLAN_PATH', 'utf8');
     const tasks = plan.match(/<task[^>]*>[\s\S]*?<\/task>/g) || [];
     const conditions = tasks.map(t => {
       const m = t.match(/<done_conditions>([\s\S]*?)<\/done_conditions>/);
       return m ? JSON.parse(m[1].trim()) : [];
     });
     console.log(JSON.stringify(conditions));
   " 2>/dev/null || echo '[]')
   node bin/execution-progress.cjs init --plan "$PLAN_FILE" --tasks "$PLAN_TASK_COUNT" --names "$PLAN_TASK_NAMES" --done-conditions "$DONE_CONDITIONS"
   ```

4. Extend the execution-progress.cjs CLI `init` command to accept `--done-conditions` flag (JSON string of array-of-arrays). Parse it and pass to `initProgress()` as `doneConditions` parameter. If parsing fails or flag not provided, default to undefined (existing behavior, empty arrays).

# Phase v0.30-08: Continuous Learning Integration Fix - Research

**Researched:** 2026-03-08
**Domain:** Hook registration, config-loader convention, integration verification
**Confidence:** HIGH

## Summary

This phase is a gap closure phase, not a greenfield implementation. Phase v0.30-04 created all the core artifacts (nf-session-end.js, skill-extractor.cjs, memory-store.cjs, learning-extractor.cjs) but failed to properly integrate them. The v0.30 milestone audit identified four specific gaps: (1) install.js does not register the SessionEnd hook -- a re-install silently drops it, (2) skill-extractor.cjs is orphaned with no automated caller, (3) context_retrieval_enabled in quorum-slot-dispatch.cjs bypasses the config-loader two-layer merge, and (4) phase v0.30-04 has no VERIFICATION.md.

All four issues are well-scoped, have clear root causes, and require small surgical changes to existing files. No new architecture or libraries are needed. The primary risk is regression -- changes to install.js and hooks must not break existing hook registration, and the hook sync pipeline (hooks/ -> hooks/dist/ -> install) must be followed.

**Primary recommendation:** Fix install.js SessionEnd registration first (blocks LRNG-01/02/04), then wire skill-extractor into nf-session-end.js (LRNG-03), fix the config-loader bypass (tech debt), and produce VERIFICATION.md last (depends on all other fixes being verified).

### Pattern 1: Hook Registration in install.js
**What:** Each hook type (PreToolUse, PostToolUse, Stop, SessionStart, SessionEnd, etc.) follows an identical registration pattern in install.js.
**When to use:** When adding a new hook registration.
**Example:**
```javascript
// Source: bin/install.js lines 1920-1931 (Stop hook registration pattern)
// Register nForma quorum gate hook (Stop)
if (!settings.hooks.Stop) settings.hooks.Stop = [];
const hasNfStopHook = settings.hooks.Stop.some(entry =>
  entry.hooks && entry.hooks.some(h => h.command && h.command.includes('nf-stop'))
);
if (!hasNfStopHook) {
  settings.hooks.Stop.push({
    hooks: [{ type: 'command', command: buildHookCommand(targetDir, 'nf-stop.js'), timeout: 30 }]
  });
  console.log(`  ${green}+${reset} Configured nForma quorum gate hook (Stop)`);
}
```

### Pattern 2: Uninstall Path for Hooks
**What:** The uninstall section (around lines 1127-1240) must also handle removing the hook for the `--uninstall` flag to work correctly.
**When to use:** Every hook registration MUST have a corresponding uninstall entry.
**Example:**
```javascript
// Source: bin/install.js lines 1229-1239 (Stop hook uninstall pattern)
if (settings.hooks && settings.hooks.SessionEnd) {
  const before = settings.hooks.SessionEnd.length;
  settings.hooks.SessionEnd = settings.hooks.SessionEnd.filter(entry =>
    !(entry.hooks && entry.hooks.some(h => h.command && h.command.includes('nf-session-end')))
  );
  if (settings.hooks.SessionEnd.length < before) {
    settingsModified = true;
    console.log(`  ${green}+${reset} Removed nForma session end hook`);
  }
  if (settings.hooks.SessionEnd.length === 0) delete settings.hooks.SessionEnd;
}
```

### Pattern 3: Config-Loader Two-Layer Merge Convention
**What:** All config keys should use `loadConfig()` from config-loader.js, which merges DEFAULT_CONFIG -> ~/.claude/nf.json -> .claude/nf.json. Direct reads of .claude/nf.json bypass the global layer and DEFAULT_CONFIG defaults.
**When to use:** Any code that reads a config value.
**Current violation (quorum-slot-dispatch.cjs lines 921-931):**
```javascript
// BAD: reads only project .claude/nf.json, misses global config and DEFAULT_CONFIG
let retrievalEnabled = true;
try {
  const nfConfigPath = path.join(cwd, '.claude', 'nf.json');
  const nfConfig = JSON.parse(fs.readFileSync(nfConfigPath, 'utf8'));
  if (nfConfig.context_retrieval_enabled === false) {
    retrievalEnabled = false;
  }
} catch { /* fail-open */ }
```
**Fix:**
```javascript
// GOOD: uses config-loader two-layer merge
const { loadConfig } = require(configLoaderPath);
const config = loadConfig(cwd);
const retrievalEnabled = config.context_retrieval_enabled !== false;
```
**Important:** quorum-slot-dispatch.cjs already uses a dual-path require pattern for other modules (lines at top of file). The config-loader require needs the same dual-path resolution (installed path first, dev path fallback).

### Pattern 4: Skill-Extractor Integration in SessionEnd Hook
**What:** nf-session-end.js currently extracts errors (LRNG-01) and corrections (LRNG-02) but does NOT invoke skill-extractor.cjs (LRNG-03).
**When to use:** After error/correction persistence, invoke skill extraction.
**Constraints:**
- Hard timeout of 5 seconds (line 8) -- skill extraction must complete within remaining time
- Must be fail-open -- if skill-extractor fails, hook still exits 0
- skill-extractor.cjs uses `readRecentEntries()` which reads all learning categories -- can be called as a module, not just CLI
- The skill extraction is a "candidate generation" step -- actual quorum validation happens later (the LRNG-03 requirement says "Quorum-validated skill extraction" but the extraction itself just generates candidates)

### Pattern 5: Hook Sync Pipeline
**What:** Any change to files in `hooks/` MUST be synced to `hooks/dist/` then installed globally.
**Pipeline:**
```bash
cp hooks/nf-session-end.js hooks/dist/
cp hooks/config-loader.js hooks/dist/  # if modified
node bin/install.js --claude --global
```

### Anti-Patterns to Avoid
- **Direct config file reads:** Reading `.claude/nf.json` directly instead of using config-loader.js loadConfig(). This is the exact bug being fixed.
- **Forgetting uninstall path:** Adding a hook registration without a corresponding uninstall filter.
- **Forgetting hooks/dist sync:** Editing hooks/ source without copying to hooks/dist/.
- **Blocking SessionEnd:** The SessionEnd hook has a 5s hard timeout. Any new work added must be fast and non-blocking.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Config reading | Direct fs.readFileSync of nf.json | `loadConfig()` from config-loader.js | Handles missing files, malformed JSON, two-layer merge, validation |
| Hook registration detection | String matching on settings.json | Existing `.some(entry => entry.hooks && entry.hooks.some(...))` pattern | Handles nested hook structure, edge cases |
| Learning memory persistence | Custom file writes | memory-store.cjs `appendEntry()` / `isDuplicate()` | Handles dedup, JSONL format, directory creation |

**Key insight:** All the building blocks exist. This phase wires existing pieces together and fixes registration/config gaps.

## Common Pitfalls

### Pitfall 1: install.js Registration Without Uninstall
**What goes wrong:** Hook gets registered but `--uninstall` flag doesn't remove it, leaving orphaned hooks.
**Why it happens:** The registration section (~line 1840+) and uninstall section (~line 1127+) are far apart in the file.
**How to avoid:** Add both registration AND uninstall filter in the same task. Verify by running `node bin/install.js --claude --global` then `node bin/install.js --claude --global --uninstall` and checking settings.json.
**Warning signs:** `grep -c SessionEnd ~/.claude/settings.json` returns non-zero after uninstall.

### Pitfall 2: config-loader Already Required in quorum-slot-dispatch.cjs?
**What goes wrong:** Assuming config-loader.js is already required in quorum-slot-dispatch.cjs. It may or may not be -- need to verify.
**Why it happens:** quorum-slot-dispatch.cjs is a large file (~950+ lines) and uses dual-path require for other modules.
**How to avoid:** Check the top of quorum-slot-dispatch.cjs for existing config-loader requires before adding one.
**Warning signs:** Duplicate require or using wrong path.

### Pitfall 3: Skill-Extractor Timeout in SessionEnd
**What goes wrong:** skill-extractor.cjs reads multiple JSONL files and does tag clustering. If there are many entries, this could exceed the 5s SessionEnd hard timeout.
**Why it happens:** readRecentEntries reads up to 50 entries from each of 4 categories (200 entries max), then does O(n^2) tag pair computation.
**How to avoid:** Call skill-extractor functions (clusterByTags, generateCandidates) directly as module imports, not via child process spawn. The functions are fast for 200 entries.
**Warning signs:** SessionEnd hook exits with no output (killed by timeout).

### Pitfall 4: DEFAULT_CONFIG Missing context_retrieval_enabled
**What goes wrong:** Adding `context_retrieval_enabled` to DEFAULT_CONFIG but quorum-slot-dispatch.cjs still reads .claude/nf.json directly.
**Why it happens:** The fix requires BOTH adding the key to DEFAULT_CONFIG AND changing the read pattern in quorum-slot-dispatch.cjs.
**How to avoid:** Do both changes together. The default should be `context_retrieval_enabled: true` (fail-open, matches current behavior).
**Warning signs:** `grep context_retrieval_enabled hooks/config-loader.js` returns no matches after fix.

### Pitfall 5: VERIFICATION.md Scope
**What goes wrong:** Writing VERIFICATION.md that only verifies the original v0.30-04 claims without verifying the v0.30-08 fixes.
**Why it happens:** The success criteria say "Phase v0.30-04 has a VERIFICATION.md" but it must validate LRNG-01..04 satisfaction WITH evidence from the v0.30-08 fixes.
**How to avoid:** VERIFICATION.md should verify the state AFTER all v0.30-08 fixes are applied. Run actual commands to demonstrate the hooks work.

### Skill-Extractor Wiring in nf-session-end.js
```javascript
// Source: Pattern from existing module resolution in nf-session-end.js
function findSkillExtractor() {
  const candidates = [
    path.join(os.homedir(), '.claude', 'nf-bin', 'skill-extractor.cjs'),
    path.join(__dirname, '..', 'bin', 'skill-extractor.cjs'),
  ];
  for (const p of candidates) {
    try { return require(p); } catch (_) {}
  }
  return null;
}

// After error/correction persistence (around line 105):
const skillExtractor = findSkillExtractor();
if (skillExtractor && memStore) {
  try {
    const entries = skillExtractor.readRecentEntries(memStore, cwd);
    if (entries.length > 0) {
      const clusters = skillExtractor.clusterByTags(entries);
      const candidates = skillExtractor.generateCandidates(clusters);
      if (candidates.length > 0) {
        // Persist candidates as pending skills for later quorum validation
        for (const candidate of candidates) {
          if (!memStore.isDuplicate(cwd, 'skill_candidates', 'skill', candidate.skill)) {
            memStore.appendEntry(cwd, 'skill_candidates', {
              type: 'skill_candidate',
              ...candidate,
              status: 'pending_validation',
            });
          }
        }
        process.stderr.write('[nf-session-end] Generated ' + candidates.length + ' skill candidates\n');
      }
    }
  } catch (e) {
    process.stderr.write('[nf-session-end] Skill extraction error (non-fatal): ' + e.message + '\n');
  }
}
```

## Open Questions

1. **Should skill candidates be persisted to 'skill_candidates' or 'skills' category in memory-store?**
   - What we know: skill-extractor.cjs CLI `--persist` mode writes to 'skills' category with validated_by field. Candidates before validation are different from validated skills.
   - What's unclear: Whether a separate 'skill_candidates' category is cleaner than using 'skills' with a status field.
   - Recommendation: Use 'skill_candidates' category to keep unvalidated candidates separate from quorum-validated skills. This avoids polluting the validated skills store.

2. **Does quorum-slot-dispatch.cjs already require config-loader.js?**
   - What we know: It does NOT appear to require config-loader at the top of the file based on the grep results. It has dual-path requires for other modules.
   - What's unclear: Whether there is a require deeper in the file.
   - Recommendation: Verify with `grep -n 'config-loader' bin/quorum-slot-dispatch.cjs` at implementation time. Add dual-path require if missing.

3. **Should context_retrieval_enabled be added to DEFAULT_CONFIG in config-loader.js?**
   - What we know: All other kill switches (learning_enabled, continuous_verify_enabled, model_routing_enabled) are in DEFAULT_CONFIG.
   - Recommendation: Yes, add `context_retrieval_enabled: true` to DEFAULT_CONFIG for consistency. This is the whole point of the config-loader convention fix.

# Requirements: nForma v0.34

**Defined:** 2026-03-11
**Core Value:** Planning decisions are multi-model verified by structural enforcement, not instruction-following

## v0.34 Requirements

Requirements for Semantic Gate Validation & Auto-Promotion. Each maps to roadmap phases.

### Gate Naming

- [x] **NAME-01**: Gate A/B/C renamed to Wiring:Evidence / Wiring:Purpose / Wiring:Coverage in all gate JSON files (gate-a-grounding.json, gate-b-abstraction.json, gate-c-validation.json)
- [x] **NAME-02**: All scripts that read/write gate scores updated to use new field names (wiring_score instead of grounding_score/gate_b_score/gate_c_score)
- [x] **NAME-03**: Solve report, TUI, and cross-layer dashboard display new gate names
- [x] **NAME-04**: Backward-compatible migration — old field names still readable during transition, new fields written on next compute

### Semantic Scoring

- [ ] **SEM-01**: Graph BFS candidate discovery script uses existing proximity-index to find unlinked (model, requirement) pairs within 3 hops with proximity score > 0.6
- [ ] **SEM-02**: Strong candidates (score > threshold) evaluated by Haiku for semantic match — returns yes/no/maybe per candidate
- [ ] **SEM-03**: Per-gate semantic_score computed as ratio of LLM-confirmed links to total expected links (from graph discovery)
- [ ] **SEM-04**: Gate JSON files carry both wiring_score and semantic_score fields
- [ ] **SEM-05**: Semantic scoring is idempotent — re-running produces same results given same graph state and model text

### N:N Pairing

- [ ] **PAIR-01**: candidate-pairings.json generated with all discovered (model, requirement) candidate pairs, their proximity scores, and Haiku verdicts
- [ ] **PAIR-02**: /nf:resolve can load candidate-pairings.json and present graph-suggested links for human confirmation
- [ ] **PAIR-03**: Confirmed pairings written back to model-registry.json requirements arrays (closing the N:N loop)
- [ ] **PAIR-04**: Rejected pairings cached to avoid re-evaluating in subsequent runs

### Auto-Promotion

- [ ] **PROMO-01**: Solve Step 8a calls promote-gate-maturity.cjs --check after convergence to identify HARD_GATE-eligible models
- [ ] **PROMO-02**: consecutive_clean_sessions counter tracked in solve-state.json — incremented when wiring >= 1.0 AND semantic >= 0.8 AND all formal checks pass; reset to 0 on regression
- [ ] **PROMO-03**: Models auto-promoted SOFT_GATE -> HARD_GATE when consecutive_clean_sessions >= 3 AND cooldown satisfied (gate-stability.cjs)
- [ ] **PROMO-04**: All promotions logged to promotion-changelog.json with timestamp, session_id, from_level, to_level
- [ ] **PROMO-05**: Flip-flop detection prevents re-promotion of unstable models (existing gate-stability.cjs enforced)

## Future Requirements

### Semantic Depth

- **SDEP-01**: Embedding-based similarity as alternative to Haiku evaluation for cost reduction at scale
- **SDEP-02**: GOLDEN gate level above HARD_GATE — models trusted as predictive blocking gates on code changes
- **SDEP-03**: Semantic scoring for test recipe quality — does the test actually exercise the model's claimed invariant?

## Out of Scope

| Feature | Reason |
|---------|--------|
| GOLDEN gate level | Requires HARD_GATE stability proof first; defer to v0.35+ |
| Embedding index | Haiku evaluation is sufficient for 180 models; optimize later if needed |
| Pre-commit blocking | HARD_GATE enforcement on commits requires GOLDEN trust level |
| Requirement auto-generation | Graph discovers links, not new requirements — that's a different capability |

## Phase Details

### Phase v0.33-01: Time Series Foundation
**Goal**: Every nf:solve run produces a durable, machine-readable history entry that downstream phases can query for trends and oscillation detection
**Depends on**: Nothing (first phase)
**Requirements**: TRACK-01, TRACK-04, STAB-03
**Success Criteria** (what must be TRUE):
  1. After running nf:solve, a new JSONL entry appears in solve-trend.jsonl containing timestamp, iteration count, per-layer residuals, total residual, converged boolean, and gate summary
  2. When a user adds new requirements between sessions, the resulting residual increase is tagged as SCOPE_GROWTH (not regression) because requirement count is tracked alongside residuals
  3. The promotion-changelog.json contains no duplicate entries (same model, same transition within 5-minute window), and the write-time guard prevents future duplicates
**Plans**: 2 plans

Plans:
- [ ] v0.33-01-01-PLAN.md — JSONL trend append with scope-growth detection
- [ ] v0.33-01-02-PLAN.md — Changelog deduplication and write-time guard

### Phase v0.33-02: Oscillation Detection
**Goal**: The solve loop detects per-layer oscillation and enforces Option C -- no layer oscillates more than once before human escalation
**Depends on**: Phase v0.33-01
**Requirements**: TRACK-02, OSC-01, OSC-02
**Success Criteria** (what must be TRUE):
  1. After 5+ solve sessions, each layer is reported as DECREASING, STABLE, INCREASING, or OSCILLATING based on Mann-Kendall trend analysis of the last 20 entries
  2. When a layer's residual increases after a prior decrease, one oscillation credit is consumed; a second oscillation blocks automated remediation for that layer and triggers human escalation
  3. When an upstream layer was remediated in the same or prior session, temporary residual increases in downstream layers are allowed without consuming oscillation credits (cascade-aware grace period)
  4. The run-collapse oscillation detection algorithm terminates on every invocation -- consistent with the formal AlgorithmTerminates invariant (enabled actions eventually fire under weak fairness)

### Phase v0.33-03: Gate Stabilization
**Goal**: Gate promotions are stable -- models that flip-flop between promoted and demoted states are flagged and require a cooldown period before re-promotion
**Depends on**: Phase v0.33-01
**Requirements**: STAB-01, STAB-02, INTG-01
**Success Criteria** (what must be TRUE):
  1. Models with 3 or more promote/demote alternations in promotion-changelog.json are flagged as UNSTABLE in per-model-gates.json
  2. Re-promotion of a flagged model is blocked until the configurable stabilization window passes (default: 3 consecutive solve sessions AND 1 hour wall time of sustained gate score)
  3. nf:solve runs --write-per-model by default during every run, persisting gate reasons to per-model-gates.json without requiring an explicit flag
**Plans**: 2 plans

Plans:

- [ ] v0.33-03-02-PLAN.md — Integration into compute-per-model-gates.cjs and --write-per-model default

### Phase v0.33-04: Predictive Power
**Goal**: Formal models are scored on their ability to predict real bugs, providing a feedback loop between observed failures and formal verification effectiveness
**Depends on**: Phase v0.33-01, Phase v0.33-02, Phase v0.33-03
**Requirements**: PRED-01, PRED-02, TRACK-03
**Success Criteria** (what must be TRUE):
  1. A bug-to-property.json mapping exists linking observed bugs (from GitHub issues, test failures, observe skill) to formal properties that could have caught them
  2. Per-model recall score (bugs_predicted / total_relevant_bugs) is computed and reported as an informational metric in the solve report (not used as a gate input)
  3. After 10+ data points, convergence velocity estimation reports estimated sessions-to-convergence per layer via exponential decay fit
**Plans**: 2 plans

Plans:
- [ ] v0.33-04-01-PLAN.md — Bug-to-property linking module with recall computation

### Phase v0.33-05: TLA+ Meta-Verification
**Goal**: The outer solve loop is formally modeled in TLA+ with TLC-verified safety and liveness properties proving Option C prevents unbounded oscillation and the loop eventually converges
**Depends on**: Phase v0.33-02, Phase v0.33-03, Phase v0.33-04
**Requirements**: FV-01, FV-02, FV-03
**Success Criteria** (what must be TRUE):
  1. NFSolveConvergence.tla models the outer solve loop with cross-session state (bounded history), Option C blocking rule, and gate maturity transitions
  2. TLC verifies the safety property: no layer oscillates more than once (oscillation_count[layer] <= 1 for all layers) with zero counterexamples
  3. TLC verifies the liveness property: solve loop eventually converges (converged = TRUE) under explicit fairness assumptions, passing the liveness fairness CI lint
  4. The convergence liveness property uses explicit fairness declarations -- consistent with the formal ConvergenceEventuallyResolves invariant pattern (weak fairness on enabling actions ensures the check eventually completes)
**Plans**: 2 plans

Plans:
- [ ] v0.33-05-01-PLAN.md — TLA+ spec, TLC config, and fairness declarations for outer solve loop
- [ ] v0.33-05-02-PLAN.md — Runner integration, TLC verification of safety + liveness properties

### Phase v0.33-06: Solve Report Integration
**Goal**: Users see convergence status, trends, and actionable recommendations directly in nf:solve output without running separate tools
**Depends on**: Phase v0.33-02, Phase v0.33-04
**Requirements**: INTG-02, OSC-03
**Success Criteria** (what must be TRUE):
  1. The solve report includes a convergence section showing per-layer trend sparklines, oscillation status, and top-3 action items derived from trend data
  2. When the oscillation breaker fires, a Haiku-based escalation classifier reports root cause as GENUINE_REGRESSION, MEASUREMENT_NOISE, or INSUFFICIENT_EVIDENCE with structured context (layer, delta, git diff)

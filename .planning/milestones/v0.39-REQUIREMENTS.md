## Milestone v0.39 Requirements

Requirements for Dual-Cycle Formal Reasoning. Each maps to roadmap phases.

### Foundation

- [ ] **FND-01**: Verification mode tagging — each model checker invocation is tagged as 'diagnostic' (Cycle 1, inverted semantics) or 'validation' (Cycle 2, normal semantics) to prevent mode contradiction
- [ ] **FND-02**: Trace comparison infrastructure — TLC counterexample traces parsed into structured JSON (ITF format) with state-by-state field extraction for diff generation
- [ ] **FND-03**: Configurable iteration limit — --max-iterations flag (default 3) controls both diagnostic refinement (Cycle 1) and simulation loop (Cycle 2), persisted in .planning/config.json

### Cycle 1 — Model Diagnostic

- [ ] **DX1-01**: Diagnostic diff generation — when a model fails to reproduce a known bug, generate targeted mismatch: "model assumes X but bug shows Y" by comparing model state space against observed bug trace
- [ ] **DX1-02**: Workflow reframing — all model-driven-fix and refinement-loop language changed from "bug not captured" to "model is incomplete/wrong" — the bug is ground truth, the oracle
- [ ] **DX1-03**: Quorum diagnostic injection — diagnostic diff context injected into quorum worker prompts during refinement iterations so workers can reason about what the model is missing
- [ ] **DX1-04**: Guided refinement — diagnostic diff used to propose specific model corrections (which invariants to add/modify, which state variables are missing) instead of blind retry

### Cycle 2 — Solution Simulation

- [ ] **SIM-01**: Fix intent normalization — accept fix ideas as natural language, constraint sets, or code sketches and normalize internally to formal model mutations
- [ ] **SIM-02**: Consequence model generation — given a reproducing model + normalized fix intent, generate a consequence model representing the system after the proposed fix is applied
- [ ] **SIM-03**: Three-gate convergence — automated convergence check: (1) original invariants still hold, (2) bug no longer triggered (inverted check passes), (3) no 2-hop neighbor regressions
- [ ] **SIM-04**: Simulation loop UX — Phase 4.5 in model-driven-fix.md with iteration display, per-gate pass/fail status, convergence progress, and escalation to user after max iterations

## Future Requirements

### Deferred

- **STG-01**: Tentative model staging area — draft/verified/promoted lifecycle for consequence models
- **STG-02**: Multi-candidate fix comparison — generate multiple consequence models for competing fix ideas, compare side-by-side

## Out of Scope

| Feature | Reason |
|---------|--------|
| Auto-fix enumeration | Brute-force fix search is computationally intractable and produces low-quality solutions |
| Parallel simulation iterations | Sequential iteration with diagnostic feedback produces better convergence than parallel exploration |
| User-editable model mutations | Direct mutation editing risks unsatisfiable specs; normalize through intent pipeline instead |
| New model checker integrations | Existing TLA+/Alloy/PRISM coverage sufficient for v0.39 |

### New Finding (from this re-audit integration check)

**TECH-01 — IS_GAP_CLOSURE false-positive (non-blocking latent risk)**
- **File:** `qgsd-core/workflows/transition.md` line 458
- **Pattern:** `grep -A 15 "Phase ${COMPLETED_PHASE}" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*'`
- **Issue:** `-A 15` matches everywhere in ROADMAP.md — including `"Depends on: Phase X"` lines within downstream phase details sections. When a downstream gap closure phase depends on primary phase X, X's name appears within 15 lines of the downstream phase's `"**Gap Closure:**"` marker, producing false-positive IS_GAP_CLOSURE count.
- **Observed:** v0.13-01 → 1 (false positive), v0.13-02 → 2 (false positive), v0.12-10 → 1 (false positive)
- **Effect:** Primary phases that are the last phase in a milestone would route to Step 2a (re-audit path) instead of Step 2b (primary completion path) — wrong-but-recoverable: audit-milestone still invoked, one extra invocation
- **Not triggered in v0.13:** The last phases (v0.13-03 and v0.13-04) ARE gap closure phases, so IS_GAP_CLOSURE=1+ is correct for them
- **Fix (for future milestone):** Anchor grep to Phase Details heading: `grep -A 3 "^### Phase ${COMPLETED_PHASE}:" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*'`
- **Severity:** Low (wrong-but-recoverable routing; no loop or halt)
- **Affects:** LOOP-01/LOOP-02 behavioral correctness in primary-phase last-step scenarios

**Impact:** Wrong-but-recoverable routing on primary-phase last-step transitions. Not triggered in v0.13. Recommend fixing before v0.14.

```bash
IS_GAP_CLOSURE=$(grep -A 15 "Phase ${COMPLETED_PHASE}" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*')
```

```bash
IS_GAP_CLOSURE=$(grep -A 3 "^### Phase ${COMPLETED_PHASE}:" .planning/ROADMAP.md | grep -c '\*\*Gap Closure:\*\*')
```
